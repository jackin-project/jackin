# Implementation Plans — TUI reusability refactoring program

Generated by the improve skill (deep TUI audit) on 2026-07-03, at commit `a2ec1b237`.
Execute in the order below unless dependencies say otherwise. Each executor: read the
plan fully before starting, honor its STOP conditions, and update your row when done.

Program goal: one shared component implementation per visual pattern, one intuitive
component API, identical hint/dialog/settings/editor presentation across the capsule,
host console, and launch TUI — per the binding canon in
`docs/content/docs/reference/tui/` (hard reuse rule, hints contract, settings↔editor
parity, ErrorPopup-only errors). The canon already exists; these plans make the code
match it and close the gaps the canon has.

Repo hard rules that bind every plan: never commit `main` — propose the plan's branch
and get operator confirmation first; `git commit -s` (DCO) and push after every
commit; `cargo nextest run` (never `cargo test`); TUI-behavior changes update the
matching `docs/content/docs/reference/tui/` page in the same PR; module moves/splits
update the codebase map in the same PR.

## Findings considered and rejected / cleared (do not re-audit)

- **hint_bar vs status_footer vs bottom_chrome vs status_popup "overlap"** — complementary components, not duplicates: hint_bar is the only keybinding-hint renderer, status_footer is the white status/chip bar, bottom_chrome is rect layout, status_popup is a modal.
- **Capsule keymap.rs as a fork** — it reuses the shared `Keymap`/`KeyBinding` engine and declares only capsule-specific binding tables; correct extend-don't-fork usage.
- **`crates/jackin` ↔ `crates/jackin-console` split as a duplication seam** — documented thin-adapter facade ("Transitional root-console TUI facades"); boundary is coherent.
- **Console footer-hints pipeline** — builds shared `HintSpan`s, renders via shared `render_wrapped_hint_bar`; clean.
- **panel vs scrollable_panel** — composition, not duplication.
- **Launch rain / progress rail / loading ripple as forks** — genuinely launch-local decorative widgets; correctly surface-local (catalog wording fixed in plan 001).
- **"32 hardcoded colors in the shared crate"** — overcount; most sites are `ansi_text.rs`'s by-design SGR decoding table. Real sites handled by plans 004/006.
- **Replacing per-dialog semantic focus enums (`ConfirmFocus`, `SaveDiscardFocus`) with a generic index** — rejected: they encode impossible-states-unrepresentable; plan 003 shares the *behavior* (cycling/index mapping), keeps the types.
- **`agent_choice` as a two-button-rule violation** — code is a generic variable-cardinality picker (`choices: Vec<A>`); the catalog prose mislabeled it. Docs fixed in plan 001; making it two-button is a product decision deliberately not planned.
- **`save_preview` should reuse `diff_view`** — expected verdict: distinct abstractions (semantic config-change summary vs text-diff renderer); plan 016 Step 0 confirms and records it in module docs.
- **`run_console` size as a lint violation** — it carries a written `#[allow(too_many_lines)]` justification; plan 017 proceeds for testability + terminal-safety reasons, not lint compliance, at P3.
- **Merging `ConfirmSaveState` into the shared save/discard dialog** — rejected: distinct component (scrollable preview + typed payload); it adopts the keymap discipline instead (plan 014).
- **Capsule `Vec<Dialog>` stack as a problem** — same semantics as the new ModalStack; migration is optional follow-up noted in plan 012, not debt.
