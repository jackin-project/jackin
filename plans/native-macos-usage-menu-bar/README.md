# Native macOS usage menu-bar program plans

Generated by the improve skill on 2026-07-21 against commit `7a52a273`; extended 2026-07-21 against commit `3c49fff0` after an operator-directed concept review against CodexBar (github.com/steipete/codexbar) and OpenUsage (github.com/robinebers/openusage). This program now has three tracks, in operator-set priority order: **(1) one host-global usage cache** shared by the menu bar, host CLI, and every container; **(2) native Tahoe-grade look and feel**; **(3) the recorded distribution residual**. No track reopens provider scope or moves probe logic into Swift.

## Dependency notes

- **005 runs first** (operator directive, 2026-07-21): a single source of truth for usage on the host — refresh anywhere, visible everywhere, at most one prober per account host-wide — outranks all cosmetic and distribution work. It is pure Rust + docs and conflicts with nothing else.
- 006 prefers 005 landed so the UI copy ("numbers match every jackin❯ surface") is true when written, and touches only Swift + a small additive FFI display API.
- 001 and 002 can run in parallel with each other (different repositories) and with 006 (001 owns packaging/scripts, 006 owns Swift sources; only a trivial `Package.swift` target listing may overlap — coordinate that one hunk).
- 003 must not begin credentialed publication until 001 and 002 land. 004 stays a separate production-evidence gate. The jackin❯ program owner — not the external tap executor — reconciles this README after the tap PR lands.

## Operator decisions required before Plan 003

Record the approved values in Plan 003 before execution. Do not let an executor infer them.

1. Confirm permanent app identity: bundle identifier `com.jackin-project.usage-menu-bar`, visible app name, signing Team ID, and cask token `jackin-usage-menu-bar`.
2. Approve a numeric `CFBundleVersion` policy. Recommendation: full-history commit count, with main-ancestry and Apple-format guards; do not use a run number that changes when a release is retried.
3. Approve immutable release recovery: never replace a published signed ZIP; conflicting bytes require a new version.
4. Choose App Store Connect API-key type. Team keys require issuer ID; individual keys omit it. Record the exact `notarytool` argument contract without recording secret values.
5. Choose secret governance. Recommendation: protected GitHub release environment, GitHub-hosted macOS runner only, with required operator approval before Apple credentials are exposed.
6. Provide Developer ID PKCS#12 secret, password, expected certificate SHA-256 fingerprint, Team ID, App Store Connect `.p8` secret, key ID, and issuer when applicable. Record credential owner and rotation/expiry responsibility, never secret values.
7. Approve tap authority. Recommendation: narrowly scoped GitHub App instead of a broad personal token; the writer may open/update a PR but cannot bypass required checks or merge the first cask.
8. Decide whether actual launch coverage on both Intel and Apple Silicon macOS 14 runners is required. Recommendation: require both before claiming universal runtime proof; if runner capacity is unavailable, keep Plan 004 blocked rather than substituting architecture inspection for execution.

## Program boundaries

- Provider scope is frozen by ADR-011: `Agent::ALL` + Z.AI/GLM + MiniMax. CodexBar and OpenUsage are **concept references only** (status-item conventions, cache honesty semantics, glass adoption patterns) — never a provider list to import, never code to copy.
- Rust owns probes, cache, cooldown, severity, and view shaping; Swift is display-only. The one cache lives on the host and is bind-mounted into containers; no daemon and no new IPC channel in this program.
- Sparkle, DMG/PKG packaging, app sandboxing, per-architecture ZIPs, daemon integration, new providers, WidgetKit, multi-status-item layouts, and speculative entitlements are out of scope. The roadmap accepts cask-only updates.
- The release artifact is `jackin-usage-menu-bar-<VERSION>-universal-apple-darwin.zip`. It contains exactly `JackinUsageMenuBar.app` and is created after stapling.
- The app remains macOS 14+, `LSUIElement`, no root daemon; Tahoe Liquid Glass arrives via the macOS 26 SDK with centralized availability fallbacks, not a raised floor.
- The final release ZIP receives three file sidecars — `.sha256`, `.bundle`, and `.sbom.json` — plus a GitHub-hosted build-provenance attestation record under the existing release trust policy.

## Considered and rejected

- **Rewrite probes in Swift to make the app "pure Swift" end to end:** rejected. Display parity with jackin❯ and the single-cache invariant both require the shared Rust core; a Swift probe matrix is exactly the parallel-copy problem this program exists to eliminate (and ADR-011 forbids it). "Pure Swift" is satisfied at the application layer: the app is a native SwiftPM SwiftUI app, no web views, no cross-platform shells.
- **Daemon-owned usage cache with a socket API (v1):** deferred. The filesystem coordination layer (account-keyed snapshots + cooldowns + best-effort locks on one shared root) already exists and satisfies refresh-anywhere semantics; the roadmap keeps the daemon as an optional later upgrade.
- **Migrate `MenuBarExtra` to AppKit `NSStatusItem` + `NSPanel` now:** deferred until evidence. CodexBar/OpenUsage did it for key-window and positioning control, but jackin❯'s popover is simpler; Plan 006 carries an explicit STOP condition if `MenuBarExtra(.window)` misbehaves on macOS 26.
- **Keep the dylib-based local build and add a separate static release build:** rejected because two assembly architectures would drift and the dylib path does not implement ADR-011's staticlib/XCFramework decision.
- **Publish the current pre-stapling ZIP:** rejected because the staple changes the app after archive creation; Homebrew must receive a ZIP created from the final stapled app.
- **Treat release + formula version as one `published` boolean:** rejected because partial releases need repair without regenerating timestamped Apple signatures.
- **Auto-merge the first cask from the secrets-bearing release job:** rejected because the tap has no cask precedent and installation validation must run without tap write credentials.
- **Ship Sparkle now:** not worth doing; the roadmap says cask-only distribution is acceptable.
