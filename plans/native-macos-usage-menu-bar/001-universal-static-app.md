## Why this matters

The app works locally, but the distributable is host-architecture-only and embeds `libjackin_usage_ffi.dylib`. The accepted architecture says UniFFI crosses a Rust staticlib/XCFramework boundary, and a future release claims universal Intel + Apple Silicon support. One secret-free, universal, statically linked assembly path must become the source of truth before signing automation is safe; otherwise PRs cannot predict release output and local/release packaging will drift.

## Current state

- `scripts/build-usage-xcframework.sh:13-65` builds arm64 and x86_64 Rust staticlibs, combines them into `JackinUsageFFI.framework`, and creates an XCFramework, but no app consumes it. Its framework module name does not yet match generated Swift's conditional `import jackin_usage_ffiFFI`.
- `native/Package.swift:4-30` links `-ljackin_usage_ffi` from host `target/release`, while `scripts/build-usage-menu-bar-app.sh:13-43` copies the host dylib and rewrites it to `@rpath`. The output architecture therefore equals the build host.
- `scripts/generate-usage-swift-bindings.sh:23-27` performs an ad-hoc `cargo install` when `uniffi-bindgen` is absent. Workflow policy requires CI tools to be pinned in `mise.toml` and installed through `jdx/mise-action`.
- `native/Tests/JackinUsageBridgeTests/ArchitectureTests.swift:6-56` is the existing Swift boundary test: no provider probes, HTTP, OAuth, or providers outside jackin❯ scope in handwritten Swift. Preserve it.
- `.github/workflows/hygiene.yml:127-161` has a scheduled macOS Rust workspace smoke job, but it does not build Swift, run Swift tests, assemble an app, or participate in required PR CI.
- `.github/workflows/ci.yml:57-190` owns path routing; `ci-required` at lines 1102-1126 aggregates required jobs. Native validation belongs there, path-routed and added to the aggregator.

## Steps

### Step 1: Make the generated C module and XCFramework consumable by SwiftPM

Pin the workspace-compatible `uniffi-bindgen` in `mise.toml`; remove ad-hoc installation from `generate-usage-swift-bindings.sh` and fail with an actionable message when the pinned tool is absent. Make `build-usage-xcframework.sh` produce a static XCFramework whose Clang module is exactly `jackin_usage_ffiFFI`, matching `native/Sources/JackinUsageBridge/jackin_usage_ffi.swift:10-12`. Preserve both `arm64` and `x86_64`, macOS 14 deployment, generated header, and module map. Add fail-closed `lipo`/`plutil` checks for both slices.

Change `native/Package.swift` to consume the local binary target/XCFramework and remove the host `target/release` dylib search path. The XCFramework must be a build input, not copied into the finished `.app`. Do not hand-edit generated UniFFI bindings.

**Verify**: build the XCFramework, then `lipo -archs <path-to-framework-binary>` must print both `x86_64 arm64` in either order; `cd native && swift test -c release` exits 0.

### Step 2: Assemble one universal, statically linked app

Rewrite `build-usage-menu-bar-app.sh` around the XCFramework path. Build both Swift executable slices (or one SwiftPM universal executable when proven equivalent), combine only when necessary, and fail unless the final executable contains both architectures. Generate the finished plist before any signing with exact bundle ID `com.jackin-project.usage-menu-bar`, executable `JackinUsageMenuBar`, `LSUIElement=true`, minimum macOS `14.0`, numeric `CFBundleShortVersionString` from required `JACKIN_APP_VERSION`, and numeric `CFBundleVersion` from required `JACKIN_APP_BUILD`.

Ad-hoc sign the local/PR app only after assembly so strict code-sign verification can exercise the final shape. The app must not contain a dylib, `.a`, framework, or XCFramework; `otool -L` must contain no repository, `target/`, or `libjackin_usage_ffi` path. Preserve the existing output path to avoid needless developer churn.

**Verify**: the build fixture and app-verification commands in the command table exit 0; `find native/dist/JackinUsageMenuBar.app -type f` shows only the executable, plist, signature metadata, and genuine resources.

### Step 3: Centralize fail-closed app validation

Add `scripts/verify-usage-menu-bar-app.sh` because PR assembly, release signing, and post-download checks need the same contract. It must accept expected version/build via environment, then verify: exact plist fields; both executable slices; each slice's encoded minimum deployment target is exactly 14.0 where the Mach-O tool reports a normalized version, and never newer than the exact plist floor; no embedded FFI binary or absolute build linkage; `codesign --verify --deep --strict`; ZIP round-trip when a ZIP argument is supplied; and exactly one top-level `JackinUsageMenuBar.app` in an archive. Keep Apple Developer/Gatekeeper assertions optional only through an explicit release mode; default PR mode must not pretend an ad-hoc signature is notarized.

Extend `ArchitectureTests.swift` only for source-level invariants that belong in Swift tests. Do not duplicate shell-level Mach-O/package checks in XCTest.

**Verify**: corrupt one copied fixture at a time (remove one architecture, alter bundle ID, inject an absolute dylib reference where practical) and confirm the verifier exits nonzero; remove temporary fixtures afterward.

### Step 4: Add a required, secret-free native PR gate

Add a `native` path output to `.github/workflows/ci.yml` covering `native/**`, the three native scripts plus verifier, `crates/jackin-usage/**`, `crates/jackin-usage-ffi/**`, `Cargo.toml`, `Cargo.lock`, Rust toolchain/mise inputs, and native/release workflow definitions. Add one GitHub-hosted macOS job that installs pinned tools through mise, runs Rust usage/FFI tests, regenerates bindings and fails on a tracked diff, runs Swift tests, builds the universal fixture, verifies the app and ZIP round-trip, and launches the app without provider credentials long enough to prove it remains alive before terminating it. It must not receive Apple or tap secrets and must never publish.
