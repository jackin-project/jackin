## Program-wide rules (bind every plan)

1. **One branch, one PR.** All `jackin❯` work lands as checkpoint commits on `feature/shared-tui-extraction` inside PR #794, which stays open until every stage and acceptance gate passes ([Decision 18](../../docs/content/docs/reference/research/shared-tui-extraction/05-decision-record.mdx)). Never open a second `jackin❯` PR for this program. Never commit to `main`.
2. **Every `jackin❯` commit is signed and pushed immediately**: `git commit -s -m "<type>(scope): …"` (Conventional Commits), then `git push` (CONTRIBUTING.md). No force pushes without explicit operator approval in-conversation.
3. **TermRock lives at `tailrocks/termrock`.** It is prepared in a dedicated local clone. Every new TermRock-authored commit after the recorded provenance boundary is DCO-signed and buildable; green checkpoints push **directly to TermRock `main`** with no TermRock PR — this is the approved bootstrap exception surfaced in the Stage 0 launch summary. Never force-push or rewrite published TermRock `main`; repair forward with new commits.
4. **`jackin❯` consumes TermRock only by exact revision**: Cargo `rev = "<full-sha>"` plus committed `Cargo.lock`. Never a branch dependency during this program.
5. **Bug-compatible extraction.** No rendering/behavior change rides along with the ownership move. Known defects live in the quality backlog (plan 001) and are fixed only in Stage 5 (plan 009). A parity diff that traces to an unrecorded "improvement" fails the gate.
6. **Scope fence.** Only `tailrocks/termrock` and `jackin❯` are touched. No checkout, change, build, or validation of Holla, Velnor, Parallax, TableRock, or any other repository. No new components beyond the donor set. No capability-degradation code (256/16-color, `NO_COLOR`, RTL/BiDi).
7. **Brand.** In every prose surface write `jackin❯` (never bare `jackin` for the brand); plain `jackin` only for code identifiers, paths, crates, env vars.
8. **Blocker protocol.** A newly discovered security defect, data-loss path, or panic on a documented valid input returns the program to Stage 0 on the same branch: fix the donor, regenerate evidence, re-freeze ([Decision 13](../../docs/content/docs/reference/research/shared-tui-extraction/05-decision-record.mdx)). A newly discovered non-blocking visual defect is appended to the quality backlog and carried through parity.

## Dependency notes

- Strictly sequential. Each plan ends at a dossier stage exit gate (or a bounded checkpoint inside Stage 2/4); the next plan assumes that gate passed. Stage boundaries are checkpoint commits, not merges.
- 003–005 all operate inside the dedicated TermRock extraction clone; nothing is pushed to the public repository until plan 005's final step.
- 007–008 repeatedly repin TermRock revisions; each repin reruns the whole `jackin❯` workspace because Ratatui-type changes can surface failures outside direct importers ([ch. 08, migration slices](../../docs/content/docs/reference/research/shared-tui-extraction/08-migration-evidence-and-gates.mdx)).
- At every stage boundary, compare the frozen donor revision `33896a504e19ef13adb8692550c1845cb86a9504` with current `origin/main`; merge-sync per [Decision 20](../../docs/content/docs/reference/research/shared-tui-extraction/05-decision-record.mdx) when upstream touched donor modules, consumers, fixtures, the dependency cell, or canonical TUI docs.
- Recorded sequencing deviation: ch. 04 lists the Fumadocs *application* under Stage 2 repository engineering; these plans scaffold it at the start of plan 006 (Stage 3) so Stage 2 checkpoints stay Rust-focused. No Stage 2 exit-gate item involves the docs application, and `docs-required` carries a placeholder until plan 006 — see the note in plan 006 "Why this matters".

## Findings considered and rejected

- **Publishing TermRock to crates.io during this program** — explicitly optional per the Decision Record; adoption artifact is an immutable Git revision. Registry publication is a separate operator decision.
- **A `termrock-testing` crate** — rejected by Decision 6 until two consumers demand the same helper API.
- **Windows CI lane** — excluded by Decision 9; adding it is a new operator decision.
- **Capsule status-bar/hint-row convergence onto shared `Tabs`/`HintBar`** — tracked post-parity follow-up, never a migration requirement (ch. 08).
