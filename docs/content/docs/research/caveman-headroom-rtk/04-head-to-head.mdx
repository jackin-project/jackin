# 04 — Head-to-head: where each wins, what each lacks

The three design teardowns ([caveman](/research/caveman-headroom-rtk/01-caveman-design/), [headroom](/research/caveman-headroom-rtk/02-headroom-design/), [RTK](/research/caveman-headroom-rtk/03-rtk-design/)) described each tool on its own terms. This page sets them against each other directly: a feature-by-feature has/lacks matrix, the internals side-by-side, and a clear statement of the *best case* for each — the workload where it beats the other two outright.

The framing to keep in mind throughout: **these are three points on one pipeline, not three solutions to one problem.** Caveman is output; headroom and RTK are two opposite points on input. So most "comparisons" between caveman and the others are category errors — the real rivalry is the narrow part of the overlap between headroom and RTK on tool-output compression, and even there they act at different interception points.

## The feature matrix — has (✓), lacks (✗), partial (◐)

| Capability | Caveman | Headroom | RTK |
|---|:---:|:---:|:---:|
| **Compresses output** (what the model writes) | ✓ | ◐ (optional shaper, off by default) | ✗ |
| **Compresses input** (what the model reads) | ✗ | ✓ (broad) | ✓ (Bash only) |
| Reaches native `Read`/`Grep`/`Glob` | ✗ | ✓ | ✗ |
| Reaches RAG chunks | ✗ | ✓ | ✗ |
| Reaches conversation history | ✗ | ✓ | ✗ |
| Reaches shell / test / build / log output | ✗ | ✓ | ✓ |
| Touches **thinking** (20% of dollars) | ✗ | ✗ | ✗ |
| Hits the 5×-priced output class | ✓ | ◐ | ✗ |
| **Deterministic** (no ML in the loop) | ✓ (it is a prompt) | ✗ (`kompress-base`) | ✓ |
| **Reversible / recoverable** compression | ✗ (re-ask) | ✓ (CCR `headroom_retrieve`) | ◐ (tee on failure only) |
| Language-aware code outlining (per-read) | ✗ (passes code verbatim) | ✓ | ✓ (8 languages) |
| Persistent queryable symbol index | ✗ | ✗ | ✗ |
| Cross-agent shared memory | ◐ (cavemem, single-agent, lossy) | ✓ (dedup, reversible) | ✗ |
| Failure-mining into memory files | ✗ | ✓ (`learn`) | ✗ |
| **Cache-safe** on Claude Code | ✓ (output side, always) | ◐ (MCP/library yes; proxy risky) | ✓ (by construction) |
| Zero MCP schema rent | ✗ (~940 tok skill listing) | ✗ (in MCP mode) | ✓ |
| Zero host-state write | ✗ (2 hooks) | ◐ (config + model download) | ✗ (PreToolUse hook) |
| Zero runtime compute | ✓ | ✗ (P50 52 ms / P99 4.2 s) | ◐ (~5–15 ms/cmd) |
| Single self-contained artifact | ◐ (plugin + hooks) | ✗ (Rust core + ML runtime + Python) | ✓ (one ~4.1 MB binary) |
| CI-safe (preserves exit codes) | n/a (output side) | n/a | ✓ |
| Multi-surface ecosystem (commit/review/memory/subagents) | ✓ (the broadest *family*) | ◐ (memory + learn) | ✗ (one job) |
| Whole-session telemetry | ✗ | ✓ (50k+ sessions) | ✗ |
| Independent third-party benchmark | ✗ | ◐ (one: 47.5%) | ✗ |
| Locally reproduced headline | ✓ (58.5% output) | ◐ (mechanisms yes; product no) | ✗ |

Read the matrix as three near-disjoint capability sets with one genuine overlap:

- **Only caveman compresses output.** Headroom's output shaper is off by default and not its strength; RTK does not touch output at all.
- **Only headroom reaches the non-shell input sources** — native reads, RAG, history. This is the column where it stands alone.
- **The overlap is exactly one row** — "reaches shell/test/build/log output" — where both headroom and RTK say ✓. That overlap is the entire substance of "headroom vs RTK," and even there they act at different points (RTK at the Bash boundary, headroom on the API request), so they compose rather than collide.
- **The bottom of the matrix is where the three diverge most**: reversibility and cross-agent memory and `learn` are headroom-only; determinism + zero-MCP + single-binary + CI-safety are RTK's cluster; zero-runtime + output-class + the broad ecosystem are caveman's.

## The internals side-by-side

| Primitive | Caveman | Headroom | RTK |
|---|---|---|---|
| Interception point | Model's own decoder (a prompt rule) | API request (proxy) or observation (MCP/lib) | Bash tool boundary (PreToolUse hook) |
| Engine type | **Markdown instruction** (no code path) | Router + typed compressors + ML model | 12 deterministic Rust filters keyed on the command |
| Parser / structural | none | per-type (AST outline, JSON, log) | per-command + a `filter.rs` code filter (8 langs) |
| ML in the loop | **No** | **Yes** (`kompress-base`, auto-downloaded) | **No** |
| Persistent state | none (hooks only track tokens) | CCR store + cross-agent memory + `learn` | SQLite history (`rtk gain`) |
| Token counter | tiktoken `o200k_base` (eval only) | own counter, no stated tokenizer | ~4 chars/token heuristic |
| Recovery on loss | **none** (re-ask) | **CCR `headroom_retrieve`** (reversible) | tee **on failure only** |
| Host-state write | `~/.claude` hooks ×2 | MCP/proxy config + model download | `~/.claude` PreToolUse hook |
| Runtime cost | ~0 compute + ~940-tok prefix | P50 52 ms / P99 4.17 s + ML + MCP rent | ~5–15 ms/cmd, ~4 MB binary |
| Hardest failure | over-terse, unrecoverable | ML drops an identifier; proxy cache-bust | truncates a needed line on a *successful* command |

The teardown confirms the [determinism gradient](/research/caveman-headroom-rtk/) from a new angle: caveman is a zero-machinery prompt; RTK is maximum determinism (fixed rules, no model, single binary); headroom is the only one that buys *breadth* by paying for an ML stage, a proxy, and a reversible store. More machinery → more reach and reversibility, but also more latency, more host effects, and a real attack surface.

## Where each one wins — the best case for each

The operator's question was: if no single product does all three, show the best case of each, where it does things much better than its competitors. Here they are.

### Caveman wins when the waste is the model talking too much

```text
   BEST CASE: CAVEMAN
   ───────────────────
   symptom   the model writes long explanations, restates code it just
             edited, narrates what it is about to do
   why it    output is the 5×-priced token class AND cache-neutral, so every
   wins      token shaved is worth ~5× an input token and costs nothing in
             cache risk; it is a free prompt with zero runtime
   margin    the ONLY tool of the three that touches output at all; headroom's
   over      shaper is off-by-default and weaker, RTK can't see output. No
   rivals    contest — caveman owns this slice outright.
   also      works under any agent/model (it is just a register instruction),
   unique    and the family extends to commits, reviews, and subagent reports
```

Caveman is uncontested on output. It is also the first tool to adopt for a separate reason: it is the only one that is *unconditionally* cache-safe and requires no runtime, no binary, no host service — minutes to adopt, nothing to provision.

### RTK wins when the waste is verbose shell output and you want zero ML

```text
   BEST CASE: RTK
   ──────────────
   symptom   Bash-heavy workload: repeated `cargo test`, `git status`/`diff`,
             build logs, `pytest`/`go test`, lint output flooding context
   why it    deterministic (no ML to mis-fire, no identifier-drop risk),
   wins      cache-safe BY CONSTRUCTION (compresses upstream of context),
             zero MCP rent, one ~4 MB binary, CI-safe (exit codes preserved)
   margin    vs headroom on the SAME shell output: RTK has no ML attack
   over      surface, no per-request model latency, no proxy, no MCP schema —
   rivals    the lower-risk way to compress command output. vs caveman: caveman
             can't see input at all.
   also      the MOST container-adoptable of the three (single binary,
   unique    deterministic, nothing to provision); 100+ command formats turnkey
```

RTK's win is on the *largest concrete* input slice — shell output — where determinism beats headroom's ML on risk and headroom can win nothing caveman can. Its published month-long head-to-head total (1.33B tokens on a Bash-heavy TypeScript/Next.js workload) reflects this: when the workload is shell-dominated, RTK dominates the raw token count.

### Headroom wins when the waste is everything RTK can't see

```text
   BEST CASE: HEADROOM
   ───────────────────
   symptom   big native-tool file reads, large JSON/API payloads, RAG chunks,
             long conversation history on the wire, cross-agent workflows that
             need shared, reversible memory
   why it    it is the ONLY one that reaches native `Read`/`Grep`/`Glob`, RAG,
   wins      and history — the rest of the 61% input bucket RTK never sees —
             AND it makes the compression reversible (CCR retrieve)
   margin    vs RTK: RTK is blind to anything not run through a shell; headroom
   over      sees the whole request. vs caveman: caveman is output-only and its
   rivals    cavemem memory is single-agent and lossy with no recovery.
   also      reversibility (CCR), cross-agent dedup memory, and failure-mining
   unique    (`learn`) — three capabilities neither rival has at all
```

Headroom's win is breadth and reversibility. It is the tool to reach for when the compressible waste lives in native-tool reads, RAG, or history — sources RTK is structurally blind to — or when an agent *platform* needs a shared, reversible, cross-tool memory. It pays for that reach with an ML stage, proxy latency, and an attack surface, so it is the highest-cost and highest-reach of the three.

## Quick selection guide

| If the waste is… | Reach for | Why |
|---|---|---|
| The model writing too much prose / restating code | **caveman** | output class, 5×-priced, cache-neutral |
| Verbose `cargo test` / `git` / build / log output run through Bash | **RTK** | deterministic, cache-safe at the tool boundary, zero MCP rent |
| Big native-tool file reads, RAG chunks, long history on the wire | **headroom** (MCP / live-zone) | broad API-layer reach + reversible recall, accepts proxy/ML cost |
| Whole files re-read just to see structure | **none of these** — a code-intelligence tool (ast-grep / codedb) | structural *retrieval*, a different lever |
| Thinking tokens (20% of dollars) | **none of them** | effort routing / model selection — the unmoved wall |

## Cache-safety, compared

Cache interaction is the make-or-break axis for any input compressor on an already-caching Claude Code, and it cleanly separates the three:

| Tool | Where it acts | Cache interaction | ML in hot path | MCP rent |
|---|---|---|---|---|
| **caveman** | Model's generated prose (output) | **Neutral** — never touches the prefix | no | ~940 tok skill listing |
| **RTK** | New Bash command output, at the tool boundary | **Safe by construction** — the compressed text is what gets cached | **no** | **none** |
| **headroom (MCP)** | A new observation, on demand | **Safe** (write-time) | yes (`kompress-base`) | yes |
| **headroom (proxy)** | Rewrites the whole request | **Risk** — can churn the prefix Claude Code already caches | yes | n/a |
| Whole-prompt proxy (LLMLingua-style) | Rewrites the whole request | **Breaks the cache** — must beat ~5.5–10× | yes | n/a |

RTK occupies the safest corner: write-time, deterministic, native-hook, no model. Headroom is safe in the modes that matter (MCP/library) and risky only in the mode you should not default (proxy in front of Claude Code). Caveman is trivially safe because it is output-side.

## Evidence quality, compared

Adoption stars are PR-inflated for all three and must be ignored as a quality signal (the star:watcher ratios are ~10× more skewed than a healthy repo). What separates them is the *kind* of evidence behind the headline:

| Tool | Best evidence | Weakest spot |
|---|---|---|
| **caveman** | Locally reproduced **58.5%** output-token cut on the Claude tokenizer; mechanism is transparent (it is a prompt) | No agentic-task quality benchmark of register-compressed output exists anywhere |
| **headroom** | Production telemetry across **50k+ sessions** (median 4.8% whole-session) + **one independent 47.5%** measurement + academic backing for the write-time pattern | Product percentages are vendor self-report; the ML stage is unbenchmarked on code quality |
| **RTK** | The underlying levers (log filter −94.2%, JSON minify −34.3%) are locally reproduced in the dossier | **No whole-session telemetry and no independent benchmark of RTK itself** — every RTK number is its own counter, on a non-Claude tokenizer |

On evidence, headroom is the best-instrumented, caveman is the most transparent (you can verify the mechanism by reading the prompt), and RTK is the least independently verified despite the highest per-command headline.

## What none of them can do

Three limits bind all three tools equally, and they are the reason this is not a 10× story:

1. **None touches thinking (20% of dollars).** Thinking bills as output, is invisible in the transcript, and on Fable 5 cannot even be disabled. No register instruction, no input filter, and no observation compressor reaches it — only the **effort** lever and model routing do. This is the largest single bucket none of the three moves.
2. **None builds a persistent symbol index.** All three can drop function bodies on a payload as it passes through (caveman by passing code verbatim does not even do this; headroom and RTK do it per-read), but none answers "where is `foo` defined?" without re-reading. That is a different lever — code intelligence — covered in the dossier's [code-intelligence chapter](/research/token-optimization-research/51-code-intelligence-tools/).
3. **None converts a per-payload ratio into a whole-bill dollar saving for free.** The headline percentages are all per-payload or per-command; the whole-bill effect is bounded by how much of the bill that payload class represents and by the 0.1× cache-read discount that most input tokens already enjoy.
