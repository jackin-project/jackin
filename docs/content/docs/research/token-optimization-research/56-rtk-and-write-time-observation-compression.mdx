## TL;DR

- **RTK is the deterministic, Claude-Code-native productization of the one cache-safe input-compression design point the dossier already named** (file 53 H1 "live-zone / write-time observation compression"; file 54's "write-time observation compression = cache-safe" class). It compresses the output of shell commands (`git`, tests, `ls`, `cat`, logs, builds) **at the tool boundary, before that output ever enters context** — so it shrinks the cache *write* of a new observation and every later 0.1× read of it, and never touches the cached prefix. No ML model, no MCP schema, no proxy in front of Claude Code.
- **It productizes a lever the dossier already validated by hand.** RTK's four strategies — filter, group, truncate, deduplicate — are record 20 (hook/preprocessing filtering, local **−94.2%** on a cargo log, file 03) plus SmartCrusher-style JSON sampling (record 14), applied deterministically across 100+ command formats. It invents no new physics; it packages the proven hook-filtering lever as a turnkey binary.
- **The "60–90%" headline is a per-command best-case ratio on verbose commands, not a whole-bill number** — the same category error the dossier corrected for caveman (K1) and headroom (H-K1). The honest whole-session effect is `(Bash-command-output share of the 61% cache bucket) × compression% × (write-share + 0.1×read-share)` — real on the largest bucket, bounded to low-double-digit % of dollars, because most observation tokens already read at 0.1× after first write. **RTK publishes no whole-session telemetry at all** (headroom at least published a 4.8% median); its numbers all trace to its own `rtk gain` counter.
- **The reach is narrower than the headline implies: RTK only intercepts Bash calls.** In Claude Code the agent reads files through the native `Read` tool and searches through native `Grep`/`Glob` — none of which route through a shell, so none are compressed by RTK. RTK bites on test/build/git/log/`gh` output the agent genuinely runs through Bash; it does nothing to native-tool reads, RAG, conversation history, or thinking (20%, untouched by all three tools).
- **Adoption is PR-inflated and must be ranked by evidence, not stars** (file 54 §A). 63,578★ on a five-month-old repo with **146 watchers** (~435:1, ~10× more skewed than a healthy repo) and a best-Hacker-News showing of **18 points / 3 comments** — the same signature flagged for headroom. There is **no independent benchmark** of RTK's savings anywhere; every figure is vendor self-report or user-reported through RTK's own counter.
- **jackin' verdict: of the three, RTK is the most directly container-adoptable** — a single Rust binary, deterministic, cache-safe by construction, zero MCP rent, no ML runtime to provision — but it registers PreToolUse hooks into the agent's config, which collides with the host-write ban and with caveman's own hook registration. Pilot it **role-scoped inside a container**, reconcile it with the existing caveman hooks, and accept it only if it beats the dossier's already-banked record-20 hook filtering net of the dropped-context risk. Keep caveman for output; RTK and headroom are **complementary input-side layers** (RTK = deterministic Bash-output compression at the tool boundary, headroom = broad API-layer everything-else) that the community in fact stacks — adopt them in risk/reach order, not necessarily all at once (see "Three-way in practice" below).

## The crux: RTK is the cache-safe write-time design point, made deterministic

File 53 (record H1) and file 54 established the single most important distinction in input compression: **whole-prompt recompression in the hot path fights prompt caching (must clear ~5.5–10× just to break even), but compressing a *new observation at the moment it is produced* — before it is ever cached — is cache-safe**, because it shrinks the cache write of that observation and all future 0.1× reads of it without ever mutating the already-cached prefix.

File 53 named headroom's MCP mode (`headroom_compress` on an observation) as the worked example of that design point. **RTK is the cleaner instance of the same idea:**

- It compresses at the **tool boundary** (the Bash call), which is upstream of context entirely — the compressed output is what gets cached in the first place, so there is no prefix to bust by construction.
- It uses **deterministic rule-based transforms**, not a learned model, so there is **no `kompress-base`-style ML model in the hot path** — none of headroom's CompressionAttack ML surface (file 46 FL3 / file 53), none of its per-request model latency.
- It adds **zero MCP schema rent** — it is a hook plus a binary, not a set of tool definitions injected every turn.

| Compressor | Where it acts | Cache interaction on Claude Code | ML in hot path | MCP rent |
|---|---|---|---|---|
| Whole-prompt proxy (LLMLingua; headroom proxy mode) | Rewrites the whole request | **Breaks the cache** unless prefix-stabilized; 5.5–10× break-even (record 19, file 53) | yes | n/a |
| headroom MCP (`headroom_compress`) | New observation, on demand | **Cache-safe** (write-time) | yes (`kompress-base`) | yes (MCP tools) |
| **RTK** | New **Bash command output**, at the tool boundary | **Cache-safe by construction** (the compressed text is what gets cached) | **no** (deterministic Rust) | **no** (hook + CLI) |
| caveman | Model's generated prose (output side) | Neutral (never touches the prefix) | no | ~940 tok skill listing |

So RTK occupies the safest corner of the input-compression space: write-time, deterministic, native-hook. Its cost for that safety is **reach** — it only sees what flows through a shell.

## The three-way: Caveman vs Headroom vs RTK

The operator runs caveman and asked to compare it with headroom (done two-way in file 53) and now RTK. The clean result: **caveman is the output side; headroom and RTK are two opposite points on the input side** — headroom is the broad, ML-driven, multi-mode compressor; RTK is the narrow, deterministic, native-hook one. They attack different token classes and stack without double-counting; the two input tools look like they overlap on tool-output compression but in fact act at different points (RTK at the Bash tool boundary, headroom on the API request) and **compose** — the community stacks all three, as the next section shows.

| Axis | Caveman | Headroom | RTK |
|---|---|---|---|
| Direction | **Output** — what the model writes | **Input** — what the model reads (broad) | **Input** — what the model reads (Bash output only) |
| Primary bucket | Visible prose (~17% of dollars) | Tool outputs / logs / RAG / files / history (the 61% cache lines) | Shell-command output (a **slice** of the 61% — only what runs through Bash) |
| Mechanism | Skill-prompt register change (terse English / Classical Chinese) | Typed compressors (AST/JSON/log/search) + a trained text model + drop-low-value | Deterministic filter / group / truncate / dedup on 100+ known command formats |
| Lossiness | **Lossy, no recovery** | **Reversible** — originals in CCR via `headroom_retrieve` | **Lossy**; tee-recovery on command **failure** only (not on truncated-but-successful output) |
| Touches code? | Passes code/diffs verbatim | Outlines bodies, passes raw code ~0% | Trims `cat`/`read`/`grep` output **and** has a language-aware code filter (`src/core/filter.rs`: strip comments / strip function bodies, 8 languages) — a **per-read** transform, **not** a persistent symbol index (that is file 51 / ast-grep) |
| Touches thinking? | **No** | **No** | **No** |
| Cache interaction | Neutral (output side) | Safe in MCP/library, risk in proxy | **Safe by construction** (write-time at the tool boundary) |
| ML in hot path | No | Yes (`kompress-base`) | **No** (deterministic) |
| MCP schema rent | ~940 tok/session skill listing | Yes in MCP mode | **None** |
| Reach limit | Visible prose only | Everything in the request | **Bash calls only** — native `Read`/`Edit`/`Grep`/`Glob` bypass it |
| Form factor | Claude Code plugin / skills | pip/npm/docker + Rust core + local ML runtime | Single Rust binary + a PreToolUse hook |
| Self-cost | ~940 tok skill prefix | Per-request ML model + proxy latency + MCP rent | Hook registration (host-state write) + a hook-conflict surface with caveman; `<10 ms` compute |
| Evidence | Output ultra **−58.5% local** (token) vs the 75% claim | **−66.1% self-report** mix; **median 4.8% whole-session** (vendor telemetry, 50k+ sessions); **47.5%** independent on a tool-heavy session | **60–90% per-command, vendor-only**; **no whole-session telemetry**, **no independent benchmark** |
| Adoption (2026-06-18) | caveman 74,446★ / 166 watchers | headroom 33,359★ / 111 watchers | RTK 63,608★ / 146 watchers — **all three PR-inflated in this niche; rank by evidence (file 54)** |

Component by component:

- **caveman (output register) is orthogonal to both.** It shrinks the 5×-priced output class the model generates; RTK and headroom shrink the input class the model reads. They stack with no double-counting because they never touch the same tokens. Keep caveman for output regardless of the input choice.
- **RTK ↔ headroom's LogCompressor + SmartCrusher + SearchCompressor.** This is the apparent overlap — headroom's typed compressors do the same *kind* of job RTK does (collapse logs, sample JSON, trim search output) — but they act at different points: RTK at the Bash tool boundary, headroom on the API request. In practice they compose (headroom further-compresses RTK's already-filtered output; measured additive — see "Three-way in practice"). RTK does it deterministically, only on Bash output, *with no reversibility on success* (tee only catches failures); headroom does it as an ML-backed proxy/MCP that also reaches RAG, files, and history, *reversibly* (CCR). **Choose by reach and risk, not as mutually exclusive:** RTK is the zero-ML, zero-MCP, cache-safe-by-construction layer for the command-output case; headroom adds RAG/file/history compression and cross-agent reversible memory at an ML/proxy cost. The only thing to avoid is expecting headroom to win much *again* on the exact bytes RTK already collapsed (diminishing returns on the same payload, not a conflict).
- **Code outlining: closer than the first draft claimed — RTK does it too, on the read path.** An earlier version of this file said RTK's `read`/`cat` filter was "line-trimming, not AST-aware." That is **wrong** and is corrected here against RTK's own architecture doc: `src/core/filter.rs` ships a **language-aware code filter** with three levels — `None` (0%), `Minimal` (strip comments, 20–40%), `Aggressive` (strip comments **and function bodies**, 60–90%) — across Rust/Python/JS/TS/Go/C/C++/Java, used by the `read` and `smart` modules. So RTK's Aggressive read overlaps headroom's CodeAwareCompressor (keep signatures, drop bodies) more than first stated. **The real, surviving distinction is statefulness, not capability:** both RTK and headroom drop bodies on *a payload as it passes through*, one read at a time; neither builds the **persistent, queryable symbol index** that file 51 (ast-grep / codedb) does. Structural *retrieval* (ask "where is `foo` defined" without re-reading the file) is still file 51 territory; one-shot body-dropping on a file the agent reads is something all three of RTK, headroom, and ast-grep can do.

**The decisive ordering (from file 53/54, now three-way):** adopt **caveman (output) first** — it is unconditionally cache-safe and hits the 5×-priced class. For input, prefer **write-time observation compression** (RTK and/or headroom-MCP) over any whole-prompt proxy, because only write-time compression composes with caching instead of fighting it. Of the two input tools, RTK is the lower-risk layer (deterministic, no ML, no MCP, cache-safe by construction, Bash-only reach) and headroom is the higher-reach layer (RAG/files/history/memory at an ML/proxy cost) — and, as the next section shows, they sit at *different* intervention points and stack rather than collide. The 20% thinking bucket stays untouched by all three — the dossier's largest unaddressed wall is unmoved.

## Three-way in practice — the community already stacks all three

The clean-room analysis above weighed the tools against each other. An internet re-sweep (2026-06-18) found the operator community does not treat them as rivals — it **stacks all three** — and one practitioner published side-by-side production numbers. Folding that in sharpens, and partly corrects, the either/or instinct.

**They sit at different intervention points, so they compose rather than collide.** The clearest published model is a four-layer stack (`sgaabdu4/claude-code-tips`, via DeepWiki): (1) prevent data from entering context at all (a Codebase-Memory MCP), (2) virtualize output (context-mode), (3) **caveman** compresses what the model writes into context, (4) **RTK + headroom** compress what is sent to the API — RTK on *shell output specifically* at the tool boundary, headroom on *general API-layer* traffic. Earlier layers shrink what the later ones must compress; the ordering "prevents redundant processing." caveman = output; RTK = Bash observations; headroom = everything else on the wire.

**The one published head-to-head with real numbers** (andrewpatterson.dev — the author's own month of production TypeScript/Next.js work, independent of all three vendors but self-measured via each tool's own counter, not a controlled A/B):

| Tool | Tokens saved (1 month) | Reported reduction | Note |
|---|---:|---|---|
| **RTK** alone | 1,327,700,000 | 60–90% per command (file reads 66.9%, lint 100%) | dominates the total because this workload was Bash-heavy |
| **headroom** alone | 189,014,601 | 31.0–59.1% per model, **96% prefix-cache-hit** | the cache-safe live-zone design *measurably* holds (record H1 confirmed in the wild) |
| **combined** | 1,516,714,601 | — | "RTK's filtered output is further compressed by headroom's proxy" — additive, not conflicting |

Two things this settles:

- **RTK and headroom are complementary, not either/or — correcting this chapter's first-draft framing.** They run in sequence (RTK filters the Bash observation at the tool boundary; headroom then compresses the request on the wire, including RTK's already-filtered output) and the author measured the combination as additive. The only real "don't" is expecting headroom to win much *again* on bytes RTK already collapsed — diminishing returns on the same payload, not a conflict. The "run one or the other" caution holds only for the narrow case of pointing *two tool-output compressors at the identical bytes*; across layers, all three stack.
- **headroom's live-zone cache-safety is real in production**, not just in source: 96% prefix-cache-hit over a real month confirms record H1 — CacheAligner keeps the prefix stable while compressing the tail. It is not free: the same author measured a **~200–500-token increase** on passthrough requests (proxy metadata overhead) — the self-cost row of the three-way table, now quantified.

**The whole-bill correction still applies to the stack headline.** Community guides advertise "90%+ token reduction" and "sessions from 30 minutes to 3+ hours" from stacking these (plus CBM/context-mode). Read them with the dossier's standing corrections:

- "1.5 billion tokens saved" / "90%+" are **token counts, not dollars.** Most of those tokens are cache *reads* priced at 0.1× (the local heavy session ran 92.83% cache reads); the dollar effect is a fraction of the token-percentage headline (the K1/K4 moves).
- "30 min → 3 hr on a 200K window" is a **context-window-occupancy / tasks-per-cap** win (file 41's subscriber metric), not a 90% dollar cut. Real and valuable for a capped subscriber, but a different axis from $-per-task.
- None of the three touches **thinking (20%)**; the 10× wall is unmoved (file 53 verdict stands).
- The independent caveats **corroborate this chapter's kills**: candido.ai and andrewpatterson both note RTK's figures have "no independent reproduction methodology, no variance statistics, no controlled baseline" and are "structurally bounded to Bash tool calls … may be significantly below the headline 60–90%" — exactly R-K1 and R-K2.

**Quick selection guide — which tool for which waste:**

| If the waste is… | Reach for | Why |
|---|---|---|
| The model writing too much prose / restating code | **caveman** | output class, 5×-priced, cache-neutral |
| Verbose `cargo test` / `git` / build / log output run through Bash | **RTK** | deterministic, cache-safe at the tool boundary, zero MCP rent |
| Big native-tool file reads, RAG chunks, long history on the wire | **headroom** (MCP / live-zone) | broad API-layer reach + reversible recall, accepts proxy/ML cost |
| Whole files re-read just to see structure | **not these** — file 51 (ast-grep / codedb outlines) | structural retrieval, a different lever |
| Thinking tokens (20% of dollars) | **none of them** | effort routing (file 15) — the unmoved wall |

**Refined verdict.** caveman (output) + RTK (Bash observations, deterministic, cache-safe) + headroom (API-layer everything-else, broad, cache-safe-in-practice but with proxy overhead and the ML/attack-surface caveat) is a **coherent three-layer stack**, and the community runs it. For jackin', adopt in risk/reach order: caveman first (output, unconditional), RTK second (Bash observations, lowest input-side risk), headroom third (only if RAG/file/history/cross-agent-memory compression is needed and the proxy/ML cost and host-write are accepted). Run all three only after each clears the harness on its own slice — and do not assume the per-tool percentages add up to the marketed stack headline.

## Technical internals — how each tool actually achieves it (equal-depth teardown)

The comparison tables above weigh the three tools on *outcomes* (what bucket, what risk, what reach). This section answers the separate question the operator asked — **how does each one work under the hood, at matched depth** — because the prior two files were lopsided: headroom got a full crate-level teardown (file 53), RTK got the mechanism table above, and caveman was only ever described as "the output side" with no look at its actual machinery. Here all three are torn down to the same level: interception point → engine type → per-unit mechanism → persistent state → self-cost → failure mode. Sources: caveman from a direct read of the installed plugin (`JuliusBrussee/caveman` `v1.9.0`); headroom from file 53's source audit plus its published latency telemetry; RTK from its architecture doc.

### caveman — a prompt, not a program

caveman has **no compression engine at all**, and that is the whole point. The installed plugin is a markdown rule-set plus two thin Node hooks:

- **The engine is `SKILL.md`** — a register-shift *instruction* the model applies to its own generation: drop articles/filler/hedging, keep code/errors/API names/commit keywords verbatim, fragments OK. The six intensity levels (`lite`/`full`/`ultra`/`wenyan-lite`/`wenyan-full`/`wenyan-ultra`) are just different strictness clauses in the same file; "ultra" adds prose-word abbreviation and causal arrows, "wenyan" switches to Classical Chinese. There is **no parser, no model, no subprocess** — the compression happens inside the model's decoding, for free, because the prompt told it to talk shorter.
- **Two hooks, both host-state writes.** `plugin.json` registers a `SessionStart` hook (`caveman-activate.js`, injects the mode prompt at session open) and a `UserPromptSubmit` hook (`caveman-mode-tracker.js`, re-asserts the active level every turn and tallies tokens for `/caveman-stats`). The installer (`bin/lib/settings.js`) writes these idempotently into `~/.claude/settings.json` with a Zod-valid shape, an `hasCavemanHook` probe to avoid duplicates, and a `removeCavemanHooks` uninstall path — the same `~/.claude` mutation surface that collides with RTK's hook (see adoption section).
- **Adjacent tools are the same trick aimed elsewhere:** `caveman-compress` rewrites memory files in place (keeping a `.original.md` backup), `caveman-shrink` compresses MCP tool descriptions, `cavecrew` subagents return caveman-compressed reports, and `/caveman-stats` reads the session JSONL — the numbers are hook-injected, the model does not compute them.
- **Self-cost:** the ~940-token skill listing carried in context, plus the two hooks' host-state writes. Runtime compute is essentially zero. **Failure mode:** over-terse prose with no recovery — the long form was never generated, so to get it back you re-ask (the project's own issue #484 is caveman shrinking PR/issue titles it should have left alone). **Eval method (so the headline is readable):** `evals/measure.py` counts with tiktoken `o200k_base` (an *approximation* of Claude's BPE) against an "Answer concisely." terse-control arm over 10 prompts, reporting median/min/max/stdev — i.e. it measures what caveman adds *on top of* already-terse output, which is why the dossier's honest re-measure (−58.5% token, ultra) lands below the README's 75% headline.

### headroom — a routing-and-typed-compressor pipeline with an ML stage

headroom is the only one of the three that is a genuine runtime system (file 53 has the full audit; condensed here for equal footing):

- **Interception:** a request passes through a `ContentRouter` that classifies each payload and dispatches it to a **typed compressor** — `LogCompressor` (keep errors/levels, drop passing noise), `CodeAwareCompressor` (keep imports/signatures/types, collapse bodies), `SearchCompressor` (`file:line:content`), `SmartCrusher` (JSON arrays → sampled/typed), `HTMLCompressor`, `IntelligentContext` (score by recency/relevance/error, drop low-value turns) — plus `kompress-base`, an **auto-downloaded transformer** trained on agentic traces that does perplexity-style prose compression. This is the one ML stage in any of the three, and it is the component most likely to drop a load-bearing identifier (the record-19 failure mode).
- **The cache-safety machinery is real Rust:** `cache_stabilization/` (`anthropic_cache_control.rs`, `volatile_detector.rs`, `tool_def_normalize.rs`, `drift_detector.rs`) + `compression/live_zone_anthropic.rs` extract the volatile tail, keep the prefix byte-identical, and insert `cache_control` breakpoints — the live-zone design point (record H1). **Persistent state:** the CCR store (SQLite/Redis/in-memory) holds originals for `headroom_retrieve`; a cross-agent memory store; `headroom learn` writes corrections into `CLAUDE.md`/`AGENTS.md`.
- **Self-cost is measured, not guessed (v0.5.18 telemetry, 50k+ sessions):** proxy overhead **P50 52 ms / P90 309 ms / P99 4,172 ms / mean 161 ms**; the internal pipeline runs **16.9 ms median** (ContentRouter 11.7 ms = 91–98% of it, SmartCrusher 50.1 ms, TextCompressor 32.0 ms), and a third party measured **+200–500 tokens** of passthrough metadata per request. Plus MCP schema rent in MCP mode and the `kompress-base` model as a hot-path attack surface (CompressionAttack, file 46 FL3). **Failure mode:** ML-dropped identifiers on code, silent cache-bust in proxy mode, double-compaction against Claude Code — all *reversible via CCR if the model knows to retrieve*.

### RTK — a six-phase deterministic Rust filter keyed on the command

RTK is the deterministic mirror of headroom's pipeline: same *kinds* of transform, no router-ML, fixed per-command rules. Its architecture doc gives the machinery the prior table only summarized:

- **Six-phase lifecycle per command:** PARSE (clap extracts command + args) → ROUTE (`main.rs` matches a command enum to a module) → EXECUTE (`std::process::Command` subprocess, capture stdout/stderr) → FILTER (the command-specific strategy) → PRINT (verbosity-gated) → TRACK (write savings to SQLite). The whole thing is a **single ~4.1 MB stripped binary, zero deps, ~5–10 ms cold start, ~2–5 MB RAM, ~5–15 ms per command**.
- **Twelve deterministic strategies, matched to output shape** (not one generic summarizer): Stats Extraction (`git status`/`log`, 90–99%), Error-Only (runner stderr), Grouping-by-Pattern (`lint`/`tsc`/`grep`, 80–90%), Deduplication (logs → unique lines + counts), Structure-Only (JSON keys+types), **Code Filtering** (`src/core/filter.rs`, the None/Minimal/Aggressive levels described in the correction above), Failure-Focus (`vitest`/`playwright`, 94–99%), Tree-Compression (`ls`), Progress-Filtering (strip ANSI), JSON/Text-Dual-Mode (`ruff`/`pip`), State-Machine-Parsing (`pytest` lifecycle), NDJSON-Streaming (`go test`). It also sniffs the package manager (`pnpm-lock.yaml` → `pnpm exec`, else `yarn`, else `npx`) before running JS/TS tools.
- **Interception is a hook, two modes:** auto-rewrite (default — the PreToolUse hook rewrites `git status` → `rtk git status` before execution, "100% adoption, zero context overhead") or suggest (emits a `systemMessage` hint, the agent chooses, ~70–85% adoption). **Persistent state:** SQLite at `~/.local/share/rtk/history.db` (per-command `saved_tokens`/`savings_pct`/`exec_time_ms`, 90-day auto-cleanup, queried by `rtk gain`). Note its token counter is a **~4-chars-per-token GPT-style heuristic**, *not* Claude's BPE — so `rtk gain` numbers are directionally right but not Claude-accurate.
- **Reliability primitives:** exit codes preserved (`std::process::exit(output.status.code())` — CI-critical), filter-failure falls back to original output (`anyhow::Result`), failure-tee saves the full unfiltered output, `-vvv` shows raw pre-filter output. **Self-cost:** the hook's host-state write (the real cost — collides with caveman's), telemetry off by default. **Failure mode:** truncation on a *successful* command can drop the one needed line, and tee only fires on *failure* — the silent-dropped-context risk, caught only by the command-re-run-rate metric in the validation harness.

### The internals, side by side

| Primitive | caveman | headroom | RTK |
|---|---|---|---|
| Interception point | Model's own decoder (prompt rule) | API request (proxy) or observation (MCP/lib) | Bash tool boundary (PreToolUse hook) |
| Engine type | **Markdown instruction** (no code path) | Router + typed compressors + ML model | 12 deterministic Rust filters keyed on command |
| Parser / structural | none | per-type (AST outline, JSON, log) | per-command + a `filter.rs` code filter (8 langs) |
| ML in the loop | **No** | **Yes** (`kompress-base`, auto-downloaded) | **No** |
| Persistent state | none (hooks only track tokens) | CCR store + cross-agent memory + `learn` | SQLite history (`rtk gain`) |
| Token counter | tiktoken `o200k_base` (eval only) | own counter, no stated tokenizer | ~4 chars/token heuristic |
| Recovery on loss | **none** (re-ask) | **CCR `headroom_retrieve`** (reversible) | tee **on failure only** |
| Host-state write | `~/.claude` hooks ×2 | MCP/proxy config + model download | `~/.claude` PreToolUse hook |
| Runtime cost | ~0 compute + ~940-tok prefix | P50 52 ms / P99 4.17 s + ML + MCP rent | ~5–15 ms/cmd, ~4 MB binary |
| Hardest failure | over-terse, unrecoverable | ML drops an identifier; proxy cache-bust | truncates a needed line on success |

The teardown confirms the dossier's framing from a new angle: **the three sit on a determinism gradient.** caveman is a zero-machinery prompt; RTK is maximum determinism (fixed rules, no model, single binary); headroom is the only one that buys *breadth* by paying for an ML stage, a proxy, and a reversible store. More machinery → more reach and reversibility, but also more latency, more host effects, and a real attack surface.

## Do you need all three, or is one enough? Standalone-vs-combined

The operator's direct question: *must a project run all three, or does one alone carry most of the value?* The answer follows from the determinism teardown — **they do not overlap on token class, so no single one of them is a superset of the others**, but they have very different standalone value depending on the workload.

**Each tool alone — what you get and what you miss:**

| If you install only… | You capture | You miss entirely |
|---|---|---|
| **caveman** | The output class (~17% of dollars, 5×-priced), cache-neutral, zero runtime, minutes to adopt | All input compression — verbose test/build/log output and big reads still hit context at full size |
| **RTK** | The Bash-observation slice of the 61% cache bucket — the single largest *concrete* coding waste, deterministically and cache-safely | Output verbosity (the model still rambles); non-Bash reads, RAG, history; reversibility on success |
| **headroom** | The broadest input surface — native-tool reads, RAG, history, files, cross-agent memory — reversibly | Output verbosity (it ships an optional shaper, off by default); and it pays ML/proxy latency + an attack surface to get the breadth |

**The decisive fact for "do I need all three":** because caveman touches *only output* and RTK+headroom touch *only input*, **caveman never double-counts with either** — running caveman plus one input tool is strictly additive and is the sweet spot for most projects. The genuine redundancy is *only* between the two input tools, and even there it is **diminishing returns on the same bytes, not a conflict** (the published month-long head-to-head measured RTK 1.33B + headroom 0.19B → 1.52B combined — additive, because RTK filters at the Bash boundary and headroom then compresses the *rest* of the request on the wire). headroom's own release notes corroborate the composition: **v0.22.4 (2026-06-01) wires a `tokens_saved_rtk` data plane and "RTK metrics" observability** — headroom explicitly tracks RTK savings rather than re-implementing shell rewriting, treating RTK as a complementary upstream layer.

**So the practical answer, by project shape:**

| Project shape | Bring | Rationale |
|---|---|---|
| Default coding agent, want the cheapest real win | **caveman + RTK** | Output + the biggest concrete input slice; both cache-safe, both deterministic, no ML, no proxy. The recommended lean stack. |
| Output verbosity is the *only* complaint; tool output already controlled | **caveman alone** | Smallest intervention; nothing else justified. |
| Bash-dominated workload (heavy `cargo test`/`git`/build), output already terse | **RTK alone** | The dominant waste is shell output; caveman adds little if the model is already brief. |
| Agent *platform*: large JSON/API/RAG, long histories, cross-agent memory, reversibility required | **headroom** (+ caveman for output) | Only headroom reaches native-tool reads, RAG, history, and offers CCR recall. RTK can't see non-Bash reads. |
| Maximal coverage, willing to pay the cost and run the harness | **all three** | caveman (output) + RTK (Bash observations) + headroom (everything else on the wire). Real, but only after each clears the harness on its own slice — and never expecting the per-tool percentages to sum to the marketed stack headline. |

**What "all three" must *not* mean:** two auto-rewrite shell hooks fighting over the same Bash bytes (run *one* shell-rewrite path), or headroom's L4 output shaper stacked on caveman `full`/`ultra` (two competing output-verbosity policies — pick one for output). The clean three-layer stack is **one output tool, one Bash-boundary tool, one API-layer tool** — never two of the same kind aimed at the same tokens. And none of the three touches **thinking (20% of dollars)** — so "all three" is still not a 10× story; the largest wall is unmoved regardless of how many you stack.

## Per-technique record

### R1. Deterministic write-time command-output compression (RTK)

- **Coverage-delta:** Productizes record 20 (hook/preprocessing filtering) + record 14 (JSON sampling) as a turnkey binary, and is the deterministic, Claude-Code-native worked example of file 53 H1 / file 54's write-time-observation-compression class (which used headroom-MCP as its example).
- **Layer:** input + cache (write-time, at the tool boundary).
- **Mechanism:** a PreToolUse hook rewrites the agent's shell command to `rtk <cmd>`; RTK runs it and returns filtered/grouped/truncated/deduplicated output. Because the compressed text is what enters context, the cache write shrinks and every later 0.1× read of it shrinks, and the cached prefix is never touched.
- **Expected savings:** on the modeled day, `(Bash-command-output share of the 61% cache bucket) × compression% × (write-share + 0.1×read-share)`. Real on the largest bucket, bounded to low-double-digit % of dollars because (a) most observation tokens already read at 0.1× after first write and (b) RTK reaches only Bash output, not native-tool reads. **NOT** the 60–90% per-command headline (ESTIMATE; arithmetic mirrors the caveman K1 / headroom H-K1 corrections).
- **Evidence tier:** **T1 for the mechanism** (the underlying filter/dedup/group/JSON-sample levers are locally reproduced in files 03/51 — log filter −94.2%, JSON minify −34.3%); **T4 for RTK's specific product numbers** (vendor self-report through its own `rtk gain` counter, no whole-session telemetry, no independent replication).
- **Quality risk:** **NEUTRAL on dedup/grouping of genuinely redundant output**; **RISKY on truncation** — a truncated-but-successful command can silently drop the one line the agent needed, and tee-recovery only fires on failure. Degradation looks like the agent acting on incomplete test/log output, or re-running commands uncompressed (eroding the saving). Falsify by A/B with RTK on/off: measure task success, tool-result tokens, and the rate at which the agent re-runs a command to see full output.
- **Availability:** `CLAUDE-CODE-TODAY` (hook + binary).
- **Effort to adopt:** minutes to install; the real cost is reconciling its hook registration with caveman's and scoping it to a container.
- **Composability:** stacks with caveman (different token class) and with code-intelligence outlines (file 51 — different mechanism, structural not line-trim); **diminishing returns (not anti-synergy) stacking headroom on the same Bash bytes** RTK already collapsed — they compose across layers (measured additive; see "Three-way in practice"), just do not double-count the same payload — and genuine anti-synergy only with any tool that re-filters the identical output. Hook-registration anti-synergy with caveman's hooks (both write agent config).
- **Validation protocol:** 20 Bash-heavy tasks (test/build/git/log), native vs RTK-hook; from JSONL require (a) `cache_read` continuity preserved (it should be, by construction), (b) tool-result tokens down, (c) task success unchanged, (d) no rise in command re-runs, (e) net tokens-per-solved-task down ≥20%.

## Benchmarks: what is real and what is self-report

RTK's published figures are internally consistent and honest about being per-command, but they are all the maintainer's own, produced by RTK's own `rtk gain` counter, with no stated tokenizer and no third-party replication.

| Workload (RTK self-report) | Claimed cut | Honest reading |
|---|---|---|
| `ls` / `tree` | −80% | Repetitive listing — matches the dossier's grouping lever |
| `cat` / `read` | −70% | Only when the agent uses Bash `cat`, not the native `Read` tool |
| `grep` / `rg` | −80% (alt. blog: 49.5%) | Search-result trimming — matches the −98% symbol-search lever's *direction*, not its magnitude (RTK is line-trim, not structural) |
| `git status` / `git diff` | −80% / −75% | Status/diff dedup — genuinely redundant content |
| `cargo` / `npm` / `pytest` / `go test` | −90% | Logs — matches the dossier's **−94.2% local** log filter (record 20) |
| "30-minute session" | ~118,000 → ~23,900 = **−80%** | A per-command best case **assuming a Bash-heavy command mix**; not a measured whole-session distribution, and ignores native-tool reads RTK never sees |

The whole-bill correction (the K1 move, on the input side): "60–90%" is a per-command ratio on verbose commands. On the modeled heavy day, Bash-command output is only part of the 61% cache traffic (the rest is the system prefix, CLAUDE.md, conversation history, native-tool file reads, and code RTK passes through). The realistic whole-bill effect is `(Bash-output share of the 61%) × compression% × (write + 0.1×read)` — low-double-digit % of dollars, the same category as the caveman and headroom corrections. RTK is a real lever on the *command-output* slice of the largest bucket; it is not the headline.

Two evidence gaps weaker than headroom's: RTK has **no whole-session production telemetry** (headroom published median 4.8% / P75 6.9% across 50k+ sessions), and **no independent third-party measurement** of any kind (headroom has Miya-Gadget's 47.5% and an HN "~50%"). Every RTK number is the vendor's, or a user reading RTK's own counter.

## jackin' adoption recommendation

RTK fits the same role-scoped, opt-in, measured-locally pattern files 51/53 set, and it is the **most directly adoptable of the three compression tools** for a jackin' container — but the host-write and hook-conflict edges are sharp.

1. **Pilot it role-scoped inside a container, never on the host.** `rtk init -g` writes a PreToolUse hook into the agent's global config (`~/.claude/settings.json` and friends). Per the host-write ban, install RTK and register its hook **inside the role container only**, never on the operator's machine.
2. **Reconcile with caveman's hooks first.** caveman already registers `SessionStart`/`UserPromptSubmit` hooks and the architect role already warns that the `~/.claude` hook/settings state is the real hazard (see <RepoFile path="docs/content/docs/reference/roadmap/architect-code-intelligence-tooling.mdx">docs/content/docs/reference/roadmap/architect-code-intelligence-tooling.mdx</RepoFile>). RTK adding a PreToolUse hook on Bash must not clobber that state; register it idempotently alongside the existing hooks, the same way the architect role registers MCP servers in `preflight`.
3. **Disable telemetry in the container.** RTK's telemetry is opt-in but the marketing copy is inconsistent ("no telemetry" vs "opt-in anonymized"); set it off explicitly for a security-conscious image, the same posture the architect role takes with `CODEDB_NO_TELEMETRY=1`.
4. **A/B against the lever the dossier already banks**, not against a naive baseline: record-20 hook filtering already captures the log/test-output win cache-safely with a hook the role can write itself. RTK earns its place only if its 100+-command coverage beats a hand-written filter net of the dropped-context risk and the hook-conflict surface.
5. **Know the reach limit going in.** In Claude Code the agent prefers the native `Read`/`Grep` tools over Bash `cat`/`grep`; RTK only compresses what actually runs through Bash. Measure how much of the session's observation tokens even flow through the shell before assuming RTK can touch them.

### Validation harness

Run the same shape as files 51/53, with cache continuity and command re-run rate as first-class metrics:

| Arm | Tools allowed |
|---|---|
| Native | Claude Code defaults (native Read/Grep, no shell-output filter) |
| Hooks | Native + a hand-written record-20 grep/log filter hook |
| RTK | Native + the RTK PreToolUse hook |
| RTK + headroom-MCP | RTK on Bash output + headroom-MCP on non-Bash observations (test the complementary stack: confirm additive, not redundant) |

Metrics: tool-result tokens; `cache_read` ratio and cache-write spikes from JSONL; **command re-run rate** (did the agent re-run a command to see full output — the dropped-context tell); fraction of observation tokens that flow through Bash at all; total tokens per solved task; task success and test pass; wall-clock.

Acceptance rule:

```text
Accept RTK for token optimization only if, versus the Hooks arm:
  task/test success            >= baseline
  cache_read ratio             >= baseline (should hold by construction)
  command re-run rate          <= baseline (no silent dropped-context cost)
  total tokens per solved task <= baseline by at least 20%
  net of the hook-conflict and host-write guardrails above
```

## Corrections and refinements to prior files

- **Refine file 53 record H1.** H1 used headroom-MCP as the worked example of cache-safe write-time observation compression. RTK is the **cleaner, deterministic instance**: it compresses at the tool boundary (upstream of context, so there is no prefix to bust), with no ML model and no MCP schema. Add RTK as the no-ML / no-MCP example of the same design point; headroom-MCP remains the ML / multi-source example.
- **Refine file 54's cache-safety classification.** RTK sits squarely in the "write-time observation compression = cache-safe" class — and is the strongest evidence yet that the class is real and shippable on Claude Code today, deterministically, without the proxy or the model.
- **Refine record 20 (hook/preprocessing filtering).** The dossier recommended writing a hook to filter logs and measured −94.2% locally. RTK is that hook generalized to 100+ command formats as a maintained binary — the turnkey version of the lever, with the same upside and the same dropped-context risk, plus a hook-registration cost the hand-written version did not have.
- **Refine file 53's caveman-vs-headroom comparison into a three-way.** The "compress what you read" side is now two distinct tools: headroom (broad, ML, proxy/MCP, reversible) and RTK (narrow, deterministic, hook, cache-safe-by-construction). caveman remains the output side. Unlike the memory layer (where file 53 drew a true cavemem *or* headroom-memory either/or), the tool-output layer is **not** either/or: RTK (tool boundary) and headroom (API request) act at different points and compose additively — the only caution is diminishing returns from re-compressing the identical bytes, not a conflict (see "Three-way in practice", with the published 1.33B + 0.19B → 1.52B/month head-to-head).
- **Self-correction (2026-06-18 verification pass): RTK *does* have a code-aware filter.** This file's first draft asserted RTK was "line-trimming, not AST-aware." RTK's architecture doc shows `src/core/filter.rs` strips comments and function bodies across 8 languages (the `read`/`smart` modules, Aggressive level, 60–90%). Corrected in the comparison table and the code-outlining bullet above. The surviving distinction from file 51 is **statefulness** (RTK outlines a payload per-read; ast-grep/codedb maintain a persistent queryable symbol index), not raw capability. Also refreshed: adoption stats to a same-day `gh api` re-pull (caveman 74,446★ / headroom 33,359★ — up from file 53's stale 28,199 — / RTK 63,608★), and added headroom's measured proxy latency percentiles (P50 52 ms / P99 4.17 s, v0.5.18) to the internals teardown.
- **New corroboration that the three compose (not compete).** headroom's own release notes — **v0.22.4, 2026-06-01: "wire `tokens_saved_rtk` data plane" + "RTK metrics + Rust observability"** — show headroom integrating RTK's savings into its data plane rather than re-implementing shell rewriting. This independently confirms the "Three-way in practice" finding from the vendor side: headroom treats RTK as a complementary upstream Bash-boundary layer, exactly the layered stack the community runs.
- **No change to the 10× verdict.** RTK attacks the command-output slice of the 61% cache bucket — the right target — but its realistic whole-bill effect is bounded (per the K1-style correction), its reach is limited to Bash, and it does not touch thinking (20%). The dossier's verdict stands: ≈2.5× defensible, ≈5–6.2× with validated routing, no honest 10× at zero quality loss. RTK is a low-risk addition to the Aggressive stack's input layer, not a new multiplier.
