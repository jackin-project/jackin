jackin❯ exports application observability directly to an OTLP/gRPC backend. The backend owns durable history. Product processes keep only bounded current-invocation state in memory for progress and operator UI; they do not write telemetry, diagnostics, or capsule log files.

This page is the living instrumentation contract. Schema changes start in the closed registry and update this reference in the same change.

## Composition and delivery

Each host, daemon, and Capsule process owns its OpenTelemetry providers and exports traces, logs, and metrics directly. There is no embedded or configured Collector. Standard `OTEL_EXPORTER_OTLP_ENDPOINT` and per-signal endpoint variables select the backend; only the `grpc` protocol is supported. `OTEL_SDK_DISABLED=true` disables providers explicitly.

Provider construction is best-effort and product work stays fail-open. Export uses bounded batches, five-second per-export deadlines, bounded retry for retryable gRPC status codes, and no retry for authentication or partial-success responses. Saturation drops telemetry instead of blocking product threads. Shutdown flushes traces, logs, then metrics and is idempotent.

`jackin diagnostics validate` emits one marker per signal and synchronously flushes them. Typed health reports active signals, attempts, success/failure counts, drops, last success/error state, and shutdown state through host-daemon protocols.

## Schema authority

<RepoFile path="crates/jackin-telemetry/registry/registry_manifest.yaml">`crates/jackin-telemetry/registry`</RepoFile> is a closed Weaver-validated OpenTelemetry semantic-convention 1.43 extension registry. Generated names and bounded enums are the only signal vocabulary. The governed facade rejects unknown names, unknown attributes, invalid bounded values, privacy violations, oversized fields, excess links/events, and cardinality overflow.

Product code emits through `jackin-telemetry`: registered events, operation guards, metric handles, and ownership-aware spawn helpers. Direct OpenTelemetry SDK use and arbitrary `tracing` telemetry targets are architecture-gate violations. Operator-facing prose remains a separate output port.

## Identity and resources

Resources are stable for one process and identical across its three providers. They contain service identity and deployment-safe process attributes such as `service.name`, `service.version`, `service.instance.id`, and `app.mode`. Invocation, session, job, workspace, role, container, and other correlation values do not belong on the Resource.

Every command mints one `cli.invocation.id`. Each interactive attachment mints a `session.id`; a reconnect rotates it and may record `session.previous_id`. Detached jobs mint `job.id`. These identifiers are UUIDs and are correlation fields on governed signals, not backend-specific resource keys.

Internal launch handoff uses `JACKIN_INVOCATION_ID` and marks a network-safe Capsule endpoint with `JACKIN_CAPSULE_OTLP_SAFE`. They are process-boundary implementation variables, not operator configuration.

## Trace model

There are no process-, console-, screen-, tab-, pane-, or session-lifetime spans. Spans represent bounded work: application startup/shutdown, a non-interactive command, UI action, launch stage, subprocess, Docker request, provider request, database operation, connection attempt, cycle, RPC request, stream segment, or prewarm attempt.

Client/server RPC boundaries use `CLIENT` and `SERVER` span kinds. Detached scheduling uses a `PRODUCER` decision and a linked `CONSUMER` attempt sharing `job.id`. Joined tasks inherit their current operation; detached, cycle, stream, and blocking helpers encode ownership explicitly. Cancellation and failure still close operation guards with governed outcomes.

W3C `traceparent` and `tracestate` cross CLI-to-daemon and host-to-Capsule control envelopes together with versioned correlation fields. A valid remote context parents the receiving server operation. Missing, malformed, unsupported, or disabled context creates a safe local root. Detached work uses span links rather than false parentage.

## Events and metrics

Events are native OpenTelemetry log records with `EventName`, severity, trace/span correlation when active, and registered attributes. One governed event produces one log record; it is not duplicated as a span event. INFO records lifecycle and state transitions, WARN handled degradation, ERROR failed operations, DEBUG diagnostic decisions, and TRACE narrowly governed high-frequency detail. Default mode excludes DEBUG and TRACE.

Screen entry/exit and widget focus are events. Operator actions are short root operations so latency and outcome remain measurable without lifetime UI traces. Agent state, PTY/session lifecycle, attach state, and periodic work use change-only events plus low-cardinality counters, histograms, and gauges.

Metric attribute sets are capped per stream. Runtime/export health, operation duration/outcome, lifecycle counts, queue pressure, drops, and process/runtime gauges remain bounded. Attribute values such as ids never become metric dimensions unless the registry explicitly permits them.

## Privacy

Telemetry is allowlist-first. The registry permits semantic categories and bounded outcomes, not arbitrary product data. Do not emit:

- absolute paths, workspace or role names, container names, tab labels, or user-entered labels;
- raw argv, commands, environment values, URLs with queries, headers, credentials, tokens, or secrets;
- PTY bytes, rendered frames, keystrokes, clipboard content, prompts, model content, or mouse coordinates.

Generative-AI signals use standard semantic conventions only where content capture remains disabled. Secret scrubbing is defense in depth for operator output and explicit evidence flows; it does not make an unregistered telemetry field acceptable.

## Runtime state and debugging

`RunDiagnostics` is a bounded in-memory progress/timing model consumed by current-run UI. It is not a sink or history store. Explicit operator-requested exports such as a copied file remain workflow artifacts, not telemetry fallback.

`--debug` enables operator troubleshooting behavior. `JACKIN_TELEMETRY_LEVEL` and `JACKIN_TELEMETRY_CATEGORIES` govern registered telemetry detail independently. Neither mode creates a telemetry file. Use the backend and `cli.invocation.id` / `session.id` for history, and use [Run Telemetry](/guides/run-telemetry/) for operator setup.

## Verification

The dev-only <RepoFile path="crates/jackin-otlp-testbed/README.md">`jackin-otlp-testbed`</RepoFile> receives real OTLP/gRPC requests and supports deterministic failure responses. Conformance tests decode every signal, validate resources, namespaces, privacy, propagation, lifecycle, volume, cardinality, shutdown, and failure behavior. Allocation and reviewed five-percent performance gates protect disabled and active hot paths.

Run focused contract checks with:

```sh
cargo xtask telemetry-registry
cargo nextest run -p jackin-diagnostics -p jackin-otlp-testbed --all-features --locked -E 'test(/conformance/)'
```
