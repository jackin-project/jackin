title: "Capsule usage and quota overlay"

**Status**: Implemented — the Capsule control channel, Turso-backed TUI overlay, focused status signal, background account refresher, and provider quota adapters for all seven providers (OpenAI, Anthropic, Amp, xAI, Z.AI, Kimi, MiniMax) have landed. The cut cost/spend, history, top-model, Instance, action-row, browser-cookie, and obsolete workspace/session usage surfaces have been removed. Live in-container verification passed for all seven providers on June 24, 2026 via `jackin-capsule usage verify` after refreshing real forwarded credentials into Turso.

## Problem

jackin' can launch several expensive agent sessions in parallel, but the operator cannot see the focused agent's remaining account headroom from inside `jackin-capsule`. The operator has to remember provider dashboards, subscription windows, reset times, and whether the current session is close to a limit while still working in the agent's terminal.

The goal is not to clone CodexBar. CodexBar is a macOS menu-bar account dashboard; Capsule is an in-terminal multiplexer for a focused agent. jackin' borrows CodexBar's data-extraction strategy and quota vocabulary while preserving Capsule's constraints: read-only, daemon-fed, focused-pane aware, keyboard-first, and safe inside low-color terminals.

## Decided design (binding)

These decisions are final and override any older description in this item.

- **Two surfaces, one source.** (1) An always-on status signal in the right of the bottom status bar for the focused pane; (2) the usage overlay (`u` prefix, palette `Usage`, or a status-bar click). Both read **exclusively from the Turso database** under one canonical provider key — neither ever calls a provider and neither recomputes a snapshot. A failed background refresh must preserve the last good value rather than overwrite it with `needs login`. Clicking the status bar opens the overlay on the focused pane's provider tab.
- **Provider-only tabs plus Overview.** Tab set and order: `Overview · OpenAI · Anthropic · Amp · xAI · Z.AI · Kimi · MiniMax`. There are no agent tabs and no Instance tab. Usage and quota are per-provider-account concepts; one agent (for example `claude`) can be routed through several providers (Anthropic, Z.AI, MiniMax, Kimi), so each tab is the provider/account and the focused pane maps to whichever provider it is routed through. Each provider tab shows the single forwarded account for that provider in the running instance.
- **Use the shared tab component.** The overlay renders its tabs with <RepoFile path="crates/jackin-tui/src/components/tab_strip.rs" /> (the same `TabStrip` used by the settings and workspace-editor surfaces), not a hand-built strip. It adopts that component's focus model: a green underline marks the tab bar as focused and `Left`/`Right` switch provider; `Tab` moves focus into the content (the underline turns white) where the operator scrolls; `Tab` or `Esc` returns focus to the strip.
- **No per-dialog footer.** The overlay does not draw its own key-hint line. It uses the standard hint bar (<RepoFile path="crates/jackin-tui/src/components/hint_bar.rs" /> with a usage keymap, the same mechanism as every other Capsule dialog), rendered below the modal.
- **Meters show remaining, full width.** Every quota bar spans the full available width of the dialog and is filled in proportion to the quota that REMAINS, so a fully-available bucket is a full bar. This is deliberate: the filled length is the at-a-glance signal of how much is left. MiniMax (which the provider reports as percent-used) is normalized to remaining like every other provider — a `0% used` bucket renders as `100% left` with a full bar, never an empty one.
- **Stacked meter lines.** Under each bar, `N% left` and `N% in reserve` are on separate lines on the left; the reset countdown and `Lasts until reset` are stacked on the right, matching CodexBar.
- **Per-provider section sets.** A tab renders only the sections that provider actually has (for example Codex's Spark buckets, Limit Reset Credits, and Credits; Claude's Sonnet, Daily Routines, and conditional Extra usage; Z.AI's Tokens/MCP/5-hour; Amp's Credits balance). No uniform scaffold is drawn on tabs that lack a section.
- **Removed from the overlay** (see Implementation plan → Cleanup): the per-message cost/spend grid (Today / 30d cost / 30d tokens / Latest tokens), the history sparkline, the `Top model` line, the spend provenance line, the Instance tab and its ledger, and the account-action rows (Buy Credits, Cost, Subscription Utilization, Add Account, Usage Dashboard, Status Page). The overlay shows account quota only.
- **Headers carry provider and account identity.** Provider name on the left, account label on the right; the second line carries freshness (`Updated Nm ago`) on the left and plan on the right. Where account identity is unknown, render `account unavailable` rather than a blank.
- **Read-only and honest.** Capsule never writes host credentials, imports cookies, scrapes dashboards, or refreshes-and-rewrites host auth files. A snapshot that cannot be trusted degrades to an explicit `stale`, `needs login`, `needs secret`, `unsupported`, or `usage unavailable` state — never a fabricated number.

## Status bar

The status bar shows, for the focused pane's provider, only the remaining session (5-hour) and weekly percentages — the single signal of whether the agent still has enough quota to keep working. There is exactly one form (no separate full / compact / lifecycle layouts). Branch/repository context stays on the left; the usage and the ids sit on the right.

```text
 Branch · feature/auth           Session 92% · Weekly 55%   18bbf3524d6cadd8 · 5vvkqps1
```

Right-hand order, left to right: the usage (`Session N% · Weekly N%`), the debug run id (only present in debug mode), then the container id (rightmost).

- **Follows the focused pane.** A screen may hold several panes, each running a different agent routed through a different provider. The bar reflects whichever pane currently has focus and switches as focus moves between panes.
- **Reads only from Turso.** The bar is built from the latest cached snapshot in the Turso database for the focused provider; it never calls a provider, so it never blocks.
- **Session and weekly** map to each provider's equivalent windows (table below); show only the one(s) the provider exposes.
- **Lifecycle fallback** in the same slot when no trustworthy percentage exists: `needs login`, `stale`, `unsupported`, or `usage unavailable` — never a fabricated number.
- **Click opens the overlay** on the tab for the focused pane's provider (a focused Anthropic pane opens the overlay on the Anthropic tab).
- **Narrow terminal:** drop the weekly value before the session value; always keep the container id and at least the session value or the lifecycle word.
- Color is advisory only; the value must read in monochrome.

## Data flow and refresh

Both surfaces — the status bar and the overlay — read **exclusively from the Turso database**. Nothing in the render path ever calls a provider. A single background refresher is the only component that calls provider endpoints (using the per-provider recipes in "How the data is extracted") and writes normalized account-quota snapshots into Turso (`account_usage_snapshots`). This keeps the UI instant and decouples how often providers are polled from how often the operator looks. Relative-time labels ("Updated 2m ago") are computed from each snapshot's `fetched_at`.

### Refresh cadence (CodexBar-aligned)

jackin' follows CodexBar's cadence model. CodexBar reference behaviour: one global automatic interval, **default 5 minutes**, refreshing all enabled providers off the main thread; opening the popover does NOT force a full refresh — it only retries providers that errored or have no snapshot, after a ~1.2s delay; server quota endpoints carry rate-limit gates (Anthropic `oauth/usage` uses a 5-minute or `Retry-After` cooldown on HTTP 429, bypassed only for a manual refresh); the expensive web scrape runs ~5× slower (~25 min); refreshes are single-flighted and coalesced; there is no periodic UI tick for the relative-time label (it updates when the menu is rebuilt).

jackin' default: poll every **~5 minutes per account** with ±60s jitter; only providers that have an active agent pane in the container are polled; the focused provider refreshes first and on overlay open; honor `Retry-After` with exponential backoff on repeated HTTP 429 capped at ~30 min; single-flight per provider. jackin' performs no OAuth refresh of its own — the refresher re-reads the in-container credential file each cycle (the agent CLI keeps it current) and degrades to `needs login` / `stale` when no fresh token is available.

### The multi-container rate-limit problem (design for it up front)

Provider usage limits are **per account / per token / per IP, not per process**. jackin' can run N containers against the SAME provider account, so naive per-container polling multiplies request volume N× — a 5-minute cadence across 50 containers is one request every ~6 seconds to one account, enough to trip 429s a single CodexBar instance never sees. The design must therefore:

- Coalesce to **one refresh per account**, fanned out to every container that shares it, rather than each container polling independently. The long-term home is the host-owned refresh daemon in [Token & Cost Telemetry](/reference/roadmap/token-cost-telemetry/#host-owned-telemetry-cache); until it exists, the capsule-local refresher uses the ~5-min interval with jitter and honors a **host-shared cooldown marker keyed by account** (a small file/db under `~/.jackin/data/daemon/`): a 429 / `Retry-After` seen by any container pauses polling for that account in every container.
- Server quota endpoints are never polled tighter than CodexBar's defaults.

### Overlay open behaviour

- Opening the overlay (including via a status-bar click) **proactively triggers a background refresh of the selected provider first**, then the others, without blocking the render — the cached Turso snapshot shows immediately.
- While a provider is refreshing, its header says so in place: the freshness line reads e.g. `Updated 4m ago · refreshing…`, and updates to `Updated just now` when the refresh completes, so the operator always knows whether the numbers are current or being refreshed.
- While the overlay is open, all relative times and computed values **tick every minute** so an open dialog never shows a frozen "Updated 2m ago". When the overlay is closed there is no UI tick; the background refresher continues on its cadence, prioritizing the active tab/provider.

| Provider | Buckets |
|---|---|
| OpenAI / Codex | `Session`, `Weekly`, `Codex Spark 5-hour`, `Codex Spark Weekly`, `Limit Reset Credits`, `Credits` |
| Anthropic / Claude | `Session`, `Weekly`, `Sonnet`, `Daily Routines`, conditional `Opus` and `Extra usage` |
| Amp | `Amp Free`, `Credits` (individual-credits balance) |
| xAI | `Weekly` (subscription/included quota) |
| Z.AI | `Tokens`, `MCP` (with `N / 100 (remaining)`), `5-hour` |
| Kimi | `Weekly`, `Rate Limit` |
| MiniMax | one bucket per model × window (`General · 5h`, `General · Weekly`, `Video`, …) |

For each bucket render: the label; a full-width remaining meter; remaining percent; reset countdown; pace/reserve label when known; and a lifecycle label when `NeedsLogin`, `Unsupported`, `Stale`, or `Error`. Conditional buckets (Opus, Extra usage, Spark, Credits) render only when the provider returns them. Each bucket may use a different unit (tokens, messages, requests, credits, plan units); the renderer never compares percentages across incompatible units.

Reset format: provider detail tabs follow CodexBar's absolute style (`Resets 19:45` same-day, `Resets tomorrow, 04:18`, `Resets Jun 26 at 13:59`, `Resets Jul 1 at 16:31`). The Overview tab uses the compact `Resets <relative> (<absolute>)` form. `Resets now` shows no time. Pace/reserve text uses the CodexBar formula: `delta = utilization - (elapsed_fraction_of_window * 100)`; `abs(delta) <= 2` is `On pace`; `delta > 0` is `N% in deficit`; `delta < 0` is `N% in reserve`.

## States and honest degradation

| State | Required rendering |
|---|---|
| Fresh provider snapshot | Quota buckets with confidence/source labels. |
| Stale provider snapshot | `stale` plus `updated Xm ago`; keep prior data visible. |
| Needs login / needs secret | Account and the repair action name; no hidden repair. |
| Unsupported provider | `unsupported` with provider/source explanation. |
| Daemon disconnected | `usage unavailable: daemon disconnected`; do not poll providers from Capsule. |
| No focused agent | `usage unavailable: no focused agent session`. |
| Narrow terminal | Collapse columns; keep provider, headline quota, reset, and status. |

Synchronized tokens can expire and Capsule deliberately never refreshes-and-rewrites host credentials, so an expired forwarded token degrades to an honest `needs login` / `stale` state and the operator re-authenticates outside Capsule. Enterprise/zero-tier accounts may not expose the consumer usage surface at all (an observed enterprise token returned HTTP 401 while the default account returned 200); the tab must show `stale` / `needs login` rather than a fabricated value.

## Terminal interaction and host-side effects

- `u` (prefix) or palette `Usage` opens the overlay; `Left`/`Right` switch provider on the focused tab bar; `Tab`/`Shift+Tab` move focus between tab bar and content; `r` requests a refresh; `Esc` closes. Keys are surfaced through the standard hint bar, not an in-box footer.
- The overlay preserves Capsule's redraw policy: no full-screen erase when opening, tabbing, refreshing, or closing.
- Capsule is read-only. It must not write provider config, Keychain entries, browser profiles, host dotfiles, Git config, or repositories, and must not poll provider APIs directly from inside the container. A repair that would change host state is surfaced as a named action pointing at the responsible surface, never performed silently.

## CodexBar parity and non-regression principle

CodexBar (`github.com/steipete/CodexBar`) is the reference jackin' borrows this feature's data extraction and information architecture from. **jackin' must never present this information at lower quality than CodexBar for any dimension it also covers** — same or richer detail, same or higher provenance honesty, same or better caching and performance. A change that makes any covered dimension worse than CodexBar is a regression. Where jackin' cannot match a CodexBar capability safely, it says so honestly rather than degrading silently or faking data. The deliberate, limited exceptions are: jackin' never runs CodexBar's token-refresh-and-rewrite flow (expired tokens degrade honestly), and jackin' never imports browser cookies, scrapes dashboards, reads the Keychain to repair accounts, or launches hidden provider CLIs.

## How the data is extracted

This is what the **background refresher** does to populate the Turso store; the status bar and overlay render paths never call these endpoints. Each provider adapter resolves a credential, calls the provider's usage endpoint, and maps the response to quota buckets. Credentials are the same accounts the operator is already logged into; the runtime forwards them into the container (see Credential delivery below). The per-provider recipes jackin' mirrors from CodexBar:

**Anthropic / Claude.** Credential: `~/.claude/.credentials.json` (`claudeAiOauth.accessToken`), forwarded from the host macOS Keychain service `Claude Code-credentials` (per-config-dir variants use service `Claude Code-credentials-<first 8 hex of sha256(config dir path)>`). Call: `GET https://api.anthropic.com/api/oauth/usage` with `Authorization: Bearer`, `Accept: application/json`, `Content-Type: application/json`, `anthropic-beta: oauth-2025-04-20`, and `User-Agent: claude-code/<version>` (no `anthropic-version`). Windows: `five_hour`→Session, `seven_day`→Weekly, `seven_day_sonnet`→Sonnet, `seven_day_opus`→Opus, `seven_day_routines` (and aliases `seven_day_claude_routines`/`claude_routines`/`routines`/`seven_day_cowork`)→Daily Routines, `extra_usage`→Extra usage; each window carries `utilization` and `resets_at`. Plan label prefers `subscriptionType`, falls back to `rateLimitTier`.
**OpenAI / Codex.** Credential: `$CODEX_HOME/auth.json` (default `~/.codex/auth.json`), `tokens.access_token` + `tokens.account_id`. Call: `GET https://chatgpt.com/backend-api/wham/usage` with `Authorization: Bearer` and `ChatGPT-Account-Id: <account_id>` when present; an alternate authority launches `codex -s read-only -a untrusted app-server` and calls JSON-RPC `account/read` + `account/rateLimits/read`. Map: `rate_limit.primary_window`→Session, `secondary_window`→Weekly (`used_percent`/`reset_at`/`limit_window_seconds`); `additional_rate_limits[]` containing `spark`→Codex Spark 5-hour/Weekly; `credits`→Credits; `plan_type`→plan. Account identity comes from the app-server RPC or the `id_token` JWT, not from a `tokens.email` field (which is usually absent).
**Amp.** Credential: `AMP_API_KEY`. Call: `POST https://ampcode.com/api/internal?userDisplayBalanceInfo` with body `{"method":"userDisplayBalanceInfo","params":{}}` and `Authorization: Bearer`; `amp --no-color usage` is a secondary source. Buckets: Amp Free plus an individual-credits balance.
**xAI / Grok.** Credential: `~/.grok/auth.json` (the scope entry's `key`). Primary call: spawn `grok agent stdio` and issue JSON-RPC `x.ai/billing` (the method must stay slash-unescaped). If the local Grok CLI accepts the token but does not implement `x.ai/billing`, Capsule uses the same fresh CLI bearer token against Grok's billing RPC without browser cookies or browser-session material. Buckets: weekly/credits from `monthlyLimit`/`usage.totalUsed`/`billingCycle` or the equivalent bearer billing response. Browser-cookie/session Grok fallbacks remain out of scope.
**Z.AI / GLM.** Credential: env `Z_AI_API_KEY` (no file, carried into the container regardless of mount mode — the reference pattern). Call: `GET https://api.z.ai/api/monitor/usage/quota/limit` with `Authorization: Bearer`. Map `TOKENS_LIMIT`/`TIME_LIMIT` entries to Tokens / MCP / 5-hour.
**Kimi.** Credential: the Kimi-For-Coding API token (`KIMI_CODE_API_KEY` or `~/.kimi-code/credentials/kimi-code.json` `access_token`). Call: `GET https://api.kimi.com/coding/v1/usages` with `Authorization: Bearer`; the response is `usage` + `limits[]`, mapped to Weekly + Rate Limit. (The `kimi.com` web-cookie billing endpoint is a separate variant that requires a browser session JWT and is not used.)
**MiniMax.** Credential: `MINIMAX_CODING_API_KEY` / `MINIMAX_API_KEY`. Call: `GET` the `/v1/token_plan/remains` and `/v1/api/openplatform/coding_plan/remains` paths on `api.minimax.io` then `api.minimaxi.com`, with `Authorization: Bearer` and `MM-API-Source`; transport errors fall through to the next host rather than aborting. `model_remains[]` yields one bucket per model × interval/weekly window, rendered as remaining.

### Credential delivery into the container

The capsule runs as PID 1 inside the Linux container and cannot read the host Keychain; the runtime reads the host credential (file or macOS Keychain) and forwards it into the container at a fixed path per provider (see <RepoFile path="crates/jackin-runtime/src/instance/auth.rs" />). File-based providers (Claude, Codex, Grok, Amp) depend on that forwarding; env-key providers (Z.AI) are carried in regardless of mount mode, which is why Z.AI works reliably and is the template the others should move toward where a provider offers an env-key REST path.

## Data model expected by the TUI

The overlay consumes a quota-only `FocusedUsageView` from <RepoFile path="crates/jackin-protocol/src/control.rs" />: a `FocusedAccountHeader` (provider, account label, plan, freshness), a list of `QuotaBucketView` (label, used/limit when known, remaining percent, reset, pace, and per-bucket `UsageSnapshotStatus`/`UsageSource`/`UsageConfidence`), and `UsageProviderTab` entries for the tab strip and Overview. The `workspace_spend` and `instance` fields and the `WorkspaceSpendView`/`InstanceUsageView` types are removed (see Cleanup).

## Scope (V1)

The only deliverables are the two surfaces and the Turso-backed data path that feeds them:

- The **status bar** (focused-provider session + weekly remaining, read from Turso, click-to-overlay).
- The **usage overlay** (Overview tab plus one tab per provider, matching the binding previews, read from Turso).
- The **background refresher** that populates Turso from provider endpoints on a rate-safe cadence, with honest fresh / stale / needs-login / unsupported states.

Nothing else is in scope. Account setup, add-account, buy-credits, settings, dashboard, status-page, and login-repair flows are NOT part of this feature, and browser-cookie / web-session sources are NOT used. Any existing code implementing those is removed (see Implementation plan → Cleanup).

## Defer

- Per-message cost/spend tracking, history, and top-model — owned by [Token & Cost Telemetry](/reference/roadmap/token-cost-telemetry/); not shown in this overlay.
- A per-instance spend ledger (the dropped Instance tab). If revived later it must not be mixed into account tabs.
- The host-owned, cross-container refresh daemon (one poller fanned out to all containers) — the target architecture for rate-safe refresh; tracked in [Token & Cost Telemetry](/reference/roadmap/token-cost-telemetry/#host-owned-telemetry-cache).

## Settled decisions

- **Token freshness without host writes.** jackin' runs no OAuth refresh. The refresher re-reads the in-container credential file each cycle; the agent CLI (Claude Code, Codex) refreshes and rewrites that file itself, so jackin' always reads a current token without writing host or implementing refresh. When no fresh token is available it degrades to honest `needs login` / `stale`.
- **Refresh cadence.** ~5 minutes per account with ±60s jitter; only providers with an active agent pane are polled; focused provider first and on overlay open; honor `Retry-After` with exponential backoff on repeated 429 (cap ~30 min); single-flight per provider.
- **Cross-container rate-limit safety.** Each container polls on its own cadence but all containers honor a host-shared cooldown marker keyed by account (a small file/db under `~/.jackin/data/daemon/`): a 429 / `Retry-After` seen by any container pauses that account everywhere. The single-poll-per-account host daemon ([Token & Cost Telemetry](/reference/roadmap/token-cost-telemetry/#host-owned-telemetry-cache)) supersedes this later.
- **Header naming.** Provider-org labels (`OpenAI`, `Anthropic`, `Amp`, `xAI`, `Z.AI`, `Kimi`, `MiniMax`) on both the tab strip and the in-tab header.
- **Codex sections.** The OpenAI tab keeps the read-only `Limit Reset Credits` and `Credits` sections (CodexBar parity, quota-relevant, no actions).
