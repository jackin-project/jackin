## Verification Findings And Remaining Work (2026-06-19)

This section is the live-verified gap analysis between the previews in this item and the shipped implementation, plus the CodexBar reference mapping the implementation must follow. It was produced by running the real binary and credential stores on the operator host, not by reading code alone. CodexBar is the canonical reference project for this feature: jackin' deliberately mirrors CodexBar's data-extraction strategy and on-screen information architecture, so every adapter and render decision below is anchored to a specific CodexBar source.

### Live Verification Performed On Host

Performed 2026-06-19 on the operator macOS host with the operator's own synchronized agent logins. All probes were read-only GETs against provider usage endpoints; no tokens were written or printed.

- **Claude / Anthropic — works when read from the right source.** The host has no `claudeAiOauth` block in `~/.claude.json` and no `~/.claude/.credentials.json` file; the real Claude Code OAuth token lives in the macOS Keychain item `Claude Code-credentials`. Reading the Keychain token and calling `GET https://api.anthropic.com/api/oauth/usage` with `anthropic-beta: oauth-2025-04-20` and `User-Agent: claude-code/2.1.0` returned **HTTP 200** with `subscriptionType: max`, `rateLimitTier: default_claude_max_20x`, and windows `five_hour` (27% utilization), `seven_day` (30%), `seven_day_sonnet` (13%), `seven_day_opus`, `seven_day_oauth_apps`, `seven_day_cowork`, `seven_day_omelette`, and `extra_usage`. The data is fully available; the adapter simply reads the wrong place.
- **Codex / OpenAI — works.** `~/.codex/auth.json` holds `tokens.access_token`, `tokens.account_id`, `tokens.refresh_token`, and a top-level `OPENAI_API_KEY`. Calling `GET https://chatgpt.com/backend-api/wham/usage` with `Authorization: Bearer <access_token>` and `ChatGPT-Account-Id: <account_id>` returned **HTTP 200** with `plan_type: pro`, `rate_limit.primary_window` + `rate_limit.secondary_window`, and `additional_rate_limits` containing `GPT-5.3-Codex-Spark`. No token refresh was required for a recently-used login.
- **Host credential inventory.** Codex `~/.codex/auth.json` present; Claude OAuth only in Keychain `Claude Code-credentials` (no `~/.claude/.credentials.json`); Amp `~/.local/share/amp/secrets.json` present; Grok `~/.grok/auth.json` present; Kimi `~/.kimi-code` + `~/.kimi` present; OpenCode `~/.local/share/opencode/auth.json` present and bound to `zai-coding-plan` (the healthy GLM/Z.AI pane). Agent CLIs are resolved through `PATH` entries in `~/.zshrc` (`~/.amp/bin`, `~/.kimi-code/bin`, `~/.grok/bin`), not shell aliases, so there is no alias indirection to resolve before probing a provider.

### Per-Workspace / Multi-Account Auth Verification

The operator runs two workspaces with different provider accounts: the jackin' workspace uses the default host logins, and the Scentbird workspace pins separate credential folders (`[claude] sync_source_dir = ~/.claude-scentbird`, `[codex] sync_source_dir = ~/.codex-scentbird`, `[amp] sync_source_dir = ~/.amp-scentbird/data/amp`). The 2026-06-19 host run confirmed per-workspace separation is supported at the forwarding layer:

- **Codex and Amp** read file credentials directly from the workspace `sync_source_dir` (`~/.codex-scentbird/auth.json` and `~/.amp-scentbird/data/amp/secrets.json` both present with distinct accounts).
- **Claude** is Keychain-backed on macOS and the token is single-slot per config dir, so the runtime derives a per-config-dir Keychain service. The default `~/.claude` config dir uses the bare service `Claude Code-credentials`; any other config dir uses `Claude Code-credentials-<first 8 hex of sha256(absolute config dir path)>`. For `~/.claude-scentbird` this resolves to `Claude Code-credentials-db49ea31`, which was verified to exist and hold the distinct Scentbird **enterprise** account (`subscriptionType: enterprise`), not the default Max account. The runtime deliberately never falls back to the default account for a workspace that pinned a source folder, so a company login cannot leak the personal Max account into the Scentbird container.
- The runtime forwards whichever credential it resolved (file or Keychain) into the container at the standard path `~/.claude/.credentials.json`, so the in-container usage daemon sees the correct per-workspace account at one fixed path regardless of workspace. This is why the adapter fix (read `~/.claude/.credentials.json`) is sufficient for both workspaces at once.
- **Honest-degradation caveat:** the Scentbird enterprise token returned **HTTP 401** from `api.anthropic.com/api/oauth/usage` on this run (stale/expired forwarded token, or an enterprise/`default_claude_zero` tier that may not expose the consumer OAuth usage surface), while the default Max account returned 200. The Claude tab must therefore degrade to `needs login` / `stale` for that account rather than show a fabricated value, and whether enterprise plans expose usage at all is an explicit open verification item.

### Confirmed Defects

1. **Claude OAuth credential source is wrong.** `claude_snapshot` loads OAuth from `~/.claude.json`. On macOS the token is in the Keychain item `Claude Code-credentials`; in a Linux/container home it is `~/.claude/.credentials.json`. CodexBar reads exactly those two sources. The adapter must read `~/.claude/.credentials.json` first and the Keychain item on macOS hosts, with `~/.claude.json` only as a last resort. This is the single reason a synchronized Claude Max login renders `account unavailable`.
2. **Render diverges from the previews.** The provider detail builder pushes scaffold rows (`Cost row`, `Token row`, `Today cost`, `30d cost`, `30d tokens`, `Latest tokens`) in addition to the real `Account cost and tokens` row, and several fallback value strings include their own label (`"today unavailable"`, `"30d cost unavailable"`) which then gets a second label prefix at render time, producing doubled text. The detail view must instead render quota meters, a single two-column cost grid, the history sparkline, `Top model`, the provenance line, and a `Source` / `Status` footer exactly as the previews show.
3. **Claude `User-Agent` differs from the verified-good value.** The adapter sends `User-Agent: jackin-capsule/usage`; the on-host 200 was obtained with `User-Agent: claude-code/2.1.0`. The OAuth usage endpoint may treat the client UA as significant, so the adapter should send a `claude-code/<version>` UA like CodexBar until a non-claude UA is proven to return 200.
4. **Claude bucket coverage is incomplete.** The live response also returns `seven_day_opus`, `seven_day_oauth_apps`, `seven_day_cowork`, and `seven_day_omelette` windows. The previews show `Sonnet`, `Daily Routines`, and `Extra usage`; the adapter should map the windows it recognizes and ignore unknown promotional windows rather than dropping `Opus` when present.
5. **Synchronized tokens can expire with no in-Capsule refresh.** Capsule deliberately never writes host credentials, so it cannot run CodexBar's refresh-token flow. When a forwarded token is expired, the tab must degrade to an honest `needs login` / `stale` state and the operator refreshes the token outside Capsule. This is a policy decision to keep, not a bug, but it must be documented in the tab state.

### CodexBar Reference Mapping

CodexBar (`github.com/steipete/CodexBar`) is the source of truth for how to extract and present this data. The exact behaviors jackin' mirrors:

Source: `Sources/CodexBarCore/Providers/Claude/ClaudeOAuth/ClaudeOAuthCredentials.swift` and `docs/claude.md`.
- Usage call: `GET https://api.anthropic.com/api/oauth/usage`, headers `Authorization: Bearer`, `Accept: application/json`, `Content-Type: application/json`, `anthropic-beta: oauth-2025-04-20`, `User-Agent: claude-code/<version>`; no `anthropic-version` header. Source: `ClaudeOAuthUsageFetcher.swift`.
- Window field to bucket: `five_hour` to `Session`, `seven_day` to `Weekly`, `seven_day_sonnet` to `Sonnet`, `seven_day_opus` to `Opus`, `seven_day_routines` to `Daily Routines`, `extra_usage` to `Extra usage` (with `monthly_limit`, `used_credits`, `utilization`, `currency`). Each window carries `utilization` (0-100) and `resets_at`. Source: `ClaudeOAuthUsageFetcher.swift`.
- Plan/tier label: prefer `subscriptionType`, fall back to `rateLimitTier`. Source: `ClaudeCredentialRouting.swift`.

Codex / OpenAI:

- Token source: `$CODEX_HOME/auth.json` (default `~/.codex/auth.json`), `tokens.access_token` and `tokens.account_id` (no JWT decode of `id_token`). Source: `Sources/CodexBarCore/Providers/Codex/CodexOAuth/CodexOAuthCredentials.swift`.

## CodexBar Parity And Non-Regression Principle

CodexBar is the reference jackin' borrows this feature's data extraction and information architecture from. That makes a non-regression rule explicit and binding for this item:

**jackin' must never present this information at lower quality than CodexBar does for any dimension it also covers. For every dimension below, jackin' targets parity-or-better: the same or richer detail, the same or higher data quality and provenance honesty, the same or better caching and refresh behaviour, and the same or better performance. A change that would make any covered dimension worse than CodexBar is a regression and must not ship without an explicitly recorded, operator-approved reason (for example, the deliberate read-only host-safety tradeoff below).** Where jackin' cannot match a CodexBar capability safely, it must say so honestly in the UI rather than degrade silently or fake data.

The deliberate exceptions, where jackin' is intentionally less capable than CodexBar in exchange for a stronger guarantee, are explicit and limited:

- jackin' never writes host credentials, so it does not run CodexBar's token-refresh-and-rewrite flow; an expired forwarded token degrades to an honest `needs login` / `stale` state instead of being silently refreshed.
- jackin' never imports browser cookies, scrapes dashboards, reads the Keychain to repair accounts, or launches hidden provider CLIs in the background. Capsule is read-only and daemon-fed.

Every other dimension must be parity-or-better.

## Problem

jackin' can launch several expensive agent sessions in parallel, but the
operator cannot see the focused agent's remaining account headroom from inside
`jackin-capsule`. The operator has to remember provider dashboards, subscription
windows, reset times, local token burn, and whether the current session is close
to a limit while still working in the agent's terminal.

The same operator also needs a per-instance accounting view. Account quota answers
"how much headroom does this provider account still have?" It does not answer
"what did this particular jackin' instance burn since it started?" A long-running
Capsule may contain several agent tabs sharing the same account, and the operator
needs a clear session ledger for the instance lifetime: wall-clock age, active
agent time, token totals, exact provider-reported spend when available,
price-table estimates when exact cost is unavailable, top models, and per-agent
breakdown from instance creation until now.

The existing [Token & Cost Telemetry](/reference/roadmap/token-cost-telemetry/)
roadmap item owns the data plane: usage adapters, account snapshots, daemon
cache, persistence, and CLI/daemon endpoints. This item owns the Capsule
operator interface: what the multiplexer should show, how it should behave in a
terminal, and which CodexBar-inspired information belongs in the focused-agent
view.

The goal is not to clone CodexBar. CodexBar is a macOS menu-bar account
dashboard; Capsule is an in-terminal multiplexer for a focused agent. jackin'
should borrow the information architecture and quota vocabulary while preserving
Capsule's constraints: read-only, daemon-fed, focused-pane aware, keyboard-first,
and safe inside low-color terminals.

## Implementation Progress

The first implementation pass has landed the read-only Capsule surface, normalized protocol shape, daemon cache, and provider adapters for every priority provider that currently has a safe non-browser extraction path. It is not yet the final persistent telemetry system described by this roadmap item.

## Extraction And Caching Plan

Telemetry extraction belongs in the host daemon and shared telemetry layer from
[Token & Cost Telemetry](/reference/roadmap/token-cost-telemetry/). Capsule
renders cached snapshots only.

### Provider Strategy Order

Each provider adapter should expose ordered candidates and a normalized failure
reason so the daemon can show `needs login`, `needs secret`, `unsupported`, or
`stale` without knowing provider internals:

| Provider | Candidate order |
|---|---|
| Claude Code / Anthropic | Authoritative account API/OAuth snapshot -> local JSONL spend scanner -> CLI PTY `/usage` only when manually requested or debugging |
| Codex / OpenAI | `codex app-server` account/rate-limit RPC -> OAuth usage endpoint from `~/.codex/auth.json`/`$CODEX_HOME/auth.json` -> optional web dashboard extras when explicitly enabled -> local JSONL spend scanner |
| Amp | `amp usage` one-shot if stable -> Amp settings/web quota source when explicitly enabled -> local Amp thread/session metadata for activity only |
| Grok Build | Read-only auth cache identity -> `grok agent stdio` ACP billing RPC -> local `~/.grok/sessions` signals -> web billing fallback when explicitly enabled |
| GLM / Z.AI | API-token quota endpoint only; no browser-cookie fallback |
| Kimi | Kimi For Coding billing endpoint from explicit auth token/cookie source -> local Kimi Code activity/spend scanner if a stable format is confirmed |
| MiniMax | Coding-plan API token -> standard MiniMax API token -> web/session fallback when explicitly enabled -> billing-history endpoint for 30-day trend when available |

Local scanners must be incremental. Cache by canonical file path, mtime, size,
and parsed offset where the log format allows it. Recompute aggregate cost/token
rollups from cached records, not by rescanning every JSONL file on each status
bar redraw. A practical floor is one cost-scan refresh per minute; account quota
refreshes should default to every five minutes unless a provider imposes a
longer safe interval.

### Refresh Policy

- Default automatic refresh: `5m`.
- Supported operator choices: manual-only, `1m`, `2m`, `5m`, `15m`, `30m`.
- Manual refresh always schedules a provider refresh but immediately returns the
  last cached snapshot to Capsule.
- Failed background child launches cool down before retrying; use a `30m`
  cooldown for Codex-style app-server launch failures unless provider evidence
  says otherwise.
- HTTP fetches need hard request/resource timeouts. CodexBar uses a `30s`
  request timeout and `90s` resource timeout as a reasonable default; providers
  with short RPC probes can use smaller timeouts such as Grok's billing probe.
- Child processes started for probes must be killed on timeout or daemon
  shutdown. Do not leave hidden CLIs running after Capsule closes.
- Stale data remains visible with `stale` and `updated Xm ago` labels. Empty
  data plus an error becomes a state row, not a blank meter.
- Browser-cookie import, Keychain access, dashboard scraping, and account repair
  are never background defaults. They require an explicit operator-controlled
  action outside Capsule.

### Provider Buckets

Provider adapters should emit display-ready bucket labels, units, limits,
remaining percentages, reset times, and provenance labels:

| Provider | Target buckets and metadata |
|---|---|
| Claude Code / Anthropic | `Session`, `Weekly`, model buckets such as `Sonnet`/`Opus`, `Daily Routines`, `Extra usage`; account plan such as `Max`; spend from local Claude logs with API-rate estimate provenance |
| Codex / OpenAI | `Session`, `Weekly`, `Codex Spark 5-hour`, `Codex Spark Weekly`, `Credits`; account plan such as `Pro 20x`; spend from local Codex sessions; optional web-only extras like code-review/credits history stay lower priority |

The `Overview` tab, if enabled, should order providers by operator priority for
this feature: Claude, Codex, Amp, Grok, GLM/Z.AI, Kimi, MiniMax, then any future
providers. The selected tab should still default to the focused pane's actual
agent/provider, even if that means jumping into the middle of the list.

## Product Shape

Capsule gets two surfaces:

1. **Always-visible active-agent status signal.** A short focused-provider
   usage summary in the status chrome for the active tab/pane. When an agent is
   active and daemon data is available, the operator must be able to see both
   current usage and remaining quota without opening the overlay.
2. **Usage overlay.** Prefix `u` and palette action `Usage` open a modal with
   CodexBar-style quota buckets, reset windows, spend/tokens, provenance,
   freshness, and provider status for the focused agent.

The default modal view is **focused provider only**. An overview tab may exist,
but opening the overlay while focused in a Codex pane must answer "can this
Codex account keep working?" before it answers anything about Claude, Amp, Kimi,
or OpenCode.

### Active-Agent Status Signal

The status chrome is the first-class surface for active-agent usage. The overlay
is for detail; the status bar is where the operator notices "I am almost out"
before sending another expensive prompt.

When the focused pane belongs to an agent, the status line must show:

- provider;
- account identity when known, or `account unavailable` when it is not;
- current window name when useful (`Session`, `Weekly`, or provider-provided
  active bucket);
- remaining quota;
- current usage when known;
- reset countdown when there is room;
- lifecycle state when data is stale, missing, or requires login.

Preferred full form:

```text
Provider · account Window: used / limit · remaining left · resets countdown
```

Examples:

```text
 jackin'  [Codex ●] [Shell]       main · Codex · alexey@chainargos.com Session: 63% used · 37% left · resets 1h21m
 jackin'  [Claude ●] [Shell]      feature/auth · Claude · alexey@chainargos.com Session: 84% used · 16% left · resets 46m
 jackin'  [Codex ●] [Shell]       docs · Codex · alexey@chainargos.com Weekly: 90% used · 10% left · resets 3h52m
 jackin'  [Codex ●] [Shell]       docs · Codex · alexey@chainargos.com stale · last 37% left · updated 24m ago
 jackin'  [Amp ●] [Shell]         main · Amp · account unavailable login
```

Compact forms when space is tight:

```text
Codex 63% used · 37% left
Codex 37% left
Codex stale
Amp login
```

Rules:

- The signal follows the focused tab/pane's provider and account. It never shows
  a global average or a sibling agent's quota.
- When both values are known, show both current usage and remaining quota. If
  only one value fits, prefer remaining quota (`37% left`) because that is the
  faster operational decision.
- When no trustworthy percentage exists, show lifecycle state: `login`,
  `secret`, `stale`, `unsupported`, or `unavailable`.
- Color is advisory only. Text/glyph state must remain clear in monochrome and
  low-color terminals.
- If the daemon is disconnected, show `usage unavailable` in the overlay and
  avoid a misleading percentage in the status line.
- The signal is secondary to the active tab label and repository/branch context.
  On tight terminals, elide in this order: reset countdown, branch/repository
  detail, current usage, then quota detail. Keep provider plus remaining/state as
  long as possible.
- Shell tabs with no active agent do not show provider usage unless a child
  agent/session is explicitly associated with the pane.

### Modal Layout

The overlay is a centered modal drawn by the existing Capsule overlay system.
The visual model is CodexBar's provider popover compressed into a terminal:
tabs at the top, account header, quota buckets, account-level cost/token summary,
provenance, provider status, then a short key-hint footer. The tab set must
distinguish provider-account state from the current jackin' instance ledger:

- `Overview`: cross-provider summary and most-constrained signal.
- `Instance`: this Capsule instance from creation/start time until now.
- Provider detail tabs are provider/account tabs, not app-instance tabs:
  `OpenAI / Codex`, `Anthropic / Claude`, `Amp`, `xAI / Grok`,
  `GLM / Z.AI`, `Kimi`, `MiniMax`, and future providers when configured.

Provider tabs answer account/subscription availability and account-wide recent
usage exactly like the CodexBar screenshots: they are not scoped to this jackin'
instance or to one launched app process. `Codex` means the OpenAI/Codex account
surface, `Claude` means the Anthropic/Claude account surface, and the same rule
applies to every other provider. The `Instance` tab answers what this one jackin'
instance has consumed across all agent tabs and accounts while it has been
running. Both views are required because they answer different operator questions.

### Account Tab Source Of Truth

The provider detail tabs are CodexBar-parity account views. Their source of truth
is the provider account plus provider/local-account logs, not the current jackin'
instance and not one running app process. When the Codex tab says `30d tokens
2.5B`, it means the OpenAI/Codex provider account. When the Claude tab says
`Today $404.61`, it means the Anthropic/Claude provider account. Those numbers
represent the selected account/provider usage surface shown in the screenshots,
even if some of that usage happened outside the current Capsule.

Required account-tab fields from the screenshots:

- provider name, account identity, plan/tier, and freshness label;
- quota buckets with remaining percent, reset countdown, reserve/deficit, and
  run-out/lasts-until-reset text when enough timing data exists;
- Codex buckets: `Session`, `Weekly`, `Codex Spark 5-hour`,
  `Codex Spark Weekly`, and `Credits`;
- Claude buckets: `Session`, `Weekly`, `Sonnet`, `Daily Routines`, and
  `Extra usage`;
- `Today`, `30d cost`, `30d tokens`, `Latest tokens`;
- compact account-level history bars;
- `Top model`;
- provenance text such as `Estimated from local Codex logs for the selected
  account`;
- account actions/status rows such as `Cost`, `Subscription Utilization`,
  `Buy Credits`, `Add Account`, `Usage Dashboard`, `Status Page`, and `Refresh`
  where Capsule can expose them read-only, disabled, or route them through
  explicit host/desktop actions without hidden host writes.

The account tab may include local-log estimates because that is how CodexBar
derives account cost/token history, but it must label those estimates clearly.
It must not pretend account-level usage is scoped to the current jackin'
instance.

Provider parity is a hard requirement, but not every provider exposes every field
through a safe API today. If a provider cannot supply a screenshot-level field
without browser-cookie import, host writes, or undocumented high-risk polling, the
adapter must expose an explicit `unsupported` / `unavailable` row for that field
and the implementation PR must call out the decision needed from the operator.
The UI must not hide the gap or synthesize fake account data.

### Visual Rules

- Use the existing Capsule palette: graphite/black background, bright active
  focus, dim secondary text, and provider accent only for meters. Avoid a full
  CodexBar-style macOS card inside the terminal.
- Active tab is filled/high-contrast. Inactive tabs are plain text with an
  optional short meter underline only when there is room.
- Meters use a stable fixed width per layout mode. Text updates must not resize
  the overlay.
- History bars are a compact spark histogram. They are useful but lower
  priority than quota buckets; hide them first in narrow layouts.
- Long account labels, provenance text, model names, and status messages are
  ellipsized with the full value available later through copy/details actions.
- Provider-specific colors are optional accents only. Status words and labels
  carry the meaning.

| `< 64 cols` | List layout: no overview tabs unless focused, one headline quota, bucket rows as text, no history bars. |

## Quota Buckets

The UI must render a list of quota buckets without hardcoding provider-specific
rows. Known examples from the reference screenshots and current provider
behavior:

| Provider | Bucket examples |
|---|---|
| Codex | `Session`, `Weekly`, `Codex Spark 5-hour`, `Codex Spark Weekly`, `Credits` |
| Claude | `Session`, `Weekly`, `Sonnet`, `Daily Routines`, `Extra usage` |
| Amp | `Amp Free`, `Balance`, provider-specific plan/free-tier buckets when a stable source exists |
| Grok Build | `Credits`, subscription/included quota, billing-window reset, local session tokens |
| GLM / Z.AI | token quota, time/MCP quota, session or five-hour token limit, model usage details |
| Kimi | Kimi For Coding weekly quota, five-hour rate limit, membership/tier quota |

Each bucket may have a different unit: tokens, messages, requests,
USD-micros, credits, or provider-defined plan units. The renderer must never
compare percentages across incompatible units. Cross-provider "most constrained"
ranking remains owned by [Token & Cost Telemetry](/reference/roadmap/token-cost-telemetry/#most-constrained-ordering).

### Bucket Rendering

For each bucket, render:

- label;
- horizontal meter when `used` and `limit` are known;
- remaining percent when known;
- reset countdown when known;
- pace/reserve label when known;
- lifecycle label when `NeedsLogin`, `Unsupported`, `Stale`, or `Error`.

If no limit is known, show the amount and source label instead of inventing a
percentage. `unsupported` is better than a blank row.

## Account Spend And History

The spend section inside provider tabs is account/provider attribution, matching
the CodexBar screenshots. It is not the current jackin' instance ledger. It
should show:

- today cost;
- last-30-days cost;
- last-30-days tokens;
- latest-token sample or current-session tokens;
- top model;
- compact history bars;
- provenance text, for example `Estimated from local Codex logs for the
  selected account`.

This section must be labeled separately from provider quota. It can explain what
the account burned over the displayed time window, but it must not be presented
as the enforced subscription limit. The `Instance` tab is the only place where
since-start jackin' instance totals are presented.

## Instance Ledger

The `Instance` tab owns lifetime accounting for one running Capsule instance.
This is additional to the account/provider tabs and must not replace them. It
must aggregate only samples whose provenance ties them to this instance:

- runtime stream samples captured from PTY output chunks for panes in this
  instance;
- agent codenames plus agent/session ids created by this Capsule daemon;
- provider account identity attached to each sample when safely available;
- workspace path and instance id stamped when the sample is recorded;
- local provider log samples only when they can be bounded to this workspace and
  the instance lifetime window without double-counting runtime stream samples.

Required instance totals:

- instance id/name;
- instance start time and age;
- active agent time when the daemon can infer it from focused/running panes;
- current workspace path;
- all agent codenames created in this instance, including closed/exited tabs;
- total tokens since instance start;
- latest/current-session tokens;
- spend since instance start;
- today spend for the same instance;
- exact-cost sample count;
- estimated-cost sample count;
- unpriced sample count;
- top model for the instance;
- compact history bars over the instance lifetime;
- per-agent-codename rows with codename, provider/account, account identity,
  tokens, spend, top model, active/idle/closed/exited state, tab/pane lineage
  when available, and last activity;
- per-provider/account rows so shared account usage caused by this instance is
  visible without mixing in other instances. These rows must include the account
  identity and plan label whenever the adapter can expose them without unsafe
  host-side behavior.

Retention rule: per-codename rows stay queryable for the lifetime of the jackin'
instance even after the visible tab is closed. The daemon may compact raw render
state, but it must keep enough normalized usage samples and lifecycle metadata to
answer "what did codename X cost in this instance?" until the instance itself is
destroyed or its telemetry cache is explicitly purged.

Cost display rules:

- Prefer provider-reported exact cost when a runtime event or local log exposes
  it.
- Use verified provider/model price tables only when exact cost is absent and the
  token split is known.
- Mark estimated totals visibly, for example `estimated at API rates`.
- Keep unpriced token totals visible instead of dropping them.
- Do not infer provider subscription dollars from quota depletion. Quota rows say
  remaining availability; instance rows say observed token/cost attribution.

The instance ledger must survive tab switches and overlay refreshes because it is
read from daemon-owned cached samples. It should not require opening provider
dashboards, importing browser cookies, or writing account state.

## Provider Status

When daemon telemetry includes provider health, the overlay should show a short
status row:

```text
Status Page
Partial System Degradation · Updated 30m ago
```

Capsule may expose a copy/open action later through the explicit host action
path, but the first version can render the row read-only. Provider status should
not block the overlay if unavailable.

## Actions

Capsule V1 actions:

- `Refresh`: asks the daemon to schedule `account/refresh` and keeps showing the
  cached snapshot immediately.
- `Copy summary`: optional later action that copies the visible usage summary
  through the existing terminal clipboard path if available.
- `Open details`: optional later action routed through Desktop/console/host
  bridge, not direct host mutation.

Actions deliberately not owned by Capsule as mutating flows, even when a
CodexBar-parity row is visible in the overlay:

- `Buy Credits` should render as a disabled/read-only billing row unless a
  future explicit host/desktop bridge owns the action;
- `Add Account` should render as a disabled/read-only auth row unless a future
  explicit host/desktop bridge owns the action;
- `Settings`;
- account login/repair flows;
- hidden browser-cookie import;
- provider config writes.

Those belong in Desktop, `jackin console`, CLI, or host bridge because they may
write host state or require richer approval UI.

## States

The overlay must have explicit states:

| State | Required rendering |
|---|---|
| Fresh provider snapshot | Show quota buckets with confidence/source labels. |
| Stale provider snapshot | Show stale label and last fetched time; keep prior data visible. |
| Needs login / needs secret | Show account and repair action name; no hidden repair. |
| Unsupported provider | Show `unsupported` with provider/source explanation. |
| Daemon disconnected | Show `usage unavailable: daemon disconnected`; do not poll providers from Capsule. |
| No focused agent | Show `usage unavailable: no focused agent session`. |
| Narrow terminal | Collapse tabs/columns, keep provider, headline quota, reset, status, and provenance. |

## Terminal Interaction

- Prefix `u` opens the overlay.
- Palette action `Usage` opens the same overlay.
- `Tab` / `Shift+Tab` move between provider tabs when tabs are visible.
- `r` requests refresh while the overlay is open.
- `Esc` closes.
- Mouse clicks may select tabs and actions only where Capsule already supports
  click regions; keyboard interaction is required.

The overlay should preserve the existing Capsule redraw policy: no full-screen
terminal erase when opening, tabbing, refreshing, or closing.

## Host-Side Effects

Capsule is read-only. It must not write provider config, Keychain entries,
browser profiles, host dotfiles, Git config, or user repositories. It must not
poll provider APIs directly from inside the container.

Allowed writes are limited to existing Capsule UI/session state and the daemon's
own host-side telemetry cache described in [Token & Cost Telemetry](/reference/roadmap/token-cost-telemetry/#host-side-effects).
If a repair action would change host state, Capsule shows the required action
and points to the responsible surface.

## Implementation Checklist

Landed in the current implementation:

1. **Contract extraction.** `jackin-protocol` exposes typed focused/account,
   workspace, session, and instance usage structs instead of TUI-specific string
   blobs.
2. **Static TUI fixture.** Capsule has fixture-backed tests for Codex, Claude,
   Overview, `Instance`, status/error states, and narrow rendering through the
   usage-specific Ratatui snapshot.
3. **Status chrome signal.** The active tab can render focused-provider usage
   from daemon cache only. Fresh labels include provider, account identity,
   selected bucket, current usage, remaining quota, and reset text when present.
4. **Daemon wiring.** `UsageFocused`, `UsageRefreshFocused`,
   `UsageAccountList`, `UsageWorkspace`, and `UsageSession` replies are shaped
   inside the daemon from cached snapshots and cached samples.
5. **Refresh action.** Prefix `u`, palette `Usage`, `Tab` /
   `Shift+Tab`, and `r Refresh` are wired through Capsule control dispatch.
   Refresh forces the daemon cache path and the renderer keeps showing cached
   data while the daemon updates.
6. **Instance ledger tab.** The overlay has a first-class `Instance` tab backed
   by daemon sample summaries scoped to the Capsule instance id. It renders
   instance age, active-agent time, workspace, today/since-start spend, tokens,
   latest-token sample, exact/estimated/unpriced counts, top model, history
   bars, per-agent-codename rows, closed-tab lifecycle state, tab/pane lineage,
   last activity, and per-provider/account rows.
7. **CodexBar information parity.** Provider/account tabs now render the
   screenshot-level account source of truth: account header, freshness,
   plan/tier, quota buckets, reserve/deficit/pace labels, reset/run-out text,
   account cost/token windows, compact history bars, top model, provenance,
   provider status, read-only `Cost`, `Subscription Utilization`,
   disabled/read-only `Buy Credits` and `Add Account` parity rows,
   `Usage Dashboard` rows, plus daemon refresh text. Unsupported fields are
   rendered as explicit unavailable/unsupported states instead of being hidden.
8. **Host-safety guardrails.** Capsule remains read-only for provider accounts:
   no browser-cookie import, account repair, hidden host writes, or unmanaged
   background CLIs.

Remaining work before retiring this roadmap item. These are explicit follow-ups,
not blockers for the current operator-testable implementation:

1. **Host-global account daemon.** Replace the explicit
   `jackin usage <instance> accounts --sync-host-cache` bridge with a long-lived
   host account refresh service that can warm provider account snapshots for
   future Desktop/console views without asking each Capsule to poll providers.
2. **Bounded local-log instance attribution.** Runtime stream samples are
   instance-stamped today. Provider-local log samples remain
   account/workspace-level unless they can be safely bounded to this instance
   without double-counting runtime samples.
3. **Provider-specific event contracts.** Keep adding verified token/cost event
   shapes as provider CLIs expose stable structured runtime usage beyond the
   common OpenAI/Anthropic-style fields already parsed.
4. **Future model pricing.** Extend verified price tables only from first-party
   provider documentation. Unknown models stay unpriced.
5. **Custom meter widget polish.** The current renderer uses roadmap-style block
   meters and history bars inside the existing dialog renderer. A later custom
   widget can add provider accent ticks and reserve markers without changing the
   daemon protocol.

## Scope V1

- Focused-agent compact signal in Capsule status chrome.
- Active-tab status bar summary that shows the focused agent's current usage
  and remaining quota whenever daemon telemetry has both values.
- Prefix `u` / palette `Usage` overlay.
- Required `Overview` and `Instance` tabs, followed by provider/account tabs.
- Provider tabs for Claude Code / Anthropic, Codex / OpenAI, Amp, Grok Build,
  GLM/Z.AI, Kimi, and MiniMax when the daemon has a safe extraction path or an
  explicit unsupported state.
- Account-wide provider tabs that match the CodexBar screenshot information
  level where safe data exists, and render unavailable/unsupported rows where it
  does not.
- Instance-lifetime ledger that is separate from account/provider tabs and
  keeps all current and closed agent codenames visible until the instance ends.
- Quota bucket list with reset/pace/source/status labels.
- Spend summary with cost, tokens, top model, history bars, and provenance.
- Provider status row when available.
- Read-only refresh request through daemon.
- Explicit disconnected/stale/needs-login/unsupported states.
- Provider extraction research and normalized adapter plan for Claude Code,
  Codex, Amp, Grok Build, GLM/Z.AI, Kimi, and MiniMax.
- Tests for rendering, input dispatch, cache behavior, status labels, narrow
  layout, and error states.

## Defer

- Full all-provider dashboard inside Capsule.
- Account setup, add-account, buy-credits, settings, or login repair flows.
- Direct browser-cookie import from Capsule.
- Provider-specific bespoke Capsule layouts.
- Budget enforcement or automatic agent pausing.
- Opening external dashboards directly from the container.
- Multi-operator/org-level spend rollups.

## Settled Product Decisions

- `Overview` is visible in Capsule and stays separate from provider detail.
- `Instance` is visible in Capsule and is the only since-start jackin' instance
  ledger.
- Provider tabs are account-wide provider/account views. They are not scoped to
  one app process, one visible tab, or the current instance.
- Account identity is required wherever usage is attributed. When it cannot be
  extracted safely, render `account unavailable`.
- The active status signal follows the focused tab/pane and elides reset text
  first, then current usage, keeping provider plus remaining/state for as long
  as space allows.
- Amp V1 uses `amp --no-color usage` under daemon-managed timeout/cooldown.
- Kimi and MiniMax Capsule adapters stay limited to explicit token/OAuth/API-key
  paths. Browser/session fallbacks belong to a future explicit Desktop/host
  approval flow, not background Capsule refresh.
