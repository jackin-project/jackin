---
title: "Per-scope auth-sync source folder"
---

**Status**: Open — design proposal

## Problem

jackin' `sync` auth mode copies an agent's host credential folder into the role-state directory and bind-mounts it into the container. The host source path is **hardcoded** per agent — `~/.claude`, `~/.codex/auth.json`, `~/.local/share/amp/secrets.json`, `~/.kimi-code`, `~/.local/share/opencode/auth.json` — in the `provision_*_auth` functions in <RepoFile path="src/instance/auth.rs" />. There is no way to point `sync` at a different folder.

Operators who keep credentials outside the default locations have no path today:

- Multiple logins for the same agent (work vs personal Claude account) kept in separate directories, with different workspaces wanting different ones.
- Credentials stored in a non-standard location (a synced/encrypted volume, a per-project secrets directory) rather than the tool's default home path.
- A single workspace that must use a completely different credential set from the global default.

The default locations must stay the default — the common case needs zero configuration — but `sync` should accept an operator-chosen source folder, settable globally or per workspace, exactly like the rest of the layered auth model.

### Motivating use case

The driving case is an operator with **two accounts for the same agent**: a personal Claude subscription and a company-provided Claude account, each logged in under a different host folder (Claude Code honors an environment override that relocates its config directory, so this is a real, supported split). The operator wants the personal account to be the global default and the company account to be used only in the workspaces tied to that company. With a per-scope source folder, the global default points at the standard `~/.claude`, and a company workspace overrides the source to the company folder — every container launched from that workspace syncs the company login, every other workspace keeps the personal one. The same pattern applies to any agent whose credentials an operator keeps in more than one place.

## Why It Matters

- The auth model is already layered per (workspace × role × agent) for the *mode* (`sync` / `api_key` / `oauth_token` / `ignore`). The *source* of a `sync` is the one axis that is not configurable, so operators with more than one identity per agent are stuck.
- Selecting a folder is a solved interaction in the console: the workspace-mount flow already has a directory picker (`FileBrowserState`). Reusing it keeps the operator experience consistent and avoids a second bespoke picker.
- This is strictly additive and read-only against the host — it changes *which* host folder jackin' reads, never writes anything host-side — so it fits the existing `sync` contract without new risk.

## Current behavior

- **Host source paths** are hardcoded in each `provision_*_auth` in <RepoFile path="src/instance/auth.rs" /> as `host_home.join("<relative-path>")`. No struct carries a source path.
- **Mode config** is `AgentAuthConfig { auth_forward: AuthForwardMode }` in <RepoFile path="src/config/mod.rs" />, embedded in both `AppConfig` (global, `config.toml`) and `WorkspaceConfig` (per-workspace files). Mode is resolved most-specific-first by `resolve_mode` in <RepoFile path="src/config/roles.rs" />.
- **The directory picker** is `FileBrowserState` in <RepoFile path="src/console/widgets/file_browser/state.rs" />, already generic: it returns a bare `PathBuf` and knows nothing about mounts. It is opened today from the workspace editor and the settings global-mounts flow.
- **Hidden folders are filtered out** by a single guard in `load_entries` (<RepoFile path="src/console/widgets/file_browser/state.rs" />): `if name.starts_with('.') { continue; }`. Because every default credential folder is a dotfile (`~/.claude`, `~/.codex`, …), the picker cannot currently show them.
- **The Auth tab** `sync` flow only sets the mode — no folder step. The settings Auth modal (`SettingsAuthModal` in <RepoFile path="src/console/manager/state.rs" />) has no `FileBrowser` variant.

## Proposal

Add an optional per-agent **source-folder override** to the auth config, resolved with the same layering as the mode, consumed by the provisioning path, and selected in the console with the existing directory picker taught to show hidden folders. The default (no override) preserves today's hardcoded paths exactly.

### Schema (versioned)

Add an optional field to `AgentAuthConfig` (and the GitHub equivalent if GitHub is in scope — see Open questions):

```toml
[claude]
auth_forward = "sync"
sync_source_dir = "/Users/me/secrets/claude-work"   # new, optional
```

`#[serde(default, skip_serializing_if = "Option::is_none")]` keeps files without the field round-tripping cleanly. Because `AgentAuthConfig` is embedded in both `AppConfig` and `WorkspaceConfig` — both versioned schemas — this addition requires the full versioned-change artifact set per the pre-release schema rule:

1. Bump `CURRENT_CONFIG_VERSION` and `CURRENT_WORKSPACE_VERSION` (currently `v1alpha5`) to the next version in <RepoFile path="src/config/migrations.rs" />.
2. A no-op migration step in both `CONFIG_MIGRATIONS` and `WORKSPACE_MIGRATIONS` (the field is additive with a serde default).
3. New fixture dirs under `tests/fixtures/migrations/config/from-v1alpha5/` and `tests/fixtures/migrations/workspace/from-v1alpha5/`.
4. Re-bake every existing fixture's `after.toml` through the new step.
5. A Timeline entry in [Schema Versions](/reference/schema-versions/).

One version bump covers all affected file kinds, per the one-bump-per-PR rule. `RoleManifest` is untouched (role manifests cannot control auth source).

### Resolution

Add a companion resolver `resolve_sync_source_dir` next to `resolve_mode` in <RepoFile path="src/config/roles.rs" />, walking the same three layers (workspace-role override → workspace → global), returning `Option<PathBuf>`. It keeps the source axis symmetric with the mode axis — the DRY rule wants both resolved by parallel functions reading the same layered structs, so they advance together.

#### Default vs explicit (the load-bearing distinction)

A layer either has an **explicit** folder set, or it carries **no value** ("default"). "Default" is never a stored path — it is the *absence* of a value at that layer, which means "inherit from the layer below." The chain is:

1. **Workspace explicit** — a folder the operator chose for this workspace. Wins outright.
2. **Global explicit** — a folder the operator chose at the global level. Used when the workspace has no explicit value.
3. **Agent hardcoded default** — `host_home.join(AGENT_DEFAULT_REL)` (e.g. `~/.claude`). Used when neither layer set anything.

So in the config, a layer with `sync_source_dir = None` is "default". The effective folder is the first explicit value walking workspace → global → hardcoded. This is exactly how `resolve_mode` already treats an absent mode, so the two resolvers behave identically and a reader who understands one understands the other.

Concretely for the motivating use case:

- **Global** Auth tab: leave Claude's source folder at default → resolves to `~/.claude` (personal).
- **Company workspace** Auth tab: explicitly select `~/company/.claude` → that workspace resolves to the company folder; every other workspace still resolves to `~/.claude`.

The displayed label must distinguish the two states so the operator always knows whether a value is explicit or inherited:

- **Global, no explicit value:** `(default) ~/.claude` — shows the hardcoded fallback that will be used.
- **Global, explicit value:** `~/secrets/claude-personal` — shows the chosen path plainly.
- **Workspace, no explicit value:** `(default — follows global) ~/.claude` — makes clear it inherits whatever global resolves to, and shows the resolved result.
- **Workspace, explicit value:** `~/company/.claude` — the override, shown plainly.

### Provisioning

Each `provision_*_auth` in <RepoFile path="src/instance/auth.rs" /> takes the resolved override and falls back to the current hardcoded relative path when it is `None`. Factor the per-agent default into a single place so the default and the override flow through one code path:

```rust
let source = resolved_source_dir.unwrap_or_else(|| host_home.join(AGENT_DEFAULT_REL));
```

This is read-only against the host (the existing `sync` contract). When the chosen folder is missing the expected credential file, fall back to the existing "no forwarded auth" warning path rather than failing the launch.

### Operator flow

The whole interaction lives inside the existing per-agent auth dialog. Today that dialog lets the operator pick the **mode** (sync / api_key / … / ignore). **Sync is the default mode** — an agent the operator never touches is already in Sync, so the **Source folder** row this feature adds is relevant by default, not only after a mode change. The row is visible whenever the mode is Sync (the default). The flow, end to end:

1. **Open the auth dialog.** In Settings (global) or in the workspace editor, go to the **Auth** tab, move the cursor to an agent (e.g. Claude), and press **Enter**. The per-agent dialog opens. The agent is already in **Sync** mode by default, so the Source folder row is shown immediately; the operator does not have to select Sync first.
2. **Sync mode (the default).** The mode row already reads **Sync** unless the operator previously changed it. The **Source folder** row sits below the mode row whenever the mode is Sync. (If the operator switches to a non-Sync mode such as api_key or ignore, the Source folder row hides, since it only governs Sync; switching back to Sync brings it back.)
3. **The Source folder row shows the pre-selected default.** Before the operator touches it, the row is pre-filled with the default value for that agent — `(default) ~/.claude` for Claude, the matching default for every other supported agent. This default is **not stored**; it is always recomputed as "the default for this setting" (hardcoded path at the global layer, "follow global" at the workspace layer). The operator can save right here and nothing about the source is persisted — the row was informational, confirming what Sync will read.
4. **Press Enter on the Source folder row to change it.** This opens a new dialog — the **same** directory picker used when adding a workspace mount (`FileBrowserState`). The operator navigates with **Up/Down** to move through entries, **Enter** to descend into a folder, and selects the current folder with **`s`** (the same select key the mount picker uses). Hidden (dot) folders are visible here, so `~/.claude`, `~/.codex`, etc. can be reached.
5. **Selection returns to the auth dialog.** After pressing `s`, the picker closes and control returns to the per-agent auth dialog. The Source folder row now shows the chosen folder (plainly, no `(default)` prefix — it is now explicit).
6. **Save to apply.** The operator saves the dialog. The auth dialog closes and the view returns to the Auth tab list exactly as before. The chosen folder is now persisted at this layer (global config or the workspace file).
7. **Reset to default.** When a folder has been explicitly selected, the Source folder row offers a **reset** action that clears the explicit value back to default (`None`). After reset the row again shows `(default) …`, and saving persists the cleared state. Reset is only meaningful when an explicit value exists — for an already-default row it is a no-op.

This is the same dialog → picker → back-to-dialog → save loop the operator already knows from workspace mounts; only the entry point (the Source folder row) and the show-hidden behavior are new.

### Picker changes

- Add a `show_hidden: bool` to `FileBrowserState` (default `false`) threaded into `load_entries`, gating the dotfile filter at <RepoFile path="src/console/widgets/file_browser/state.rs" />. The auth flow constructs the picker with `show_hidden = true`; the mount flow keeps `false`, so mount behavior is unchanged. No other picker behavior changes — `s` to select, Enter to descend, the sandbox-root and `~/.jackin/` reject rules in `commit_or_reject` all carry over.
- Add a `FileBrowser` variant to `SettingsAuthModal` (settings Auth tab, <RepoFile path="src/console/manager/state.rs" />) and to the workspace-editor auth modal, dispatched the same way the mount file browser is dispatched in <RepoFile path="src/console/manager/input/global_mounts.rs" />. The committed `PathBuf` is written to a pending field on the auth state, then persisted on save.

### Both surfaces behave identically

The Settings Auth tab (global) and the workspace-editor Auth tab offer the **same** Source folder row, the **same** picker, the **same** keys, and the **same** reset action. The only difference is which config layer the saved value lands in: global writes to `config.toml`, the workspace editor writes to that workspace's file. Per the reuse rule, both surfaces route through **one** shared "render Source-folder row / open picker / persist or reset result" helper rather than two parallel copies — mirroring how `resolve_mode` already serves both layers from one function. Building it as two siblings is the exact drift the DRY rule blocks (one surface gets reset, the other does not; one shows the inherited-from-global label, the other does not).

## Host-side effects

None that mutate the host. The feature only changes **which** host folder `sync` *reads*; jackin' never writes to the operator-chosen folder. The chosen path is stored in jackin' own config (`config.toml` / workspace files), not in any host dotfile. This is the same read-only contract the current `sync` mode honors.

## Per-agent applicability

| Agent | Default source | Override applies |
|---|---|---|
| Claude Code | `~/.claude` + `~/.claude.json` | Yes |
| Codex | `~/.codex/auth.json` | Yes |
| Amp | `~/.local/share/amp/secrets.json` | Yes |
| Kimi | `~/.kimi-code` | Yes |
| OpenCode | `~/.local/share/opencode/auth.json` | Yes |
| GitHub CLI | `gh auth token` subprocess, `~/.config/gh/hosts.yml` fallback | See Open questions |

## Phases

1. **Schema + resolution** — add `sync_source_dir` to `AgentAuthConfig`, the version bump + migrations + fixtures, and `resolve_sync_source_dir` with the default-vs-explicit layering above. No behavior change yet (nothing reads the field).
2. **Provisioning** — wire the resolved override into `provision_*_auth` with default fallback and the missing-file warning path. Now CLI-set overrides take effect.
3. **Console UX** — `show_hidden` on the picker; the Source folder row (with the default/explicit/inherited labels) on both Auth-tab surfaces; the shared render/open-picker/persist helper; reset-to-default. The row appears only in Sync mode and pre-fills the recomputed default.

## Open questions

- **GitHub CLI.** `sync` for GitHub primarily reads via the `gh auth token` subprocess, not a folder, with `~/.config/gh/hosts.yml` only as a fallback. A folder override is meaningful only for the file fallback. Decide whether to expose the override for GitHub at all, or scope Phase 1–3 to the five file-based agents and treat GitHub separately.
- **Relationship to the unified [Credential Source Pattern](/reference/roadmap/credential-source-pattern/).** This folder override is the "filesystem source" axis. If the cross-cutting credential-source abstraction lands, the source folder should become one source kind under it rather than a standalone field. Decide whether to ship this independently first (small, self-contained) or hold for the abstraction.
- **Interaction with [Live bidirectional auth sync](/reference/roadmap/live-auth-sync/).** Live sync reconsiders the `sync` name and adds bidirectional follow-up. A per-scope source folder is orthogonal (it picks the read location) but both touch the same provisioning path; sequence them so the live-sync work consumes the resolved source rather than the hardcoded path.
- **Validation depth.** Whether to validate at config-set time that the chosen folder contains the expected credential file, or only warn at launch (the current provisioning behavior).
