## Remaining undocumented modules

`src/` coverage is complete as of 2026-06-02 — only `main.rs`, `bin/role.rs`, and two `_tests.rs` files lack `//!` blocks (all intentional: binary entry points and test modules do not need orientation headers).

The next pass is `crates/`: `grep -rL '^//!' crates/ --include='*.rs'` to get the current list. The `jackin-tui` and `jackin-console` crates have higher `//!` coverage already; the `jackin-capsule` crate's large files are the most urgent.

## Steps

1. For each file, read the first 50 lines and the public item list.
2. Write a `//!` block at the top: purpose, scope claim, key invariants, what callers can assume.
3. Model on <RepoFile path="src/env_model.rs" /> (the 3-element exemplar: purpose + scope + history).
4. Do not add `///` item-level docs in this pass — only `//!` module-level.

## What a good `//!` block says

- **Purpose:** What does this module do?
- **Scope:** What is it NOT responsible for? (crucial for AI agents to know where not to look)
- **Key invariants:** What must always be true when this module's public functions return?

## Caveats

Purely additive — no behavior change, no CI risk. Do before the behavioral specs since the spec presupposes the module has a stated contract.
