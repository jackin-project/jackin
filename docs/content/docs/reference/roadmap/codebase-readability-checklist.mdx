## Prerequisite decisions (operator sign-off)

Three judgment calls gate the rest of Stage 1 / Stage 4. Each ripples across many files, so decide once before any code moves. None should be resolved silently by an agent.

- [ ] **Core error-type policy.** `DockerApi` (16 methods) and `CommandRunner` (3 methods) return `anyhow::Result<ŌĆ”>` today. `jackin-core` currently carries only `serde` + `thiserror`. Choose: **(a)** add `anyhow` to `jackin-core` and move the traits as-is (low friction, ~50-file ripple is zero), or **(b)** introduce typed `thiserror` errors in core (`DockerError`, `RunnerError`) and rewrite the 29 + 21 consumer sites. This is the single biggest Stage 1 blocker ŌĆö it gates both trait moves. Recommendation in the research: (a), because the typed-error rewrite touches the bollard impl's error conversions too.
- [ ] **`JackinPaths` purity vs ergonomics.** `JackinPaths::detect()` needs `directories` + `anyhow`; `ensure_base_dirs()` does `std::fs::create_dir_all` (forbidden in a "fs-free" core regardless). Choose: **(a)** add `directories` + `anyhow` to core, move `detect()` in, keep only `ensure_base_dirs()` as a binary free fn (preserves method-call ergonomics across 73 consumer files; only the handful of `ensure_base_dirs` sites change), or **(b)** keep core fs/`directories`-free ŌĆö move only the struct + pure constructors (`resolve_with_env`, `for_tests`), keep `detect`/`ensure_base_dirs` as binary-side free fns / an extension trait (more call-site churn). The roadmap line "add `directories` to core" assumes (a) but cannot move `ensure_base_dirs` either way.
- [ ] **Selector prefix source.** `is_reserved_builtin_role_name` / `is_valid_container_name` use `CONTAINER_PREFIX_DASH` (`"jk-"`), which actually lives in `jackin-protocol` (re-exported via `instance/naming/mod.rs:9-11`), **not** `instance/naming` directly. Choose: **(a)** add `jackin-protocol = { path = "../jackin-protocol" }` to `jackin-core` and re-export `CONTAINER_PREFIX` / `CONTAINER_PREFIX_DASH` (no literal drift; `jackin-protocol` is serde-only and acyclic with core), or **(b)** hardcode `"jk-"` in core with a source-of-truth comment pointing at `jackin_protocol::CONTAINER_PREFIX_DASH`. The earlier checklist line said "hardcode the `jk-` prefix" ŌĆö that is option (b).

## Target repository structure (end state)

The program is finished when the workspace is a **tiered DAG**: a small, stable, dependency-light vocabulary at the bottom; the heavy machinery that reads/writes/orchestrates it in the middle; a thin binary on top. Cargo forbids cycles between crates, so every Stage-4 extraction is converging on this shape. Dependencies point **only downward** ŌĆö a higher tier may name a lower tier, never the reverse.

### DAG invariants ŌĆö final verification checklist
- [ ] The full inter-crate graph is acyclic ŌĆö `cargo build --workspace` succeeds with no `cyclic package dependency` error; spot-check with a dependency-graph tool
- [ ] `jackin-core` carries no `tokio`, no `bollard`/docker, no `std::fs` write path, no `*Config` type ŌĆö `cargo tree -p jackin-core` shows only the sanctioned leaf deps
- [ ] `jackin-config` does not name `jackin-env`, `jackin-runtime`, `instance`, or `isolation` ŌĆö `cargo check -p jackin-config` passes with no binary-crate dep
- [ ] `jackin-env` does not name `jackin-runtime` ŌĆö `cargo check -p jackin-env` passes standalone
- [ ] `jackin-runtime` is the **only** crate depending on `bollard` and the privileged-docker code paths
- [ ] The root binary contains only `cli/`, `app/`, `main.rs`, `bin/` glue ŌĆö every domain `use` resolves to `jackin_{core,config,env,runtime}`
- [ ] **Layering cleanup ŌĆö break the `jackin-env ŌåÆ jackin-console` edge:** `OpCli` imports the 1Password data aliases (`OpAccount`, `OpVault`, `OpItem`, `OpField`, `OpCache`) from `jackin_console::tui::components::op_picker`. A non-visual logic crate depending on a TUI crate inverts the tier order. Move those plain data types **down** to `jackin-core` (or `jackin-config`) and have both `jackin-env` and `jackin-console` import them from there, so `jackin-env`'s dependencies are exactly `core` + `config`
- [ ] Every new crate uses the self-named Rust 2024 layout (`foo.rs` + `foo/`, no `mod.rs`) and enforces `clippy::mod_module_files = "deny"`

## Rust best-practices acceptance gates (every PR under this program)

These are the per-diff checks a reviewer verifies, drawn from the roadmap's "Conventions every refactor follows" and the repo rules in <RepoFile path="AGENTS.md" />. They are written as pass/fail gates, not aspirations ŌĆö a PR that fails one is incomplete. They compose with, and do not replace, the structural tasks above.

### Crate boundaries (Stages 1 & 4)
- [ ] Public API is deliberate: `pub(crate)` by default, `pub` only on what the next crate up imports
- [ ] Struct fields stay private unless the representation is part of a stable contract (the serde `*Config` structs expose fields intentionally; most other types do not)
- [ ] `jackin-core` stays types + traits only ŌĆö no `tokio`, no subprocess, no filesystem; the I/O seam is the `DockerApi` / `CommandRunner` traits (dependency inversion), concrete impls in `jackin-runtime`
- [ ] Invariants encoded in types ŌĆö newtypes and enums over `bool` / `String` / bare primitives, so each higher crate inherits the constraint instead of re-checking it
- [ ] Standard traits derived where they make semantic sense on the `jackin-core` vocabulary ŌĆö `Debug`, `Clone`, `PartialEq`/`Eq`, `Hash`, `Default` for data; `Display` + `std::error::Error` for errors; `From` / `TryFrom` for conversions
- [ ] `derive(Serialize, Deserialize)`, public re-exports, and public dependencies treated as intentional compatibility commitments ŌĆö present only where the serialized shape or the re-export is part of the contract
- [ ] Sealed trait used where a downstream `impl` would block future evolution (the `AgentRuntime` trait ŌĆö private `sealed::Sealed` supertrait)

### Shared extractions (Stage 2)
- [ ] Extracted in a shape future callers reuse ŌĆö a parameter or small enum on the existing function beats a second near-copy
- [ ] Borrowed parameters when ownership is not required: `&str` over `String`, `&[T]` over `Vec<T>`, `&Path` over `PathBuf`, `Option<&T>` over `&Option<T>`
- [ ] `.clone()` is a deliberate design decision with a reason, never a borrow-checker appeasement; no needless intermediate `Vec` / `String` allocations
- [ ] Any new third-party dependency is justified in the PR (internal crate splits are free; an external crate carries compile, supply-chain, and API cost)

### File splits & functions (Stage 3)
- [ ] First-time-reader order: entry-point items first, important types before helpers, implementation near its use
- [ ] Control flow stays local and visible; preconditions expressed in types or adjacent checks, not a distant helper; clear branching over a clever combinator chain that hides errors/ownership/side effects
- [ ] Boring, searchable names over abbreviations (short names only for `ctx`, `idx`, `res`, `db`-style conventions)
- [ ] Every newly split module gets a `//!` contract stating the non-obvious why (invariants, ordering, race windows), not a restatement of the code

### Errors
- [ ] `Result<T, E>` for expected failure; `Option<T>` only when absence carries no useful detail
- [ ] **Typed errors (`enum` + `thiserror`) at crate boundaries** ŌĆö the new `jackin-core` / `-config` / `-env` / `-runtime` library crates return typed errors; `anyhow`-style context only at the binary / CLI / app edge
- [ ] `?` for propagation, with context added at the I/O, parse, network, and user-facing boundaries
- [ ] `unwrap` / `expect` reserved for tests, unreachable invariants, and programmer errors with precise messages; any production `unwrap` touched is converted to `expect("why")` or a typed error ŌĆö no new ones added
- [ ] Public library functions document failure with a rustdoc `# Errors` section (and `# Panics` where a panic is reachable, `# Safety` for any `unsafe`)

### Tests & docs (Stages 0 & 3)
- [ ] Tests exercise a crate's **stable public boundary**, not the incidental private helpers about to move (a test that freezes a private signature blocks the very refactor this program performs)
- [ ] `insta` snapshots kept deterministic and redacted, so a structure-only refactor produces an empty diff
- [ ] Public-API doc-test examples propagate with `?`, not `unwrap`, so the example documents the error path too
- [ ] The failure path is covered, not only the success path

### Lints & validation
- [ ] A Clippy warning is fixed before it is suppressed; a justified suppression uses `#[expect(clippy::lint_name, reason = "ŌĆ”")]` so it fails loudly when stale; a new `#[allow(clippy::too_many_lines)]` is a request to extract, not annotate
- [ ] The new library crates carry no crate-level `#![deny(warnings)]` and do not switch on `pedantic` / `restriction` / `nursery` wholesale ŌĆö strict lints applied selectively, by name
- [ ] Each PR runs `cargo fmt --check`, `cargo clippy --all-targets --all-features --locked -- -D warnings`, and the test suite before it is marked ready
