---
title: "Codebase readability — implementation checklist"
---

The ordered, agent-executable checklist for the [codebase readability program](/reference/roadmap/codebase-readability/). The end state is **no root `src/` crate**: every domain module becomes a single-responsibility workspace crate, the only binary (`jackin`) is itself a crate (`crates/jackin/`), the inter-crate graph is a DAG, and no `mod.rs` file remains anywhere.

**How to use this document.** Work top to bottom. Each numbered **Phase** is one dependency-gated unit; most phases are several PRs. Do not start a phase until its **Gate** is satisfied — the gates encode the only safe order. Every `[ ]` is a concrete, independently verifiable task. Re-measure any count before relying on it; where a number is given it was measured on `feature/tui-architecture` on 2026-06-03 and is flagged stale if it drifts.

**Legend:** `[ ]` not done · `[~]` partial · `[x]` done. Counts like *(61 sites)* are blast-radius; they size the re-export shim, not extra work.

## Groundwork already complete (do not redo)

The TUI surface crates (`jackin-tui`, `jackin-console`, `jackin-launch`, `jackin-capsule`, `jackin-protocol`, `jackin-tui-lookbook`, `jackin-build-meta`) are extracted. `jackin-core` ships the universal vocabulary (`Agent`, `MountIsolation`, `AuthForwardMode`, `EnvValue`/`OpRef`/`FieldTarget`, `RoleManifest` + manifest types, `Selector`/`RoleSelector`, `JackinPaths`, the `DockerApi`/`CommandRunner` traits + their data types, `op_types`, constants). `jackin-config` ships the config/workspace **schema** (`WorkspaceConfig`, `MountConfig`, `*AuthConfig`, `RoleSource`, `DockerConfig`, planner, `validate_mounts`, `ResolvedWorkspace`, version constants). `jackin-env` ships `OpRunner`, `resolve_env_value`, `EnvLayer`, `merge_layers`, `OpStructRunner`/`OpWriteRunner`/`OpItemCreateParams`. Every module dependency **cycle** is cut (`config↔workspace`, `config↔operator_env`, `env_model↔manifest`, `agent→config`, `config→runtime::list_role_names`). The remaining work is the bottom-up extraction below.

## Pre-flight decisions (locked — do not re-litigate)

These five forks were resolved during planning. Each is recorded with its rationale so the executing agent applies it without re-deciding.

- [ ] **D1 — host observability/terminal substrate → one `jackin-diagnostics` crate (Tier 1).** The `debug_log!` macro (defined `tui/mod.rs:181`, `#[macro_export]`, expands to `$crate::tui::is_debug_mode()`/`emit_debug_line()`), the terminal-ownership flags (`rich_terminal_owned`, `host_screen_owned`, `set_rich_surface_active`, `reassert_alt_screen`), and `RunDiagnostics`/`active_run`/`active_debug` are one mutually-coupled cluster (`emit_debug_line` calls `diagnostics::active_debug`). They all move into **`jackin-diagnostics`** (deps: core, tui). `tui/animation` + `tui/prompt` + `prune_output` + `ansi_text` are pure terminal formatting and go to **`jackin-tui`**. Rationale: `jackin-docker` (Tier 1) already calls all three families, so the substrate must sit at/below Tier 1 or nothing in the runtime tier compiles.
- [ ] **D2 — `FakeDockerClient` / `FakeRunner` → `jackin-core` behind a `test-support` feature.** They implement the `DockerApi`/`CommandRunner` traits (which live in core) and are consumed by tests in app, config, console, isolation, capsule, runtime (6+ crates). They cannot live in `jackin-runtime` (Tier 4). Put them in `jackin-core` under `#[cfg(feature = "test-support")]`; consumers add `jackin-core = { …, features = ["test-support"] }` as a dev-dependency. Rationale: lowest common crate that owns the traits.
- [ ] **D3 — `repo` + `repo_contract` → `jackin-manifest` (not runtime).** `repo_contract::validate_agent_dockerfile` returns `repo::RoleRepoValidationError`, and `derived_image` (image tier) depends on `repo::ValidatedRoleRepo` + `repo_contract`. Putting `repo` in runtime would create an `image → repo → runtime → image` cycle. Move `repo` (locate/validate role-repo structure), `repo_contract`, and `RoleRepoValidationError` together into `jackin-manifest`. The git clone/update half already lives separately in `runtime/repo_cache`. Rationale: role-repo *validation* is a manifest-contract concern; only *fetching* is runtime.
- [ ] **D4 — `jackin-env` gains a `jackin-console` dependency.** `operator_env` needs `OpCache` (alias of `jackin_console::tui::components::op_picker::OpPickerCache`) and `parse_op_reference` (`jackin_console::op_reference`). Adding `env → console` is acyclic (console depends on neither env nor runtime — verified). Accept the dependency now; an optional later cleanup may relocate `op_reference`/`op_cache` down to `jackin-core` to restore a strict `core → config → env` line, but that is not required.
- [ ] **D5 — `prune_output` → `jackin-tui`.** It is owo_colors terminal formatting consumed by `diagnostics::prune_all_runs` and the cleanup CLI. Folding it into `jackin-tui` lets `jackin-diagnostics` depend on it through the tui crate. Rationale: it is presentation, and diagnostics already depends on tui for `ansi_text`.

## Conventions every extraction PR follows

- [ ] **No `mod.rs`.** Each file moved into a crate is renamed to the self-named Rust 2024 layout (`foo/mod.rs` → `foo.rs`, `foo/bar.rs` unchanged) in the same PR. Every new crate sets `clippy::mod_module_files = "deny"` in its crate manifest. The root binary enables the lint **last**, once it reaches zero `mod.rs`.
- [ ] **Re-export shim during transition.** When a module's definitions move down, leave the binary-side module as `pub use jackin_<crate>::…;` so the consumer sites (counted per phase) compile unchanged. The shims are deleted in Phase 8.
- [ ] **Serde byte-identical for versioned schemas.** Moving `AppConfig`, `WorkspaceConfig`, or `RoleManifest` must not change any `#[serde(...)]` attribute, field order, `default`, `skip_serializing_if`, or `rename`. A relocation that preserves the serialized shape needs **no** `CURRENT_*_VERSION` bump, migration step, or fixture re-bake. Acceptance check: diff every `#[serde]` line of the moved type against the original, and confirm `tests/migration_fixtures.rs` still passes. (`isolation.json` / `instances.json` carry a plain `version: u32` hard-fail and are not migrated families — byte-identical moves only.)
- [ ] **Typed errors at crate boundaries.** Library crates return `thiserror` enums; `anyhow` context stays at the binary/CLI edge. `pub(crate)` by default; `pub` only what the next crate up imports.
- [ ] **Per-PR validation.** `cargo fmt --check`, `cargo clippy --all-targets --all-features --locked -- -D warnings`, the test suite, and `cargo check -p <new-crate>` **standalone** (proves no accidental upward edge). On the dev box run heavy `cargo` with `-j 1` (`CARGO_BUILD_JOBS=1`) to avoid the link-time OOM.
