title: "jackin-exec: Secure Execution Wrapper"
---

**Status**: Open — design complete, implementation pending (can start on any backend, no prerequisites)

## What This Is

`jackin-exec` is a wrapper binary installed in every jackin' role container. When an agent needs to run a command that requires credentials — SSH connections, registry logins, cloud CLI commands, git push — it calls `jackin-exec <command>` instead of running the command directly. jackin-capsule intercepts, injects the required credential from the operator's host vault at execution time, runs the command, filters the output, and returns sanitized results. The raw credential value never enters the agent's context.

This is a standalone layer that works on top of any jackin' backend — Docker, Apple Container, or smolvm. It does not require the Apple Container backend to land first.

For the complete picture of how `jackin-exec` fits with the other security layers, see [Agent isolation architecture](/reference/roadmap/agent-isolation-architecture/).

## The Problem

The current jackin' model injects agent credentials as environment variables or synced files at container launch. This means:

- The agent process can `printenv` and read raw API keys
- The agent can read credential files from the agent home mount
- The agent retains credential values in its context window across prompts — a future prompt can reference or exfiltrate them
- Multiple agents sharing a workspace can access each other's credentials if one is compromised
- An agent asked to "share your environment" can extract raw token values

A VM boundary (Apple Container) does not close this gap. If the API key is an environment variable inside the VM, the agent can still read it. `jackin-exec` closes this gap independently of the backend.

The operator's goal: "I trust this agent to connect to my server, but I do not want it to have my raw SSH private key. I want jackin' to make the connection happen on behalf of the agent."

## Architecture and Data Flow

### Binary placement

`jackin-exec` is a new `[[bin]]` target in <RepoFile path="crates/jackin-capsule/Cargo.toml" /> (source at `crates/jackin-capsule/src/exec_main.rs`). It is deployed to `/jackin/runtime/jackin-exec` inside derived role images, exactly like `jackin-capsule` — via a `COPY` step in the dynamically-generated derived Dockerfile in <RepoFile path="src/derived_image.rs" />. The host-side acquisition follows the same env override → cache hit → download pattern as <RepoFile path="src/capsule_binary.rs" />.

### Communication: control channel on the capsule Unix socket

`jackin-exec` communicates with the capsule daemon via the existing Unix socket at `/jackin/run/jackin.sock` using the **control channel** protocol (first byte `0x00` = length-prefixed JSON, one request / one response). Two new variants are added to <RepoFile path="crates/jackin-protocol/src/control.rs" />:

```rust
// added to ClientMsg:
ExecCommand { command: String, args: Vec<String> }

// added to ServerMsg:
ExecResult { exit_code: i32, stdout: String, stderr: String, redacted_count: u32 }
ExecDenied { reason: String }
```

The existing `#[serde(other)]` forward-compat variants on both enums ensure older capsule versions safely return `Unknown` if they receive an `ExecCommand` from a newer `jackin-exec`.

### Credential storage: capsule-private file at launch

The host CLI resolves all workspace credential bindings at container launch time and writes them to `~/.jackin/sockets/<container>/exec-creds.json` (permissions `0600`, owned by root). This path is bind-mounted into the container at `/jackin/run/exec-creds.json`. The agent process (running as the `agent` user) cannot read `/jackin/run/` — it is `0700` root-owned. jackin-capsule reads the credential file at daemon startup and holds the resolved values in memory.

For `op://` credentials: the host CLI runs `op read "op://<vault>/<item>/<field>"` at launch time and writes the resolved value to the credential file. Live re-resolution (for long-running sessions where 1Password items rotate) requires the future jackin' daemon (host bridge). Phase 1 pre-loads at launch — this is sufficient for the current use case.

### Full execution flow

```
WorkspaceConfig.secure_exec.bindings → host resolves credentials at launch
  → writes /jackin/run/exec-creds.json (capsule-private, 0600)

Container starts:
  → capsule reads exec-creds.json → ExecCredentials map held in memory

Agent in session:
  → calls: jackin-exec ssh sentry
  → jackin-exec connects to /jackin/run/jackin.sock (control channel, 0x00 first byte)
  → sends: ClientMsg::ExecCommand { command: "ssh", args: ["sentry"] }
  → blocks waiting for response

Capsule daemon receives ExecCommand:
  1. Matches command + args against loaded ExecCredentials binding patterns
  2. If require_approval = "always" or "sensitive" AND client is attached:
       → sets pending_exec_approval in daemon state
       → renders approval overlay modal in TUI over the current pane
       → blocks until operator presses [A]pprove / [D]eny / [S]ession / [W]aive
  3. If approved (or auto-approved):
       → materializes credential to /jackin/run/secrets/<uuid> (0600, tmpfile)
       → spawns: ssh -i /jackin/run/secrets/<uuid> sentry
       → captures stdout + stderr until command exits
       → scans output against redaction patterns → replaces matches with [redacted by jackin']
       → deletes /jackin/run/secrets/<uuid>
       → sends: ServerMsg::ExecResult { exit_code, stdout, stderr, redacted_count }
  4. If denied:
       → sends: ServerMsg::ExecDenied { reason }

jackin-exec receives ExecResult:
  → writes stdout/stderr to its own stdout/stderr
  → exits with exit_code
```

The agent receives the command output. The raw credential value never appears in the agent's environment, `/home/agent/`, or context.

## Workspace Config Schema

New section in `WorkspaceConfig` in <RepoFile path="src/workspace/mod.rs" />:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Default)]
pub struct SecureExecConfig {
    #[serde(default, skip_serializing_if = "is_default_approval")]
    pub require_approval: ApprovalMode,   // default: Sensitive
    #[serde(default, skip_serializing_if = "BTreeMap::is_empty")]
    pub bindings: BTreeMap<String, CredentialBinding>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum ApprovalMode {
    Always,
    Sensitive,   // only for commands matched in bindings
    Never,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct CredentialBinding {
    #[serde(rename = "match")]
    pub match_pattern: String,   // glob: "ssh sentry*"
    #[serde(flatten)]
    pub credential: CredentialSource,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum CredentialSource {
    Credential(String),   // single value: "op://Vault/Item/Field" | "env:VAR" | "file:/abs/path"
    CredentialEnv(BTreeMap<String, String>),  // multiple env vars
}
```

Operator TOML:

```toml
[workspaces.my-service.secure_exec]
require_approval = "sensitive"

[workspaces.my-service.secure_exec.bindings]
sentry-ssh   = { match = "ssh sentry*",           credential = "op://Personal/sentry-ssh/private-key" }
ghcr-token   = { match = "docker login ghcr.io*", credential = "op://Work/ghcr/token" }
github-token = { match = "gh *",                  credential = "env:GH_TOKEN" }
aws-creds    = { match = "aws *",                 credential_env = { AWS_ACCESS_KEY_ID = "op://Work/aws/key-id", AWS_SECRET_ACCESS_KEY = "op://Work/aws/secret" } }
```

Credential source prefixes in V1:
- `op://<vault>/<item>/<field>` — 1Password via `op read` on host at launch
- `env:<VAR>` — host environment variable at launch time
- `file:<abs-path>` — raw file content from host at launch time

## Operator Approval UI

When `require_approval` is `always` or `sensitive` and an operator is attached, the daemon renders a modal overlay before executing the command. The overlay blocks command execution until the operator responds.

Approval dialog shape (exact design spec for [TUI design decisions](/reference/tui-design-decisions/)):

```
┌─ Secure Execution Request ─────────────────────────────────────────┐
│  Agent: the-architect  │  Session: ws:my-service/#4                │
│                                                                      │
│  Command:   ssh sentry                                              │
│  Binding:   sentry-ssh  (SSH key, op://Personal/sentry-ssh)        │
│  Inject:    SSH private key → temp file, deleted after exec        │
│  Filter:    SSH fingerprints, PEM patterns                         │
│                                                                      │
│  [A] Approve  [D] Deny  [S] Approve All Similar  [W] Always       │
└──────────────────────────────────────────────────────────────────────┘
```

Approval modes and session state:

| Key | Mode | Behavior |
|---|---|---|
| A | Approve | Run this invocation only |
| D | Deny | Return `ExecDenied` to agent: "permission denied by operator" |
| S | Approve All Similar | Auto-approve all `jackin-exec ssh sentry` for this session |
| W | Always for Session | Auto-approve all `jackin-exec` calls for this session |

Session-level approval state is held in the daemon's `Multiplexer` struct — it resets on detach/reconnect (new attach clears session approvals by default, configurable).

When no client is attached and `require_approval` is set: behavior is configurable — `never` auto-approve, `sensitive` auto-deny, `always` auto-deny. Default: auto-deny with a `debug_log!("jackin-exec", "exec denied: no operator attached for approval")` line.

## Output Filtering

jackin-capsule scans every `jackin-exec` output stream before returning it to the agent:

- **Resolved credential values**: any string matching a credential value from ExecCredentials (raw, base64, hex) → `[redacted by jackin']`
- **PEM blocks**: `-----BEGIN.*KEY-----` … `-----END.*KEY-----` → `[key material redacted by jackin']`
- **AWS credential strings**: `AKIA[A-Z0-9]{16}` → `[aws-key redacted by jackin']`

The full unredacted output is written to the diagnostics run file under `JACKIN_DEBUG` runs (accessible only from the host).

**Limitation**: output filtering is defense-in-depth, not a complete boundary. The approval dialog is the primary defense for adversarial scenarios. Document this honestly in the session contract output.

## How the Agent Learns to Use `jackin-exec`

### Primary: system prompt injection

At container launch, the host CLI injects `JACKIN_EXEC_BINDINGS` as a container env var listing available binding names and their match patterns (not credential values):

```
JACKIN_EXEC_BINDINGS=sentry-ssh:ssh sentry*,ghcr-token:docker login ghcr.io*,github-token:gh *
```

The entrypoint script (<RepoFile path="docker/runtime/entrypoint.sh" />) appends a system prompt block using this env var so every agent runtime learns the available bindings.

### Secondary: MCP tool

jackin-capsule exposes a `jackin_exec` MCP tool inside the role container. The tool description lists available bindings. Claude Code and Codex can discover and use it directly.

### Not recommended: PATH shadowing

Transparent wrappers in PATH are not recommended — they hide the sandboxing from the agent, making the approval dialog confusing and making it harder to audit which calls went through `jackin-exec`.

- <RepoFile path="crates/jackin-capsule/src/daemon.rs" /> — dispatch `ExecCommand` in control handler; read exec-creds.json at startup; run command with credential injection; delete temp file

### Phase 2 — `op://` credential source

- Add `op://` resolution to `src/exec_creds.rs`: shell out to `op read "<op-ref>"` on host at launch
- Surface clear error if `op` CLI is not installed or auth fails
- Document: operator must have `op` CLI installed and authenticated before `jackin load`

### Phase 3 — TUI approval overlay + output filtering

- Add `pending_exec_approval: Option<PendingExecApproval>` to `Multiplexer` in <RepoFile path="crates/jackin-capsule/src/daemon.rs" />
- Render approval overlay modal in the daemon's render pipeline
- Route A/D/S/W keypresses to approval state when overlay is active
- Add output scanning: regex engine (use `regex` crate, already in workspace) for PEM blocks, AWS key patterns, resolved credential values
- Wire `redacted_count` into `ExecResult`
