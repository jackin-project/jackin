title: "jackin-exec: Secure Execution Wrapper"

**Status**: Open — design complete, implementation pending (can start on any backend, no prerequisites)

## What This Is

`jackin-exec` is a subcommand of `jackin-capsule`. The agent calls `jackin-exec ssh sentry`; under the hood this is `jackin-capsule exec ssh sentry`. jackin-capsule intercepts, injects the selected credentials at execution time, runs the command, filters the output for leaked values, and returns sanitized results. The raw credential values never appear in the agent's process environment.

`jackin-exec` is a symlink to `jackin-capsule` — no separate binary, no separate release artifact, no separate build job.

Works on top of any jackin' backend. Does not require the Apple Container backend to land first. See [Agent isolation architecture](/reference/roadmap/agent-isolation-architecture/) for the complete picture.

## The Problem

Current jackin' model injects all credentials as container env vars at `docker run` time. Every process inside the container — the agent, any subprocess — can call `printenv` and see raw API keys, SSH tokens, GitHub credentials.

The operator's goal: "I trust this agent to connect to my server, but I don't want it to have my raw SSH private key."

## Two Visibility Tiers for Workspace Env Vars

The `on_demand` flag is added directly to the existing `OpRef` struct. No new sections, no new syntax — the same `env` map, the same `OpRef` inline table form, one new optional field.

### Tier 1 — always-available (default, no change)

Existing behavior unchanged. Env var is injected into the container at `docker run` time. Agent sees it via `printenv`.

```toml
[workspaces.my-service.env]
PUBLIC_ENDPOINT = "https://api.example.com"
BUILD_ENV       = "${BUILD_ENV}"
API_KEY         = { op = "op://abc.../def.../fld...", path = "Work/Anthropic/api key" }
```

### Tier 2 — on-demand

Add `on_demand = true` to an existing `OpRef`. The env var is **not** injected at launch. It does not appear in the container's process environment. It is only materialized — ephemerally — when the operator selects it in the picker dialog triggered by `jackin-capsule exec`.

```toml
[workspaces.my-service.env]
PUBLIC_ENDPOINT = "https://api.example.com"
API_KEY         = { op = "op://abc.../def.../fld...", path = "Work/Anthropic/api key" }

# On-demand: not in container env, only via jackin-exec picker:
SSH_KEY  = { op = "op://abc.../def.../fld...", path = "Personal/Sentry SSH/key",  on_demand = true }
GH_TOKEN = { op = "op://abc.../def.../fld...", path = "Work/GitHub/token",         on_demand = true, suggest_for = "gh *" }
AWS_KEY  = { op = "op://abc.../def.../fld...", path = "Work/AWS/access-key",       on_demand = true, suggest_for = "aws *" }
```

`suggest_for` is a glob pattern. When the command matches, that env var is pre-selected in the picker. The operator can still change the selection before pressing Enter.

## Rust Change: Two Fields Added to `OpRef`

In <RepoFile path="src/operator_env.rs" />, `OpRef` gains two optional fields:

```rust
#[derive(Clone, Debug, PartialEq, Eq, serde::Serialize, serde::Deserialize)]
#[serde(deny_unknown_fields)]
pub struct OpRef {
    pub op: String,
    pub path: String,
    #[serde(default, skip_serializing_if = "Option::is_none")]
    pub account: Option<String>,

    // NEW:
    #[serde(default, skip_serializing_if = "std::ops::Not::not")]
    pub on_demand: bool,
    #[serde(default, skip_serializing_if = "Option::is_none")]
    pub suggest_for: Option<String>,
}
```

`#[serde(default)]` on both fields means: existing TOML files without these fields parse with `on_demand = false` and `suggest_for = None` — fully backward compatible.

This is a `WorkspaceConfig` schema change → `CURRENT_WORKSPACE_VERSION` bump from `v1alpha5` → `v1alpha6`.

## The Picker Dialog

When the agent calls `jackin-exec <command>`, jackin-capsule renders a picker overlay in the TUI. Only `on_demand = true` env vars appear. Vars with a matching `suggest_for` pattern are pre-selected.

```
┌─ Select credentials to attach ────────────────────────────────────┐
│  Command: gh pr create --title "fix: ..."                        │
│                                                                    │
│  [ ] SSH_KEY    Personal/Sentry SSH/key      (1Password)         │
│  [x] GH_TOKEN   Work/GitHub/token            (1Password)  ← suggested
│  [ ] AWS_KEY    Work/AWS/access-key          (1Password)         │
│                                                                    │
│  Space: toggle  ↑↓: navigate  Enter: execute  Esc: cancel       │
└────────────────────────────────────────────────────────────────────┘
```

`path` from `OpRef` is shown as the human-readable label. Never the UUID `op://` URI.

Operator presses Enter → capsule sends the selected credential refs to the host process → host runs `op read` for each selected item → 1Password Touch ID / biometric prompt appears on macOS → values flow back to capsule → injected as temp env vars for the duration of the command → command runs → output filtered for leaked values → temp values deleted.

## Host Process Callback (Phase 1 Design)

On-demand credentials are resolved at exec time, not at container launch. The capsule (inside the container) needs to call back to the host to invoke `op read`.

**Phase 1 mechanism**: the `jackin load` host process stays running for the lifetime of the session. It listens on a Unix socket at `~/.jackin/sockets/<container>/host.sock`, which is bind-mounted into the container at `/jackin/run/host.sock`. When the operator selects credentials in the picker, capsule connects to this socket, sends a credential request (list of `op://` URIs to resolve), the host process runs `op read` for each, and writes the resolved values back. Capsule injects them as temp env vars and runs the command.

**This is an ad-hoc Phase 1 solution.** The host process listening on a socket is not a proper daemon — it's the `jackin load` process staying alive. In a future iteration, this will migrate to the jackin' daemon ([Host bridge](/reference/roadmap/host-bridge/)). The daemon roadmap item explicitly tracks this migration: the daemon will handle all credential resolution requests from all running containers, and the ad-hoc per-process socket approach will be retired. See [jackin' daemon](/reference/roadmap/jackin-daemon/) for that migration path.

## Binary Design: Symlink + Subcommand

`jackin-exec` is a symlink to `/jackin/runtime/jackin-capsule` placed at `/usr/local/bin/jackin-exec`. Created via a `RUN ln -sf` step in <RepoFile path="src/derived_image.rs" />.

When `jackin-capsule` is invoked as `jackin-exec` (argv[0] check) or with `exec` as first argument, it routes to the exec subcommand handler. Same binary, same socket protocol, new subcommand.

```bash
# These are identical:
jackin-exec gh pr create
jackin-capsule exec gh pr create
```

**New subcommand in <RepoFile path="crates/jackin-capsule/src/main.rs" />:**

```rust
Some("exec") | None if invoked_as_jackin_exec(&args) => {
    exec::run(&args).await
}
```

The exec module: connects to `/jackin/run/jackin.sock` (control channel, first byte `0x00`), sends `ClientMsg::ExecCommand { command, args }`, waits for `ServerMsg::ExecResult`, writes output, exits with command exit code.

**New control protocol variants in <RepoFile path="crates/jackin-protocol/src/control.rs" />:**

```rust
// added to ClientMsg:
ExecCommand { command: String, args: Vec<String> }

// added to ServerMsg:
ExecResult { exit_code: i32, stdout: String, stderr: String, redacted_count: u32 }
ExecDenied { reason: String }
```

## Output Filtering

Capsule scans every `jackin-exec` output stream before returning it to the agent. All injected credential values (resolved from `op read`) are added to the redaction set:

- Exact value match (raw string) → `[redacted by jackin']`
- PEM blocks (`-----BEGIN.*KEY-----` … `-----END.*KEY-----`) → `[key material redacted by jackin']`

Full unredacted output logged to diagnostics run file under `JACKIN_DEBUG`. Document the limitation honestly: the approval picker + output filtering is defense-in-depth, not an absolute barrier.

## How the Agent Learns to Use `jackin-exec`

### System prompt injection

At container launch, `JACKIN_EXEC_BINDINGS` is injected as an always-available env var listing the names and `suggest_for` patterns of all on-demand vars (not their values):

```
JACKIN_EXEC_BINDINGS=SSH_KEY:ssh sentry*,GH_TOKEN:gh *,AWS_KEY:aws *
```

The entrypoint script (<RepoFile path="docker/runtime/entrypoint.sh" />) uses this to append a system prompt block for every agent runtime.

### MCP tool

jackin-capsule exposes a `jackin_exec` MCP tool. Claude Code and Codex can discover available on-demand bindings and use the tool directly.
