title: "jackin-exec: Secure Execution Wrapper"

**Status**: Open ‚Äî design complete, implementation pending (can start on any backend, no prerequisites)

## What This Is

`jackin-exec` is a subcommand of `jackin-capsule`. The agent calls `jackin-exec ssh sentry`; under the hood this is `jackin-capsule exec ssh sentry`. jackin-capsule intercepts, injects the required credential at execution time, runs the command, filters the output, and returns sanitized results. The raw credential value never appears in the agent's process environment.

`jackin-exec` is a symlink to `jackin-capsule` ‚Äî no separate binary, no separate release artifact, no separate build job.

This is a standalone layer that works on top of any jackin' backend ‚Äî Docker, Apple Container, or smolvm. It does not require the Apple Container backend to land first.

For the complete picture of how `jackin-exec` fits with the other security layers, see [Agent isolation architecture](/reference/roadmap/agent-isolation-architecture/).

## The Problem: Env Var Visibility

The current jackin' model injects all agent credentials as environment variables at container launch. Every process inside the container ‚Äî the agent, any subprocess, any tool the agent runs ‚Äî can call `printenv` and see raw API keys, SSH tokens, and GitHub credentials.

This design introduces the concept of **protected env vars**: credentials that are configured in the workspace but deliberately withheld from the container's process environment. They are only materialized during `jackin-capsule exec` invocations, for the duration of a single command, then deleted.

## Two Tiers of Workspace Env Vars

```
env.*          ‚Üí always_available = true  (current behavior)
secure_exec.*  ‚Üí protected, capsule-private only
```

### Tier 1: Always-available (`env.*`)

Injected into the container's process environment at `docker run` time (current behavior). All processes see them. Used for non-sensitive configuration: API endpoints, feature flags, tool paths, public identifiers.

```toml
[workspaces.my-service.env]
PUBLIC_API_ENDPOINT = "https://api.example.com"
REGION = "us-east-1"
```

### Tier 2: Protected (`secure_exec.*`)

**Not injected into the container process environment.** The agent cannot see these via `printenv`, `/proc/<PID>/environ`, or environment file reads. They are resolved by the host CLI at container launch and stored in a capsule-private file at `/jackin/run/exec-creds.json` (permissions `0600`, owned by root ‚Äî the agent user cannot read `/jackin/run/`). They are only materialized, ephemerally, when the agent calls `jackin-capsule exec` (or its alias `jackin-exec`), then immediately deleted.

```toml
[workspaces.my-service.secure_exec]
require_approval = "sensitive"  # always | sensitive | never

[workspaces.my-service.secure_exec.bindings]
sentry-ssh   = { match = "ssh sentry*",           credential = "op://Personal/sentry-ssh/private-key" }
ghcr-token   = { match = "docker login ghcr.io*", credential = "op://Work/ghcr/token" }
github-token = { match = "gh *",                  credential = "env:GH_TOKEN" }
aws-session  = { match = "aws *",                 credential_env = { AWS_ACCESS_KEY_ID = "op://Work/aws/key-id", AWS_SECRET_ACCESS_KEY = "op://Work/aws/secret" } }
```

Credential sources:
- `op://<vault>/<item>/<field>` ‚Äî 1Password, resolved via `op read` on host at launch
- `env:<VAR>` ‚Äî host environment variable, read at launch (not passed to container)
- `file:<abs-path>` ‚Äî raw file content from host, read at launch

The host CLI resolves these at container launch (when the host has access to `op`, host env, and host filesystem), writes resolved values to the capsule-private file. This means:
- The agent never sees the raw values in its environment
- The values are resolved once (not on every exec call), so `op` auth is only needed at launch
- Future improvement: truly lazy resolution via the jackin' daemon (see [Host bridge](/reference/roadmap/host-bridge/)), where `op read` is called on-demand at exec time rather than upfront

## Binary Design: `jackin-exec` = `jackin-capsule exec`

`jackin-exec` is **not a separate binary**. It is a symlink to `jackin-capsule` placed in PATH inside the role container. When `jackin-capsule` is invoked as `jackin-exec` (via symlink, argv[0] check) or with `exec` as the first argument, it routes to the exec subcommand handler.

```bash
# These three are identical:
jackin-exec ssh sentry
jackin-capsule exec ssh sentry
/jackin/runtime/jackin-capsule exec ssh sentry
```

The symlink is created in the derived image entrypoint setup or via a `RUN ln -sf /jackin/runtime/jackin-capsule /usr/local/bin/jackin-exec` step in <RepoFile path="src/derived_image.rs" />.

**What this eliminates:**
- No new binary artifact to build, cross-compile, or release
- No new `src/exec_binary.rs` acquisition module
- No new GitHub Actions job
- No separate `COPY` in the derived Dockerfile
- No new capsule version pinning for a second binary

**New subcommand in <RepoFile path="crates/jackin-capsule/src/main.rs" />:**

```rust
// In the non-PID-1 client dispatch:
Some("exec") | None if invoked_as_jackin_exec(&args) => {
    exec::run(&args).await
}
```

The exec module connects to `/jackin/run/jackin.sock`, sends `ClientMsg::ExecCommand`, waits for `ServerMsg::ExecResult`, writes output, exits with the command's exit code.

## Communication Protocol

`jackin-capsule exec` connects to the existing Unix socket at `/jackin/run/jackin.sock` using the **control channel** (first byte `0x00` = length-prefixed JSON). Two new variants added to <RepoFile path="crates/jackin-protocol/src/control.rs" />:

```rust
// added to ClientMsg:
ExecCommand { command: String, args: Vec<String> }

// added to ServerMsg:
ExecResult { exit_code: i32, stdout: String, stderr: String, redacted_count: u32 }
ExecDenied { reason: String }
```

The existing `#[serde(other)]` forward-compat sinks ensure older capsule versions return `Unknown` if they receive an `ExecCommand` from a newer client.

## Full Execution Flow

```
Host at container launch:
  ‚Üí reads workspace.secure_exec.bindings
  ‚Üí resolves each source (op read, host env var, host file)
  ‚Üí writes ~/.jackin/sockets/<container>/exec-creds.json (0600, root-owned)
  ‚Üí starts container (bind-mounts socket dir ‚Üí /jackin/run/)
  ‚Üí does NOT pass resolved values as -e flags to docker run

Capsule daemon startup:
  ‚Üí reads /jackin/run/exec-creds.json ‚Üí ExecCredentials held in memory
  ‚Üí agent user cannot read /jackin/run/ (0700, root-owned)

Agent calls: jackin-exec ssh sentry
  ‚Üí connects to /jackin/run/jackin.sock (control channel)
  ‚Üí sends ClientMsg::ExecCommand { command: "ssh", args: ["sentry"] }

Capsule daemon:
  1. Matches "ssh sentry" against ExecCredentials patterns
  2. If require_approval + operator attached: shows TUI approval overlay
  3. If approved:
       ‚Üí materializes credential to /jackin/run/secrets/<uuid> (0600 tmpfile)
       ‚Üí spawns: ssh -i /jackin/run/secrets/<uuid> sentry
       ‚Üí captures stdout + stderr
       ‚Üí scans output for secret patterns ‚Üí redacts
       ‚Üí deletes /jackin/run/secrets/<uuid>
       ‚Üí sends ExecResult { exit_code, stdout, stderr, redacted_count }
  4. If denied: sends ExecDenied { reason }

jackin-exec:
  ‚Üí writes ExecResult.stdout/stderr to its stdout/stderr
  ‚Üí exits with ExecResult.exit_code
```

| Key | Behavior |

## Output Filtering

Capsule scans every `jackin-exec` output stream before returning it to the agent:
- Resolved credential values (raw, base64, hex encoding) ‚Üí `[redacted by jackin']`
- PEM blocks (`-----BEGIN.*KEY-----` ‚Ä¶ `-----END.*KEY-----`) ‚Üí `[key material redacted by jackin']`
- AWS credential strings (`AKIA[A-Z0-9]{16}`) ‚Üí `[aws-key redacted by jackin']`

Full unredacted output logged to diagnostics run file under `JACKIN_DEBUG`. Output filtering is defense-in-depth ‚Äî the approval dialog is the primary defense for adversarial cases.

## How the Agent Learns to Use `jackin-exec`

### System prompt injection (primary, works with all agents)

The host CLI injects `JACKIN_EXEC_BINDINGS` as a container env var (always-available tier) listing binding names and patterns ‚Äî not credential values:

```
JACKIN_EXEC_BINDINGS=sentry-ssh:ssh sentry*,ghcr-token:docker login ghcr.io*,github-token:gh *
```

The entrypoint script (<RepoFile path="docker/runtime/entrypoint.sh" />) appends a system prompt block using this env var for every agent runtime.

### MCP tool (Claude Code / Codex)

jackin-capsule exposes a `jackin_exec` MCP tool. The tool description lists available bindings. Agents that support MCP can discover and use it directly.

### PATH (agent calls `jackin-exec` naturally)

`/usr/local/bin/jackin-exec` ‚Üí symlink to `/jackin/runtime/jackin-capsule`. The agent calls `jackin-exec ssh sentry` in bash ‚Äî no alias, no PATH trick needed beyond the symlink.

## Rust Struct Shapes

In <RepoFile path="src/workspace/mod.rs" />:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Default)]
pub struct SecureExecConfig {
    #[serde(default, skip_serializing_if = "is_default_approval")]
    pub require_approval: ApprovalMode,
    #[serde(default, skip_serializing_if = "BTreeMap::is_empty")]
    pub bindings: BTreeMap<String, CredentialBinding>,
}

#[derive(Debug, Clone, Serialize, Deserialize, Default)]
#[serde(rename_all = "snake_case")]
pub enum ApprovalMode {
    Always,
    #[default]
    Sensitive,
    Never,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct CredentialBinding {
    #[serde(rename = "match")]
    pub match_pattern: String,
    #[serde(flatten)]
    pub credential: CredentialSource,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum CredentialSource {
    Credential(String),                           // "op://...", "env:VAR", "file:/path"
    CredentialEnv(BTreeMap<String, String>),      // multiple env vars
}
```

Add to `WorkspaceConfig`:

```rust
#[serde(default, skip_serializing_if = "SecureExecConfig::is_empty")]
pub secure_exec: SecureExecConfig,
```

## Implementation Phases

### Phase 1 ‚Äî exec subcommand + symlink + env/file credentials

**Modified files only** (no new binary artifact):

- <RepoFile path="crates/jackin-capsule/src/main.rs" /> ‚Äî add `exec` subcommand dispatch; detect argv[0] == "jackin-exec" for symlink invocation
- Add `crates/jackin-capsule/src/exec.rs` ‚Äî exec client: connect to socket, send ExecCommand, receive ExecResult, write output, exit with code
- <RepoFile path="crates/jackin-protocol/src/control.rs" /> ‚Äî add `ExecCommand` to `ClientMsg`; `ExecResult` + `ExecDenied` to `ServerMsg`
- <RepoFile path="crates/jackin-capsule/src/daemon.rs" /> ‚Äî dispatch `ClientMsg::ExecCommand`; read exec-creds.json at daemon startup; run command with credential injection; delete temp file
- <RepoFile path="crates/jackin-capsule/src/socket.rs" /> ‚Äî route `ExecCommand` in the control channel dispatcher
- <RepoFile path="src/workspace/mod.rs" /> ‚Äî add `SecureExecConfig`, `ApprovalMode`, `CredentialBinding`, `CredentialSource`; add `secure_exec` field to `WorkspaceConfig`
- <RepoFile path="src/config/migrations.rs" /> ‚Äî bump `CURRENT_WORKSPACE_VERSION` to `v1alpha6`, add `noop_migration` step
- Add `src/exec_creds.rs` ‚Äî host-side: reads `secure_exec.bindings`, resolves `env:` and `file:` sources, writes `exec-creds.json`
- <RepoFile path="src/runtime/launch.rs" /> ‚Äî call `exec_creds::resolve_and_write()` at launch; inject `JACKIN_EXEC_BINDINGS` always-available env var
- <RepoFile path="src/derived_image.rs" /> ‚Äî add `RUN ln -sf /jackin/runtime/jackin-capsule /usr/local/bin/jackin-exec` to derived image
- <RepoFile path="docker/runtime/entrypoint.sh" /> ‚Äî use `JACKIN_EXEC_BINDINGS` to inject system prompt block
- Add `tests/fixtures/migrations/workspace/from-v1alpha5/` ‚Äî `meta.toml`, `before.toml`, `after.toml`

### Phase 2 ‚Äî `op://` credential source

- Add `op://` resolution to `src/exec_creds.rs`: shell out to `op read` on host at launch
- Surface clear error if `op` CLI is not installed or not authenticated
- Document: operator must have `op` CLI installed and authenticated before `jackin load`

### Phase 3 ‚Äî TUI approval overlay + output filtering

- Add `pending_exec_approval: Option<PendingExecApproval>` to `Multiplexer` in <RepoFile path="crates/jackin-capsule/src/daemon.rs" />
- Render approval overlay modal in the daemon render pipeline (new state in the render loop)
- Route A/D/S/W keypresses to approval state machine when overlay is active
- Add output filtering: `regex` crate (already in workspace) for PEM blocks, AWS key patterns, resolved credential value patterns
- Wire `redacted_count` into `ExecResult`

### Phase 4 ‚Äî MCP tool

- Expose `jackin_exec` as an MCP tool from jackin-capsule (uses the same exec subcommand handler)

### Future ‚Äî lazy credential resolution

When the jackin' daemon ([Host bridge](/reference/roadmap/host-bridge/)) is available: instead of resolving `op://` at launch and writing to `exec-creds.json`, the capsule can request resolution at exec time via a vsock/IPC call to the host daemon. The daemon runs `op read` on the host when the agent calls `jackin-exec`, eliminating the startup-time resolution entirely. This means: credentials are never pre-resolved, never stored anywhere, resolved only on-demand for each exec invocation.
