**Status**: Open — design proposal (Isolation & security track)

## Problem

jackin' has two credibility gaps that this roadmap item addresses together.

**Gap 1 — undocumented security implementation.** jackin' markets itself on isolation and security, but the security implementation is undocumented. Real security engineering — symlink rejection in <RepoFile path="crates/jackin-runtime/src/instance/auth.rs" />, `shellfirm` and `tirith` in the construct image, the YOLO flag contract in <RepoFile path="docker/runtime/entrypoint.sh" /> — is hidden in code and invisible to operators, role authors, and contributors who need to reason about the trust boundary.

**Gap 2 — unsigned capsule binary.** The `jackin-capsule` binary is downloaded into every operator's container at runtime (<RepoFile path="crates/jackin-image/src/capsule_binary.rs" />), verified by SHA256 only. There is no cryptographic signing, no SLSA provenance, and no SBOM. A compromised release artifact that matches the expected SHA256 would go undetected.

## Part 1 — Security threat model & implementation document

A contributor-facing internals doc at `docs/content/docs/reference/security-implementation.mdx` covering the following.

### 1. Threat model table

| Threat | Defended? | Where in code | Residual risk |
|---|---|---|---|
| Agent reads/writes files on host | Yes | Mount scoping in <RepoFile path="crates/jackin-config/src/mounts.rs" /> | Explicitly mounted rw paths |
| Credential theft via symlink | Yes | `reject_symlink` in <RepoFile path="crates/jackin-runtime/src/instance/auth.rs" /> | TOCTOU window between symlink check and file use |
| Container escape | Out of scope | Docker hardening tracked in [Docker runtime hardening contract](/reference/roadmap/docker-runtime-hardening-contract/) and [smolvm backend research](/reference/roadmap/smolvm-backend-research/) | Shared kernel; future microVM work addresses this |
| Agent exfiltration over network | Needs policy | Tracked in [Network egress policy](/reference/roadmap/network-egress-policy/) | Open egress by default today |
| Supply chain (capsule binary) | SHA256 only | <RepoFile path="crates/jackin-image/src/capsule_binary.rs" /> | No signing, no provenance — Part 2 below |

### 2. Implementation walk

One paragraph each for every security function:

- `reject_symlink` — rejects symlink paths when writing credential files to prevent agent-created symlinks from redirecting auth writes to hostile locations
- `repair_permissions` — fixes file permission drift on credential and state files that may have been loosened by tooling inside the container
- `write_private_file` — atomically writes a file with owner-only permissions, closing the window where a file briefly exists world-readable
- `set_file_perms` — sets explicit permissions on a file after creation, used when the file may have been created by a tool that applied a default umask
- `host_home_is_real` — verifies that the host home directory is a real directory (not a symlink or mount artifact) before trusting it for credential resolution
- `parse_gh_hosts_yml` — safely reads and validates `~/.config/gh/hosts.yml` without exposing the parser to malformed or hostile input

### 3. External tools rationale

- `shellfirm` — intercepts dangerous shell commands (`rm -rf /`, `chmod 777`, `curl | bash`, etc.) at the shell level before execution. Protects against both accidental operator mistakes and agent-generated destructive commands. Installed in <RepoFile path="docker/construct/Dockerfile" />.
- `tirith` — security scanner that validates container runtime state against a policy. Protects against configuration drift and unexpected privilege escalation during a session. Installed in <RepoFile path="docker/construct/Dockerfile" />.
- Both can be disabled by the operator via environment variables when they interfere with legitimate workflows.

### 4. Container boundary contract with YOLO flags

When `--dangerously-skip-permissions` (YOLO) flags are active:

- jackin' still enforces mount scoping, network isolation, and the host Docker socket exclusion
- jackin' does **not** guarantee that the agent cannot damage the workspace content on writable mounts
- jackin' does **not** guarantee that the agent cannot exfiltrate data over the network when egress is open
- The flag name (`dangerously-skip-permissions`) is the contract: the operator explicitly opted into reduced safety for agent productivity

## Part 2 — Signed releases + SLSA provenance + SBOM

### What to add to publish workflows

1. **SLSA provenance** — `actions/attest-build-provenance@v2` for every published binary, producing a signed provenance attestation that chains back to the GitHub Actions run
2. **Cosign signing** — `cosign sign-blob` with keyless OIDC signing for every release artifact, so operators can verify the binary came from the jackin' CI without pre-sharing a public key
3. **SBOM generation** — `syft scan` to produce a CycloneDX SBOM for each binary, uploaded as a release asset alongside the binary itself
4. **Operator verification docs** — document verification steps (`cosign verify-blob`, `slsa-verifier`) in a new page at `docs/content/docs/getting-started/verifying-releases.mdx`
5. **Runtime verifier upgrade** — upgrade the capsule binary download in <RepoFile path="crates/jackin-image/src/capsule_binary.rs" /> `ensure_available` to verify against a signed manifest instead of downloading and checking SHA256 alone

### Affected workflows

- <RepoFile path=".github/workflows/preview.yml" /> — preview/rolling release
- <RepoFile path=".github/workflows/release.yml" /> — tagged releases

## Non-goals

- Do not claim hostile multi-tenant safety from Docker hardening alone — that is the [Docker runtime hardening contract](/reference/roadmap/docker-runtime-hardening-contract/) scope.
- Do not replace future microVM work — [smolvm backend research](/reference/roadmap/smolvm-backend-research/) tracks that separately.
- Do not retroactively sign historical releases — provenance starts from the first workflow that includes signing.

## Implementation phases

### Phase 1 — Security implementation doc

Write the contributor-facing security implementation document at `docs/content/docs/reference/security-implementation.mdx`. Pure docs, no code changes. Estimated ~2-3 days.

### Phase 2 — SLSA provenance + cosign signing

Add `actions/attest-build-provenance@v2` and `cosign sign-blob` to the publish workflows. Both preview and tagged release pipelines produce signed artifacts. Estimated ~1 day.

### Phase 3 — SBOM generation

Add `syft scan` to produce CycloneDX SBOMs, uploaded as release assets alongside each binary. Estimated ~half day.

### Phase 4 — Operator verification docs

Write the verification guide at `docs/content/docs/getting-started/verifying-releases.mdx` with `cosign verify-blob` and `slsa-verifier` instructions. Estimated ~half day.

### Phase 5 — Capsule binary signed manifest

Upgrade `ensure_available` in <RepoFile path="crates/jackin-image/src/capsule_binary.rs" /> to verify against a signed manifest (signature over the SHA256 digest + version tag) instead of bare checksum comparison. The manifest is produced and signed in the same CI step that publishes the binary. Estimated ~1 day.
