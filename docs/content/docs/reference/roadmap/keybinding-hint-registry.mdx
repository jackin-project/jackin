## Problem

Across every jackin' TUI surface, the keys a screen *handles* and the keys its hint bar *advertises* are maintained as two independent lists. Nothing binds them together, so they drift. The result is a class of bug where a key does something the operator was never told about — or a hint names a key that does nothing, or names one key while a different key fires.

The reported instance: on the console workspaces screen the hint bar reads `↑↓ · ↵ launch   N new · S settings   Ctrl+Q quit`, yet pressing bare `Q` (or `q`, or `Esc`) exits jackin' immediately with **no confirmation** — a path that is both unadvertised and more destructive than the advertised `Ctrl+Q`, which only opens a confirm dialog. The handler is `KeyCode::Esc | KeyCode::Char('q' | 'Q') => Ok(InputOutcome::ExitJackin)` in <RepoFile path="crates/jackin/src/console/tui/input/list.rs" />, while `should_open_quit_confirm()` in <RepoFile path="crates/jackin-console/src/tui/run.rs" /> deliberately does not fire for bare `q` on the main list (because `is_on_main_screen` returns true there), so bare `q`/`Esc` bypass even the confirm that `Ctrl+Q` would open.

An audit of all surfaces (host console, launch cockpit, capsule, shared `jackin-tui`, plus the lookbook) found the same divergence pattern everywhere — dozens of live instances catalogued below.

## Why It Matters

Keybindings are a contract with the operator, and the hint bar is the only place that contract is published. When a key is handled but not advertised, the operator cannot discover it and can trigger it by accident (a reflexive `q` quits the whole app). When a hint advertises a key the widget does not handle, the operator presses it and nothing happens, eroding trust the way an inert `Ctrl+C` does. A correct architecture makes these states unrepresentable; today it permits all of them, and the count of live divergences is the evidence that manual discipline does not hold.

## Design

### Root-cause analysis

There is **no single source of truth binding a key → action → hint label.** Dispatch and advertisement are two disjoint code paths that share facts only by hand.

- `HintSpan::Key(&str)` in <RepoFile path="crates/jackin-tui/src/geometry.rs" /> is free-text presentation. The string `"Ctrl+Q"` has no connection to any `KeyCode`; the hint renderer only measures and styles strings.
- Every component's `handle_key` is a standalone `match key.code` with no reference to any hint list: `ConfirmState::handle_key` in <RepoFile path="crates/jackin-tui/src/components/confirm_dialog.rs" />, `TextInputState::handle_key` in <RepoFile path="crates/jackin-tui/src/components/text_input.rs" />, `SelectListState::handle_key` in <RepoFile path="crates/jackin-tui/src/components/select_list.rs" />, plus every per-surface dispatcher in <RepoFile path="crates/jackin/src/console/tui/input/list.rs" />, <RepoFile path="crates/jackin-launch/src/tui/subscriptions.rs" />, and <RepoFile path="crates/jackin-capsule/src/tui/input.rs" />.
- Hints are hand-written constants/builders in different files (often different crates) from the handler: <RepoFile path="crates/jackin-console/src/tui/components/footer_hints.rs" />, <RepoFile path="crates/jackin-launch/src/tui/components/prompts.rs" />, <RepoFile path="crates/jackin-capsule/src/tui/components/dialog/hint.rs" />.

No test asserts that a handler's key set matches its hint; the only hint tests check column-width math. So a key can be handled without a hint, a hint can name a key no handler acts on, and a hint can name a different key than the one handled.

The pattern is already proven achievable in this codebase — just not applied to keys. Mouse geometry is unified: `StatusBarPlan` in <RepoFile path="crates/jackin-capsule/src/tui/components/status_bar.rs" /> is the single source for both paint and hit-test of clickable regions, and `box_rect` in <RepoFile path="crates/jackin-capsule/src/tui/components/dialog.rs" /> is shared between render and click hit-testing. Scroll is near-unified: `ScrollAxes` feeds both `scroll_hint_spans(axes)` (advertise) and `handle_key_for_axes` (dispatch) in <RepoFile path="crates/jackin-tui/src/components/dialog_layout.rs" />, so advertised arrows and accepted arrows agree on availability. Keys have no analogue, and that absence is the enabling structure.

### The fix: one binding table per context, feeding both render and dispatch

Replace "free-text hint + independent match" with a binding entry that couples everything currently split. The hint bar renders the `Shown` entries — it can no longer advertise a key with no binding nor omit a `Shown` one. The dispatcher matches the incoming key against the table's chords and returns the action — a key absent from the table is, by construction, not handled, so no hidden behaviour is possible. Aliases are explicit rows (`HiddenAlias`), so `h/j/k/l` and lowercase `y`/`n` keep working but are registered, not silently bolted onto a match arm. `HintSpan::Key` stops being an author-supplied string and is derived from the binding's chord, so labels and handlers cannot disagree.

### Registry API spec (concrete)

A new module `crates/jackin-tui/src/keymap.rs` (self-named, per the crate's no-`mod.rs` rule), declared `pub mod keymap;` in <RepoFile path="crates/jackin-tui/src/lib.rs" /> and re-exported through <RepoFile path="crates/jackin-tui/src/components.rs" />. Tests in `keymap/tests.rs`. It builds on existing `HintSpan` (geometry.rs), `ModalOutcome` (lib.rs), and `ScrollAxes` (scroll.rs) — none replaced; the registry sits beside them and *produces* `HintSpan`, *consumes* `ScrollAxes` for axis-gated visibility.

**Neutral logical key** — both `crossterm::event::KeyEvent` (host/launch) and the capsule's raw-byte parser must construct and match the same type:

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub enum LogicalKey {
    Char(char),
    Enter, Esc, Tab, BackTab,
    Up, Down, Left, Right,
    Home, End, PageUp, PageDown,
    Backspace, Delete,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, Default)]
pub struct Mods { /* CTRL | ALT | SHIFT bit flags (u8 mask) */ }

#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct KeyChord { pub key: LogicalKey, pub mods: Mods }
```

**Binding + map:**

```rust
pub enum Visibility { Shown, HiddenAlias, Internal }

pub struct KeyBinding<A> {
    pub chords: &'static [KeyChord], // first chord drives the hint glyph; rest are aliases
    pub action: A,
    pub hint: Option<&'static str>,  // label after the glyph ("confirm", "cancel", …); None for Internal
    pub visibility: Visibility,
}

pub struct Keymap<A: 'static> { bindings: &'static [KeyBinding<A>] }

impl<A: Copy + 'static> Keymap<A> {
    pub fn dispatch(&self, chord: KeyChord) -> Option<&A>;      // first binding whose chords contain chord
    pub fn hint_spans(&self) -> Vec<HintSpan<'static>>;          // Shown bindings → Key+Text spans, GroupSep-joined
    pub fn hint_spans_for_axes(&self, axes: ScrollAxes) -> Vec<HintSpan<'static>>; // fold in scroll_hint_spans gating
}
```

`A` is a per-widget action enum mapped back to `ModalOutcome<T>` by the widget's thin dispatch shim, so the `ModalOutcome` contract is untouched. Case-folding (`y`/`Y`, `o`/`O`) uses the explicit-alias model: list both `Char('y')` and `Char('Y')` as chords so the `HiddenAlias` visibility stays representable.

**Glyph derivation** — `chord_glyph` must reproduce the exact glyphs already in use so output is byte-identical to today: `Ctrl+Q`; `↑↓` (`\u{2191}\u{2193}`); `↑↓←→`; `←→`; `↵` (`\u{21b5}`); `Esc`; `Y`; `⇥` (`\u{21e5}`). Grouped glyphs (`↑↓`, `↵/Esc`) map from a small binding-level lookup, not a single chord.

**Per-surface adaptation (incremental — no event-loop rewrite):** crossterm surfaces add `impl From<KeyEvent> for KeyChord`; a widget's `handle_key` body becomes `match keymap.dispatch(key.into()) { Some(a) => self.apply(a), None => /* text Char insert fallthrough */ }`. The capsule adds a byte→`KeyChord` parser; `handle_raw_key_for_axes` stays as a migration fallback. Migrate one widget at a time; `ConfirmState` is the pilot (worst divergence). Delete the hand-built footer functions only once every consumer reads `keymap.hint_spans()`.

**Enforcement (so the class cannot return):**

1. The binding table makes dispatch and hint the *same data* — divergence becomes unrepresentable for `Shown`/`HiddenAlias` bindings.
2. Per-context test: drive the real `handle_key` with every chord over the full `LogicalKey × Mods` space; any chord the handler *consumes* (not `ModalOutcome::Continue` and no focus change) must resolve via `keymap.dispatch(chord).is_some()`, and vice-versa. This is exactly the test that catches the ContainerInfo `↵ copy value` phantom (Enter is advertised but `handle_scroll_key` falls through).
3. CI lint/test forbidding `handle_key` match arms outside a binding table on migrated surfaces. The repo already proves this is enforceable for modals: `modal_footer_items` is an exhaustive `match` (compile-enforced) in the console footer code.

## Mandatory hint bar on every operational screen

An additional requirement was added after the audit: every operational TUI screen must carry a visible hint bar. Intro and outro animations are the only exceptions. Screens where an operator makes a selection, confirms an action, or navigates a list must always advertise the available keys — the hint bar is the only discovery surface.

An audit of all four surfaces (console, launch cockpit, capsule, lookbook) confirmed no screen was completely missing a hint bar call. The gaps found were incomplete hints — keys handled but not advertised in an existing hint bar. Six sites were fixed in this branch:

| Surface | File | Fix |
|---|---|---|
| Launch | `prompts.rs CONFIRM_HINT` | Added `↵ confirm` |
| Console | `run.rs CONFIRM_HINT` | Added `↵ confirm` |
| Capsule | `dialog/hint.rs CONFIRM_HINT` | Added `↵ select` and `⇥ toggle`; normalised `N/Esc no` |
| Capsule | `dialog/hint.rs main_view_hint` | Added `Ctrl+Q quit` to the non-scrollback branch |
| Lookbook | `main.rs SIDEBAR_HINT` | Added `Home/End first/last` |
| Lookbook | `main.rs PREVIEW_FOCUS_HINT` | Added `PgUp/PgDn page` |

Remaining hint gaps (handled but unadvertised) are in the divergence catalogue above and are addressed as part of Phase 2.
