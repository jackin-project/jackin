This roadmap item no longer repeats shipped runtime selector, state-path, or manifest details. Use the standard docs above for current behavior.

## Remaining Work

The project now has basic built-in runtime support. The remaining work is parity depth:

- runtime-native extension surfaces for Codex, Amp, Kimi, and OpenCode;
- richer Codex/Amp/Kimi/OpenCode config defaults and documented tuning;
- MCP / skill integration beyond simple config-file injection;
- runtime-specific auth and state behavior where Codex, Amp, Kimi, or OpenCode need something more than `sync`, `api_key`, and `ignore`;
- clearer version/update behavior for non-Claude runtimes.

With the [AgentRuntime registry](/reference/getting-oriented/architecture/#agentruntime-and-provider-registry) in place, adding a built-in runtime is a centralized adapter registration instead of scattered edits across `runtime/`, `config/`, `console/`, and the entrypoint. That registry is the structural prerequisite for most of the parity-depth items above.

## Non-Goals

- A third-party runtime marketplace.
- Automatic translation of Claude plugins into Codex config, Amp skills, or MCP servers.
- Pretending all runtimes should expose the same feature set.

## Open Questions

- Which Codex, Amp, Kimi, and OpenCode settings deserve first-class role-manifest fields rather than generated defaults?
- Should runtime-native skills/MCP wiring live in each runtime adapter or in a shared extension model?
- Do Codex, Amp, Kimi, and OpenCode need additional auth modes beyond `sync`, `api_key`, and `ignore`?
- How much runtime update detection should jackin own for non-Claude CLIs?
