---

The ordered, agent-executable checklist for the [post-restructure correctness & polish program](/reference/roadmap/post-restructure-fixes/).

**How to use this document.** Each `## Defect` is one self-contained fix. Before starting any defect, re-read the **Fix the whole class, never one instance** rule on the program page: every entry is a *class* of identical sites, and the fix must converge **all** of them on one shared pattern in a single pass — never patch only the screenshot the operator captured. Each entry states the root cause against the current tree, the canonical pattern to converge on, and a repo-wide scan command to enumerate sibling sites. `[ ]` not done · `[~]` partial · `[x]` done.

**Mandatory per-item analysis (hard rule).** This checklist is a starting point, not the full truth — it was written from screenshots and a first-pass scan and is necessarily incomplete. When you execute any entry, **first** re-scan the repo for that defect class, verify the listed sites and root cause, and discover the sites the checklist missed; **then** fix the whole superset. Each entry's site list is a hypothesis to confirm, not a closed set. If analysis reveals more sites, more root causes, or a better shared fix, expand the entry and fix all of it. Leaving a site unfixed because "it wasn't listed" is a miss.

**Validation gate for every defect:** `cargo fmt --check`, `cargo clippy --workspace --all-targets --all-features --locked -- -D warnings`, the test suite, and — for any lookbook visual change — regenerate SVGs (`cargo run -p jackin-tui-lookbook -- docs/public/tui-lookbook`) then verify no drift (`cargo run -p jackin-tui-lookbook -- --check docs/public/tui-lookbook`). Any cross-cutting TUI rule established goes into [TUI Design Decisions](/reference/tui/) in the same PR.

**Smoke-test recipe (operator-facing).** Per the repo's debug-flag rule, validate fixes with `cargo run --bin jackin -- console --debug` (host TUI: lists, dialogs, focus, status bar, hover) and, for capsule changes, build + point at the local capsule via `eval "$(cargo run --bin build-jackin-capsule -- --export)"` then `cargo run --bin jackin -- console --debug` and attach. The lookbook (`cargo run -p jackin-tui-lookbook -- --terminal`) is the fast visual surface for shared-component fixes. These are pure TUI/UX fixes: **no versioned-schema or serde changes** are expected — if a fix appears to need one, stop and reconsider.

## Execution phases & defect clusters

The 30+ defects are not independent — they converge on a few shared primitives and split into clusters that should be fixed together to avoid churn. Recommended order:

1. **Phase 0 — build the shared primitives** (see *Build-first shared primitives* below): `HoverTracker`, `DialogBodyLayout`, shared scrollable-dialog, `FocusOwner`, action-row component, modal input/lifecycle helper, and the capsule full-invalidate path. Everything else routes through these.
2. **Phase 1 — capsule render/lifecycle cluster** (Defects 26, 27, 28, 29, 30): one shared invalidate-and-full-repaint path + a capsule panic hook. These are crash/corruption bugs — highest severity — and the panic hook unblocks triage of the rest. Do first among the defects.
3. **Phase 2 — focus cluster** (9, 17, 18, 19, 20, 23): land the `FocusOwner` model, then every focus/`▸`/tab-content/picker defect derives from it.
4. **Phase 3 — dialog cluster** (5, 6, 7, 8, 9, 10, 11): land `DialogBodyLayout` + scrollable-dialog + modal-lifecycle, then route every dialog/picker through them.
5. **Phase 4 — input/affordance cluster** (11, 13, 22): single-consumer mouse routing + `HoverTracker` (cursor + hover color).
6. **Phase 5 — layout/styling cluster** (1, 2, 4, 14, 15, 16, 21, 24, 25, plus 31–35 below): scrollbars, padding, spacers, selection, action rows, status-bar reservation, palette, empty states.
7. **Phase 6 — surface parity + coverage** (3, 12, capsule parity audit, lookbook coverage, testing, design-decision updates) and the program acceptance gates.

Each phase ends green on `fmt`/`clippy`/tests; the capsule phases also re-run the capsule smoke build.

## Defect 1 — scrollable lists must show a scrollbar and accept the wheel everywhere

**Symptom:** the lookbook `stories` sidebar lists more components than fit the viewport but renders no vertical scrollbar, and the mouse wheel over the sidebar does nothing — only keyboard `↑↓` moves the (auto-following) selection. Long story ids are also hard-truncated (`status-footer/launch-progr`) with no affordance. The sidebar is the reference list widget; an operator reading it learns that lists in jackin' silently overflow.

**Root cause:** <RepoFile path="crates/jackin-tui-lookbook/src/main.rs" /> builds the sidebar with raw `ratatui::widgets::List` + `ListState` (the `List::new(items)` block around the `render_stateful_widget` call) instead of routing through the canonical scrollable pattern. The canonical pattern lives in <RepoFile path="crates/jackin-tui/src/components/scrollable_panel.rs" /> (`render_scrollable_block` / `cursor_follow_offset` / `render_vertical_scrollbar`) and <RepoFile path="crates/jackin-tui/src/components/select_list.rs" /> (`render_select_list`, which already draws a `full_cell_thumb` scrollbar and follows the cursor). The raw `List` path has none of that and the sidebar region is excluded from the wheel handler (only the preview panel forwards mouse events to the interactor).

**Canonical pattern to converge on:** every overflowing list renders a vertical scrollbar via `jackin_tui::scroll` (`is_scrollable` + `full_cell_thumb`) and follows the cursor via `cursor_follow_offset`, and every scrollable region accepts mouse-wheel `ScrollUp`/`ScrollDown` mapped by pointer position (per the wheel-routing rule in [TUI Navigation](/reference/tui/navigation/)).

**Tasks:**
- [ ] Scan repo-wide for raw list rendering that bypasses the canonical helper: `rg -l "ListState|widgets::List|List::new" crates/*/src --glob '!**/select_list.rs' --glob '!**/tests.rs'`. For each hit, determine whether it (a) renders a scrollbar when content overflows and (b) accepts the wheel. Enumerate every offender.
- [x] Fix the lookbook `stories` sidebar in `main.rs`: render the two-row-per-story list through a scrollbar-bearing helper (either `render_scrollable_block` with pre-built `Vec<Line>` + a tracked `sidebar_scroll` offset, or a 2-line-capable list helper added to `jackin-tui`). The selected story must stay visible via `cursor_follow_offset`; a thumb must appear when stories overflow the sidebar height.
- [x] Add mouse-wheel handling for the sidebar region in the `Event::Mouse` arm of `run_terminal`: `ScrollUp`/`ScrollDown` over `last_sidebar_inner_area` scrolls the sidebar offset, mirroring the existing preview-scroll affordance.
- [ ] Resolve the long-id truncation: either widen the sidebar to fit the longest id, wrap the id, or horizontally scroll the focused row — pick the option already used elsewhere in the repo for the same situation and apply it.
- [ ] If a 2-row scrollable list helper is the right shape and `jackin-tui` lacks it, add it to `jackin-tui` (not lookbook-local) and route the sidebar through it (lookbook public-API rule).
- [ ] Audit and fix every other offender found in the first task so no list in the workspace overflows without a scrollbar + wheel.

## Defect 2 — every preview component must sit on the same dark "card" background

**Symptom:** some lookbook previews (e.g. `ConfirmDialog`) read as a clearly-bounded widget on a dark backdrop, while others (`Panel`, `BrandHeader`) blend into the preview canvas so it is unclear where the component starts and ends. The dark background must be uniform across **all** previews.

**Root cause:** the preview canvas charcoal fill in <RepoFile path="crates/jackin-tui-lookbook/src/main.rs" /> (the `Color::Rgb(28, 28, 28)` block applied to `preview_inner`) is one layer, but individual components differ in whether they paint their own bounded surface: `modal_block`-based dialogs read as a card because the dialog occupies a centered sub-rect with visible canvas framing it, while a `Panel` story sized to fill the canvas has no framing and its transparent interior is indistinguishable from the backdrop. The treatment is per-story and inconsistent rather than a single shared rule.

**Canonical pattern to converge on:** one shared preview-canvas treatment so **every** story renders on an identical dark card with consistent framing, regardless of component type — the operator should never have to guess a component's bounds.

No per-story background special-casing.
- [ ] Verify against the full story list in <RepoFile path="crates/jackin-tui-lookbook/src/stories.rs" /> that no story renders with a background that differs from the shared treatment.

## Defect 4 — consistent padding on titled panels (title spacing + body inset) everywhere

**Symptom:** the lookbook `Panel` story renders `┌Workspace───┐` with the title flush against the corner and the body rows (`Name:`, `Role:`…) flush against the left border — "thick and ugly." The `about` block in the lookbook chrome has the same problem (content flush to top/left/right). Dialogs, by contrast, render `┌ Confirm ─┐` with a padded title and inset body. Padding must be uniform across every titled panel.

**Root cause:** two separate inconsistencies in the same class:
1. **Title spacing.** The repo convention is to pass titles pre-padded as `" Title "` (30+ call sites already do, e.g. `.title(" preview ")`, `.title(" Select Role ")`). A few sites pass bare titles — `story_panel_focused` in <RepoFile path="crates/jackin-tui-lookbook/src/stories.rs" /> uses `.title("Workspace")`, producing `┌Workspace` with no spaces. `Panel::title()` in <RepoFile path="crates/jackin-tui/src/components/panel.rs" /> does not normalize spacing, so the convention is unenforced and drifts.
2. **Body inset.** Panel bodies are rendered directly into `block.inner(area)` with no horizontal/top margin, so content touches the border. The preview canvas in `main.rs` applies `Margin { horizontal: 1, vertical: 1 }` but the `about` description block and the `Panel` story body do not.

**Canonical pattern to converge on:** every titled panel shows a space-padded title (`┌ Title ┐`) and insets its body content off the border with a consistent margin. Pick one enforcement point — ideally normalize in the shared `Panel` API so callers cannot get it wrong — and route every site through it.

**Tasks:**
- [x] Decide the enforcement point for **title spacing**: either (a) make `Panel::title()` normalize to `" {} "` internally and strip the now-redundant manual spaces from all `" X "` call sites, or (b) keep the caller convention and fix only the bare-title offenders. Whichever is chosen, apply it to **every** `.title(...)` site — scan with `rg "\.title\(" crates/*/src` and converge all of them.
- [x] Decide the enforcement point for **body inset**: define a shared inner-padding helper/margin (or a `Panel` method) and apply it to every titled-panel body so content never touches the border. Scan for panel bodies rendered into `block.inner(...)` without a margin and fix each.
- [x] Fix the lookbook `about` block and `Panel` story (`main.rs` + `stories.rs`) as part of the same pass — they are instances of this class, not special cases.
- [x] Record the title-spacing + body-inset rule in [TUI Design Decisions](/reference/tui/) so new panels inherit it.
- [x] Regenerate SVGs and confirm padded titles + inset bodies across every titled-panel preview.

## Defect 5 — dialogs need symmetric vertical padding (top spacer + bottom spacer) across all dialog types

**Symptom:** dialog internal vertical spacing is inconsistent and asymmetric across the family. `ConfirmDialog` (`confirm/default`) renders its prompt flush against the **top** border — no blank line above the first content row. `ErrorDialog` (`error/default`) is the mirror image: it has a top blank line but the `OK` button is flush against the **bottom** border (no trailing blank). `SaveDiscardDialog` (`save-discard/default`) over-corrects: it has the top spacer and the content→buttons spacer, but **two** blank lines below the buttons instead of one. `StatusPopup` (`status-popup/default`) has its `Please wait` status row flush against the bottom border (no trailing spacer) — it must move up one row. No two dialogs agree.

**Canonical dialog vertical layout (operator-specified, exact):**

```text
┌ Title ────────────┐
│                    │  ← 1 leading spacer row
│      content       │  ← content (1+ rows)
│                    │  ← 1 spacer row
│   Save  …  Cancel  │  ← action/button row
│                    │  ← exactly 1 trailing spacer row
└────────────────────┘
```

Every dialog must match this exactly: one leading spacer, content, one spacer, the action row, exactly one trailing spacer — no more, no less.

**Root cause:** each dialog hand-rolls its own vertical `Layout` constraints, and they disagree:
- <RepoFile path="crates/jackin-tui/src/components/confirm_dialog.rs" /> splits the inner area as `[prompt_lines, 1 (spacer), 1 (buttons)]` — **no leading spacer row**, so the prompt touches the top border.
- <RepoFile path="crates/jackin-tui/src/components/error_dialog.rs" /> splits as `[1 (top spacer), body, 1 (spacer), 1 (button)]` — the button is `chunks[3]`, the **last** inner row, so it touches the bottom border (no trailing spacer).

There is no shared dialog body-layout helper enforcing symmetric top/bottom insets, so each dialog drifts independently.

**Canonical pattern to converge on:** every modal dialog reserves a one-row blank spacer **both** above its first content row and below its action row, so content and buttons float with equal breathing room. Ideally express this once (a shared dialog body-layout helper or a Margin convention) and route every dialog through it.

**Tasks:**
- [x] Scan repo-wide for every modal dialog renderer: `rg -l "modal_block|render_confirm_dialog|render_error_dialog|render_save_discard|render_status_popup|Clear.render" crates/jackin-tui/src/components`. Enumerate `confirm_dialog.rs`, `error_dialog.rs`, `save_discard` dialog, `status_popup.rs`, and any other modal — confirm each one's top/bottom inset.
- [x] Define the canonical dialog vertical layout (leading spacer + content + spacer + action row + trailing spacer) once — a shared helper in `jackin-tui` if the dialogs can share it, otherwise the identical constraint shape applied to each.
- [x] Fix `ConfirmDialog`: add the leading spacer row so the prompt no longer touches the top border. (Confirmed offender: the `Git repository detected` confirm — content flush to top, bottom spacer present, top spacer missing — is the same `render_confirm_dialog` path.)
- [x] Fix `ErrorDialog`: add the trailing spacer row so the `OK` button sits one row above the bottom border.
- [x] Fix `SaveDiscardDialog`: collapse the **two** trailing spacer rows to exactly one.
- [x] Fix `StatusPopup`: add the trailing spacer so the `Please wait` row moves up one row off the bottom border (and confirm it has exactly one leading spacer + one content spacer like the rest).
- [x] Fix the `Mount destination` choice dialog: it has **neither** a leading nor a trailing spacer (content flush to top, buttons flush to bottom) — add both.
- [ ] Apply the same symmetric inset to every other dialog found in the scan — fix the whole class in one pass, not just the screenshotted dialogs.
- [x] Record the dialog symmetric-vertical-padding rule in [TUI Design Decisions](/reference/tui/dialogs/) so new dialogs inherit it.
- [x] Regenerate SVGs and confirm symmetric top/bottom spacing across every dialog preview.

## Defect 6 — one shared cross-screen debug-info dialog, full-screen on open, additive content per surface

**Symptom:** clicking the run-id chip must hide the **whole** screen and show **only** a debug-info dialog. That dialog must be **one shared dialog used across every surface** — console, capsule, launch, and any other screen — with the **same style and the same base content everywhere**. Content is **additive**, never subtractive: the base is Run ID + where to find the log + the run log contents; on surfaces that know more (capsule/launch know the container ID and what was just started), the dialog shows that **additional** information too — but it is never drawn differently or with less.

**Root cause:** today there is no single cross-screen debug dialog. The console builds a static two-row `ContainerInfoState` (Run ID + Run log *path*) via `debug_run_info_state` (<RepoFile path="crates/jackin-console/src/tui/components/container_info.rs" />) with no log contents and no scroll; the capsule has its own `container_info_dialog` (<RepoFile path="crates/jackin-capsule/src/tui/components/container_info_dialog.rs" />); the launch surface has yet another. They are separate implementations that drift in style and content. `ContainerInfoState` (<RepoFile path="crates/jackin-tui/src/components/container_info.rs" />) has no scroll offset and `render_container_info` draws no scrollbar.

**Canonical pattern to converge on:** **one** shared "Container info" / debug-info dialog component in `jackin-tui`, opened from the container-ID chip and/or the run-id chip on every surface. On open it paints the full-screen opaque backdrop (Defect 10) so only the dialog is visible. Same component, same style everywhere. Its content is additive along **two axes**:

1. **Surface axis** — the fields a surface can supply. Where a container exists (capsule/launch) the dialog shows: Container ID, Role, Agent, Workdir, `jackin` version, `jackin-capsule` version. The console (no container) omits the container-only rows.
2. **Debug axis** — `--debug` makes the dialog a richer **superset**: non-debug shows the simpler base; debug **adds** the diagnostics rows (Run ID, Run log path, run log contents, Docker build log location, and other debug pointers so the operator knows where to look). Debug never *removes* a field — it only adds.

Reference layout (capsule, debug):
```text
┌ Container info ──────────────────────────────┐
│                                              │  ← leading spacer (Defect 5)
│  Container ID  : jk-sk76zdat-thearchitect    │
│  Role          : the-architect               │
│  Agent         : claude                      │
│  Workdir       : /…/jackin                   │
│  jackin        : 0.6.0-dev                    │
│  jackin-capsule: 0.6.0-dev+444004b           │
│  Run ID        : jk-run-cc5ff2               │  ← debug-only rows
│  Run log       : ~/.jackin/…/jk-run-cc5ff2.jsonl │
│                                              │  ← trailing spacer
└──────────────────────────────────────────────┘
```

Same component, same style, same row formatting everywhere; surfaces and debug-mode only *add* rows. Adding a field later is a one-place change every surface inherits.

**Tasks:**
- [ ] Introduce (or promote) one shared "Container info" / debug-info dialog in `jackin-tui` with the two-axis additive content model (surface fields + debug-mode superset). Replace the console `debug_run_info_state` two-row state and the capsule's separate `container_info_dialog` with this one component (DRY — no per-surface dialog).
- [ ] Implement the debug gate: non-debug shows the simpler container rows; `--debug` adds the diagnostics rows (Run ID, Run log path + contents, Docker build log pointer, …) as a superset — never fewer rows in debug.
- [ ] On open, paint the full-screen backdrop (Defect 10) so the dialog supersedes the entire screen — nothing else visible.
- [ ] Base content on every surface: Run ID value, run log path, and the run log **contents** (read from the active run's `.jsonl`), in a scrollable body (Defect 8: `jackin_tui::scroll` + bottom-border `full_cell_thumb`, no scrollbar when it fits, `↑↓`/`J`/`K`/wheel).
- [ ] Additive content: on capsule and launch, include the container ID and the started-container details; design the model so new fields are added in one place and inherited everywhere, never removed per surface.
- [ ] Wire the run-id chip → this dialog on **every** surface that shows the chip (console, launch, capsule): `rg -n "status_footer_right_chip_rect|debug_bar_chip_area|last_debug_chip_area" crates/*/src`. In the capsule, **both** the container-ID chip and the (restored, Defect 12) red run-id chip open this same dialog.
- [ ] Record the cross-screen debug-info-dialog contract (shared component, full-screen on open, additive content, scrollable, same style everywhere) in [TUI Design Decisions](/reference/tui/chrome/).

## Defect 7 — dialogs keep a stable preferred size on resize; shrink only when the screen is too small

**Symptom:** resizing the terminal resizes the dialog **immediately and proportionally** — shrinking the screen shrinks the dialog and truncates its content (e.g. the Container info dialog truncates the Run log path to `/Users/donbeave/.jackin-pr-523/data` even though the full path would fit at the dialog's natural width). The dialog should hold a consistent preferred size and only begin shrinking once the terminal is genuinely too small to contain that preferred size.

**Root cause:** `centered_rect_fixed` in <RepoFile path="crates/jackin-console/src/tui/layout.rs" /> computes width as `outer.width * pct_w / 100` — a pure percentage of the terminal, so the dialog rescales continuously with every resize. Every modal routed through `modal_rect` / `modal_rect_for_mode` in <RepoFile path="crates/jackin-console/src/tui/components/modal_rects.rs" /> (ConfirmSave, ErrorPopup, ContainerInfo, AuthForm) inherits this, as do the launch-surface dialogs (<RepoFile path="crates/jackin-launch/src/tui/components/failure_dialog.rs" />, <RepoFile path="crates/jackin-launch/src/tui/components/container_info_dialog.rs" />, <RepoFile path="crates/jackin-launch/src/tui/components/prompts.rs" />) and any capsule dialog using the same idiom.

**Canonical pattern to converge on:** dialog width is a **content-derived preferred width, capped at a maximum**, taken as `preferred.min(outer.width - margin)` — constant while the terminal is at least that wide, shrinking only when the terminal is narrower. Same idea for height (already `rows.min(outer.height)` in places — confirm it does not over-shrink).

**Tasks:**
- [x] Scan repo-wide for percentage-based dialog sizing: `rg -n "centered_rect_fixed|outer.width \* |width \* pct|Percentage" crates/*/src`. Enumerate every modal that scales with the screen.
- [x] Replace the percentage width with a preferred-width-capped-at-max model in `centered_rect_fixed` (or a new shared helper): `width = preferred.min(outer.width.saturating_sub(margin))`, where `preferred` is content-derived (or a per-dialog constant) and stays stable until the terminal is narrower than it.
- [x] Route every dialog (`modal_rects.rs` modes, launch dialogs, capsule dialogs) through the stable-preferred-size helper so none rescales proportionally on resize. Fix the whole class.
- [ ] Verify the Container info / Run info dialog holds its width (no path truncation) until the terminal is genuinely too narrow, then degrades gracefully.
- [x] Record the modal-sizing rule (preferred size, shrink-only-when-too-small) in [TUI Design Decisions](/reference/tui/dialogs/).

## Defect 8 — every dialog is scrollable (horizontal + vertical) when content exceeds the dialog area

**Symptom:** once a dialog has shrunk to the screen limit (Defect 7) and its content still does not fit — too many rows, or a line wider than the dialog — the content is clipped/truncated with no way to reach the rest. This must never happen: **any** jackin' dialog, of any kind, must become scrollable in whichever axis overflows, following the same scroll approach used everywhere else in the TUI.

**Root cause:** dialogs render their bodies with fixed `Paragraph`/`Layout` widgets and no scroll state or scrollbar. There is no shared "dialog body scroll" path, so each dialog clips independently. The canonical scroll machinery already exists (`jackin_tui::scroll`, `render_scrollable_block`, `render_vertical_scrollbar` / `render_horizontal_scrollbar`, `full_cell_thumb`, `cursor_follow_offset`) but dialogs do not use it.

**Canonical pattern to converge on (universal dialog rule):** every dialog body routes through the shared scroll machinery. When content height exceeds the body viewport, a vertical scrollbar thumb appears on the right/bottom border and `↑↓`/`J`/`K`/`PageUp`/`PageDown`/wheel scroll it; when a line is wider than the body, a horizontal scrollbar appears and `←→` scroll it. When content fits in either axis, that axis shows no scrollbar. This is the same rule as `render_scrollable_block` — dialogs are not exempt.

**Tasks:**
- [ ] Enumerate every dialog in the workspace: `rg -ln "modal_block|render_confirm_dialog|render_error_dialog|render_save_discard|render_status_popup|render_container_info|failure_dialog|ConfirmSave|ErrorPopup|AuthForm|op_picker" crates/*/src`. Build the full dialog inventory (console + launch + capsule + jackin-tui shared).
- [ ] Define one shared dialog-body scroll path in `jackin-tui` (a helper or a state mixin) that wraps a dialog body in the canonical H/V scroll machinery — scrollbar-on-overflow, none-when-fits, standard key + wheel bindings — so every dialog inherits identical behavior.
- [ ] Route every dialog through it: Confirm, Error, SaveDiscard, StatusPopup, Container/Run info, AuthForm, op-picker, launch failure/prompt dialogs, capsule dialogs, and the **filter/selection pickers** (`Select Role`, `Unfinished jackin instances`, etc.). Confirmed offender: the `Unfinished jackin instances` picker has a row (`Recover other role with hardline 3bekkf7b status:failed_setup agent:claude role:the-architect upd…`) far wider than the dialog — it must scroll horizontally (and vertically when rows overflow) instead of clipping. Fix the whole class in one pass — no dialog left clipping — by reusing **one** shared scrollable-dialog component (DRY; never a second implementation).
- [ ] Compose with Defect 7: a dialog first holds its preferred size, shrinks only when the screen is too small, and **then** (if content still overflows the shrunk body) becomes scrollable. The two behaviors must work together, not fight.
- [ ] Compose with Defect 5: the symmetric vertical padding (leading/trailing spacer) is part of the body that scrolls — confirm the spacers stay correct under scroll.
- [x] Record the universal dialog-scroll rule in [TUI Design Decisions](/reference/tui/dialogs/): "every dialog scrolls H/V when its content exceeds the dialog area, using the shared scroll machinery; no scrollbar when content fits."

## Defect 26 — capsule resize is reactive: full erase + reflow, no stale or duplicated content

**Symptom:** resizing the terminal while attached to the capsule leaves the screen not properly refreshed — stale/duplicated content remains (e.g. the bottom PR status line renders twice). The mux does not reactively repaint the whole surface on resize.

**Root cause:** the SIGWINCH path sends a `Resize` frame (<RepoFile path="crates/jackin-capsule/src/tui/run.rs" />) → `mux.resize` (<RepoFile path="crates/jackin-capsule/src/daemon/pane_layout.rs" />), which calls `ratatui_terminal.clear()` whose `SocketBackend::clear()` **deliberately does not emit a screen erase** ("so the next frame is a full repaint without a blank flash"). That bet fails on shrink/reflow: cells from the previous, differently-sized frame (and on the attach client's outer terminal) are never erased, so old rows persist and overlap the new layout — the duplicated status line. The dirty-row renderer also assumes prior-buffer geometry that the resize invalidated.

**Canonical pattern to converge on:** a resize is a reactive full-repaint event — the mux marks the entire surface dirty and the renderer (and the attach client's outer terminal) performs a real clear + full reflow so no stale or duplicated cells survive. Resizing never shows doubled or leftover content; the next frame after a resize is authoritative for every cell.

**Tasks:**
- [ ] Reproduce and instrument the resize path with `cdebug!` (per the repo's debug-telemetry rule) to capture the frame geometry before/after resize and confirm where stale cells originate (daemon double-buffer vs attach-client terminal).
- [ ] On resize, force a full repaint: mark all rows/panes dirty and emit a real erase for the new geometry (reconcile `SocketBackend::clear()`'s no-erase optimization with correctness — erase on resize even if not on steady-state frames), so shrink/grow/reflow leaves no leftover cells.
- [ ] Ensure the attach client's **outer** terminal is also cleared/repainted on resize so duplicated chrome (the doubled PR/status line) cannot persist.
- [ ] Invalidate any cached geometry on resize: `pane_body_caches`, dirty-row state, `reconcile_content_rows`, status-bar/tab-strip caches.
- [ ] Compose with Defect 15 (status bar reserved) and Defect 7 (dialog sizing) so a resize re-lays-out chrome + modals within the new content area without overdraw.
- [ ] Add a resize regression test (shrink then grow) asserting the rendered buffer has no duplicated status/chrome rows.
- [ ] Record any capsule-specific resize/redraw rule in [TUI Design Decisions](/reference/tui/) and the capsule architecture reference.

## Defect 27 — rapid resizing must never crash the capsule (exit 101 panic)

**Symptom:** doing a lot of resizing quickly crashes the container: `container jk-…-thearchitect exited during session (exit 101) and produced no log output`. Exit code 101 is a Rust panic. A resize storm (many `SIGWINCH` in quick succession) panics the `jackin-capsule` process, and the crash produces **no diagnostics log** — so it is currently un-triageable. Resizing must never crash anything.

**Root cause (to confirm — requires runtime triage):** a SIGWINCH burst drives many back-to-back `mux.resize` / re-render cycles (<RepoFile path="crates/jackin-capsule/src/daemon/pane_layout.rs" />, <RepoFile path="crates/jackin-capsule/src/tui/run.rs" />). Candidate panic sources to check: unsaturated `u16` subtraction in render/layout geometry that underflows when dimensions shrink fast; out-of-bounds buffer/cell indexing when the ratatui/`SocketBackend` buffer and the new geometry disagree mid-resize (related to Defect 26's no-erase clear); a `RwLock`/`Mutex` poisoned by a panic in the PTY-reader or render task then re-locked elsewhere; or an `unwrap()`/`expect()` on a PTY/`bollard` resize call that errors under load. The "no log output" means the panic path bypasses the diagnostics writer and/or no panic hook routes the backtrace to the run log.

**Canonical pattern to converge on:** resize is panic-free and self-throttling. Geometry math saturates (no underflow), buffer access is bounds-checked against the *current* geometry, locks are not held across a panic-prone render, and a SIGWINCH burst is coalesced to the latest size rather than processed one-per-event.
