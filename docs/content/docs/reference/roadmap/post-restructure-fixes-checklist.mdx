The ordered, agent-executable checklist for the [post-restructure correctness & polish program](/reference/roadmap/post-restructure-fixes/).

**How to use this document.** Each `## Defect` is one self-contained fix. Before starting any defect, re-read the **Fix the whole class, never one instance** rule on the program page: every entry is a *class* of identical sites, and the fix must converge **all** of them on one shared pattern in a single pass вҖ” never patch only the screenshot the operator captured. Each entry states the root cause against the current tree, the canonical pattern to converge on, and a repo-wide scan command to enumerate sibling sites. `[ ]` not done В· `[~]` partial В· `[x]` done.

**Mandatory per-item analysis (hard rule).** This checklist is a starting point, not the full truth вҖ” it was written from screenshots and a first-pass scan and is necessarily incomplete. When you execute any entry, **first** re-scan the repo for that defect class, verify the listed sites and root cause, and discover the sites the checklist missed; **then** fix the whole superset. Each entry's site list is a hypothesis to confirm, not a closed set. If analysis reveals more sites, more root causes, or a better shared fix, expand the entry and fix all of it. Leaving a site unfixed because "it wasn't listed" is a miss.

**Validation gate for every defect:** `cargo fmt --check`, `cargo clippy --workspace --all-targets --all-features --locked -- -D warnings`, the test suite, and вҖ” for any lookbook visual change вҖ” regenerate SVGs (`cargo run -p jackin-tui-lookbook -- docs/public/tui-lookbook`) then verify no drift (`cargo run -p jackin-tui-lookbook -- --check docs/public/tui-lookbook`). Any cross-cutting TUI rule established goes into [TUI Design Decisions](/reference/tui/) in the same PR.

**Smoke-test recipe (operator-facing).** Per the repo's debug-flag rule, validate fixes with `cargo run --bin jackin -- console --debug` (host TUI: lists, dialogs, focus, status bar, hover) and, for capsule changes, build + point at the local capsule via `eval "$(cargo run --bin build-jackin-capsule -- --export)"` then `cargo run --bin jackin -- console --debug` and attach. The lookbook (`cargo run -p jackin-tui-lookbook -- --terminal`) is the fast visual surface for shared-component fixes. These are pure TUI/UX fixes: **no versioned-schema or serde changes** are expected вҖ” if a fix appears to need one, stop and reconsider.

## Execution phases & defect clusters

The 30+ defects are not independent вҖ” they converge on a few shared primitives and split into clusters that should be fixed together to avoid churn. Recommended order:

1. **Phase 0 вҖ” build the shared primitives** (see *Build-first shared primitives* below): `HoverTracker`, `DialogBodyLayout`, shared scrollable-dialog, `FocusOwner`, action-row component, modal input/lifecycle helper, and the capsule full-invalidate path. Everything else routes through these.
2. **Phase 1 вҖ” capsule render/lifecycle cluster** (Defects 26, 27, 28, 29, 30): one shared invalidate-and-full-repaint path + a capsule panic hook. These are crash/corruption bugs вҖ” highest severity вҖ” and the panic hook unblocks triage of the rest. Do first among the defects.
3. **Phase 2 вҖ” focus cluster** (9, 17, 18, 19, 20, 23): land the `FocusOwner` model, then every focus/`в–ё`/tab-content/picker defect derives from it.
4. **Phase 3 вҖ” dialog cluster** (5, 6, 7, 8, 9, 10, 11): land `DialogBodyLayout` + scrollable-dialog + modal-lifecycle, then route every dialog/picker through them.
5. **Phase 4 вҖ” input/affordance cluster** (11, 13, 22): single-consumer mouse routing + `HoverTracker` (cursor + hover color).
6. **Phase 5 вҖ” layout/styling cluster** (1, 2, 4, 14, 15, 16, 21, 24, 25, plus 31вҖ“35 below): scrollbars, padding, spacers, selection, action rows, status-bar reservation, palette, empty states.
7. **Phase 6 вҖ” surface parity + coverage** (3, 12, capsule parity audit, lookbook coverage, testing, design-decision updates) and the program acceptance gates.

Each phase ends green on `fmt`/`clippy`/tests; the capsule phases also re-run the capsule smoke build.

## Defect 1 вҖ” scrollable lists must show a scrollbar and accept the wheel everywhere

**Symptom:** the lookbook `stories` sidebar lists more components than fit the viewport but renders no vertical scrollbar, and the mouse wheel over the sidebar does nothing вҖ” only keyboard `вҶ‘вҶ“` moves the (auto-following) selection. Long story ids are also hard-truncated (`status-footer/launch-progr`) with no affordance. The sidebar is the reference list widget; an operator reading it learns that lists in jackin' silently overflow.

**Root cause:** <RepoFile path="crates/jackin-tui-lookbook/src/main.rs" /> builds the sidebar with raw `ratatui::widgets::List` + `ListState` (the `List::new(items)` block around the `render_stateful_widget` call) instead of routing through the canonical scrollable pattern. The canonical pattern lives in <RepoFile path="crates/jackin-tui/src/components/scrollable_panel.rs" /> (`render_scrollable_block` / `cursor_follow_offset` / `render_vertical_scrollbar`) and <RepoFile path="crates/jackin-tui/src/components/select_list.rs" /> (`render_select_list`, which already draws a `full_cell_thumb` scrollbar and follows the cursor). The raw `List` path has none of that and the sidebar region is excluded from the wheel handler (only the preview panel forwards mouse events to the interactor).

**Canonical pattern to converge on:** every overflowing list renders a vertical scrollbar via `jackin_tui::scroll` (`is_scrollable` + `full_cell_thumb`) and follows the cursor via `cursor_follow_offset`, and every scrollable region accepts mouse-wheel `ScrollUp`/`ScrollDown` mapped by pointer position (per the wheel-routing rule in [TUI Navigation](/reference/tui/navigation/)).

**Tasks:**
- [x] Scan repo-wide for raw list rendering that bypasses the canonical helper: `rg -l "ListState|widgets::List|List::new" crates/*/src --glob '!**/select_list.rs' --glob '!**/tests.rs'`. For each hit, determine whether it (a) renders a scrollbar when content overflows and (b) accepts the wheel. Enumerate every offender.
- [x] Fix the lookbook `stories` sidebar in `main.rs`: render the two-row-per-story list through a scrollbar-bearing helper (either `render_scrollable_block` with pre-built `Vec<Line>` + a tracked `sidebar_scroll` offset, or a 2-line-capable list helper added to `jackin-tui`). The selected story must stay visible via `cursor_follow_offset`; a thumb must appear when stories overflow the sidebar height.
- [x] Add mouse-wheel handling for the sidebar region in the `Event::Mouse` arm of `run_terminal`: `ScrollUp`/`ScrollDown` over `last_sidebar_inner_area` scrolls the sidebar offset, mirroring the existing preview-scroll affordance.

- [x] If a 2-row scrollable list helper is the right shape and `jackin-tui` lacks it, add it to `jackin-tui` (not lookbook-local) and route the sidebar through it (lookbook public-API rule).
- [x] Audit and fix every other offender found in the first task so no list in the workspace overflows without a scrollbar + wheel.

## Defect 4 вҖ” consistent padding on titled panels (title spacing + body inset) everywhere

**Symptom:** the lookbook `Panel` story renders `в”ҢWorkspaceв”Җв”Җв”Җв”җ` with the title flush against the corner and the body rows (`Name:`, `Role:`вҖҰ) flush against the left border вҖ” "thick and ugly." **Root cause:** two separate inconsistencies in the same class:

Pick one enforcement point вҖ” ideally normalize in the shared `Panel` API so callers cannot get it wrong вҖ” and route every site through it.

## Defect 5 вҖ” dialogs need symmetric vertical padding (top spacer + bottom spacer) across all dialog types

**Symptom:** dialog internal vertical spacing is inconsistent and asymmetric across the family. `ConfirmDialog` (`confirm/default`) renders its prompt flush against the **top** border вҖ” no blank line above the first content row. `ErrorDialog` (`error/default`) is the mirror image: it has a top blank line but the `OK` button is flush against the **bottom** border (no trailing blank). `SaveDiscardDialog` (`save-discard/default`) over-corrects: it has the top spacer and the contentвҶ’buttons spacer, but **two** blank lines below the buttons instead of one. `StatusPopup` (`status-popup/default`) has its `Please wait` status row flush against the bottom border (no trailing spacer) вҖ” it must move up one row. No two dialogs agree.

**Canonical dialog vertical layout (operator-specified, exact):**

```text
в”Ң Title в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ                    в”Ӯ  вҶҗ 1 leading spacer row
в”Ӯ      content       в”Ӯ  вҶҗ content (1+ rows)
в”Ӯ                    в”Ӯ  вҶҗ 1 spacer row
в”Ӯ   Save  вҖҰ  Cancel  в”Ӯ  вҶҗ action/button row
в”Ӯ                    в”Ӯ  вҶҗ exactly 1 trailing spacer row
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

Every dialog must match this exactly: one leading spacer, content, one spacer, the action row, exactly one trailing spacer вҖ” no more, no less.

**Root cause:** each dialog hand-rolls its own vertical `Layout` constraints, and they disagree:
- <RepoFile path="crates/jackin-tui/src/components/confirm_dialog.rs" /> splits the inner area as `[prompt_lines, 1 (spacer), 1 (buttons)]` вҖ” **no leading spacer row**, so the prompt touches the top border.
- <RepoFile path="crates/jackin-tui/src/components/error_dialog.rs" /> splits as `[1 (top spacer), body, 1 (spacer), 1 (button)]` вҖ” the button is `chunks[3]`, the **last** inner row, so it touches the bottom border (no trailing spacer).

There is no shared dialog body-layout helper enforcing symmetric top/bottom insets, so each dialog drifts independently.

**Canonical pattern to converge on:** every modal dialog reserves a one-row blank spacer **both** above its first content row and below its action row, so content and buttons float with equal breathing room. Ideally express this once (a shared dialog body-layout helper or a Margin convention) and route every dialog through it.

**Tasks:**
- [x] Scan repo-wide for every modal dialog renderer: `rg -l "modal_block|render_confirm_dialog|render_error_dialog|render_save_discard|render_status_popup|Clear.render" crates/jackin-tui/src/components`. Enumerate `confirm_dialog.rs`, `error_dialog.rs`, `save_discard` dialog, `status_popup.rs`, and any other modal вҖ” confirm each one's top/bottom inset.
- [x] Define the canonical dialog vertical layout (leading spacer + content + spacer + action row + trailing spacer) once вҖ” a shared helper in `jackin-tui` if the dialogs can share it, otherwise the identical constraint shape applied to each.
- [x] Fix `ConfirmDialog`: add the leading spacer row so the prompt no longer touches the top border. (Confirmed offender: the `Git repository detected` confirm вҖ” content flush to top, bottom spacer present, top spacer missing вҖ” is the same `render_confirm_dialog` path.)
- [x] Fix `ErrorDialog`: add the trailing spacer row so the `OK` button sits one row above the bottom border.
- [x] Fix `SaveDiscardDialog`: collapse the **two** trailing spacer rows to exactly one.
- [x] Fix `StatusPopup`: add the trailing spacer so the `Please wait` row moves up one row off the bottom border (and confirm it has exactly one leading spacer + one content spacer like the rest).
- [x] Fix the `Mount destination` choice dialog: it has **neither** a leading nor a trailing spacer (content flush to top, buttons flush to bottom) вҖ” add both.
- [x] Apply the same symmetric inset to every other dialog found in the scan вҖ” fix the whole class in one pass, not just the screenshotted dialogs.
- [x] Record the dialog symmetric-vertical-padding rule in [TUI Design Decisions](/reference/tui/dialogs/) so new dialogs inherit it.

## Defect 7 вҖ” dialogs keep a stable preferred size on resize; shrink only when the screen is too small

**Symptom:** resizing the terminal resizes the dialog **immediately and proportionally** вҖ” shrinking the screen shrinks the dialog and truncates its content (e.g. the Container info dialog truncates the Run log path to `/Users/donbeave/.jackin-pr-523/data` even though the full path would fit at the dialog's natural width). The dialog should hold a consistent preferred size and only begin shrinking once the terminal is genuinely too small to contain that preferred size.

**Root cause:** `centered_rect_fixed` in <RepoFile path="crates/jackin-console/src/tui/layout.rs" /> computes width as `outer.width * pct_w / 100` вҖ” a pure percentage of the terminal, so the dialog rescales continuously with every resize. Every modal routed through `modal_rect` / `modal_rect_for_mode` in <RepoFile path="crates/jackin-console/src/tui/components/modal_rects.rs" /> (ConfirmSave, ErrorPopup, ContainerInfo, AuthForm) inherits this, as do the launch-surface dialogs (<RepoFile path="crates/jackin-launch/src/tui/components/failure_dialog.rs" />, <RepoFile path="crates/jackin-launch/src/tui/components/container_info_dialog.rs" />, <RepoFile path="crates/jackin-launch/src/tui/components/prompts.rs" />) and any capsule dialog using the same idiom.

**Canonical pattern to converge on:** dialog width is a **content-derived preferred width, capped at a maximum**, taken as `preferred.min(outer.width - margin)` вҖ” constant while the terminal is at least that wide, shrinking only when the terminal is narrower. Same idea for height (already `rows.min(outer.height)` in places вҖ” confirm it does not over-shrink).

**Tasks:**
- [x] Scan repo-wide for percentage-based dialog sizing: `rg -n "centered_rect_fixed|outer.width \* |width \* pct|Percentage" crates/*/src`. Enumerate every modal that scales with the screen.
- [x] Replace the percentage width with a preferred-width-capped-at-max model in `centered_rect_fixed` (or a new shared helper): `width = preferred.min(outer.width.saturating_sub(margin))`, where `preferred` is content-derived (or a per-dialog constant) and stays stable until the terminal is narrower than it.
- [x] Route every dialog (`modal_rects.rs` modes, launch dialogs, capsule dialogs) through the stable-preferred-size helper so none rescales proportionally on resize. Fix the whole class.
- [x] Verify the Container info / Run info dialog holds its width (no path truncation) until the terminal is genuinely too narrow, then degrades gracefully.
- [x] Record the modal-sizing rule (preferred size, shrink-only-when-too-small) in [TUI Design Decisions](/reference/tui/dialogs/).

## Defect 26 вҖ” capsule resize is reactive: full erase + reflow, no stale or duplicated content

**Symptom:** resizing the terminal while attached to the capsule leaves the screen not properly refreshed вҖ” stale/duplicated content remains (e.g. the bottom PR status line renders twice). The mux does not reactively repaint the whole surface on resize.

**Root cause:** the SIGWINCH path sends a `Resize` frame (<RepoFile path="crates/jackin-capsule/src/tui/run.rs" />) вҶ’ `mux.resize` (<RepoFile path="crates/jackin-capsule/src/daemon/pane_layout.rs" />), which calls `ratatui_terminal.clear()` whose `SocketBackend::clear()` **deliberately does not emit a screen erase** ("so the next frame is a full repaint without a blank flash"). **Canonical pattern to converge on:** a resize is a reactive full-repaint event вҖ” the mux marks the entire surface dirty and the renderer (and the attach client's outer terminal) performs a real clear + full reflow so no stale or duplicated cells survive. Resizing never shows doubled or leftover content; the next frame after a resize is authoritative for every cell.

## Defect 43 вҖ” async-first: nothing on a render thread may block; parallelize independent work everywhere

**Symptom:** the launch cockpit's digital rain stutters/freezes and the screen feels unresponsive while provisioning runs, and provisioning stages feel serial. The operator's intent: "jackin should be designed around async as much as possible вҖ” blocking the rendering or a parallel operation slows performance and the visual representation. Refactor heavily; use async everywhere it makes sense."

**Root cause:** the host CLI runs on a **single-threaded** runtime вҖ” `#[tokio::main(flavor = "current_thread")]` in <RepoFile path="crates/jackin/src/main.rs" /> (the capsule daemon is the same, <RepoFile path="crates/jackin-capsule/src/main.rs" />). The launch cockpit's rain is rendered by a `tokio::spawn` task (`RichDriver::spawn` in <RepoFile path="crates/jackin-launch/src/tui/run.rs" />, ticking ~33 ms) that **shares that one OS thread** with the provisioning task. Cooperative scheduling means any **blocking** call on the provisioning task вҖ” a synchronous `std::process::Command`, a blocking `std::fs` read, or un-offloaded CPU work вҖ” prevents the runtime from polling the render task, so the rain freezes and input lags until the blocking call returns. The architecture is *already* partly async (`ShellRunner` wraps `tokio::process` in <RepoFile path="crates/jackin-docker/src/shell_runner.rs" />; hashing is offloaded via `spawn_blocking` in <RepoFile path="crates/jackin-image/src/agent_binary.rs" /> and <RepoFile path="crates/jackin-image/src/capsule_binary.rs" />; `tokio::join!`/`try_join!` parallelize in <RepoFile path="crates/jackin/src/runtime/cleanup.rs" />, <RepoFile path="crates/jackin/src/runtime/image.rs" />, <RepoFile path="crates/jackin/src/runtime/launch/launch_pipeline.rs" />) вҖ” so the work is to make that discipline **total**, not to introduce async from scratch.

**Two distinct problems, do not conflate them:**
1. **Thread-starving blockers** вҖ” blocking calls executed on the runtime thread while a render task wants to tick. These cause the visible rain freeze. Fix: route every subprocess through `ShellRunner` (async `tokio::process`) or wrap unavoidable sync work in `tokio::task::spawn_blocking`.
2. **Missed parallelism** вҖ” independent provisioning stages awaited sequentially (<RepoFile path="crates/jackin/src/runtime/launch.rs" /> has ~25 sequential `.await`s) where no data dependency forces an order. Fix: fan independent stages out with `try_join!` / `FuturesUnordered`.

**Canonical pattern to converge on:** no blocking call ever runs on a thread that also renders or handles input; independent async work runs concurrently; CPU-bound and unavoidably-sync work is offloaded to `spawn_blocking`. "Async everywhere it makes sense" вҖ” subprocess, network, fs-at-scale, and multi-step provisioning are async and concurrent by default; only genuinely-sequential data-dependent steps stay ordered.

**Tasks (analysis/plan only until operator says fix):**

*Inventory & classification*
- [ ] **Enumerate every blocking call reachable from an async context.** `rg -n "std::process::Command|\.output\(\)|\.status\(\)(?!\.await)|std::fs::(read|write|read_to_string|copy)|std::thread::sleep" crates/ src/ --glob '!**/tests*'`.
