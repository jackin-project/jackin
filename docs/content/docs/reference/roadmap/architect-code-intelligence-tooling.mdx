**Status**: Open — ready to implement (pilot: the-architect role only, opt-in, no jackin-core code change)

## Problem

Agents waste tokens and wall-clock re-discovering code by ad-hoc text search and by guessing at definitions instead of resolving them. A reasonable proposal to fix this for Rust roles is a four-tool stack: a fast file searcher, a structural code searcher (`ast-grep`), the official LSP (`rust-analyzer`), and the cargo verifiers (`nextest`, `clippy`, `fmt`). The tool *choices* are the current consensus stack. The risk is in the *assembly*: wiring every tool as a Model Context Protocol (MCP) server injects tool-schema tokens into every agent turn (a five-server setup is ~55K tokens before the first message), and a single repo-level `.mcp.json` only configures `Claude Code`, not the four other runtimes the-architect supports. For a multi-agent orchestrator, CLI tools on `PATH` are the uniform, zero-permanent-cost surface; MCP earns its schema cost only where a warm long-lived process (like a resident file index) can't be had from a cold CLI spawn.

This item pilots the stack on **one role — the-architect** — so the approach can be measured before any broader rollout, and so nothing changes for operators who do not load the-architect.

## Scope and principles

- **One role only.** All changes land in the external [`jackin-project/jackin-the-architect`](https://github.com/jackin-project/jackin-the-architect) repository. No other role is touched.
- **Opt-in by construction.** A tool only exists in a container the operator explicitly loads. Operators who never `jackin load the-architect` see no change.
- **No jackin-core change.** The pilot rides plumbing that already exists: role `Dockerfile` tool installs, role `hooks` for MCP registration, and default-home seeding of agent memory files. Nothing in <RepoFile path="src/derived_image.rs" />, the manifest schema, or the runtime needs to change. This keeps the experiment fully decoupled and reversible — reverting the the-architect commit removes every trace.
- **Reversible and measurable.** The goal is to evaluate whether the stack makes the-architect measurably more productive (fewer tokens per code lookup, fewer wrong-path edits) before deciding whether to generalize it to a shared base or a manifest feature.

## Corrections to the original proposals

Two external write-ups motivated this item. Both contained errors that an implementer must not copy forward:

| Claim in the source material | Correct form | Why it matters |
|---|---|---|
| `cargo install ast-grep-cli` | `cargo install ast-grep --locked` (or a prebuilt binary) | `ast-grep-cli` is not the crate; the build fails. |
| `cargo install ast-grep-mcp-server` | `cargo install ast-grep-mcp --locked` | `ast-grep-mcp` is the crate; `ast-grep-mcp-server` is the binary it produces. |
| Invoke the structural tool as `sg` | Invoke as `ast-grep` | `sg` is the system set-group command (shadow-utils, Debian `login` package). The construct is Debian Trixie, so `sg` is taken. Upstream recommends the full `ast-grep` name on Linux. |
| "avoid `rg`/`grep` unless explicitly plain-text" | Prefer `ast-grep` **for code structure**; keep `rg`/built-in grep for plain text | Most agent searches are plain text (string literals, messages, paths, config keys). Forcing structural search on those slows the agent and produces awkward patterns. |

## The tool stack and how each is exposed

| Capability | Tool | Exposure | Rationale |
|---|---|---|---|
| Structural Rust search / refactor / lint | `ast-grep` | **CLI** on `PATH` | Agents call `ast-grep -l rust -p '…'` in the shell. Zero permanent schema cost; works for every runtime the role supports. |
| Definitions / references / types / impls | `rust-analyzer` | **LSP** (rustup component on `PATH`) | `Claude Code`'s native LSP integration drives it directly — no MCP bridge needed. Other runtimes fall back to text + ast-grep. |
| Test / coverage / lint / format | `cargo-nextest`, `cargo-llvm-cov`, `clippy`, `rustfmt` | **CLI** | Already present in the-architect (the Rust dev role). Verification stays CLI-native. |
| Fast file / text search at scale | `fff` (`fff-mcp`) | **MCP** (role-scoped, user-scope registration) | The one tool whose value is a warm resident index across many searches in one session — exactly what a cold CLI can't provide, so MCP is the right interface. Included in the pilot to evaluate it on the-architect even though the repo is medium-sized (the dramatic win is monorepo-scale; the pilot will tell us whether it helps here). |

The honest trade-off on `fff`: it carries a permanent MCP schema cost in every turn, and on a ~91K-line repo its index warmth saves less than on a 500K-file monorepo. The pilot includes it deliberately so we measure rather than assume.

## Implementation (in the-architect repo)

### 1. Install the tools (`Dockerfile`)

The-architect already builds on the construct and already carries the Rust toolchain plus `nextest`/`clippy`/`rustfmt`. Add the missing pieces. Prefer prebuilt binaries so image builds stay fast:

```dockerfile
# --- Code-intelligence tooling (pilot — see roadmap) ---

# rust-analyzer: official LSP as a rustup component (no compile)
RUN rustup component add rust-analyzer

# ast-grep: structural search/refactor as a CLI. Prebuilt binary via mise's
# aqua backend (layer-cached, no source compile). Invoke as `ast-grep` — never
# `sg`, which is the system set-group command on Debian.
RUN mise use --global "aqua:ast-grep/ast-grep"

# fff: resident file-search MCP server. Upstream-blessed installer (the project
# ships no Debian package); pins land via the installer's release resolution.
RUN curl -fsSL https://dmtrkovalenko.dev/install-mcp.sh | bash
```

A `cargo install ast-grep --locked` is an acceptable fallback for `ast-grep` since the toolchain is present, but it compiles from source and slows the build; prefer the prebuilt path.

### 2. Register `fff-MCP` for Claude (role hook)

Role hooks already register MCP servers (the `preflight` example in the [role manifest reference](/developing/role-manifest/) runs an MCP setup). Register `fff` at **user scope** so it writes to the container's `~/.claude.json`, never to a project `.mcp.json` inside the operator's mounted repo. Use `setup_once` so it runs once per instance, after auth handoff:

```bash title="hooks/setup-once.sh"
#!/bin/bash
set -euo pipefail

# Register the fff file-search MCP server for Claude at user scope.
# User scope keeps this out of the operator's mounted repo.
if command -v fff-mcp >/dev/null 2>&1; then
  claude mcp list 2>/dev/null | grep -q '^fff' || claude mcp add -s user fff -- fff-mcp
fi
```

Wire it in the manifest if a `setup_once` hook is not already declared:

```toml title="jackin.role.toml"
[hooks]
setup_once = "hooks/setup-once.sh"
```

No manifest *schema* change is required — `[hooks]` already exists. If the-architect already has a `setup_once` hook, append the block instead of adding a second hook.

### 3. Tell the agent the tools exist (baked memory file)

Installing tools is not enough — the model must be told they are available and when to prefer them. Bake a guidance file into the agent's home memory in the `Dockerfile`. For Claude this is `~/.claude/CLAUDE.md`:

```dockerfile
COPY --chown=agent:agent code-intelligence.md /home/agent/.claude/CLAUDE.md
```

```markdown title="code-intelligence.md"
# Code-intelligence tooling (the-architect)

This container ships dedicated code-intelligence tools. Prefer them over ad-hoc text search when the query is about code structure.

## Structural search & refactor — ast-grep
`ast-grep` is installed. Invoke it as `ast-grep`, never `sg` (`sg` is the system set-group command on Debian).

Reach for ast-grep first whenever the target is a *syntax structure*, not literal text: functions, `impl` blocks, trait impls, macro definitions/calls, `async fn`, derives, match arms, or specific call shapes (`$X.unwrap()`, `$X.clone()`).

- Search: `ast-grep --lang rust --pattern '<pattern>'` (short: `ast-grep -l rust -p '<pattern>'`).
- Rewrite: add `--rewrite '<replacement>'`.
- Lint/scan: `ast-grep scan` against rules in `sgconfig.yml`.
- Metavariables: `$X` matches one node, `$$$ARGS` matches a list. Every `.unwrap()`: `ast-grep -l rust -p '$X.unwrap()'`.

Use ripgrep (`rg`) or built-in grep for *plain text* — string literals, log/error messages, file paths, config keys, comments. That is the common case; do not force ast-grep on it.

## Semantic navigation — rust-analyzer
Definitions, references, types, and implementations come from rust-analyzer (on `PATH`). Resolve symbols with go-to-definition / find-references / hover rather than guessing from text matches.

## Verify
- Tests: `cargo nextest run`
- Lints: `cargo clippy --all-targets --all-features -- -D warnings`
- Format: `cargo fmt --check`
```

For the other runtimes the-architect supports, bake the same body into each one's home memory file. The default-home capture in <RepoFile path="src/derived_image.rs" /> copies `~/.claude/`, `~/.codex/`, and the other agent homes, so a role-baked file in any of them survives into the live container:

| Runtime | Memory file to bake |
|---|---|
| `claude` | `~/.claude/CLAUDE.md` |
| `codex` | `~/.codex/AGENTS.md` |
| `amp` / `kimi` / `opencode` | the runtime's user-level instructions file |

For the pilot, Claude is the primary target; baking the Codex file too is cheap and recommended.

## How the model learns about the tools

This is the load-bearing question, and it is already answered by existing plumbing — no jackin-core change:

1. The the-architect `Dockerfile` writes `code-intelligence.md` to `/home/agent/.claude/CLAUDE.md` in the **agent layer**, on top of the construct.
2. The jackin-generated **derived layer** runs after the agent layer. Its default-home capture step (`cp -a /home/agent/.claude/. /jackin/default-home/.claude/`, see <RepoFile path="src/derived_image.rs" />) snapshots that file into `/jackin/default-home/`.
3. On **first boot**, the runtime seeds `/jackin/default-home/` into `/home/agent/`, so the file lands in the live agent home.
4. `Claude Code` reads `~/.claude/CLAUDE.md` as **user-level memory** at startup — so the guidance is in context for every session, regardless of which repo the operator mounted.

The same chain delivers `~/.codex/AGENTS.md` to Codex. Because the guidance lives in the agent's *home*, not in the operator's repo, it applies to every workspace the-architect runs against and never mutates a host repository.

## Why no jackin-core change

Every piece rides an existing extension point:

- **Tool installs** → the role `Dockerfile` (agent layer). The construct's "prefer apt over `cargo install`" rule is construct-scoped; role Dockerfiles use `mise` and prebuilt binaries by convention (see [Creating a Role](/developing/creating-roles/)).
- **MCP registration** → a role `setup_once` hook running `claude mcp add -s user` (the same pattern the [role manifest reference](/developing/role-manifest/) documents for other MCP servers).
- **Agent guidance** → a baked home memory file carried by the existing default-home seeding in <RepoFile path="src/derived_image.rs" />.

Keeping the pilot out of jackin core is the point: it can be trialed, measured, and reverted entirely within one role repo.

## Verification

Smoke-test from a jackin checkout (always `--debug`, and share the printed run id if something misbehaves):

```bash
cargo run --bin jackin -- load the-architect . --debug
```

Inside the container, confirm each surface:

- `ast-grep -l rust -p '$X.unwrap()'` returns structural matches.
- `command -v rust-analyzer` resolves; `Claude Code` go-to-definition works on a symbol.
- `claude mcp list` shows `fff`; ask the agent to "use fff" for a file search.
- `cargo nextest run` runs the suite.
- The agent, when asked to find all trait impls or `.unwrap()` calls, reaches for `ast-grep` rather than `rg` — confirming the baked guidance took effect.

## Evaluation criteria

The pilot decides whether the stack generalizes. Track, over a handful of representative the-architect sessions:

- **Token cost per code lookup** — does ast-grep/LSP reduce tokens versus repeated `rg` + file reads, net of the `fff` MCP schema overhead carried every turn?
- **Tool adoption** — does the agent actually invoke ast-grep/LSP/fff when appropriate, or ignore the guidance? (If ignored, the guidance phrasing or a skill-based delivery is the lever, not more tools.)
- **Correctness** — fewer wrong-path edits from resolving symbols instead of guessing.
- **`fff` specifically** — does the warm index measurably beat the construct's ripgrep on this repo size, or is its value only monorepo-scale? This is the one tool whose inclusion is a genuine open question.

A positive result motivates the follow-ups; a negative `fff` result drops it from any general rollout while keeping the CLI + LSP tools.

## Optional: ast-grep rules as a lint gate

The structural rules from the original proposal (flag `todo!()`/`unimplemented!()`, audit `.unwrap()`) are better as a CI/pre-commit lint than as agent prose. If adopted, add `sgconfig.yml` plus `.ast-grep/rules/*.yml` to the-architect (or the linted repo) and run `ast-grep scan` in CI. This is independent of the agent-guidance pilot and can land separately.

## Follow-ups (deferred — not part of the pilot)

- **Declarative MCP servers in the manifest.** A first-class `[claude].mcp_servers` (and per-runtime equivalents) field in `jackin.role.toml` would let roles register MCP servers without a hook. This is a versioned-schema change (one `CURRENT_MANIFEST_VERSION` bump with migration + fixtures, per the schema-evolution rules) and should only be designed if the pilot proves MCP servers are worth generalizing.
- **Skill-based guidance.** Delivering tool guidance as a `Claude Code` skill (loaded on demand) instead of always-on `CLAUDE.md` would cut the standing memory cost and is more discoverable; the-architect already installs plugins via a marketplace, so a `code-intelligence` skill is a natural next step if the pilot succeeds.
- **Generalize to a shared layer.** If the stack proves out, move the CLI tools (ast-grep, rust-analyzer) into a shared base so other Rust roles inherit them, keeping `fff`/MCP opt-in.
