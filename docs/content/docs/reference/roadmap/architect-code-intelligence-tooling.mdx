**Status**: Open — ready to implement (pilot: the-architect role only, opt-in, no jackin-core code change)

## Objective

Equip the the-architect role with a Rust code-intelligence stack and make the agent use it: structural search and refactor (`ast-grep`), semantic navigation (`rust-analyzer`), the cargo verifiers (`nextest`, `clippy`, `rustfmt`), and a resident file-search server (`fff`). Deliver everything inside the external [`jackin-project/jackin-the-architect`](https://github.com/jackin-project/jackin-the-architect) repository, opt-in and role-scoped, with no change to jackin core.

## Scope

- **the-architect only.** No other role is touched.
- **Opt-in.** The tools exist only inside a container the operator loads; operators who never `jackin load the-architect` see no change.
- **No jackin-core change.** The pilot rides existing extension points: the role `Dockerfile`, a role `setup_once` hook, and default-home seeding of the agent memory file (<RepoFile path="src/derived_image.rs" />).
- **Reversible.** Reverting the the-architect commit removes every trace.

## Tools and exposure

| Capability | Tool | Exposure |
|---|---|---|
| Structural Rust search / refactor / lint | `ast-grep` | CLI on `PATH` (invoke as `ast-grep`, never `sg`) |
| Definitions / references / types / impls | `rust-analyzer` | LSP (rustup component on `PATH`) |
| Test / coverage / lint / format | `cargo-nextest`, `clippy`, `rustfmt` | CLI (already present in the-architect) |
| Resident file / text search | `fff` (`fff-mcp`) | MCP, registered at Claude user scope |

CLI tools sit on `PATH` so every runtime the role supports reaches them with no per-turn schema cost. `fff` is exposed over MCP because its value is a warm resident index that a cold CLI spawn cannot provide.

## Implementation (the-architect repo)

### 1. Install the tools (`Dockerfile`)

The-architect already carries the Rust toolchain plus `nextest`/`clippy`/`rustfmt`. Add the missing pieces, preferring prebuilt binaries so image builds stay fast:

```dockerfile
# --- Code-intelligence tooling ---

# rust-analyzer: official LSP as a rustup component (no compile)
RUN rustup component add rust-analyzer

# ast-grep: structural search/refactor as a CLI. Prebuilt binary via mise's
# aqua backend (layer-cached, no source compile). Invoke as `ast-grep` — never
# `sg`, which is the system set-group command on Debian.
RUN mise use --global "aqua:ast-grep/ast-grep"

# fff: resident file-search MCP server, installed via the upstream installer
# (the project ships no Debian package).
RUN curl -fsSL https://dmtrkovalenko.dev/install-mcp.sh | bash
```

`cargo install ast-grep --locked` is an acceptable fallback for `ast-grep` since the toolchain is present, but it compiles from source; prefer the prebuilt path.

### 2. Register fff-MCP (role hook)

Register `fff` at **user scope** so it writes to the container's `~/.claude.json`, never to a project `.mcp.json` inside the operator's mounted repo. Use a `setup_once` hook so it runs once per instance, after auth handoff:

```bash title="hooks/setup-once.sh"
#!/bin/bash
set -euo pipefail

# Register the fff file-search MCP server for Claude at user scope.
if command -v fff-mcp >/dev/null 2>&1; then
  claude mcp list 2>/dev/null | grep -q '^fff' || claude mcp add -s user fff -- fff-mcp
fi
```

Wire it in the manifest if a `setup_once` hook is not already declared (no manifest *schema* change is needed — `[hooks]` already exists):

```toml title="jackin.role.toml"
[hooks]
setup_once = "hooks/setup-once.sh"
```

### 3. Bake agent guidance

Bake a guidance file into the agent's home memory so the model knows the tools exist and when to use them. For Claude this is `~/.claude/CLAUDE.md`:

```dockerfile
COPY --chown=agent:agent code-intelligence.md /home/agent/.claude/CLAUDE.md
```

```markdown title="code-intelligence.md"
# Code-intelligence tooling (the-architect)

This container ships dedicated code-intelligence tools. Prefer them over ad-hoc text search when the query is about code structure.

## Structural search & refactor — ast-grep
`ast-grep` is installed. Invoke it as `ast-grep`, never `sg` (`sg` is the system set-group command on Debian).

Reach for ast-grep first whenever the target is a *syntax structure*, not literal text: functions, `impl` blocks, trait impls, macro definitions/calls, `async fn`, derives, match arms, or specific call shapes (`$X.unwrap()`, `$X.clone()`).

- Search: `ast-grep --lang rust --pattern '<pattern>'` (short: `ast-grep -l rust -p '<pattern>'`).
- Rewrite: add `--rewrite '<replacement>'`.
- Lint/scan: `ast-grep scan` against rules in `sgconfig.yml`.
- Metavariables: `$X` matches one node, `$$$ARGS` matches a list. Every `.unwrap()`: `ast-grep -l rust -p '$X.unwrap()'`.

Use ripgrep (`rg`) or built-in grep for *plain text* — string literals, log/error messages, file paths, config keys, comments. That is the common case; do not force ast-grep on it.

## Semantic navigation — rust-analyzer
Definitions, references, types, and implementations come from rust-analyzer (on `PATH`). Resolve symbols with go-to-definition / find-references / hover rather than guessing from text matches.

## Verify
- Tests: `cargo nextest run`
- Lints: `cargo clippy --all-targets --all-features -- -D warnings`
- Format: `cargo fmt --check`
```

Bake the same body into each supported runtime's home memory file. The default-home capture in <RepoFile path="src/derived_image.rs" /> copies `~/.claude/`, `~/.codex/`, and the other agent homes, so a role-baked file in any of them survives into the live container:

| Runtime | Memory file to bake |
|---|---|
| `claude` | `~/.claude/CLAUDE.md` |
| `codex` | `~/.codex/AGENTS.md` |
| `amp` / `kimi` / `opencode` | the runtime's user-level instructions file |

Claude and Codex are the priority targets for the pilot.

## How the agent learns the tools

1. The the-architect `Dockerfile` writes `code-intelligence.md` to `/home/agent/.claude/CLAUDE.md` in the agent layer, on top of the construct.
2. The jackin-generated derived layer runs after the agent layer. Its default-home capture step (`cp -a /home/agent/.claude/. /jackin/default-home/.claude/`, see <RepoFile path="src/derived_image.rs" />) snapshots that file into `/jackin/default-home/`.
3. On first boot, the runtime seeds `/jackin/default-home/` into `/home/agent/`, so the file lands in the live agent home.
4. `Claude Code` reads `~/.claude/CLAUDE.md` as user-level memory at startup, so the guidance is in context for every session regardless of which repo the operator mounted. The same chain delivers `~/.codex/AGENTS.md` to Codex.

Because the guidance lives in the agent's home, it applies to every workspace the-architect runs against and never mutates a host repository.

## Verification

Smoke-test from a jackin checkout (always `--debug`; share the printed run id if something misbehaves):

```bash
cargo run --bin jackin -- load the-architect . --debug
```

Inside the container, confirm each surface:

- `ast-grep -l rust -p '$X.unwrap()'` returns structural matches.
- `command -v rust-analyzer` resolves; `Claude Code` go-to-definition works on a symbol.
- `claude mcp list` shows `fff`; ask the agent to "use fff" for a file search.
- `cargo nextest run` runs the suite.
- When asked to find all trait impls or `.unwrap()` calls, the agent reaches for `ast-grep` rather than `rg`.

## Success criteria

The pilot succeeds when, across a handful of representative the-architect sessions:

- the agent invokes `ast-grep` for structural queries and `rust-analyzer` for symbol resolution instead of ad-hoc text search;
- code lookups cost fewer tokens than repeated `rg` plus file reads, net of the `fff` MCP schema carried each turn;
- fewer edits land on wrong paths because symbols were resolved rather than guessed;
- `fff`'s resident index measurably beats the construct's ripgrep on this repo — or the pilot concludes it does not, and `fff` is dropped while the CLI and LSP tools stay.

## Follow-ups (deferred)

- **ast-grep lint gate.** Add `sgconfig.yml` plus `.ast-grep/rules/*.yml` (flag `todo!()`/`unimplemented!()`, audit `.unwrap()`) and run `ast-grep scan` in CI. Independent of the agent-guidance pilot; can land separately.
- **Declarative MCP servers in the manifest.** A first-class `[claude].mcp_servers` (and per-runtime equivalents) field in `jackin.role.toml` would register MCP servers without a hook. This is a versioned-schema change (one `CURRENT_MANIFEST_VERSION` bump with migration and fixtures) and is only worth designing if the pilot proves MCP servers worth generalizing.
- **Skill-based guidance.** Deliver the guidance as a `Claude Code` skill loaded on demand instead of always-on `CLAUDE.md`, cutting standing memory cost.
- **Generalize to a shared layer.** Move the CLI tools (`ast-grep`, `rust-analyzer`) into a shared base so other Rust roles inherit them, keeping `fff`/MCP opt-in.
