This feature is expected to merge first because it is far smaller. To keep the eventual reconciliation mechanical rather than surgical, every phase here is implemented in shapes that match the refactor's target architecture. An implementer working on any remaining phase must follow these alignment rules.

The refactor's relevant end-state (for status work):

- The capsule renders through Ratatui. A custom `SocketBackend` serializes the Ratatui `Buffer` diff to the attach socket. Dialog overlays are built from shared `jackin-tui` components, not raw ANSI byte pushes.
- `crates/jackin-capsule/src/statusbar.rs` is removed; the status bar lives in `crates/jackin-capsule/src/tui/components/status_bar.rs`.
- `daemon.rs` is decomposed into submodules under `crates/jackin-capsule/src/daemon/` (compositor, session lifecycle, pane layout, multiplexer utilities, …).

- The refactor does **not** remove the silence-timer heuristic and does **not** add `Unknown` — those are this feature's job, so the two are complementary, not redundant.

Alignment rules to minimize future conflicts:

1. **Keep status derivation and token data in pure modules (`agent_status/`, `token_monitor/`) with no rendering.** These are new files the refactor does not touch, so they merge clean. The bulk of every phase's logic must live here. Rendering is a thin shim layered on top.
2. **Protocol `AgentState` stays the single wire enum; `Unknown` is added there.** When the refactor merges, it gains one new arm in `visible_agent_state_from_protocol` and one new `VisibleAgentState::Unknown` (plus `Stuck` later). Treat that two-line addition as the documented merge cost; do not invent a competing render enum on `main`.
3. **Tab-glyph logic is keyed on the effective status and kept in one cohesive function.** On `main` it lives in `statusbar.rs::tab_label`; on merge it moves whole to `tui/components/status_bar.rs` and switches its key type from `AgentState` to `VisibleAgentState`. Keep the glyph→state mapping a single match so the move is copy-paste.
4. **Daemon additions stay cohesive and ticker-local.** The per-tick status refresh is one method (`refresh_session_statuses`) plus one `Multiplexer` field (`detectors`) plus one `state_ticker` arm edit. On merge it relocates to the refactor's session-lifecycle submodule as a block, not interleaved lines.
5. **The `Session.state` field → `Session::state()` accessor swap is deliberate.** The refactor keeps a `state: AgentState` field with ~3 read sites; on merge those become `session.state()` and the refactor's `refresh_state`/`state_after_*` bodies are deleted (this feature's Phase 0). The accessor shape makes that a find-replace.
6. **Phase 6 token dialog must be Ratatui-ready.** Build the dialog's data model and layout (`ProviderUsageSnapshot`, `RateWindow`, the line/span computation) as a pure function in `token_monitor/`. The render shim that paints it is the only piece that touches the dialog system. On `main` that shim may be raw-ANSI to match the pre-refactor dialog path; on merge only the shim is rewritten to a `jackin-tui` Ratatui widget — the layout logic is reused unchanged. Do not bury layout decisions inside ANSI emit code.

## Implemented Shape (As Shipped)

The shipped implementation diverges from the original design proposal below in three deliberate ways. This section is the source of truth for what exists in the code today; the design sections that follow describe the remaining work and retain the original proposal's vocabulary where it has not yet been built.

- **`AgentState` keeps its name; `Unknown` added.** The proposal called for renaming the wire enum to `AgentEffectiveStatus`. Instead, the existing `AgentState` in <RepoFile path="crates/jackin-protocol/src/control.rs" /> gained a fifth variant `Unknown` (`#[serde(rename_all = "snake_case")]` → `"unknown"`). Renaming would have broken every host-CLI read site for no functional gain; the enum already _is_ the effective-status type. Variants: `Working`, `Blocked`, `Done`, `Idle`, `Unknown`.
- **`SessionStatus::advance()` state machine instead of `arbitrate_session_status()`.** The proposal described a pure `arbitrate_session_status(hook, screen, process)` function consuming three evidence structs. The shipped design is a small state machine in <RepoFile path="crates/jackin-capsule/src/agent_status/mod.rs" />: `SessionStatus { effective, seen, revision }` with `advance(AgentRawState) -> Option<AgentState>` (returns `Some` only on a real transition) and `acknowledge() -> Option<AgentState>` (Done → Idle on focus). `Done` is derived inside `advance` from raw-idle + `!seen`, never reported directly. The full evidence-struct arbitration (hook authority + sequence validation + process consistency) is deferred to Phases 2–3 when those signal sources exist.
- **`AgentRawState` is the input-signal enum, not the detector output.** The proposal's `AgentRawState` was `Unknown|Working|Blocked|Idle` (the detector's verdict). The shipped `AgentRawState` is a 10-variant _signal_ enum — `PromptVisible`, `WorkingVisible`, `BlockedVisible`, `Osc133PreExec`, `HookTaskStart`, `HookTaskDone`, `CursorProbeOk`, `CursorProbeTimeout`, `OperatorInput`, `ProcessExited` — that every signal source emits and the state machine folds into the effective `AgentState`. This keeps each source (screen detector, future hook, future cursor probe) emitting one vocabulary while the machine owns the transition policy.
- **`Detector` trait + `DetectorRegistry` for screen detection.** Each built-in runtime has a `Detector` impl in <RepoFile path="crates/jackin-capsule/src/agent_status/detectors/mod.rs" /> (`claude`, `codex`, `amp`, `kimi`, `opencode`) returning `Option<AgentRawState>` from the bottom `DETECTION_ROWS = 24` of the `vt100::Screen`. Adding a new agent is one file + one registry line. The registry runs once per session on the 1Hz ticker via `Multiplexer::refresh_session_statuses()` in <RepoFile path="crates/jackin-capsule/src/daemon.rs" />, which also acknowledges the focused pane so its `Done` clears to `Idle`.
- **`Session` owns one `status: SessionStatus`.** The old `state: AgentState` field is gone; `Session::state()` derives it from the machine. `mark_operator_input()` and `acknowledge()` route through the machine. PTY output no longer mutates state directly — it only stamps `last_output_at` (a weak signal for future stale/stuck detection).
- **TUI.** The tab strip in <RepoFile path="crates/jackin-capsule/src/statusbar.rs" /> renders `Unknown` as a mid-gray `·`. Roll-up priority `Blocked > Done > Unknown > (Working/Idle)`. The `Stuck` overlay glyph is not yet wired (lands with stuck detection in Phase 4).

## Problem

jackin' needs to know what each in-container agent is actually doing: actively working, waiting on the operator, finished but unseen, idle and reviewed, crashed, or suspiciously stuck. The current multiplexer code has the protocol labels but not the authority behind them. In <RepoFile path="crates/jackin-capsule/src/session.rs" />, `Session::refresh_state` still reduces state to "output in the last few seconds means working; otherwise blocked." That is not good enough for a first-class agent orchestrator. A quiet agent can be thinking, waiting for network, running a long tool, sitting at an input box, or dead. Treating all quiet periods as `blocked` creates false notifications, noisy dashboards, bad queue dispatch, and weak operator trust.

The roadmap item is the state authority jackin' needs before agent attention prompts, autonomous task queues, Desktop Agent Hub, or cost/resource correlation can be reliable. It replaces timer-only inference with a layered model that is conservative by default and precise when the runtime exposes semantic events.

## Target

jackin' should become the native, agentic terminal orchestrator: agents remain visible in real PTYs, but jackin' understands enough of their lifecycle to route attention and automation without babysitting. The operator should see one consistent status vocabulary across `jackin console`, `jackin hardline`, the daemon, Desktop Agent Hub, and scripts:

| Public status | Meaning |
|---|---|
| `working` | The agent is actively generating, running tools, processing a task, or otherwise not ready for operator input. |
| `blocked` | The agent is waiting on an operator decision, approval prompt, question, auth prompt, or other interactive input. |
| `done` | The agent has returned to idle after work and the operator has not yet reviewed that pane/session. |
| `idle` | The agent is idle and already seen, or the pane is a shell/no active agent. |
| `unknown` | jackin' cannot identify a reliable state yet. Unknown is safer than pretending silence means blocked. |
| `stuck` | Diagnostic overlay for "we expected progress, but no reliable state transition happened before a timeout." This should not replace `blocked`; it is a health signal used for escalation and debugging. |

Internally, the raw runtime detector should track `unknown | working | blocked | idle`, plus per-pane `seen` metadata. `done` is derived from raw `idle + !seen`, not reported as a raw runtime state. This matches the useful part of Herdr's model and avoids conflating "the process is idle" with "the operator has reviewed the result."

## Prior Art

### Herdr

[Herdr](https://herdr.dev/) is the strongest current public reference for this problem. Its docs describe the operator-facing feature directly: the sidebar shows blocked, working, done, and idle agents; workspaces roll up to their most urgent state; detection works through foreground process and terminal output with zero config, with a socket API for runtimes that expose hooks. Its source confirms that the important behavior is not a single regex or timer; it is the arbitration layer that decides which signal is authoritative when signals disagree.

**Concepts to borrow, not code:**

- **Process ownership first.** Herdr identifies which agent owns a pane by reading the foreground process group, then matching known agent binaries. It handles wrappers such as `node`, `bun`, `python`, and shells before deciding the real agent label. jackin' can do this more cleanly inside the container because it sees the real agent process instead of a host-side terminal wrapper.
- **Visible terminal evidence is current-screen evidence.** Herdr reads the bottom/recent terminal screen, not arbitrary scrollback. Its default detection window is the last 24 rows (`DEFAULT_DETECTION_ROWS = 24`). Screen polling happens every 300ms in a per-pane async task.
- **Per-agent visible signals are deliberately narrow.** Herdr separates `visible_blocker`, `visible_idle`, and `visible_working` as separate boolean fields in `AgentDetection`. Only strong current UI states should override semantic reports.
- **Semantic reports are primary when they are fresh and consistent.** Herdr's socket reports carry source, agent label, state, message/custom status, sequence, and session references using nanosecond timestamps as monotonic sequence numbers.
- **Central arbitration is the product.** Hook authority, visible blockers, visible idle, visible working, process exit, stale reports, and sequence numbers flow through one state machine in `recompute_effective_state()`. The UI consumes the result; it does not re-detect state.
- **Done is an acknowledgement overlay.** Herdr maps raw idle plus `seen = false` to public `done`; once the pane is viewed, it becomes `idle`.
- **Attention roll-up prioritizes attention, not activity.** Herdr's effective priority is `blocked(4) > idle+unseen/done(3) > working(2) > idle+seen(1) > unknown(0)`. A finished unseen pane deserves attention before a pane that is merely busy in the background.

**Herdr's arbitration algorithm in detail** (to be reimplemented from scratch for jackin'):

```
Priority (highest first):
1. visible_blocker AND screen not older than hook → override to Blocked
2. visible_working AND (hook reports Idle or Blocked) AND (screen is fresher
   than hook, OR hook is within CLAUDE_WORKING_HOLD=1200ms of screen) → override to Working
3. visible_idle AND hook reports Working or Blocked AND
   STALE_HOOK_IDLE_GRACE=2s elapsed since idle signal started → stale hook, use Idle
4. Hook authority state (if present, process-consistent, and sequence valid)
5. Screen fallback state (visible detector result)
6. Unknown
```

**Edge cases Herdr handles that jackin' must design for from the start:**

- Stale hook authority after process exit: hook authority for an agent is cleared when the process exits AND the hook is not newer than the process-exit time.
- Cross-agent stale authority: if the process probe sees a different agent than the one in hook authority, and the hook is not newer than the screen observation, clear the hook.
- Codex install-process false positives: the visible detector for Codex must not match installer/update output as an active agent state.
- Claude working hold of 1200ms: if Claude screen was `Working` and flips to `Idle`, hold `Working` for 1200ms to prevent blink between tool calls.
- Transcript viewer skip: if Claude Code shows a transcript/history viewer (`showing detailed transcript` + `ctrl+o to toggle`), suppress all state updates to avoid an idle-looking transcript overriding a working state.
- SubagentStop exclusion: the Claude hook must explicitly ignore `SubagentStop` events to avoid reviving an idle pane from background agent completions.
- Agent miss confirmation: 6 consecutive foreground process misses before clearing the current agent, preventing brief process-group transitions from flipping state.

### multicode

[multicode](https://github.com/graemerocher/multicode) is the best comparison for structured runtime state. It does not solve status as a terminal-observability problem first; it asks the runtime when possible.

**Useful concepts:**

- **Codex app-server status.** multicode talks JSON-RPC 2.0 over WebSocket to `codex app-server --listen ws://<host>:<port>`. The handshake is `initialize` → `initialized` then subscriptions. The primary push event is `thread/status/changed` with the status shape `{"type":"notLoaded"|"idle"|"systemError"|"active","activeFlags":["waitingOnApproval"|"waitingOnUserInput"]}`. multicode treats `active+waitingOnApproval` or `active+waitingOnUserInput` as `blocked`, plain `active` as `working`, and `idle` as `idle`. It also keeps a turn active even if Codex races back to `idle` by tracking unmatched `turn/started` notifications via `active_turn_thread_id`. **Critical risk**: the app-server and the visible TUI session are separate clients — running `codex app-server` alongside the visible `codex` TUI may not observe the same session. Do not use app-server as truth until same-session observation is verified.
- **OpenCode HTTP/SSE state.** multicode uses a generated OpenAPI client to subscribe to `GET /global/event` as SSE, polls `GET /session/status` for the map of all sessions, and lists `GET /question` for pending user questions. Session statuses are `idle`, `busy`, or `retry`. Pending questions override status to `Question`. There is no session-scoped SSE stream; all clients must subscribe to the global bus and filter by session ID.
- **Descendant session roll-up.** multicode promotes a root session to `Busy` while any descendant subagent session is busy, using a `parentID` chain walk with cycle guard. A parent pane returning to a prompt must not hide child work that is still active.
- **Cooperative state files.** multicode's autonomous-state skill writes a one-line file such as `working`, `question`, `review`, or `idle`, optionally with a session id, and polls it every two seconds (`STATE_REFRESH_INTERVAL`). Stale states are preserved when a file is missing or unreadable to prevent brief write-gap loss. This is useful as a heartbeat/evidence channel but cannot be proof because the agent can crash, forget, or race.

### Agent Session Manager

`izll/agent-session-manager` (MIT, commit `7432789`) is the strongest source for Claude Code structural prompt-box detection. Key implementation details:

**tmux capture-pane arguments**: `tmux capture-pane -t <session>:<window> -p -S -50` — captures the last 50 lines (scrollback + screen) and prints to stdout. jackin' does not need tmux; it has direct `vt100::Screen` access. The line depth is a useful reference point: 50 lines is generous; Herdr's 24 is tighter and avoids false positives from older output.

**Claude Code structural detection** (verified in `status_detector.go`): rather than scanning all lines with regexes, the detector finds Claude's horizontal separator lines — lines where `count('─') + count('━') > 20`. With two separators present: the area between them is the input box; scan 15 lines above the top separator for the spinner/busy area. With one separator: everything below it is a potential permission dialog. With zero separators: fall back to the last 10 lines. Lines starting with `╭`, `╰`, `└`, `Tip:`, `Next:`, lines containing `⎿` or starting with `├`, `│`, and lines containing ` for ` (completed timing like "Churned for 1m") are all skipped before pattern matching.

**Waiting-over-busy two-pass approach**: waiting patterns are checked on `inputAreaLines + belowSeparatorLines` only — NOT on `aboveSeparatorLines` where old dialogs might linger. Busy patterns are checked on `inputAreaLines + aboveSeparatorLines`. The first-pass waiting check returns immediately on a match; busy is only checked after. This prevents stale dialogs in scrollback from triggering false Waiting results.

**Followed-window aggregation priority**: `Waiting > Busy > Idle` — the first Waiting result across any followed window short-circuits and returns immediately.

### WezTerm Agent Deck

`Eric162/wezterm-agent-deck` (MIT, commit `0bdd442`) provides the most detailed process-identity logic of any reviewed project. Key implementation details:

**Three-tier process identification** (Lua, WezTerm API):
1. `pane:get_foreground_process_info()` — primary. Returns `.executable`, `.name`, `.argv`, `.children`. If the foreground process is `node`, iterate `.children` to find the real agent (Claude Code, OpenCode) since Node.js wraps the agent JS bundle.
2. `pane:get_foreground_process_name()` — secondary fallback when `get_foreground_process_info()` fails.
3. `pane:get_title()` and `pane.title` — tertiary fallback. Matches against `title_patterns` per agent (e.g. Claude has `{ 'claude code', 'claude' }`).

**Detection cache TTL**: `CACHE_TTL_MS = 1500` ms in the active implementation (the standalone `detector.lua` module uses 5000ms but the inlined `init.lua` version that actually runs uses 1500ms). A stale-cache cleanup pass runs every 30 seconds (`CLEANUP_INTERVAL_MS = 30000`); entries older than 60 seconds are evicted.

**Line window sizes for detection**: idle check uses last 5 lines (checks for `>` or `> ` prompt); working/waiting checks use last 10 lines in the active implementation. The rationale in source comments: "only in last 10 lines to avoid stale matches." The max scan depth is bounded by `config.max_lines` (default 100).

**Stale working false positive handling** (confirmed by tests): idle is checked first (last 5 lines for `>` prompt), and if found it returns immediately without reaching the working/waiting pattern scan. `Esc to interrupt` that appeared earlier in scrollback will not trigger `working` if the current last 5 lines show `> `.

**Subagent counter (sticky blocked rule)**: On `SubagentStart`, increment a counter. On `SubagentStop`, decrement. While counter > 0, `PostToolUse`-equivalent events must not clear a `Blocked` state — only explicit `PermissionResult`/`UserPromptSubmit` events or a new `Blocked` report may transition out. This is the same mechanism as wezterm-agent-cards' sticky-waiting rule.

### Broader Comparison Lessons

Screen classifiers need fixture tests, debounce, and false-positive fixtures. They should parse visible/current terminal UI, not historical scrollback. Agent identity detection should combine child PID, foreground process group, argv/cmdline, wrapper process handling, and short cache TTLs. Attention notifications should fire on transitions and be rate-limited. Polling or redraw frequency should never become notification frequency. Waiting beats busy, and unseen finished work beats merely working background panes. The roll-up priority is an operator-attention rule, not a CPU-activity rule. Parent panes must remain working while descendant/subagent work is active. Hook/status-file ideas are useful only when installed inside the container. Host-side hook mutation remains out of bounds by default. Health scores, circuit breakers, and keep-awake behavior are adjacent follow-ups. They should consume the status authority; they should not become new detectors.

The comparison also shows what to avoid. Regex-only and silence-only detectors drift quickly as agent TUIs change. Cooperative state files are useful as heartbeat/evidence channels, but they cannot prove the agent is not stuck. Host-side hook mutation is common in the ecosystem, but jackin' default path must stay container-local so the operator's host config is never silently changed.

### Comparative Validation

The researched projects validate the concept, but they also change the recommended implementation order. Most tools are external observers: they cannot assume control over the agent installation, cannot safely rewrite agent config, and often only see a tmux/WezTerm pane. That forces them toward pane scraping, prompt regexes, host hook setup, or host-side status files. jackin' has a better boundary because it owns the role image, the container filesystem, `/jackin/runtime`, `/jackin/state`, `/jackin/run`, and the container-local agent homes. Therefore jackin' should not copy an observer-first architecture. It should install deterministic in-container reporters and treat screen/process detection as validation and fallback.

| Implementation family | Projects that use it | Borrow | Do not borrow |
|---|---|---|---|
| External screen/process observation | Herdr, CCManager, Agent Session Manager, WezTerm Agent Deck, ccmux, claudeye, TUICommander | Current-screen windows, foreground process identity, idle debounce, prompt-box-region parsing, recent-line priority, false-positive fixtures, and hermetic PTY/tmux tests. | Observer-first truth model, broad historical regexes, silence-as-input-needed, and independent UI-level terminal parsers. |
| Host hook/status-file reporting | Codemux, tmux-agent-status, agent-deck | Hook events as semantic transitions, atomic status writes, freshness windows, done/unseen acknowledgement, source-specific stale handling, fsnotify overflow recovery, and hook schema drift repair. | Silent host `~/.claude`, `~/.gemini`, shell rc, tmux, or project-file mutation. jackin' can install the same idea inside the container instead. |
| Structured runtime APIs | multicode, OpenCode/Codex integrations in other tools | Runtime-native waiting flags, SSE/JSON-RPC subscriptions, turn lifecycle events, pending-question priority, and descendant/subagent roll-up. | Polling a structured API from the host as a separate truth path. The bridge should live in the container and report through Capsule. |
| Agent-written completion/health signals | Codeman, agent-deck | Explicit done sentinels, stuck/circuit-breaker evidence, health consumers, and conservative "when unsure, still working" policy. | Treating an agent-authored line as proof of runtime state without process identity, sequence, and freshness validation. |
| Terminal protocol readiness | Zylos, VS Code shell integration, Windows Terminal shell integration, Warp-style shell markers | Cursor-position probes, output-stability windows, OSC prompt/command markers, cwd/command metadata, and graceful fallback when probes fail. | Using cursor readiness as a blocker detector for full-screen agent TUIs, or assuming shell markers explain an embedded agent runtime. |

The best implementation is therefore reporter-first and detector-verified: install a versioned jackin' status reporter into the container, wire every supported runtime to it when a stable hook/API exists, validate reports against process and visible-screen evidence, and keep `unknown` as the conservative state when neither semantic events nor strong current-screen evidence are available.

### Terminal Shell Integration

Terminal ecosystems already solved a narrower version of this problem for shell commands. VS Code's terminal shell integration uses custom OSC sequences such as prompt start (`OSC 133 ; A`), prompt end (`OSC 133 ; B`), pre-execution (`OSC 133 ; C`), and command-finished (`OSC 133 ; D ; <exit_code>`) markers; it also supports `OSC 633` for VS Code-specific extensions. Windows Terminal documents the same family of prompt/command markers. These signals are excellent when the foreground program is an ordinary shell, but they do not fully solve embedded agent TUIs that draw their own prompt, approval UI, or full-screen interface.

**Important limitation**: Claude Code itself does NOT emit OSC 133 prompt markers as of June 2026. Four GitHub issues on `anthropics/claude-code` have requested this feature: issue #1465 was closed as "not planned" (the stated rationale is that the Claude Code TUI is not a standard shell prompt loop), issue #22528 was closed as duplicate, and issues #26235 and #32635 remain open feature requests. When Claude Code is the foreground program — not a shell — OSC 133 does not identify Claude's prompt vs. command boundaries. jackin' should still listen for OSC 133 markers that pass through the PTY (they are useful for shell command boundaries, cwd tracking, and "agent returned to shell" transitions) but must not rely on them for Claude Code status detection.

Terminals with confirmed OSC 133 support: Ghostty, iTerm2, Kitty, WezTerm, VS Code Terminal, Windows Terminal, Contour Terminal. Shells that emit them natively when integration is enabled: Zsh (built-in), Fish, Bash.

**Exact byte sequences:**

All sequences use ESC (`\033`) + `]` + numeric code + `;` + letter + string terminator. Both BEL (`\007`) and ST (`\033\\`) are valid terminators; BEL is more compact and universally supported.

| Sequence | Escape string | Meaning |
|---|---|---|
| `OSC 133 ; A` | `\033]133;A\007` | FTCS_PROMPT — prompt start |
| `OSC 133 ; B` | `\033]133;B\007` | FTCS_COMMAND_START — prompt end / start of command input |
| `OSC 133 ; C` | `\033]133;C\007` | FTCS_COMMAND_EXECUTED — pre-execution (user pressed Enter, before output) |
| `OSC 133 ; D ; exitcode` | `\033]133;D;0\007` | FTCS_COMMAND_FINISHED — command done, with optional exit code |
| `OSC 7 ; path` | `\033]7;file://HOSTNAME/percent-encoded-path\033\\` | Current working directory |
| `OSC 633 ; A/B/C/D` | Same letters as OSC 133 | VS Code extension of OSC 133; guard behind `$TERM_PROGRAM == vscode` |
| `OSC 633 ; E ; cmdline` | `\033]633;E;cmdline\007` | Explicit command line capture (semicolons escaped as `\x3b`) |
| `OSC 633 ; P ; Cwd=/path` | `\033]633;P;Cwd=/path\007` | VS Code property set (`Cwd`, `IsWindows`, etc.) |

The canonical per-cycle ordering is: `D → A → [PS1 renders between A and B] → B → [user types] → C → [command output] → D → A → …`

**Shell prompt instrumentation**: by writing to `/home/agent/.zshrc` or `/home/agent/.bashrc` inside the container to add `precmd`/`PROMPT_COMMAND` hooks, jackin' can get reliable shell-command-boundary detection regardless of terminal emulator support. This belongs in the container injection setup path, not in the host shell config. The complete Zsh integration block for Docker containers:

```zsh
# Shell integration: OSC 133 + OSC 7
_si_urlencode() {
    local -a chars
    chars=( ${(s::)1} )
    printf '%s' ${(j::)chars/(#b)([^A-Za-z0-9\/._~!$&\'()*+,;=:@-])/%${(l:2::0:)$(([##16]#match))}}
}

_si_osc7() {
    printf '\033]7;file://%s%s\033\\' "${HOSTNAME}" "$(_si_urlencode "${PWD}")"
}

# precmd fires after each command, before the prompt is drawn.
# Emits D (command finished + exit code), then A (prompt start), then OSC 7.
_si_precmd() {
    local exit_code=$?
    printf '\033]133;D;%d\007' "${exit_code}"
    printf '\033]133;A\007'
    _si_osc7
}

# preexec fires after the user presses Enter, before the command executes.
# Emits B (prompt end / command committed), then C (pre-execution).
_si_preexec() {
    printf '\033]133;B\007'
    printf '\033]133;C\007'
}

autoload -Uz add-zsh-hook
add-zsh-hook precmd  _si_precmd
add-zsh-hook preexec _si_preexec
add-zsh-hook chpwd   _si_osc7
```

`HOSTNAME` is set automatically by Docker (to the container ID or `--hostname` value) and is accessible in Zsh as `$HOSTNAME`. The `_si_precmd` function captures `$?` as its first act — any intervening operation would clobber the exit code. The `add-zsh-hook` approach is cleaner than appending to `precmd_functions` directly.

### Cursor-Position And Screen-Stability Probes

The strongest non-Herdr conceptual work is cursor-position readiness detection: query the PTY cursor position with `CSI 6n` (DECXCPR), combine that with a short output-stability window, and infer whether input would land at a prompt. Zylos' research argues this is more robust than prompt regexes because it asks "where will the next character appear?" instead of "does the screen text match this prompt string?"

This belongs in jackin' as a fallback/probe, not as the primary blocker detector. It is useful for "is this pane input-ready?" and "is a submitted task suspiciously stuck?", but full-screen TUIs, nested multiplexers, and terminals that intercept cursor reports can produce false positives. The state authority should treat cursor probes as one signal with confidence metadata, not as truth. Cursor probes must run in a separate task with a hard timeout (suggested: 2 seconds) and must never block the PTY reader, attach channel, or daemon event loop while waiting for the response.

Of these, only `Working` and `Blocked` are ever set on a live session. `Idle` and `Done` are never set from any live code path. `Session::refresh_state()` fires on a tick; if `last_output_at.elapsed() >= BLOCKED_AFTER` (3 seconds) and state is `Working` or `Idle`, it flips to `Blocked`. ## jackin' Design

This section is the implementation contract. Future agents working on this feature should treat the requirements here as the source of truth unless a later PR updates this roadmap item first.

### Container Injection Advantage

jackin' should use the control it has and other projects usually lack. Host-side terminal tools must be zero-config observers because they cannot assume they can install files into Claude Code, Codex, OpenCode, Kimi, Amp, tmux, WezTerm, or a user's shell without surprising the operator. jackin' builds and launches the role container, so it can install status assets owned by jackin' on every launch, repair drift, and keep all mutations inside `/jackin/` plus container-local agent homes.

The default status architecture should be:

1. Install a small reporter command under `/jackin/runtime/agent-status/` that talks to the existing Capsule socket at `/jackin/run/jackin.sock`.
2. Launch every built-in agent with stable environment including `JACKIN_SESSION_ID`, `JACKIN_AGENT_RUNTIME`, `JACKIN_STATUS_SOCKET=/jackin/run/jackin.sock`, and `JACKIN_STATUS_SOURCE=<runtime-or-hook-id>`.
3. Merge runtime hook/plugin/API bridge configuration into the container-local agent home during `runtime_setup`; repair drift on every launch the same way agent-deck repairs stale hook configuration, but never against the host home.
4. Let hooks/API bridges report `working`, `blocked`, `idle`, `heartbeat`, `child-start`, `child-stop`, and `clear` events with source id, sequence, timestamp, runtime label, and session id.
5. Use foreground process, visible screen, shell markers, cursor probes, and weak activity to validate, stale, or repair those reports. They are guardrails and fallbacks, not the preferred truth source.

This is the main architectural difference from Herdr, CCManager, ccmux, WezTerm Agent Deck, and tmux-agent-status. Their zero-config observation constraint is a product strength for host tools, but it is not the best default for jackin'. jackin' can still keep zero host setup: the injected reporter lives in the role container and is removed with the container/image state.

### Implementation Surface

The first implementation should add a dedicated status authority module at `crates/jackin-capsule/src/agent_status/mod.rs`. `Session` in <RepoFile path="crates/jackin-capsule/src/session.rs" /> should grow fields for child PID, detected foreground agent, `AgentRawState`, `AgentEffectiveStatus`, `seen` acknowledgement state, current authority source, evidence metadata, revision counter `u64`, and last-seen revision `u64`. The protocol type in <RepoFile path="crates/jackin-protocol/src/control.rs" /> should add `Unknown` to `AgentState`; since there are no external users before first release, replace the enum shape directly rather than adding compatibility shims.

The raw detector state should stay separate from the public status. Raw states are `unknown`, `working`, `blocked`, and `idle`. Effective statuses are `unknown`, `working`, `blocked`, `done`, `idle`, and optional diagnostic overlays such as `stuck`. `done` is computed from raw idle plus the `seen` flag and should never be accepted from a hook as runtime truth.

**The concrete model:**

| `AgentEffectiveStatus` | `enum { Unknown, Working, Blocked, Done, Idle }` | What UI, CLI, and host consumers read. Also in protocol crate. |

The protocol should expose `AgentRawState`, `AgentEffectiveStatus`, `AgentStatusSource`, `AgentStatusConfidence`, and a summary `AgentStatusReport` as structured JSON fields in `SessionInfo` and `PaneSnapshot`. UI labels, colors, icons, and workspace roll-ups are consumers of the model.

### Signal Sources

1. **Semantic reports.** Add a control-channel method for trusted in-container runtime integrations to report state: source, agent/runtime, raw state, optional message, optional custom status, sequence number, timestamp, and session reference. Reporters include Claude Code hooks, Codex structured status when available, OpenCode plugins/API bridges, Amp/Kimi equivalents when available, and wrapper hooks created by jackin' inside the container at launch. This is the preferred source when fresh and process-consistent.
2. **Process ownership.** Track the child PID for each session, read Linux `/proc/<pid>/stat` to get the foreground terminal process group (`tpgid` field, **field index 8** in `/proc/<pid>/stat`), scan `/proc` for that process group, and identify known agent binaries. Because jackin' runs inside Linux containers, this can be Linux-first and simpler than Herdr's cross-platform path. Use the `procfs` crate (`procfs = "0.18"`, crates.io canonical crate by `eminence`, actively maintained as of 2025-08-30). Process exit should clear hook authority for that agent and transition toward `idle` or `exited`, not leave a stale working/blocking state behind.
3. **Visible screen detection.** Read the current `vt100::Screen` snapshot for the pane and match only strong, current-screen signals: approval prompts, explicit input-required prompts, visible idle input UI, and visible working chrome. Avoid treating arbitrary scrollback text as authority. The detection window should be the last 24 rows of the current viewport (matching Herdr's `DEFAULT_DETECTION_ROWS`).
4. **Shell-integration markers.** Capture `OSC 133`, `OSC 633`, `OSC 7`, and related markers when shells emit them. `OscCapture` in <RepoFile path="crates/jackin-capsule/src/session.rs" /> already intercepts OSC sequences; extend it to detect OSC 133 markers. Use them for shell prompt/execution boundaries and cwd/title enrichment.
5. **Cursor-position probes.** Optionally probe `CSI 6n`/cursor position through the PTY for readiness/stuck diagnostics, with timeout and fallback. Never block the PTY reader, attach channel, or daemon event loop while waiting for a probe response.
6. **Output activity.** Keep output timestamps, but demote them to a weak signal: recent output supports `working`; silence alone does not prove `blocked`.

**Process ownership — Rust implementation details:**

Use the `procfs` crate. The `Stat` struct from `procfs::process::Process::stat()` exposes `tpgid` at field index 8 (1-indexed per kernel documentation). Key distinction: `stat.pgrp` (field 5) is the process's own PGID; `stat.tpgid` (field 8) is the PGID of whichever process group currently owns the controlling terminal.

```rust
use procfs::process::{all_processes, Process};
use std::path::Path;

/// Returns the foreground process group ID for the terminal attached to `pid`.
fn read_tpgid(pid: u32) -> Option<u32> {
    let stat = Process::new(pid as i32).ok()?.stat().ok()?;
    Some(stat.tpgid as u32)
}

/// Returns all PIDs in a given process group.
fn pids_in_pgrp(target_pgrp: i32) -> Vec<i32> {
    let Ok(iter) = all_processes() else { return vec![] };
    iter.filter_map(|e| {
        let proc = e.ok()?;
        let stat = proc.stat().ok()?;
        if stat.pgrp == target_pgrp { Some(stat.pid) } else { None }
    })
    .collect()
}

#[derive(Debug, PartialEq)]
enum AgentKind { ClaudeCode, Codex, Amp, Kimi, OpenCode, Unknown }

fn identify_agent(pid: i32) -> Option<AgentKind> {
    let proc = Process::new(pid).ok()?;
    let cmdline = proc.cmdline().ok()?;  // Vec<String>, split on NUL bytes
    let exe = proc.exe().ok()?;
    let exe_name = exe.file_name()?.to_string_lossy();

    // Native binaries: exe name is the agent slug directly.
    match exe_name.as_ref() {
        "codex"     => return Some(AgentKind::Codex),
        "amp"       => return Some(AgentKind::Amp),
        "kimi"      => return Some(AgentKind::Kimi),
        "opencode"  => return Some(AgentKind::OpenCode),
        _           => {}
    }

    // Node-wrapped agents: exe is "node"; inspect argv[1] for the JS entry point.
    // Claude Code installs as: node /.../node_modules/@anthropic-ai/claude-code/cli.js
    if exe_name == "node" {
        if let Some(script) = cmdline.get(1) {
            let script_path = Path::new(script);
            if script_path.components().any(|c| c.as_os_str() == "claude-code") {
                return Some(AgentKind::ClaudeCode);
            }
        }
    }

    // stat.comm is capped at 15 chars by the kernel — only useful as secondary check.
    let stat = proc.stat().ok()?;
    match stat.comm.as_str() {
        "claude" | "claude-code" => Some(AgentKind::ClaudeCode),
        _ => Some(AgentKind::Unknown),
    }
}
```

|---|---|---|
| Claude Code (npm, current) | `npm install -g @anthropic-ai/claude-code` | `/usr/local/bin/node` (node wraps `cli.js`) |
| Claude Code (native stub) | `~/.local/bin/claude` installer | `/usr/local/bin/claude` or `~/.local/bin/claude` |
| Codex (native binary, current) | GitHub Releases — 96% Rust binary | `/usr/local/bin/codex` |
| Codex (npm, older) | `npm install -g @openai/codex` | `/usr/local/bin/node` (wraps a JS shim) |
| Amp | binary distribution | `/usr/local/bin/amp` |
| Kimi | binary distribution | `/usr/local/bin/kimi` |
| OpenCode | binary distribution | `/usr/local/bin/opencode` |

### Runtime Flow

The 1Hz Capsule ticker in <RepoFile path="crates/jackin-capsule/src/daemon.rs" /> should be the normal recomputation point. PTY output can mark the screen/process evidence dirty, but it should not directly decide `blocked`. Each tick should:

1. Refresh foreground process evidence from `/proc` for live sessions. Read `/proc/<child_pid>/stat` to find `tpgid`. If `tpgid` differs from the known agent PGID, scan `/proc/*/stat` to find the process group and identify the binary via `/proc/<pid>/exe` symlink or `/proc/<pid>/cmdline`.
2. Read the current parser screen snapshot for each pane and run the built-in visible detectors for the detected agent. Only inspect the last 24 rows of the current viewport.
3. Drop or stale any reported authority whose process has exited, sequence moved backward, source conflicts with the foreground agent, or age exceeds the source policy.
4. Run the arbitration function once per session to compute raw state, confidence, and evidence.
5. Derive effective status from raw state plus `seen`.
6. Increment the session revision and emit an event only when raw state, effective status, seen status, authority, confidence, or evidence summary changes.
7. Recompute tab/workspace/fleet roll-up from effective statuses using `blocked > done > working > idle > unknown`.

Runtime reports should take a separate path into the same state machine. A `report_agent_state` message validates the session reference, source id, sequence, agent label, and timestamp, stores the candidate authority, and requests an immediate recomputation. A `clear_agent_authority` message removes only the matching source authority and also requests recomputation. Report handlers must not update UI state directly.

Seen handling is also centralized. Focusing a pane or acknowledging a session records the current status revision as seen. If the raw state later transitions back through `working` or `blocked` and then to `idle`, the effective state becomes `done` again until the new revision is seen.

### Arbitration

All signals flow through one arbitration function: `fn arbitrate_session_status(hook: Option<&HookAuthority>, screen: &ScreenDetection, process: &ProcessEvidence, now: Instant) -> (AgentRawState, AgentStatusConfidence, AgentStatusEvidence)`.

**Full precedence order (implemented as explicit if-chain, not a match cascade):**

1. `hook_blocked_and_visible_blocker_agrees`: hook says `Blocked` for the current detected agent, valid sequence, AND `screen.visible_blocker = true` → `Blocked, Authoritative`.
2. `visible_blocker_overrides_hook`: `screen.visible_blocker = true`, screen observation is not older than hook, and hook exists but reports non-Blocked for the same agent type → `Blocked, Strong`.
3. `visible_working_overrides_hook`: `screen.visible_working = true`, hook reports `Idle` or `Blocked` for the same agent, AND (screen is fresher than hook OR hook is within 1200ms of screen) → `Working, Strong`.
4. `visible_idle_stales_hook`: `screen.visible_idle = true`, hook reports `Working` or `Blocked`, AND `STALE_HOOK_IDLE_GRACE = 2s` has elapsed since the idle signal started → clear hook authority, `Idle, Strong`.
5. `fresh_hook_authority`: hook present, process-consistent, sequence valid, not stale → hook's `raw_state` with `Authoritative`.
6. `screen_fallback`: use screen detection result with `Strong`.
7. `process_fallback`: process alive, no screen signals → `Working, Weak` (conservative: presence without screen evidence means we cannot tell).
8. `default`: `Unknown, Unknown`.

**Sequence validation** (per source_id): accept only if `incoming_seq > last_accepted_seq[source_id]`. If `source_id` has never reported before, accept the first report unconditionally. If `source_id` has reported before without a sequence, reject future sequenced reports from the same source (prevents replay). Use nanosecond timestamps as sequence values (same convention as Herdr).

**Process consistency check**: if the hook's `agent_label` does not match the foreground-detected agent binary and the process probe is not older than the hook, clear the hook authority. This prevents a stale Claude hook from holding authority when Codex is now foreground.

**Process exit clearing**: on process exit, clear any hook authority whose agent matches the exited process and whose timestamp is not newer than the process-exit event. Then treat screen detection as the only signal and transition toward `Idle` (not `Blocked`).

**Subagent counter / sticky blocked rule**: maintain a `subagent_count: u32` per session. On `SubagentStart` hook event, increment. On `SubagentStop`, decrement but do not go below zero. While `subagent_count > 0`, `PostToolUse`-equivalent events must not clear a `Blocked` state — only explicit `PermissionResult`/`UserPromptSubmit` events or a new `Blocked` report may transition out. This prevents the wezterm-agent-cards false-negative where a child `PostToolUse` clears a parent `PermissionRequest`'s blocked state.

### Done, Seen, And Roll-Up

`done` is derived outside the raw detector:

- When a pane transitions from `working` or `blocked` to raw `idle` while not focused/visible, mark `seen = false`; public status becomes `done`.
- When the operator focuses or explicitly acknowledges the pane, mark `seen = true`; public status becomes `idle`.
- Tab, workspace, and fleet roll-up should use attention priority: `blocked(4) > done(3) > working(2) > idle(1) > unknown(0)`.

The priority function in Rust: `fn attention_priority(effective: AgentEffectiveStatus) -> u8 { match effective { Blocked => 4, Done => 3, Working => 2, Idle => 1, Unknown => 0 } }`. Tab/workspace aggregate is `tabs.iter().flat_map(|t| t.panes()).map(|p| attention_priority(p.effective())).max().map(AgentEffectiveStatus::from_priority).unwrap_or(Unknown)`.

### Runtime Setup

All status assets that jackin' installs inside a role container should live under `/jackin/`: reporter binaries/scripts under `/jackin/runtime/agent-status/`, ephemeral sockets/pidfiles under `/jackin/run/`, and per-session status cache or sequence state under `/jackin/state/agent-status/`. The existing Capsule socket at `/jackin/run/jackin.sock` should remain the control-plane endpoint unless a measured throughput issue proves a separate socket is needed.

Runtime setup belongs in <RepoFile path="crates/jackin-capsule/src/runtime_setup.rs" /> and the launch scaffolding in <RepoFile path="docker/runtime/entrypoint.sh" />. The setup path should copy or generate:

- `/jackin/runtime/agent-status/report` — small shell script or compiled binary that sends `report_agent_state`, `heartbeat`, and `clear_agent_authority` messages to the Capsule socket using the JSON framing in <RepoFile path="crates/jackin-protocol/src/control.rs" />.
- `/jackin/runtime/agent-status/wrap` — optional launch wrapper that stamps `JACKIN_SESSION_ID`, `JACKIN_AGENT_RUNTIME`, `JACKIN_STATUS_SOURCE`, and the child PID/process start/exit events before `exec`ing the real runtime.
- `/jackin/runtime/agent-status/hooks/<runtime>/` — runtime-specific hook/plugin/API bridge assets written by jackin' and safe to reinstall every launch.
- `/jackin/state/agent-status/<session>/` — sequence, heartbeat, and diagnostic state owned by Capsule, not by a host cache directory.

#### Codex App-Server Bridge

**Wire protocol**: JSON-RPC 2.0 over WebSocket. Launch with `codex app-server --listen ws://127.0.0.1:<port>`.

**Handshake** (always performed on a fresh connection):
```json
→ {"jsonrpc":"2.0","id":1,"method":"initialize","params":{"clientInfo":{"name":"jackin-capsule","version":"<ver>"},"capabilities":{"experimentalApi":true}}}
← {"jsonrpc":"2.0","id":1,"result":{...}}
→ {"jsonrpc":"2.0","method":"initialized"}
```

**Primary push event** (subscribe on a second WebSocket connection):
```json
{"method":"thread/status/changed","params":{"threadId":"...","status":{"type":"active","activeFlags":["waitingOnApproval"]}}}
```

**Thread status mapping**:
- `{"type":"notLoaded"}` → `unknown` + `requires_replacement = true`
- `{"type":"idle"}` → `idle` (unless `active_turn_thread_id` is set, then `working`)
- `{"type":"systemError"}` → `unknown` + `requires_replacement = true`
- `{"type":"active","activeFlags":[]}` → `working`
- `{"type":"active","activeFlags":["waitingOnApproval"]}` → `blocked`
- `{"type":"active","activeFlags":["waitingOnUserInput"]}` → `blocked`

**Turn lifecycle**: on `turn/started` notification, set `active_turn_thread_id`. On `turn/completed`, clear it. While `active_turn_thread_id` is set, override an `idle` status to `working`.

**Error classification** (from `codexErrorInfo` on turn failure): `ContextWindowExceeded`, `UsageLimitExceeded`, `HttpConnectionFailed`, `ResponseStreamDisconnected`, `SandboxError`. Each maps to a `stuck` diagnostic reason with different recovery hints.

**Critical warning**: the app-server and the visible TUI session are separate clients. Running `codex app-server` alongside the visible `codex` TUI may not observe the same session. Do not run both by default. Use the Capsule reporter wrapper plus visible/process evidence until same-session observation is verified in Phase 3.

### Visible Screen Detectors — Exact Patterns

This section provides verified patterns for each built-in runtime. All patterns should be extracted from the **last 24 rows** of the current viewport, not from scrollback. All patterns are case-insensitive unless noted.

#### Claude Code Visual Patterns

**Idle detection** — prompt box must be visible. Claude Code renders a rounded-corner prompt box:
```
╭──────────────────────────────────────────────────────────────╮
│ > [cursor]                                                    │
╰──────────────────────────────────────────────────────────────╯
```

Two approaches to detect this (use both; if either matches, `visible_idle = true`):

1. **Border-line scan** (CCManager approach): scan from the bottom, count lines matching `/^─+$/` or `/^[╰╯]─+[╰╯]$/`. On the second border match, the content between borders is the input area. If this structure is present and no `working` or `blocked` content is in the area above it, `visible_idle = true`.
2. **Rounded corner check** (ccmux approach): last non-empty line contains at least one of `╭╮╰╯` AND has ≥2 distinct hits from the set `{╭, ╮, ╰, ╯, │, ─, >}`. Hard requirement: must contain at least one rounded corner character (rules out `tree` output and status bars that use sharp corners `┌┐└┘`).

**Agent Session Manager structural approach** (alternative, verified): find lines where `count('─') + count('━') > 20`. With two separators, the area between them is the input box; scan 15 lines above the top separator for spinner/busy area. Skip lines starting with `╭`, `╰`, `└`, `Tip:`, `Next:`, lines containing `⎿` or starting with `├`, `│`, and lines containing ` for ` (completed timing like "Churned for 1m") before matching busy/blocked patterns.

Additional idle signals: line containing `⌕ Search…` (search mode, always idle after debounce); line containing `ctrl+r to toggle` (do not change state, pass through).

**Working detection** (must be above the prompt box, in the last contiguous content block):
- Strings (case-insensitive): `esc to interrupt`, `ctrl+c to interrupt`
- Spinner pattern (verified from CCManager): regex `[✱✲✳✴✵✶✷✸✹✺✻✼✽✾✿❀❁❂❃❇❈❉❊❋✢✣✤✥✦✧✨⊛⊕⊙◉◎◍⁂⁕※⍟☼★☆·•⏺▸▹∙⋅○●] \S+ing…` — a spinner character followed by a space, a word ending in `ing`, and an ellipsis (U+2026). Examples: `✻ Tempering…`, `✳ Simplifying…`, `· Misting…`, `⏺ Coding…`.
- Token stats pattern: `/\([^)]*\d[^)]*tokens\s*\)/i` — parenthesized content with a digit and the word `tokens`. Example: `(9m 21s · ↓ 13.7k tokens)`. Explicitly does NOT match `(see tokens in docs)` — a digit is required.
- Background task line (last 3 lines): `/(\d+)\s+(?:background\s+task|local\s+agent)/` or `content.includes('(running)')`.

**Blocked detection** (must be in current visible screen, not historical):
- `has_live_blocked_form`: any line after the last `─────` rule contains ALL of: `enter to select` + `esc to cancel` + `↑/↓ to navigate` (multi-choice approval dialog).
- `do you want to proceed?` followed by yes/no choice lines.
- `waiting for permission` anywhere in visible screen.
- `tab to amend` — plan amendment prompt.
- `ctrl+e to explain` — explain prompt.
- `review your answers` — review prompt.
- Regex from CCManager: `/(?:do you want|would you like).+\n+[\s\S]*?(?:yes|❯)/` — multiline match requiring `yes` or `❯` after the question.
- String include: `esc to cancel` (covers many blocking dialogs).

**Transcript viewer skip** (suppress all state updates when visible): bottom 3 non-empty lines contain `showing detailed transcript` + `ctrl+o to toggle` + `ctrl+e to show all/collapse`.

**Working hold**: when screen shows `Working` and then transitions to `Idle`, hold `Working` for 1200ms (`CLAUDE_WORKING_HOLD`) before emitting the `Idle` transition. Apply only to screen-detected state, not to hook-reported state.

**Idle debounce**: `visible_idle = true` only after the last 30 lines of terminal content have been byte-for-byte identical for 1500ms (`IDLE_DEBOUNCE_MS`). Any change — including cursor position updates — resets the clock.

**Content isolation**: before applying working/idle patterns, isolate the "current content block" above the prompt box. Walk from the last non-empty line backward until a blank line or separator is found. Patterns from old turns are excluded. This prevents a spinner from a previous turn triggering `working` when the agent is now idle.

#### Codex Visual Patterns

**Idle detection**: `›` or `› ` prompt line present as the last non-empty line, with no block marker (`•`, `■`, `✗`, `✓`) below it.

**Working detection** (last `•` block marker before prompt):
- `• Working (` — active working state
- `Waiting for background terminal (` — waiting for subprocess
- `esc to interrupt` — active
- `/ps to view` — process list active
- `Booting MCP server:` — init in progress

**Blocked detection**:
- Strong (match as-is): `press enter to confirm or esc to cancel`, `enter to submit answer`, `allow command?`
- Weak (only if no `›` prompt present): `[y/n]`, `yes (y)`, general confirmation prompt patterns

**False-positive guard**: Do NOT match Codex installer or update output as an active agent state. The Codex binary may print update messages at startup; these look like working state but are not. After detecting `working` from screen, verify the foreground process is the Codex agent process and not the installer subprocess.

#### Amp Visual Patterns

Amp uses `esc to cancel` interrupt chrome when working, and approval/waiting headers when blocked. Current launch uses `--dangerously-allow-all`, so most tool approval prompts are bypassed. Treat auth prompts, operator questions, and runtime modal prompts as blockers; otherwise prefer `unknown` over overfitting Amp text.

Start with narrow patterns until stable fixtures are captured:
- Working: `esc to cancel` in viewport.
- Blocked: explicit question/approval prompt with an input affordance visible.
- Idle: clean prompt line at the bottom.

#### Kimi Visual Patterns

Current launch uses `--yolo` mode. Kimi renders:
- Working: braille spinner characters (`⠋⠙⠹⠸⠼⠴⠦⠧⠇⠏`) or moon-phase indicators.
- Blocked: `approve once`, `approve for session`, `reject` choice lines — tool approval prompts not bypassed by yolo mode.
- Idle: clean Kimi prompt line.

#### OpenCode Visual Patterns

OpenCode uses Bubble Tea (Go). Bubble Tea continuously redraws cursor positions and status bars even when idle — raw output silence will never be reliable as a `working` signal. Any content-silence detector must strip ANSI cosmetic escape sequences before measuring stability. Patterns:

- Blocked: `permission required` UI, question prompts with `dismiss`/`enter`/`select` affordances.
- Working: `interrupt` chrome (Bubble Tea `Ctrl+C to cancel` style).
- Idle: input box visible at bottom.

### Stuck Detection

`stuck` should be a diagnostic overlay driven by expectations, not a normal lifecycle state. Examples:

- A prompt/task was submitted, the foreground agent process is still alive, and no reliable `working`, `blocked`, or `idle` transition arrived within the runtime's timeout.
- A hook source reports `working` for too long while the visible screen and process state disagree.
- A process remains foreground but stops responding to cursor/screen probes.

Stuck events should be rate-limited, include the evidence trail, and surface as "inspect this agent" rather than "agent needs approval." This prevents false positives from training operators to ignore attention prompts.

Initial stuck constants: `STUCK_WARNING_THRESHOLD = 5 minutes`, `STUCK_RECOVERY_THRESHOLD = 10 minutes`, `MAX_STUCK_RECOVERIES = 3`. After max recoveries, emit a "manual intervention needed" diagnostic. These constants should be placed in a `stuck_detection` config struct for future runtime-specific tuning.

### Telemetry

Status authority needs durable debug evidence. Compact `clog!` lines should record lifecycle transitions, authority changes, process-exit clearing, and externally visible notifications. Verbose `cdebug!` lines should record per-tick evidence, foreground process-group reads, visible detector booleans, hook/API report acceptance/rejection, sequence validation, raw/effective state calculations, and roll-up results. When a rich TUI owns the terminal, debug output must go only to the diagnostics run file, following the existing debug-output rule.

### TUI Visual State Representation

Every agent state must be visible in the tab strip **without the operator needing to open the tab**. When multiple panes live in a workspace with five tabs open, the operator must know at a glance which tabs need attention. Visual clarity is the product: a stuck or blocked agent in a background tab should scream; an idle reviewed agent should be quiet.

The existing `TabGlyph` enum in <RepoFile path="crates/jackin-capsule/src/statusbar.rs" /> currently has three variants: `None` (space, Working/Idle), `Done` (`○`), and `Blocked` (`●` bright red). Replacing `AgentState` with `AgentEffectiveStatus` requires extending this enum for the new states. Implement the full glyph map in Phase 1 alongside the state model.

**Full glyph + color map** (one slot per tab, rightmost character of the tab cell — already reserved in the layout, so cell width stays stable):

| `AgentEffectiveStatus` | Glyph | Color | Meaning |
|---|---|---|---|
| `Working` | `◌` | Phosphor green dim (`#005012` / `PHOSPHOR_DARK`) | Active work in progress |
| `Blocked` | `●` | Bright red (`#ff4455`) | Waiting for operator — highest urgency. Existing behavior, keep. |
| `Done` | `○` | Phosphor green (`#00ff41` / `PHOSPHOR_GREEN`) | Work finished, operator hasn't seen it yet |
| `Idle` | ` ` | (space, no glyph) | Idle and reviewed or no active agent |
| `Unknown` | `·` | Mid-gray (`#606060`) | State not yet determined — not a problem, just starting up |
| `Stuck` | `!` | Amber (`#ffaa00`) | Diagnostic: expected progress, none arrived |

Rationale for color choices: `Working` gets a dim green that reads as "things happening" without urgency. `Done` gets bright green as a positive "finished, come look" signal. `Stuck` gets amber — it's not blocking the operator but does need attention. `Unknown` gets gray — neutral, not alarming, common at session start.

**Tab label roll-up for multi-pane tabs**: a tab contains multiple panes. Roll up using the same attention priority as workspace roll-up: `Blocked(4) > Done(3) > Stuck(2.5) > Working(2) > Idle(1) > Unknown(0)`. The highest-priority pane's state drives the tab glyph. `Stuck` sits between `Done` and `Working` in the tab glyph because it is diagnostic, not blocking.

**Implementation in statusbar.rs**: extend `TabGlyph` with `Working`, `Unknown`, `Stuck` variants. Update `tab_label()` to consume `AgentEffectiveStatus` instead of `AgentState`. Update `emit_tab_row0()` to emit the correct color sequence per variant. Keep the placeholder slot reserved so tab widths do not shift on state transitions.

**Color constraint**: use only the existing shared palette from `crates/jackin-tui/src/lib.rs`. Never define local color constants. The amber and dim-red values must be added to the shared palette if not already present.

## Control Protocol

The control plane should grow from one-shot `status`/`snapshot` toward state events. New message types to add to <RepoFile path="crates/jackin-protocol/src/control.rs" />:

| Surface | Purpose | Wire format |
|---|---|---|
| `session.report_agent_state` | Runtime hook/plugin/API bridge reports raw state and optional metadata for one session. | `{type:"session.report_agent_state", session_id, source_id, agent_label, raw_state, seq, ts_ns, message?}` |
| `session.heartbeat_agent_authority` | Runtime reporter confirms that its source is still alive without changing raw state. | `{type:"session.heartbeat_agent_authority", session_id, source_id, seq}` |
| `session.clear_agent_authority` | Runtime hook/plugin/API bridge releases its authority when it exits or knows it is stale. | `{type:"session.clear_agent_authority", session_id, source_id}` |
| `session.report_child_agent_state` | Runtime bridge reports descendant/subagent lifecycle so parent panes do not become done while children are still working. | `{type:"session.report_child_agent_state", parent_session_id, child_session_id, raw_state, seq}` |
| `events.subscribe` | Host/daemon/console subscribe to session lifecycle and effective state transitions. | `{type:"events.subscribe", filter?:{session_ids?, event_types?}}` |
| `wait_session_status` | CLI scripts block until a session reaches one of the target statuses without polling. Returns immediately if already in a target state. | `{type:"wait_session_status", session_id, target_statuses:["blocked","done","idle","exited"], timeout_ms?}` |
| `session.read_visible` | Debugging path to read visible/recent pane text when state looks wrong. | `{type:"session.read_visible", session_id, rows?:24}` |

**`wait_session_status` semantics**: the handler must check the current effective status before blocking. If the session is already in one of the `target_statuses`, return immediately with the current status and revision. Otherwise, subscribe to state-change events and block until a matching transition arrives or the timeout fires. This prevents callers from missing an already-satisfied status when the transition happened between the subscribe and the check.

**Event stream payload**: `{type:"agent_state_changed", session_id, raw_state, effective, seen, source, confidence, detected_agent, foreground_pgid, visible_blocker, visible_idle, visible_working, process_exited, stale_report, seq, ts_ns, revision, last_seen_revision, reason?}`. Consumers should subscribe; they should not poll rendered terminal previews.

**Reporter CLI** (`/jackin/runtime/agent-status/report`) wire format: the reporter sends a length-prefixed JSON message over a Unix socket connection to `/jackin/run/jackin.sock`. The first byte `0x00` (control channel marker, per existing framing in <RepoFile path="crates/jackin-capsule/src/socket.rs" />) precedes the 4-byte big-endian length prefix. The body is UTF-8 JSON matching one of the `session.*` message types above.

### Daemon-Ready Subscription Protocol

**Design now so the daemon never needs a protocol change.** The future jackin' daemon will connect to multiple capsule instances and aggregate state across all of them — watching for blocked/done agents, forwarding macOS native notifications, managing the task queue, and serving the Desktop Agent Hub. The control protocol must support this use case without modifications once the daemon is implemented.

**Subscription model** (`events.subscribe`): after a subscriber sends an `events.subscribe` message, the connection transitions from one-shot request/reply mode to a persistent streaming mode. The server keeps the connection open and pushes event messages as JSON frames (same 4-byte-length-prefix + UTF-8 JSON framing) whenever state changes. The subscriber can send additional one-shot queries on the same connection while subscribed — the server processes them as before but does not close the connection. The subscriber disconnects when done.

**Subscriber identification**: subscriptions carry an optional `subscriber_id: String` field. The daemon uses this to identify itself; the capsule uses it for `clog!` messages. If absent, an anonymous subscriber id is generated. Multiple concurrent subscribers are allowed (the existing concurrency semaphore already permits this).

**Event types the daemon will consume** (all must be present in Phase 1+):
- `agent_state_changed` — effective status changed, with full evidence payload
- `session_spawned` — new session created (session_id, agent, label)
- `session_exited` — session removed
- `token_usage_changed` — token totals updated
- `provider_usage_changed` — quota/window percentages updated (Phase 6)
- `workspace_status_changed` — roll-up status for the whole workspace changed

**`workspace_status_changed` event** (new, daemon-specific): `{type:"workspace_status_changed", effective: AgentEffectiveStatus, session_count: u32, blocked_count: u32, done_count: u32, working_count: u32, ts_ns}`. This lets the daemon watch one event instead of aggregating per-session events. Emitted whenever the roll-up result changes.

**Protocol versioning**: add a `jackin_protocol_version: "1"` field to the `Welcome` response (the first frame any client receives on the attach channel). The control channel's `status` reply should include `protocol_version: 1`. The daemon checks this on connect and logs a warning if the version is higher than what it knows — never crashes, but reports so the operator knows to update the daemon.

**Future daemon connection pattern** (no code needed now, just design constraint): the daemon connects to `/jackin/run/jackin.sock`, sends `{type:"status"}` to verify the capsule is alive and read the current state, then immediately sends `{type:"events.subscribe", subscriber_id:"jackin-daemon", filter:{event_types:["agent_state_changed","workspace_status_changed","session_spawned","session_exited","token_usage_changed"]}}`. It then drives macOS notifications and the Desktop Agent Hub from the event stream without polling.

### Cost Calculation

**Always prefer `costUSD` from the JSONL line when it is present** (Claude Code and Codex write it). When absent, calculate from a static pricing table. Do not make network calls to fetch pricing data at runtime.

For token-based calculation when `costUSD` is absent, the formula (per ccusage):
```
cost = (input_tokens * input_price)
     + (output_tokens * output_price)
     + (cache_creation_input_tokens * cache_create_price)
     + (cache_read_input_tokens * cache_read_price)
```
With tiered pricing above 200k tokens (when the model has an above-threshold price):
```
if tokens > 200_000:
  cost = (200_000 * base_price) + ((tokens - 200_000) * above_price)
```
For Claude `speed == "fast"` entries, multiply the total by a per-model `fast_multiplier`.

The embedded static pricing table in `crates/jackin-capsule/src/token_monitor/pricing.rs` should cover the current main models for all five providers, be clearly marked as approximate with its last-updated date, and return `None` for unknown models rather than guessing.

### Architecture: Where Token Monitoring Lives

Token data should be read inside the container and reported through the Capsule control socket, not fetched from outside. The container has access to the local JSONL/SQLite files and holds the credentials. The daemon on the host side should consume a structured `token_usage_update` event rather than accessing agent home directories itself.

The token monitor belongs in a new module `crates/jackin-capsule/src/token_monitor/mod.rs` with the following responsibilities:

1. **Session-scoped polling**: when a session spawns, start a background task that reads the provider-specific usage files for that session's agent and session ID. Poll at a configurable interval (default 30 seconds; configurable down to 10 seconds for interactive dashboards).
2. **Incremental reads**: track the last-seen byte offset for JSONL files. On each poll, read only new lines. For SQLite, track `max(rowid)` and query `WHERE rowid > last_seen`. Never re-read the entire file.
3. **Cumulative aggregation**: maintain session-level totals: `input_tokens`, `output_tokens`, `cache_read_tokens`, `cache_write_tokens`, `cost_usd` (from file or calculated), `model` (last seen), `5h_window_start` (rolling, for Claude).
4. **Cost calculation**: if `costUSD` is present in the JSONL, use it. If absent, calculate from the static pricing table. Do not make network calls to fetch pricing.
5. **Event emission**: emit `token_usage_changed` events through the Capsule event stream when totals change. Include `session_id`, `agent`, `model`, `input_tokens`, `output_tokens`, `cache_read_tokens`, `cache_write_tokens`, `cost_usd`, `window_start` (for 5-hour billing window), `ts`.

**Caching**: store the last-known token totals per session in `/jackin/state/token-monitor/<session_id>.json`. On daemon restart, load from cache so the resource panel is populated immediately rather than waiting for the next poll cycle.

**Polling frequency and back-off**: poll every 30 seconds by default. If no new data is found in 5 consecutive polls (2.5 minutes of silence), back off to 60 seconds. Resume 30-second polling when the session produces output (indicated by the existing `last_output_at` field in `Session`).

### Token Usage TUI — Status Bar Indicator and Detail Dialog

The operator must be able to see token/quota usage without leaving the capsule. Two surfaces:

**1. Compact indicator in the bottom status bar** (always visible when a session is focused):

The rightmost section of the status bar (after the branch name) shows a compact per-provider usage indicator for the currently focused session. Pricing/cost is NOT shown in this indicator (deferred to a future phase). The format is a percentage of the primary usage window remaining:

```
 claude-sonnet-4-6  main  89% (5h)
                    ^^^^  ^^^^^^^^
                    branch  token indicator
```

The percentage is the primary window's `remainingPercent` (100 − usedPercent). The window label in parentheses identifies the time window: `5h` = 5-hour rolling window (Claude), `1h` = OpenAI hourly, `wk` = weekly fallback. When no usage data is available yet, the indicator is omitted entirely (not shown as `?%` — absence means not loaded, not unknown).

When usage drops below 20%, the color switches to amber (`#ffaa00`). Below 10%, it switches to red (`#ff4455`) to match the blocked-state urgency.

**2. Detail dialog** (opened by pressing a configurable keybind — default `Ctrl+u` — when focused inside the capsule):

A centered modal dialog, using the existing `modal_block` pattern, with a scrollable content area. The dialog shows a `RateWindow` card per available window, per provider, for the currently focused session's agent.

Each card row layout:
```
 Claude Sonnet 4.6 (session)
 ████████████░░░░░░░░░  89%   Resets in 2h 14m
 
 Claude (weekly)
 ████████░░░░░░░░░░░░░  41%   Resets in 4d 12h
 
 Claude Opus (weekly)
 ███░░░░░░░░░░░░░░░░░░  16%   Resets in 4d 12h
 
 Claude Sonnet (weekly)
 ████████████░░░░░░░░░  63%   Resets in 4d 12h
```

Progress bar: ASCII block characters (`█` filled, `░` empty), 20 chars wide, rendered with the provider brand color for the filled portion and `PHOSPHOR_DARK` for the empty portion. Percentage label right-aligned after the bar. Reset time shown as a countdown ("Resets in 2h 14m") or absolute time ("Resets Mon 3:00 AM") depending on whether the time is > 24 hours away (use absolute past 24h).

**`RateWindow` data model** (mirrors CodexBar's design, documented from its source):

```rust
pub struct RateWindow {
    pub label: String,           // "Session", "Weekly", "Claude Sonnet (weekly)", etc.
    pub used_percent: f64,       // 0–100
    pub window_minutes: Option<u32>,  // Duration: 300 = 5h, 10080 = 1 week
    pub resets_at: Option<SystemTime>, // Next reset timestamp
    pub reset_description: Option<String>, // Custom text e.g. "Resets Monday 3:00 AM"
}

pub struct ProviderUsageSnapshot {
    pub provider: String,        // "claude", "codex", etc.
    pub model: Option<String>,   // Last used model
    pub primary: Option<RateWindow>,    // Session-level quota
    pub secondary: Option<RateWindow>,  // Weekly quota
    pub tertiary: Option<RateWindow>,   // Monthly / model-specific quota
    pub extra_windows: Vec<RateWindow>, // Per-model breakdowns (Claude Opus/Sonnet separate)
    pub fetched_at: SystemTime,
}
```

**Reset countdown format**:
- `< 1 min`: "Resets in < 1m"
- `< 1 hour`: "Resets in Xm"
- `< 24 hours`: "Resets in Xh Ym"
- `< 7 days`: "Resets in Xd Yh"
- `>= 7 days`: absolute "Resets Mon 3:00 AM"
- Past reset (stale data): "Reset at [time]" in gray

| Provider | Primary | Secondary | Extra |
|---|---|---|---|
| Claude | 5-hour session (`five_hour` from OAuth API) | Weekly (`seven_day`) | `seven_day_opus` and `seven_day_sonnet` separately if both present |
| Codex/OpenAI | 1-hour | Monthly | — |
| Kimi | Session | Weekly | — |
| OpenCode/Z.ai | Session (token) | Daily (time) | Session session-token limit |
| Amp | Session | Weekly | — |

`↑/↓ scroll · Ctrl+u close`

## Acceptance Criteria

This roadmap item is implementation-ready when an agent can build from these requirements without needing to re-research the product shape. The first PR that claims a meaningful implementation slice should satisfy these criteria:

- A quiet live session no longer becomes `blocked` solely because no bytes arrived for a few seconds.
- `AgentState` or its replacement has an explicit `Unknown` state in the protocol, and every current status/snapshot path can carry it.
- `done` is derived from raw idle plus seen/unseen revision state; no reporter or detector can directly report public `done`.
- `jackin-capsule` has one status authority path that owns raw state, effective status, evidence, revision, and roll-up. Console, daemon, host commands, and future desktop surfaces consume that path instead of parsing terminal text.
- The container image/setup path installs the status reporter under `/jackin/runtime/agent-status/`, launches built-in runtimes with stable `JACKIN_*` status environment, and has tests proving reporter events flow through the Capsule socket.
- Reported status authority is accepted only with valid session id, source id, monotonic sequence, freshness, and process/runtime consistency; stale, conflicting, or exited reports are rejected or cleared with diagnostic evidence.
- Built-in runtime detection starts with Claude Code, Codex, and OpenCode, with Amp and Kimi represented as supported runtimes that produce `unknown` or weak fallback state until their stable integration surfaces are verified.
- Every built-in visible detector has sanitized fixture coverage for at least one working, blocked, idle, and false-positive case before it can drive notifications.
- Runtime hooks/API reporters are installed only inside the container or derived image state under `/jackin/` and container-local agent homes. The implementation does not write host agent config, host shell files, host terminal config, host git config, or user repositories.
- Runtime hook/plugin setup does not write into bind-mounted project directories by default; any integration that requires project-tree files is explicit and surfaced to the operator.
- The status event stream includes enough evidence to debug a wrong state from diagnostics: detected agent, authority source, confidence, foreground process group, visible detector booleans, sequence/timestamp, and reason for stale/rejected reports.
- Workspace/tab roll-up uses `blocked(4) > done(3) > working(2) > idle(1) > unknown(0)`, and this priority is covered by tests.
- Wait semantics work when the session is already in the requested state; callers must not miss an already-satisfied status.
- The Claude Code hook entry for `Stop` is registered as `async: false`, checks `background_tasks` in the hook input, and blocks the stop (returning `{"decision":"block"}`) when background tasks are running.
- The subagent counter prevents `PostToolUse` events from clearing a `Blocked` state set by a `PermissionRequest` while subagents are active.
- Per-session token totals are read from provider-specific local files inside the container, emitted through the Capsule event stream, and included in `SessionInfo` responses.
- A `ModelCatalog` is populated at session start by querying each provider's model listing API; it falls back to an embedded list when the API is unreachable. The `token.get_models` control message returns the cached catalog. The catalog drives the model availability display in the console.
- The process identity module uses the `procfs` crate, correctly reads `tpgid` from field 8 of `/proc/<pid>/stat`, handles the Node.js wrapper case for Claude Code by inspecting `cmdline[1]` for the `@anthropic-ai/claude-code` path, and distinguishes native Codex (exe = `codex`) from npm-wrapped Codex (node wrapping a JS shim).

## Scope

**In scope**:

- One canonical state authority inside `jackin-capsule`.
- Runtime-neutral public status vocabulary (`unknown`, `working`, `blocked`, `done`, `idle`).
- Per-session raw state, public derived status, `seen`, evidence metadata, and revision.
- Container-local reporter assets under `/jackin/runtime/agent-status/` and reporter state under `/jackin/state/agent-status/`.
- Process detection through Linux `/proc` using the `procfs` crate (Linux-first; containers are always Linux).
- Screen heuristics for built-in runtimes with fixture test coverage.
- Container-local hook/plugin/API reporters for supported runtimes where each runtime exposes a stable surface.
- Shell integration (OSC 133 + OSC 7) injected into `/home/agent/.zshrc` for shell-command boundary detection.
- Event subscription and wait semantics over the control socket.
- Token monitoring: local JSONL/SQLite reads per provider, session-level totals, emission through the event stream, display in `SessionInfo`.
- Tests for arbitration, sequence rejection, stale hook clearing, process exit, `done`/`idle` acknowledgement, roll-up priority, false-positive suppression, token parsing, and incremental reads.

**Out of scope for the first implementation**:

- Host-side hook installation without an explicit operator action.
- API/network interception of LLM provider traffic.
- Runtime-specific tool-level status for every command/tool.
- Windows ConPTY support.
- Autonomous recovery actions after stuck detection. Phase 1 only reports and routes attention.
- Support for future agents such as Gemini beyond the custom-agent fallback contract.
- Anthropic Admin API for organizational Claude usage (requires org-level `sk-ant-admin` key; individual Pro/Max users cannot use it).
- Anthropic OAuth API usage polling from inside the container (requires an OAuth access token; only available when explicitly passed as `ANTHROPIC_OAUTH_TOKEN`).
- OpenAI billing API for Codex credit balance (requires session bearer token; not stable for programmatic use).
- Real-time token streaming (the monitor reads local files after-the-fact, not during generation).

## Do Not Do

- Do not copy Herdr source or wire format.
- Do not silently edit host agent homes, shell rc files, terminal settings, git config, or user repositories.
- Do not create container paths owned by jackin' outside `/jackin/`.
- Do not silently write status hooks/plugins into bind-mounted project directories.
- Do not treat silence as `blocked`.
- Do not let every UI surface parse terminal text independently.
- Do not add broad regex libraries of historical output; visible-screen heuristics must be narrow and fixture-backed.
- Do not let cooperative state files or hooks prove health without process/session validation and stale handling.
- Do not run `codex app-server` alongside the visible TUI session until same-session observation is verified.
- Do not write OpenCode plugins into `.opencode/plugins/` by default; that is a bind-mounted project directory.
- Do not register Claude Code hooks with `async: false` unless the hook needs to block Claude (only `Stop`, `PermissionRequest`); unnecessary synchronous hooks add latency.
- Do not register `SubagentStop` hook events as a source of `idle` state; they must only update the subagent counter.
- Do not trust the `StopFailure` event as an `idle` signal; it maps to `working` with a diagnostic marker.
- Do not make network calls for pricing data; use the embedded static pricing table.
- Do not fetch token totals from host-side agent homes via `docker exec`; read from inside the container.
- Do not rely on `CLAUDE_CONFIG_DIR` to redirect Claude Code settings; it is unsupported. Use `/home/agent/.claude/settings.json` directly.
- Do not scan arbitrary scrollback for working/blocked patterns; scan only the last 24 rows of the current viewport. For Claude Code, use the structural separator approach (two `─`/`━` border lines) to isolate the current input region before applying patterns, and skip lines containing ` for ` (completed timing), `⎿`, or starting with `├`, `│`, `Tip:`, `Next:`.
- Do not use `stat.comm` as the primary agent identification method; it is capped at 15 characters by the kernel. Use `Process::exe()` and `Process::cmdline()` as primary sources.

## Future Decisions

- **Role-authored visible detectors.** A future role manifest contract may let role authors declare small named visible states for custom agents. That should come after the built-in status authority exists.
- **Runtime-specific stuck policy.** Initial stuck timeouts can be conservative constants; later work can make them role-specific or task-class-specific.
- **Amp/Kimi semantic integrations.** Add structured reporters only after validating stable hook/plugin/API surfaces in the installed runtimes. Amp Neo (May 2026) has a production plugin API; Kimi has semantic hook events. Both are candidates for Phase 3 but require runtime version detection before installing.
- **Health and keep-awake consumers.** Sleep inhibition, health scores, circuit breakers, and escalation policies should consume the authority after it ships.
- **Stop-hook as coordination bus.** A pattern where the `Stop` hook long-polls a coordination channel (up to 30 seconds) to receive messages from other agents before Claude truly stops is viable over the Capsule socket. This enables interrupt-driven inter-agent messaging without polling. Belongs in a future autonomous-task-queue feature that consumes the status authority.
- **OpenCode ACP stability.** The `opencode acp` stdio JSON-RPC path avoids port binding and SQLite concurrency issues. Verify its stability and event schema before committing to it as the primary OpenCode integration path in Phase 3.
- **Anthropic Admin API for org-level usage.** For organizational deployments with Admin API keys, the usage report (`/v1/organizations/usage_report/messages`) and cost report (`/v1/organizations/cost_report`) endpoints provide richer token data than local JSONL parsing. Add this as an opt-in overlay when `ANTHROPIC_ADMIN_API_KEY` is set in the container environment.
- **Anthropic OAuth usage windows.** For individual Pro/Max users who provide an OAuth token, the `GET https://api.anthropic.com/api/oauth/usage` endpoint returns 5-hour and 7-day utilization percentages with reset times. This enables a real-time usage bar in the resource panel. Add as an opt-in path when `ANTHROPIC_OAUTH_TOKEN` is set.
- **Token budget alerts.** Once token totals are flowing through the event stream, a future alert policy can notify the operator when a session approaches a configurable token or cost threshold.
- **Tiered model pricing support.** The embedded static pricing table should eventually support per-model fast-mode multipliers (`speed == "fast"` entries in Claude JSONL). Defer until the pricing table is otherwise complete.
