**Status**: Open — design proposal (Codebase health; complements the two-tier `clog!` / `cdebug!` telemetry contract in <RepoFile path="AGENTS.md" />)

## Problem

jackin' has three hand-rolled observability surfaces that solve the same problem three different ways:

| Surface | Where | Shape | Sink |
|---|---|---|---|
| `eprintln!` | host CLI (88 call sites) | unstructured, untimestamped strings | stderr only |
| `clog!` / `cdebug!` | <RepoFile path="crates/jackin-capsule/src/logging.rs" /> (9 capsule files) | tagged + ISO-8601-stamped strings, debug gated on `JACKIN_DEBUG` | stderr **and** `/jackin/state/multiplexer.log` |
| `RunDiagnostics` | <RepoFile path="src/diagnostics.rs" /> (~14 host files) | JSON-per-line events with `trace_id` / `span_id` / `kind` / `stage` / `detail` | `~/.jackin/data/diagnostics/runs/<run-id>.jsonl` |

The third surface is the tell. <RepoFile path="src/diagnostics.rs" /> already emits this per event:

```rust
struct JsonEvent<'a> {
    ts_ms: u128,
    run_id: &'a str,
    trace_id: &'a str,          // always == run_id today
    span_id: Option<&'a str>,   // always None today
    kind: &'a str,              // "stage_started", "auth", "git_pull", …
    message: &'a str,
    stage: Option<&'a str>,     // "identity" | "role" | "credentials" | …
    detail: Option<&'a str>,
}
```

This is a span tree drawn freehand. `trace_id` is hard-wired equal to `run_id`; `span_id` is *defined in the struct but never set* (the `stage()` method's own comment says "`span_id` is left unset"); `stage` and `detail` are weakly-typed strings that callers in <RepoFile path="src/runtime/progress.rs" /> fill from a `LaunchStage` enum (`identity`, `role`, `credentials`, `construct`, `agent binaries`, `derived image`, `workspace`, `network`, `sidecar`, `capsule`, `hardline`) by calling `.label()` and stringifying. The launch pipeline is exactly the concurrent, multi-stage workload that span correlation exists for — pull → construct → agent-binary fetch → derived-image build → DinD sidecar spawn → capsule start can overlap, and today the only way to know which event belongs to which stage is to string-match the `stage` field.

The cost of three surfaces:

- **Plumbing duplication.** The compact-vs-verbose split (`clog!` always on, `cdebug!` gated) is the exact behaviour `tracing` gives for free via levels + an `EnvFilter`. The capsule hand-rolls ISO-8601 stamping, a `OnceLock<Mutex<Option<File>>>` sink, a panic hook, and a debug-enabled `AtomicBool`. <RepoFile path="src/diagnostics.rs" /> hand-rolls JSON serialization, a `BufWriter` behind a `Mutex`, file rotation/pruning, and the run-scoped `ACTIVE_RUN` thread-local-ish global.
- **No real correlation.** Because `span_id` is never set and `trace_id` is just the run id, you cannot ask "show me every event under the derived-image build" without grepping the `stage` string — and nested work (a `docker build` inside the derived-image stage that itself shells out) has no parent/child link at all.
- **Three formats to learn.** An operator pasting a bug report might paste stderr (`eprintln!`), the capsule log file (`clog!`), or a diagnostics run id (`RunDiagnostics`). They carry overlapping but differently-shaped information, and only one is machine-parseable.
- **No metrics.** Stage durations, build cache hit/miss, bytes pulled, agent-binary cache reuse — all reconstructable from the JSONL timestamps today, none queryable. There is no counter/histogram surface at all.

The `tracing` ecosystem collapses all three into one instrumentation API with multiple **output layers**: a compact ANSI layer reproducing today's `clog!` behaviour, a JSON file layer reproducing today's `RunDiagnostics` JSONL, and an OpenTelemetry layer (via `tracing-opentelemetry`) for operators who want to ship spans/metrics to a collector. Spans carry real parent/child structure and typed fields (`instance_id`, `role`, `agent`, `stage`) instead of stringly-typed `stage` / `detail`.

The design commits to **both** ends of the stack, kept at distinct layers so each carries its own cost only when used:

- **`tracing` is the instrumentation front, mandatory and always present.** Every call site uses `info!` / `debug!` / `#[instrument]` — vendor-neutral, the de-facto Rust standard, ubiquitous across the tokio ecosystem. Code never calls a `tracing-subscriber` sink or an `opentelemetry` exporter directly; it only emits `tracing` events and spans.
- **`tracing-opentelemetry` is the committed OpenTelemetry bridge.** It is the chosen `Layer` that translates `tracing` spans into OpenTelemetry spans — not an alternative to evaluate later. The deliberate distinction is between the *bridge concept* (committed: this is how jackin' speaks OTel) and the *exporter + transport dependency tree* (`opentelemetry-otlp` + `tonic`/`prost`), which is feature-gated so a default `jackin` build pays nothing for it. "We will have both" means `tracing` everywhere and `tracing-opentelemetry` as the one sanctioned OTel path — enabled behind a Cargo feature, off by default, on when an operator runs a collector.

This is not an either/or between a "tracing crate" and an "OpenTelemetry crate": they sit at different layers. `tracing` is the API you instrument against; OpenTelemetry is a wire protocol / backend you export to. Writing raw `opentelemetry` calls through the codebase would couple every call site to the wire format — the bridge exists precisely so jackin' instruments once with `tracing` and chooses the OTel backend at the subscriber layer.

## Why this is worth doing

This is **not** a "swap the logging crate" cosmetic change. The leverage is that one instrumentation call site feeds every sink, and the sinks become composable:

- **The two-tier contract becomes a one-line filter.** <RepoFile path="AGENTS.md" />'s hard rule — compact always-on, verbose gated on `JACKIN_DEBUG` — maps directly onto `tracing` levels: `clog!` sites become `info!`/`warn!`/`error!`, `cdebug!` sites become `debug!`/`trace!`, and the gate becomes `EnvFilter::new(if debug { "debug" } else { "info" })`. The "skip the format + write entirely when the flag is off" guarantee the rule demands is exactly what `tracing`'s level check already does (the macro short-circuits before evaluating its fields). No behaviour change for operators; far less plumbing.
- **`span_id` stops being a lie.** A `#[instrument]` on each launch stage gives every event under it a real parent span id, automatically. The diagnostics JSONL gains true correlation with zero extra caller code — the fields that are hand-threaded today (`stage`, `detail`) become span fields recorded once at span entry.
- **Concurrent stages become legible.** When construct, agent-binary fetch, and DinD sidecar spawn run concurrently, their events interleave in the JSONL. With span ids the consumer (or a future TUI timeline) can demultiplex them. This directly serves the [launch progress TUI](/reference/roadmap/launch-progress-tui/) and the [agent runtime status authority](/reference/roadmap/agent-runtime-status/) work, both of which want per-stage timing.
- **Metrics arrive for free-ish.** `tracing` span timings + a metrics layer (or `tracing-opentelemetry`'s metrics bridge) give stage-duration histograms, cache hit/miss counters, and bytes-pulled gauges without a second instrumentation pass.
- **The OpenTelemetry path is opt-in at two levels, never mandatory.** The local-first JSONL stays the default. The `tracing-opentelemetry` layer is gated by a compile-time Cargo feature (`otlp`) and a runtime env var: a default build has no `opentelemetry` dependency at all, and even a feature-enabled build only exports when the operator points it at a collector. jackin' never *requires* a network sink.

## Best practices this should follow

Grounded in the current `tracing` / `tracing-subscriber` / `tracing-opentelemetry` guidance:

1. **One `Registry`, many `Layer`s.** `tracing_subscriber::registry()` is the span store; each output is a `Layer` composed onto it. The canonical shape is `registry().with(fmt_layer).with(json_layer).with(otlp_layer)`. This is the single mechanism that replaces all three current surfaces.
2. **Per-layer filtering, not one global filter.** Each layer gets its own filter via `.with_filter(...)` so the compact ANSI layer can run at `info` while the JSON file layer captures `debug`/`trace` — the firehose lands in the file, the terminal stays quiet, both from the same events. Use `EnvFilter` (honours `JACKIN_DEBUG` → `RUST_LOG`-style directives) for the dynamic surface and the lighter-weight `Targets` filter where dynamic field-level filtering is not needed.
3. **Optional layers without changing the subscriber type.** Wrap a layer in `Option<Layer>` (a `None` layer is a no-op) or `Box<dyn Layer<S> + Send + Sync>` (`.boxed()`) so the OTLP layer can be absent on a default build and present under the feature flag without the registry type diverging across `cfg`s.
4. **A custom `Layer` for the JSONL, not the stock JSON formatter.** The existing `~/.jackin/data/diagnostics/runs/<run-id>.jsonl` schema (`ts_ms`, `run_id`, `trace_id`, `span_id`, `kind`, `message`, `stage`, `detail`) is operator- and agent-facing — the `--debug` walkthrough rule in <RepoFile path="AGENTS.md" /> tells operators to share a run id and tells agents to read that file. A bespoke `Layer` implementing `on_new_span` / `on_event` (saving span fields into `span.extensions()` so they are visible at event time) preserves that exact schema while sourcing it from real spans. Do **not** let the stock `fmt().json()` formatter dictate a new field shape — that would break every existing reader and fixture.
5. **Non-blocking file writes.** Use `tracing-appender`'s non-blocking writer for the file layers (capsule log + diagnostics JSONL) so the draw thread / launch path never blocks on disk — matching the current "logging must never block startup" comment in <RepoFile path="crates/jackin-capsule/src/logging.rs" />. Keep the returned `WorkerGuard` alive for the process lifetime (drop = flush). Do **not** wrap stdout/stderr in the non-blocking writer — that path is effectively free and the extra thread is pure overhead.
6. **`#[instrument]` on the stage boundaries, not everywhere.** Annotate the launch-stage functions and the capsule session lifecycle with `#[instrument(skip(...), fields(instance_id, role, agent, stage))]`. Skip large/non-`Debug` args explicitly. Per-byte / per-frame capsule traces stay as `trace!` events inside the existing hot loops — `#[instrument]` on a per-byte function would be a performance regression.
7. **OTLP via the batch processor, protobuf, opt-in.** If the OTLP layer ships, use `opentelemetry-otlp` with the `BatchSpanProcessor` (not the simple processor) and protobuf-over-HTTP by default (fewer moving parts than gRPC-tonic). Gate the whole dependency tree behind a Cargo feature so the default build does not pull `opentelemetry` and its transitive `tonic`/`prost` tree.
8. **Initialize once, capture the debug flag once.** Mirror today's `logging::init()` / `RunDiagnostics::start` lifetime: a single `observability::init(debug, paths, command)` that builds the registry, opens the run file, installs the panic hook (via `tracing-panic` or the existing hand-rolled hook), and returns the `WorkerGuard`(s). The `JACKIN_DEBUG` flag flows into the container via `env_passthrough` exactly as it does now.

## Target shape

```rust
// src/observability/mod.rs (new) — host side
pub struct Observability {
    _guards: Vec<tracing_appender::non_blocking::WorkerGuard>,
    run: Arc<RunDiagnostics>,   // retained: still owns run-id minting + pruning
}

pub fn init(debug: bool, paths: &JackinPaths, command: &str) -> anyhow::Result<Observability> {
    let level = if debug { "debug" } else { "info" };

    // Layer 1 — compact ANSI to stderr (today's clog!/eprintln! surface).
    let (stderr_nb, g1) = tracing_appender::non_blocking(std::io::stderr());
    let fmt_layer = tracing_subscriber::fmt::layer()
        .with_writer(stderr_nb)
        .compact()
        .with_filter(EnvFilter::new(level));

    // Layer 2 — the existing JSONL schema, sourced from real spans.
    let run = RunDiagnostics::start(paths, debug, command)?;
    let json_layer = JackinDiagnosticsLayer::new(run.clone())
        .with_filter(EnvFilter::new("trace")); // file always gets the firehose

    // Layer 3 — OpenTelemetry via tracing-opentelemetry, gated twice:
    // compiled only under `--features otlp`, attached only when the
    // endpoint env var is set. None on a default build.
    let otlp_layer = build_otlp_layer()?; // Option<BoxedLayer>

    tracing_subscriber::registry()
        .with(fmt_layer)
        .with(json_layer)
        .with(otlp_layer)
        .init();

    Ok(Observability { _guards: vec![g1], run })
}
```

```rust
struct JackinDiagnosticsLayer { run: Arc<RunDiagnostics> }

impl<S> tracing_subscriber::Layer<S> for JackinDiagnosticsLayer
where S: Subscriber + for<'a> LookupSpan<'a> {
    fn on_new_span(&self, attrs: &Attributes<'_>, id: &Id, ctx: Context<'_, S>) {
        // Record `stage`, `instance_id`, `role`, `agent` into span.extensions()
        // so they're available when on_event fires under this span.
    }
    fn on_event(&self, event: &Event<'_>, ctx: Context<'_, S>) {
        // Build the same JsonEvent { ts_ms, run_id, trace_id, span_id, kind,
        // message, stage, detail } from the event + its parent span, then hand
        // it to self.run for the BufWriter+Mutex append that already exists.
    }

```

Call sites change from this:

```rust
self.diagnostics.stage("stage_started", stage.label(), &detail, None);
```

to this:

```rust
// inside #[instrument(fields(stage = %stage.label(), instance_id, role, agent))]
info!(detail = %detail, "stage started");
```

— and the JSONL row that lands in `~/.jackin/data/diagnostics/runs/<run-id>.jsonl` is byte-compatible with today's, now with a real `span_id`.

The capsule (<RepoFile path="crates/jackin-capsule/src/logging.rs" />) gets the same treatment: `clog!` → `info!`, `cdebug!` → `debug!`, the file sink becomes a `tracing-appender` non-blocking writer pointed at `/jackin/state/multiplexer.log`, and the `[jackin-capsule]` / `[jackin-capsule debug]` prefixes become a static `target` field so existing log readers still grep cleanly.

## Migration plan

Land incrementally; the tree stays green and operator-facing JSONL stays byte-stable at every step.

1. **PR 1 — Registry + layers behind the existing API, zero call-site churn.** Add `tracing`, `tracing-subscriber`, `tracing-appender` to the host <RepoFile path="Cargo.toml" />. Build `observability::init` with the fmt layer + the custom `JackinDiagnosticsLayer`. Re-implement `RunDiagnostics::compact` / `stage` / `debug` as thin shims that emit `tracing` events (so existing callers keep compiling) while the layer produces the identical JSONL. Lock the schema with a golden-file test: capture a known launch's JSONL before and after and assert byte-equality of every field except the new `span_id`.
2. **PR 2 — `#[instrument]` the launch stages.** Annotate the `LaunchStage` functions in <RepoFile path="src/runtime/launch.rs" /> and <RepoFile path="src/runtime/progress.rs" /> so `span_id` becomes real and the weakly-typed `stage`/`detail` strings become span fields. Retire the `RunDiagnostics::stage` shim once its callers are spans. Concurrent-stage correlation lands here.
3. **PR 3 — Migrate the capsule.** Replace `clog!`/`cdebug!` bodies with `tracing` macros and a `tracing-appender` file layer in <RepoFile path="crates/jackin-capsule/src/logging.rs" />, keeping the `[jackin-capsule]` prefix as a `target`. Per-byte/per-frame `cdebug!` sites become `trace!`. The `jackin-capsule` smoke-test mandate in <RepoFile path=".github/AGENTS.md" /> applies — the eval one-shot build must pass.
4. **PR 4 — Sweep the 88 `eprintln!` call sites.** Convert operator-facing error/lifecycle `eprintln!`s to leveled `tracing` events; leave genuine stdout *output* (command results the operator reads, not telemetry) as `println!`. This is a judgement sweep, not a mechanical one — distinguish "telemetry" from "program output."
5. **PR 5 — Metrics + the OpenTelemetry layer (deferred / separate).** Add stage-duration histograms and cache hit/miss counters, and the `tracing-opentelemetry` bridge layer wired to `opentelemetry` + `opentelemetry-otlp` (protobuf/HTTP, batch processor). The OTel path is toggled at **two levels**: a compile-time Cargo feature (`otlp`) that decides whether the dependency tree and the layer are built at all, and a runtime env var (e.g. `JACKIN_OTLP_ENDPOINT`) that decides whether the layer is actually attached and exporting when the feature is compiled in. Default build: feature off, no `opentelemetry` / `tonic` / `prost` in the tree, no behaviour change. Operator with a collector: build with `--features otlp`, set the endpoint env var, get spans/metrics over OTLP. This is the only PR that adds a heavy dependency tree, and it is explicitly opt-in at both the build and run boundaries.

PRs 1–4 add only `tracing` + `tracing-subscriber` + `tracing-appender` (small, ubiquitous, the canonical choice — satisfies the "prefer libraries / prefer popular canonical crates" rule in <RepoFile path="AGENTS.md" />). PR 5's OTLP tree stays behind a feature flag.

## Non-goals

- **Changing the operator-facing JSONL schema.** The `~/.jackin/data/diagnostics/runs/<run-id>.jsonl` field shape is a contract the `--debug` triage workflow depends on. This item *sources* it from spans; it does not rename fields. A future schema change is a separate, explicitly-versioned decision.
- **Requiring a network collector.** Local-first JSONL is the default forever. OTLP is opt-in behind a feature flag and an env var; a default `jackin` build has no `opentelemetry` dependency.
- **Replacing the two-tier telemetry *contract*.** The compact-vs-verbose rule in <RepoFile path="AGENTS.md" /> stays. This item changes the *implementation* (levels + filters instead of two macros) while preserving the operator-visible behaviour: quiet by default, firehose under `--debug`.
- **Distributed tracing across host ↔ container as a hard requirement.** Propagating a trace context from the host launch into the capsule is attractive (one trace spanning both processes) but is a stretch goal noted under Open Questions, not a PR-1 requirement.
- **A `tracing`-based replacement for the launch-progress TUI view model.** The TUI reads launch state from its own view struct; this item feeds the same stage transitions but does not rewrite the renderer (that is [launch progress TUI](/reference/roadmap/launch-progress-tui/) work).

## Open questions

- **Host ↔ container trace propagation.** Should the host's launch trace id flow into the capsule (via an env var carrying a W3C `traceparent`) so a single trace spans both processes? Attractive for "one timeline per launch," but couples the two crates' span models. Recommendation: ship per-process traces first (run id == trace id, as today), evaluate cross-process propagation after PR 3.
- **Keep `RunDiagnostics` as the file owner, or fold it into the layer?** PR 1 keeps `RunDiagnostics` owning run-id minting, file creation, and pruning (battle-tested, with `restrict_to_owner` permissions and age/count-based retention) and makes the layer a thin writer into it. Folding pruning into the layer is possible but risks losing the existing retention logic; recommendation is to keep the split.
- **`Targets` vs `EnvFilter` for the default surface.** `EnvFilter` honours `RUST_LOG`-style dynamic directives (useful: operators could narrow to one module) but is heavier. `Targets` is lighter but static. Recommendation: `EnvFilter` driven off `JACKIN_DEBUG` for the compact layer, `Targets` (or a constant `LevelFilter::TRACE`) for the always-firehose file layer.
- **Metrics backend when OTLP is off.** With no collector, where do counters/histograms go? Options: a periodic summary line in the JSONL at run end, or nowhere (spans-only). Recommendation: emit a compact end-of-run summary event (stage durations, cache hits) into the existing JSONL so metrics are useful without a collector.
- **`tracing-opentelemetry` version cadence.** The `opentelemetry` crate family versions in lockstep and bumps often (0.2x series at time of writing); pinning the OTLP layer behind a feature flag isolates the default build from that churn. Confirm the exact compatible `tracing-opentelemetry` ↔ `opentelemetry` ↔ `opentelemetry-otlp` version triple at PR-5 time.
