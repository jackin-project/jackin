## TL;DR

`jackin-capsule` used to depend on a **single-author git fork of `vt100` pinned at a SHA** for its terminal model. Two independent pressures made us own this layer instead: (1) **upstream `vt100` had effectively stalled** ŌĆö last code push July 2025, ten open PRs, and the exact patch our fork carried sitting unmerged as PR #31 ŌĆö so the fork was not temporary; and (2) **the fast off-the-shelf engines are built for GPU rendering, not for the re-emit-to-a-host-terminal hot path the capsule runs**, so "just switch to Alacritty" did not apply. The production answer is now `jackin-term`: `vte` parses bytes, `DamageGrid` owns terminal state and damage, `PaneBodyWidget` renders snapshots into Ratatui, and `SocketBackend` emits the attach stream. The remaining work on this page is verification/performance closure, not replacing an active `vt100` dependency.

## Hard constraint: pure Rust, no foreign bindings

This decision has exactly **two** possible outcomes: **adopt a maintained, public Rust crate**, or **build our own Rust crate**. There is no third option. jackin' will not take an FFI dependency on a C or Zig terminal library ŌĆö not libvterm, not `libghostty-vt`, not any `-sys` wrapper. The entire toolchain stays in Rust with no non-Rust build dependency in the container.

Non-Rust emulators (libvterm, Ghostty) appear in this document **only as idea sources**. If Ghostty's memory model is the best in the field, the move is to **re-implement that design in Rust inside `jackin-term`**, with attribution to the MIT-licensed original ŌĆö never to link the Zig library. Wherever this page says "bind," read it as "ruled out by this constraint; kept only to show the option was considered and rejected on principle, not overlooked."

## Where this idea came from

### Why the first rewrite used `vt100`

`jackin-capsule` is an in-container PTY multiplexer. It runs as PID 1 inside a role container, spawns one PTY per agent session, and presents the focused session to whatever host terminal the operator attached. To do that it needs a **terminal model**: something that consumes the raw bytes an agent program writes to its PTY (escape sequences, UTF-8, cursor moves, colors, scroll regions) and maintains an in-memory picture of what the screen *should* look like ŌĆö so the multiplexer can redraw that picture on demand, swap between panes, restore state on focus changes, and forward OSC sequences.

`vt100` (by Jesse Luehrs, `doy`) was the natural first pick, for reasons that remain important historical context:

- It is the **canonical small "terminal parser as a library"** crate on crates.io ŌĆö exactly "the parser component of a terminal emulator pulled out into a separate crate."
- Its model is shaped for **re-emission**, not pixels. The signature API is `Screen::contents_diff(prev)` / `contents_formatted()`: given two screen states it emits the **minimal escape byte sequence** that turns one into the other. That is precisely what a tmux/Zellij-style multiplexer needs, and it is unusual ŌĆö most terminal libraries model a grid for a GPU to draw, not a grid to *re-serialize back into escape sequences*.
- It ships a `Callbacks` trait (`set_window_title`, `set_window_icon_name`, `copy_to_clipboard`, `unhandled_csi`, ŌĆ”) that lets us intercept OSC and unhandled-CSI traffic for passthrough.

That old path was built directly on `vt100`'s `Screen` + `Callbacks` shape. The current path is `vte ŌåÆ DamageGrid ŌåÆ GridSnapshot/GridPatch ŌåÆ PaneBodyWidget ŌåÆ SocketBackend`; `OscCapture`, the `vt100` feature flag, and the dependency itself are gone.

### The problem with `vt100` and our current implementation

Two distinct problems, which matter for different reasons.

**Problem 1 ŌĆö we are not on canonical `vt100`; we are on a forked SHA, and upstream has stalled.**

Our dependency is:

```toml
# crates/jackin-capsule/Cargo.toml
vt100 = { git = "https://github.com/donbeave/vt100-rust", rev = "527f0715..." }
```

- The four callbacks the TODO named **already shipped** in canonical [`vt100` 0.16.2](https://docs.rs/vt100/latest/vt100/trait.Callbacks.html) (January 2025). They are not what the fork adds.
- The fork's **real** delta over upstream is a single additive method, `Screen::clear_scrollback()`, used by `clear_scrollback_and_request_screen_clear` in <RepoFile path="crates/jackin-capsule/src/session.rs" /> and the focused-pane handler in <RepoFile path="crates/jackin-capsule/src/daemon.rs" />.
- **That exact patch is already submitted upstream as `doy/vt100-rust` PR #31 ("Support clearing scrollback", May 22 2026) ŌĆö and it is not merged.**

This is the realization that reframed the whole question: **the fork is not a temporary bridge to an upstream that will catch up.** Look at the merge history, not just the commit date ŌĆö it is the sharper signal. The maintainer's last merge of any kind was a CI-only change in July 2025, and the last *substantive* merge before that was in **April 2023**. Meanwhile a wave of **eight community pull requests landed between December 2025 and May 2026 ŌĆö and not one was merged**, including our own `clear_scrollback` as PR #31 and outright bug fixes (#29 1├Ś1 scroll, #30 a panic on truncated wide characters). The oldest open PR has waited since January 2021. The pattern is unambiguous: **the author has been effectively absent for years**, the crate is in maintenance limbo, and any feature or fix the capsule needs that upstream lacks means **we carry the patch ourselves, indefinitely.** We are *already* maintaining a fork. The only question the fork raises is whether to keep it as an ad-hoc patch on someone else's abandoned-in-practice crate, or to own the layer deliberately.

**Problem 2 ŌĆö `vt100`'s model is correct in shape but not optimized for our hot path.**

Independent of maintenance, `vt100`'s implementation has structural costs in the stage the capsule lives in:

- Its `present` stage diffs **two full `Screen`s every frame** ŌĆö an O(rows ├Ś cols) compare ŌĆö even when an agent changed three lines of a tall pane.
- `contents_formatted` / `contents_diff` **allocate `String`s per frame**.
- Its grid is `Vec<Row>` of fat cells; scroll rotates rows; scrollback is a separate concern. Fine, but not the cache-friendly ring buffer the fast engines use.

Whether any of this *matters* for the capsule's actual workload is **unknown and unmeasured** ŌĆö which is the central thing the verification plan exists to settle.

## Usage-grounded analysis (deep dive ŌĆö corrects the framing above)

A pass over what the capsule **actually** calls from `vt100` changes the conclusion materially. The single most important fact:

> **The capsule does not use `vt100`'s emit stage at all.** `rg "contents_diff|contents_formatted" crates/jackin-capsule/src` returns **nothing**. The "diff two full `Screen`s and serialize to escapes" path ŌĆö the thing the survey calls `vt100`'s defining strength and the reason GPU engines "don't fit" ŌĆö is **dead code for us**. We replaced it long ago with our own snapshot + diff + emit in <RepoFile path="crates/jackin-capsule/src/tui/render.rs" />.

### What the old path used `vt100` for (the real coupling surface)

A replacement must satisfy exactly this, and nothing more:

1. **Parser** ŌĆö `Parser::process(bytes)` (a `vte`-class state machine: UTF-8, wide chars, CSI/OSC/DEC modes).
2. **A readable cell grid** ŌĆö `Screen::cell(row, col)` ŌåÆ `Cell::contents()`, `is_wide()`, `is_wide_continuation()`, `has_contents()`, fg/bg/attrs (`render.rs::snapshot_screen_row`, `cell_attrs`).
3. **Geometry + modes** ŌĆö `Screen::size()`, `alternate_screen()`, `screen_mut().set_size()`.
4. **Scrollback** ŌĆö `screen_mut().set_scrollback(n)` + a filled count; the capsule layers its own `scrollback_offset` / inline-scrollback bookkeeping on top (<RepoFile path="crates/jackin-capsule/src/session.rs" />).
5. **`Callbacks`** ŌĆö OSC/title/clipboard/`unhandled_csi` passthrough + the fork's `clear_scrollback()`.
6. **Mouse-mode enums** ŌĆö `vt100::MouseProtocolMode` / `MouseProtocolEncoding` (<RepoFile path="crates/jackin-capsule/src/tui/input.rs" />).

That is a **grid + parser + passthrough** dependency, not an emit dependency.

### Where our time and allocations actually go (the true hot path)

Every render tick (33 ms, `RENDER_TICK_INTERVAL`) the focused pane runs:

- `pane_snapshot` ŌåÆ `snapshot_screen_row` for **every row**, calling `screen.cell(r, c)` across the **whole rect** and building a fresh `Vec<RowSnapshot>` ŌĆö **O(rows ├Ś cols) per frame** even on a partial redraw, because `PaneBodyCache::render_partial` rebuilds the full snapshot before diffing it (<RepoFile path="crates/jackin-capsule/src/tui/render.rs" /> ~line 211).
- **Per non-blank cell: `cell.contents().to_string()`** ŌĆö a heap `String` allocation per cell, per frame; blanks allocate via `" ".repeat(width)`. A tall pane = thousands of `String`s allocated and dropped every 33 ms even when three lines changed.
- Then an O(grid) row-by-row diff against the cached snapshot, and finally an **O(╬ö) emit** (`render_snapshot_rows`).

So the real cost is **per-cell snapshot rebuild + per-cell `String` alloc + full-grid diff**, recomputed from scratch each frame. `vt100`'s internal full-screen compare is *not* in the path (we never call it); the redundant full-grid work is in **our** `render.rs`, because `vt100` exposes **no damage information**, forcing us to re-read and re-diff the entire grid to discover what changed.

### What this does to the build-vs-buy reasoning

- **The roadmap's "Justification B win 1" (O(grid) ŌåÆ O(╬ö) present) is real, but the O(grid) cost is in our snapshot/diff layer, not `vt100`'s emit.** The lever is **source-recorded damage** (mark cells dirty as `Parser::process` mutates them) so the renderer walks only dirty spans and never rebuilds the full snapshot. This is the single biggest, most certain win, and it is an architecture change regardless of which crate backs the grid.
- **"Win 2" (zero per-frame alloc) is concrete and visible right now:** the per-cell `cell.contents().to_string()` is the alloc storm. A packed cell (interned grapheme + style id) read straight into a reused emit buffer removes it.
- **The survey disqualifies Alacritty/Rio for "having no diff-to-escape emit" ŌĆö but that disqualifier does not apply to us, because we do not use `vt100`'s emit either.** We already own the emit (`render_snapshot_rows`). What we actually need from a grid is the two things `vt100` lacks and **`alacritty_terminal` has**: **line/cell-level damage tracking** and a **packed, cache-friendly cell** in a ring-buffer grid. That reframes the candidate set: adopting `alacritty_terminal` purely as the **grid + damage source** (keeping our own emit and passthrough) is a legitimate "buy" path the original survey dismissed for the wrong reason.
- **Our `RowSnapshot` + `PaneBodyCache` layer is itself a hand-rolled damage tracker** ŌĆö a worse one, because it reconstructs damage by full re-read instead of recording it at mutation time. Whatever we adopt or build, that layer is what gets absorbed/deleted.

### Recommended implementation shape (if "build" wins)

The build is **smaller than the survey implies**, because we keep our emit and only replace the grid+damage:

```text

   Ō¢╝

   Ōöé  - dirty set recorded AS           ring-buffer rows, scrollback in-store;

   Ō¢╝
```

The capsule-facing API nouns are already known from the coupling surface: `process(bytes)`, `cell(r,c)`, `size()`/`set_size()`, `alternate_screen()`, `set_scrollback()`, a `dirty_spans()` iterator (the new capability), and a typed passthrough stream. Building to that surface is a drop-in for `session.rs` + a deletion of most of `render.rs`'s snapshot/cache machinery.

**Net:** the concept is *more* attractive than the original page argued, but for a *different reason* ŌĆö not "vt100's emit is slow" (we don't use it) but "vt100 hides damage and allocates per cell, forcing an O(grid)+alloc-storm rebuild every frame in our code." And the cheapest route to test the upside may be adopting `alacritty_terminal`'s grid, not building from scratch. Everything below stays gated on the experiments; this section only sharpens *what to measure and which candidate to spike*.

## The concept we want to test

The concept has three nested claims, each of which must be checked:

1. **The capsule is a re-emitting multiplexer, not a GPU terminal ŌĆö so the "fast" libraries do not fit.** A GPU terminal (Alacritty, Rio, WezTerm, Ghostty) presents by uploading the grid to the GPU; it has **no escape-sequence emission stage at all**. The capsule presents by *serializing the focused grid back into escape bytes* over a wire that may be SSH. The expensive, correct stage for us is **diff-to-escape**, the very thing `vt100` provides and the GPU engines lack.

2. **The performance-optimal design for our hot path is a hybrid that no single crate ships:** the canonical fast **parser** (`vte`, which Alacritty and Rio's `copa` both build on) + a **ring-buffer, damage-tracked grid** (Alacritty's `Storage` idea) + a **wire-minimal diff-to-escape emit** (vt100's idea, but driven by source-recorded damage instead of full-screen compare) + a **typed passthrough event stream** that makes the capsule's OSC/kitty/focus/cwd/scrollback needs first-class instead of bolted on through callbacks.

3. **Because the old fork was permanent and owning crates is not a problem here, the marginal cost of a purpose-built crate (`jackin-term`) is small relative to the architectural and maintenance benefit.** A multiplexer-shaped terminal model stops tracking someone else's stalled SHA, makes every capsule-specific need a first-class API rather than a patch, and can become a reusable building block for the broader jackin' ecosystem.

The concept is therefore: **use the internal `jackin-term` terminal model as the only capsule terminal substrate**: `vte` for parsing, an owned damage grid for terminal state, typed passthrough events for OSC/CSI side channels, and capsule-owned focused-patch emission. The verification plan now proves this implementation under real load; it no longer decides whether to keep a `vt100` path.

What the comparison proves:

- **The GPU-fast engines (Alacritty, Rio) are structurally missing our stage.** Adopting them means hand-writing the diff-to-escape emit ŌĆö exactly what `vt100` gives free. So "just use Alacritty" trades our problem for a bigger one.
- **The reusable, fast, canonical piece everyone shares is `vte`.** That is the part to depend on; rebuilding a VT parser would violate <RepoFile path="AGENTS.md" /> "Prefer libraries over hand-rolled parsers" for no gain.
- **`termwiz` is the only off-the-shelf crate with both a model and a diff-emit** ŌĆö making it the mandatory "buy" candidate to rule out before any "build."
- **Even libvterm ŌĆö the mature C model neovim and emacs-vterm embed ŌĆö does not ship the wire-emit stage.** It gives damage callbacks and abstract state; the host still renders or re-emits itself. That is strong evidence that the diff-to-escape present stage is a genuinely rare, multiplexer-specific piece that *nobody hands you*, in any language. (libvterm is also non-Rust and therefore ruled out as a dependency per the pure-Rust constraint ŌĆö it informs the design, it does not become one.)
- The hybrid we'd build = **`vte` (theirs) + ring-buffer damage grid (Alacritty's / libvterm's idea, our code) + wire-minimal emit (vt100's idea, our code, but damage-driven) + typed passthrough (ours).** Best of each world, none of them shippable as-is for our exact job.

## Why our own crate could make sense

Two **independent** justifications. Either alone could be sufficient; together they are the case.

### Justification A ŌĆö the maintenance reality (already proven, no numbers needed)

We are not choosing between "clean upstream" and "our fork." We are choosing between "**our ad-hoc patch on a stalled crate**" and "**our deliberately-owned crate**." Upstream `vt100` has not pushed code since July 2025, sits on 10 open PRs (one open since 2021), and has left our own `clear_scrollback` patch unmerged as PR #31. Every future capsule need that upstream lacks is a new patch we carry. Given that **owning crates is explicitly not a problem for this project**, a forced-fork-forever situation tips strongly toward owning the layer on purpose ŌĆö *provided* the owned crate gives us something the patched fork does not. That "something" is Justifications B (performance) and the architecture wins below.

### Justification B ŌĆö the architecture/performance fit (must be proven with numbers)

If a jackin'-owned crate is built around source-recorded damage + wire-minimal emit + typed passthrough, the potential wins are:

1. **Per-frame focused-pane present drops from O(rows ├Ś cols) to O(╬ö).** Walk only changed cell spans instead of diffing two full screens every frame.
2. **Zero per-frame allocation.** Emit straight from packed cells into a reused byte buffer; no per-frame `String`.
3. **Fewest bytes on the wire** ŌĆö diff tuned for escape-byte count (tight cursor-move + run-length), the metric that limits an SSH-reached host terminal; no GPU engine targets this.
4. **Passthrough becomes first-class and the fork disappears** ŌĆö kitty-keyboard stack, `modifyOtherKeys`, focus-event mode, OSC 7 cwd, unhandled-CSI forwarding, and scrollback-clear become a typed event stream designed *for*, not patched *around*; `clear_scrollback()` is just a method on our grid.
5. **Scales to many panes** ŌĆö arena/slab storage and one shared scrollback ring across the daemon's N sessions instead of N independent heap grids.
6. **The API's nouns are the capsule's nouns** ŌĆö `Session`, `FocusedEmit`, `PassthroughEvent` ŌĆö the "more intuitive" half of the motivation.

Win 4 is a maintenance/architecture win that holds **regardless of performance**. Wins 1ŌĆō3, 5 are performance claims that are **unproven until measured**. The verification plan exists to measure them.

## Proposed architecture (only if verification says "build")

```text
vte (dependency ŌĆö canonical parser, do not rebuild)
   Ōöé  byte slices in, Perform events out (zero-copy borrow of the input buffer)
   Ō¢╝
DamageGrid          ŌåÉ pooled arena pages (Ghostty PageList idea) or a ring
   Ōöé                  buffer (Alacritty Storage idea): O(1) scroll, packed
   Ōöé                  cache-friendly Cell with interned style ids, scrollback
   Ōöé                  in the same store, dirty-span tracking recorded AS bytes
   Ōöé                  arrive (not recomputed by full-screen compare)
   Ō¢╝
PassthroughEvents   ŌåÉ typed stream: TitleChanged, IconName, ClipboardWrite,
   Ōöé                  KittyPush(n)/KittyPop, FocusMode(bool), ModifyOtherKeys,
   Ōöé                  CwdChanged (OSC 7), PassthroughCsi, ScrollbackCleared.
   Ō¢╝
GridSnapshot        ŌåÉ present stage: DamageGrid::dump() produces an owned screen
                      snapshot, rendered by the capsule's PaneBodyWidget (a
                      Ratatui widget) into the multiplexer frame buffer.
                      SocketBackend then diffs the Ratatui buffer and emits the
                      minimal ANSI byte sequence over the attach socket.
```

Cross-cutting: a multi-session arena so N sessions share slab row storage and one scrollback ring.

### Zero-copy / zero-alloc specifics

- **Parse:** `vte` consumes borrowed input slices; no copy into the parser.
- **Model:** cells store style as packed bitflags + indexed color; grapheme content references interned storage rather than a per-cell heap `String`.
- **Present:** the capsule renders the `GridSnapshot` through `PaneBodyWidget` into the Ratatui buffer; `SocketBackend` diffs the buffer and writes the minimal escape bytes over the attach socket.

### Ideas worth borrowing from Ghostty (MIT-licensed)

Ghostty is **not a candidate dependency** ŌĆö it is Zig, and the pure-Rust constraint above forbids linking it (and, like every engine surveyed, it ships no diff-to-escape stage anyway). But its terminal core has the **best memory model of any emulator surveyed**, and Ghostty is MIT-licensed, so the *patterns* (re-implemented in Rust, with attribution) are fair game for `DamageGrid`. The ideas worth porting into Rust:

1. **`PageList` ŌĆö pooled arena pages instead of a flat ring buffer.** The screen + scrollback is a doubly-linked list of fixed-size pages drawn from a pre-heated `MemoryPool`; each page is one contiguous arena holding its rows, cells, styles, graphemes, and hyperlinks. Scroll and scrollback-trim become page-pointer operations with near-zero per-cell allocation, and on hitting the scrollback limit Ghostty *reuses the oldest page as the newest* ŌĆö a free ring without a fixed-size ring's wide-terminal waste. This is a strictly better `DamageGrid` backing store than the naive ring buffer the design first sketched.
2. **`RefCountedSet` style interning.** Cells store a small style **id**; the actual style lives once in a per-page refcounted set, released on cell overwrite with lazy id reuse. Exactly the "packed cell + interned style" goal, with a proven shape. Hyperlinks use the same set.
3. **Grapheme side-map.** A `grapheme_map` from cell offset to a codepoint slice keeps the common single-codepoint case in the cell and spills only complex clusters to a grapheme allocator ŌĆö our grapheme-interning idea, concretely.
4. **Offset-addressed memory (`OffsetBuf`).** Everything inside a page is stored as an offset into the page arena, not a pointer, so pages are relocatable and copyable. This is what makes the multi-session arena and cheap session teardown realistic.
5. **Serial numbers + pins for invalidation.** Monotonic page serials detect stale references without requiring pointer stability, and a `PinSet` keeps cursor/scrollback anchors correct across mutations. This is the principled replacement for the capsule's current ad-hoc `scrollback_offset` + `inline_scrollback` bookkeeping in <RepoFile path="crates/jackin-capsule/src/session.rs" />.

What Ghostty does **not** give us is the one thing that matters most ŌĆö the wire-emit stage ŌĆö which reconfirms the survey's central finding: the memory model is borrowable from the best in the field, but the diff-to-escape present stage is ours to write no matter which engine we study.

## The synthesis ŌĆö one library, best of all worlds

This is the heart of the proposal. `jackin-term` is not "yet another terminal emulator." It is a deliberate **assembly of the single best idea from each project**, each chosen because it serves the one job the capsule actually does: re-emit a focused PTY's screen to a host terminal with the fewest bytes, the least allocation, and the cleanest passthrough. Every borrowed idea is either a Rust crate we depend on, a pattern we re-implement in Rust (with attribution), or a correctness reference ŌĆö never a foreign binding.

### Store / borrow / concept ŌĆö the full survey (build-decided framing)

Now that the decision is **build**, every project is classified by **how** we take from it, into three precise buckets:

- **DEPEND** ŌĆö take the actual Rust crate as a dependency, unchanged. Reserved for the canonical, stable, correct pieces it would be malpractice to rewrite.
- **STORE** ŌĆö re-implement a **data-structure / memory model** in Rust inside `jackin-term`, with attribution. These are the "how the grid is laid out in memory" ideas.
- **BORROW** ŌĆö re-implement an **algorithm / technique / discipline** in Rust. These are the "how the grid behaves" ideas.
- **REFERENCE** ŌĆö never code from it directly; use it as a **correctness oracle or benchmark target** (differential tests, conformance, "are we as fast as X").

It is not "a new terminal emulator" ŌĆö it is the owned, harness-guarded terminal **model** assembled from the single best answer each project already proved, re-implemented in Rust.

### Prior art for the use case ŌĆö and why the niche is still open

We are explicitly not the first to need this. The *use case* ŌĆö a terminal multiplexer that re-emits a pane to a host terminal ŌĆö has three notable solvers:

- **Zellij (Rust)** ŌĆö the closest. It built precisely `vte` + own `Grid` + `OutputBuffer` damage-emit, and **had to hand-roll the grid because no crate fit**. It never extracted that work into a reusable library; it lives inside `zellij-server`. Aram Drevekenin's [performance write-up](https://poor.dev/blog/performance/) is essentially a field report on optimizing the exact stage `jackin-term` targets.
- **tmux (C)** ŌĆö the canonical multiplexer; its grid + redraw logic is hand-rolled C, not reusable from Rust.
- **WezTerm mux (Rust)** ŌĆö uses `termwiz`'s `Surface`/`Change` model for its multiplexer, the one published Rust path that already has a diff-emit.

The pattern across all three: **the re-emit-multiplexer terminal model is always either hand-rolled and locked inside an application (Zellij, tmux) or available only as part of a heavier surface (`termwiz`).** No one has published a small, focused, pure-Rust crate whose single job is "model a PTY screen and re-emit it to a host terminal with minimal bytes." That gap is exactly the shape of `jackin-term`. If we build it, it is plausibly the first reusable crate of its kind ŌĆö which is why an open question below asks whether it should ship publicly and feed the broader ecosystem, not stay capsule-private.

## How we verify the implementation

This is the core of the remaining item. **`jackin-term` is the implementation; verification proves it is good enough under real load.** All live measurements run inside a role container against the real `jackin-capsule` daemon, with `--debug` so the numbers land in the diagnostics run file (`~/.jackin/data/diagnostics/runs/<run-id>.jsonl`). There is no fallback terminal model and no `vt100` baseline left in the build.

### Step 0 ŌĆö retire the fork the cheap way first (do regardless)

Superseded cleanup: the old fork-retirement baseline is complete because `vt100` is gone from jackin' manifests, lockfiles, fuzz targets, benchmarks, and source policy. <RepoFile path="TODO.md" /> no longer carries the retired fork entry.

### Experiment 1 ŌĆö profile the present path (the load-bearing measurement)

**Goal:** find out whether the `jackin-term` capsule path has *any* per-frame wall, and if so, in which stage.

**Instrumentation:** add `cdebug!`-gated timers around the three stages, in <RepoFile path="crates/jackin-capsule/src/session.rs" /> and <RepoFile path="crates/jackin-capsule/src/tui/render.rs" />:

- `t_parse` ŌĆö `Parser::process(bytes)` per PTY read.
- `t_diff` ŌĆö `GridPatch`/Ratatui buffer diff and encode per rendered frame.
- `t_write` ŌĆö host-PTY write per frame.

Per frame, also record: bytes-in, bytes-out, grid size (rows ├Ś cols), and number of changed cells from `dirty_spans()`.

**Workload datasets** (each run captured to its own diagnostics run id):

1. `seq 1 100000` dumped fast (dense scroll).
2. A real long `claude` / `codex` response rendered in a tall pane.
3. A full-screen TUI redraw storm ŌĆö `htop`, `vim` scrolling a large file.
4. `yes` for 5 seconds (pathological throughput).
5. Multi-pane: 4, 8, and 16 sessions active, all producing output, focus held on one.

**Metrics + decision gate:**

- If **p99 `t_parse` + `t_diff` + `t_write` per frame < 8 ms** and **daemon CPU stays modest** across datasets 1ŌĆō4, the single-pane path has **no wall** ŌåÆ Justification B wins 1ŌĆō3 are worthless ŌåÆ the performance case collapses to multi-pane scaling (Experiment 4) and maintenance (Justification A).
- If **`t_diff` dominates** (full-screen compare cost) on dataset 2/3 where few cells changed, that is direct evidence for **O(grid) ŌåÆ O(╬ö)** being a real win ŌåÆ Justification B win 1 is live.
- If **`t_write` dominates** and bytes-out Ōē½ changed cells, the emit is wasteful ŌåÆ win 3 is live.

### Experiment 2 ŌĆö bytes-on-the-wire efficiency

**Goal:** test win 3 (wire-minimal emit) directly.

For a sample of representative frames from Experiment 1, compute the **theoretical minimal escape byte count** for the actual cell delta (cursor-move + styled run-length) and compare to what `SocketBackend` emits from `GridPatch`.

**Decision gate:** if emitted bytes are within **~15%** of theoretical minimum, the emit stage is near-optimal. If overhead is large (e.g. it re-emits whole lines for single-cell changes), the direct patch encoder needs more work.

### Experiment 3 ŌĆö allocation profiling

**Goal:** test win 2 (zero per-frame alloc).

Run the daemon under `dhat` (or `heaptrack`) across datasets 1ŌĆō3. Count **allocations attributable to the focused-pane render path per frame.**

**Decision gate:** if allocations per frame are already negligible, the zero-alloc rewrite buys nothing ŌåÆ win 2 is dead. If each frame allocates `String`s proportional to screen size, win 2 is live and quantified (allocs/frame ├Ś frame rate = the saving).

### Experiment 4 ŌĆö multi-pane scaling

**Goal:** test win 5 (arena across N sessions).

Spawn 1, 4, 8, 16, 32 sessions all producing steady output. Measure daemon **RSS** and **CPU** vs session count, and per-session spawn/teardown latency.

**Decision gate:** if memory and CPU scale **linearly and cheaply** to 32 panes, the arena win is marginal ŌåÆ win 5 is low priority. If per-session heap grids cause superlinear RSS or cache-thrash CPU, the arena is justified and quantified.

### Experiment 5 ŌĆö focused-path acceptance

The old prototype-spike gate is retired because `jackin-term` is already the production path. The remaining acceptance work is to run the complete focused render path under DHAT and live diagnostics, then record the run ids in the post-restructure checklist.

**Decision gate:** the focused dirty-patch path must show zero heap blocks after warmup for the direct encoder and acceptable p99 present time in a real capsule run. Complex full/chrome/dialog frames remain Ratatui-backed and are measured separately.

### Experiment 6 ŌĆö external-crate watch

The build-vs-buy gate is closed for this PR. `termwiz`, Alacritty, Ghostty, Zellij, tmux, and libvterm remain reference points for future targeted improvements, but replacing the active terminal model means changing `jackin-term` itself rather than restoring an external terminal dependency.

## The decision

The decision is fixed: **`jackin-term` is the only terminal model in `jackin-capsule`.** The old build-vs-buy matrix is retired. External projects remain references, not runtime alternatives, and no PR should reintroduce a `vt100` dependency, feature flag, fuzz target, benchmark baseline, source-policy exception, or fallback path.

The verification is complete when the checklist's remaining live acceptance rows are backed by diagnostics run ids and allocation/RSS/CPU evidence, not guesses.
