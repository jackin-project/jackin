title: "Agent runtime trait — replace the closed Agent enum with per-runtime modules"

**Status**: Open — design proposal (Phase 1.5 follow-up under the [Codebase readability program](/reference/roadmap/codebase-readability/))

## Problem

<RepoFile path="src/agent/mod.rs" /> defines `enum Agent { Claude, Codex, Amp, Kimi, Opencode }` — five closed variants, behind a `pub const` slug table and a hand-rolled `FromStr`. Every layer of the codebase then re-matches on it. The cost of adding a sixth runtime today is not "one new variant"; it is a sweeping, cross-file edit that an agent or human reviewer cannot mechanically verify:

| File | `Agent::Variant` references | Role in the dispatch fan-out |
|---|---|---|
| <RepoFile path="src/runtime/launch.rs" /> | **70** | Container launch, capsule config, mode-resolution trace, install-block emission |
| <RepoFile path="src/derived_image.rs" /> | **42** | Dockerfile generation, entrypoint dispatch, per-agent COPY/CHMOD scaffolding |
| <RepoFile path="src/agent/mod.rs" /> | **37** | The enum itself, plus its install-block / required-env / supported-modes tables |
| <RepoFile path="src/instance/auth.rs" /> | **27** | Credential provisioning (5 near-identical `provision_*` functions) |
| <RepoFile path="src/config/roles.rs" /> | **26** | Per-agent `AgentAuthConfig` field resolution at global / workspace / role layers |
| <RepoFile path="crates/jackin-console/src/tui/components/agent_choice.rs" /> | **23** | TUI agent picker |
| <RepoFile path="crates/jackin-console/src/tui/auth.rs" /> | **20** | Auth-tab row construction |
| <RepoFile path="src/agent_binary.rs" /> | **19** | Per-agent release-resolution and cache layout |
| <RepoFile path="src/instance/mod.rs" /> | **18** | Per-instance manifest state, model accessors |
| <RepoFile path="src/version_check.rs" /> | **15** | Per-agent CLI version probing |
| 21 more files | **117** | Manifest validation, attach, image build, cleanup, save flow, workspace resolve, … |
| **Total** | **414** | spread across **31 production files** |

The same shape leaks into the data model. `AppConfig`, `WorkspaceConfig`, and `RoleConfig` each carry five parallel optional fields (`claude`, `codex`, `amp`, `kimi`, `opencode`) of *the same type* (`AgentAuthConfig`), and the TOML parser uses five separate newtypes (`CodexAuthConfig`, `AmpAuthConfig`, `KimiAuthConfig`, `OpencodeAuthConfig`) just to reject `oauth_token` on the four runtimes that don't support it. A single function in <RepoFile path="src/runtime/launch.rs" /> — `build_mode_resolution` — contains *three* byte-for-byte-symmetric 5-arm matches in 30 lines, one per layer of the global / workspace / role resolution stack:

```rust
let agent_at_global = match agent {
    Agent::Claude => cfg.claude.as_ref().map(|c| c.auth_forward),
    Agent::Codex => cfg.codex.as_ref().map(|c| c.auth_forward),
    Agent::Amp => cfg.amp.as_ref().map(|c| c.auth_forward),
    Agent::Kimi => cfg.kimi.as_ref().map(|c| c.auth_forward),
    Agent::Opencode => cfg.opencode.as_ref().map(|c| c.auth_forward),
};
// …and the same shape repeated for `agent_at_workspace` and `agent_at_ws_role`.
```

[Phase 1.5 of the readability program](/reference/roadmap/codebase-readability/#3-auth-provisioning--agent-dispatch-deduplication) names this — "7+ agent match-arm dispatch blocks" — and proposes "a macro or trait dispatch" as the fix, but stops at that two-word framing. This item commits to the trait shape, defines the per-runtime module layout, and sequences the migration so it can land incrementally on top of the existing Phase 1.5 work without forcing a big-bang rewrite of all 31 dispatch sites at once.

## Why this is the highest-leverage DRY target

The two largest duplication-by-the-numbers entries in the readability program — `Style::default()` (296 sites) and `"jackin.role.toml"` (154 sites) — are *literal-level* duplication. They are real, they are worth fixing, and the readability program already plans to fix them with shared constants and named-style helpers.

The `Agent` dispatch is the same anti-pattern at a higher level of abstraction: instead of a string literal repeated, it is *behavior dispatch* on a closed enum repeated. The cure is the same in spirit (one source of truth) but lives at the type-system layer, not the constants layer. And unlike the constants work, the trait extraction directly enables:

- **A sixth runtime is one file + one registration line.** Today, adding a (hypothetical) `Gemini` variant requires editing all 31 files above and risks compile-clean code where one of the 18 explicit `Agent::Claude => …` arms was missed because the field wasn't required to be exhaustive (e.g. the `_ => None` fallback in a TUI picker).
- **Operator- or role-defined runtimes via TOML** — the existing [Multi-runtime support](/reference/roadmap/multi-runtime-support/) item lists "richer Codex/Amp/Kimi/OpenCode config defaults" and "MCP / skill integration" as remaining work, and lists "third-party runtime marketplace" as a non-goal. The trait shape is the missing prerequisite that lets *us* keep shipping runtimes faster (six is not the ceiling) without committing to a public marketplace. It also opens the door to a future operator-defined `[runtime."myorg/my-cli"]` block whose adapter is a small TOML-described shape rather than five edits across five subsystems.
- **Behavioral specs become writable.** The trait surface (`slug`, `install_block`, `auth_kinds`, `required_env_var`, `dockerfile_fragment`, `runtime_args`, `state_paths`) gives the [Behavioral spec: runtime/launch.rs](/reference/roadmap/behavioral-spec-runtime-launch/) work a stable contract to specify *against*. Today the spec would have to enumerate "for each of the 5 agents, …" everywhere; with the trait the spec describes one interface and the per-runtime modules become the test surface for adapter compliance.

This is not bikeshedding. Every duplication number in the readability program — 296 `Style::default`, 154 `"jackin.role.toml"`, 227 `"Dockerfile"`, 414 `Agent::Variant` — is a symptom of the same anti-pattern at different scales: hand-rolled dispatch and string literals where a constant or a trait object would do. The `Agent` abstraction is the highest-leverage place to break the cycle because it cuts across **5 modules** (config, runtime, instance, console, manifest) rather than being localized to one rendering layer or one filename string.

## Target shape

Replace the closed enum's *behavior* (not necessarily its identity) with a trait whose methods are exactly the questions the codebase asks today:

```rust
// src/runtime/adapter.rs (new)
pub trait AgentRuntime: Send + Sync + 'static {
    /// Stable identifier — matches the slug used in TOML (`claude`,
    /// `codex`, …) and in container labels.
    fn slug(&self) -> &'static str;

    /// Human-facing name for TUI labels and error messages.
    fn display_name(&self) -> &'static str;

    /// Dockerfile snippet to install this runtime into the derived
    /// image. `source` is the path under `.jackin-runtime/agent-binaries/`.
    fn install_block(&self, source: &str) -> String;

    /// `Dockerfile` entrypoint dispatch arm for this runtime — emitted
    /// into the generated entrypoint script today by `derived_image.rs`.
    fn entrypoint_arm(&self) -> String;

    /// Auth modes this runtime supports (`sync`, `api_key`,
    /// `oauth_token`, `ignore`) and the env var each mode resolves to.
    fn auth_modes(&self) -> &'static [AuthModeSpec];

    /// CLI version probe — command + arg list to invoke at startup
    /// for the `version_check` path.
    fn version_probe(&self) -> &'static [&'static str];

    /// Container-side paths this runtime owns (state directory, model
    /// cache, credential file), used by `instance/auth.rs` for
    /// provisioning and by `cleanup.rs` for reclaim.
    fn state_paths(&self) -> RuntimeStatePaths;

    /// Optional model accessor — today's `claude_model()` /
    /// `codex_model()` / `kimi_model()` accessors on
    /// `AgentRuntimeState` collapse into one method that reads from
    /// the per-runtime manifest section.
    fn manifest_model<'m>(&self, manifest: &'m crate::manifest::RoleManifest)
        -> Option<&'m str>;
}
```

Each of the five built-ins becomes a small module behind a `#[derive(Default)]` zero-sized type:

```
src/runtime/adapters/
├── mod.rs        ← `pub fn registry() -> &'static [&'static dyn AgentRuntime]`
├── claude.rs     ← `pub struct ClaudeRuntime;  impl AgentRuntime for ClaudeRuntime`
├── codex.rs      ← `pub struct CodexRuntime;   impl AgentRuntime for CodexRuntime`
├── amp.rs        ← `pub struct AmpRuntime;     impl AgentRuntime for AmpRuntime`
├── kimi.rs       ← `pub struct KimiRuntime;    impl AgentRuntime for KimiRuntime`
└── opencode.rs   ← `pub struct OpencodeRuntime; impl AgentRuntime for OpencodeRuntime`
```

The registry is a static slice; adding a sixth runtime is *one new file* under `adapters/` plus one line in `mod.rs::registry()`. The TOML serde representation keeps a single canonical `AgentAuthConfig` (already exists) keyed by slug — the four agent-specific newtypes (`CodexAuthConfig`, `AmpAuthConfig`, `KimiAuthConfig`, `OpencodeAuthConfig`) collapse into one type, and the "reject `oauth_token` on agents that don't support it" check moves from compile-time `serde(deny)` newtypes to a runtime validator that consults the adapter's `auth_modes()` table.

The `Agent` enum itself does not have to disappear. The realistic shape is:

- Keep `enum Agent` as a thin *identity* enum used for parser ergonomics (`FromStr`, `serde`, exhaustive matches in tests).
- Move every *behavioral* table (`slug`, `install_block`, `required_env_var`, `supported_modes`) onto the trait, with the enum delegating: `impl Agent { pub fn runtime(self) -> &'static dyn AgentRuntime { … } }`.
- Open question (deliberately deferred): once operator-defined TOML runtimes are committed to, drop the enum entirely and use the slug string + a `HashMap<&str, &'static dyn AgentRuntime>` registry.

## Migration plan

The 31-file fan-out is exactly why a big-bang rewrite is wrong. Land in four small PRs, each of which leaves the tree green and incrementally retires one class of duplication:

1. **PR 1 — Trait + adapters, no callers changed.** Introduce `AgentRuntime`, the five adapter modules, and `Agent::runtime()`. The enum's existing methods (`slug`, `install_block`, `required_env_var`, `supported_modes`) become two-line delegators to the trait. Zero behavioral change; the diff is one new directory and a trimmed <RepoFile path="src/agent/mod.rs" />. Snapshot tests in <RepoFile path="src/agent/mod.rs" /> still pass byte-for-byte against the new install-block formatters.
2. **PR 2 — Retire the 18 explicit `Agent::Claude => …` arms.** Replace each match site with `agent.runtime().<method>(…)`. The three identical 5-arm matches in `build_mode_resolution` collapse to one line. Auth provisioning in <RepoFile path="src/instance/auth.rs" /> picks up the `state_paths()` accessor.
3. **PR 3 — Collapse per-agent serde newtypes.** Replace `CodexAuthConfig` / `AmpAuthConfig` / `KimiAuthConfig` / `OpencodeAuthConfig` with a single `AgentAuthConfig` plus an adapter-driven validator. This is a schema-versioned change ([per the AGENTS.md schema-versioning rule](/reference/runtime/schema-versions/)): bump `CURRENT_CONFIG_VERSION` and `CURRENT_WORKSPACE_VERSION`, add a migration step that re-keys the parallel `[claude]` / `[codex]` / … tables under a single shape if the storage layout changes (it likely doesn't — the *parser* changes, not the on-disk format), and re-bake all existing migration fixtures.
4. **PR 4 — Collapse the parallel struct fields where it pays.** `AppConfig.claude` / `.codex` / `.amp` / `.kimi` / `.opencode` become `AppConfig.agents: BTreeMap<Agent, AgentAuthConfig>` *if and only if* the resulting call sites read more clearly than the named fields. This is a judgement PR — not every collapse is a win, and BTreeMap access at every call site can be worse than five named fields. The criterion is: if the next runtime would need to add a sixth named field to *N* structs in three modules, the map shape pays for itself; otherwise keep the named fields. Run the call-site read and make the call in the PR.

Each PR has a clear stop point. The work is parallelisable with the [Split `runtime/launch.rs`](/reference/roadmap/split-runtime-launch/) and [Split `instance/auth.rs`](/reference/roadmap/codebase-readability/#tier-a--files-over-2000-lines-14-files-highest-priority) plans — in fact, retiring the dispatch fan-out from `launch.rs` and `auth.rs` is a prerequisite that makes those splits *smaller*, because much of what looks like "logic to relocate" is actually "match arm to delete." Sequencing recommendation: land PR 1 and PR 2 of *this* item before the file-split work starts.

## Non-goals

- **A third-party runtime marketplace.** Adapters land as `pub mod` entries in the binary; we are not committing to a plugin ABI or dynamic loading. Operator-defined TOML runtimes (a future possibility) are explicitly deferred — see Open Questions.
- **Hiding the closed set.** The five built-ins remain the only runtimes jackin' ships and supports. `Agent::ALL` and the exhaustive matches in tests are intentional — they catch the "you added a runtime and forgot to wire X" class of bug. The trait extraction reduces the *number* of sites that need to be exhaustive, not the existence of the exhaustivity gate.
- **Replacing the per-agent manifest sections.** `[claude]`, `[codex]`, `[amp]`, `[kimi]`, `[opencode]` blocks in `jackin.role.toml` are operator/author-visible TOML surface and stay. The trait describes how the *codebase* reads them, not how role authors write them.
- **Re-litigating the auth model.** This item does not touch the `sync` / `api_key` / `oauth_token` / `ignore` modes themselves — that work lives under [Reliable Claude authentication strategy](/reference/roadmap/claude-auth-strategy/) and [Live auth sync](/reference/roadmap/live-auth-sync/).

## Open questions

- **Operator-defined runtimes via TOML — commit or defer?** A `[runtime."myorg/cli"]` block describing slug, install command, auth-mode list, and entrypoint shape is attractive (closes a real gap in [Multi-runtime support](/reference/roadmap/multi-runtime-support/)) but trades the static registry for `HashMap<&str, Box<dyn AgentRuntime>>` and forces decisions about who validates the install-block's shell content. Current recommendation: ship the trait with a static registry, *design* the TOML shape, gate landing it on a second PR with operator confirmation.
- **Drop the `Agent` enum entirely, or keep it as identity?** Keeping the enum gives us exhaustive matches in tests and a small typed surface for the picker. Dropping it removes a second source of truth (slug strings already exist). The migration plan above keeps the enum; if step 4's call-site read shows the enum adds no value on top of the slug, retire it in a follow-up.
- **Where does the trait live — `src/agent/runtime.rs` or `src/runtime/adapter.rs`?** Adjacent to `Agent` (current `agent/` module) makes the relationship clearest; adjacent to `runtime/` (where the trait is consumed most heavily) reduces the number of `use` paths in `launch.rs`. Pick during PR 1.
- **Should `AgentRuntime` be `dyn`-compatible or generic?** `&'static dyn AgentRuntime` is the simplest registry shape and matches the static-dispatch fallback (`agent.runtime()` returns `&'static dyn`). Generic methods would force every caller to know the concrete adapter at compile time, defeating the registry. `dyn` it is unless a profiler shows a hot path that cares — none of the current dispatch sites are hot.
- **Schema-versioning interaction.** The serde collapse in PR 3 may or may not require a `CURRENT_CONFIG_VERSION` bump depending on whether the on-disk shape changes. Confirm during implementation; if a bump is required, follow the [config / workspace / manifest fixture-and-chain rules](/reference/runtime/schema-versions/) in the same PR.
