title: "Process-Level Sandboxing"
---


**Status**: Proposed — research captured, no implementation committed

## Problem

jackin' current isolation model operates at two coarse granularities:

- **Git-level isolation** (shipped): `shared | worktree | clone` modes control how each mount is materialized per agent session. This governs filesystem branching, not command-level access control.
- **Runtime-level isolation** (deferred): `dind | microvm` modes determine which compute boundary the entire workspace runs inside. This governs the VM or container boundary, not individual operations inside it.

Neither layer addresses the gap in between: **individual tool calls made by an agent during a session have unrestricted access to everything inside their sandbox boundary.**

When an agent runs `npm install` to set up a project, that process can read `~/.ssh/id_rsa`, write to `/etc`, reach arbitrary network endpoints, and inherit every secret injected at container launch. When an agent generates and runs a shell snippet it composed itself, that snippet runs with the same trust level as the entrypoint. There is no per-operation access control — the entire container or VM is the trust boundary, and once the agent is inside it, every subprocess inherits the same permissions.

This is the gap process-level sandboxing fills. It sits *below* the container/VM boundary, wrapping individual commands in OS-native isolation so that each operation gets only the access it needs.

## The Isolation Tier Stack

A complete isolation strategy for AI-agent workspaces has at least three tiers:

```
Tier                     Boundary granularity         Overhead
─────────────────────────────────────────────────────────────────
Process sandbox          Per operation                < 10 ms
Container / DinD         Per workspace session        100–500 ms
MicroVM                  Per workspace session        1–5 s
```

The [selectable sandbox backends](/reference/roadmap/selectable-sandbox-backends/) roadmap item covers the container-to-microVM step. This item covers the process sandbox tier — the innermost ring that wraps individual shell commands, build steps, and AI-generated code invocations.

The two tiers are complementary, not competing. A microVM boundary prevents a container escape from reaching the host. A process sandbox prevents a single bad command from reading every secret inside the VM.

## Zerobox: The Reference Implementation

[Zerobox](https://github.com/afshinm/zerobox) is a lightweight, cross-platform process sandboxing tool written in Rust. It originates from OpenAI's production Codex sandbox runtime and is now open source.

### What it does

Zerobox executes a command with deny-by-default access controls:

- **Filesystem**: reads and writes are blocked unless explicitly permitted via `--allow-read`, `--allow-write`, `--deny-read`, `--deny-write` path flags.
- **Network**: all outbound connections are blocked unless the target domain is listed via `--allow-net`. Restricted to named domains, not IP ranges.
- **Environment**: only a minimal set of variables (`PATH`, `HOME`, `TMPDIR`) pass through by default. Secrets injected at container launch are invisible to sandboxed subprocesses.
- **Credential injection**: a built-in network proxy intercepts outbound HTTPS connections and substitutes real API keys *at the proxy layer*, so the sandboxed process never sees the real credential values. The process sends requests using placeholder values; the proxy rewrites them to the real tokens only for approved host names. This is the same architecture Docker Sandboxes uses for credential isolation.

The overhead is roughly 10 ms per invocation and about 7 MB of memory. There is no daemon, no separate container layer, no image build step.

### Platform implementation

| Platform | Mechanism |
|---|---|
| Linux | Bubblewrap (unprivileged user namespaces), Seccomp (syscall filtering), Landlock (kernel 5.13+, filesystem restrictions via LSM) |
| macOS | Apple Seatbelt (`sandbox-exec`, available since macOS 10.5) |
| Windows | Planned (restricted tokens + ACLs + firewall rules) |

On Linux, the combination of Bubblewrap + Seccomp + Landlock creates defense-in-depth without requiring root or a privileged sidecar. On macOS, Seatbelt runs kernel-enforced policy without any extra dependencies. Neither path uses a container engine — process sandboxing and container isolation compose independently.

### Snapshot and restore

Zerobox includes a snapshot module (Blake3 + Merkle trees + LZ4 compression) that can capture filesystem state before a command runs and restore it afterward. This is relevant to the [session snapshot and rollback](/reference/roadmap/session-snapshot-rollback/) roadmap item: a lightweight, per-operation snapshot is faster and cheaper than a full container or VM checkpoint and is sufficient for many "undo the last build step" use cases.

### Maturity

Zerobox is in early active development (v0.2.x as of May 2026, roughly 580 GitHub stars, daily release cadence). It is not yet production-stable. APIs may change. The Rust SDK (`zerobox` crate), TypeScript SDK (npm `zerobox`), and Python SDK (PyPI `zerobox`) are all available.

Given the pre-1.0 state, the right posture for jackin' is to track zerobox and treat it as a reference implementation that defines the architecture — not to depend on it as a pinned production dependency today.

## Codex CLI Sandbox Mode

OpenAI's [Codex CLI](https://github.com/openai/codex) runs each shell command it issues through a process sandbox in its default `suggest` mode and in `full-auto` mode. The underlying runtime is zerobox: when Codex CLI tells the user it is running a command in "sandbox mode", it is invoking zerobox with a policy derived from the workspace's declared network allow-list and the set of paths Codex is currently working in.

This is the production pattern worth understanding for jackin':

- The *agent entrypoint* (Claude Code, Codex, Amp) runs with normal container permissions.
- Each *tool call* (shell command, file write, subprocess) is wrapped in a process sandbox scoped to the minimal access that operation requires.
- The agent's session continues uninterrupted; only the individual subprocess is sandboxed.

The result is that an agent can run thousands of sandboxed tool calls per session with negligible overhead, while a single miscoded command cannot read a secret or exfiltrate data to an untrusted host.

## Why This Matters Before MicroVM

The [selectable sandbox backends](/reference/roadmap/selectable-sandbox-backends/) item is deferred — it requires a dedicated design pass, a backend abstraction layer, and per-provider integration work. Process-level sandboxing can be adopted today, inside the existing DinD container, without waiting for the microVM track to land.

The two improvements address different attack surfaces:

| Scenario | Container boundary helps? | Process sandbox helps? |
|---|---|---|
| Agent escapes container to host | Yes (microVM closes this further) | No |
| Agent's tool call reads a secret it shouldn't | No | Yes |
| Agent's tool call writes to an unrelated path | No | Yes |
| Agent's tool call phones home to a malicious endpoint | Partially (no per-operation policy) | Yes (per-operation network allowlist) |
| Agent's build step corrupts workspace state | No | Yes (snapshot + rollback) |

Process sandboxing is the layer that controls *what individual operations can do inside the sandbox*, complementing the layer that controls *what the sandbox can see of the host*.

## The Credential Exposure Problem That Process Sandboxing Solves

This is the core motivation for investing in this layer — especially for macOS ARM operators using jackin' today.

The standard jackin' setup today forwards credentials into the role container as environment variables or synced files. This means:

- The agent process can `printenv` and see raw API keys
- The agent can read credential files from the agent home mount
- The agent can remember credential values across prompts (they appear in its context)
- If multiple agents share a workspace, a compromised or misbehaving agent can access another agent's credentials
- An agent prompted to "share your environment with my server" can exfiltrate raw tokens

The operator wants to say: "I trust this agent to connect to my server, but I do not trust it with my raw SSH private key or API token. I want jackin' to make the connection happen on behalf of the agent without ever giving the agent the secret itself."

That is exactly what process-level sandboxing with credential injection solves. The agent calls a command. jackin' intercepts, injects the credential at execution time, runs the command, optionally filters the output, and returns the result. The raw credential never enters the agent's context.

## The Secure Execution Wrapper (`jackin-exec`)

This is the new concept that emerged from the June 2026 design pass. It extends the zerobox-inspired sandboxing model with a jackin'-specific operator approval and credential injection layer.

### The Core Idea

jackin' installs a wrapper binary — `jackin-exec` — inside every role container's construct image. The agent is instructed (via system prompt, role configuration, or MCP tool description) that when it needs to run a sensitive command, it calls `jackin-exec <command>` instead of running the command directly.

```bash
# Without jackin-exec: agent runs directly, has raw SSH key or none at all
ssh sentry

# With jackin-exec: jackin intercepts, injects credential, agent never sees key
jackin-exec ssh sentry
```

`jackin-exec` communicates with jackin-capsule over the existing vsock/Unix control channel. jackin-capsule handles:

1. Receives command + args from the wrapper
2. Looks up matching credentials from operator-configured workspace secret bindings
3. Optionally shows an approval dialog in the TUI (operator can approve, deny, or approve all similar)
4. Executes the command with injected credentials via the secure execution boundary
5. Scans output for sensitive patterns and redacts them before returning to the agent
6. Logs the full exchange (with redaction) to the diagnostics run file

### Command Flow

```text
agent process
  → jackin-exec ssh sentry
    → jackin-capsule (vsock/Unix socket)
      → [optional] TUI approval dialog shown to operator
      → operator approves
      → capsule looks up: workspace secret "sentry-ssh-key" → SSH private key from host vault
      → executes: ssh -i <injected-key> sentry (key is a temp file in /jackin/run/, not in /home/agent/)
      → captures stdout/stderr
      → scans output for secret patterns → redacts
      → returns sanitized output to jackin-exec
    → jackin-exec writes sanitized output to agent's stdout
  agent sees: connection established, no key material
```

The key property: the SSH private key never exists in `/home/agent/`, never appears in `env`, never enters agent context. It is materialized as a temp file under `/jackin/run/<instance>/secrets/` (following the container path convention), used for the single invocation, then deleted. The agent sees only the command's output — which is filtered before return.

### Naming

`jackin-exec` is the proposed binary name. Alternative names considered: `jsecure`, `jx`, `jackin-safe`. The binary name matters because agents must call it by name in shell. `jackin-exec` is explicit, namespaced, and reads naturally in agent tool calls. The concept is called the "secure execution boundary" in docs; operators can think of it as "jackin' runs this on your behalf."

### What Commands Make Sense for `jackin-exec`

| Command type | Why `jackin-exec` adds value |
|---|---|
| `ssh <host>` | SSH key never exposed to agent |
| `gh pr create`, `gh auth login` | GitHub token injected, not stored in container |
| `docker login <registry>` | Registry credential injected per-invocation |
| `aws s3 ...`, `kubectl ...` | Cloud credentials injected without `~/.aws/credentials` in container |
| `curl -H "Authorization: Bearer ..."` | Bearer token constructed and injected, not passed as agent arg |
| `git push` (with SSH or HTTPS) | Git credential injected, not stored in `.git/config` |

Commands that do NOT need `jackin-exec`: read-only operations, package installs, builds, lint/test runs — these don't require credentials the agent shouldn't see.

## Operator Approval UI

jackin' controls the entire TUI display. When an agent calls `jackin-exec`, jackin' can interrupt the session and surface a native approval dialog before execution. This is not possible with any other sandbox approach — it is unique to jackin' because jackin' owns the terminal layer.

Proposed approval dialog shape (exact design belongs in [TUI design decisions](/reference/tui-design-decisions/)):

```
┌─ Secure Execution Request ─────────────────────────────────────┐
│  Agent: the-architect  │  Session: ws:my-service/#4            │
│                                                                  │
│  Command:   ssh sentry                                          │
│  Credential: sentry-prod-ssh-key (SSH key, read from host)     │
│  Will inject: SSH_KEY → temp file, deleted after               │
│  Output filter: redact SSH host key fingerprints                │
│                                                                  │
│  [Approve]  [Deny]  [Approve All Similar]  [Always for Session] │
└──────────────────────────────────────────────────────────────────┘
```

Approval modes:

| Mode | Behavior |
|---|---|
| `Approve` | Run this invocation only |
| `Deny` | Return "permission denied" to the agent |
| `Approve All Similar` | Auto-approve all `jackin-exec ssh sentry` calls for this session |
| `Always for Session` | Auto-approve all `jackin-exec` calls for this session (removes friction once operator trusts the session direction) |

The approval dialog is configurable:

```toml
[runtime.secure_exec]
require_approval = "sensitive"  # always | sensitive | never
```

`sensitive` means approval is required for commands matched in the workspace secret bindings; other `jackin-exec` calls (e.g. commands with no credential injection) run without approval.

When approval is disabled or "always for session" is active, jackin-capsule still logs every `jackin-exec` call with command, credential used, output hash, and timestamp to the diagnostics run file. The operator can review the full log after the session.

## Output Filtering and Redaction

Even with credential injection, command output can contain sensitive content that the operator does not want the agent to see: host key fingerprints, server banners with internal hostnames, partial credential echoes, or error messages that expose infrastructure details.

jackin-capsule scans every `jackin-exec` output stream before returning it to the agent. Redaction rules:

- **Configured secret patterns**: any value matching a workspace-configured secret (its value, its base64 encoding, its hex encoding) is replaced with `[redacted by jackin']`
- **PEM blocks**: `-----BEGIN * KEY-----` … `-----END * KEY-----` → `[key material redacted by jackin']`
- **AWS-style credential strings**: `AKIA...` 20-char uppercase alphanumeric → redacted
- **Operator-defined patterns**: regex patterns declared in workspace config → redacted

The agent receives sanitized output. The full unredacted output is written to the diagnostics run file under the operator's JACKIN_DEBUG run, accessible only from the host.

### Limitations of Output Filtering

Output filtering is not a complete defense. A sufficiently adversarial agent could:

- Exfiltrate in pieces across multiple invocations and reconstruct
- Use encoding tricks (base64, ROT13, steganography in filenames)
- Ask jackin-exec to run a command that re-reads credentials from a known path

The operator approval step is the higher-value defense for these cases: if the operator reviews the command before it runs, they can spot suspicious patterns. Output filtering is defense-in-depth, not a complete boundary. The design is honest about this in the session contract output.

## How the Agent Learns to Use `jackin-exec`

The agent must know to call `jackin-exec` for sensitive operations instead of running commands directly. Three mechanisms, in order of preference:

**1. Role system prompt** (highest coverage, works with any agent runtime): the role's construct image configures the agent's system prompt with an instruction block:

```text
When you need to execute commands that require credentials — SSH connections, registry logins,
cloud CLI commands, git push — call jackin-exec <command> instead of running the command directly.
jackin' will securely inject any required credentials. Available credential bindings for this
workspace: sentry-ssh-key (ssh), ghcr-token (docker login), github-token (gh).
```

**2. MCP tool** (Claude Code / MCP-capable agents): jackin-capsule exposes a `jackin_exec` MCP tool inside the role container. The tool description tells the agent when and how to use it. This surfaces naturally in Claude Code's tool panel. The MCP approach is more structured than shell prompt instructions — the agent can inspect the tool's `inputSchema` to know which commands are supported and which credential bindings are available for the current workspace.

**3. Shell alias / PATH shadowing** (fallback, automatic): the construct image can shadow specific binaries with wrappers that transparently call `jackin-exec`. For example, `~/.local/bin/ssh` (higher in `PATH` than `/usr/bin/ssh`) calls `jackin-exec ssh "$@"`. The agent runs `ssh sentry` naturally and gets jackin-exec behavior without needing to know the wrapper exists. Risk: the agent cannot distinguish sandboxed vs. direct execution, which makes the approval dialog less transparent. Use only for commands where the credential binding is unambiguous.

The recommended approach is (1) + (2): system prompt for universal coverage, MCP tool for structured invocation from Claude Code and Codex.

## Workspace Secret Bindings

The operator declares credential bindings per workspace. These map command patterns to host-side secrets:

```toml
[workspaces.my-service.secure_exec.bindings]
# name = { command_pattern, credential_source, output_filters }
sentry-ssh   = { match = "ssh sentry*", credential = "op://Personal/sentry-ssh/private-key" }
ghcr-token   = { match = "docker login ghcr.io*", credential = "op://Work/ghcr/token" }
github-token = { match = "gh *", credential = "op://Work/github/token" }
aws-creds    = { match = "aws *", credential_env = { AWS_ACCESS_KEY_ID = "op://Work/aws/key-id", AWS_SECRET_ACCESS_KEY = "op://Work/aws/secret" } }
```

`credential` sources supported initially: `op://` (1Password via `op` CLI), `env:<VAR>` (from operator host environment), `file:<path>` (from host file path). Future: native host keychain, HashiCorp Vault, jackin' host bridge.

When a command matches a binding, jackin-capsule resolves the credential at execution time from the host — never pre-loading it into the container. No credential is ever decrypted until the moment of execution, and no decrypted value persists longer than the single invocation.

## Integration Shape for jackin'

This section captures how the above pieces wire into the existing codebase. It is not a committed design.

### Entrypoint wrapping and zerobox

The construct image and runtime entrypoint install both `zerobox` (for general per-operation filesystem/network sandboxing) and `jackin-exec` (for credential-aware secure invocations). For Claude Code, this aligns with its `--dangerously-skip-permissions` full-auto model: the agent still runs without interactive permission prompts, but each shell command it issues is constrained by a jackin'-managed sandbox policy rather than running unrestricted.

`jackin-exec` and `zerobox` compose: `jackin-exec` handles credential injection and operator approval, then delegates actual sandboxed execution to `zerobox` with the appropriate filesystem/network policy derived from mount config.

### Policy derivation

jackin' already knows, per workspace mount, which host paths are shared and with what intent. A natural extension is to derive a per-mount process-sandbox policy from the same mount configuration:

```toml
[[mounts]]
path = "~/Projects/my-service"
# read-write access → allow-write for this path
# nothing else → deny writes elsewhere

[[mounts]]
path = "~/Projects/shared-libs"
readonly = true
# read-only access → allow-read for this path, deny-write
```

The sandbox policy for any command running against these mounts would be derived automatically: allow writes to `~/Projects/my-service`, allow reads from `~/Projects/shared-libs`, deny everything else.

### Credential proxy

Zerobox's built-in network proxy implements the same credential injection architecture that Docker Sandboxes uses: secrets never enter the sandboxed process; the proxy substitutes them only for approved host names. This is a direct implementation path for the credential proxy item in the security roadmap. Rather than building a standalone proxy, jackin' could delegate this to the process sandbox layer for commands that run inside the agent session.

`jackin-exec` and the zerobox credential proxy are complementary: zerobox handles network-layer HTTPS token substitution for arbitrary outbound calls; `jackin-exec` handles command-level credential injection for specific commands with explicit operator approval. Both are needed for full coverage.

### Sandboxed DinD commands

Some agents (role repos that invoke Docker to build sibling containers) issue Docker CLI commands as part of their workflow. These could be sandboxed separately from the agent entrypoint: the agent runs with normal permissions, but its `docker build` or `docker run` invocations are wrapped with a policy that restricts which paths the build context can include and which registries the pull can reach.

## Alternatives

### gVisor

gVisor interposes at the syscall layer rather than the filesystem and network layer, providing a user-space kernel that intercepts every system call a process makes. It offers strong process isolation but adds meaningful latency per syscall and does not provide per-process network policy without additional infrastructure. It is more relevant as a container runtime hardening layer than as a per-operation sandboxing mechanism.

### WebAssembly + WASI

WASI (WebAssembly System Interface) provides capability-based isolation for portable Wasm modules. The isolation model is strong and the overhead is low, but it requires code to be compiled to WebAssembly — not applicable to arbitrary shell commands or native build toolchains.

### Custom seccomp profiles

The DinD container already applies a default seccomp profile. jackin' could tighten this by supplying a custom profile tuned to what agent workloads actually need. This is a weaker and more brittle mechanism than process sandboxing: it restricts syscalls globally for the whole container rather than per-operation, and maintaining correct seccomp profiles for diverse agent toolchains is operationally expensive.

## Open Questions

**`jackin-exec` design:**
1. **Binary name.** `jackin-exec` proposed. Alternatives: `jsecure`, `jx`, `jackin-safe`. Final name must be decided before the construct image bakes it in — renaming later breaks existing role system prompts and MCP tool descriptions.
2. **MCP vs binary vs PATH-shadowing.** Which mechanism should be primary? MCP gives structured tool use for Claude Code / Codex; binary works for any agent; PATH-shadowing is transparent but opaque. Recommendation: binary + MCP, not PATH-shadowing — operators should be able to see when `jackin-exec` is being used.
3. **Credential source priority.** When a command matches multiple bindings, which binding wins? Longest match? First declared? Named explicitly in invocation?
4. **Agent circumvention risk.** A determined agent can still exfiltrate in pieces, use encoding, or run the raw command via a different path. How much of this is acceptable for the current threat model (trusted operator, semi-trusted agent, untrusted external code the agent runs)? The approval dialog is the primary defense; output filtering is secondary. Document this honestly in the session contract.
5. **UX for approval latency.** The operator must approve before the command runs. For an agent making many sensitive calls in sequence, this creates friction. The "Approve All Similar" and "Always for Session" modes address this but need UX design. What session-level state does jackin' track to honor "always for session" across reconnects?

**Zerobox integration:**
6. **Which zerobox integration surface?** Shell-out to the `zerobox` CLI, embed the Rust crate directly, or wait for a more mature API surface before committing to either.
7. **Policy authoring burden.** Who specifies which paths and hosts each agent operation may access? Should jackin' derive this from mount configuration automatically, expose it in `jackin.role.toml`, or leave it to a default-open policy with operator override?
8. **Interaction with Bubblewrap inside the DinD container.** DinD currently runs a privileged container. Bubblewrap requires unprivileged user namespaces, which may be restricted by AppArmor on some hosts. Compatibility testing is needed before committing to the Linux mechanism.
9. **Performance at scale.** Zerobox reports 10 ms overhead per invocation. An agent making 500 tool calls in a session adds 5 seconds of overhead. At 1000 calls, 10 seconds. Whether this is acceptable depends on the operation mix.
10. **Snapshot policy.** Should snapshots be taken before every command, only before commands classified as risky, or only on explicit operator opt-in?
