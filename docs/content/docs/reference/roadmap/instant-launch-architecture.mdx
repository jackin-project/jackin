## Product Contract

An agent instance has one lifecycle. Startup speed, crash recovery, image validity, durable home seeding, and cleanup are not separate features; they are the same runtime contract observed at different moments.

The target behavior is:

- **Launch resumes before it rebuilds.** If the selected workspace/role/agent has a running, stopped, or restorable instance, jackin' offers resume and reuses that same `container_base` and `~/.jackin/data/<container>/` before role refresh, credential resolution, image rebuild, workspace freshness checks, or sibling-runtime preparation enter the foreground.
- **Valid images make container creation cheap.** If no resumable container can be used but the stored image recipe is valid, jackin' creates the container from that image without Docker build, runtime agent install, runtime Claude plugin install, or foreground sibling-agent checks.
- **Builds are deterministic and explainable.** If the image is missing or invalid, jackin' rebuilds from a pinned recipe that names the role commit, base/construct image, supported agent installs, Claude plugin/default-home state, hooks, Capsule payload, mount plan, and Docker resource shape. Every rejected faster path has a typed reason.
- **Runtime setup never repairs a non-empty home.** Container startup only seeds empty durable homes from `/jackin/default-home`, copies auth handoff on that first seed, runs cheap runtime hooks, and starts the selected agent. It does not install agents, install plugins, merge default state, or overwrite existing auth/session state.
- **Exit leaves either resumable state or no state.** A kept session leaves exactly the data needed to resume. A clean terminal outcome deletes the per-instance data directory, socket directory, name-claim lock, and instance-index row in the same path. There is no orphaned middle state that needs a later manual prune.

This page replaces the old separate instant-launch and session keep/resume roadmap items. The merged goal is a fast path that reaches an already-materialized agent in roughly 1-3 seconds, and a cold path that does only the smallest required repair while preserving the same operator state.

## Resolved Decisions

These are settled. The rest of this page assumes them; an implementation that diverges from one of these is wrong, not an alternative.

| D22 | Prewarm | Prewarm builds/refreshes the baked image when the recipe or version labels are stale, off the operator attach path (the `PrewarmOnly` plan), invoked by operator or a background trigger. Launch never waits on prewarm. |

| D16 | Base image pinning | Restore pins the base/construct image by **digest** and rebuilds from it whenever the digest is still pullable (faithful Tier 3). Digest is always preferred; the tag is only a fallback. |
| D15 | Instance surfaces | The console (workspace tree + Running panel + instance detail) lists **all** instances — live and failed — each with its state, and selecting one enters the shared restore ladder at its current rung (live → Tier 0 hardline; stopped/died → Tier 1; image gone → Tier 2/3), with delete/restart available on a failed instance. See <RepoFile path="docs/adr/0002-role-hooks-are-the-role-authors-domain.md" />. |
| D13 | Launch vs reconnect | **Launch never reconnects to a live instance.** A launch (pick workspace + role + agent) only creates a new instance or restores an un-cleanly-terminated one. Reconnect (hardline) happens only when the operator explicitly selects a concrete instance. The two surfaces are orthogonal: **launch = create or restore-dead; instance selection = reconnect/manage live.** Unlimited live instances per workspace+role+agent are allowed. Reverses PR #576's silent auto-attach — see <RepoFile path="docs/adr/0001-launch-never-reconnects-to-a-live-instance.md" />. |

## Target Design

### Image materialization

The derived role image must prove that all supported runtime materialization needed for cheap startup is already present before `docker run`:

1. Install every agent in `manifest.supported_agents()` using the normal official/default paths.
2. Install Claude marketplaces/plugins during Docker build, writing to the standard Claude home `/home/agent/.claude`.
3. Run deterministic role dependency materialization during image build, not from runtime `setup-once.sh`, when the work does not require per-instance mounted state.
4. As the final setup step, move each supported agent's initialized home/state root into `/jackin/default-home`.
5. Recreate empty standard homes afterward so the final image remains sane, even though runtime bind mounts normally cover them.
6. Stamp **Docker image labels** carrying the role commit, base/construct image reference and digest, generated runtime Dockerfile section hashes, each agent's install recipe and binary version/checksum, the Claude plugin/default-home recipe hash, hooks hash, Capsule payload identity, and host identity strategy.

Example final snapshot shape:

```dockerfile
RUN install -d -o agent -g agent /jackin/default-home \
 && mv /home/agent/.claude /jackin/default-home/.claude \
 && mv /home/agent/.claude.json /jackin/default-home/.claude.json \
 && mv /home/agent/.codex /jackin/default-home/.codex \
 && mv /home/agent/.local/share/amp /jackin/default-home/.local/share/amp \
 && mv /home/agent/.config/amp /jackin/default-home/.config/amp \
 && mv /home/agent/.kimi-code /jackin/default-home/.kimi-code \
 && mv /home/agent/.local/share/opencode /jackin/default-home/.local/share/opencode \
 && mv /home/agent/.config/opencode /jackin/default-home/.config/opencode \
 && mv /home/agent/.grok /jackin/default-home/.grok \
 && install -d -o agent -g agent \
      /home/agent/.claude \
      /home/agent/.codex \
      /home/agent/.local/share/amp \
      /home/agent/.config/amp \
      /home/agent/.kimi-code \
      /home/agent/.local/share/opencode \
      /home/agent/.config/opencode \
      /home/agent/.grok
```

The generated Dockerfile should only move roots for supported agents and only for roots/files that exist after setup. The snapshot step must be last among agent/default-home setup steps so all plugin and agent initialization is captured once. The generated command must not run runtime auth setup during image build; image-baked default-home represents tool defaults, plugins, skills, agent config, and non-secret state only.

Image-preparation work can be optimized later with a recipe-keyed artifact cache or a `jackin-image-prepare` helper, but that optimization must preserve the same contract: plugin and agent materialization are image/default-home inputs, not container-start side effects.

### Image validity

Image validity is decided by reading the Docker labels from step 6, not by re-deriving the whole build. Launch computes the expected recipe (role SHA, base digest, agent versions/checksums, plugin/hooks/Capsule hashes) and compares it against the labels stamped on the local image:

- All labels match the expected recipe → the image is valid; create the container from it with no `docker build`, no runtime agent install, no runtime plugin install (`CreateFromValidImage`).
- Any label missing or mismatched → the image is invalid; rebuild from the pinned recipe (`BuildAndCreate`). The specific mismatched label is the typed reason.

Because agent binaries and plugins are now baked (D1/D2), an agent CLI version bump or a plugin recipe change *should* invalidate the image — that is the intended cost of a self-proving image, and the version/checksum labels are what make the bump visible. The trigger is automatic (D20): on launch jackin resolves the latest available agent binary versions plus the current recipe and compares them to the baked image's labels, so a newer binary is detected and rebuilt/prewarmed without a manual rebuild.

### Runtime seeding

Runtime setup must treat the host-mounted durable home as authoritative:

- If the mounted home/root is empty, copy the full matching `/jackin/default-home/...` tree into it.
- If it is not empty, do nothing.
- No merge.
- No partial plugin repair.
- No runtime plugin install.
- No runtime agent install.

Authentication handoff is also first-start-only:

- If the mounted home was empty and got seeded, copy auth handoff into that home when that agent's resolved auth mode requires it.
- If the mounted home was already non-empty, do not copy, remove, or overwrite auth files.

This prevents jackin' from clobbering refreshed in-container OAuth tokens or session state when an operator resumes an old instance. Existing durable state wins.

`/jackin/default-home` must not contain operator auth secrets. It contains image-baked defaults, plugins, agent config, and tool state. Auth enters durable homes only from the runtime handoff on first seed. If auth was not forwarded at first seed, jackin does not re-inject it later (D19) — the operator authenticates inside the container with the agent's normal login, which writes into the durable home and persists.

Per-agent first-seed auth copy rules:

| Agent | Empty-root check | Auth copy on first seed only |
|---|---|---|
| Claude | `/home/agent/.claude` empty | copy `/jackin/claude/account.json` to `/home/agent/.claude.json`; copy `/jackin/claude/credentials.json` to `/home/agent/.claude/.credentials.json` when forwarded |
| Codex | `/home/agent/.codex` empty | copy `/jackin/codex/auth.json` to `/home/agent/.codex/auth.json` when forwarded |
| Amp | `/home/agent/.local/share/amp` empty | copy `/jackin/amp/secrets.json` to `/home/agent/.local/share/amp/secrets.json` when forwarded |
| Kimi | `/home/agent/.kimi-code` empty | copy `/jackin/kimi-code/` contents into `/home/agent/.kimi-code/` when forwarded |
| OpenCode | `/home/agent/.local/share/opencode` empty | copy `/jackin/opencode/auth.json` to `/home/agent/.local/share/opencode/auth.json` when forwarded |
| Grok | `/home/agent/.grok` empty | copy `/jackin/grok/auth.json` to `/home/agent/.grok/auth.json` when forwarded |

For agents with both data and config roots, first-seed detection uses the primary data root and seeds/copies the paired config root in the same transaction. If either root is already non-empty, treat the agent as existing durable state and leave both roots untouched.

There is no separate seed marker (D6). The home's own **emptiness is the signal**: empty → first start → seed defaults and copy auth; non-empty → already set up → do nothing. The current `/jackin/state/agent-auth/<agent>.done` marker is removed — it lives apart from the home it describes and can disagree with it (a wiped home with a surviving marker never gets auth; a lost marker re-copies auth over a refreshed token). Emptiness is one fact that moves, deletes, and restores together with the home.

For emptiness to stay trustworthy, the first seed must be **atomic** per agent: copy the default-home tree and the auth handoff into a temporary location on the same mount, then `rename` into place as the last step. A crash before the rename leaves the home empty, so the next start retries cleanly from scratch — a half-copied home never survives to count as durable.

This is the same lifecycle for first start, restart, and recreate-after-crash (D6a): the container init and all role hooks run normally every start; only this seed/auth step is conditional on emptiness. Restore is not a special path — it is an ordinary start whose home already exists, so seeding no-ops.

### Role hooks

Role hooks are the **role author's domain**. jackin runs them exactly as they are implemented today and does not police, sandbox, rewrite, or rate-limit them. The contract is intentionally narrow (see <RepoFile path="docs/adr/0002-role-hooks-are-the-role-authors-domain.md" />):

- jackin runs the hooks as the standard start flow: `setup-once.sh` once (marker-gated), `source.sh` sourced every start, `preflight.sh` every start. No new build hook is added (D14) — deterministic dependency installation belongs in the role's **own Dockerfile**, which is the build surface jackin already builds on top of.
- A hook that exits non-zero is a **hard failure**: jackin aborts the start and surfaces the hook's error and exit code directly. This is always fatal, not gated on `--debug`. The role author decides what the hook does; jackin only reports faithfully when it fails.
- jackin does not detect, warn about, or prevent what a hook does or how long it takes. If a hook installs dependencies at runtime, that is the author's choice and the author's latency to own; jackin neither blocks it nor runs it in a restricted sandbox.
- Documentation guides authors on intended use — `setup-once.sh` for per-instance state init, `source.sh` for env/PATH, `preflight.sh` for cheap validation, and deterministic deps in the Dockerfile — but this is guidance for role authors, not an enforced jackin contract.

### Launch planning

Launch planning chooses the smallest valid foreground repair:

| Plan | Foreground work allowed | Foreground work forbidden |
|---|---|---|
| `AttachExisting` | inspect running container, verify Capsule, open hardline | role repo refresh, workspace pull, credential resolution, image decision, binary prep, Docker build |
| `StartStopped` | `docker start`, verify Capsule, open hardline | role repo refresh, workspace pull, credential resolution, image decision, binary prep, Docker build |
| `CreateFromValidImage` | re-resolve required secret values, re-materialize mounts from stored recipe, `docker run`, seed empty homes, attach | Docker build, role `HEAD` refresh as restore input, runtime agent/plugin install |
| `BuildAndCreate` | rebuild from pinned or current recipe, then create | untyped rebuilds, rebuilding from moved `HEAD` during restore, storing resolved secrets |
| `PrewarmOnly` | refresh role/image/runtime resources outside operator attach path | mutating host repos or external tool state without explicit opt-in |

Fresh starts may use current role/config inputs. Restores must use the saved launch recipe. This is the key difference: “start new” means “launch what the workspace says today,” while “resume” means “return to the state that was already working.”

### Restore ladder

Resume always reuses the same `container_base` and the same `~/.jackin/data/<container>/` directory. The tiers are ordered by fidelity:

```text
operator chooses Resume <id>
  │  inspect_container_state(container_base)
  ├─ Running                -> Tier 0: hardline reconnect
  ├─ Stopped / exists        -> Tier 1: docker start + reconnect
  ├─ NotFound, image exists  -> Tier 2: docker run using stored image_tag
  └─ NotFound, image missing -> Tier 3: pinned rebuild, then Tier 2
```

| Tier | Meaning | What comes back | Build allowed? |
|---|---|---|---|
| Tier 0 | The original container is still running. | The literal live process/session and writable layer. | No |
| Tier 1 | The original container exists but is stopped. | The same container and writable layer after `docker start`. | No |
| Tier 2 | The container was removed, but its stored image tag still exists. | A recreated container with the same host-backed homes, mounts, auth handoff refs, and image defaults. Container writable-layer-only changes are gone. | No |
| Tier 3 | Both container and stored local image are gone. | A recreated container after rebuilding the same image from launch-time pinned inputs. | Yes, pinned rebuild only |

The ladder is one shared mechanism keyed on `inspect_container_state` of the **chosen instance**; the entry rung depends on the instance's current state, and the two surfaces enter it differently (D15):

- **Console instance selection** lists every instance for the workspace — live and failed — with its state, and selecting one enters the ladder at its current rung: a live instance → **Tier 0** hardline (the only path that reaches Tier 0); a stopped/died instance → **Tier 1**; an instance whose image is gone → **Tier 2/3**. The console also offers **delete** and **restart** on a failed instance.
- **The launch dialog** (role+agent pick with no instance selected) shows only failed/restartable candidates plus "Start new"; it **never lists live instances**. So a launch-restore always enters at **Tier 1 or below** — never Tier 0.

Restore must replay the launch-time recipe stored in the instance manifest rather than resolving current role `HEAD` or current config as if this were a fresh launch. If the role repo, workspace config, or base image changed after the original launch, resume still targets the old pinned recipe. Secrets are never stored; only env var names, `op://` references, and auth mode/source references are persisted and re-resolved when a new container must be created.

Required launch-recipe fields:

| Field | Restore use |
|---|---|
| `image_tag` plus role commit SHA | Reuse exact image at Tier 2; rebuild from exact role commit at Tier 3. |
| base / construct image reference and digest | Tier 3 rebuilds from the pinned **digest** when it is still pullable (faithful); if the digest is gone, jackin auto-falls back to the recorded tag and continues, emitting a typed diagnostic that restore is no longer byte-faithful (D16). Digest is always preferred over tag. |
| selected agent and supported-agent set | Avoid prompting for a different runtime during resume. |
| mount plan and isolation records | Reuse the same worktree/clone/shared layout. |
| env var names and source references | Re-resolve values without storing secrets. |
| auth mode/source references | Rebuild auth handoff when a new container must be created. |
| Docker resource shape | Recreate network, DinD, volumes, labels, and runtime flags consistently. |
| image recipe labels/hashes | Explain whether Tier 2 is valid and why Tier 3 is needed. |

### Exit, keep, and cleanup lifecycle

The same instance identity is the unit of resume. A kept instance retains `~/.jackin/data/<container>/`, its materialized worktree/clone state, the per-instance name-claim lock (`jk-<id>-<role>.lock`), the Capsule socket directory if needed for reconnect metadata, and the `instances.json` row marked restorable. A clean terminal outcome removes all of those immediately.

The name-claim lock is not a sibling-coordination lock: it guards the unique container name during launch and is otherwise inert detritus once the instance is up. Sibling agent tabs in one container are multiplexed in-process by the Capsule daemon, with no host-side lock. Cleanup reaps the lock file because it is leftover per-instance state, not because anything still holds it (D9).

The policy governs **one decision only: what to do when a session ends with dirty or unpushed isolated work.** A session whose `worktree`/`clone` mounts are all clean and fully pushed has nothing at risk and is **always auto-cleaned regardless of policy**. The policy is a single config key — global on `AppConfig`, optionally overridden per-`[workspaces.<name>]` — and its value only changes the dirty/unpushed branch:

| Policy | When the session is clean + pushed | When the session is dirty or unpushed |
|---|---|---|
| `ask` (default) | auto-clean | prompt the operator (keep/discard) |
| `keep` | auto-clean | auto-**preserve** the instance for resume — no prompt |
| `discard` | auto-clean | auto-**discard** everything, including uncommitted edits and unpushed commits — no prompt (D17) |

`keep` and `discard` only take effect when there is dirty/unpushed work; neither keeps a clean session around. Resolution order: workspace override, then global default, then built-in `ask`. The default is always `ask` — the conservative choice that never silently keeps or discards at-risk work.

For `ask`, jackin' must assess `worktree` and `clone` mounts, show concrete uncommitted files and unpushed branches before discard or reuse, and treat `shared` mounts as host-owned data that jackin' never cleans. On resume, `git_pull_on_entry` is skipped; the preserved worktree/clone is authoritative because resume means “return to this exact session,” not “advance the workspace before returning.”

A terminal cleanup outcome must:

1. stamp the terminal status so an interrupted cleanup leaves an honest record;
2. remove `~/.jackin/data/<container>/`, the per-instance name-claim lock `jk-<id>-<role>.lock`, and `~/.jackin/sockets/<container>/`;
3. remove the instance row from `~/.jackin/data/instances.json`.

Only keep/resume outcomes may leave an instance directory or index row behind. Launch or prune should also reconcile stale `active` rows against Docker and reap orphaned `jk-*.lock` / repo-lock detritus through the existing cleanup helpers.

Adding the global and per-workspace `dirty_exit_policy` changes versioned schema surfaces, so the implementation PR must ship the full schema-migration set for affected files: version bump, migration step, migration fixtures, re-baked `after.toml` files, and a [Schema Versions](/reference/runtime/schema-versions/) timeline entry.

### TUI flow

The operator-facing flow must be designed as session continuity, not as an error recovery path. It renders through the launch/exit rich surface. jackin requires a rich terminal (D18); a non-rich, dumb, or headless context is refused with a clear upfront error rather than falling back to a degraded flow. This rich surface is the only place these decisions are made (D11): the console instance list may show status, but resume-or-new, delete-candidate, and the dirty-exit decision are not duplicated as separate console stages or CLI subcommand prompts.

Every screen below follows the [jackin' TUI design rules](/reference/tui/): brand-chrome header, the shared `HintBar` footer (all keys live in the footer, never inside the box — vocabulary `↑↓ navigate · ↵ <action> · Esc back · Ctrl-Q quit`), the `▸` cursor with full-width row selection, the canonical `Confirm` Yes/No dialog (box title `Confirm`, default focus **No**) for destructive confirms, and the sub-dialog stack where each `Esc` walks back exactly one step. The mockups are intent sketches; the implementation reuses the existing `jackin-tui` components (`SelectList`, `ConfirmDialog`, `HintBar`, the scrollable-block and full-screen-overlay primitives) rather than drawing new chrome. Footer hints below are written outside the box to reflect that the footer is a separate reserved region.

Exit flow:

```text
foreground session ends
  │
  ├─ policy = discard ───────────────────────────► DISCARD
  ├─ policy = keep  ─────────────────────────────► KEEP
  └─ policy = ask
        │  assess isolated worktree/clone mounts
        ├─ all clean / all pushed ───────────────► DISCARD
        └─ any dirty or unpushed ─────────────────► Exit dialog (one screen)
                 ├─ Return to agent ─────────────► reconnect now
                 ├─ Exit and keep ───────────────► KEEP
                 └─ Exit and discard ────────────► DISCARD
```

The exit dialog is a **single screen**: a list of the dirty isolated repos (worktree/clone only — D24) plus the three outcome choices. The operator must explicitly choose one; there is no implicit cleanup (D25). The three outcomes:

- **Reconnect** — hardline back into the instance: jackin keeps the container we were about to stop, re-enters the agent, and the operator finishes the work, then exits again later. The no-loss option.
- **Keep** — silently exit and keep the instance. The container stops but the instance is preserved; from `jackin console` the operator can always select it and restart it later (to resume this exact state).
- **Discard** — delete everything. The instance and its dirty work disappear and are no longer selectable anywhere.

`I`nspect opens the read-only change viewer (D24) so the operator can examine the changes before deciding. It does not show per-file detail inline.

```text
 jackin'  · session ended
 the-architect (claude) · jackin · jk-sz2v4p0e

 Unsaved work in 2 isolated repos (worktree/clone only):
   ▸ jackin       worktree · uncommitted · 1 branch unpushed
     holla-apt    clone    · uncommitted

 End this session:
   Reconnect   (hardline back, finish the work — no loss)
   Keep        (stop now, resume from console later)
   Discard     (delete the instance and its changes)
```

Footer hint: `↑↓ navigate · ↵ inspect changes · R reconnect · K keep · D discard · Ctrl-Q quit`. Clean repos are not listed; if nothing is dirty there is no dialog (auto-clean per `dirty_exit_policy`). `↵`/`I` on a repo opens its change viewer; `R`/`K`/`D` are the outcome shortcuts, `R`econnect being the safe default. `Discard` routes through the canonical `Confirm` dialog before deleting.

Launch flow:

Matching candidates are the **un-cleanly-terminated** instances for the same workspace + role + agent: `RestoreAvailable`, `PreservedDirty`, `PreservedUnpushed`, `Crashed`, and stale index rows whose container is gone (D10). A healthy live instance is never a candidate — it is reached only by explicit instance selection (reconnect/hardline), never from a launch pick (D13). The operator can restore one, delete one in place, or start new; a launch with no candidate starts new directly even while another instance is live. Deleting an instance runs the same terminal cleanup as a clean exit — but a candidate with dirty/unpushed work first shows a confirm screen listing what will be lost (D21); a clean one deletes immediately. Deleting every candidate collapses the screen to a plain "Start new".

```text
operator selects workspace + role + agent
  │  find matching candidate instances
  ├─ none ───────────────────────────────────────► Start new
  └─ one or more ────────────────────────────────► Launch dialog (one screen)
        ├─ Start new ────────────────────────────► fresh instance
        ├─ Resume <id> ──────────────────────────► restore ladder Tier 1 → 2 → 3
        └─ Del <id>
              ├─ candidate clean ────────────────► delete immediately
              └─ candidate dirty ────────────────► Delete-confirm screen
```

This dialog opens from the **launch surface**: the operator picks a workspace + role + agent and launches (without having selected a concrete instance). If that workspace+role+agent has no un-cleanly-terminated instances, launch starts a new one directly. If it has one or more, the launch surface opens this dialog instead of silently starting.

The launch dialog is a **single screen**: `Start new` plus each candidate with its **state** (`crashed`, `dirty`, `unpushed`, `kept`) and age, and whether its image is ready. A highlighted dirty candidate lists its dirty *repo names* (not file detail). Resume, delete, start-new, and `I`nspect are all reachable here — no separate verify step. `Start new` stays first so launch never enters old state without selection. Choosing a candidate resumes it: it enters the restore ladder at its current rung (D15) — a stopped/crashed instance is restarted (`docker start`) and the operator hardlines back in; the preserved tree is authoritative and workspace pull is skipped (D7).

```text
 jackin'  · resume or start new
 the-architect (claude) · jackin

 Filter:
 ▸ Start new session
   Resume  jk-sz2v4p0e · 2h · kept · clean · image ready
   Resume  jk-fme29j3j · 5h · crashed · dirty
        dirty repos: jackin (worktree), holla-apt (clone)
   Resume  jk-az81mq4d · 1d · crashed · unpushed · image ready
```

Footer hint: `↑↓ navigate · ↵ resume · I inspect · Del delete · type to filter · Ctrl-Q quit`. The `Filter:` row sits directly under the header per the picker convention. `↵` on `Start new` launches fresh; `↵` on a candidate resumes it (restart + hardline). `I` opens the read-only Inspect viewer (D24) for the highlighted candidate. Once the last candidate is deleted, only `Start new` remains.

Pressing `Del` on a clean candidate deletes it immediately; on a dirty/unpushed candidate it opens the **canonical `Confirm` dialog** (D21) — box title `Confirm`, one-line question, a dim summary of what is lost, two side-by-side buttons with default focus on **No**. The full per-file detail is available via `V`iew beforehand, so the confirm only summarizes:

```text
 ┌ Confirm ─────────────────────────────────┐
 │                                           │
 │           Delete jk-fme29j3j?             │
 │                                           │
 │   Discards 3 uncommitted files and 1      │
 │   unpushed branch — cannot be undone.     │
 │                                           │
 │              Yes        No                │
 │                                           │
 └───────────────────────────────────────────┘
```

Footer hint: `←→ select · ↵ confirm · Y yes · N back · Esc cancel · Ctrl-Q quit`. Default focus is `No`, so a reflexive `↵` does not delete.

### Dirty inspection

`I`nspect (from the exit dialog, or `V`iew from the launch dialog) opens a read-only change viewer. It copies the **concept of GitUI's status tab** — a changed-files list beside a live diff of the selected file — using jackin components only, never gitui itself. It is strictly **preview-only**: no staging, committing, pushing, hunk-discard, or any other mutation. The operator examines changes to inform the reconnect/keep/discard (or resume/delete) decision, and nothing else.

The viewer is one full-screen surface with up to three live panes (`⇥` moves between them):

1. **Repos** (left) — the dirty `worktree`/`clone` repos; shown only when more than one is dirty. `shared` mounts are never inspected.
2. **Changed files** (middle) — the selected repo's changed paths as GitUI-style status rows (`M`/`A`/`D`/`??` glyph + path); nothing clean is listed.
3. **Diff** (right) — the selected file's content, updating live as the selection moves.

```text
 jackin'  · inspect changes · jk-fme29j3j

 ┌ Repos ────────┐┌ Changed files ─────────┐┌ Diff · mod.rs · modified ───────────┐
 │▸ jackin       ││▸ M runtime/launch/mod  ││ ── before ──────│── after ───────    │
 │  holla-apt    ││  M isolation/finalize  ││  fn resolve() { │ fn resolve() {     │
 │               ││  A docs/new-guide.md   ││- attach_now();  │+ create_new();     │
 │               ││  D old/legacy.rs       ││  return Ok(p);  │ return Ok(p);      │
 │               ││  ?? notes.md           ││ }               │ }                  │
 └───────────────┘└────────────────────────┘└─────────────────────────────────────┘
```

Footer hint: `⇥ switch pane · ↑↓ navigate · ↑↓←→ scroll diff · Esc back · Ctrl-Q quit`. With a single dirty repo the Repos pane is omitted, leaving the two-pane files │ diff layout exactly like GitUI's status tab.

Diff rendering by change kind, in the Diff pane:

- **Modified** (`M`) — side-by-side, before │ after.
- **Added / untracked** (`A` / `??`) — single-pane full content as additions; untracked is identical to added because nothing is committed yet.
- **Deleted** (`D`) — single-pane full content as deletions.

```text
 ┌ Diff · docs/new-guide.md · added ───────┐   ┌ Diff · old/legacy.rs · deleted ─────────┐
 │ + # New Guide                           │   │ - fn legacy() {                         │
 │ + ...                                   │   │ -   ...                                 │
 └─────────────────────────────────────────┘   └─────────────────────────────────────────┘
```

The viewer is a sub-dialog on the modal stack: `Esc` returns to the originating exit/launch dialog; a terminal action (choosing an outcome, confirming a delete) clears the chain. Inspecting mutates nothing.

**Crash safety (D25) — critical.** An instance is removed only by explicit operator approval (intentional delete or an explicit discard choice). Any interruption or crash — including while the operator is partway through this inspection — leaves every instance preserved as resumable dirty state, exactly like a crashed instance. There is no implicit cleanup; the operator can always resume/restart. (Only a clean + fully-pushed session auto-cleans, and that is not dirty state.)

**Build notes.** jackin already has ratatui 0.30 + crossterm and a modal/scroll/select-list/dialog component library, plus an Elm-style `ConsoleManagerStage` to add a screen to. To build the viewer: extend <RepoFile path="crates/jackin-runtime/src/isolation/finalize.rs" /> (`assess_cleanup`) to also parse `git status --porcelain` into a per-file change list (added/modified/deleted/untracked), source per-file content via `git diff` (and the file contents for added/deleted), and render with the `similar` crate (already vendored transitively via `insta`; promote to a direct dependency). [gitui](https://crates.io/crates/gitui) is the reference for the status-tab layout and read-only diff rendering — concept only; the repos and changed-files panes reuse jackin's `SelectList`, and the diff pane is one new `diff_view` component. Syntax highlighting is optional and deferred.

### TUI components

Following the [component reuse rule](/reference/tui/components), each screen maps to a jackin' component — reusing an existing one wherever a shape already exists, and adding a new shared component only for a genuinely new pattern (D26). The drill-down as a whole is a [wizard-style modal-stack flow](/reference/tui/components), like the Claude-token flow: each step is one component and `Esc` walks back one step.

| Screen | Component | Reuse / new |
|---|---|---|
| Exit dialog — dirty-repo list + outcomes | `SelectList`/`render_picker_list` for the repo list; outcomes (`R`/`K`/`D`) are footer action keys via `HintBar` (the workspace-list action pattern). Modal shell + backdrop. | Reuse |
| Launch dialog — resume-or-new | `filter_input` + `SelectList` picker with a `Start new session` sentinel row and candidate rows (modeled on `role_picker`). | Reuse (compose) |
| Delete confirm | `ConfirmDialog` — canonical `Confirm` Yes/No, default `No`, with a note row summarizing what is lost. | Reuse |
| Inspect — repos pane + changed-files pane | `SelectList` ×2 (repos list; GitUI-style changed-files status list), arranged in the multi-pane surface. | Reuse (compose) |
| Inspect — diff pane | **`diff_view`** — scrollable diff with a mode for side-by-side (modified) and single-pane (added/untracked, deleted), rendered from `similar` hunks through the shared scrollbar. | **New** shared `jackin-tui` component |

The Inspect surface composes the two `SelectList` panes and the new `diff_view` into a GitUI-status layout; only `diff_view` is genuinely new. It lives under `crates/jackin-tui/src/components/` with a `jackin-tui-lookbook` story exercising focused/unfocused and side-by-side/single-pane states for visual-regression coverage. No screen draws its own border, scrollbar, hint row, or selection chrome — those come from the shared primitives.

## Bug — launch-exit confirmation hangs during a build (fix in this PR)

Reproduced live: with a cold role-base build running on the launch cockpit, opening the "Exit jackin'?" confirmation and answering Yes does not exit. The screen stays on "Building role base image…" and the process never terminates. Parallax confirms it: run `18bbf8288aa08ca8` stayed `status: running` 14 minutes after the exit attempt, the TUI render task was still responsive (mouse scroll log lines 2 minutes before inspection), and `buildx` spans (`FileSync/DiffCopy`, `Secrets/GetSecret`) were still firing 12–13 minutes in — the build never stopped and the launch never unwound.

Two distinct defects, both to fix in this PR:

**LX1 — the exit confirmation defaults focus to No, so Enter dismisses it.** `ConfirmState::new` in <RepoFile path="crates/jackin-tui/src/components/confirm_dialog.rs" /> sets `focus: ConfirmFocus::No` (commented "Default focus = No, so Enter does not accidentally confirm"). The dialog renders the buttons left-to-right as `Yes  No`, which reads as if Yes is the default, but `Enter` maps to `CommitFocused` → `Commit(false)` → `QuitConfirmOutcome::Dismissed` in `apply_quit_confirm_key` (<RepoFile path="crates/jackin-launch/src/tui/subscriptions.rs" />), so the build silently continues. The operator believes they chose Yes; nothing happens. Decision still open and user-facing: either default the **exit** dialog's focus to Yes (a focused-Yes variant, leaving destructive confirmations like delete defaulting to No), or make the rendered button order/highlight unambiguous about which one Enter triggers. Do not silently flip the global default for every `ConfirmState` — the delete/data-loss dialogs intentionally default to No. Pick the exit-specific fix and keep the destructive ones conservative.

**LX2 — Yes is a graceful cancel that waits, but it must be an immediate hard exit.** Today `QuitConfirmOutcome::Confirmed` calls `cancel_token.cancel()` and returns `CockpitOutcome::Continue` (<RepoFile path="crates/jackin-launch/src/tui/subscriptions.rs" /> around the quit-confirm branch). The pipeline is supposed to observe the token via `while_waiting` (role-base build and derived build in <RepoFile path="crates/jackin-runtime/src/runtime/image.rs" />) and unwind through `LoadCleanup` before exiting. That graceful unwind waits on the in-flight build (and, because the dropped build future does not necessarily kill the `docker`/buildkit child, on an orphaned daemon-side build), which is the observed hang. The decided behavior: **answering Yes exits immediately — cancel and quit without waiting for the build, cleanup, or any result.** Implementation: route `QuitConfirmOutcome::Confirmed` to `CockpitOutcome::HardExit` instead of `cancel_token.cancel()` + `Continue`. The render task already implements `HardExit` as restore-terminal-then-`std::process::exit(0)` (<RepoFile path="crates/jackin-launch/src/tui/run.rs" />, the `CockpitOutcome::HardExit` arm), the same path Ctrl+C uses — so Yes becomes the menu equivalent of Ctrl+C: an explicit, immediate, improper exit. This deliberately skips `LoadCleanup`; half-built containers, networks, and volumes are reclaimed by the next launch's `gc_orphaned_resources`, exactly as the existing Ctrl+C path documents. Two side-decisions to settle while implementing: (a) whether to additionally kill the in-flight `docker build` child on the way out (set `kill_on_drop` / send SIGTERM to the buildx process) so the daemon-side build stops promptly rather than running to completion orphaned — preferred, since a multi-minute orphaned build wastes the host; and (b) confirm we want Yes to *always* hard-exit, including the no-build case where graceful cleanup would have been cheap — the user's intent is "always immediate," so the simple single-path behavior wins over branching on whether a build is in flight.

**LX3 — double Ctrl+C must be an unconditional immediate exit from every screen, and currently is not.** Observed: during the build, pressing Ctrl+C repeatedly does nothing — the process does not exit. Ctrl+C detection today lives in `handle_cockpit_input` (<RepoFile path="crates/jackin-launch/src/tui/subscriptions.rs" />), which runs inside the cooperative render task spawned in <RepoFile path="crates/jackin-launch/src/tui/run.rs" /> and is gated on `renderer.try_lock()` plus a 33ms tick. That task is starved while a heavy build is in flight (the same trace shows input recovering only after the build stage changed: mouse events logged at 2 minutes versus the build still running at 12–13 minutes), so the `HardExit` arm never runs and Ctrl+C is effectively dead exactly when the operator most needs it. Note this is in crossterm raw mode, where `ISIG` is disabled, so Ctrl+C arrives as a key event, not a `SIGINT` — a plain `SIGINT` handler would not fire and cannot be the fix.

Required behavior, as a hard cross-screen guarantee: **two Ctrl+C presses within a short window force an immediate process exit from any screen — cockpit, any modal, any prompt, mid-build — with no confirmation, no cleanup, no waiting, no lock acquisition.** The first Ctrl+C may keep its current best-effort meaning; the second is an absolute escape hatch. Implementation must not depend on the tokio runtime or the render task being schedulable, because that is precisely what the build starves. The robust shape is a dedicated OS thread (`std::thread`, outside the async runtime) that owns terminal input, counts consecutive Ctrl+C presses, and on the second within the window calls `std::process::exit` (or `libc::_exit`) directly after restoring the terminal — bypassing `cancel_token`, `LoadCleanup`, the pipeline, and every mutex. Moving input ownership to a never-starved thread also fixes LX2's `HardExit` reliability (the menu Yes → `HardExit` route only helps if something is actually polling input). Orphaned docker resources are reclaimed by the next launch's `gc_orphaned_resources`, identical to the existing accepted Ctrl+C contract. This guarantee applies to the launch cockpit and every other rich-surface screen (exit/launch/inspect dialogs, prompts) — not just the build screen.

Verification: launch into a cold build (e.g. a custom-construct dev sync so the role base rebuilds), and confirm each of these exits in well under a second with the terminal restored — (a) Ctrl+Q then Yes; (b) a single Ctrl+C where the input thread is responsive; (c) **double Ctrl+C mid-build, the case that currently hangs**; and from a non-build screen (an open modal/prompt) confirm double Ctrl+C also exits immediately. Then confirm the next launch's `gc_orphaned_resources` reclaims the abandoned container/network/volume. Add regression tests at the `handle_cockpit_input`/input-thread layer asserting that a confirmed quit yields `HardExit` and that a second Ctrl+C forces exit regardless of pending state, mirroring the existing quit-confirm tests in <RepoFile path="crates/jackin-launch/src/tui/subscriptions/tests.rs" />.

## Follow-up — derived-image build performance (implemented)

The lifecycle contract above is implemented, and this PR also implements the derived/role image build optimizations below. The dominant cost was not the `RUN` steps themselves but the BuildKit **export + unpack** phase: a representative build showed `exporting layers 87.1s` followed by an `unpacking` pass, plus an attestation manifest and a manifest list for a single-platform, never-pushed image. The checklist records the implementation and verification evidence for each build-performance decision.

Context that frames every item below: the derived image is **local-only** — built on the operator's host, loaded into the local engine, and never pushed to a registry. So compression, provenance attestations, and OCI-index packaging buy nothing here. The primary target is macOS, split between OrbStack and Docker Desktop; OrbStack already runs the containerd image store (`overlayfs` + `io.containerd.snapshotter.v1`) by default, so the slow export/unpack on OrbStack is compression + attestations + redundant layers, not a missing store.

| Item | Change | Decision | Risk |
|---|---|---|---|
| BP1 | Disable BuildKit default provenance/SBOM attestations | Do | None — no consumer of attestations on a local throwaway image |
| BP2 | Eliminate the recursive `chgrp -R 0 && chmod -R g=u` by making baked files born group-0 + group-writable | Do, with arbitrary-UID test | Medium — touches the construct base user + arbitrary-UID read/write/seed correctness |
| BP3 | Collapse the per-plugin `claude plugin install` layers into one readable step | Do | Low — loses per-plugin cache granularity; plugins already move as one recipe unit (D2) |
| BP4 | Stop compressing exported layers for the local-only build | Do (confirm exact flag by test) | Medium — likely requires switching from plain `docker build` to explicit `docker buildx build --output …` |
| BP5 | Exploit / require the containerd image store on the daemon | Detect + document only | None to code — store type is operator-daemon config |

**BP1 — provenance/SBOM off.** Log shows `exporting attestation manifest` + `exporting manifest list` for a single-platform image because buildx defaults `--provenance=true` (≥v0.10). That builds an OCI index instead of a plain image and blocks the fast loader, forcing a full OCI export then unpack. Preferred knob is the env var `BUILDX_NO_DEFAULT_ATTESTATIONS=1` added to `docker_build_env()` in <RepoFile path="crates/jackin-runtime/src/runtime/image.rs" />, not a `--provenance=false` CLI flag: the build currently shells out to plain `docker build` (`build_args` starts `["build", …]`), whose classic frontend rejects `--provenance`, while the env var works whether the command resolves to buildx or the classic builder. No detection logic, one line.

**BP2 — narrow the recursive permission pass to the correct boundary.** The old final derived-image step (`RUN chgrp -R 0 /home/agent /jackin/default-home && chmod -R g=u …` in <RepoFile path="crates/jackin-image/src/derived_image.rs" />) rewrote both the live runtime home and the seed snapshot. That was structurally wrong: `/jackin/default-home` is seed data and only needs group readability for runtime seeding, while `/home/agent` is the runtime-mutable home left behind after seed state moves out. The container runs as the host UID/GID (`docker run --user <hostuid>:<hostgid> --group-add 0`, the bind-mount-writability pattern), so every image-baked file that remains under `/home/agent` must be writable through supplementary group 0. jackin cannot predict role-authored tool paths (`.cargo`, `.npm`, `.rustup`, or future dotdirs), so enumerating selected children is the wrong abstraction. Replacement: create the `agent` user with primary group 0 (`useradd -u 1000 -g 0`, the OpenShift idiom) in the construct base <RepoFile path="docker/construct/Dockerfile" />, make `/home/agent` and initial home roots group-writable, set `umask 0002` across build `RUN` steps (files 664 / dirs 775, group-writable by birth), switch generated `COPY --chown=agent:agent` lines to `COPY --chown=agent:0`, copy construct shell configs as `0664`, recreate live-home placeholders with `install -d -o agent -g 0 -m 0775`, explicitly seed/repair Git and shell config files as `0664 agent:0` after runtime finalization, then normalize **only `/home/agent`** to group `0` + group-write and fail the build if anything under it is not group `0` or group-writable. `/jackin/default-home` gets a separate stat-only group-read guard and is not made runtime-writable. Transient installer artifacts are not seed state and must be removed before the snapshot rather than normalized into `/jackin/default-home`. This spans the construct base and the derived template, so it must be validated by an end-to-end arbitrary-UID/GID launch (read + write + first seed + `git config --global` during runtime setup + role hook tool installs) before it is trusted; it is a correctness/security change, not a cosmetic one.

**BP3 — collapse plugin layers, keep them readable.** The generated Dockerfile emits one `RUN claude plugin install …` per plugin (four separate layers in the sampled build), each re-exporting a growing node-heavy plugin store. Fold them into a single `RUN`, but keep it readable: every plugin on its own continued line so the Dockerfile still reads as a clear list of what is installed. Readability of the generated Dockerfile is a hard requirement — an operator must be able to open it and understand each step at a glance. Rendered in `claude_plugin_section` of <RepoFile path="crates/jackin-image/src/derived_image.rs" />.

**BP4 — drop layer compression.** Export gzip-compresses every layer; for an image that is loaded locally and immediately unpacked, that compression is pure wasted CPU that the unpack step then reverses. Remove it entirely for now (a smarter compression policy can come later). The active builder is the `docker` driver writing to the containerd store, and compression is still occurring, so the exact mechanism must be confirmed by a measured test build — the likely shape is switching to an explicit `docker buildx build --output type=image,compression=uncompressed …` invocation in <RepoFile path="crates/jackin-runtime/src/runtime/image.rs" />, which is a larger change than BP1's env var.

**BP5 — containerd image store.** The export-then-unpack round-trip is inherent to the BuildKit image exporter; a daemon backed by the containerd image store avoids the tarball materialization. On macOS this splits by engine: OrbStack already enables it by default (verified: `Storage Driver: overlayfs`, `driver-type: io.containerd.snapshotter.v1`), and Docker Desktop defaults it on since roughly v4.34 with a Settings toggle on older versions. Scope here is therefore detection and guidance, not engine configuration: read the daemon storage driver (e.g. from `docker info`) and surface a launch/diagnostics note when the slow non-containerd store is in use, plus an operator doc for enabling it on Docker Desktop. No jackin-owned code can flip an operator's daemon setting.
