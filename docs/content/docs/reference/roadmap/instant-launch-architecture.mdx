## Confirmed defect — the dirty-exit confirmation never fires

The headline behavior of this work — "if you exit with dirty or unpushed work, jackin' asks what to do" — does not happen. Reproduced three times: with a dirty workspace and the default `ask` policy, exiting the agent normally shows **no dialog**; the container just disappears and the work is silently left preserved. The operator is never asked.

Root cause (code-confirmed):

- The `jackin load` foreground path, `launch_role_runtime` in <RepoFile path="crates/jackin-runtime/src/runtime/launch.rs" />, **never calls `finalize_foreground_session`** at all. On success it returns `Ok(())`; on the attach result being an error it returns a `capsule attach failed` error. There is no dirty-exit dialog on the load path, on any exit.
- The `jackin hardline` reconnect path, `finalize_reconnected_foreground_session` in <RepoFile path="crates/jackin-runtime/src/runtime/attach.rs" />, runs `attach_outcome?` **before** it calls finalize, so a clean agent exit short-circuits past the dialog.
- On a clean last-session exit the in-container capsule client (`client::run_client`) hits the daemon's socket close as `early eof` and **exits non-zero**, so `docker exec … jackin-capsule` returns non-zero. The host then treats a normal exit as a failed attach.

Net effect: the host-side exit dialog is unreachable on the everyday "quit the agent" flow. The feature, as a thing an operator can see and use, is not implemented.

## Decision — the dirty-exit confirmation moves inside the capsule

The confirmation is **redesigned to run inside the capsule** (the in-container multiplexer), not on the host after the container exits. This supersedes the old host-side post-exit exit dialog for the dirty case.

Why in-capsule (operator's call, accepted):

- **Ask before teardown.** A host-side dialog can only prompt *after* the container has exited — at which point "want to resume?" means rebuild/restart to get back to work. In-capsule, the container is still warm; starting a new agent is instant.
- **Decision at the place of work.** The operator is already inside the container; the capsule already owns the TUI and a full set of dialogs. No new surface, no second binary.
- **Less resource churn.** No stop → ask → recreate cycle.

The split of responsibility: the **capsule** detects dirty/unpushed state and **asks** the operator before any teardown; the **host** still **executes** the filesystem keep/discard, because it owns the isolated worktrees. The capsule decides; the host carries it out.

## Architecture — share the dirty-detection logic

The detection logic must be shared, not duplicated, so the capsule and the host can never disagree on what "dirty" means.

- Today the detection lives only host-side, in `assess_cleanup` in <RepoFile path="crates/jackin-runtime/src/isolation/finalize.rs" /> — pure git work (`git -C <path> status --porcelain` for uncommitted/untracked, `rev-list <upstream>..<branch>` for unpushed), coupled only to a worktree path and a command runner.
- Both `jackin-runtime` (host) and `jackin-capsule` (in-container) already depend on `jackin-core`, which already holds the shared `CleanupStatus` and `IsolationRecord` types in <RepoFile path="crates/jackin-core/src/isolation_record.rs" />. `jackin-core` is the natural shared home — no new crate, no new binary.
- Plan: extract the path-based assessment into `jackin-core` as a function over a repo path returning `Clean | Dirty | Unpushed` (plus the `PreservedReason` it already produces). The host's `assess_cleanup` becomes a thin wrapper over it; the capsule calls the same function in-container against its workdir at last-session-close. One source of truth.

## Target workflow — in-capsule dirty-exit

The hinge is the last-session-close branch in <RepoFile path="crates/jackin-capsule/src/daemon.rs" /> (`no_live_sessions()`, which today calls `drain_and_exit` immediately):

```text
last live session exits (agent /exit, Ctrl-D, or crash)
  │  run the shared dirty assessment on the workspace mount(s) in-container
  ├─ clean / pushed ─────────────► drain_and_exit  (unchanged — silent exit)
  └─ dirty / unpushed ───────────► push in-capsule modal; container stays alive
        ├─ Start a new agent ─────► open the New-tab agent picker (verbatim) → spawn → back to work
        ├─ Inspect changes ───────► read-only diff view (Esc back to modal)
        ├─ Exit & keep changes ───► drain_and_exit → host preserves the instance (resumable)
        └─ Exit & discard changes ► drain_and_exit → host discards the dirty work
```

"Start a new agent" reuses the capsule's existing `new_agent_picker` (`PickerIntent::NewTab`) in <RepoFile path="crates/jackin-capsule/src/tui/components/dialog.rs" /> — the exact dialog "New tab" already opens. The confirmation reuses the capsule's existing dialog / modal-stack infrastructure; no new chrome.

### Settled decisions (resolved with operator)

- **Host signal**: the capsule writes the chosen action to a small state file under the per-instance dir — `~/.jackin/data/<container>/.jackin/exit-action.json` (`{ "action": "keep" | "discard" }`) — before `drain_and_exit`. The host finalize reads it and executes that action. Chosen over an exit-code convention (range-limited, conflated with real errors) and a control frame (more plumbing); the file survives the exit and is inspectable.
- **`Ctrl+Q` from the exit modal** maps to `Exit & keep` — the safe default that never loses work.
- **Discard takes no extra sub-confirm**: selecting `Exit & discard changes` is itself the explicit destructive approval (the operator chose the split-Exit shape). No second Yes/No.

## In-capsule dirty-exit UI — agreed design

Settled with the operator by walking faithful TUI previews. The modal renders in the capsule's existing chrome (row 0 ` jackin' ` brand pill + tab strip, row 1 `━` underline) as a centered choice list, reusing the capsule's existing dialog / modal-stack components — no new chrome.

The exit modal, `Unsaved work — exit?`, shows **one summary line per dirty repo** (`<repo>  N changed · N unpushed`) above four choices:

```text
 jackin'  [the-architect]
 ━━━━━━━━━━━━━━━━━━━━━━━━━━

  ┌─ Unsaved work — exit? ───────────────┐
  │ jackin      2 changed · 1 unpushed    │
  │ holla-apt   3 changed                 │
  │                                       │
  │ ▸ Start a new agent                   │
  │   Inspect changes                     │
  │   Exit & keep changes                 │
  │   Exit & discard changes              │
  └───────────────────────────────────────┘
  ↑↓ · ↵ select · Ctrl-Q quit
```

Settled decisions:

- **Default focus** is the first row, `Start a new agent`. **`Esc` is ignored** — the modal is a forced explicit choice; there is no ambiguous cancel.
- **`Start a new agent`** opens the **exact New-tab agent picker** — the same `Dialog::new_agent_picker(agents, PickerIntent::NewTab)` in <RepoFile path="crates/jackin-capsule/src/tui/components/dialog.rs" />, rendered identically: title `New tab`, full-width `── agents ──` / `── shells ──` section dashes, agent display names, type-to-filter, and the `Shell` row. It is the New-tab flow verbatim, not a lookalike. `Esc` in the picker walks back one step to the exit modal (modal stack).
- **`Inspect changes`** opens a read-only diff view inside the capsule, reusing the shared `jackin-tui` `diff_view` plus select-list panes: a changed-files pane beside a live diff, with a repos pane shown only when more than one repo is dirty. Strictly preview-only. `Esc` returns to the exit modal.
- **Keep vs discard is decided in-capsule**: the two `Exit & …` rows are the decision. The host only **executes** the chosen action — it never prompts. Selecting `Exit & discard changes` is itself the explicit destructive approval, so there is no separate host confirmation; if a delete guard is wanted it is an in-capsule sub-confirm, never a post-teardown host prompt.
- **Multiple dirty repos** are listed one line each, with full per-file detail behind `Inspect`.

Still open (minor, settle during implementation): exact row wording, the `Ctrl+Q` meaning from this modal (tentatively treat as `Exit & keep`), and unpushed-only vs uncommitted summary phrasing. The cross-screen double-`Ctrl+C` immediate-exit guarantee applies here too.
