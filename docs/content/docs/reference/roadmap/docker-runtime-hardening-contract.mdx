**Status**: Partially implemented. PR #528 ships the profile/grant *model* and a subset of enforcement, but the end-to-end hardening contract is **not** live yet. What is wired and verified: the four-profile model with per-profile capability/network/dind/sudo defaults; `no-new-privileges` enforced for `standard` (and every non-sudo grant); cgroup-version + AppArmor probes with a fail-closed cgroup-v1 abort for `hardened`/`locked`; the in-capsule `firewall-apply` primitive; removal of the baked `NOPASSWD:ALL` from the base image; and the schema bump to `v1alpha7`. What is **not** wired (claimed shipped earlier but absent or dead in the code) closes the core of the contract: egress allowlist injection + `firewall-apply` exec at launch, the `locked` Docker-internal network, the launch session contract, most per-decision telemetry, the rootless DinD tier, sudo runtime provisioning, and the host-socket regression test. Until those land, `hardened`/`locked` advertise an egress boundary that does not exist — the exact false-isolation gap this item was opened to close. This state violates Decision 2 (#528 ships the enforcement bundle, not a skeleton); **#528 is not mergeable** until the remaining-in-#528 work below closes.

This item tracks the undelivered work — both the in-#528 gaps that block merge and the deferred follow-ups (WP5/WP6/WP7). Operator-facing detail for the parts that genuinely shipped lives in the [Docker Security Profiles guide](/guides/docker-profiles/), the [role manifest reference](/developing/role-manifest/), [Configuration](/reference/runtime/configuration/), and [Schema Versions](/reference/runtime/schema-versions/).

## Decisions (finalized)

These are agreed and drive the plan below. Do not re-litigate without flagging it.

1. **Land everything wired, enforce nothing by default.** Every feature below ships fully wired but takes effect only when an operator opts into a profile. The `compat` default stays a verified no-op, so no existing user's workflow breaks. This is the core guarantee for the whole effort.
2. **PR #528 ships the enforcement bundle, not a skeleton.** #528 does not merge until the opt-in features actually work end to end. Scope: WP0 + WP1 + WP2 + WP3 + WP4 in one PR.
3. **Fail-closed on firewall failure.** Under `hardened`/`locked`, if `firewall-apply` cannot install the egress allowlist, the launch aborts and the container is torn down with an actionable error. The agent never runs with open egress when the operator asked for a boundary.
4. **`standard` becomes the default eventually — not now.** The long-term goal is security-by-default (`standard`: resource limits + `no-new-privileges` on). The compiled-in default flip is the one change that alters behavior for non-opt-in users, so it is deferred (WP6). It is no longer blocked on a sudo audit — WP-SUDO handles sudo in #528; only the flip itself waits (for the WP0 matrix + an explicit changelog call-out). Until then `compat` stays the default.
5. **Deferred out of #528:** credential env-only posture (WP7 → owned by [Container credential exposure](/reference/roadmap/container-credential-exposure/)), making rootless DinD a *default* (WP5 — the rootless *tier* itself ships selectable in WP4), and the `standard` default flip (WP6). Each lands as its own follow-up; none blocks #528.
6. **Docs never describe unshipped behavior.** Every WP updates its own docs in the same commit, and a final claim-by-claim audit (WP-DOC) blocks merge. The user-facing guide must not present egress enforcement, the session contract, or telemetry as shipped until the code that backs them lands, must mark deferred work as deferred, and must not leak internal paths or Rust symbols (three-audience rule).
7. **WP0 ships the full automated matrix now, two-tier.** Profiles must be proven, not spot-checked: Tier 1 mechanism probes gate every PR (fast, no creds/internet, via a local stub host); the full Tier 2 workflow matrix proves real workloads run and must pass once for #528 as the readiness artifact, then runs nightly. Optimized so the full matrix never slows the per-PR signal (image built once, parallel cells, stub host, cheapest workloads).
8. **DinD is fully configurable — all three tiers, nothing removed.** `none` / `rootless` / `privileged` are all operator-selectable (capability-grant model). Rootless DinD is wired in #528 (WP4) so users can choose it and it gets validated, but privileged stays available and rootless is not forced as any profile's default until the WP0 matrix is green (WP5/WP6). Rootless requires cgroup v2 and fails with a clear error — never a silent fallback — on v1.
9. **OTLP endpoint is always allowlisted.** The capsule OTLP telemetry endpoint (`host.docker.internal`) is jackin'-owned infrastructure egress, so it is auto-added to the allowlist on every profile and is not operator-removable. Telemetry keeps flowing under `hardened`/`locked`.
10. **Sysbox is dropped from this roadmap.** Not needed; rootless DinD covers the inner-Docker-without-host-root need without the host-install friction. Revisit only if rootless proves insufficient.
11. **Sudo is opt-in outside `compat`, landing in #528 (WP-SUDO).** The baked `NOPASSWD:ALL` is removed from the base image; sudo is provisioned at runtime only when the `sudo` grant is on. `compat` keeps `sudo = true` (the default profile is unchanged — nobody is broken); `standard`/`hardened`/`locked` are sudo-off-by-default and require an explicit `sudo = true`. This is a deliberate posture change for the non-default profiles only, and it replaces the per-binary `setcap` sudo audit — removing the WP6 blocker so `no-new-privileges` turns on for `standard` in #528.
12. **DinD is off by default everywhere except `compat`; rootless is the recommended tier when enabled (lands in #528, WP4).** Same opt-in philosophy as sudo. Defaults: `compat` = privileged (unchanged); `standard` = none (changed from privileged); `hardened`/`locked` = none. When an operator enables DinD: `standard` → rootless recommended (privileged allowed); `hardened` → rootless only (privileged defeats the profile — emit the partial-enforcement warning if forced); `locked` → stays none, reachable only via an explicit grant that downgrades the enforcement label. Smallest-blast-radius default (no inner engine), unprivileged when you do need it.
13. **Firewall-install signal is informational; marker file deferred.** The host's fail-closed `firewall-apply` exec result (Decision 3) is the single source of truth that the allowlist is active. The container entrypoint's `JACKIN_FIREWALL_INSTALLED` check is informational only and must not error on "not yet installed." An in-container confirmation marker file is deferred unless a concrete need appears.
14. **cgroup v2 is required for `hardened`/`locked`; fail-closed on v1.** On a cgroup-v1 host, `hardened`/`locked` abort with a clear error rather than deliver weaker resource enforcement than promised. `standard` degrades `memory_reservation` with a warning; `compat` is unaffected. (Rootless DinD also requires v2.)

## What actually shipped in #528 (verified against code)

Only the items below are wired and tested. Mechanics live in the canonical pages above; this list is the honest as-merged set.

- **Profile/grant model + per-profile defaults** — four profiles (`compat`/`standard`/`hardened`/`locked`); `profile_base_grants` defaults: `standard` DinD `None`, `standard` sudo `false`, `hardened`/`locked` DinD `None` + sudo `false`, `compat` unchanged (privileged DinD, sudo on). `apply_implicit_grants` adds `NET_ADMIN`/`NET_RAW` under `allowlist` and sets `no_new_privileges = true` whenever `!grants.sudo`. Wired into `run_args` and unit-tested.
- **cgroup + AppArmor probes** — `probe_cgroup_version()`, `parse_apparmor_from_docker_info()`, and `validate_cgroup_for_profile()` exist and are called from `launch_role_runtime`; the cgroup-v1 abort for `hardened`/`locked` is genuinely fail-closed (returns before `run_args` assembly) and tested. `standard` degrades with a warning; `compat` unaffected.
- **`firewall::apply()` primitive** — `crates/jackin-capsule/src/firewall.rs`: `-P OUTPUT DROP` first, accept loopback/ESTABLISHED/DNS, resolve the allowlist into an ipset, empty list = fail-closed. Correct and unit-tested — **but never invoked at launch** (see Gap WP1 below).
- **Base image sudo removal** — the baked `NOPASSWD:ALL` `/etc/sudoers.d/agent` write is removed from <RepoFile path="docker/construct/Dockerfile" />; the `sudo` package is kept; the `TODO(docker-security-profile-sudo-audit)` comment is deleted.
- **Reserved env names** — `JACKIN_NETWORK_MODE`, `JACKIN_ALLOWED_HOSTS`, `JACKIN_NETWORK_ENFORCEMENT`, `JACKIN_FIREWALL_INSTALLED`, `JACKIN_SUDO` are added to the reserved runtime env table (`RESERVED_RUNTIME_ENV_VARS`) so role manifests cannot set them. Only `JACKIN_NETWORK_MODE` is actually injected today.
- **Schema bump to `v1alpha7`** — config/workspace/manifest grant schema, migration step, fixtures, and the [Schema Versions](/reference/runtime/schema-versions/) timeline. This is a real migration; PR-body "no migration" framing is wrong.

## Remaining work to ship in #528 (blocks merge — Decision 2)

These were claimed shipped but are absent or dead in the code. Each is verified below; the per-WP plan in [Implementation Plan](#implementation-plan) still applies. The structural root cause is one defect, not seven (see [Test-architecture gap](#wp0--test-architecture-gap-root-cause) below): the model is *computed* but never *applied*, and the tests cannot see the difference.

1. **WP1 — Egress allowlist enforcement is unwired (highest priority).** `launch_role_runtime` never pushes `JACKIN_ALLOWED_HOSTS` or `JACKIN_NETWORK_ENFORCEMENT`, never builds the allowlist union, and never `docker exec`s `jackin-capsule firewall-apply`. There is no fail-closed teardown and the OTLP host is never added to any allowlist. `default_allowed_hosts_for_agent` is dead code (zero callers). Only `firewall::apply()` and the env reservation landed — so `hardened`/`locked` still advertise an allowlist that does not exist.
2. **WP2 — `locked` internal network is never set.** `create_network`'s `internal: bool` is threaded through all five layers, but every production caller passes `false`; no `profile == Locked ⇒ internal = true` branch exists. `locked` gets an ordinary bridge.
3. **WP3 — Session contract + most telemetry are missing.** `format_session_contract()` is **dead code** — never called from any launch path, with a doc-comment that falsely claims it is emitted under `--debug`. Of the ~11 per-decision `debug_log!` lines specified, only `profile_selected`, `apparmor`, and `cgroup_version` are emitted. The host-socket regression test `role_container_never_mounts_host_docker_socket` (a hard rule) does **not** exist.
4. **WP4 — Rootless DinD tier (Part B) is unimplemented.** `DIND_IMAGE` is hardcoded to `docker:dind` and `privileged: true` is a hardcoded literal; the sidecar builder never inspects the grant. A `dind = "rootless"` grant silently launches a privileged `docker:dind` sidecar — the exact silent-fallback the spec forbids. No rootless cgroup-v2 check, no DinD-without-named-tier → `Rootless` coercion, no privileged-under-`hardened`/`locked` warning. (Part A — per-profile defaults + skip-sidecar-when-`None` — did ship.)
5. **WP-SUDO — sudo provisioning (grant side) is dead.** `JACKIN_SUDO` is never injected into the container, and `sudo_provision::provision()` is reachable only as a manual capsule subcommand — never called from launch or the entrypoint. Granting `sudo = true` therefore produces no working sudo. The deny side (sudo off by default + `no-new-privileges`) did ship. A doc-comment in `sudo_provision.rs` contradicts the Dockerfile about whether sudo is baked-then-revoked or provisioned-on-demand.
6. **WP0 — Test matrix is partial and structurally blind.** Tier 1 probes cover capabilities / read-only-root / `no-new-privileges` only — no egress, no internal-network, no DinD probe — and the 8-cap minimum is inferred from "SETPCAP absent" rather than counted. Tier 2 is a comment block plus a `TODO`, with zero executable code. Critically, the probes hand-write the `docker run` flags they expect and assert Docker honoured them; they never invoke jackin's profile→flags path, so they cannot catch any of WP1–WP-SUDO above.
7. **WP-DOC — Guide is inaccurate.** The [Docker Security Profiles guide](/guides/docker-profiles/) presents egress enforcement, the session contract, full `--debug` telemetry, and the sudo opt-in as shipped — all non-functional today. Rootless DinD is not marked experimental, and the page leaks internal `/jackin/run/...` paths and debug-symbol lines onto an Operator-audience page (three-audience violation). The guide must be reconciled to the as-merged set, and re-reconciled as each gap above closes.

### WP0 — Test-architecture gap (root cause)

The six functional gaps above share one enabling structure: the model layer (`profile_base_grants`, `apply_implicit_grants`, `format_session_contract`, the allowlist union, the DinD grant) is *computed* correctly, but the launch layer never *applies* it, and the WP0 Tier 1 tests assert Docker's behavior given hand-written flags instead of asserting that jackin's launch path emits those flags. That makes "computed but never applied" invisible — dead `format_session_contract`, unset `JACKIN_ALLOWED_HOSTS`/`JACKIN_SUDO`, hardcoded `DIND_IMAGE`, never-`true` `internal` all pass green. The fix that removes the whole class: a **launch-plan assertion layer** — build the `run_args` + env set + post-run exec plan for each profile in a unit test and assert the firewall-apply exec, the injected env vars, the session-contract emission, the rootless DinD image, and the `internal` flag are present per profile. Build this first; it is the acceptance gate every other in-#528 WP cites, and it is what would have caught these gaps.

## Implementation Plan

Detailed, sequenced plan for the undelivered work. Each work package (WP) lists the concrete integration points (file + function), the changes, schema/env additions, tests, telemetry, and exit criteria. The PR breakdown and ordering are at the end.

### Sequencing rationale

The order is driven by dependencies and by user-visible risk, not by the original Phase 1–5 numbering:

1. **WP1 (egress enforcement)** is first because it closes the one active correctness gap: `hardened`/`locked` advertise an allowlist that does not exist. Until it lands, those profiles give a false sense of isolation. Everything else is additive polish on top of already-correct behavior.
2. **WP2 (`internal` network for `locked`)** is a small, self-contained hardening that pairs naturally with WP1 (both touch the network path) and can ship in the same PR.
3. **WP3 (observability)** comes next because once enforcement exists, operators need to *see* what was applied — and the session contract is the surface that tells them the difference between `full` and `partial` enforcement that WP1 introduces.
4. **WP4 (DinD-free auto-skip)** is independent and can land any time after WP1; it makes `hardened` meaningfully lighter and improves WP1's enforcement label (no DinD ⇒ `full` instead of `partial`).
5. **WP5 / WP6** are research/blocked and must not block the above.
6. **WP0 (test matrix harness)** is cross-cutting and should land first or alongside WP1, because every later WP cites it as its acceptance gate.

```text
WP0 (test matrix harness) ─┬─> WP1 (egress) ─┬─> WP2 (internal net)   [same PR as WP1]
                           │                 └─> WP3 (observability)
                           ├─> WP4 (DinD tiers) ──> WP-SUDO (sudo opt-in)
                           ├─> WP6 (default flip) ── deferred (flip only)
                           └─> WP5 (rootless-as-default) ── research
                                WP7 (credential posture) ── after WP3
```

### WP0 — Acceptance test matrix harness (cross-cutting, land first)

**Goal:** prove every profile is genuinely ready — that `hardened`/`locked` actually enforce what they promise *and* still run real agent workflows — with an automated, repeatable test instead of human judgment. **Decision: build the full automated matrix now (in #528)**, structured in two tiers so it proves everything without making PR CI slow.

**Two-tier design (the optimization):** the expensive part is running real workloads and making real network calls; the cheap part is checking the container's effective posture. Split them:

- **Tier 1 — mechanism probes (runs on every PR; fast, deterministic, no credentials, no internet).** Launch each profile **once** (4 containers total), then `docker exec` quick assertions of the *effective state* the WPs produce:
  - capability set — `NET_RAW`/`NET_ADMIN` present only when expected; the 8-cap minimum applied under `hardened`/`locked`;
  - read-only root + tmpfs paths writable (touch a file in `/`→fail, in `/tmp`→ok);
  - `no-new-privileges` active where expected;
  - network egress — curl to a **local stub HTTP server** on the docker network: blocked when not allowlisted, allowed when allowlisted; `locked` network is `internal`;
  - DinD present/absent per profile.
  These verify the *mechanism* in seconds and form the regression guard that gates merges.
- **Tier 2 — real workflow matrix (dedicated CI job + `JACKIN_E2E=1` for local; must pass once for #528 as the readiness artifact, then runs nightly).** The actual workloads per profile prove the cap set/tmpfs/allowlist genuinely run real work: `cargo build`, native-dep `npm install`, `pip install <wheel>`, `git clone+push`, `gh pr create`, in-DinD `docker build`, Compose, Testcontainers. `locked`/`hardened` cells expected to reject a workflow (e.g. `apt install` under read-only root) assert the **documented named error**, not a crash.

**Cell scope (full set — proves all features):** agent runtimes `claude`, `codex`, `amp`, `kimi`, `opencode` × the workflow list above × profiles `locked`/`hardened`/`standard`/`compat`.

**Speed techniques (so the full matrix is not slow):**
- Build the role image **once**, reuse it across all cells (no per-cell rebuild).
- Run cells **in parallel** (nextest / tokio), bounded to host capacity.
- Use a **local stub HTTP server** container for egress cells — deterministic, no real internet, no model-API credentials for the network assertions.
- Pick the **cheapest workload** that still exercises each capability (a hello-world `cargo build`, a single native-dep package), not heavy real builds.
- Tier 1 (mechanism probes) carries the per-PR regression-guard load; Tier 2 (real workloads) runs as its own job so it never blocks the fast PR signal.

**Harness location:** `crates/jackin/tests/profile_matrix.rs`, gated by `JACKIN_E2E=1` like the existing `dind_e2e` test, reusing its container scaffolding. Tier 1 probes may live in the same file behind a fast default-on path; Tier 2 behind the env gate.

**Exit criteria:** Tier 1 probes pass on every PR; the full Tier 2 matrix passes at least once for #528 and its outcome is recorded in a checked-in dated table; every later WP references the relevant rows.

### WP1 — Network egress allowlist enforcement (highest priority)

**Goal:** make `network = allowlist` actually restrict egress. Today the allowlist is computed into `EffectiveGrants.allowed_hosts` and then dropped on the floor.

**Integration points:**
- <RepoFile path="crates/jackin-core/src/env_model.rs" /> — add `JACKIN_NETWORK_MODE_ENV_NAME = "JACKIN_NETWORK_MODE"`, `JACKIN_ALLOWED_HOSTS_ENV_NAME = "JACKIN_ALLOWED_HOSTS"`, `JACKIN_FIREWALL_INSTALLED_ENV_NAME = "JACKIN_FIREWALL_INSTALLED"`; add all three to the `RUNTIME_OWNED_ENV_VARS` table so role manifests cannot set them.
- <RepoFile path="crates/jackin-runtime/src/runtime/launch.rs" />, `launch_role_runtime` — env injection near the `env_strings` block (~line 648). When `grants.network == Allowlist`, push `JACKIN_NETWORK_MODE=allowlist` and `JACKIN_ALLOWED_HOSTS=<comma-joined>`. The host list is `grants.allowed_hosts` ∪ `default_allowed_hosts_for_agent(agent)` ∪ forwarded GitHub host(s) ∪ **the OTLP endpoint host** (always — see below). For `open`, push `JACKIN_NETWORK_MODE=open` (informational, no firewall). For `none`, the existing `--network none` path already covers it.
- <RepoFile path="crates/jackin-runtime/src/runtime/launch.rs" />, `launch_role_runtime` — firewall invocation in the post-`run_role.await?` window (~line 860, after `docker run -d` succeeds, before `reconnect_or_create_session_with_focus`). When `grants.network == Allowlist`, run `docker exec --user root <container> /jackin/runtime/jackin-capsule firewall-apply` via `runner.run(...)`. Root-via-`exec` needs no setuid, so it composes with `no-new-privileges`. On non-zero exit, fail the launch (fail-closed) with an actionable error — do **not** drop the operator into a session that believes it is firewalled but is not.
- <RepoFile path="crates/jackin-capsule/src/firewall.rs" /> — no behavior change needed; `apply()` already does `-P OUTPUT DROP` first, accepts loopback/ESTABLISHED/DNS, resolves the allowlist into an ipset, and treats an empty list as fail-closed (no egress).

**Verified prerequisites (no extra work):** the runtime image already installs `iptables`, `ipset`, and `dnsutils` (<RepoFile path="docker/construct/Dockerfile" />); `firewall-apply` resolves hosts via libc `getaddrinfo`, not `dig`. So WP1 needs no Dockerfile change — only the host-side env injection + exec.

**Firewall-install signal (decided — informational; marker file deferred):** the host's fail-closed `firewall-apply` exec result is the single source of truth for whether the allowlist is active (Decision 3). The entrypoint's `JACKIN_FIREWALL_INSTALLED` check is therefore **informational only** — it must not treat "not yet installed" as an error (the firewall legitimately installs a moment after start). An in-container confirmation marker file is **deferred** as an unnecessary refinement; revisit only if a real need to verify firewall state from inside the container appears.

**OTLP telemetry under allowlist (decided):** the launch wires `OTEL_EXPORTER_OTLP_ENDPOINT` at `host.docker.internal` for capsule telemetry, which jackin' itself relies on. The OTLP endpoint host is therefore **always** added to the auto-injected allowlist set (every profile, never operator-removable), so telemetry keeps flowing under `hardened`/`locked`. It is jackin'-owned infrastructure egress, not agent egress.

**Enforcement honesty:** reuse `network_enforcement_label(grants)` (already written) to report `full` vs `partial (sudo grants iptables access)` vs `partial (DinD inner containers bypass host iptables)`. This label must appear in the session contract (WP3) so operators are never told "allowlist" without the caveat.

**Tests:**
- Unit: env injection present for `allowlist`, absent for `open`/`none`; host list composition (agent endpoints + GitHub + configured hosts, deduped).
- Unit: `firewall-apply` exec is emitted only for `allowlist`, with `--user root`.
- E2E (WP0 rows): under `hardened`, a curl to a non-allowlisted host fails; an allowlisted host succeeds; DNS resolves.

**Telemetry:** `debug_log!("launch", "network_enforcement mode=allowlist label=<full|partial:…> hosts=<n>")` and `firewall_apply exit=<code>`.

**Exit criteria:** an agent launched under `hardened` cannot reach a non-allowlisted host; the launch fails closed if `firewall-apply` errors; the enforcement label is truthful.

### WP3 — Launch observability (session contract + probes + telemetry)

**Goal:** make every applied control visible at launch. The data already exists; the surface does not.

**Integration points:**
- <RepoFile path="crates/jackin-runtime/src/runtime/docker_profile.rs" /> — `format_session_contract(...)` exists but is never called. Call it from `launch_role_runtime` and print under `--debug` (and emit via `debug_log!`).

On cgroup **v1**: `hardened`/`locked` **abort the launch** with a clear error ("requires cgroup v2; this host is v1") — never deliver a weaker boundary than the profile promises. `standard` degrades `memory_reservation` with a warning; `compat` is unaffected. (rootless DinD also requires v2 — same probe, WP4.)

- <RepoFile path="crates/jackin-runtime/src/runtime/launch.rs" /> — replace the single summary `debug_log!` line with the full per-decision telemetry set (below).

**Telemetry surface (minimum lines):**

- `seccomp profile=<docker-default|custom|unconfined>`
- `apparmor available=<yes|no> profile=<…> layer=<host|backend-vm>`
- `read_only_root enforced=<yes|no> tmpfs=<list>`
- `cgroup_version v=<v1|v2|hybrid>`
- `resource_limit kind=<memory|cpus|pids> value=<v> flag=<docker-flag>`
- `dind enabled=<yes|no> mode=<privileged|rootless> tls_certs_path=<path>`
- `host_socket_check passed=<yes|no>`

**Tests:**
- **Host-socket regression test** (`role_container_never_mounts_host_docker_socket`) — specified for Phase 1 but missing. Assert no `-v …/docker.sock` ever appears in `run_args`. This is a hard rule; the test must exist.

- Unit: probe parsers handle apparmor present/absent and cgroup v1/v2/hybrid.

**Exit criteria:** `jackin load … --debug` prints a complete, accurate per-dimension contract; the host-socket test guards the hard rule.

### WP4 — DinD configuration: all three tiers selectable (auto-skip + rootless)

**Goal:** make every `DindGrant` tier (`none` / `rootless` / `privileged`) a real, operator-selectable option — the capability-grant model applied to DinD. No tier is removed: privileged DinD stays available; `none` skips the sidecar; `rootless` is newly wired. Rootless is *selectable* now, validated against the WP0 matrix, but is **not** any profile's default until that matrix is green and `standard` becomes the default (WP6). It requires cgroup v2 and must fail with a clear error (not silently) on v1 hosts.

**Part A — DinD-free (`none`) and per-profile defaults (Decision 12):** stop starting the sidecar when the resolved grant is `None`, and set the defaults so DinD is off everywhere except `compat`.
- <RepoFile path="crates/jackin-runtime/src/runtime/docker_profile.rs" />, `profile_base_grants` — `compat` = `Privileged` (unchanged); change `standard` from `Privileged` to **`None`**; `hardened`/`locked` stay `None`. Recommended-tier-when-enabled is policy, not a default: a `standard`/`hardened` launch that grants DinD without naming a tier resolves to `Rootless`; `privileged` under `hardened`/`locked` is allowed only by explicit grant and emits the partial-enforcement warning.
- <RepoFile path="crates/jackin-runtime/src/runtime/launch/launch_pipeline.rs" /> — grant resolution already downgrades `dind` to `None` when the role declares it; with the new defaults, only `compat` (or an explicit grant) reaches a DinD sidecar.
- <RepoFile path="crates/jackin-runtime/src/runtime/launch.rs" /> — sidecar block is already gated on `dind_enabled(grants)`; remaining work is telemetry + verifying the no-DinD path skips cert volume / network wiring cleanly.

**Part B — rootless tier (`rootless`):** wire `DindGrant::Rootless` end to end.
- <RepoFile path="crates/jackin-runtime/src/runtime/launch/launch_dind.rs" />, `run_dind_sidecar` — when the grant is `Rootless`, use the `docker:dind-rootless` image and **drop `--privileged`** (rootless runs in a user namespace); keep the existing TLS-cert wiring. When `Privileged`, keep today's `docker:dind` + `--privileged` path unchanged.
- **cgroup-v2 requirement:** rootless DinD needs cgroup v2. Reuse the WP3 cgroup probe; if the host is v1 and the grant is `rootless`, fail the launch with a clear error naming the requirement — never silently fall back to privileged (that would defeat the operator's choice).
- **Defaults (Decision 12):** only `compat` defaults to `Privileged`; `standard`/`hardened`/`locked` default to `None`. Rootless is reached when an operator enables DinD on a non-`compat` profile (the recommended tier) or via an explicit `dind = "rootless"` grant. Making rootless a *standard default* (vs `none`) is the separate WP5/WP6 question.

**Tests:**
- Unit (Part A): `hardened` with no inner-engine need starts no DinD, mounts no certs, reports `network_enforcement = full`.
- Unit (Part B): `dind = "rootless"` selects `docker:dind-rootless` and omits `--privileged`; `privileged` keeps `docker:dind` + `--privileged`; `rootless` on a probed-v1 host fails with the named error.
- E2E (WP0 rows): rootless DinD runs the inner-Docker workflows (`docker build`, Compose, Testcontainers) on a cgroup-v2 runner — these rows are exactly the rootless compatibility matrix that gates making it a default later.

**Telemetry:** `debug_log!("launch", "dind enabled=<yes|no> mode=<privileged|rootless> reason=<…>")`.

### WP-SUDO — sudo becomes opt-in (drop baked `NOPASSWD:ALL`)

**Goal:** stop baking blanket passwordless sudo into the base image. Sudo becomes a real, grant-controlled capability: present by default **only under `compat`** (so today's default behavior is unchanged), off by default under `standard`/`hardened`/`locked` and enabled only when the operator explicitly grants it. This is the one deliberate per-feature change to a non-default profile's posture (`standard` loses default sudo) — `compat`, the actual default, is untouched, so no current launch breaks. Removing the blanket grant also **unblocks `no-new-privileges` for `standard`** (the old WP6 blocker), replacing the per-binary `setcap` audit entirely.

**Integration points:**
- <RepoFile path="docker/construct/Dockerfile" /> — remove the baked `echo "agent ALL=(ALL) NOPASSWD:ALL" > /etc/sudoers.d/agent` (currently ~line 135). Keep the `sudo` package installed so the grant can provision it. Delete the `TODO(docker-security-profile-sudo-audit)` comment — this WP resolves it.
- <RepoFile path="crates/jackin-runtime/src/runtime/docker_profile.rs" />, `profile_base_grants` — change `standard` from `sudo: true` to `sudo: false`; keep `compat` `sudo: true`; `hardened`/`locked` stay `false`. Make `grants.sudo` *enforced*, not just informational.
- Provision sudo at runtime when granted: the host passes a `JACKIN_SUDO=1` env (new constant in <RepoFile path="crates/jackin-core/src/env_model.rs" />, added to `RUNTIME_OWNED_ENV_VARS`) from `grants.sudo`; <RepoFile path="crates/jackin-capsule/src/runtime_setup.rs" /> (or the entrypoint) writes `/etc/sudoers.d/agent` only when that env is set. The setup step already runs at container start with the privilege needed to write there.
- **`no-new-privileges` tie-in:** enforce `no_new_privileges` whenever `grants.sudo == false` (so `standard` with default-off sudo gets it; a `standard` launch that opts into `sudo = true` keeps `no-new-privileges` off to avoid the silent-sudo-failure trap). Fold this into the WP3 flag emission.

**Tests:**
- Unit: `compat` resolves `sudo = true`; `standard`/`hardened`/`locked` resolve `sudo = false`; explicit `sudo = true` grant flips it on.
- Unit: `JACKIN_SUDO` env injected only when `grants.sudo`; `no_new_privileges` emitted iff `!sudo`.
- E2E (WP0 rows): `compat` container can `sudo apt install`; `standard` cannot until `sudo = true` is granted; the image ships no `/etc/sudoers.d/agent` until provisioned.

**Telemetry:** `debug_log!("launch", "sudo enabled=<yes|no> source=<profile|grant>")`.

**Docs (WP-DOC):** the guide's profile table currently shows `standard` Sudo = **Yes** — must change to opt-in; document that sudo is now off outside `compat` and how to enable it.

**Exit criteria:** the base image ships no blanket sudo; `compat` launches still have sudo (unchanged); non-`compat` profiles have sudo only when granted; `no-new-privileges` is active on `standard` by default.
