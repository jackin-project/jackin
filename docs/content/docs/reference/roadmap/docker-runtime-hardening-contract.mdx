**Status**: Partially implemented. The profile/grant model, per-launch flag emission (capabilities, read-only root, resources, `no-new-privileges`), and the config/workspace/manifest schema shipped in PR #528 behind the `compat` default тАФ no behavior change for existing launches. The egress-allowlist enforcement, the launch observability surface (session contract + AppArmor/cgroup probes + full telemetry), and the DinD-reduction track are **not yet wired**. Remaining work is below.

This item tracks only undelivered work. Operator-facing detail for what shipped lives in the [Docker Security Profiles guide](/guides/docker-profiles/), the [role manifest reference](/developing/role-manifest/), [Configuration](/reference/runtime/configuration/), and [Schema Versions](/reference/runtime/schema-versions/).

## What shipped (PR #528)

Baseline, so the remaining work below has context. Do not re-document the mechanics here тАФ link to the canonical pages above.

- `DockerSecurityProfile` (`locked | hardened | standard | compat`, ascending capability) with full precedence resolution: CLI `--docker-profile` > workspace `[docker] profile` > config `[docker] profile` > compiled-in `compat` default.
- `[docker.grants]` (config + workspace) and role manifest `[docker]` (`min_profile`, `dind`, `allowed_hosts`, `capabilities_add`), with grant validation, per-dimension layering, `min_profile` launch gate, and DinD downgrade.
- Per-launch Docker flag emission for `hardened`/`locked`: `--cap-drop=ALL` + 8-cap minimum set, `--read-only` + tmpfs preset, `--memory`/`--memory-reservation`/`--cpus`/`--pids-limit`/`--ulimit`, `--security-opt no-new-privileges`. `network = "none"` grant emits `--network none`.
- DinD privileged sidecar with TLS certs under `/jackin/run/dind-certs`.
- Schema: config + workspace `v1alpha7`, manifest `v1alpha6`, with migrations and fixtures.
- A single launch telemetry line (`debug_log!("launch", "docker profile=тАж source=тАж dind_enabled=тАж network=тАж")`).

The `compat` default is a verified no-op: it emits no new Docker flags, so any launch without explicit profile/grant configuration behaves exactly as before this PR.

## Remaining work

### 1. Network egress allowlist enforcement (highest priority)

The allowlist is computed (`EffectiveGrants.allowed_hosts`, the `firewall-apply` capsule subcommand) but never enforced: nothing sets `JACKIN_ALLOWED_HOSTS` / `JACKIN_NETWORK_MODE` on the container, and nothing invokes `jackin-capsule firewall-apply`. Today `hardened`/`locked` resolve `network = allowlist` but run on an open bridge with full egress тАФ a silent gap operators could mistake for isolation.

- Inject `JACKIN_NETWORK_MODE` and `JACKIN_ALLOWED_HOSTS` (operator/role hosts + the agent's API endpoints from `default_allowed_hosts_for_agent` + forwarded GitHub) as `-e` flags in <RepoFile path="crates/jackin-runtime/src/runtime/launch.rs" /> when `grants.network == Allowlist`.
- After the container starts, run `jackin-capsule firewall-apply` via `docker exec --user root` (root without setuid, so it composes with `no-new-privileges`), gated on `network == Allowlist`, and set `JACKIN_FIREWALL_INSTALLED=1`. The capsule entrypoint already warns when `network=allowlist` but the firewall is not installed.
- Fail-closed semantics already live in <RepoFile path="crates/jackin-capsule/src/firewall.rs" /> (`-P OUTPUT DROP` first; empty allowlist = no egress). Surface a `partial` enforcement label when sudo, `user=root`, or DinD can bypass host iptables (`network_enforcement_label` already computes this).
- The allowlist is IPv4-only (ipset). IPv6 / `ip6tables` egress policy is deferred.

### 2. `internal` network for `locked`

`create_network` in <RepoFile path="crates/jackin-docker/src/docker_client.rs" /> has no `internal` parameter. For `locked`, create the bridge with `internal: true` (blocks external egress at the Docker network layer) rather than relying solely on the iptables allowlist. Verify DinD inner containers inherit the restriction or emit a `partial` warning.

### 3. Launch observability

`format_session_contract` exists but is never called, and the per-decision telemetry lines specified for this contract are not emitted (only the one summary line ships).

- Call `format_session_contract` at launch and surface it under `--debug` so operators see the effective per-dimension contract (profile, caps, root fs, DinD, network mode + enforcement, resources, credentials, residual risk).
- Add an AppArmor probe (`docker info --format '{{.SecurityOptions}}'` or `{{.OperatingSystem}}`) and a cgroup-version probe (`stat -fc %T /sys/fs/cgroup/` or `/proc/self/cgroup`) at launch; feed both into the contract and telemetry. Cache per session тАФ the probe adds 100тАУ500 ms on some Docker Desktop setups. On cgroup v1 under `hardened`/`locked`, fail-closed; under `standard`, downgrade `memory_reservation` with a warning.
- Emit the per-decision `debug_log!("launch", тАж)` lines: `cap_drop_all` + `cap_add`, `no_new_privileges enforced=тАж reason=тАж`, `seccomp`, `apparmor`, `read_only_root`, `cgroup_version`, `resource_limit`, `dind тАж tls_certs_path=тАж`, `host_socket_check`.
- Add the host-socket regression test (`role_container_never_mounts_host_docker_socket`) тАФ it was specified for Phase 1 but does not exist. It must fail if any change mounts `/var/run/docker.sock` into the role container.

### 4. DinD reduction track (research / blocked)

- **DinD-free roles:** auto-skip the sidecar when `profile >= hardened` and the role does not require an inner engine. The schema (`dind` grant) exists; the launch-time skip + `debug_log` is the remaining work.
- **Rootless DinD** (`docker:dind-rootless`, drop `--privileged`): blocked on cgroup v2 + passing the compatibility matrix. Research only.
- **Sysbox on Linux:** blocked тАФ needs a host-side runtime registration that conflicts with the "never mutate host silently" rule. Decide operator-confirmed opt-in vs hard-skip before any work.

### 5. `compat` тЖТ `standard` default flip + `no-new-privileges` for `standard` (blocked)

Blocked on the base-image `NOPASSWD:ALL` sudo audit. With `--security-opt no-new-privileges`, `sudo` fails at the kernel level тАФ silent, confusing runtime breakage for `apt`/`dpkg`/`npm` scripts, worse than not shipping. Resolution: audit every `sudo` call site in <RepoFile path="docker/construct/Dockerfile" /> and replace each with a file capability (`setcap`), then flip the compiled-in default to `standard` and enable `no-new-privileges` there. Until then `standard` keeps `no-new-privileges` off.

### 6. Credential posture integration

Under `hardened`/`locked`, prefer env-only credential injection over file copies; emit a `credential_posture` telemetry line per agent auth type. Tracked jointly with [Container credential exposure](/reference/roadmap/container-credential-exposure/).

## Open questions

- **Acceptance test matrix.** The exact agent-runtime ├Ч workflow set that validates the 8-cap minimum and each profile (e.g. `apt install build-essential`, native-dep `npm install`, `cargo build`, in-DinD `docker build`, Testcontainers, `gh pr create`). Without it, "validate the cap set" has no objective answer.
- **AppArmor on macOS.** What jackin' prints under `standard` on Docker Desktop / OrbStack, derived from the backend probe (item 3).
- **Read-only root `$HOME` audit.** Before tightening read-only root, enumerate every non-bind-mount runtime write target under `/home/agent/` (shell history, `.zsh_sessions/`, oh-my-zsh markers, `mise` shims) and give each a tmpfs entry or state bind mount. `runtime-setup` writes are already on bind-mount paths.

## Constraints that stay true

- **Agent bypass flags are not per-profile.** Every agent launches with its maximum-bypass flag in <RepoFile path="docker/runtime/entrypoint.sh" /> (`claude --dangerously-skip-permissions`, etc.). The Docker boundary is the only enforceable isolation layer; the session contract must state this so operators trust Docker isolation, not the agent's own permission system.
- **Host Docker socket exclusion is a hard rule** (see the regression test in item 3).
- Hardened Docker is still a shared-kernel boundary тАФ the contract must not overstate it.
