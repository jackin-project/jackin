title: "Docker Runtime Hardening Contract"

**Status**: Open — design proposal (Docker-first containment track)

## Problem

jackin' is intentionally Docker-first today. That choice is still correct for the current product shape: local operator-controlled agents, explicit mounts, agent state that survives reconnects, and a Docker-compatible development environment that most role authors already understand.

The weakness is not "Docker exists." The weakness is that the current jackin' Docker runtime does not yet expose a precise hardening contract. Operators see "container isolation" but do not get a structured answer to:

- which Docker security controls were requested
- which controls the selected host can actually enforce
- whether the role container, DinD sidecar, and inner containers have different risk postures
- what remains open after launch
- which profile is appropriate for trusted local work, suspicious repos, or long autonomous runs

The current model is coherent but broad:

- build the role image through the selected host Docker engine
- create a per-agent Docker network
- start a privileged `docker:dind` sidecar
- start the role container on that network
- mount only operator-approved host paths
- connect the role container to the DinD sidecar over TLS
- keep the host Docker socket out of the agent container

That is already better than a normal development container with the host Docker socket mounted. But it is not the same thing as a hardened sandbox. This roadmap item defines the Docker-side hardening path before jackin' adds any new backend family.

## Design Philosophy: Capability Grants, Not Restrictions

jackin' is intentionally a no-interruption autonomous agent environment. The agent binaries run with maximum bypass: `claude --dangerously-skip-permissions`, `codex --dangerously-bypass-approvals-and-sandbox`, `amp --dangerously-allow-all`. This is not a weakness — it is the product's reason for existing. An agent that stops to ask permission for every file write is not useful.

The Docker container IS the sandbox. The agent is free inside it. The operator controls what the container can do, not what the agent chooses to do inside the container. These are the same outcome — if the container has no outbound network, it does not matter that the agent might try to exfiltrate data, because the packet never leaves.

This shapes the entire design of the profile system:

> **Capability grants, not restrictions.** A container starts from a minimal locked baseline. The operator (or role) explicitly grants what they need. The profile is a named bundle of grants. Explicit grants can extend any profile. The result is: "this container is exactly as capable as I allowed it to be — nothing more."

This is different from the alternative approach (start permissive, optionally restrict), which has the failure mode of "I thought I restricted X but I forgot to add the flag." A grant model fails safe: if you forget to grant something, it does not work, and the operator knows immediately what to add. A restriction model fails open: if you forget to restrict something, the capability is silently available.

**Practical implications:**
- `compat` is a full-grant profile — it is what you opt into when you need everything, not the default
- `locked` is the starting point for any use case where you want tight control
- profiles are bundles of grants for common use cases; explicit grants compose with any profile
- a role declares minimum grants it requires; the operator can grant more but cannot grant less than the role needs
- the session contract output tells the operator exactly what was granted and what was not

**The agent bypass flags are not configurable** — they are immutable inside the container. The Docker boundary is the only enforceable isolation layer. This is correct: jackin' is not trying to sandbox the agent from itself; it is trying to sandbox the container from the host. The operator's trust model is "I trust the agent to do its job within the limits I set on the container."

## Thesis

The next step should be:

> Keep Docker as the default runtime, but make Docker launches managed by jackin'
> run under an explicit, auditable capability-grant contract. The container has
> exactly the capabilities it was granted — network, Docker, privilege, filesystem
> access, and resources — and no more.

This is different from:

- replacing Docker with microVMs
- claiming hardened containers equal hypervisor isolation
- forking Docker or owning a custom runtime
- silently changing host Docker, host firewall, or host config
- restricting what the agent binary itself chooses to do inside the container (it runs with full bypass; the container boundary is the enforcement layer)

Docker remains the packaging and execution substrate. jackin' becomes the grant policy layer that chooses, applies, reports, and tests a precise Docker capability posture.

## Source Material

This design depends on Docker primitives that exist today:

- [Docker rootless mode](https://docs.docker.com/engine/security/rootless/) - runs the daemon and containers inside a user namespace without daemon root privileges.
- [Docker seccomp profiles](https://docs.docker.com/engine/security/seccomp/) - Docker's default profile is an allowlist that blocks a set of high-risk syscalls while preserving broad compatibility.
- [Docker AppArmor profiles](https://docs.docker.com/engine/security/apparmor/)
  - Docker applies `docker-default` on supported Linux hosts unless overridden.
- [Docker run reference](https://docs.docker.com/reference/cli/docker/container/run/)
  - the CLI surface for capabilities, `--security-opt`, `no-new-privileges`, read-only root filesystems, tmpfs mounts, networking, and ulimits.
- [Docker resource constraints](https://docs.docker.com/engine/containers/resource_constraints/)
  - memory, CPU, swap, and related cgroup controls.
- [Docker daemon attack surface](https://docs.docker.com/engine/security/) - daemon control is privileged enough to mount host paths and create powerful containers, so jackin' must keep the host socket out of agent reach.

Existing jackin' roadmap items folded into this contract:

- [Network egress policy](/reference/roadmap/network-egress-policy/) - outbound policy and enforcement-quality reporting.
- [Declarative resource limits](/reference/roadmap/declarative-resource-limits/)
  - role and launch resource budgets.
- [Session contract and explain mode](/reference/roadmap/session-contract-explain-mode/)
  - preflight explanation of the actual boundary.
- [Process-level sandboxing](/reference/roadmap/process-level-sandboxing/) - per-operation controls inside the runtime.

Concrete Docker upstream references collected during the May 2026 research pass:

- Docker default capability set (the 14 caps granted unless `--cap-drop` overrides) is defined in `moby/moby/daemon/pkg/oci/caps/defaults.go`: `CAP_CHOWN`, `CAP_DAC_OVERRIDE`, `CAP_FSETID`, `CAP_FOWNER`, `CAP_MKNOD`, `CAP_NET_RAW`, `CAP_SETGID`, `CAP_SETUID`, `CAP_SETFCAP`, `CAP_SETPCAP`, `CAP_NET_BIND_SERVICE`, `CAP_SYS_CHROOT`, `CAP_KILL`, `CAP_AUDIT_WRITE`.
- Docker default masked `/proc` and `/sys` paths (from `moby/moby/oci/defaults.go`): masked = `/proc/asound`, `/proc/acpi`, `/proc/kcore`, `/proc/keys`, `/proc/latency_stats`, `/proc/timer_list`, `/proc/timer_stats`, `/proc/sched_debug`, `/proc/scsi`, `/sys/firmware`, `/sys/devices/virtual/powercap`; read-only = `/proc/bus`, `/proc/fs`, `/proc/irq`, `/proc/sys`, `/proc/sysrq-trigger`. `--security-opt systempaths=unconfined` lifts both lists (per `docs.docker.com/reference/cli/docker/container/run/`).
- Docker rootless storage drivers (per `docs.docker.com/engine/security/rootless/`): `overlay2` (kernel 5.11+), `fuse-overlayfs` (4.18+), `btrfs`, `vfs`. AppArmor, overlay networks, and SCTP port exposure are explicitly unsupported in rootless mode.
- `dind` image TLS posture (per Docker Hub `_/docker` image docs): from 18.09+ the dind variants auto-generate `ca/`, `server/`, `client/` cert directories in `DOCKER_TLS_CERTDIR`. No rotation/revocation tooling shipped upstream; an operator-built PKI is the only path beyond first launch.
- `docker sandbox` (Docker Sandboxes) CLI surface as of May 2026 (per `docs.docker.com/reference/cli/docker/sandbox/`): `create`, `exec`, `inspect`, `ls`, `network`, `reset`, `rm`, `run`, `save`, `stop`, `version`. No documented events stream, no `--format json`, no per-sandbox stdout/log hooks. Not yet a viable programmatic backend for jackin'.

## Non-Goals

- Do not claim hostile multi-tenant SaaS safety from Docker hardening alone.
- Do not mount the host Docker socket into agent containers.
- Do not mutate host firewall rules, host Docker daemon config, host git config, or host credential stores silently.
- Do not make every role compatible with the strictest profile on day one.
- Do not replace the future backend work tracked in [Selectable sandbox backends](/reference/roadmap/selectable-sandbox-backends/).
- Do not keep separate one-off roadmap pages for individual Docker hardening flags when they belong to this shared contract.

## Current Docker Boundary

The current Docker backend has useful properties:

| Property | Current behavior |
|---|---|
| Host file visibility | Only explicit workspace/global mounts enter the role container. |
| Mount write access | `:ro` mounts are enforced by Docker and the kernel. |

| Host Docker daemon | The role container does not receive the host Docker socket. |

## User Perspective and Practical Impact

This section evaluates each hardening control from the operator's actual usage pattern and answers the question: what does this give a real user and when?

### The baseline user and their workflow

jackin' operators are developers who run AI coding agents against their local or worktree-isolated repositories. The daily workflow is `jackin console` → pick workspace → pick role → agent launches and works autonomously in the background. The operator does not supervise every command. This is by design: jackin' is deliberately a no-interruption, full-permission environment. The agents run with `--dangerously-skip-permissions` / `--dangerously-bypass-approvals-and-sandbox` / `--dangerously-allow-all` at the entrypoint level — these are not flags operators can override per-session. The agents have full shell access plus Docker access via DinD.

**The threat the operator actually cares about** is: "I left an AI agent running against my codebase overnight. What is it allowed to touch that it should not be?" The answer today is: anything in mounted paths, any outbound network, anything in the DinD Docker daemon. The operator wants to reduce that surface without abandoning the reason they use jackin' (autonomous, fast, effective agents).

### Per-control user impact

**Resource limits — `standard` profile, Phase 2**

Impact: universally useful. Zero compatibility breakage. Prevents one agent from exhausting host memory, CPU, or PID table and degrading other containers or the host machine. The operator running three or four agents in parallel against different repos today has no protection against one agent hitting a runaway loop. A declared `memory_max = "8G"` + `pids = 2048` prevents OOM cascades and fork bombs. This is the item with the best effort-to-impact ratio.

**Host Docker socket guard — Phase 1**

Impact: regression test prevents the single most dangerous mistake. The hard rule already exists; the test just makes it machine-enforceable. If a future change accidentally mounts `/var/run/docker.sock` into the role container, the test fails before merge. No user-visible change; pure safety net.

**DinD TLS cert lifecycle — Phase A**

Impact: makes the current security posture auditable. The certs already die with the launch (per-launch generation today, confirmed in code). Emitting `debug_log!("dind", "TLS certs generated, …")` + storing under `/jackin/run/<instance>/dind-certs/` makes this visible and testable. Low effort, real signal for debugging auth issues.

**DinD-free roles — Phase D / `hardened` profile**

Impact: the highest-value item in the entire contract for roles that genuinely do not need Docker. A code review role, a documentation agent, or a read-only analysis role does not need DinD. Skipping the sidecar eliminates the largest blast-radius component: the privileged container that can reach the host kernel on Linux. Without DinD, capability drops on the role container become genuinely meaningful rather than circumventable.

**Who does this help?** Role authors who build purpose-specific restricted roles. The two current built-in roles (`agent-smith` and `the-architect`) both need DinD for Testcontainers, Docker Compose, and inner build workflows. A future `reviewer` or `researcher` role that only reads code and calls the model API does not. The `requires_inner_engine = false` manifest field is what enables that use case.

**`no-new-privileges` — Phase 2, blocked by sudo audit**

Impact when unblocked: prevents the agent from gaining new privileges through setuid/setgid binaries or file capabilities beyond what the container was started with. Marginal benefit while DinD is active (agent can `docker run --privileged` to regain capabilities). Real benefit under `hardened` with DinD disabled. Currently blocked by `NOPASSWD:ALL` sudo in the base image — enabling it today would silently break `apt install`, tool scripts that call `sudo`, and `gh auth` operations that need elevated file access.

**Capability drops — Phase 4, `hardened` profile only**

Impact: only meaningful under `hardened` + DinD disabled. Under `compat`/`standard` with DinD active, a determined agent can run `docker run --privileged` against the DinD daemon and regain any dropped cap inside that inner container. The 8-cap minimum set (`CHOWN`, `DAC_OVERRIDE`, `FOWNER`, `FSETID`, `SETUID`, `SETGID`, `SETFCAP`, `KILL`) is a good starting hypothesis for roles that do not need `MKNOD` (device nodes), `NET_RAW` (raw sockets/ping), `SYS_CHROOT` (chroot calls), or `AUDIT_WRITE` (kernel audit log). Dropping `NET_RAW` is safe for most coding workflows since modern kernels allow ICMP via `ping_group_range` without it.

**Read-only root — Phase 4, `hardened` profile only**

Impact: eliminates the class of attacks where the agent modifies its own runtime binaries or system libraries mid-session. But it is incompatible with `apt install` at runtime: `dpkg` extracts package files to `/usr/bin/`, `/usr/lib/`, `/usr/share/`, etc., which are image-layer paths and cannot be tmpfs without losing the pre-installed binary tree. The tmpfs preset covers `/var/cache/apt/archives/` (download cache) and `/var/lib/dpkg/` (package database), so `apt-get update` works and `apt-get download` works, but `apt install` (extraction step) fails on a read-only root.

**Who does this help?** Roles that are fully pre-built: everything the agent needs is installed in the image at `docker build` time, and the agent never runs `apt install` during a session. For those roles, `--read-only` is meaningful. For the current built-in roles that call `apt install` in hooks or during tasks, `hardened` is incompatible. This is acceptable — `hardened` is an opt-in, and the role's `min_profile = "standard"` declares incompatibility cleanly.

**Network egress — Phase 5**

Impact: high, but the hardest to get right. Blocking outbound connections prevents credential exfiltration over the network — the most realistic post-compromise attack. The complication is DinD: a role container whose egress is blocked can still create inner containers via DinD with unrestricted networking unless the DinD daemon itself is also egress-limited. Any implementation that only filters the role container must explicitly report `partial` enforcement to avoid giving operators false confidence. Under `locked` with DinD disabled, a network deny policy is fully enforceable at the Docker bridge layer.

**`locked` profile — when is it useful?**

The `locked` profile as designed (no DinD, no writable mounts, read-only root, network deny except agent API, no persistent auth) is only usable for a narrow but real class of roles: **pure read-only analysis agents**. An agent that reads code from a `:ro` mount, calls its model API, and outputs analysis (as terminal text or by writing to a single explicitly-writable output path) fits `locked`. Examples: a PR reviewer that reads a diff and posts a comment via the `gh` CLI (but `gh` needs auth, which needs persistent state — so `locked` would need an explicit auth exception), a code quality scanner that reads source and emits a report.

In practice: `locked` is too restrictive for any current built-in role and for most user-authored roles today. Its value is as a future target for specialized read-only roles. Document it clearly so role authors understand what they're designing toward.

**The honest one-sentence summary for users:** The most impactful near-term items are (1) resource limits on `standard`, and (2) `requires_inner_engine = false` enabling DinD-free roles under `hardened`. Everything else is either blocked by the sudo audit, only relevant for purpose-built restricted roles, or largely theater while DinD remains privileged. The hardening contract makes sense structurally but its Phase 2 deliverable (resource limits) is what most operators will actually experience; Phases 3–5 are for specialized roles.

## Capability Taxonomy

Six independent capability dimensions define the full grant space. Each dimension has discrete tiers. A profile bundles a choice across all six. Explicit grants extend any profile on specific dimensions.

### Dimension 1 — Network

| Tier | Docker mechanism | What the agent can do |
|---|---|---|
| `none` | `--network none` | No IP connectivity at all. Only Unix sockets within the container. Cannot reach model API. Only useful for offline filesystem-analysis roles. |
| `api_only` | In-container iptables/ipset allowlist (see below) — requires `NET_ADMIN`+`NET_RAW` implicitly granted | Only explicitly allowlisted domains. Agent can do its job but cannot exfiltrate to arbitrary URLs. Enforcement degrades to `open` with warning if `api_only` is requested but iptables implementation is unavailable. |
| `open` | Default bridge network with external route | Full outbound internet access. No restrictions. |

**`api_only` implementation — in-container iptables.** The Anthropic upstream devcontainer (see [Devcontainer Parity](/reference/roadmap/devcontainer-parity/)) ships a working reference: `init-firewall.sh` sets default DROP policy on OUTPUT, then adds an `ipset` of allowed domains seeded from live DNS lookups. jackin' would ship its own version as a runtime hook with a jackin-specific default allowlist — described below. This approach requires `CAP_NET_ADMIN` and `CAP_NET_RAW` on the role container; these caps are **implicitly granted** as prerequisites whenever `network = "api_only"` is selected, even under `hardened`/`locked` profiles that otherwise drop `NET_RAW`. This is not a contradiction: without these two caps, the iptables-based allowlist cannot be installed and the network grant cannot be enforced.

**`api_only` implementation sketch for jackin' (jackin-aware, configurable):**

```sh
#!/bin/bash
# /jackin/runtime/init-firewall.sh — runs at container start under api_only network grant
# Requires: NET_ADMIN + NET_RAW capabilities, iptables, ipset, jq, curl
set -euo pipefail

ipset create allowed-domains hash:net maxelem 65536

# Agent model API endpoints — auto-populated from resolved JACKIN_ALLOWED_DOMAINS env var
# (set by jackin host based on configured agents + operator allow list)
for domain in $(echo "$JACKIN_ALLOWED_DOMAINS" | tr ',' '\n'); do
    for ip in $(dig +short "$domain" | grep -E '^[0-9]'); do
        ipset add allowed-domains "$ip" 2>/dev/null || true
    done
done

# Apply: default DROP on OUTPUT, allow loopback, established, DNS, allowlisted
iptables -P OUTPUT DROP
iptables -A OUTPUT -o lo -j ACCEPT
iptables -A OUTPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
iptables -A OUTPUT -p udp --dport 53 -j ACCEPT
iptables -A OUTPUT -m set --match-set allowed-domains dst -j ACCEPT

# Self-test (log only; does not abort if --debug not set)
if ! curl -sf --max-time 5 https://api.github.com/zen &>/dev/null; then
    echo "[jackin firewall] WARNING: api.github.com unreachable — check DNS lookup succeeded" >&2
fi
```

`JACKIN_ALLOWED_DOMAINS` is assembled by the host CLI at launch from: (a) all configured agent API endpoints for the selected agents, (b) `github.com` + `api.github.com` if gh auth is forwarded, (c) any operator-declared `network.allow` entries, (d) any role-declared `network.allow` entries. It is injected as a comma-separated env var before the firewall init runs.

**Default jackin allowlist by agent** (seeded into `JACKIN_ALLOWED_DOMAINS`):
- Claude: `api.anthropic.com` (or `ANTHROPIC_BASE_URL` hostname for Z.AI/MiniMax/Kimi providers)
- Codex: OpenAI API base hostname (from Codex config)
- Amp: `ampcode.com`
- Kimi: Kimi service endpoints (from `~/.kimi-code/config.toml`)
- OpenCode: configured provider hostname
- All agents: `github.com`, `api.github.com` (if gh auth forwarded)
- Operator additions: `network.allow = ["registry.npmjs.org", "pypi.org", "crates.io"]` in config/workspace

**The `api_only` tier is fully configurable** — the operator can add or remove domains. jackin does not force a single fixed allowlist (unlike the Anthropic devcontainer which hard-codes Anthropic-specific endpoints). The operator who wants npm access adds `registry.npmjs.org`; the operator who wants strictly model-API-only removes even `github.com`. The default is a reasonable starting point, not the only option.

**Enforcement quality note:** When DinD is active, inner containers created via DinD have their own network path that bypasses the role container's iptables rules. Session contract must report `partial` enforcement in this case. Under `locked`/`hardened` where DinD is disabled, enforcement is full.

Minimum for any agent to function: network must reach at least the agent's model API endpoint. Per-agent required endpoints: `api.anthropic.com` (Claude, or `ANTHROPIC_BASE_URL` override for Z.AI/MiniMax/Kimi via provider system), OpenAI API (Codex), `ampcode.com` (Amp), Kimi service endpoints, configured provider for OpenCode.

### Dimension 2 — Inner Docker (DinD)

| Tier | Docker primitive | What the agent can do |
|---|---|---|
| `none` | No DinD sidecar started; `DOCKER_HOST`, `DOCKER_TLS_VERIFY`, `DOCKER_CERT_PATH`, `JACKIN_DIND_HOSTNAME`, `TESTCONTAINERS_HOST_OVERRIDE` env vars NOT injected; no certs volume | No Docker CLI, no `docker build`, no Docker Compose, no Testcontainers. Agent cannot run any inner containers. |
| `rootless` | `docker:dind-rootless` sidecar (no `--privileged`), cgroup v2 required | Docker available but daemon runs unprivileged. Most Compose/build workflows work. Kernel exploits inside the sidecar do not reach the host as easily as privileged DinD. |
| `privileged` | `docker:dind --privileged` (current default) | Full Docker daemon including `docker run --privileged`, `docker run -v /:/host`. On Linux hosts, privileged DinD is approximately root + kernel namespace access. On macOS (Docker Desktop/OrbStack), escaping reaches the VM, not macOS directly. |

When `inner_docker = "none"` and `requires_inner_engine` is absent or `false` in the role manifest, the entire DinD sidecar block (lines 675–724 in `launch_role_runtime`) is skipped. No cert volume, no DinD container, no five DinD env vars.

### Dimension 3 — User Privilege

| Tier | What the agent can do | Mechanism |
|---|---|---|
| `agent` (no sudo, no-new-privs) | Runs as `agent` user with no privilege escalation path. `sudo` fails. Cannot `apt install`. Cannot write to `/usr/`. | Remove `NOPASSWD:ALL` from Dockerfile; `--security-opt no-new-privileges` |
| `agent_scoped_sudo` | Runs as `agent`; sudo allowed ONLY for an explicit allowlist of commands. E.g. sudo permitted for `/jackin/runtime/init-firewall.sh` and nothing else. | `/etc/sudoers.d/agent: agent ALL=(root) NOPASSWD: /jackin/runtime/init-firewall.sh` + `no-new-privileges` on the remaining process tree |
| `agent_with_sudo` | Runs as `agent`; can sudo any command. Can `apt install`, system writes, arbitrary privilege. Current default. | `NOPASSWD:ALL` in `/etc/sudoers.d/agent` (Dockerfile line 113); no `no-new-privileges` |
| `root` | Full root inside the container. | `--user root` on `docker run` |

**The scoped sudo tier** (`agent_scoped_sudo`) is directly inspired by the Anthropic devcontainer, which scopes sudo to exactly one command: `sudo /usr/local/bin/init-firewall.sh`. jackin' equivalent: sudo scoped to `/jackin/runtime/init-firewall.sh` (needed for `api_only` network enforcement) plus any additional commands the role or operator explicitly names. The `network = "api_only"` grant **implicitly selects `agent_scoped_sudo`** minimum (firewall init must run with elevated privilege). The operator can layer `agent_with_sudo` on top if broader sudo is also needed.

This makes the privilege grant composable: `network = "api_only"` + default user tier → scoped sudo for firewall init only. `network = "api_only"` + `sudo = true` → open sudo AND firewall init.

`agent_with_sudo` is the current default. `agent` and `agent_scoped_sudo` are both blocked by the unconditional `NOPASSWD:ALL` in the base image until the sudo audit (Open Question #5) is resolved. `root` is an explicit operator opt-in.

### Dimension 4 — Filesystem

| Tier | What the agent can do | Docker mechanism |
|---|---|---|
| `read_only` | Cannot write to container root filesystem (image layer). Can write to bind-mounted paths (workspace `:rw`, `/jackin/state`, agent home mounts) and tmpfs paths. Cannot `apt install` — package extraction targets image-layer paths. | `--read-only` + tmpfs preset entries |
| `writable` | Full read-write container root filesystem. `apt install`, modifying system libraries, writing anywhere. Current default. | No `--read-only` flag |

Note: `--read-only` applies only to the overlay image layer. Bind-mounted paths (workspace mounts, `/jackin/run`, `/jackin/state`, agent home credential dirs) are always writable if mounted `:rw` regardless of `--read-only`. Workspace mounts have their own per-mount `readonly` flag independent of this dimension.

### Dimension 5 — Resources

Explicit numeric budgets. No profile provides `unlimited` — every profile should have at least a soft ceiling.

| Field | Docker flag | Purpose |
|---|---|---|
| `memory_high` | `--memory-reservation` | Soft ceiling; triggers throttling before OOM-kill |
| `memory_max` | `--memory` | Hard ceiling; OOM-kill if exceeded |
| `cpus` | `--cpus` | CPU share quota |
| `pids` | `--pids-limit` | Fork bomb prevention; also limits inner container spawning |
| `nofile` | `--ulimit nofile=N:N` | Open file descriptor limit |

Profile defaults provide sensible starting values; role manifest and workspace can override.

### Dimension 6 — Linux Capabilities

Only meaningful under `inner_docker = "none"` (otherwise the agent can regain any cap via `docker run --privileged` against DinD). Default profile values:

| Tier | Capability set |
|---|---|
| `default_set` | Docker's 14-cap default; no explicit `--cap-add`/`--cap-drop`. Applied under `compat`, `standard`. |
| `minimum_set` | `--cap-drop=ALL --cap-add {CHOWN,DAC_OVERRIDE,FOWNER,FSETID,SETUID,SETGID,SETFCAP,KILL}` — 8 caps. Applied under `hardened`, `locked`. |
| `custom` | Explicit per-cap additions via `capabilities.add = ["NET_RAW"]` in role manifest. Added on top of the profile's base cap set. |

## Explicit Capability Grants

Profiles are bundles for common cases. Explicit grants let operators and role authors extend any profile on specific dimensions without choosing a more permissive profile for everything.

Operator-level grants (in `~/.config/jackin/config.toml` or workspace config):

```toml
[docker]
default_profile = "hardened"

[docker.grants]
inner_docker = "rootless"           # allow rootless DinD even under hardened
root_access = true                  # run as root instead of agent user
network = "open"                    # open egress even under hardened
system_writes = true                # writable root filesystem even under hardened
sudo = true                         # agent_with_sudo (full sudo) even under hardened
resource_override = { memory_max = "32G", cpus = 8.0 }

[docker.grants.network]
mode = "api_only"                   # use iptables allowlist
allow = ["registry.npmjs.org", "pypi.org"]   # add domains beyond the default agent-endpoint set
```

Role-level declarations (in `jackin.role.toml`):

```toml
[docker]
min_profile = "standard"
requires_inner_engine = "rootless"    # "none" | "rootless" | "privileged"

[capabilities]
sudo = true                   # requires agent_with_sudo user tier
net_raw = true                # requires CAP_NET_RAW (for tcpdump/nmap roles, beyond the implicit NET_RAW granted by api_only)
system_writes = true          # requires writable filesystem tier

[network]
min_mode = "api_only"         # role requires at least api_only network; open also satisfies
allow = ["crates.io", "static.crates.io"]   # role-declared extra domains added to JACKIN_ALLOWED_DOMAINS
```

**Profile + grant composition rule:** take the profile's per-dimension tier, apply any explicit grants that elevate a dimension, apply implicit cap dependencies (e.g. `api_only` → `NET_ADMIN`+`NET_RAW`), and report the resulting per-dimension effective grant in the session contract. The session contract shows the effective grant set — not just the profile name — so the operator sees exactly what the container can do.

**Conflict resolution:** if an operator grant and a role requirement conflict (operator says `network = "none"`, role declares `network.min_mode = "api_only"`), the launch fails with a clear message: `"role requires network = api_only or higher; workspace override sets network = none — add [docker.grants.network] mode = 'api_only' to your workspace config or choose a less restrictive profile"`. Silent downgrade is never acceptable — the operator must always know what they granted.

**The jackin' philosophy here:** Anthropic's devcontainer hard-codes one fixed allowlist for one fixed use case. jackin' ships the same mechanism but makes every component configurable: the profile sets the baseline, explicit grants add dimensions, the network allow list is assembled from role + operator + agent-endpoint sources, and the operator can always see the result in the session contract before committing. Nobody is forced into one experience — the tool adapts to what the operator actually needs.

## Security Profiles

Profiles are named bundles of capability grants across the six dimensions (network, inner Docker, user privilege, filesystem, resources, Linux capabilities). They exist for convenience — instead of specifying every dimension individually, an operator picks a profile that matches their intent. Explicit grants can extend any profile on specific dimensions.

**The ordering is ascending capability, not descending restriction.** `locked` is the starting point for tightly controlled use cases. `compat` is what you opt into when you need everything. The default profile for a fresh install is `compat` until the sudo audit and compatibility matrix are resolved; the intended eventual default is `standard`.

CLI and config shape:

```sh
jackin load the-architect . --docker-profile standard
jackin load reviewer . --docker-profile hardened
jackin load researcher . --docker-profile locked
jackin load fullstack . --docker-profile compat
```

```toml
[docker]
default_profile = "standard"

[workspaces.my-service.docker]
profile = "hardened"
```

### `locked` — minimal, tightly controlled

Purpose: analysis, review, investigation, or any role where the operator wants maximum confidence about what the agent can and cannot do. The agent can read, think, and call its model API. Everything else requires an explicit grant.

| Dimension | Default grant | What agent can do |
|---|---|---|
| Network | `api_only` | Reach configured model API endpoint only. No arbitrary outbound. |
| Inner Docker | `none` | No `docker` CLI, no DinD sidecar. |
| User privilege | `agent` (no sudo, `no-new-privileges`) | Runs as `agent` user; cannot escalate. No `apt install`. |
| Filesystem | `read_only` | Cannot write outside bind-mounted paths. No system writes. |
| Resources | strict: `memory_max=4G, cpus=2.0, pids=512` | Hard ceiling. |
| Linux capabilities | minimum 8-cap set | No `MKNOD`, `NET_RAW`, `SYS_CHROOT`, etc. |

**Who uses `locked`?** Roles purpose-built for read-only analysis. A PR reviewer that reads diffs and calls the model. A code quality scanner that emits a report. An agent that reads documentation and answers questions. Any role where the operator's guarantee is "this agent cannot change anything on my machine or reach the internet beyond its model API."

**What `locked` requires from the image:** all tools the agent needs must be pre-installed in the role Dockerfile at build time. No runtime `apt install`. No `sudo`. The role's `min_profile` must be `"locked"`.

### `hardened` — restricted, DinD-disabled by default

Purpose: untrusted or unfamiliar role repos, long autonomous runs, and roles that need to do real coding work but do not need inner Docker workflows.

| Dimension | Default grant | What agent can do |
|---|---|---|
| Network | `api_only` + `github.com` | Model API + GitHub. No other outbound. |
| Inner Docker | `none` (override with `requires_inner_engine = "rootless"`) | No DinD by default. Roles that need Docker declare it. |
| User privilege | `agent` (no sudo, `no-new-privileges`) | No sudo. Pre-installed toolchains only. |
| Filesystem | `read_only` + tmpfs preset | Writable workspace mounts + tmpfs system paths. No image-layer writes. |
| Resources | generous: `memory_max=16G, cpus=4.0, pids=2048` | Enough for typical Rust/Node/Python builds. |
| Linux capabilities | minimum 8-cap set | Defense-in-depth when DinD is disabled. |

**Who uses `hardened`?** An agent running against a repo from an unfamiliar source. A long-running autonomous agent where the operator wants network egress limited to the model and GitHub. A role that does all its work through pre-installed tools and does not need to install packages at runtime. When DinD is not needed and the role's image is purpose-built, this profile provides the strongest practical guarantee available while still allowing real coding tasks.

**What `hardened` requires from the image:** same as `locked` — all tools pre-installed. No sudo. If DinD is needed, the role must declare `requires_inner_engine = "rootless"` (cgroup v2 host required) or `"privileged"` (explicit operator grant).

### `standard` — typical dev work

Purpose: the intended eventual default for everyday coding agent sessions once the sudo audit is resolved. Real dev work: editing files, running tests, installing npm/pip/cargo packages, git push, GitHub CLI.

| Dimension | Default grant | What agent can do |
|---|---|---|
| Network | `open` | Full outbound. Agent can reach any URL. |
| Inner Docker | `rootless` (cgroup v2) or `privileged` (cgroup v1 / no rootless support) | DinD for Docker-based workflows. |
| User privilege | `agent_with_sudo` (blocked by sudo audit until resolved) | `sudo apt install`, package installs, system writes. |
| Filesystem | `writable` | Full read-write container root. |
| Resources | moderate: `memory_max=16G, cpus=4.0, pids=2048` | **Required** even if declared by role/workspace. Warn if absent on multi-agent launches. |
| Linux capabilities | Docker 14-cap default | Drops are deferred until test matrix confirms minimum safe set for `standard`. |

**Who uses `standard`?** The operator who trusts the role repo (e.g. built by their team or one of the built-in roles) and wants full dev capability with resource protection. This is jackin's intended everyday profile.

**Current blocker for making `standard` the default:** the `NOPASSWD:ALL` sudo audit (Open Question #5). Until the base image drops unconditional sudo, `standard` cannot enable `no-new-privileges`, which means the user privilege grant cannot be tightened from `agent_with_sudo`. The eventual `standard` default enables `no-new-privileges` once that audit is done.

### `compat` — maximum compatibility, today's behavior

Purpose: roles that need everything: privileged DinD, writable system, sudo, open network, no resource limits. The explicit "I know what I'm doing, give the agent everything" profile. The current default behavior before this contract ships.

| Dimension | Default grant | What agent can do |
|---|---|---|
| Network | `open` | Full outbound. |
| Inner Docker | `privileged` | Privileged DinD sidecar. |
| User privilege | `agent_with_sudo` | `NOPASSWD:ALL` passwordless sudo. |
| Filesystem | `writable` | Full read-write root. |
| Resources | none | No limits. Parallel agents can exhaust host. |
| Linux capabilities | Docker 14-cap default | No drops. |

**Who uses `compat`?** Roles with complex Docker workflows (Docker Compose, Testcontainers, multi-stage `docker build`), roles that run `apt install` during sessions, roles that need privileged operations inside the container, and any operator who wants exact parity with today's behavior. `compat` must be explicitly declared — it is not the intended default after `standard` is validated.

**Explicit compat grant:** An operator can also grant `compat` behavior on specific dimensions without taking the whole profile:

```toml
[docker]
default_profile = "standard"

[docker.grants]
inner_docker = "privileged"   # grant privileged DinD on top of standard
```

## Control Matrix

| Control | Docker primitive | First profile | Notes |
|---|---|---|---|
| Seccomp | `--security-opt seccomp=...` | `compat` | Docker default profile stays on unless explicitly disabled. The default is itself an allowlist that blocks high-risk syscalls while preserving compatibility. |
| AppArmor | `--security-opt apparmor=...` | `standard` | Linux-only at the kernel layer. On macOS through Docker Desktop or OrbStack the role container runs inside the backend's LinuxKit/OrbStack VM, so `docker-default` AppArmor is enforced inside that VM, not by macOS. The launch contract must say which layer enforces it on the active host. |
| SELinux label | `--security-opt label=...` | Future | Relevant on SELinux Linux hosts (Fedora, RHEL, openSUSE Tumbleweed with Crio/Podman backends). Not on macOS. |
| Capabilities | `--cap-drop`, `--cap-add` | `hardened` | Hypothesis grounded in Docker's documented default (`CAP_CHOWN`, `CAP_DAC_OVERRIDE`, `CAP_FSETID`, `CAP_FOWNER`, `CAP_MKNOD`, `CAP_NET_RAW`, `CAP_SETGID`, `CAP_SETUID`, `CAP_SETFCAP`, `CAP_SETPCAP`, `CAP_NET_BIND_SERVICE`, `CAP_SYS_CHROOT`, `CAP_KILL`, `CAP_AUDIT_WRITE` — 14 caps). See Capability Minimum Set below. |
| No privilege escalation | `--security-opt no-new-privileges` | `standard` | Breaks `sudo` (errors with "sudo: effective uid is not 0, is sudo installed setuid root?"). Mitigation: install role-required privilege paths via file caps (`setcap cap_setuid,cap_setgid+ep <binary>`) in the role Dockerfile, or keep `sudo`-using roles on `compat`. |
| Masked paths | implicit, `--security-opt systempaths=unconfined` to lift | `standard` | The default masked set (see Source Material) stays on. `systempaths=unconfined` is only allowed under `compat` and must be reported in the launch contract. |
| Read-only root | `--read-only`, `--tmpfs` | `hardened` | Requires the writable-path inventory (see Read-Only Root Tmpfs Preset). |
| Non-root user | `--user`, image user setup | Existing | Today the role process maps to host UID/GID. Sidecars (DinD especially) must be audited separately — DinD runs as root with `--privileged` today. |
| User namespace remap | `--userns-remap`, `--userns=host` | Future | Daemon-level, not per-container. Incompatible with `--pid=host`/`--network=host`/`--privileged`, and with several volume-driver setups. Out of scope for V1; tracked here so it does not get forgotten. |
| Resource limits | `--memory`, `--cpus`, `--pids-limit`, `--ulimit` | `standard` | Translation depends on cgroups version (see Cgroup Detection). Integrate with [Declarative resource limits](/reference/roadmap/declarative-resource-limits/). |
| Network isolation | per-agent network, `--network none` | Existing | Per-agent network exists. Egress policy is open by default; see [Network egress policy](/reference/roadmap/network-egress-policy/). |
| Host socket exclusion | omit `/var/run/docker.sock` | Existing | Hard rule. Backed by a launch-time guard plus regression test. |
| Private engine | DinD/rootless DinD/Sysbox/future backend | Existing | Keep inner Docker workflows away from the host daemon. |
| Credential location | env/file/brokered auth | Future | Coordinate with [Container credential exposure](/reference/roadmap/container-credential-exposure/) and [Host bridge](/reference/roadmap/host-bridge/). |

### Capability Minimum Set

**Critical constraint: capability drops on the role container are only meaningful under `hardened` profile with DinD disabled.** Under `compat` and `standard` (where DinD is active), the agent process has `DOCKER_HOST` pointing to a privileged DinD sidecar. Even if `CAP_NET_RAW` is dropped from the role container, the agent can run `docker run --privileged` against the DinD daemon and get the full capability set inside an inner container. Capability dropping is therefore a defense-in-depth measure for the `hardened` profile where DinD is absent, not a meaningful constraint for `compat`/`standard` with DinD active. This does not make it worthless — defense-in-depth against prompt injection attacks that try to exploit specific syscalls directly is real — but operators should not treat capability drops as a primary security boundary unless DinD is disabled.

Docker's 14-cap default is the upper bound. The validated jackin' starting set, derived from common role workflows during the May 2026 research pass, is:

| Capability | Why needed | Worflows it unblocks |
|---|---|---|
| `CAP_CHOWN` | Change file owner | `apt`, `npm install`, file extraction |
| `CAP_DAC_OVERRIDE` | Bypass file read/write/exec checks | `apt`, `npm install`, `pip`, `cargo`, `git` |
| `CAP_FOWNER` | Bypass perm checks on file owner | `apt`, `tar`, `unzip` |
| `CAP_FSETID` | Clear setuid/setgid on chown | `apt`, `tar` |
| `CAP_SETUID` | `setuid()` calls | drop privileges, `su`, `gosu`, `runuser` |
| `CAP_SETGID` | `setgid()` calls | drop privileges, group ops |
| `CAP_SETFCAP` | Set file caps | `apt`/`dpkg` (sets file caps on installed binaries) |
| `CAP_KILL` | Signal other procs | shells, process supervisors, `bash -c "cmd & wait"` |

Drop from Docker default in the jackin' starting set:

| Capability | Reason to drop |
|---|---|
| `CAP_MKNOD` | Role containers do not create device nodes; jackin' provides `/dev` via Docker defaults. |
| `CAP_NET_RAW` | Allows raw socket creation (`ping`, `tcpdump`). Modern Linux exposes ICMP via `ping_group_range` sysctl without raw sockets. Keep behind an opt-in `network.allow_raw_sockets = true` for roles that need `tcpdump`/`nmap`. |
| `CAP_SETPCAP` | Allows changing other procs' caps. Not needed once jackin' itself sets the cap set at exec. |
| `CAP_NET_BIND_SERVICE` | Allows binding privileged ports (`<1024`). Role containers should bind unprivileged ports inside the container; host port mapping is unaffected. |
| `CAP_SYS_CHROOT` | `chroot()` calls. Not needed for normal role workflows. |
| `CAP_AUDIT_WRITE` | Write to kernel audit log. Not needed; reduces audit-stream noise. |

Result: 8-cap default (down from 14). This is the V1 hypothesis; roles that fail with the smaller set declare `min_profile = "standard"` or `min_profile = "compat"` in their manifest, and `hardened` either drops the role's per-workflow caps to the same set or refuses to launch with a clear message.

DinD inside the role container is a separate concern: BuildKit and Compose empirically require near-full default set plus `CAP_SYS_ADMIN` (for mount namespacing) and `CAP_NET_ADMIN` (for bridge setup). DinD is therefore gated behind `compat` and `standard` by default, and explicitly opt-in under `hardened`.

### Read-Only Root Tmpfs Preset

The minimum writable-path inventory for `--read-only` with `--tmpfs`, derived from the Linux base-image + common package managers:

```text
/tmp                       — POSIX requirement
/run                       — runtime daemons, pid files
/var/run                   — typically a symlink to /run, but explicit on some images
/var/tmp                   — long-lived tmp
/var/cache                 — apt/dpkg/pip
/var/log                   — process logs that some tools (rsyslogd, systemd-journald, agents) write
/var/lib/apt/lists         — apt-get update writes here
/var/cache/apt/archives    — apt-get install downloads here (download phase only)
/var/lib/dpkg              — dpkg state, including /var/lib/dpkg/lock-frontend
$HOME/.cache               — pip, cargo, npm, fastly tools
```

Each path becomes a `--tmpfs <path>:rw,nosuid,nodev,size=<sized>` flag. Sizes need profile-level defaults plus a per-role override. `/tmp` and `$HOME/.cache` are the largest expected consumers (npm/yarn cache, pip wheel cache).

**`apt install` is NOT compatible with read-only root.** The download phase (`apt-get download`, fetching `.deb` files) works because it only writes to `/var/cache/apt/archives/` and `/var/lib/dpkg/`. The extraction phase (`dpkg --unpack` + `dpkg --configure`) writes to `/usr/bin/`, `/usr/lib/`, `/usr/share/`, `/etc/`, and other FHS paths under the image layer. These paths are not in the tmpfs preset because making them tmpfs would shadow the pre-installed binary tree and make tools installed at `docker build` time invisible. Therefore, under `--read-only`, `apt install <pkg>` fails when `dpkg` tries to extract to `/usr/`. This is an expected and accepted tradeoff for `hardened` — roles that need `apt install` at runtime declare `min_profile = "compat"` or `min_profile = "standard"`. Only roles that pre-install everything in the role Dockerfile can use `hardened` with read-only root.

**Shell history and runtime home paths.** The `agent` user's home directory (`/home/agent/`) has several subdirectories that are bind mounts (agent credential files seeded by `jackin-capsule runtime-setup`) and several that are image-layer paths (oh-my-zsh, `.zshrc`, `.config/`, etc.). Under `--read-only`, shell history (`~/.zsh_history`, `~/.bash_history`), zsh session files (`~/.zsh_sessions/`), and oh-my-zsh update markers will fail to write. These need either a tmpfs entry (acceptable for ephemeral zsh session state) or a bind mount (for persistent history). Shell history persistence as a feature is deferred — see [Devcontainer Parity Finding 4](/reference/roadmap/devcontainer-parity/) and [Déjà Vu](/reference/roadmap/deja-vu/) for the full discussion. For Phase 4 read-only root, the minimum fix is adding these to the tmpfs preset so container startup does not break: `$HOME/.zsh_sessions`, `$HOME/.zsh_history`, `$HOME/.bash_history`. An explicit audit of every write target in `$HOME` is required before Phase 4 ships.

**`jackin-capsule runtime-setup` writes under read-only root.** The capsule runtime setup (in `crates/jackin-capsule/src/runtime_setup.rs`) writes auth credential files (`~/.claude/`, `~/.claude.json`, `~/.codex/auth.json`, `~/.local/share/amp/secrets.json`, `~/.kimi-code/`, `~/.local/share/opencode/`) and git hook markers (`/jackin/state/container-init.done`, `/jackin/state/git-hooks/`). All of these are on bind-mounted paths (either the per-container home state volume or `/jackin/state/`), so they are writable regardless of `--read-only`. No capsule changes needed for read-only root support.

Paths that should NOT be tmpfs even under `hardened`:

- jackin'-owned persistent state mounts (under `/jackin/state/` and any agent-home state bind mounts) — explicit volume mounts, already writable.
- `/jackin/run` — already a bind mount at `launch.rs:1007–1008`; adding it as tmpfs would shadow the bind mount. Do not include in the tmpfs preset.
- Workspace mounts — bind mounts from the host, not tmpfs.

Under `locked` the inventory is smaller because the agent is not expected to install packages: `/tmp`, `/run`, `/var/run` only (plus the shell history entries).

### Cgroup Detection

Resource limits translate differently across cgroups v1 and v2. Detection at jackin' launch time:

```sh
# v2 if cgroup2fs; v1/hybrid if tmpfs
stat -fc %T /sys/fs/cgroup/
# or: read /proc/self/cgroup — v2 has a single line starting "0::/..."
```

| jackin' contract | cgroup v2 mapping | cgroup v1 mapping |
|---|---|---|
| `memory_high` | `--memory-reservation` → `memory.high` (soft throttle, no OOM-kill) | not supported; downgrade to `--memory-reservation` → `memory.soft_limit_in_bytes` and warn that throttling differs |
| `memory_max` | `--memory` → `memory.max` | `--memory` → `memory.limit_in_bytes` |
| `cpus` | `--cpus` → `cpu.max` | `--cpus` → `cpu.cfs_quota_us`/`cpu.cfs_period_us` |
| `pids` | `--pids-limit` → `pids.max` | `--pids-limit` → `pids.max` (controller may not be delegated in rootless v1) |
| `nofile` | `--ulimit nofile=N:N` | same; LimitNOFILE differs by host |

`hardened` and `locked` should require cgroups v2 on Linux. On macOS, Docker Desktop and OrbStack both expose v2 inside their VMs by default. Older Linux hosts on v1/hybrid should get a `debug_log!("launch", "cgroup v1 detected, downgrading memory_high to soft limit")` line and degrade gracefully under `standard`, but fail-closed under `hardened`/`locked`.

## DinD Hardening Track

DinD is the highest-priority Docker-specific risk because jackin' keeps Docker available inside the agent environment.

### Phase A — make the current DinD contract explicit

- Report sidecar image, privilege status, transport, cert volume, and network in the launch/session contract.
- Ensure all docs say TLS-authenticated DinD, not plain TCP.
- Keep the host Docker socket blocked from the role container.
- Add regression tests that fail if the host socket is mounted into either the role container or the DinD sidecar by accident.

DinD TLS reality check (May 2026): the upstream `_/docker` image generates `ca/`, `server/`, `client/` cert directories under `DOCKER_TLS_CERTDIR` on first boot. Docker ships no rotation, no revocation, and no CA-management tooling — `docs.docker.com/engine/security/protect-access/` explicitly warns "Using TLS and managing a CA is an advanced topic" and gives no guidance beyond that. jackin' inherits a per-launch self-signed CA today and never rotates. If a DinD client cert leaks (e.g. via `docker inspect` on a misconfigured role), the leaker has full daemon control inside the sidecar — and with `--privileged` the sidecar has kernel access to the outer host. Phase A scope:

- regenerate certs on every launch (already true today; document it)
- store cert volume under `/jackin/run/<instance>/dind-certs/` per the container path convention so `purge` removes them
- emit `debug_log!("dind", "TLS certs generated, lifetime=<launch>")` so the operator can trace the cert lifecycle
- do not invest in CA rotation tooling under Phase A — the certs already die with the launch

### Phase B — evaluate `docker:dind-rootless`

Known-good upstream limits to design against (per `docs.docker.com/engine/security/rootless/troubleshoot/`):

- Storage drivers supported in rootless: `overlay2` (kernel 5.11+), `fuse-overlayfs` (4.18+), `btrfs`, `vfs`. Other drivers fail-closed.
- Explicitly unsupported: AppArmor, checkpointing, overlay networks, exposing SCTP ports.
- Cgroups: resource limits delegated only on cgroup v2 + systemd; only memory + pids controllers by default.
- NFS as data-root unsupported.
- Privileged ports (`<1024`) require explicit setup (`/etc/sysctl.d/87-net.ipv4.ip_unprivileged_port_start.conf` or systemd `AmbientCapabilities=CAP_NET_BIND_SERVICE`).

Research questions remaining:

- Can role Dockerfiles build under rootless DinD? Most Dockerfiles should; ones that need `--add-host`, custom sysctls, or non-`overlay2` drivers will not.
- Does Java Testcontainers work against rootless DinD inside the role container? Reach-out path: try with `testcontainers-java` against the `testcontainers/ryuk` reaper, which has historically had rootless issues.
- Do BuildKit, Compose, bind mounts, and container networking behave acceptably? BuildKit yes (rootless-supported upstream), Compose yes for v2.
- What extra writable paths or sysctls are required? `subuid`/`subgid` maps must exist for the role user.
- Is startup time acceptable for interactive `jackin load`?

Known limitations from `moby#42910` (cgroup-v2 issues) and `docker-library/docker#451` (nested rootless): host-rootless + dind-rootless on cgroup v1 fails entirely. Decision pivot: rootless DinD is only viable on cgroup-v2 hosts. Document the prereq, do not silently downgrade.

Decision rule:

- If rootless DinD passes normal role workflows on cgroup v2 hosts, make it the `standard` DinD implementation on those hosts.
- On cgroup v1 hosts, keep privileged DinD under `standard` and gate `hardened` behind an upgrade path.
- If it only passes some workflows, make it opt-in under `hardened`.
- If it breaks common workflows even on cgroup v2, keep it documented as unsupported and revisit only when Docker changes.

### Phase C — evaluate Sysbox on Linux

Sysbox is a Linux-only container runtime that can run Docker-in-Docker style workloads without the usual privileged-container model. It is not a microVM.

Research questions:

- Can jackin' select Sysbox only for the DinD sidecar?
- Does it require host runtime installation that violates the "never mutate host silently" rule?
- Does it work with the selected Docker contexts jackin' supports?
- Is it useful enough if it cannot work on macOS?

Decision rule:

- Treat Sysbox as a Linux operator opt-in, not a default dependency.
- Surface any host-side installation requirement before launch.

### Phase D — support DinD-free roles

Some roles do not need Docker inside the sandbox. For those roles, `hardened` should skip the sidecar entirely.

Suggested role manifest shape for future design:

```toml
[runtime.docker]
requires_inner_engine = false
```

If this field lands in `jackin.role.toml`, it is a versioned manifest schema change and must ship with a migration.

## Filesystem Policy

Profiles should separate three filesystem concepts:

- host mounts, controlled by workspace/global mount config
- role container root filesystem, created from the role image
- jackin'-owned runtime/state paths inside the container

Rules:

- Host mounts remain explicit. No profile may add host mounts silently.
- `:ro` host mounts stay the primary read-only guarantee.
- `hardened` and `locked` should use read-only role root filesystems.
- Writable paths for jackin'-owned runtime state must remain under `/jackin/`.
- Temporary writable paths must be explicit tmpfs mounts.
- Persisted agent state must remain easy to purge and visible in the session contract.

Likely writable paths for read-only root evaluation:

```text
/tmp
/var/tmp
/run
/jackin/run
/jackin/state
agent home state mounts
```

The exact list must come from a compatibility test matrix, not guesswork.

## Decided In This Pass

These are the V1 defaults the contract commits to. Code work that implements them does not need to re-debate the choice:

- **Model: capability grants, not restrictions.** Profiles are capability bundles. `locked` is the tightest; `compat` grants everything. Explicit `[docker.grants]` extend any profile on specific dimensions. Operators grant up, not restrict down.
- Profile enum is `locked | hardened | standard | compat` (ordered ascending capability). The conceptual default from the user's perspective is `locked` as the starting point; the practical default while `standard` is not yet validated is `compat`.
- **`compat` is the initial default for all launches** until the sudo audit (Open Question #5) is resolved and the compatibility test matrix passes. `standard` becomes the default after that. Phase 1 ships only the enum + `compat`-only behavior — no functional change for any user, just bookkeeping.
- **Profile + explicit grant composition:** take the profile's per-dimension tier, apply any explicit grants that elevate a dimension, report the effective per-dimension grant in the session contract. Conflict (operator restricts a dimension below what a role declares minimum) = launch failure with a named-conflict message.
- **Ascending-capability interpretation:** `locked` is not "locked down from compat"; it is "explicitly granted only what's needed for model-API-only agent work." The operator who adds `inner_docker = "privileged"` to a `locked` profile is granting one dimension beyond the locked bundle, not weakening a restriction.
- Capability minimum hypothesis for `hardened`/`locked`: 8 caps (`CHOWN`, `DAC_OVERRIDE`, `FOWNER`, `FSETID`, `SETUID`, `SETGID`, `SETFCAP`, `KILL`). Caps only matter when DinD is disabled — under `hardened`/`locked`, DinD is `none` by default. Validate against the test matrix before the 8-cap set is considered confirmed.
- `no-new-privileges` is required under `hardened` and `locked`. Under `standard` it is the goal but blocked by the sudo audit. Under `compat` it is not applied.
- `locked` and `hardened` default network grant is `api_only`. `api_only` requires the allowlist proxy implementation (tracked in [Network egress policy](/reference/roadmap/network-egress-policy/)). Until that is implemented, `locked`/`hardened` network grant degrades to `open` with a visible warning in the session contract — launching with an unimplemented grant is better than refusing to launch.
- Masked-paths default stays on for all profiles; `systempaths=unconfined` allowed only under `compat` and reported.
- Read-only root: only for `locked`/`hardened`, only for purpose-built images (pre-installed tools, no runtime `apt install`). The `apt install` + read-only root conflict is a feature: if the role needs `apt install`, it declares `min_profile = "standard"` and gets a writable root.
- Cgroup v2 required for rootless DinD under `hardened`/`standard`; privileged DinD falls back on cgroup v1 for `standard`/`compat`. `hardened` fails-closed on cgroup v1 if rootless DinD is the only option and rootless fails.
- DinD TLS certs regenerate per launch under `/jackin/run/<instance>/dind-certs/`. No CA rotation under V1 — certs already die with the launch.
- Host Docker socket exclusion is a hard rule; the contract adds a launch-time regression test.
- Profile selection precedence: CLI flag > workspace override > role manifest `min_profile` > global default. CLI cannot go below `min_profile` without explicit operator grant override.
- **The implementation priority order, grounded in user impact:** (1) resource limits (`standard` profile, Phase 2) — universally useful, zero compatibility issues, real protection for parallel agent users; (2) DinD-free roles (`requires_inner_engine = "none"`, Phase 3) — enables `hardened`/`locked` to be meaningful; (3) read-only root + capability minimum set (Phase 4) — for purpose-built roles after DinD is disabled; (4) network allowlist (`api_only` tier, Phase 5) — completes the `locked` profile's network grant. Phase 1 is infrastructure only.

## Open Before Implementation Can Start

These need answers before Phase 1 code work begins:

1. **Test fixture matrix.** Which exact agent runtimes + workflows are the acceptance set? (claude-code, codex, amp, kimi, opencode × `apt install build-essential`, `npm install <pkg-with-native-deps>`, `pip install <wheel>`, `cargo build`, `git clone+push`, `docker build` in-DinD, Compose, Testcontainers, `gh pr create`). Without this, "validate the cap set" has no objective answer.
2. **`min_profile` field shape.** Role manifest needs a way to say "I need at least `compat`" or "I am `locked`-compatible". Open: enum vs ordered list vs constraint expression. Lowest-friction choice: an ordered enum field, default `standard`. The `DockerSecurityProfile` enum in `src/runtime/docker_profile.rs` (see Codebase Map) should derive `PartialOrd`/`Ord` so precedence is `max(workspace_override, role_min_profile)`.
3. **AppArmor on macOS launch contract.** What exact text does jackin' print under `standard` when running on Docker Desktop or OrbStack? Current code does not know which backend Docker is using; needs a probe (`docker info --format '{{.OperatingSystem}}'` or `docker info --format '{{.SecurityOptions}}'`) at launch. The probe result flows into the session contract output and the `debug_log!("launch", "apparmor available={} layer=…")` telemetry line. The `clap` definition for `--docker-profile` will live in `LoadArgs` in <RepoFile path="src/cli/role.rs" />.
4. **Sysbox host-install policy.** Sysbox needs a host-side runtime registration that violates the "never mutate host silently" rule. Either operator-confirmed opt-in install or hard-skip on hosts without Sysbox. Decide which before Phase C.
5. **`NOPASSWD:ALL` sudo audit.** Before `no-new-privileges` can be applied to any built-in agent runtime, audit every `sudo` call site in <RepoFile path="docker/construct/Dockerfile" /> and all role Dockerfiles. The base image adds `NOPASSWD:ALL` at line 113. With `--security-opt no-new-privileges`, `sudo` fails because setuid elevation is blocked at the kernel level (`sudo: effective uid is not 0, is sudo installed setuid root?`). Resolution path: identify each privileged binary that roles need and replace the `sudo` call with a file capability (`setcap cap_setuid,cap_setgid+ep <binary>`) in the role Dockerfile. Until the audit is complete, `standard` profile cannot safely enable `no-new-privileges` for built-in agent images. Custom role authors who control their own Dockerfile can opt in earlier by declaring `min_profile = "standard"` after removing `sudo` from their image.
6. **Two-Docker-path sync policy.** The production role container uses the CLI subprocess path (`run_args: Vec<&str>` in `launch_role_runtime`); the Bollard `ContainerSpec` / `create_container` path is used by tests and restore code. Both must be kept in sync. Decide whether to: (a) keep both paths and add `ContainerSpec` security fields mirroring the CLI path (more maintenance, but tests can exercise security config), or (b) have the Bollard path defer to the same helper that builds CLI flags (eliminates duplication). Option (b) is cleaner but requires refactoring the Bollard path to accept a pre-built `HostConfig` rather than a `ContainerSpec`. Option (a) is the lower-risk Phase 1 choice.
7. **`/jackin/run` tmpfs exclusion.** The socket-dir bind mount `-v <socket_dir>:/jackin/run` is appended at `launch.rs:1007–1008` and provides write access to `/jackin/run` through the bind mount, not a tmpfs. The read-only root tmpfs preset listed above correctly excludes `/jackin/run`. Confirm before Phase 4 that no other path under `/jackin/` needs an explicit tmpfs entry — the bind mounts for `/jackin/state`, `/jackin/claude/`, `/jackin/codex/` etc. already provide write access to those subtrees.
8. **$HOME write-path audit required before read-only root.** The `agent` user's home directory mixes bind-mount paths (writable regardless of `--read-only`) and image-layer paths (blocked under `--read-only`). Before Phase 4 ships, enumerate every runtime write target under `/home/agent/` that is NOT a bind mount: shell history (`.zsh_history`, `.bash_history`), zsh session files (`.zsh_sessions/`), oh-my-zsh update markers, any `.gitconfig` local overrides, and `.local/share/mise/` shim paths if `mise` activates at runtime. Each needs either a tmpfs entry or a per-container state bind mount. `jackin-capsule runtime-setup` writes are all on bind-mount paths (`.claude/`, `.codex/`, `.local/share/amp/`, etc.) and are fine.
9. **Agent bypass flags are not configurable per profile.** Every agent binary is launched with its maximum-bypass flag in <RepoFile path="docker/runtime/entrypoint.sh" /> (`claude --dangerously-skip-permissions`, `codex --dangerously-bypass-approvals-and-sandbox`, `amp --dangerously-allow-all`, `kimi --yolo`, `opencode` with `OPENCODE_CONFIG_CONTENT='{"permission":"allow"}'`). These are hard-coded in the entrypoint — not configurable per Docker profile. Any security model that assumes the agent binary enforces its own sandboxing is incorrect. The Docker boundary is the only enforceable isolation layer. This is intentional design (jackin' is a no-interruption autonomous environment), but it must be stated clearly in the session contract output so operators understand the trust model: they are trusting the Docker isolation, not the agent's own permission system.

## Risks and Design Traps

- Overstating security. Hardened Docker is still a shared-kernel boundary.
- Breaking role author workflows by making strict profiles the default too soon.
- Adding flags without session-contract reporting, which makes the policy invisible and hard to debug.
- Filtering only the role container's network while inner DinD containers still have open egress.
- Treating Docker Desktop, OrbStack, and Linux Docker as if they expose the same security controls.
- Mutating host Docker daemon settings to make a profile pass.
- **CLI path and Bollard path diverging.** Security flags are built in two separate places (`run_args` vec in `launch_role_runtime` for production, `HostConfig` in `create_container` for the Bollard path). If one is updated and the other is not, tests pass against a different security posture than production runs. Enforce this via a shared `docker_profile.rs` helper that both paths call, or add an integration test that exercises both paths with an explicit non-default profile.
- **`NOPASSWD:ALL` sudo + `no-new-privileges` silent failure.** If `standard` profile enables `no-new-privileges` before the base image removes `NOPASSWD:ALL` sudo, every `sudo` call inside the container silently fails with a non-obvious error. This is worse than the feature not being there — the container starts successfully but tools that call `sudo` (apt, dpkg post-installs, npm scripts) fail at runtime with confusing permission errors. The `standard` profile must not enable `no-new-privileges` until the base-image audit in Open Question #5 is resolved.
- **Profile precedence not enforced before launch.** If the precedence logic (`CLI > workspace > role min_profile > global default`) is only applied after some early-exit path (e.g., image not found, trust gate rejection), operators may see an inconsistent profile in logs vs what actually ran. Resolve the effective profile once at the start of `load_role_with` and record it in `LaunchContext` before any early exits.
- **`docker info` probe adds latency to every launch.** The AppArmor / cgroup version probe requires a `docker info` call which can take 100–500 ms on some Docker Desktop configurations. Cache the result per-session or run it in parallel with the image pull check.
- **`run_args` tests brittle to flag ordering.** Many existing tests (search `docker run` in `launch.rs` test section starting ~line 5200) assert exact `run_args` contents. Adding new security flags changes arg ordering and will break these tests. Budget time for test updates in each phase.
- **`/jackin/run` bind-mount before read-only flag.** Docker processes bind mounts and the `--read-only` flag together. The socket-dir bind mount at `launch.rs:1008` must remain a bind mount (not tmpfs) even under `hardened` because the capsule daemon socket lives there. Confirm that Docker's behavior with `--read-only` + explicit bind mounts does not prevent writes to those bind-mount paths (it does not — `--read-only` applies only to the image layer, not bind mounts — but document this explicitly so future maintainers don't add an unnecessary tmpfs entry for `/jackin/run`).
- **DinD cert volume and `--read-only`.** The DinD sidecar mounts its cert volume at `/certs/client` (from `launch.rs:688`). The role container mounts the same volume read-only at `/certs/client:ro`. Under `hardened`, the DinD sidecar itself is disabled by default, so this is moot. Under `standard`, the cert volume mount at `/certs/client:ro` in the role container is already read-only — no change needed there.

## Lessons from Anthropic's Claude Code Devcontainer

The [devcontainer-parity](/reference/roadmap/devcontainer-parity/) roadmap item documents a side-by-side analysis of jackin' construct vs the upstream [anthropics/claude-code `.devcontainer`](https://github.com/anthropics/claude-code/tree/main/.devcontainer). Several of those findings directly inform this contract. Key extractions:

**What the upstream devcontainer proves works (and jackin' makes configurable instead of fixed):**

| Upstream choice | What it solves | jackin' equivalent | jackin' difference |
|---|---|---|---|
| `iptables`/`ipset` OUTPUT allowlist in `init-firewall.sh` | Prevents arbitrary outbound from agent | `network = "api_only"` tier in capability taxonomy | Configurable: operator adds/removes domains; default list is assembled from configured agent endpoints; fully toggleable per profile/workspace/role |
| `--cap-add=NET_ADMIN --cap-add=NET_RAW` in `devcontainer.json` | Required for iptables init | Implicit cap grants when `network = "api_only"` is selected | Surfaced in session contract; not hidden; reverts to default cap set if `api_only` is not selected |
| Sudo scoped to `/usr/local/bin/init-firewall.sh` only | Firewall init needs root; agent should not | `agent_scoped_sudo` user privilege tier; `network = "api_only"` implicitly selects it | Fully toggleable; operator can layer `sudo = true` on top for broader sudo if the role needs it |
| Hard-coded allowlist (Anthropic endpoints, GitHub, npm, Sentry, VS Code) | Catches 90% of real Claude Code use cases | Per-session assembled `JACKIN_ALLOWED_DOMAINS` from agent endpoints + operator config + role config | Not hard-coded; adapts to which agents are running and what packages/services the role needs |
| Named Docker volumes for `~/.claude` and shell history | State survives container recreation | `~/.jackin/data/{name}/home/.claude` bind mount (existing). Shell history persistence is a separate concern — see note below. | `~/.claude` bind mount already survives container recreation and eject. Shell history and **agent conversation history** are different in importance: conversation history is tracked by [Déjà Vu](/reference/roadmap/deja-vu/). |
| Default DROP OUTPUT policy before allowlist is applied | Fail-safe: if script fails, no traffic leaves | Same posture: iptables -P OUTPUT DROP runs before any ipset additions | Same semantics; if firewall init fails, container should surface the error (jackin-capsule can detect exit status) |

**What the upstream devcontainer does NOT do that jackin' provides:**
- DinD for inner Docker workflows
- Per-agent namespaced networks (each agent isolated)
- Provider routing (Z.AI, MiniMax, Kimi via provider system)
- Multi-agent, workspace-level configuration
- Configurable profiles instead of a fixed setup

**Key philosophical difference:** the upstream devcontainer makes one set of choices for one use case (Claude Code in VS Code/Codespaces). Those choices are correct for that use case. jackin' operators have different use cases — some want `open` network for unrestricted development, some want `api_only` for suspicious repos, some want `none` for offline analysis. jackin does not force a single experience; it provides the mechanism and lets the operator choose. The upstream devcontainer is a valuable proof of concept for the `api_only` + `agent_scoped_sudo` combination specifically.

**The `init-firewall.sh` approach is the V1 implementation of `api_only`.** It is simpler than a proxy sidecar (no additional container, no TLS interception), proven in production use at Anthropic's scale, and directly implementable as a jackin runtime hook. The proxy sidecar approach (tracked in [Network egress policy](/reference/roadmap/network-egress-policy/)) is the V2 implementation that covers non-HTTP protocols and provides connection logs.
