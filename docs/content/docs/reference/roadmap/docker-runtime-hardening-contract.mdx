**Status**: Partially implemented. The profile/grant model, per-launch flag emission (capabilities, read-only root, resources, `no-new-privileges`), and the config/workspace/manifest schema shipped in PR #528 behind the `compat` default — no behavior change for existing launches. The egress-allowlist enforcement, the launch observability surface (session contract + AppArmor/cgroup probes + full telemetry), and the DinD-reduction track are **not yet wired**. Remaining work is below.

This item tracks only undelivered work. Operator-facing detail for what shipped lives in the [Docker Security Profiles guide](/guides/docker-profiles/), the [role manifest reference](/developing/role-manifest/), [Configuration](/reference/runtime/configuration/), and [Schema Versions](/reference/runtime/schema-versions/).

## Decisions (finalized)

These are agreed and drive the plan below. Do not re-litigate without flagging it.

1. **Land everything wired, enforce nothing by default.** Every feature below ships fully wired but takes effect only when an operator opts into a profile. The `compat` default stays a verified no-op, so no existing user's workflow breaks. This is the core guarantee for the whole effort.
2. **PR #528 ships the enforcement bundle, not a skeleton.** #528 does not merge until the opt-in features actually work end to end. Scope: WP0 + WP1 + WP2 + WP3 + WP4 in one PR.
3. **Fail-closed on firewall failure.** Under `hardened`/`locked`, if `firewall-apply` cannot install the egress allowlist, the launch aborts and the container is torn down with an actionable error. The agent never runs with open egress when the operator asked for a boundary.
4. **`standard` becomes the default eventually — not now.** The long-term goal is security-by-default (`standard`: resource limits + `no-new-privileges` on). That flip is the one change that alters default behavior, so it is deferred behind the base-image `NOPASSWD:ALL` sudo audit (WP6). Until then `compat` stays the default.
5. **Deferred out of #528:** credential env-only posture (WP7 → owned by [Container credential exposure](/reference/roadmap/container-credential-exposure/)), rootless DinD / Sysbox research (WP5), and the `standard` default flip (WP6). Each lands as its own follow-up; none blocks #528.
6. **Docs never describe unshipped behavior.** Every WP updates its own docs in the same commit, and a final claim-by-claim audit (WP-DOC) blocks merge. The user-facing guide must not present egress enforcement, the session contract, or telemetry as shipped until the code that backs them lands, must mark deferred work as deferred, and must not leak internal paths or Rust symbols (three-audience rule).

## What shipped (PR #528)

Baseline, so the remaining work below has context. Do not re-document the mechanics here — link to the canonical pages above.

- `DockerSecurityProfile` (`locked | hardened | standard | compat`, ascending capability) with full precedence resolution: CLI `--docker-profile` > workspace `[docker] profile` > config `[docker] profile` > compiled-in `compat` default.
- `[docker.grants]` (config + workspace) and role manifest `[docker]` (`min_profile`, `dind`, `allowed_hosts`, `capabilities_add`), with grant validation, per-dimension layering, `min_profile` launch gate, and DinD downgrade.
- Per-launch Docker flag emission for `hardened`/`locked`: `--cap-drop=ALL` + 8-cap minimum set, `--read-only` + tmpfs preset, `--memory`/`--memory-reservation`/`--cpus`/`--pids-limit`/`--ulimit`, `--security-opt no-new-privileges`. `network = "none"` grant emits `--network none`.
- DinD privileged sidecar with TLS certs under `/jackin/run/dind-certs`.
- Schema: config + workspace `v1alpha7`, manifest `v1alpha6`, with migrations and fixtures.
- A single launch telemetry line (`debug_log!("launch", "docker profile=… source=… dind_enabled=… network=…")`).

The `compat` default is a verified no-op: it emits no new Docker flags, so any launch without explicit profile/grant configuration behaves exactly as before this PR.

## Implementation Plan

Detailed, sequenced plan for the undelivered work. Each work package (WP) lists the concrete integration points (file + function), the changes, schema/env additions, tests, telemetry, and exit criteria. The PR breakdown and ordering are at the end.

### Sequencing rationale

The order is driven by dependencies and by user-visible risk, not by the original Phase 1–5 numbering:

1. **WP1 (egress enforcement)** is first because it closes the one active correctness gap: `hardened`/`locked` advertise an allowlist that does not exist. Until it lands, those profiles give a false sense of isolation. Everything else is additive polish on top of already-correct behavior.
2. **WP2 (`internal` network for `locked`)** is a small, self-contained hardening that pairs naturally with WP1 (both touch the network path) and can ship in the same PR.
3. **WP3 (observability)** comes next because once enforcement exists, operators need to *see* what was applied — and the session contract is the surface that tells them the difference between `full` and `partial` enforcement that WP1 introduces.
4. **WP4 (DinD-free auto-skip)** is independent and can land any time after WP1; it makes `hardened` meaningfully lighter and improves WP1's enforcement label (no DinD ⇒ `full` instead of `partial`).
5. **WP5 / WP6** are research/blocked and must not block the above.
6. **WP0 (test matrix harness)** is cross-cutting and should land first or alongside WP1, because every later WP cites it as its acceptance gate.

```text
WP0 (test matrix harness) ─┬─> WP1 (egress) ─┬─> WP2 (internal net)   [same PR as WP1]
                           │                 └─> WP3 (observability)
                           ├─> WP4 (DinD-free auto-skip)
                           ├─> WP6 (default flip) ── blocked on sudo audit
                           └─> WP5 (rootless DinD / Sysbox) ── research
                                WP7 (credential posture) ── after WP3
```

### WP0 — Acceptance test matrix harness (cross-cutting, land first)

**Goal:** an objective, repeatable way to answer "does profile X still run workflow Y" so every later WP has a pass/fail gate instead of a judgment call.

**Changes:**
- Define the acceptance set as data: agent runtimes (`claude`, `codex`, `amp`, `kimi`, `opencode`) × workflows (`apt install build-essential`, native-dep `npm install`, `pip install <wheel>`, `cargo build`, `git clone+push`, in-DinD `docker build`, Compose, Testcontainers, `gh pr create`).
- Add a gated integration test (feature flag or `JACKIN_E2E=1`, like the existing `dind_e2e`) that launches each (profile, workflow) cell and asserts success/expected-failure. `locked`/`hardened` rows that are *expected* to reject a workflow (e.g. `apt install` under read-only root) assert the failure is the documented named error, not a crash.
- Record the matrix outcome in a checked-in table so "the 8-cap minimum is validated" has a dated artifact.

**Exit criteria:** matrix runs in CI (or a documented manual gate) and every later WP references the relevant rows.

### WP1 — Network egress allowlist enforcement (highest priority)

**Goal:** make `network = allowlist` actually restrict egress. Today the allowlist is computed into `EffectiveGrants.allowed_hosts` and then dropped on the floor.

**Integration points:**
- <RepoFile path="crates/jackin-core/src/env_model.rs" /> — add `JACKIN_NETWORK_MODE_ENV_NAME = "JACKIN_NETWORK_MODE"`, `JACKIN_ALLOWED_HOSTS_ENV_NAME = "JACKIN_ALLOWED_HOSTS"`, `JACKIN_FIREWALL_INSTALLED_ENV_NAME = "JACKIN_FIREWALL_INSTALLED"`; add all three to the `RUNTIME_OWNED_ENV_VARS` table so role manifests cannot set them.
- <RepoFile path="crates/jackin-runtime/src/runtime/launch.rs" />, `launch_role_runtime` — env injection near the `env_strings` block (~line 648). When `grants.network == Allowlist`, push `JACKIN_NETWORK_MODE=allowlist` and `JACKIN_ALLOWED_HOSTS=<comma-joined>`. The host list is `grants.allowed_hosts` ∪ `default_allowed_hosts_for_agent(agent)` ∪ forwarded GitHub host(s). For `open`, push `JACKIN_NETWORK_MODE=open` (informational, no firewall). For `none`, the existing `--network none` path already covers it.
- <RepoFile path="crates/jackin-runtime/src/runtime/launch.rs" />, `launch_role_runtime` — firewall invocation in the post-`run_role.await?` window (~line 860, after `docker run -d` succeeds, before `reconnect_or_create_session_with_focus`). When `grants.network == Allowlist`, run `docker exec --user root <container> /jackin/runtime/jackin-capsule firewall-apply` via `runner.run(...)`. Root-via-`exec` needs no setuid, so it composes with `no-new-privileges`. On non-zero exit, fail the launch (fail-closed) with an actionable error — do **not** drop the operator into a session that believes it is firewalled but is not.
- <RepoFile path="crates/jackin-capsule/src/firewall.rs" /> — no behavior change needed; `apply()` already does `-P OUTPUT DROP` first, accepts loopback/ESTABLISHED/DNS, resolves the allowlist into an ipset, and treats an empty list as fail-closed (no egress).

**Open design point — `JACKIN_FIREWALL_INSTALLED` timing:** the entrypoint checks this env at container start and warns if `network=allowlist` but it is unset. The host applies the firewall *after* start, so it cannot satisfy that check via a `docker run -e`. Resolve one of: (a) treat the entrypoint check as informational only and rely on the host's fail-closed exec result as the source of truth (simplest); or (b) have `firewall-apply` itself write a marker file the capsule reads. Recommend (a).

**Enforcement honesty:** reuse `network_enforcement_label(grants)` (already written) to report `full` vs `partial (sudo grants iptables access)` vs `partial (DinD inner containers bypass host iptables)`. This label must appear in the session contract (WP3) so operators are never told "allowlist" without the caveat.

**Tests:**
- Unit: env injection present for `allowlist`, absent for `open`/`none`; host list composition (agent endpoints + GitHub + configured hosts, deduped).
- Unit: `firewall-apply` exec is emitted only for `allowlist`, with `--user root`.
- E2E (WP0 rows): under `hardened`, a curl to a non-allowlisted host fails; an allowlisted host succeeds; DNS resolves.

**Telemetry:** `debug_log!("launch", "network_enforcement mode=allowlist label=<full|partial:…> hosts=<n>")` and `firewall_apply exit=<code>`.

**Exit criteria:** an agent launched under `hardened` cannot reach a non-allowlisted host; the launch fails closed if `firewall-apply` errors; the enforcement label is truthful.

### WP2 — `internal` network for `locked` (ship with WP1)

**Goal:** block external egress at the Docker network layer for `locked`, independent of in-container iptables.

**Integration points:**
- <RepoFile path="crates/jackin-core/src/docker.rs" /> — `DockerApi::create_network` gains an `internal: bool` parameter.
- <RepoFile path="crates/jackin-docker/src/docker_client.rs" />, `create_network` — set `NetworkCreateRequest.internal = Some(internal)`.
- <RepoFile path="crates/jackin-runtime/src/runtime/test_support.rs" /> — update the fake `create_network` to record the flag.
- <RepoFile path="crates/jackin-runtime/src/runtime/launch/launch_dind.rs" />, `create_role_network` (~line 90) — thread the flag; pass `internal: true` when `profile == locked`.

**Open verification:** when DinD is enabled under `standard`, confirm inner containers created by the sidecar inherit the restriction; if not, emit a `partial` enforcement warning rather than implying full coverage.

**Tests:** unit — `locked` creates the network with `internal=true`; other profiles `internal=false`. E2E — `locked` container has no route off-bridge even before `firewall-apply`.

**Telemetry:** `debug_log!("launch", "network internal=<bool> profile=<name>")`.

### WP3 — Launch observability (session contract + probes + telemetry)

**Goal:** make every applied control visible at launch. The data already exists; the surface does not.

**Integration points:**
- <RepoFile path="crates/jackin-runtime/src/runtime/docker_profile.rs" /> — `format_session_contract(...)` exists but is never called. Call it from `launch_role_runtime` and print under `--debug` (and emit via `debug_log!`).
- New probe helpers (in `docker_profile.rs` or a sibling `host_probe.rs`):
  - **AppArmor:** `docker info --format '{{.SecurityOptions}}'` (or `{{.OperatingSystem}}`) via `runner.capture(...)`; parse for `apparmor`. Feeds `apparmor_available` + `apparmor_layer` (host vs backend-VM on Docker Desktop/OrbStack).
  - **cgroup version:** `stat -fc %T /sys/fs/cgroup/` or read `/proc/self/cgroup`. Feeds the `cgroup_version` contract field. On v1 under `hardened`/`locked`, fail-closed; under `standard`, downgrade `memory_reservation` with a warning.
  - **Cache + parallelism:** the `docker info` probe can cost 100–500 ms on Docker Desktop; cache per session and/or run concurrently with the image-pull check.
- <RepoFile path="crates/jackin-runtime/src/runtime/launch.rs" /> — replace the single summary `debug_log!` line with the full per-decision telemetry set (below).

**Telemetry surface (minimum lines):**
- `profile_selected profile=<name> source=<cli|workspace|config|default>`
- `cap_drop_all` + `cap_add cap=<name> reason=<jackin-default|role-required|opt-in>`
- `no_new_privileges enforced=<yes|no> reason=<profile|sudo-audit-pending>`
- `seccomp profile=<docker-default|custom|unconfined>`
- `apparmor available=<yes|no> profile=<…> layer=<host|backend-vm>`
- `read_only_root enforced=<yes|no> tmpfs=<list>`
- `cgroup_version v=<v1|v2|hybrid>`
- `resource_limit kind=<memory|cpus|pids> value=<v> flag=<docker-flag>`
- `dind enabled=<yes|no> mode=<privileged|rootless> tls_certs_path=<path>`
- `host_socket_check passed=<yes|no>`

**Tests:**
- **Host-socket regression test** (`role_container_never_mounts_host_docker_socket`) — specified for Phase 1 but missing. Assert no `-v …/docker.sock` ever appears in `run_args`. This is a hard rule; the test must exist.
- Unit: contract string renders each dimension correctly per profile (snapshot-style).
- Unit: probe parsers handle apparmor present/absent and cgroup v1/v2/hybrid.

**Exit criteria:** `jackin load … --debug` prints a complete, accurate per-dimension contract; the host-socket test guards the hard rule.

### WP4 — DinD-free auto-skip (independent)

**Tests:** unit — `hardened` with no inner-engine need starts no DinD container, mounts no certs, and reports `network_enforcement = full` (no DinD bypass). E2E — a no-DinD `hardened` launch runs an agent successfully.

**Telemetry:** `debug_log!("launch", "dind enabled=no reason=profile-no-inner-engine")`.

## Constraints that stay true

- **Agent bypass flags are not per-profile.** Every agent launches with its maximum-bypass flag in <RepoFile path="docker/runtime/entrypoint.sh" /> (`claude --dangerously-skip-permissions`, etc.). The Docker boundary is the only enforceable isolation layer; the session contract must state this so operators trust Docker isolation, not the agent's own permission system.
- **Host Docker socket exclusion is a hard rule** (see the regression test in WP3).
- Hardened Docker is still a shared-kernel boundary — the contract must not overstate it.
