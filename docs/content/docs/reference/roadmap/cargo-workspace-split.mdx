**Status**: Partially implemented — the surface crates are extracted; the non-visual core still lives in the root `src/` binary crate. Re-baked 2026-06-02.

## What shipped

`jackin` is now a Cargo workspace using the virtual-manifest + flat `crates/` pattern. The 2026 TUI architecture work extracted every terminal-UI surface into its own crate:

```
jackin/
├── Cargo.toml             ← virtual workspace manifest + root `jackin` binary
├── crates/
│   ├── jackin-tui/        ← shared TUI component library + theme/geometry/scroll
│   ├── jackin-console/    ← host console TUI (Elm layers + screens)
│   ├── jackin-launch/     ← launch progress cockpit
│   ├── jackin-capsule/    ← in-container multiplexer
│   ├── jackin-protocol/   ← capsule wire protocol
│   ├── jackin-tui-lookbook/ ← component lookbook + SVG generator
│   └── jackin-build-meta/ ← build-time version metadata
└── src/                   ← root binary: runtime, config, instance, isolation, app, manifest, cli
```

This split was **surface-oriented**: it drew the boundary around terminal UI, which is where the worst duplication lived (see [TUI Design](/reference/tui/)). It did not extract the non-visual core.

## What remains — the non-visual core split

The root `src/` crate still holds ~108K lines of runtime, config, instance, isolation, app-dispatch, manifest, and CLI code. The remaining work is to extract the tier-based core crates from it:

- **`jackin-core`** — Tier 0 leaf types with no cross-module deps: `workspace/`, `manifest/`, `selector.rs`, `paths.rs`, and the docker-runner trait.
- **`jackin-config`** — TOML persistence (`AppConfig`, `ConfigEditor`, migrations).
- **`jackin-runtime`** — the container bootstrap pipeline (`runtime/`, `instance/`, `isolation/`, `derived_image.rs`).

The root `src/` then becomes a thin CLI-dispatch binary over those crates.

## Trigger — now met

The original LOC trigger (production Rust over ~150K) is met: the workspace totals ~170K lines (108K in `src/`, 62K in `crates/`). But LOC was always a heuristic; the real blockers are sequencing, not size. **Do not start the core split until the prerequisites below are done** — extracting monolithic, duplicated, undocumented files into crates would freeze that entropy into crate APIs.

## Prerequisites before the core split

1. **File splits** — the root-crate monoliths (`runtime/launch.rs` at 9,482 L, `app/mod.rs` at 3,966 L, `operator_env.rs` at 4,209 L, `instance/auth.rs` at 3,302 L) must become focused modules before they can become focused crates. Tracked under [Codebase readability → Split the root-crate monoliths](/reference/roadmap/codebase-readability/).
2. **DRY extractions** — shared string constants, auth-provisioning, and agent dispatch must be deduplicated so the duplication is not frozen into the crate boundary.
3. **Visibility pass** — continue the [`pub(crate)` migration](/reference/roadmap/pub-crate-visibility/). The root crate's `pub(crate)` share rose from ~11% to ~31% as the surface crates drew explicit public boundaries, but a meaningful core split needs each new crate's public API to be a deliberate, auditable surface rather than an accidental one.

## Already resolved

The earlier blocker — a `console/` → `runtime/` production dependency in TUI input code — is resolved. That call (`register_agent_repo`) now lives correctly inside a service adapter at <RepoFile path="src/console/services/role_load.rs" />, outside the TUI boundary, so a future `jackin-runtime` crate boundary no longer cuts through TUI code.
