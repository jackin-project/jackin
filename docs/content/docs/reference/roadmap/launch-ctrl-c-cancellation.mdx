---
title: "Launch Screen Exit Keybindings (Ctrl+C / Ctrl+Q)"
---

**Status**: Implemented

## Problem

Pressing `Ctrl+C` during the launch screen has no effect. While the rich TUI cockpit is active ‚Äî covering image pulls, Docker builds, DinD network setup, workspace materialization, and every other pre-attach pipeline step ‚Äî `Ctrl+C` is silently swallowed. The operator cannot abort a long-running launch without resorting to `kill` from another terminal window or closing the terminal tab entirely. Neither is acceptable UX for a CLI tool.

The bug has two layers:

1. **Raw-mode signal suppression.** `crossterm::terminal::enable_raw_mode()` on Unix clears the `ISIG` `termios` flag. This disables the kernel's Ctrl+C ‚Üí SIGINT translation, so `tokio::signal::ctrl_c()` never fires. The key event reaches the process as `KeyCode::Char('c') + KeyModifiers::CONTROL` instead. No signal means no signal handler helps.

2. **Input polling ignores the key event during pipeline phases.** `handle_cockpit_input()` in <RepoFile path="crates/jackin-launch/src/tui/subscriptions.rs" /> drains queued terminal events every 33 ms with `event::poll(Duration::ZERO)`, but it dispatches only mouse events, scroll keys, build-log toggles, and similar cockpit interactions. `Ctrl+C` falls through unhandled during every non-dialog phase of the launch pipeline.

`read_pressed_key()` in <RepoFile path="crates/jackin-launch/src/tui/run.rs" /> correctly detects and reacts to `Ctrl+C` ‚Äî but it is called only when an interactive dialog is on screen (role picker, text prompt, confirm prompt). During the much longer stretches when the pipeline is doing I/O (git clone, Docker pull, image build, container startup), nothing reads or acts on the key event.

## Why It Matters

`Ctrl+C` is the universal exit convention for interactive CLI processes. Every major terminal application ‚Äî `docker`, `git`, `cargo`, `kubectl`, and the agents themselves ‚Äî honours it. Operators reach for `Ctrl+C` by reflex. Finding it inert on the launch screen erodes trust in the tool and forces disruptive workarounds (`kill -9`, closing the terminal).

The impact is highest during slow operations: large image pulls on a metered connection, Docker builds that should have used `--rebuild`, and DinD network setup stalls. These are exactly the moments when operators most want to bail.

Fixing this also unblocks a clean cancellation path for the pipeline generally: error propagation, `LoadCleanup` (container, DinD, certs volume, network teardown), and terminal-state restoration on abort are all already implemented ‚Äî they just need a signal to trigger on.

## Design

### Root-cause analysis

The relevant code paths:

| Location | Role |
|---|---|
| <RepoFile path="crates/jackin-launch/src/tui/subscriptions.rs" /> `handle_cockpit_input()` | Polls events every 33 ms during pipeline phases ‚Äî does not check `Ctrl+C` |
| <RepoFile path="crates/jackin-launch/src/tui/run.rs" /> `read_pressed_key()` | Correctly handles `Ctrl+C` but is only called during interactive dialogs |
| <RepoFile path="crates/jackin-launch/src/tui/run.rs" /> `RichDriver::spawn()` | Owns the background render task; the render loop needs access to a shared cancel signal |
| <RepoFile path="crates/jackin-runtime/src/runtime/launch/launch_pipeline.rs" /> | Hosts `load_role_with()` ‚Äî the top-level pipeline future; needs to await cancellation in parallel with each pipeline step |
| <RepoFile path="crates/jackin-runtime/src/runtime/launch/progress_helpers.rs" /> | `StepCounter::while_waiting()` wraps long-running docker/git futures ‚Äî the choke point for wiring cancellation into individual steps |

### Cancellation primitive: `tokio_util::CancellationToken` ‚úì settled

`tokio_util::CancellationToken` is the primitive. It is:

- Clone-able and `Send + Sync` ‚Äî safe to share between the render task and the pipeline task.
- Awaitable ‚Äî `token.cancelled()` is a future that resolves when `token.cancel()` is called.
- Composable with `tokio::select!` without wrapping every future in an `Option`.
- Provides sync `is_cancelled()` for the render loop's per-tick check.

`tokio-util` is not a current direct dep in any crate. Add it to the workspace `[workspace.dependencies]` in the root <RepoFile path="Cargo.toml" />, then reference it from `jackin-launch/Cargo.toml` with the `sync` feature. Only `jackin-launch` needs it directly ‚Äî `LaunchProgress`, `RichDriver`, and `handle_cockpit_input()` all live there; callers in `jackin-runtime` just call `progress.while_waiting(fut).await?` with no token type visible to them.

### Keybinding model ‚Äî revised after operator testing

Two bindings, both cause immediate exit without confirmation:

| Key | Label | Behaviour | Phase |
|---|---|---|---|
| `Ctrl+C` | `abort` | Immediate hard cancel ‚Äî no dialog, no double-press | All pipeline phases |
| `Ctrl+Q` | `quit` | Immediate clean exit ‚Äî no dialog | All pipeline phases |

**Single-press Ctrl+C** is the established standard for pipeline contexts. Double-press Ctrl+C is an emerging pattern for *interactive AI sessions* (Codex CLI, OpenCode) where an accidental Ctrl+C on an idle prompt is a real risk. The launch screen is a pipeline watcher (analogous to `docker build` or `cargo build`), not an interactive session ‚Äî single-press abort is correct and matches operator expectations.

**Ctrl+Q now exits immediately** ‚Äî the previous design planned a "Ctrl+Q ‚Üí confirmation dialog ‚Üí pipeline keeps running" flow. Operator testing feedback: the dialog is unnecessary; Ctrl+Q should just quit. Both Ctrl+C and Ctrl+Q result in immediate pipeline abort and terminal restore; the semantic difference is the label ("abort" vs. "quit") only.

**Bare `q` must be migrated to `Ctrl+Q` everywhere it was used** in the launch surface ‚Äî the previous design note that said "q is already used for closing dialogs and must not be repurposed" was reversed by operator feedback. Any dismiss binding that was `Esc | q` (build log, container info) must become `Esc | Ctrl+Q`, and hint text must update accordingly. `Ctrl+Q = XON` (flow control resume) is inert in raw mode, so no conflict.

**Both bindings must be available on all screens** within the launch surface ‚Äî cockpit, build log overlay, container info overlay, and every interactive dialog during the pipeline.

`Ctrl+C` in the capsule TUI is forwarded to the running agent's PTY as SIGINT ‚Äî this must not be intercepted there. Capsule-level `Ctrl+Q` is a related but separate concern tracked as a follow-on.

### Implementation plan

**Phase 1 ‚Äî Wire the cancellation token and exit-confirm state through the render driver (jackin-launch)**

1. Add `cancel_token: CancellationToken` field to `LaunchProgress`. Expose via `cancel_token() -> CancellationToken` (returns a clone).
2. Thread the token into `RichDriver::spawn()`; the render task checks `cancel_token.is_cancelled()` as an early-break guard alongside the existing `stop` atomic.
3. Add `exit_confirm_open: bool` to `LaunchView` with `ExitConfirmOpened` / `ExitConfirmClosed` `LaunchMessage` variants.
4. In `handle_cockpit_input()`: detect `Ctrl+C` ‚Üí `cancel_token.cancel()`. Detect `Ctrl+Q` ‚Üí dispatch `ExitConfirmOpened`. When `exit_confirm_open` is true, route Y/Enter ‚Üí `cancel_token.cancel()`, N/Esc ‚Üí `ExitConfirmClosed`. Add `cancel_token: &CancellationToken` parameter to `handle_cockpit_input`.
5. Also update `read_pressed_key()` to bail on `Ctrl+Q` in addition to `Ctrl+C`, so interactive dialog phases handle it consistently.

**Phase 2 ‚Äî Propagate cancellation into the pipeline (jackin-runtime)**

1. In `StepCounter` (or wherever `while_waiting()` lives), add a `cancel_token: Option<CancellationToken>` field.
2. Rewrite `while_waiting()`:

```rust
pub async fn while_waiting<T>(
    &self,
    future: impl Future<Output = anyhow::Result<T>>,
) -> anyhow::Result<T> {
    if let Some(token) = &self.cancel_token {
        tokio::select! {
            result = future => result,
            _ = token.cancelled() => {
                anyhow::bail!("launch cancelled by operator")
            }
        }
    } else {
        future.await
    }
}
```

3. Thread the `CancellationToken` from `RichDriver` ‚Üí `StepCounter` ‚Üí every `while_waiting()` call site in `launch_role_runtime()` and the DinD sidecar path.
4. For pipeline stages that do not use `while_waiting()` (e.g., `resolve_restore_candidate`, trust prompts), wrap the await with `tokio::select!` directly or extract a `cancellable()` helper.

**Phase 3 ‚Äî Coverage audit and cooperative cancellation guarantee**

A `tokio::select!` wrapping `load_role_with()` at the top level is NOT the right guard. Dropping the pipeline future abruptly skips `LoadCleanup::run()` (called inside the existing error-propagation chain, not via `Drop`). The correct model is **cooperative cancellation**: `while_waiting()` returning `Err` propagates via `?` through the existing error handlers, which call `cleanup.run()`.

The slow pipeline operations (image build, docker pull, DinD startup probe, workspace materialization) all go through `while_waiting()`. Fast operations (docker API calls, git identity, container name claim) do not ‚Äî but they complete in under 1 s, so cancellation is felt at the next `while_waiting()` boundary. This latency is acceptable.

Phase 3 work: audit every pipeline step to confirm it either (a) goes through `while_waiting()` or (b) is fast enough that waiting for the next `while_waiting()` boundary is fine. If any non-`while_waiting()` step can block for seconds (e.g., a slow docker API call), wrap it in a direct `tokio::select!` that returns the same cancellation error ‚Äî still cooperative, still hits `?`.

**Phase 4 ‚Äî Cleanup and terminal restoration (BUG: terminal freezes on Ctrl+C)**

Operator testing revealed that pressing `Ctrl+C` during a running pipeline causes the screen to freeze for 10‚Äì30 seconds before the terminal is restored. The root cause:

`LaunchProgress` owns `RichDriver` which owns `Arc<Mutex<RichRenderer>>`. `RichRenderer::drop()` is the mechanism that restores the terminal (disables raw mode, leaves the alternate screen). The Arc ref count reaches zero ‚Äî and `RichRenderer::drop()` fires ‚Äî only when **both** the render task and the `RichDriver` struct release their Arc clones. `RichDriver` is held inside `LaunchProgress`. `LaunchProgress` is dropped **after** `LoadCleanup::run()` completes (Docker container stop, network teardown ‚Äî typically 10‚Äì30 s). The render task breaks its loop on the next tick (~33 ms after cancel fires), releasing its Arc clone, but that only lowers the count from 2 ‚Üí 1. The count reaches 0 only after `LaunchProgress` drops, which happens post-cleanup.

Fix required: terminal must be restored **before** cleanup runs, not after. Two approaches:

1. **Explicit early restore** ‚Äî add a `restore_terminal_now()` method to `LaunchProgress` that (a) calls `stop_detached()` on the driver (aborts the render task, releasing its Arc clone immediately), (b) drops the `renderer` field explicitly (the driver's Arc clone goes to 0 ‚Üí `RichRenderer::drop()` fires). Call this in the pipeline the moment the cancellation error is detected, before `LoadCleanup::run()`. Cleanup then runs with the terminal in normal mode; any output goes to stderr.
2. **Separate terminal handle** ‚Äî extract terminal ownership from `LaunchProgress` into a `TerminalGuard` with explicit scope, so the terminal is restored when the guard drops at the cancel boundary, independently of when `LaunchProgress` drops.

Either approach must not break the case where `LaunchProgress` is dropped on an error path that does **not** go through cancellation (e.g., an actual Docker error), where the terminal should also be restored promptly before the error is printed.

Also verify `RichRenderer::drop()` fires correctly on non-cancel error paths (it always fires on drop; confirm no path panics before drop, which would skip `Drop`). Add unit test stub for `while_waiting()` cancellation path.

**Phase 5 ‚Äî Hints in the correct location (BUG: hints on status bar, not hint bar)**

Operator testing revealed that `Ctrl+C abort ¬∑ Ctrl+Q exit?` was added to the **status bar** (white bar) via a `hint: Option<&str>` parameter on `render_status_footer`. This is wrong. Per the [TUI chrome design](/reference/tui/chrome): the status bar is the white instance/branch bar; hints belong in the **hint bar** (the separate row above the chrome separator, not on the white bar).

Fix required:

1. Revert `hint: Option<&str>` from `render_status_footer` / `StatusFooter::new()` in `jackin-tui` and all callers ‚Äî remove this slot entirely.
2. Add `Ctrl+C cancel ¬∑ Ctrl+Q exit?` to the **launch hint bar** (the row that currently shows e.g. `Q close ¬∑ ‚Üë‚Üì scroll` when the build log or container info is open, and the activity-level hint row during pipeline phases). The hint bar is the canonical place for all keybinding hints per `/reference/tui/navigation`.
3. Update the [TUI chrome docs page](/reference/tui/chrome) to remove the incorrect status-bar hint slot description added in the previous implementation pass, and document the correct hint-bar placement instead.

### Visual feedback on cancellation ‚Äî revised

Immediate teardown. When Ctrl+C or Ctrl+Q fires, the render task immediately calls `restore_terminal()` in the same tokio task step as the cancel (before the pipeline task can resume), then `while_waiting()` returns `Err`, the error propagates via `?`, `LoadCleanup::run()` removes Docker resources with the terminal already in normal mode. No "Cancelled" cockpit state is rendered ‚Äî the cockpit simply disappears, matching `docker build`, `cargo build`, `git clone` behaviour. If cleanup emits any output, it goes to stderr after the terminal is restored.

The previous design relied on `RichRenderer::drop()` for terminal restoration, which fires only after `LaunchProgress` drops (post-cleanup). This caused the 10‚Äì30 s freeze. The fix: explicitly call `restore_terminal()` in the render task when cancel is detected, before breaking the render loop, making `Drop` a no-op via `entered_alt_screen = false`.

### What does NOT change

- `tokio::signal::ctrl_c()` is not the right fix (blocked by `ISIG` suppression in raw mode) but can be added as defence-in-depth for the narrow path where the launcher runs without raw mode active (e.g., non-interactive CI invocations).
- The cancellation error message matches the existing `"launch cancelled by operator"` string used in `read_pressed_key()` and the cancel branches of `update_text_prompt` / `update_select_prompt`.
- `exit_confirm_open` state and the `ExitConfirmOpened` / `ExitConfirmClosed` `LaunchMessage` variants should be removed ‚Äî the exit confirm dialog is no longer part of the design.
