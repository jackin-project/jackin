---
title: "Launch Screen Exit Keybindings (Ctrl+C / Ctrl+Q)"
---

**Status**: Partially implemented — Phase 1–3 shipped; Phase 3 audit and Phase 4–5 docs in same PR

## Problem

Pressing `Ctrl+C` during the launch screen has no effect. While the rich TUI cockpit is active — covering image pulls, Docker builds, DinD network setup, workspace materialisation, and every other pre-attach pipeline step — `Ctrl+C` is silently swallowed. The operator cannot abort a long-running launch without resorting to `kill` from another terminal window or closing the terminal tab entirely. Neither is acceptable UX for a CLI tool.

The bug has two layers:

1. **Raw-mode signal suppression.** `crossterm::terminal::enable_raw_mode()` on Unix clears the `ISIG` termios flag. This disables the kernel's Ctrl+C → SIGINT translation, so `tokio::signal::ctrl_c()` never fires. The key event reaches the process as `KeyCode::Char('c') + KeyModifiers::CONTROL` instead. No signal means no signal handler helps.

2. **Input polling ignores the key event during pipeline phases.** `handle_cockpit_input()` in <RepoFile path="crates/jackin-launch/src/tui/subscriptions.rs" /> drains queued terminal events every 33 ms with `event::poll(Duration::ZERO)`, but it dispatches only mouse events, scroll keys, build-log toggles, and similar cockpit interactions. `Ctrl+C` falls through unhandled during every non-dialog phase of the launch pipeline.

`read_pressed_key()` in <RepoFile path="crates/jackin-launch/src/tui/run.rs" /> correctly detects and reacts to `Ctrl+C` — but it is called only when an interactive dialog is on screen (role picker, text prompt, confirm prompt). During the much longer stretches when the pipeline is doing I/O (git clone, Docker pull, image build, container startup), nothing reads or acts on the key event.

## Why It Matters

`Ctrl+C` is the universal exit convention for interactive CLI processes. Every major terminal application — `docker`, `git`, `cargo`, `kubectl`, and the agents themselves — honours it. Operators reach for `Ctrl+C` by reflex. Finding it inert on the launch screen erodes trust in the tool and forces disruptive workarounds (`kill -9`, closing the terminal).

The impact is highest during slow operations: large image pulls on a metered connection, Docker builds that should have used `--rebuild`, and DinD network setup stalls. These are exactly the moments when operators most want to bail.

Fixing this also unblocks a clean cancellation path for the pipeline generally: error propagation, `LoadCleanup` (container, DinD, certs volume, network teardown), and terminal-state restoration on abort are all already implemented — they just need a signal to trigger on.

## Design

### Root-cause analysis

The relevant code paths:

| Location | Role |
|---|---|
| <RepoFile path="crates/jackin-launch/src/tui/subscriptions.rs" /> `handle_cockpit_input()` | Polls events every 33 ms during pipeline phases — does not check `Ctrl+C` |
| <RepoFile path="crates/jackin-launch/src/tui/run.rs" /> `read_pressed_key()` | Correctly handles `Ctrl+C` but is only called during interactive dialogs |
| <RepoFile path="crates/jackin-launch/src/tui/run.rs" /> `RichDriver::spawn()` | Owns the background render task; the render loop needs access to a shared cancel signal |
| <RepoFile path="crates/jackin-runtime/src/runtime/launch/launch_pipeline.rs" /> | Hosts `load_role_with()` — the top-level pipeline future; needs to await cancellation in parallel with each pipeline step |
| <RepoFile path="crates/jackin-runtime/src/runtime/launch/progress_helpers.rs" /> | `StepCounter::while_waiting()` wraps long-running docker/git futures — the choke point for wiring cancellation into individual steps |

### Cancellation primitive: `tokio_util::CancellationToken` ✓ settled

`tokio_util::CancellationToken` is the primitive. It is:

- Clone-able and `Send + Sync` — safe to share between the render task and the pipeline task.
- Awaitable — `token.cancelled()` is a future that resolves when `token.cancel()` is called.
- Composable with `tokio::select!` without wrapping every future in an `Option`.
- Provides sync `is_cancelled()` for the render loop's per-tick check.

`tokio-util` is not a current direct dep in any crate. Add it to the workspace `[workspace.dependencies]` in the root `Cargo.toml`, then reference it from `jackin-launch/Cargo.toml` with the `sync` feature. Only `jackin-launch` needs it directly — `LaunchProgress`, `RichDriver`, and `handle_cockpit_input()` all live there; callers in `jackin-runtime` just call `progress.while_waiting(fut).await?` with no token type visible to them.

### Keybinding model ✓ settled

Two bindings, distinct semantics:

| Key | Behaviour | Phase |
|---|---|---|
| `Ctrl+C` | Immediate hard cancel — no dialog | All pipeline phases |
| `Ctrl+Q` | Exit confirmation dialog — pipeline continues while user decides | All pipeline phases |

`q` (bare) is already used for closing dialogs (build log, container info) and must not be repurposed. `Ctrl+Q` is an unused slot across all surfaces; `Ctrl+Q = XON` (flow control resume) is inert in raw mode.

`Ctrl+C` in the capsule TUI is forwarded to the running agent's PTY as SIGINT — this must not be intercepted there. Capsule-level `Ctrl+Q` ("Disconnect session?") is a related but separate concern tracked as a follow-on.

### Signal flows

**Ctrl+C — hard cancel:**

```
Operator presses Ctrl+C during pipeline
        │
        ▼
handle_cockpit_input() detects KeyCode::Char('c') + KeyModifiers::CONTROL
        │
        ▼
cancel_token.cancel()
        │
        ├──► while_waiting() tokio::select! fires cancelled branch
        │    → Err("launch cancelled by operator")
        │    → LoadCleanup::run() — removes container, DinD, certs volume, network
        │
        └──► RichDriver tick checks cancel_token.is_cancelled() → breaks
             RichRenderer::drop() — cursor, raw mode, alt screen, mouse restored
```

**Ctrl+Q — exit confirmation (non-blocking):**

```
Operator presses Ctrl+Q during pipeline
        │
        ▼
handle_cockpit_input() detects KeyCode::Char('q') + KeyModifiers::CONTROL
        │
        ▼
update_launch_view(ExitConfirmOpened)     ← pipeline KEEPS RUNNING
        │
        ▼
cockpit renders "Abort launch? [Y / N]" modal overlay (ConfirmState)
        │
        ├── Y pressed → cancel_token.cancel() → same path as Ctrl+C above
        └── N / Esc pressed → update_launch_view(ExitConfirmClosed) → modal gone
```

The confirm modal follows the same overlay pattern as `container_info_open` / `build_log_open` — a boolean flag on `LaunchView`, rendered as a `ConfirmState` widget on top of the cockpit. The pipeline does not pause; if the launch completes before the user answers, the modal is irrelevant and torn down with the cockpit.

### Implementation plan

**Phase 1 — Wire the cancellation token and exit-confirm state through the render driver (jackin-launch)**

1. Add `cancel_token: CancellationToken` field to `LaunchProgress`. Expose via `cancel_token() -> CancellationToken` (returns a clone).
2. Thread the token into `RichDriver::spawn()`; the render task checks `cancel_token.is_cancelled()` as an early-break guard alongside the existing `stop` atomic.
3. Add `exit_confirm_open: bool` to `LaunchView` with `ExitConfirmOpened` / `ExitConfirmClosed` `LaunchMessage` variants.
4. In `handle_cockpit_input()`: detect `Ctrl+C` → `cancel_token.cancel()`. Detect `Ctrl+Q` → dispatch `ExitConfirmOpened`. When `exit_confirm_open` is true, route Y/Enter → `cancel_token.cancel()`, N/Esc → `ExitConfirmClosed`. Add `cancel_token: &CancellationToken` parameter to `handle_cockpit_input`.
5. Also update `read_pressed_key()` to bail on `Ctrl+Q` in addition to `Ctrl+C`, so interactive dialog phases handle it consistently.

**Phase 2 — Propagate cancellation into the pipeline (jackin-runtime)**

1. In `StepCounter` (or wherever `while_waiting()` lives), add a `cancel_token: Option<CancellationToken>` field.
2. Rewrite `while_waiting()`:

```rust
pub async fn while_waiting<T>(
    &self,
    future: impl Future<Output = anyhow::Result<T>>,
) -> anyhow::Result<T> {
    if let Some(token) = &self.cancel_token {
        tokio::select! {
            result = future => result,
            _ = token.cancelled() => {
                anyhow::bail!("launch cancelled by operator")
            }
        }
    } else {
        future.await
    }
}
```

3. Thread the `CancellationToken` from `RichDriver` → `StepCounter` → every `while_waiting()` call site in `launch_role_runtime()` and the DinD sidecar path.
4. For pipeline stages that do not use `while_waiting()` (e.g., `resolve_restore_candidate`, trust prompts), wrap the await with `tokio::select!` directly or extract a `cancellable()` helper.

**Phase 3 — Coverage audit and cooperative cancellation guarantee**

A `tokio::select!` wrapping `load_role_with()` at the top level is NOT the right guard. Dropping the pipeline future abruptly skips `LoadCleanup::run()` (called inside the existing error-propagation chain, not via `Drop`). The correct model is **cooperative cancellation**: `while_waiting()` returning `Err` propagates via `?` through the existing error handlers, which call `cleanup.run()`.

The slow pipeline operations (image build, docker pull, DinD startup probe, workspace materialisation) all go through `while_waiting()`. Fast operations (docker API calls, git identity, container name claim) do not — but they complete in <1 s, so cancellation is felt at the next `while_waiting()` boundary. This latency is acceptable.

Phase 3 work: audit every pipeline step to confirm it either (a) goes through `while_waiting()` or (b) is fast enough that waiting for the next `while_waiting()` boundary is fine. If any non-`while_waiting()` step can block for seconds (e.g., a slow docker API call), wrap it in a direct `tokio::select!` that returns the same cancellation error — still cooperative, still hits `?`.

**Phase 4 — Cleanup and terminal restoration**

Verify that cancellation on every path reaches `LoadCleanup::run()`. The existing error-propagation chain (`?` on every step returns to the top-level error handler that calls `cleanup.run()`) should cover this already — confirm with a manual test and add a unit test stub for the cancellation error path.

Also verify `RichRenderer::drop()` fires correctly on the cancellation path (it always fires on drop; confirm no path panics before drop, which would skip `Drop`).

**Phase 5 — Hint in the cockpit footer ✓ in scope**

Add `Ctrl+C  cancel` as a dim hint in the launch footer while the pipeline is running (before any interactive dialog and before handoff to capsule). Implemented by adding `hint: Option<&str>` to `render_status_footer` in `jackin-tui` — rendered as dim text on the left side after the activity text. All callers updated; the launch footer passes `Some("Ctrl+C  cancel")` during pipeline phases and `None` otherwise. TUI docs updated in `docs/content/docs/reference/tui/` per the cross-cutting TUI behaviour rule in `RULES.md`.

### Visual feedback on cancellation ✓ settled

Immediate teardown. When Ctrl+C fires, `while_waiting()` returns `Err`, the error propagates via `?`, `LoadCleanup::run()` removes Docker resources, `RichProgress::drop()` stops the render task, and `RichRenderer::drop()` restores the terminal. No "Cancelled" cockpit state is rendered — the cockpit simply disappears, matching `docker build`, `cargo build`, `git clone` behaviour. If cleanup emits any output, it goes to stderr after the terminal is restored.

### What does NOT change

- `read_pressed_key()` is already correct for the interactive dialog phases. No change needed there.
- `tokio::signal::ctrl_c()` is not the right fix (blocked by `ISIG` suppression in raw mode) but can be added as defence-in-depth for the narrow path where the launcher runs without raw mode active (e.g., non-interactive CI invocations).
- The cancellation error message matches the existing `"launch cancelled by operator"` string used in `read_pressed_key()` and the cancel branches of `update_text_prompt` / `update_select_prompt`.

### Dependency check

Before starting implementation, confirm `tokio_util` appears in `Cargo.toml` as a direct dependency in the relevant crates (`jackin-launch`, `jackin-runtime`). If absent, add it with the `sync` feature only (`tokio-util = { workspace = true, features = ["sync"] }`).
