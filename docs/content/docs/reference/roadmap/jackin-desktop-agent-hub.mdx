**Status**: Open — design reset, building incrementally. The native macOS app is being rebuilt around an embedded [libghostty](https://github.com/ghostty-org/ghostty) terminal and a [CMUX](https://github.com/manaflow-ai/cmux)-style window: a left sidebar that groups sessions by workspace, and per-space tabs across the top. The earlier consolidated host-daemon plus menu-bar prototype was removed; v1 drives everything through the `jackin` CLI, and the daemon returns later as a pure enrichment plane. **Current point:** milestone 1 (a SwiftUI window that launches) is scaffolded under `desktop/macos/`, awaiting operator preview on macOS. The live milestone checklist is in [Implementation plan](#implementation-plan) below — that section is the resume anchor for this work.

## Problem

jackin' is already the control plane for isolated developer agents: workspaces define project boundaries, roles define runtime shape, and `jackin load` gives every agent its own isolated, reconnectable container session. The missing piece is a native macOS **operator surface** that answers, without forcing the operator to keep every terminal tab in view:

- **What is running** right now, across every project?
- **What needs review** — which agent has a PR ready or is waiting on me?
- **What accounts are near exhaustion** — is any built-in runtime about to hit a quota wall?

The app is not a replacement for Claude Code, Codex, or Amp — the agent UI stays the agent's own TUI. jackin' Desktop is the native shell *around* those agents: pick a workspace, start a session, see active containers, jump to pull requests, monitor account state, and get an attention prompt when an agent is blocked or finished. That full control-surface goal is unchanged from the original proposal; the [Direction reset](#direction-reset) below changes only how it is built.

## Direction reset — same goal, new foundation

The destination above is the original Desktop Agent Hub goal. What changed is the **foundation** the goal is built on, not the goal itself.

The original proposal assumed a **menu-bar-first** hub, backed by a long-running **host daemon**, that launched agents into **external** Ghostty windows via `open -na Ghostty`. It was, in effect, all control-surface chrome with **no terminal of its own** — the operator still ended up in a separate terminal app to actually watch or drive an agent. That daemon-plus-menu-bar prototype was built ahead of the product and has been removed (`src/daemon.rs`, the daemon CLI, the daemon-client menu-bar Swift app, and the `jackin-notify` bridge); it stays recoverable from git history.

[CMUX](https://github.com/manaflow-ai/cmux) reframed the foundation: the control surface should *be* the terminal host. If the app already embeds the terminal, then "what's running / what needs review / which account is low" stop being a separate dashboard and become annotations on the sessions the operator is already looking at. So the new foundation is:

- **Window-first, not menu-bar-first.** The product is a full window modeled on CMUX, not a status-item popover. (A menu-bar glance can come back later as a convenience, not the primary surface.)
- **Embedded terminal, not external Ghostty.** The app embeds **libghostty** and renders terminal surfaces *inside* its own window. Ghostty itself is built on libghostty, and CMUX embeds the same library, so the surface matches Ghostty's look and feel exactly.
- **CLI-only for v1.** Each terminal surface runs the `jackin` CLI directly (`jackin load`). The app needs no daemon, no `jackin-protocol`, and no Capsule wiring of its own — jackin' already owns the container launch and attach.
- **Daemon deferred, not dropped.** The original control-surface features (what's running, PR inbox, account/quota state, attention routing) re-enter as the [enrichment plane](#deferred--the-daemon-enrichment-plane) once the terminal foundation exists — now wrapped around real embedded sessions instead of pointing at external Ghostty windows.

In short: the original idea is the destination; the terminal window is the foundation the original idea was missing. Build the foundation first, then grow the original surfaces on top of it.

## Primary reference — CMUX

Reference: https://github.com/manaflow-ai/cmux

CMUX describes itself as a *"Ghostty-based macOS terminal with vertical tabs and notifications for AI coding agents."* It is a native Swift/AppKit app that embeds libghostty for GPU terminal rendering, reads the operator's existing Ghostty config (themes, fonts, colors), and presents a left vertical-tab sidebar with per-session git branch, PR status, working directory, listening ports, and notification state. It supports native splits, an optional in-app browser, a scriptable socket/CLI, and layout/scrollback restore.

What jackin' borrows:

- The **libghostty-embedded native terminal** as the core surface.
- The **left sidebar** that lists work and groups it, so many concurrent sessions stay legible.
- **Attention affordances** (pane rings / tab highlights) for "this session needs you", to be wired when the event plane lands.
- An **in-app browser surface** alongside the terminal.

The jackin' delta — and the reason this is not just a CMUX clone:

- CMUX runs agents on **host worktrees**. jackin' runs every session in an **isolated container** via `jackin load`. Same native terminal experience, isolated substrate. The isolation is the product's whole point and the thing CMUX deliberately does not do.

## Product shape (v1)

The window has two tab axes:

- **Left sidebar** — workspaces, each grouping the sessions ("spaces") currently running under it. This is the operator's map of concurrent work.
- **Top tabs** — the surfaces *within* the active space. Surface types are the **jackin' terminal** (a libghostty surface) and a **browser** (WKWebView). v1 ships the terminal surface; the browser is the first additional surface type so the operator can, for example, open a preview or a PR next to the agent without leaving the app.

```text
┌────────────┬──────────────────────────────────────────┐
│ SIDEBAR    │ [ jackin term ] [ browser ] [ + ]   ← top tabs (this space)
│            │ ┌──────────────────────────────────────┐ │
│ ▾ acme     │ │                                      │ │
│   • api    │ │   libghostty surface                 │ │
│   • web    │ │   pty: jackin load acme              │ │
│ ▾ jackin   │ │   (agent runs isolated in container) │ │
│   • docs   │ │                                      │ │
│            │ └──────────────────────────────────────┘ │
└────────────┴──────────────────────────────────────────┘
```

New-session flow:

1. Operator picks a workspace from the sidebar and chooses "new session".
2. A native launch screen (SwiftUI) confirms workspace and role/agent. Workspace data comes from the `jackin` CLI as JSON — no daemon.
3. The app opens a new libghostty surface whose PTY runs `jackin load <workspace>`. jackin' provisions and attaches the container; the agent TUI renders inside the surface.
4. The session appears as a space under its workspace in the sidebar.

v1 starts **new sessions only**. Attaching to already-running sessions, splits, layout restore, and session persistence are deferred.

## Architecture (v1, CLI-only)

1. **`Jackin.app`** — native macOS app (Swift/AppKit + SwiftUI). Owns the window, sidebar, tab model, launch screen, and the embedded surfaces. This is net-new code; no part of the removed menu-bar app is reused.
2. **libghostty** — embedded via its C API for terminal rendering. The app builds a SwiftUI terminal-surface view around a libghostty surface and runs the `jackin` CLI as the surface's PTY process. Because the PTY runs `jackin load`, jackin' owns the container lifecycle and the Capsule attach; the app carries no `jackin-protocol` or Capsule code.
3. **WKWebView** — the browser surface type, a native WebKit view embedded as a top tab.
4. **`jackin` CLI** — the only backend in v1. The launch screen reads the saved workspace list as JSON; "new session" execs `jackin load`. Tab/session state is owned by the app, since the app is what spawned each surface.

The app must stay optional: installing the `jackin` CLI must not require the app, and installing the app must require or bootstrap a compatible `jackin` CLI.

## Deferred — the daemon enrichment plane

This is where the original control-surface goal lands. The removed host daemon was a real, tested control plane; it returns later as an **enrichment** layer the app reads when present, never a v1 dependency. These are exactly the "what's running / what needs review / which account is low / who is waiting" surfaces from the [Problem](#problem) — now drawn on top of the embedded sessions the operator is already looking at, rather than as a standalone dashboard:

- GitHub PR surfaces — "my open PRs", project inbox, and per-session branch→PR correlation with GitHub and DiffsHub review links.
- Account status — built-in-runtime (Claude, Codex, Amp, Kimi, OpenCode) credential-source presence, and later quota/reset/usage.
- Live session discovery — Capsule-backed running-session state (panes, runtime status, stale containers) rather than the app's own spawn list.
- Notifications and attention routing — waiting / blocked / ready-for-review events surfaced as sidebar rings and Notification Center clicks.

These live in their own roadmap items: [jackin' daemon](/reference/roadmap/jackin-daemon/), [Live bidirectional auth sync](/reference/roadmap/live-auth-sync/), [Agent attention prompts](/reference/roadmap/agent-attention-prompts/), [Token & Cost Telemetry](/reference/roadmap/token-cost-telemetry/), and [Host bridge — secrets and approved host actions](/reference/roadmap/host-bridge/). When the app grows these surfaces, the menu-bar references below come back into scope.

### Account / PR / menu-bar references (enrichment era)

These native macOS apps are references for the *enrichment* surfaces, not for the v1 terminal window. Borrow their UI patterns when those surfaces are built; do not inherit their broad provider catalogs.

- [CodexBar](https://github.com/steipete/CodexBar) — provider/account quota display, refresh loops, status-item split (AppKit owns the status item, SwiftUI renders rich cards). Avoid its broad provider catalog and browser-cookie-first Amp model as jackin' defaults.
- [RepoBar](https://github.com/steipete/RepoBar) — repository/PR submenus, cache-first GitHub state with ETags, bounded local checkout status. Keep jackin' centered on its own workspaces and running sessions, not all accessible GitHub repositories.
- [OpenUsage](https://github.com/robinebers/openusage) — menu-bar usage panel, cached-not-blocking data model, loopback-only local API.
- Tokemon, Token Tracker, Brim, MeterBar, Control Tower — quota ring / multi-account dashboard patterns; the data model and confidence labels are the [telemetry roadmap's contract](/reference/roadmap/token-cost-telemetry/).

## Native macOS design

Use a native SwiftUI/AppKit app targeting macOS Tahoe 26 where available while remaining usable on the minimum supported macOS.

- Embed **libghostty** for terminal surfaces; embed **WKWebView** for browser surfaces.
- Use SwiftUI `NavigationSplitView`, `List`, `Table`, `Toolbar`, and standard controls for the shell; keep content dense and developer-oriented.
- Use Liquid Glass only where system components provide it naturally (toolbars, sidebars, popovers). Do not build a custom frosted dashboard.
- Respect Reduce Transparency, Increase Contrast, and keyboard navigation; prefer SF Symbols for indicators.

Apple references:

- Liquid Glass overview: https://developer.apple.com/documentation/TechnologyOverviews/liquid-glass
- SwiftUI menu bar guidance: https://developer.apple.com/documentation/SwiftUI/Building-and-customizing-the-menu-bar-with-SwiftUI
- WWDC25 "Build a SwiftUI app with the new design": https://developer.apple.com/videos/play/wwdc2025/323

## Open questions

- **libghostty embedding contract.** The C API exists in `include/ghostty.h`: `ghostty_init`, `ghostty_app_new`, `ghostty_surface_new`, and a `ghostty_surface_config_s` that carries `command`, `working_directory`, `env_vars`, and a `platform.macos.nsview` handle — enough to embed a surface in an NSView and run `jackin load` as its command. The header explicitly warns it "isn't meant to be a general purpose embedding API (yet)", so it is experimental and will churn; CMUX is the working proof it is usable today. The bindings, the pinned libghostty/Zig version, and the update cadence need deciding at the libghostty spike (milestone 5).
- **Build and CI.** libghostty pulls in a Zig toolchain, and the Swift app only builds on macOS. CI on Linux cannot build or test the app — verification for the app is operator-side on macOS. Decide how the app is built, pinned, and smoke-tested outside Linux CI.
- **Launch-screen data.** The exact `jackin` CLI JSON shape the launch screen reads for the saved workspace list (and the command surface for "new session").
- **Tab / space model.** How spaces are grouped under workspaces, whether top tabs persist across relaunch, and how the browser surface is scoped (WKWebView only for v1; cookie import and scripting later).
- **Daemon re-entry.** When and how the enrichment plane re-attaches without becoming a hard dependency of the terminal window.

## Implementation plan

The app is built in small, independently previewable milestones. Each one is a single buildable step the operator runs on macOS (`cd desktop/macos && swift run JackinDesktop`) before the next begins — the agent runs on Linux and cannot build Swift/libghostty, so visual verification is operator-side. This checklist is the resume anchor for the work: the box marked in progress is where to pick up.

The app is a SwiftPM executable (Swift 6, macOS 14+) under `desktop/macos/`, CLI-only for v1. Workspace creation/editing stays in `jackin console`; the app launches and observes, it does not edit config.

| # | Milestone | Build | Preview | State |
|---|---|---|---|---|
| 1 | SwiftUI window that launches (`@main App`, activation delegate) | `swift run` | a window titled `jackin'` comes forward | **done — awaiting operator preview** |
| 2 | `NavigationSplitView` shell — left sidebar + empty detail | `swift run` | sidebar + detail panes render | next |
| 3 | Sidebar lists real **workspaces** | needs `jackin workspace list --json` (small Rust add, buildable on the Linux agent — dispatch is in <RepoFile path="src/app/mod.rs" />, no `--json` exists yet) | saved workspace names appear | — |
| 4 | Per-space **top-tab bar** in the detail area (empty tabs + new-tab button) | `swift run` | tabs switch | — |
| 5 | **libghostty spike** — one embedded surface running a shell | libghostty FFI + Zig toolchain; pin version here | a real terminal renders in-window | — |
| 6 | "New session" → surface runs `jackin load <workspace>` | `swift run` + `jackin` on PATH | an agent runs isolated inside a tab | — |
| 7 | **Browser** surface type (WKWebView) as a tab | `swift run` | a browser tab beside the terminal | — |

Milestone 5 (libghostty) is the highest-risk step because the embedding API is experimental; isolate it so churn there does not block the shell milestones. Milestone 3 is the only step that needs a jackin' CLI change, and that change is verifiable on Linux CI.

Once the foundation (milestones 1–7) is solid, the [enrichment plane](#deferred--the-daemon-enrichment-plane) layers the original control-surface features (running state, PR inbox, accounts, attention rings) on top.
