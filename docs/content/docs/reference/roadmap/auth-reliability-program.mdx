**Status**: Partially implemented — new-tab overwrite fix shipped; all other phases open

## Purpose of this document

This is the single source of truth for jackin's authentication reliability and operator convenience goals. Every individual roadmap item that touches auth design links here, and this document links back to each of them. The goal is that a contributor or operator who wants to understand where jackin' auth is heading, what is broken today, why it is broken, and what the complete fix looks like reads this page first — and then follows links to the individual items for implementation detail.

The problem this program addresses is not one bug but a class of failure: **auth that works at first but fails later, silently, in ways that force the operator to interrupt their work**. Every individual fix in this program targets one mechanism of that class. The program is complete only when the entire class is structurally closed — not just patched at the surface.

## The operator experience today

An operator working with jackin' across multiple projects or companies will eventually hit each of these failure modes. They are listed in rough order of frequency, from most to least common.

**Silent 401 after opening a new tab.** The operator opens a second agent tab in a running container. Moments later, all agent sessions in the container — including the one that was already working — receive a `401 authentication_error`. The cause: `jackin-capsule`'s runtime setup unconditionally re-copied the stale launch-time credential snapshot over the agent's refreshed token. The operator has no warning before it happens and no indication that the new tab caused it.

*This failure is fixed. See [Auth overwrite on new tab](/reference/roadmap/auth-overwrite-on-new-tab/).*

**Silent 401 after host token rotation.** The operator is running a long session. The host's Claude Code or GitHub CLI rotates an OAuth token in the background — this is normal and automatic. The container, holding a snapshot of the pre-rotation token, continues using the old value. The server invalidates the old token as a side effect of the rotation. The next API call from inside the container fails with `401`. The operator has no warning before it happens and no indication that token rotation is the cause.

*Not yet fixed. Requires the [jackin' daemon](/reference/roadmap/jackin-daemon/) and [live bidirectional auth sync](/reference/roadmap/live-auth-sync/).*

**Silent 401 in one of two parallel containers.** The operator is running two containers with the same agent and the same account — for example, two `the-architect` sessions against two different repositories under one Claude subscription. Container A refreshes the OAuth token. The server rotates the refresh token as part of that exchange, invalidating the grant Container B holds. Container B's next API call fails. The two containers competed for the same OAuth refresh grant without knowing about each other.

*Not yet fixed. Requires the same daemon-based live-sync shared store.*

**No way to check auth health without entering each container.** The operator has three workspaces running. They want to know whether all three are currently authenticated before starting a long task. The only way to check is to enter each container and look for a `401`. There is no `jackin auth status` command, no console indicator, no launch-time probe that surfaces credential validity before the container starts.

*Not yet fixed. Covered by [Auth health and operator visibility](/reference/roadmap/auth-health-and-visibility/).*

**Multi-company auth requires manual per-workspace configuration.** An operator working for two companies has two Claude accounts — a personal subscription and a company account. Both companies use Claude Code. Today the only way to point different workspaces at different accounts is to store credentials in non-default locations and manually configure each workspace, which requires knowing the internal `sync_source_dir` field that does not yet exist. Without it, every workspace reads from the same hardcoded `~/.claude` path — the operator must manually switch accounts on the host before launching each workspace, or accept that all workspaces share one account.

*Not yet fixed. Covered by [Per-scope auth-sync source folder](/reference/roadmap/auth-sync-source-folder/).*

**Token setup is a manual multi-step copy-paste process.** For `oauth_token` mode (the recommended mode for long-lived sessions), the operator must run `claude setup-token` on the host, copy the output, navigate to the correct workspace's auth tab, and paste it in — with no validation that the value they pasted is correct until the container starts and either works or fails. Rotation, revocation, and expiry are also manual.

*Partially fixed. Covered by [Workspace Claude token setup](/reference/roadmap/workspace-claude-token-setup/).*

**Credential files are visible in `docker inspect` and container filesystem.** Tokens forwarded via `api_key` mode appear in `docker run -e KEY=VALUE` and are readable via `docker inspect` by any process with Docker socket access on the host. The `sync`-mode state files under `~/.jackin/data/<container>/` are plaintext on the host filesystem.

*Not yet fixed. Tracked in [Container credential exposure](/reference/roadmap/container-credential-exposure/).*

## Why these failures share a root cause

All of the above — except the credential-exposure hardening — stem from the same fundamental model: **jackin' treats authentication as a one-time provisioning event at container launch, not as a runtime relationship that must be maintained**.

The `sync` mode reads the host credential file once, copies it, and binds it into the container. From that moment, the container's credential state is frozen. The host can change (token rotation, re-login, account switch), the agent inside the container can change (in-container refresh), and other containers can change (parallel session rotation) — none of it propagates anywhere. The frozen snapshot is the only input the container has. When the world moves and the snapshot does not, auth breaks.

The `oauth_token` and `api_key` modes are better — they inject a static value from the operator's env rather than a live file — but they share the same frozen-at-launch property for the env values themselves. An expired `CLAUDE_CODE_OAUTH_TOKEN` is not discovered until the container starts using it.

The new-tab overwrite bug was a special case of the frozen-snapshot model: the snapshot was overwriting a *live* in-container value, destroying a refresh that had already succeeded. Fixing that was the first step. The rest of the program is about making the snapshot either unnecessary (live sync) or at least transparent (health checks and visibility).

## The complete fix, phase by phase

### Phase 0 — new-tab overwrite fix (shipped)

The `jackin-capsule` runtime-setup path now captures `is_first_init` before the container-init marker is written and passes `copy_auth: bool` through `run_agent_setup()` and each `setup_*()` function. Credential copy only runs on first boot; subsequent `jackin-capsule new` invocations (new tabs) skip the copy and leave whatever the agent has on disk untouched.

This closes the most frequent failure mode with no daemon dependency.

**Shipped in:** this PR. **Detail:** [Auth overwrite on new tab](/reference/roadmap/auth-overwrite-on-new-tab/).

### Phase 1 — multi-company auth isolation (no daemon required)

Add optional `sync_source_dir` to `AgentAuthConfig` at both the global and per-workspace layers, resolved by a companion function `resolve_sync_source_dir` that mirrors `resolve_mode`'s three-layer logic. Each `provision_*_auth` function takes the resolved path and falls back to the current hardcoded default when it is `None`, so the change is purely additive.

Wire the field into the console's per-agent auth dialog: a **Source folder** row appears below the mode row whenever the mode is `sync` (the default), pre-filled with `(default) ~/.claude` (or the per-agent equivalent). Pressing Enter opens the existing directory picker with `show_hidden = true` so dotfiles are visible. The operator can override the folder per workspace and reset to default.

This closes the multi-company isolation gap without requiring any runtime machinery. An operator with a personal Claude account and a company Claude account can point each workspace at the correct credential directory and never manually switch accounts on the host.

This is a versioned schema change: `CURRENT_CONFIG_VERSION` and `CURRENT_WORKSPACE_VERSION` both advance, with migration steps, fixtures, and a Timeline entry.

**Not yet shipped.** **Detail:** [Per-scope auth-sync source folder](/reference/roadmap/auth-sync-source-folder/).

### Phase 2 — auth health and visibility (no daemon required)

Add a pre-launch credential validity probe that runs as part of `RoleState::prepare()` and emits results into the existing launch summary. The probe is local-only — file existence, non-empty check, basic JSON parse, and JWT `exp` claim decode for oauth_token mode. It never makes a network call. Result icons (`✓` / `⚠` / `✗`) appear in the per-agent auth rows that jackin' already prints before starting the container.

Add `jackin auth status` as a new CLI subcommand. It reads the same config resolution chain that the launch path uses, runs the probe for every workspace × agent combination, and prints a structured health table. `--json` flag for scripting.

Add health indicators to the console Auth tab: every agent row gains a column showing the last-probe result, computed lazily when the tab is focused. Add a small health glyph to running instance rows in the workspace sidebar.

This surfaces "your credentials are missing / expiring" at launch time and at glance, rather than mid-session when the first API call fails.

**Not yet shipped.** **Detail:** [Auth health and operator visibility](/reference/roadmap/auth-health-and-visibility/).

### Phase 3 — daemon foundation (lifecycle + socket)

Ship `jackin daemon serve` / `start` / `stop` / `status` / `logs` with the lifecycle, install (launchd LaunchAgent on macOS, systemd user unit on Linux), control socket at `~/.jackin/run/jackin-daemon.sock`, and log redaction. No watchers yet. The daemon is the prerequisite for everything in Phase 4.

`jackin auth status --live` gains the ability to query the daemon socket when present, returning real-time health for running instances rather than a static file-based snapshot.

**Not yet shipped.** **Detail:** [jackin' daemon](/reference/roadmap/jackin-daemon/).

### Phase 4 — live bidirectional auth sync (requires daemon Phase 3)

Add per-axis watcher adapters to the daemon: one for `gh`, one for Claude, one for Codex, and one for each additional agent axis. Each adapter watches the host credential source (via inotify on Linux, polling with optional Keychain callbacks on macOS) and writes changes to a flock-protected shared store at `~/.jackin/auth-shared/<axis>/`.

Add a small static `jackin-auth-watcher` binary to the construct image. It uses inotify on the agent credential files inside the container. When an in-container refresh writes a new token, the watcher pushes the new value to the shared store bind-mount, making it visible to the daemon and through the shared store to sibling containers.

Replace the per-container provisioned-snapshot bind-mount (today's `/jackin/claude/credentials.json`) with a bind-mount of the shared store path for containers in `live` mode. Containers in `sync` (`forward`) mode continue using the snapshot for backward compatibility.

Rename `sync` to `forward` (or `snapshot`) in the same PR to free `sync` for bidirectional meaning. This is a versioned schema rename across `config.toml`, workspace files, and role manifests, with migrations.

This closes the host-rotation and parallel-container-invalidation failure modes. With live sync active, token rotation propagates within seconds to all running containers. The `401`-from-rotation class of failure becomes structurally impossible for operators using live mode.

**Not yet shipped.** **Detail:** [Live bidirectional auth sync](/reference/roadmap/live-auth-sync/).

### Phase 5 — unified credential source abstraction

Replace the ad-hoc per-agent credential resolution in `provision_*_auth` with a unified `CredentialSource` enum covering literal, `env`, `command`, `op://` (1Password), OS secret store (Keychain on macOS, libsecret on Linux), and file-path sources. The enum is lazily resolved at consumer need, values are wrapped in a `SecretString` newtype (zeroize-on-drop, redacted in Debug), and resolution failures carry contextual errors.

Every auth axis (`[claude]`, `[codex]`, `[github]`, and any future axis) accepts `Option<CredentialSource>` instead of the current per-axis custom field. The `sync_source_dir` from Phase 1 becomes the `file` source kind. The `oauth_token` env reference becomes the `env` source kind. 1Password refs become the `op` source kind.

This makes every auth path first-class, composable, and auditable through one abstraction rather than five separate implementations that can drift.

**Not yet shipped.** **Detail:** [Credential source pattern](/reference/roadmap/credential-source-pattern/).

### Phase 6 — credential exposure hardening (no strict ordering)

Move from env-injected credentials (visible in `docker inspect`) to file-mount-based injection (Compose-secrets style, not in inspect output) for the agents that accept file-path credential inputs. Medium-term: per-command opaque handles via the daemon — the agent gets a handle, not the raw token value, and the daemon verifies the handle before issuing the actual credential. Long-term: a Docker Sandboxes-style credential proxy that intercepts outbound API calls and substitutes tokens at the network layer, so tokens never appear in container memory.

This is independent of all other phases — it can be developed in parallel and does not block or depend on live sync.

**Not yet shipped.** **Detail:** [Container credential exposure](/reference/roadmap/container-credential-exposure/).

### Phase 7 — mid-session secret and host-action requests (requires daemon Phase 3)

Add a host-bridge MCP server, auto-registered in containers, that lets an agent request a secret or an approved host action mid-session via a TouchID/polkit/password-gated approval flow. This closes the last remaining "operator must restart to give the container a new credential" gap.

**Not yet shipped.** **Detail:** [Host bridge — secrets and approved host actions](/reference/roadmap/host-bridge/).

## The "magical" end state

When all phases are complete, the operator experience is:

- Launch any workspace → auth loads automatically from the correct account, with a clear one-line health result in the launch summary. No manual setup if credentials exist at the default or configured source.
- Token rotates on the host or inside a container → all running containers see the new token within seconds. `401` from rotation is no longer possible.
- Two parallel containers running the same agent account → the shared store serializes refresh-token exchanges; neither container holds a grant the server has already revoked.
- Credentials expire → the launch probe warns at least 7 days before expiry, and the console Auth tab and `jackin auth status` show the approaching expiry with a clear path to rotate.
- Multi-company operator → each workspace reads from the correct company account, configured once via the source-folder picker, never needing a manual host-side account switch.
- Agent needs a new credential mid-session → host-bridge MCP tool prompts the operator once; no container restart.
- Credentials never appear in `docker inspect`; tokens are opaque handles inside containers.

## Design constraints that apply across all phases

**The host is never mutated silently.** Every write jackin' makes to host-side credential state — even in live-sync mode — must be surfaced in the launch summary or daemon status, and must be opt-in through the workspace's chosen auth mode. Read-only access to host credential files is unrestricted. The full rationale is in `AGENTS.md` under "Never mutate the host machine silently."

**Token values are never logged.** Every code path that handles raw token values must use the same redaction pattern as `GithubAuthContext`'s manual `Debug` impl: token fields are replaced with `[REDACTED]` in debug output. This applies to the pre-launch probe, the `jackin auth status` output, the daemon log, and any future credential-source resolver. No "temporary debug logging" exceptions.

**Container-path convention: `/jackin/` only.** All credential mounts into containers remain under `/jackin/<agent>/`. The shared-store bind-mount from Phase 4 goes under `/jackin/auth-shared/<axis>/`. No new top-level container paths.

**One schema version bump per PR.** Phase 1 (`sync_source_dir`), Phase 4 (`sync` → `forward` rename), and Phase 5 (credential source field) are each a separate schema bump with the full migration artifact set. A single PR must not introduce more than one version.

**Backward compatibility.** Each phase must be a clean upgrade for existing configs. `sync` mode behavior must not change until Phase 4's rename PR, and even then migration must be automatic and tested end-to-end by the fixture harness.
