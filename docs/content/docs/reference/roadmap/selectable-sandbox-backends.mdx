title: "Selectable Sandbox Backends"

**Status**: Open — umbrella for Docker and Apple Container as the two runtime families

## Problem

jackin' currently has one production runtime backend: Docker with a per-instance role container, per-agent Docker network, and private Docker-in-Docker sidecar. That backend is useful and should remain the default while jackin' is local-first.

The broader product routes agent sessions to one of two runtime families:

- **Docker** — standard local development backend; remains the default
- **Apple Container** (`apple-container`) — macOS 26 ARM VM backend; each session gets its own Linux kernel via Virtualization.framework

This page is the umbrella. Detailed backend work lives in focused items linked below. The backend selection model, instance registry design, and operator console surface live here.

## Product Direction

1. **Docker-first, always.** Docker matches how role authors package tools, how agents build and test software, and how operators inspect what happened. It stays the default.
2. **Harden Docker before switching runtime families.** The immediate gap is not that jackin' uses Docker; it is that jackin' does not yet expose a precise hardening contract. See [Docker runtime hardening contract](/reference/roadmap/docker-runtime-hardening-contract/).
3. **Apple Container is the macOS 26 VM backend.** Each container gets its own Linux kernel via Virtualization.framework, with zero install friction on macOS 26. Apple Container is not a Docker replacement — it is a VM primitive that jackin' builds its credential injection, network policy, and session model on top of. See [Apple Container backend](/reference/roadmap/apple-container-backend/).
4. **Docker Sandboxes is a benchmark, not a backend.** Docker Sandboxes is a complete agent runner — it runs Claude Code, Codex, and others in managed VMs with built-in credential and network proxies. jackin' learns from its security model but does not integrate with it as a backend (it is a competitor, not a provider). See [Agent isolation architecture](/reference/roadmap/agent-isolation-architecture/) for the full comparison.
5. **Kubernetes is future platform expansion.** Not part of the current Docker or Apple Container design pass.

The core product bet:

> jackin' is the orchestration, policy, and session layer. Runtimes are pluggable enforcement providers below that layer.

## Current Focus

Work proceeds in this order:

1. **[Docker runtime hardening contract](/reference/roadmap/docker-runtime-hardening-contract/)** — make the Docker backend explicit, auditable, and progressively stricter. Also the prerequisite for Apple Container: rootless DinD (replacing `--privileged`) must validate before Apple Container Phase 0 can start.
2. **[Apple Container backend](/reference/roadmap/apple-container-backend/)** — macOS 26 VM backend, Phase 0 empirical validation after rootless DinD lands.
3. **[jackin-exec](/reference/roadmap/jackin-exec/)** — secure execution wrapper; credential injection without exposing raw secrets to agents. Can start before Apple Container lands.
4. **Kubernetes platform support** — future, not part of this design pass.

**[smolvm backend research](/reference/roadmap/smolvm-backend-research/) is deferred.** Research in June 2026 established that Apple Container (zero install friction, gRPC/vsock attach, Virtualization.framework) is the better macOS 26 VM choice when rootless DinD validates. smolvm remains the fallback if Apple Container Phase 0 fails the Docker-inside compatibility test.

**[OrbStack isolated machine backend](/reference/roadmap/orbstack-isolated-machine-backend/) is deferred.** OrbStack isolated machines share OrbStack's Linux kernel — namespace isolation, not a hypervisor boundary.

## Threat Model Map

| Scenario | User problem | Recommended direction | Why |
|---|---|---|---|
| Local trusted operator | Run personal agents without trashing the host workflow. | Docker `standard` profile. | Best UX, existing role compatibility, explicit mounts, private DinD. |
| Suspicious repository | Let an agent inspect/build unknown code without broad host access. | Docker `hardened` first; Apple Container `apple-container` if Docker controls are insufficient. | First wins are network deny/allowlist, resource caps, read-only root, no unnecessary DinD. Apple Container adds per-session kernel boundary. |
| Credential-sensitive workflows | Agent must connect to servers, registries, cloud APIs without seeing raw keys. | Docker or Apple Container + [jackin-exec](/reference/roadmap/jackin-exec/). | VM boundary doesn't protect credentials in env vars; jackin-exec does. |
| Long autonomous agent | Limit runaway cost, resource, network, credential exposure. | Docker `hardened` + session contract + network policy + resource limits + jackin-exec. | VM boundary helps kernel escapes; egress/credentials/resources fail without the policy layers. |
| Shared workstation or team laptop | Keep agents from reaching local services, broad files, or shared Docker state. | Apple Container (`apple-container`) — each session gets its own kernel, own Docker daemon, own IP. | True kernel isolation per session. |
| Hosted or multi-tenant future | Run code for strangers or customers. | Not supported today. Requires audited hypervisor isolation + policy proxy + brokered credentials. | Containers share the host kernel; hostile multi-tenant workloads need a stronger tenant boundary. |

## Runtime Comparison

| Concern | Hardened Docker | Apple Container | Docker Sandboxes |
|---|---|---|---|
| Primary value | Improve the runtime jackin' already ships. | True VM per session (own kernel) on macOS 26 ARM. | Reference design: private Docker daemon + credential proxy + network proxy in a VM. Not a jackin' backend. |
| Isolation boundary | Containers on selected Docker engine; shared OrbStack VM kernel. | Own Linux kernel per session via Virtualization.framework. | Dedicated microVM per sandbox via proprietary VMM. |
| Docker workflow fit | Excellent; current default. | Rootless DinD inside VM (requires Phase 0 validation). | Private Docker Engine is a core product feature. |
| Host Docker risk | Host daemon still builds/starts resources. | No outer Docker daemon needed for runtime; build path separate. | No path to host Docker daemon. |
| Host filesystem risk | Explicit mounts; writable mounts can still be modified. | Explicit bind mounts; same absolute host path. | Explicit workspace mount; same absolute host path. |
| Network policy | Needs jackin' egress work. | Per-container IP via vmnet; session-level egress policy. | Host-side proxy, allow/deny policy, raw non-HTTP blocked. |
| Credential handling | Env/file injection; jackin-exec for per-command injection. | Same + jackin-exec works inside VM. | Host-side credential proxy built in. |
| Persistence | jackin' state/reconnect/eject/purge model. | Same model; AppleContainerResources backend variant. | `sbx rm` lifecycle; not aligned with jackin' session model. |
| Install friction | None — Docker already present. | Zero — built into macOS 26. | Requires Docker Desktop. |
| UX maturity | Highest for current users. | New in macOS 26; rough edges (DNS after sleep/wake). | Polished runner, but less aligned with jackin' role/session model. |
| Licensing/control | Docker Engine open. | `apple/container` open source. | Docker product surface; paid governance features. |
| Implementation complexity | Lowest. | Medium: capsule PID-1 patch, Phase 0 validation, backend adapter. | Not a backend integration path. |

## Backend Vocabulary

| Backend | Status | Meaning |
|---|---|---|
| `docker` | Active default | Current Docker engine backend, hardened by policy profiles over time. |
| `apple-container` | Active (Phase 0 pending) | macOS 26 ARM backend via Virtualization.framework; own kernel per session. |
| `kubernetes` | Future | Cluster backend for team/production debug workflows. |
| `smolvm` | Deferred — fallback if Apple Container Phase 0 fails | Open-source Rust/libkrun VM backend. Revisit if apple-container cannot support Docker workflows. |
| `orbstack-isolated` | Deferred — shared-kernel, namespace only | OrbStack isolated machine backend. Shares OrbStack Linux kernel; does not add hypervisor boundary. |

## Non-Goals

- Do not replace Docker as the default local backend before the Docker hardening contract lands.
- Do not claim every backend has the same risk posture.
- Do not hide provider limitations behind `auto`.
- Do not silently mutate host state to make a backend work.
- Do not integrate Docker Sandboxes as a backend — it is a competitor product, not a provider API.
- Do not move Kubernetes into the Docker hardening or Apple Container scope.

## Product Contract

Every backend must implement the same high-level jackin' operations:

| Operation | Requirement |
|---|---|
| launch | Start a role environment with resolved mounts, auth, env, and runtime policy. |
| attach | Connect the operator to the in-runtime capsule session. |
| inspect | Report instance state without destructive actions. |
| reconnect | Recover a stopped/running runtime when possible. |
| eject | Stop runtime resources while preserving recovery state. |
| purge | Delete jackin'-owned runtime state for that instance. |
| explain | Print the exact backend, provider, policy, mounts, credentials, resources, network behavior, and residual risks before launch. |

## Backend-Neutral Instance Registry

Docker container names cannot remain the source of truth once two backend families exist. Today per-instance metadata lives in <RepoFile path="src/instance/manifest.rs" /> as `InstanceManifest` with a Docker-specific `DockerResources` field. The registry shape needs a `BackendResources` tagged union:

```text
instance_id
backend_kind         (docker | apple-container | kubernetes | ...)
workspace identity or ad-hoc path
role selector
agent runtime
mount materialization handles
auth/state handles
backend-specific resource handles (tagged union, one shape per backend_kind)
created_at
last_seen_at
cleanup status
```

Concrete layout:

- Per-instance manifest stays at `~/.jackin/data/<instance>/manifest.json`. The `DockerResources` field becomes `backend_resources: BackendResources` — a tagged union with `DockerResources` and `AppleContainerResources` variants. See [Apple Container backend](/reference/roadmap/apple-container-backend/) for the `AppleContainerResources` struct shape.
- A top-level index at `~/.jackin/data/instances.json` lists `{ instance_id, backend_kind, workspace, last_seen_at }` so `jackin inspect` / `jackin list` does not walk every per-instance directory.
- `InstanceManifest` is per-instance ephemeral metadata, not a versioned schema. Add `manifest_version: u32` for graceful fallback when a stale manifest is read.

## Runtime Capability Contract

Every backend must provide:

- shell execution
- `jackin-capsule` as the session authority
- role image or equivalent role environment
- explicit workspace/global mounts
- configured agent state
- configured authentication mode
- Git identity setup
- network policy reporting
- resource policy reporting
- optional Docker-compatible inner engine when the role requires Docker

The optional inner engine matters: the current role ecosystem expects Docker workflows. A backend that cannot provide a private Docker-compatible engine should not be treated as a full replacement for the Docker backend.

## Experiment Plan

Evidence gathering in this order:

1. **Docker hardening smoke matrix** — compat behavior per agent runtime, contract reporting for seccomp/AppArmor/caps/no-new-privileges/resource limits/DinD, test candidates against common workflows.
2. **Rootless DinD validation** — `docker:dind-rootless` against `docker build`, Compose, Testcontainers; compare startup/storage/cleanup against privileged DinD.
3. **Apple Container Phase 0** — boot jackin' role OCI image, capsule non-PID-1 with `JACKIN_CAPSULE_FORCE_DAEMON`, `container exec` PTY attach, rootless DinD inside VM, workspace bind mounts, network policy. Decision gate: if rootless DinD passes → proceed to Apple Container Phase 1; if fails → smolvm Phase 0 fallback.
4. **Docker Sandboxes benchmark** — run the same agent workflow where access/licensing permits; use as the reference for what credential proxy + network proxy + private Docker Engine looks like in production.

Each experiment must answer: what user problem became meaningfully easier or safer? What jackin' UX, role compatibility, or debuggability did it make worse?

## Backend Selection and `--backend auto`

The `--backend` flag precedence:

1. **CLI flag** — `jackin load … --backend apple-container`. Operator-explicit always wins.
2. **Workspace override** — `[workspaces.X.runtime] backend = "..."` in workspace config.
3. **Role manifest hint** — `[runtime] backend_preference = "..."` in `jackin.role.toml` (advisory; operator can override).
4. **Global default** — `[runtime] default_backend = "..."` in `config.toml`. Ships as `"docker"`.

`--backend auto` decision rule (deterministic):

```text
1. If JACKIN_BACKEND env var is set, use it.
2. Else if a workspace override is set, use it.
3. Else if a global default is set in config.toml, use it.
4. Else:
   - macOS 26+ host + apple/container CLI present
     → apple-container (only if [runtime.apple_container] enabled = true; else fall through)
   - Any host with Docker available
     → docker (the safe default)
   - Otherwise: error with "no backend available" and list what was probed.
```

`auto` is never a backend identity at runtime; the launch contract always names the resolved backend.

## TUI / Operator Console Surface

Multi-backend changes what `jackin console` shows per instance. Today the per-instance details pane renders Docker-specific state. The pane needs a backend-aware variant:

```text
backend:   apple-container
provider:  apple/container 0.11.0 on macOS 26 ARM
container: jackin-abcd1234
kernel:    own (Virtualization.framework)
inner:     rootless Docker inside VM
mounts:    2 explicit bind mounts
network:   per-container IP, allowlist (4 hosts)
attach:    container exec (gRPC/vsock)
```

vs the current Docker shape:

```text
backend:   docker
provider:  OrbStack Docker engine on macOS ARM
role:      jackin-the-architect-abc
dind:      jackin-dind-abc (rootless, TLS)
network:   per-agent bridge, open egress
attach:    docker exec
```

## Later Provider Research

These remain possible only if a later operator problem justifies them:

- **smolvm**: deferred; revisit if Apple Container Phase 0 fails the Docker-inside test. smolvm uses Hypervisor.framework + libkrun (open-source, cross-platform), documented Docker-in-VM path in v0.7.0.
- **Kata Containers** through containerd or Kubernetes RuntimeClass.
- **Docker Sandboxes direct integration**, if the `sbx` CLI/API ever proves suitable as a lifecycle backend rather than an agent runner.
- **gVisor** as a syscall-sandboxing runtime — confirmed broken on OrbStack macOS ARM in 2026 (runsc symlink issue), Claude Code hangs in gVisor on macOS ARM.
