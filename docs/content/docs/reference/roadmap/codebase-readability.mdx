## Goal

Make AI-generated code verifiable and maintainable. The codebase is ~170K lines of production Rust. The TUI half is already restructured ŌĆö every terminal surface lives in its own crate behind one Elm Architecture boundary (see [TUI Design](/reference/tui/)). The unsolved half is the **root `src/` binary crate**: ~108K lines of runtime, config, instance, isolation, app-dispatch, manifest, and CLI code that is monolithic, duplicated, undocumented, and ŌĆö the part no previous audit admitted ŌĆö **tangled into a cyclic dependency graph that cannot be split into crates as-is**.

The end state is a tiered Cargo workspace where each crate has one clear responsibility and the dependency graph is a DAG. This single program covers both the readability work (deduplication, file splits, module docs) and the crate extraction, because **they are the same motion**: you cannot extract a clean crate from a 9,482-line file full of duplicated literals and circular dependencies. The crate boundary is what forces the cleanup, and the cleanup is what makes the boundary reviewable.

## The target: a tiered core-crate workspace

The non-visual core splits into four crates plus a thin binary. The boundary rule is **data and contracts versus behavior**: the small, universal vocabulary every layer speaks lives at the bottom; the heavy machinery that reads, writes, and orchestrates it lives above.

```
jackin-core            ŌåÉ leaf. no deps, no tokio, no docker, no filesystem
  universal non-config vocabulary + contracts:
  Agent, MountIsolation, AuthForwardMode, RoleSource,
  RoleManifest + manifest types, paths, selector,
  and the DockerApi / CommandRunner traits (the I/O seam)

jackin-config          ŌåÆ core
  config + workspace, types AND behavior together:
  AppConfig, WorkspaceConfig, MountConfig, *AuthConfig,
  KeepAwakeConfig, version constants, ConfigEditor, migrations,
  validate_mounts, plan_collapse, planner, workspace resolution

jackin-env             ŌåÆ core, config
  operator_env / 1Password reference resolution (CLI + console both use it)

jackin-runtime         ŌåÆ core, config, env
  the container bootstrap pipeline: runtime, instance, isolation,
  derived_image. Holds the concrete DockerApi / CommandRunner impls.

jackin (binary)        ŌåÆ all of the above
  cli argument structs + app dispatch, kept thin

+ existing surface crates: jackin-tui, jackin-console,
  jackin-launch, jackin-capsule, jackin-protocol
```

**Why `jackin-core` is separate from `jackin-config` (and is not just "more config").** The `*Config` structs are config-file-shaped data with config behavior, so they stay in `jackin-config` ŌĆö that crate owns everything named "config." But a handful of types are *universal vocabulary* that is not about config at all: `Agent` is named by nine modules (`instance` 49├Ś, `runtime` 90├Ś, `console` 33├Ś, `cli`, `app`, `manifest`, `workspace`, `config`); `MountIsolation` is a four-variant enum `config` needs 49├Ś; the `DockerApi` / `CommandRunner` traits are the seam every test and runtime path depends on. If those lived in `jackin-config`, then `instance` would have to depend on the entire config crate ŌĆö TOML parsing, the migration engine, `ConfigEditor`, workspace planning ŌĆö just to name `Agent::Claude`. `jackin-core` keeps that vocabulary cheap, stable, and fast to compile, separate from the heavy config machinery that churns. The split line in one sentence: **config-file-shaped struct or config behavior ŌåÆ `jackin-config`; universal enum, trait, or path that even foundation modules need regardless of config ŌåÆ `jackin-core`.**

This was verified, not assumed: nothing destined for `jackin-core` reaches up for a `*Config` type (`selector` needs only `instance`; the `manifest` *types* need only `Agent`; the docker traits reference no config), so the `*Config` types cleanly stay up in `jackin-config`.

## The real blocker: a cyclic dependency graph

The previous plan claimed "verified tiers" with `workspace/` and `manifest/` as clean leaves. That was false. The actual graph has cycles:

- **`config Ōåö workspace`** ŌĆö `config ŌåÆ workspace` 47 refs, `workspace ŌåÆ config` 50 refs. A direct mutual cycle.
- **`config ŌåÆ isolation ŌåÆ workspace ŌåÆ config`** ŌĆö a three-node cycle (`config` needs `MountIsolation`; `isolation` needs `MountConfig`/`ResolvedWorkspace`; `workspace` needs `AppConfig`).
- **`config Ōåö operator_env`** (missed in the original audit) ŌĆö `config/persist` and `config/editor` call `operator_env::validate_reserved_names`. Cut in the feature/tui-architecture branch.
- **`config ŌåÆ runtime::list_role_names`** (claimed cut in Stage 1, but was still live) ŌĆö `detect_workspace_edit_drift` in `config/workspaces/mod.rs` called `runtime::list_role_names`. Cut by moving `detect_workspace_edit_drift` to `runtime/drift.rs`.
- **`env_model Ōåö manifest`** (missed in the original audit) ŌĆö `env_model::topological_env_order` took a `&BTreeMap<String, manifest::EnvVarDecl>` param. Cut by moving `EnvVarDecl` to `jackin-core`.
- **`agent ŌåÆ config`** ŌĆö `Agent` and `AuthForwardMode` enums needed by config. Cut by moving them to `jackin-core`.
- `manifest ŌåÆ agent` / `manifest ŌåÆ config` ŌĆö these are **clean one-directional downward edges**, not bidirectional pulls. No cycle to cut.

Cargo forbids cycles *between* crates, so no crate boundary can be drawn until these are broken. The decisive finding: **the cycles are ~95% shared data types, not shared logic.** `isolation ŌåÆ workspace` is `MountConfig` and `ResolvedWorkspace`; `agent ŌåÆ config` is `AuthForwardMode`; `manifest ŌåÆ agent` is `Agent`. All `#[derive(Serialize, Deserialize)]` structs and enums.

The standard Rust fix for a mutual-type cycle is a foundation crate: move the shared *type definitions* down into `jackin-core`, leaving the *behavior* above. The moment the shared vocabulary lives below everything that names it, every one of these cycles dissolves at once. The residual `config`-logic Ōåö `workspace`-logic coupling (config calls `validate_mounts`/`plan_collapse`; workspace calls `migrations`/`ConfigEditor`) is genuine logic coupling ŌĆö which is exactly why config and workspace become **one** `jackin-config` crate rather than two.

## Conventions every refactor follows

This program is a long sequence of moves that look mechanical ŌĆö extract a constant, split a file, draw a crate boundary. Each one is a chance to raise or lower the bar, so every PR under this program holds to the same Rust standard. The checks below are what a reviewer verifies against the diff; they compose with the existing repo rules in <RepoFile path="AGENTS.md" /> ("Reuse before writing ŌĆö DRY", "Prefer libraries over hand-rolled parsers", "Code comments ŌĆö explain only what is not obvious").

**Crate boundaries (Stages 1 and 4).**
- Public API is deliberate: `pub(crate)` by default, `pub` only what the next crate up imports. A crate's public surface is designed, not leaked. Keep struct fields private unless the representation is part of the stable contract (the serde `*Config` structs expose fields intentionally; most other types should not).
- `jackin-core` is types + traits only ŌĆö no `tokio`, no subprocess, no filesystem. The I/O seam is the `DockerApi` / `CommandRunner` traits (dependency inversion); the concrete implementations live in `jackin-runtime`.
- Encode invariants in types: newtypes and enums over `bool` / `String` / bare primitives, so every higher crate inherits the constraint instead of re-checking it.
- Derive the standard traits wherever they make semantic sense on the `jackin-core` vocabulary types ŌĆö `Debug`, `Clone`, `PartialEq`/`Eq`, `Hash`, `Default` for data; `Display` + `std::error::Error` for error types; `From` / `TryFrom` for conversions ŌĆö so downstream crates get debuggable, comparable, ergonomic values.
- Treat `derive(Serialize, Deserialize)`, public re-exports, and public dependencies as intentional compatibility commitments, not defaults ŌĆö they belong only where the serialized shape or the re-export is part of the contract.
- Use a sealed trait where a downstream `impl` would block future evolution (the agent-runtime trait is the candidate).

**Shared extractions (Stage 2).**
- Extract in a shape future callers reuse ŌĆö a parameter or small enum on the existing function beats a second near-copy.
- Borrowed parameters when ownership is not required: `&str` over `String`, `&[T]` over `Vec<T>`, `&Path` over `PathBuf`, `Option<&T>` over `&Option<T>`. Take ownership only when the function stores, sends, or moves the value, so the caller decides where any clone happens.
- `.clone()` is a design decision, never a borrow-checker appeasement. Skip intermediate `Vec` / `String` allocations when an iterator or slice does the job.
- Add external dependencies conservatively. Internal crate splits are free; a new third-party crate carries compile, supply-chain, and public-API cost. This composes with the repo's "prefer a maintained crate over hand-rolling" rule ŌĆö reach for an existing ecosystem crate before writing a parser, but do not add a dependency the workspace does not already carry without justifying it in the PR.

**File splits and functions (Stage 3).**
- Optimize for the first-time reader: entry-point items first, important types before helpers, implementation near its use.
- Keep control flow local and visible; express preconditions in types or adjacent checks, not in a distant helper. Prefer clear control flow over a clever iterator/combinator chain when the chain hides errors, branching, ownership, or side effects.
- Boring, searchable names over abbreviations ŌĆö short names only for established conventions (`ctx`, `idx`, `res`, `db`).
- Every newly split module gets a `//!` contract that states the non-obvious why ŌĆö invariants, ordering, race windows ŌĆö not a restatement of the code.

**Errors.**
- `Result<T, E>` for expected failure; `Option<T>` only when absence carries no useful detail. Typed errors (`enum` + `thiserror`-style) at crate boundaries; `anyhow`-style context only at the binary / CLI / app edge. The new `jackin-core` / `-config` / `-runtime` crates are libraries ŌĆö they return typed errors, and the root binary adds the operator-facing context.
- `?` for propagation, with context added at the I/O, parse, network, and user-facing boundaries.
- Reserve `unwrap` / `expect` for tests, unreachable invariants, and programmer errors with precise messages. Convert any production `unwrap` you touch to `expect("why")` or a typed error; do not add new ones.
- Document public failure on library functions with a rustdoc `# Errors` section (and `# Panics` where a panic is reachable, `# Safety` for any `unsafe`).

**Tests and docs (Stages 0 and 3).**
- Tests are readable behavior examples at a crate's **stable boundary** ŌĆö test what the public surface does, not the incidental private helpers you are about to move. A test that freezes a private signature blocks the very refactor this program performs; that is the failure mode to avoid when splitting files.
- Keep the `insta` snapshot net for rendered, serialized, or CLI output; keep snapshots deterministic and redacted so a refactor that changes structure but not output produces an empty diff.
- Public-API examples go in doc tests that propagate with `?`, not `unwrap`, so the example also documents the error path.
- Cover the failure path, not just the success path ŌĆö the behavioral specs in Stage 0 exist precisely to pin error behavior before a file moves.

**Lints and validation.**
- Fix a Clippy warning before suppressing it. A justified suppression uses `#[expect(clippy::lint_name, reason = "ŌĆ”")]` so it fails loudly when it goes stale ŌĆö and a new `#[allow(clippy::too_many_lines)]` is a request to extract, not annotate.
- The new library crates carry no crate-level `#![deny(warnings)]` (it makes a future compiler warning a hard build break) and do not switch on `pedantic` / `restriction` / `nursery` groups wholesale ŌĆö apply strict lints selectively, by name, with agreement.
- Each PR runs `cargo fmt --check`, `cargo clippy --all-targets --all-features --locked -- -D warnings`, and the test suite before it is ready.

## Module layout migration plan

As part of the crate extraction (Stage 4), every new workspace crate under `crates/` follows **Rust 2024 self-named module layout** (`foo.rs` + `foo/` instead of `foo/mod.rs`). This is enforced by `clippy::mod_module_files = "deny"` in each new crate's Cargo.toml.

The migration applies in three phases:

**Phase 1 (current):** New crates (`jackin-config`, `jackin-env`, `jackin-runtime`) start clean with the self-named layout and enforce it via Clippy.

**Phase 2:** When a module is extracted from the legacy root `src/` crate into a new workspace crate, its files are migrated to the self-named layout as part of that extraction PR. No big-bang migration needed.

**Phase 3:** When the last module leaves `src/`, the root binary crate itself enforces `mod_module_files = "deny"`. The `crates/AGENTS.md` convention document already captures this rule for contributors.
