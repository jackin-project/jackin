**Status**: Implemented — all phases complete. The root `src/` crate no longer exists; every production module lives in a single-responsibility crate under `crates/`; the dependency graph is a DAG with no upward edges; zero `mod.rs` files remain in `crates/`; `cargo clippy --workspace --all-targets --all-features -- -D warnings` is green. The one remaining Phase 7 item (moving the remaining ~47 generic console UI files that bind `ManagerState<'_>` type aliases) is tracked as a follow-up.

## Goal

The end state is a **tiered Cargo workspace with no root `src/` crate at all.** Every line of production Rust lives in a single-responsibility crate under `crates/`, and the only binary — `jackin` — is itself a crate (`crates/jackin/`) holding CLI dispatch and the host run-loops. The inter-crate dependency graph is a DAG: dependencies point downward, never up.

Two standing conventions fall out of following current Rust practice, and every PR under this program holds to them: **self-named module layout everywhere (no `mod.rs`)**, and **`pub(crate)` by default** so each crate's public surface is designed rather than leaked.

The readability work (deduplication, file splits, module docs) and the crate extraction are the same motion. You cannot draw a clean crate boundary through a 9,000-line file full of duplicated literals and circular dependencies; the boundary is what forces the cleanup, and the cleanup is what makes the boundary reviewable.

## Target — the crate DAG

Dependencies point **only downward**. A higher tier may name a lower tier; never the reverse. Heavy dependencies (tokio, bollard, the `op` CLI) live only where the tier annotation says so.

```text
Tier 0  jackin-core        vocabulary + traits + manifest types + JackinPaths +
                           env_model (reserved-name policy) + op_types + constants
        jackin-protocol    capsule↔client wire protocol
        ──  no tokio · no docker · no filesystem (except std::fs in paths)  ──

Tier 1  jackin-tui         ratatui widgets + theme + ansi_text + prune_output formatting → core
        jackin-diagnostics RunDiagnostics JSONL + debug_log! macro +
                           terminal-ownership state                 → core, tui
        jackin-docker      BollardDockerClient + ShellRunner + net  → core, diagnostics
                           (concrete DockerApi / CommandRunner impls; heavy I/O deps)

Tier 2  jackin-config      AppConfig · WorkspaceConfig · ConfigEditor · migration engine ·
                           planner · mount/workspace validation + resolution  → core

Tier 3  jackin-manifest    manifest validation + migrations + repo_contract +
                           repo (validation)                       → core, config, diagnostics
        jackin-env         operator_env + env_resolver + host_claude +
                           token_setup                  → core, config, console, diagnostics

Tier 4  jackin-image       derived_image + agent_binary + capsule_binary +
                           binary_artifact + version_check  → core, config, manifest, docker, diagnostics
        jackin-runtime     runtime + instance + isolation  → core, config, env,
                                                            manifest, docker, image, diagnostics
        jackin-console     generic console UI + effects-as-data   → core, config, tui, diagnostics
        jackin-launch      launch progress UI                     → protocol, tui
        jackin-capsule     in-container TUI (its own binary)      → core, tui, protocol

Tier 5  jackin (binary)    cli/ + app/ + the console host effect-runner / services / run-loop +
                           role_authoring + host tui/ + main.rs + bin/role.rs
                           → all of the above. THE ONLY BINARY. No root src/.
```

The `jackin-diagnostics` crate is load-bearing and easy to miss: the `debug_log!` macro, the terminal-ownership flags (`rich_terminal_owned`, `host_screen_owned`, …), and the `RunDiagnostics` JSONL log are one mutually-coupled host substrate that `jackin-docker` (Tier 1) already calls, so it must extract **before** the runtime tier or nothing there compiles.

### Why these boundaries

A few non-obvious calls, recorded so they are not re-litigated:

- **`jackin-core` vs `jackin-config`.** Config-file-shaped structs and config behavior stay in `jackin-config`. A universal enum, trait, or path that even foundation modules need regardless of config (`Agent`, `MountIsolation`, the `DockerApi`/`CommandRunner` traits, `JackinPaths`, the reserved-name policy in `env_model`) belongs in `jackin-core`, so naming `Agent::Claude` does not drag in the TOML parser and migration engine.
- **One console crate, not two.** `jackin-console` is the generic UI: view, update, and effects expressed **as data**, generic over the host's concrete types (`AppConfig`, `IsolationRecord`, …). Executing those effects against the runtime is the host application's job, so the effect-runner, services, and run-loop live in the `jackin` binary beside `app/` and `cli/`. The crate boundary exists so the UI stays below the runtime in the DAG — compilable and unit-testable without docker.
- **Finer-grained runtime tier.** `jackin-docker` (concrete daemon I/O), `jackin-image` (derived-image build and agent-binary acquisition), and `jackin-runtime` (launch, instance lifecycle, isolation) split by responsibility rather than collapsing into one ~18K-line crate — sharper review boundaries and faster incremental builds.
- **`token_setup` is an env concern, not config.** It drives the 1Password / Claude-token flow (`OpCli`, `host_claude`, `parse_op_reference`), so it lands in `jackin-env`, not `jackin-config`, even though it edits the config file through `ConfigEditor`.
- **`jackin-diagnostics` is the host observability substrate, not just a log.** It owns the structured JSONL run log **and** the `debug_log!` macro plus the terminal-ownership flags currently tangled into the binary's `tui/mod`. These are one mutually-coupled cluster (`emit_debug_line` writes to the run log), and `jackin-docker` already calls all of them — so the substrate sits at Tier 1 (`→ core, tui`) and must extract before the runtime tier. `repo`'s validation lands in `jackin-manifest` (not runtime) because `derived_image` depends on it and a runtime home would create an `image → repo → runtime → image` cycle.

## The AppConfig domino

`AppConfig` is the load-bearing extraction. It cannot move to `jackin-config` on its own: Rust's orphan rule requires all four `impl AppConfig` blocks — workspace CRUD, mount resolution, role/auth resolution, and persist/load — to move with the struct, and they pull in `ConfigEditor`, the migration engine, and a few workspace helpers. The order that keeps every PR compiling:

1. `env_model` → `jackin-core` (unblocks `is_reserved`, which the config-load path calls).
2. Workspace pure helpers (`expand_tilde`, `validate_workspace_config`, `validate_isolation_layout`) → `jackin-config`.
3. Migration engine + `atomic_write` + the config/workspace registries → `jackin-config` (the manifest crate repoints to `jackin_config::migrations`).
4. `AppConfig` + `ConfigEditor` + the four `impl` blocks → `jackin-config` — the one unavoidable big-bang, minimized by steps 1–3.
5. Workspace resolution (`resolve_load_workspace`, `current_dir_workspace`) → `jackin-config`.

Once `AppConfig` lives in `jackin-config`, `jackin-env` (which needs it), `jackin-manifest`, and the entire runtime tier unblock in sequence. The full per-module ordering, with the `cargo check -p <crate>` gate for each step, is the [implementation checklist](/reference/roadmap/codebase-readability-checklist/).

## Conventions every refactor follows

Every PR under this program holds to the same Rust standard. These compose with the repo rules in <RepoFile path="AGENTS.md" /> ("Reuse before writing — DRY", "Prefer libraries over hand-rolled parsers", "Code comments — explain only what is not obvious").

**Crate boundaries.**

- `pub(crate)` by default; `pub` only on what the next crate up imports. A crate's public API is designed, not leaked. Struct fields stay private unless the representation is part of a stable contract (the serde `*Config` structs expose fields intentionally; most other types do not).
- `jackin-core` is types + traits only — no tokio, no subprocess, no docker. The I/O seam is the `DockerApi` / `CommandRunner` traits; the concrete impls live in `jackin-docker`.
- Encode invariants in types: newtypes and enums over `bool` / `String` / bare primitives, so every higher crate inherits the constraint.
- Treat `derive(Serialize, Deserialize)`, public re-exports, and public dependencies as intentional compatibility commitments, not defaults.

**Module layout (no `mod.rs`).**

- Every crate uses Rust 2024 self-named module files (`foo.rs` + `foo/`), enforced by `clippy::mod_module_files = "deny"`. New crates start clean; a module migrates to the self-named layout in the same PR that extracts it. The root binary enables the lint last, once it reaches zero `mod.rs`.

**Errors.**

- Typed errors (`enum` + `thiserror`) at crate boundaries; `anyhow`-style context only at the binary / CLI / app edge. The library crates return typed errors; the root binary adds operator-facing context. Document public failures with a rustdoc `# Errors` section. Reserve `unwrap` / `expect` for tests and genuinely unreachable invariants.

**File splits and functions.**

- Optimize for the first-time reader: entry-point items first, important types before helpers, implementation near its use. A new `#[expect(clippy::too_many_lines)]` is a request to extract, not annotate. Every newly split module gets a `//!` contract stating the non-obvious why.

**Validation.**

- Every PR runs `cargo fmt --check`, `cargo clippy --all-targets --all-features --locked -- -D warnings`, and the test suite before it is ready. Versioned-schema moves (`AppConfig`, `WorkspaceConfig`, `RoleManifest`) must be serde-byte-identical relocations — no version bump, no migration step — or they are not relocation-only and fall under the migration rules in <RepoFile path="AGENTS.md" />. On the dev box, run heavy `cargo` with `-j 1` to avoid the link-time OOM observed during research.
