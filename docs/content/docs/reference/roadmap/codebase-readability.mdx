**Status**: Open — active program. Stages 0 and 1 complete (2026-06-02): `jackin-core` leaf crate ships with `Agent`, `MountIsolation`, `AuthForwardMode`, and shared constants; cyclic dependency graph broken. Stage 2 DRY pass ongoing. Stage 3 (monolith splits) next.

## Goal

Make AI-generated code verifiable and maintainable. The codebase is ~170K lines of production Rust. The TUI half is already restructured — every terminal surface lives in its own crate behind one Elm Architecture boundary (see [TUI Design](/reference/tui/)). The unsolved half is the **root `src/` binary crate**: ~108K lines of runtime, config, instance, isolation, app-dispatch, manifest, and CLI code that is monolithic, duplicated, undocumented, and — the part no previous audit admitted — **tangled into a cyclic dependency graph that cannot be split into crates as-is**.

The end state is a tiered Cargo workspace where each crate has one clear responsibility and the dependency graph is a DAG. This single program covers both the readability work (deduplication, file splits, module docs) and the crate extraction, because **they are the same motion**: you cannot extract a clean crate from a 9,482-line file full of duplicated literals and circular dependencies. The crate boundary is what forces the cleanup, and the cleanup is what makes the boundary reviewable.

## The target: a tiered core-crate workspace

The non-visual core splits into four crates plus a thin binary. The boundary rule is **data and contracts versus behavior**: the small, universal vocabulary every layer speaks lives at the bottom; the heavy machinery that reads, writes, and orchestrates it lives above.

```
jackin-core            ← leaf. no deps, no tokio, no docker, no filesystem
  universal non-config vocabulary + contracts:
  Agent, MountIsolation, AuthForwardMode, RoleSource,
  RoleManifest + manifest types, paths, selector,
  and the DockerApi / CommandRunner traits (the I/O seam)

jackin-config          → core
  config + workspace, types AND behavior together:
  AppConfig, WorkspaceConfig, MountConfig, *AuthConfig,
  KeepAwakeConfig, version constants, ConfigEditor, migrations,
  validate_mounts, plan_collapse, planner, workspace resolution

jackin-env             → core, config
  operator_env / 1Password reference resolution (CLI + console both use it)

jackin-runtime         → core, config, env
  the container bootstrap pipeline: runtime, instance, isolation,
  derived_image. Holds the concrete DockerApi / CommandRunner impls.

jackin (binary)        → all of the above
  cli argument structs + app dispatch, kept thin

+ existing surface crates: jackin-tui, jackin-console,
  jackin-launch, jackin-capsule, jackin-protocol
```

**Why `jackin-core` is separate from `jackin-config` (and is not just "more config").** The `*Config` structs are config-file-shaped data with config behavior, so they stay in `jackin-config` — that crate owns everything named "config." But a handful of types are *universal vocabulary* that is not about config at all: `Agent` is named by nine modules (`instance` 49×, `runtime` 90×, `console` 33×, `cli`, `app`, `manifest`, `workspace`, `config`); `MountIsolation` is a four-variant enum `config` needs 49×; the `DockerApi` / `CommandRunner` traits are the seam every test and runtime path depends on. If those lived in `jackin-config`, then `instance` would have to depend on the entire config crate — TOML parsing, the migration engine, `ConfigEditor`, workspace planning — just to name `Agent::Claude`. `jackin-core` keeps that vocabulary cheap, stable, and fast to compile, separate from the heavy config machinery that churns. The split line in one sentence: **config-file-shaped struct or config behavior → `jackin-config`; universal enum, trait, or path that even foundation modules need regardless of config → `jackin-core`.**

This was verified, not assumed: nothing destined for `jackin-core` reaches up for a `*Config` type (`selector` needs only `instance`; the `manifest` *types* need only `Agent`; the docker traits reference no config), so the `*Config` types cleanly stay up in `jackin-config`.

## The real blocker: a cyclic dependency graph

The previous plan claimed "verified tiers" with `workspace/` and `manifest/` as clean leaves. That was false. The actual graph has cycles:

- **`config ↔ workspace`** — `config → workspace` 47 refs, `workspace → config` 50 refs. A direct mutual cycle.
- **`config → isolation → workspace → config`** — a three-node cycle (`config` needs `MountIsolation`; `isolation` needs `MountConfig`/`ResolvedWorkspace`; `workspace` needs `AppConfig`).
- **`agent → config`** and **`manifest → agent` / `manifest → config`** — the `Agent` and `AuthForwardMode` enums pulled in both directions.

Cargo forbids cycles *between* crates, so no crate boundary can be drawn until these are broken. The decisive finding: **the cycles are ~95% shared data types, not shared logic.** `isolation → workspace` is `MountConfig` and `ResolvedWorkspace`; `agent → config` is `AuthForwardMode`; `manifest → agent` is `Agent`. All `#[derive(Serialize, Deserialize)]` structs and enums.

The standard Rust fix for a mutual-type cycle is a foundation crate: move the shared *type definitions* down into `jackin-core`, leaving the *behavior* above. The moment the shared vocabulary lives below everything that names it, every one of these cycles dissolves at once. The residual `config`-logic ↔ `workspace`-logic coupling (config calls `validate_mounts`/`plan_collapse`; workspace calls `migrations`/`ConfigEditor`) is genuine logic coupling — which is exactly why config and workspace become **one** `jackin-config` crate rather than two.

## Conventions every refactor follows

This program is a long sequence of moves that look mechanical — extract a constant, split a file, draw a crate boundary. Each one is a chance to raise or lower the bar, so every PR under this program holds to the same Rust standard. The checks below are what a reviewer verifies against the diff; they compose with the existing repo rules in <RepoFile path="AGENTS.md" /> ("Reuse before writing — DRY", "Prefer libraries over hand-rolled parsers", "Code comments — explain only what is not obvious").

**Crate boundaries (Stages 1 and 4).**
- Public API is deliberate: `pub(crate)` by default, `pub` only what the next crate up imports. A crate's public surface is designed, not leaked. Keep struct fields private unless the representation is part of the stable contract (the serde `*Config` structs expose fields intentionally; most other types should not).
- `jackin-core` is types + traits only — no `tokio`, no subprocess, no filesystem. The I/O seam is the `DockerApi` / `CommandRunner` traits (dependency inversion); the concrete implementations live in `jackin-runtime`.
- Encode invariants in types: newtypes and enums over `bool` / `String` / bare primitives, so every higher crate inherits the constraint instead of re-checking it.
- Derive the standard traits wherever they make semantic sense on the `jackin-core` vocabulary types — `Debug`, `Clone`, `PartialEq`/`Eq`, `Hash`, `Default` for data; `Display` + `std::error::Error` for error types; `From` / `TryFrom` for conversions — so downstream crates get debuggable, comparable, ergonomic values.
- Treat `derive(Serialize, Deserialize)`, public re-exports, and public dependencies as intentional compatibility commitments, not defaults — they belong only where the serialized shape or the re-export is part of the contract.
- Use a sealed trait where a downstream `impl` would block future evolution (the agent-runtime trait is the candidate).

**Shared extractions (Stage 2).**
- Extract in a shape future callers reuse — a parameter or small enum on the existing function beats a second near-copy.
- Borrowed parameters when ownership is not required: `&str` over `String`, `&[T]` over `Vec<T>`, `&Path` over `PathBuf`, `Option<&T>` over `&Option<T>`. Take ownership only when the function stores, sends, or moves the value, so the caller decides where any clone happens.
- `.clone()` is a design decision, never a borrow-checker appeasement. Skip intermediate `Vec` / `String` allocations when an iterator or slice does the job.
- Add external dependencies conservatively. Internal crate splits are free; a new third-party crate carries compile, supply-chain, and public-API cost. This composes with the repo's "prefer a maintained crate over hand-rolling" rule — reach for an existing ecosystem crate before writing a parser, but do not add a dependency the workspace does not already carry without justifying it in the PR.

**File splits and functions (Stage 3).**
- Optimize for the first-time reader: entry-point items first, important types before helpers, implementation near its use.
- Keep control flow local and visible; express preconditions in types or adjacent checks, not in a distant helper. Prefer clear control flow over a clever iterator/combinator chain when the chain hides errors, branching, ownership, or side effects.
- Boring, searchable names over abbreviations — short names only for established conventions (`ctx`, `idx`, `res`, `db`).
- Every newly split module gets a `//!` contract that states the non-obvious why — invariants, ordering, race windows — not a restatement of the code.

**Errors.**
- `Result<T, E>` for expected failure; `Option<T>` only when absence carries no useful detail. Typed errors (`enum` + `thiserror`-style) at crate boundaries; `anyhow`-style context only at the binary / CLI / app edge. The new `jackin-core` / `-config` / `-runtime` crates are libraries — they return typed errors, and the root binary adds the operator-facing context.
- `?` for propagation, with context added at the I/O, parse, network, and user-facing boundaries.
- Reserve `unwrap` / `expect` for tests, unreachable invariants, and programmer errors with precise messages. Convert any production `unwrap` you touch to `expect("why")` or a typed error; do not add new ones.
- Document public failure on library functions with a rustdoc `# Errors` section (and `# Panics` where a panic is reachable, `# Safety` for any `unsafe`).

**Tests and docs (Stages 0 and 3).**
- Tests are readable behavior examples at a crate's **stable boundary** — test what the public surface does, not the incidental private helpers you are about to move. A test that freezes a private signature blocks the very refactor this program performs; that is the failure mode to avoid when splitting files.
- Keep the `insta` snapshot net for rendered, serialized, or CLI output; keep snapshots deterministic and redacted so a refactor that changes structure but not output produces an empty diff.
- Public-API examples go in doc tests that propagate with `?`, not `unwrap`, so the example also documents the error path.
- Cover the failure path, not just the success path — the behavioral specs in Stage 0 exist precisely to pin error behavior before a file moves.

**Lints and validation.**
- Fix a Clippy warning before suppressing it. A justified suppression uses `#[expect(clippy::lint_name, reason = "…")]` so it fails loudly when it goes stale — and a new `#[allow(clippy::too_many_lines)]` is a request to extract, not annotate.
- The new library crates carry no crate-level `#![deny(warnings)]` (it makes a future compiler warning a hard build break) and do not switch on `pedantic` / `restriction` / `nursery` groups wholesale — apply strict lints selectively, by name, with agreement.
- Each PR runs `cargo fmt --check`, `cargo clippy --all-targets --all-features --locked -- -D warnings`, and the test suite before it is ready.

## The plan — staged, bottom-up

### Stage 0 — Safety net and an honest map

- Land the [behavioral spec for `runtime/launch.rs`](/reference/roadmap/behavioral-spec-runtime-launch/) and [op_picker](/reference/roadmap/behavioral-spec-op-picker/), and extend the [snapshot net](/reference/roadmap/snapshot-tests-tui/) — the regression guard for everything below.
- Generate the real module dependency graph and mark every cycle. The "tiers" must be measured, not assumed.

### Stage 1 — Build `jackin-core`, break the cycles

Move the universal vocabulary into a new leaf crate: `Agent`, `MountIsolation`, `AuthForwardMode`, `RoleSource`, the `RoleManifest` + manifest types, `paths`, `selector`, and the `DockerApi` / `CommandRunner` traits. This is type-level surgery — a type definition moves to `jackin-core` while the behavior that uses it (e.g. `isolation::materialize`, `config::ConfigEditor`) stays put and references `jackin_core::…`. Cut the two trivial upward edges (`config → runtime::list_role_names`, `config → instance::naming`) by moving those functions down or inverting them through a trait. Result: the root crate's internal graph becomes a DAG, and `jackin-core` compiles standalone.

### Stage 2 — DRY pass

Reduce duplication before it is frozen into crate APIs. Every extraction has 3+ real call sites.

- **Shared string constants** — none extracted yet: `MANIFEST_FILENAME` (`"jackin.role.toml"`, 160×), `DOCKERFILE_NAME` (`"Dockerfile"`, 235×), `BASE_DOCKERFILE_FROM` (~100×). These belong in `jackin-core`.
- **Named style constants** — collapse the 235+ `Style::default().fg(…).add_modifier(BOLD)` chains in the TUI crates into `BOLD_WHITE` / `DIM` / `GREEN` / `BORDER` / `DANGER` in `jackin-tui`.
- **Auth provisioning** — <RepoFile path="src/instance/auth.rs" /> (3,302 L) has six near-identical `provision_*_auth` functions; extract `provision_file_credential(target, host_path, mode)`.
- **Agent dispatch** — adding an agent currently edits eight files with `Agent::Claude => …` arms (<RepoFile path="src/config/roles.rs" />, <RepoFile path="src/instance/mod.rs" />, <RepoFile path="src/runtime/launch.rs" />, <RepoFile path="src/runtime/image.rs" />, <RepoFile path="src/derived_image.rs" />, <RepoFile path="src/agent_binary.rs" />, <RepoFile path="src/version_check.rs" />, <RepoFile path="src/manifest/validate.rs" />). Replace with trait dispatch — tracked as the [Agent runtime trait](/reference/roadmap/agent-runtime-trait/) item.
- **Docker resource derivation** and **test infrastructure** — extract a resource-name struct + `run_dind_sidecar` helper, and consolidate the repeated `FakeRunner` / `seed_minimal_role_repo` / `test_workspace` / `simple_mount` helpers into a shared support module.

### Stage 3 — Split and document the monoliths

Snapshot/behavioral coverage and the Stage 2 DRY pass must land on a file before it is split. Four files have dedicated design pages:

- [Split `runtime/launch.rs`](/reference/roadmap/split-runtime-launch/) — **9,482 L** (was 7,966), the hardest file; behavioral spec required first.
- [Split `console/tui/input/editor.rs`](/reference/roadmap/split-input-editor/) — **4,212 L**, one module per editor tab.
- [Split `operator_env.rs`](/reference/roadmap/split-operator-env/) — **4,209 L**, the future `jackin-env`.
- [Split `app/mod.rs`](/reference/roadmap/split-app-mod/) — **3,966 L**; `fn run` alone is 1,406 L.

Other root files over 2,000 L: <RepoFile path="src/instance/auth.rs" /> (3,302, after the auth DRY pass), <RepoFile path="src/console/tui/input/global_mounts.rs" /> (2,871), <RepoFile path="src/console/tui/state.rs" /> (2,707), <RepoFile path="src/console/tui/input/mouse.rs" /> (2,676), <RepoFile path="src/isolation/materialize.rs" /> (2,441), <RepoFile path="src/config/editor.rs" /> (2,434), <RepoFile path="src/workspace/token_setup.rs" /> (2,388), <RepoFile path="src/isolation/finalize.rs" /> (2,134). In parallel, write `//!` module contracts — six `src/` modules (`isolation`, `instance`, `app`, `manifest`, `agent`, `tui`) have **zero** coverage; see [Module contracts](/reference/roadmap/module-contracts/). Mega-functions regressed to **53** `too_many_lines` suppressions (was 35); a new suppression in a PR is a request to extract, not annotate.

Three crate-side files also exceed 2,000 L and should be split, lower priority than the root work: <RepoFile path="crates/jackin-capsule/src/daemon/mod.rs" /> (3,901 L, but ~2,900 of that is one `#[cfg(test)]` module — split the tests into per-submodule files first), <RepoFile path="crates/jackin-capsule/src/tui/components/dialog.rs" /> (3,207), and <RepoFile path="crates/jackin-console/src/tui/components/op_picker.rs" /> (2,877).

### Stage 4 — Extract the crates bottom-up

With the cycles broken (Stage 1) and the monoliths split (Stage 3), draw the boundaries leaf-first:

1. **`jackin-core`** already stands from Stage 1.
2. **`jackin-config`** — config + workspace logic over the core types. Confirm no residual upward edge into runtime/instance survives.
3. **`jackin-env`** — `operator_env` after its split.
4. **`jackin-runtime`** — runtime + instance + isolation + `derived_image`, holding the concrete `DockerApi` / `CommandRunner` impls (dependency inversion: the traits are in core, the subprocess implementation is here).
5. The root `src/` collapses to a thin CLI-dispatch binary over the crates.

Each extraction is gated on the visibility pass for that slice — `pub(crate)` by default, `pub` only what the next crate up imports — so every crate's public API is a deliberate surface, not an accidental one.
