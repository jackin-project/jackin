## Goal: idiomatic Rust is a first-class deliverable of Phase 2 (best-practices charter)

Phase 2 is a refactor/restructure at scale, so **applying Rust's own best practices is an explicit goal, not a side effect.** Every Phase 2 change is held to this bar; reviewers block on violations the same way they block on a failing test. The charter (each item is enforceable against a diff):

1. **Total functions at boundaries — never panic on data.** Functions that consume runtime input (parsers, render/ANSI encoders, config readers) must not `unwrap()`/`expect()`/`panic!`/index on values derived from that input. Return `Result`/`Option`, match exhaustively, fall back dynamically. *This is the exact class that crashed every container this session (`rgb_bg(DANGER_RED)` panic, Defect 40); a `const` allowlist that panics is fine for compile-time tables, never for render-time values.*
2. **Make illegal states unrepresentable.** Prefer the type system over runtime checks: the `AgentRuntime`/`Provider` registry replaces 414 sites of closed-enum dispatch; newtypes carry invariants; enums + exhaustive matches catch "added a variant, forgot a site." The agent-runtime-trait epic *is* this principle.
3. **DRY through traits/parameters, never parallel copies** (AGENTS DRY rule). Symmetric variants are *data*, not duplicated control flow — one adapter table, not five `provision_*` functions; one resolver, not parallel `resolve_*`.
4. **Prefer maintained crates; own the layer only when the crate is wrong-shaped.** `vte` is a dependency (never rebuild a parser); `jackin-term` is a *deliberate, documented* exception because no crate ships our diff-to-escape stage. Justify every hand-roll in the PR.
5. **Rust 2024 module layout** — self-named files, no `mod.rs` (crates/AGENTS); `snake_case`/`UpperCamelCase`/`SCREAMING_SNAKE_CASE` naming.
6. **Error handling discipline.** `anyhow` + `.with_context()` at app edges; typed errors in library crates; `?` propagation; **no silent failures** (compose with the single-shareable-debug-log goal, Defect 41 — every failure path emits a durable, surfaced error).
7. **Async-first; nothing blocks a render or runtime thread** (Defect 43). Subprocess/network/large-fs through `tokio` or `spawn_blocking`; `Send`/`Sync` correctness; structured concurrency (`join!`/`try_join!`) for independent work.
8. **Zero-alloc / zero-cost in hot paths.** Packed types, reused buffers, borrow over clone, iterators over intermediate `Vec`s — the per-frame `String` storm is the anti-pattern `jackin-term` kills (Defect 45).
9. **Ownership clarity.** Take `&str`/`&[T]`, return owned only when ownership is needed; kill needless clones/lifetimes (clippy).
10. **`cargo fmt --check` + `cargo clippy --workspace --all-targets --all-features --locked -- -D warnings` are non-negotiable gates** on every commit.
11. **Tests as design pressure.** Unit + property + (for `jackin-term`) fuzz + differential/golden; `#[must_use]` where dropping the result is a bug; doc examples on public API.
12. **API design.** `&'static dyn Trait` registries for dispatch; builders for many-arg constructors; `#[non_exhaustive]` on enums that will grow; newtypes over bare `String`/`bool` where the value has meaning.
13. **`unsafe` is forbidden unless justified, encapsulated, and `// SAFETY:`-commented.** None is expected in Phase 2.
14. **Comments explain WHY, not WHAT** (AGENTS rule); every borrowed design (Ghostty/Alacritty/Zellij patterns in `jackin-term`) carries an attribution + license comment.

**Library-crate refinements (canonical `rust-best-practices`).** The new and touched crates (`jackin-term`, `jackin-protocol`, `jackin-tui`, the `observability` surface, the config/runtime crates) are *libraries* in a workspace, so they carry the stricter library bar on top of the 14 items above:

- **Borrowed params, precisely:** `&str`/`&[T]`/`&Path` over owned; `Option<&T>` over `&Option<T>`; take ownership only when the function stores/moves/sends the value. `.clone()` is a design decision, never a borrow-checker appeasement.
- **Typed errors in libraries; `anyhow` only at binary/CLI edges.** A library returns a typed error enum (`thiserror`-style) so callers can match; `anyhow` + `.with_context()` belongs in the `jackin` binary / CLI / test helpers. Document public failure with rustdoc `# Errors` / `# Panics` / `# Safety` sections.
- **Encode invariants in types, not `bool`/`String`/`Option` parameters.** Newtypes, enums, builders, type-state, validated wrappers. Keep public fields private unless the representation is a stable contract. The `AgentRuntime` registry trait should be **sealed** (downstream impls would block evolution of the built-in set); the built-ins are the only implementors.
- **Suppression discipline.** Fix Clippy findings before suppressing; if justified, `#[expect(clippy::lint, reason = "…")]` (not bare `#[allow]`) so stale suppressions surface. Do **not** enable `pedantic`/`nursery`/`restriction` groups wholesale, and do **not** put crate-level `#![deny(warnings)]` in a reusable library — the `-D warnings` gate is a CI flag, not baked into the crate, so a future compiler warning cannot brick downstream builds.
- **Conservative dependencies / stable boundaries.** Every dep carries compile-time + supply-chain + public-API cost; avoid public re-exports, blanket generic impls, and `serde` derives on public types unless they are intentional compatibility commitments. IO, serialization (the JSONL schema), protocols (`jackin-protocol`), and feature flags (`otlp`) are explicit boundaries.
- **Tests/docs as the library contract.** Behavior tests at stable boundaries (not tests freezing incidental helpers); doc-tests on public API that propagate with `?`; **snapshot tests for rendered/serialized output** (the `jackin-term` golden wire-emit, the diagnostics JSONL golden) kept deterministic and redacted.

When a Phase 2 entry's "Done when" is met but the diff violates the charter, it is **not** done. The charter lives here once and is referenced by the checklist so every executor shares one bar. **Run the `rust-best-practices` review pass** (`references/review-checklist.md`) over every Phase 2 diff before marking work complete, with the canonical validation: `cargo fmt --check`, `cargo clippy --workspace --all-targets --all-features --locked -- -D warnings`, `cargo test --all-targets --all-features --locked`.

## Goal: zero dead code (operator directive, Round 3 — Defect 61)

The workspace ends this branch with **zero dead code**, enforced by a layered toolchain rather than any single tool, because "dead code" is several distinct classes in Rust and each needs its own detector. The full executable plan is [Defect 61 in the checklist](/reference/roadmap/post-restructure-fixes-checklist/); the policy, summarized once here:

**Hard rule — deny always, never keep.** Every dead-code lint is `deny` from the moment it is configured, permanently. We never stage a lint at `warn`, never downgrade a level to turn a red build green, and never keep dead code "for later" without a written `#[expect(..., reason)]`. When a deny fires, the resolution is always to adjust the code — delete, narrow, remove — iteration by iteration until the build is clean at the bar. The bar never moves down; the code moves up to it.

1. **Delete what no lint can see first.** Source that the module system never compiles (the Defect 49 orphaned trees) is invisible to every lint and scanner — deletion is the only fix for that class, and it is the prerequisite for everything below.
2. **Compiler lints in one workspace table.** The hoisted `[workspace.lints.rust]` table (Defect 56) carries `dead_code = "deny"`, `unused_imports = "deny"`, `unused_variables = "deny"`, and `unreachable_pub = "deny"` — all four at deny from day one per the hard rule — with every crate opting in via `lints.workspace = true`. `unused_crate_dependencies` deliberately stays at its `allow` default — rustc documents its false positives across Cargo target layouts; the dedicated dependency scanners below cover that class better.
3. **Narrow visibility first, scan second, delete third.** rustc's `dead_code` lint is blind on `pub` items, so the `pub`-everywhere habit hides dead API from the compiler. The sweep demotes every `pub` item not consumed outside its crate to `pub(crate)`/`pub(super)`, which arms `dead_code` to find — and the sweep to delete — the genuinely dead items. The workspace is the full world (no published crates, pre-release), so an unused `pub` item really is dead. The sweep runs crate by crate under the always-on deny: each iteration shrinks the error list, never the lint level.
4. **Dependency + unlinked-file scanner.** `cargo-shear` (fast, PR-blocking, in the CI `check` job) detects unused dependencies, misplaced dependencies (`[dependencies]` entries only tests use), and **unlinked source files** — the last being exactly the Defect 49 orphan class, so the CI step doubles as the permanent guard against orphaned source re-accumulating. `cargo-udeps` was retired because it needs nightly and overlaps the dependency hygiene class that shear already enforces on stable. `cargo-machete` is not used — shear covers its class and more.
5. **Retired temporary unused-`pub` scanner.** `cargo-workspace-unused-pub` was useful during the restructure sweep, but the branch now relies on the strict workspace lint table (`dead_code`/`unreachable_pub` at deny), narrow-visibility policy, and clippy gate with `-D warnings` rather than carrying the niche wrapper in CI.
6. **Suppression discipline.** Anything intentionally kept while unused carries `#[expect(dead_code, reason = "…")]` — never bare `#[allow]` — so stale suppressions self-report. No blanket module- or crate-level allows.

The enforcement shape: `fmt → check → clippy → shear → nextest` on every PR; cargo-deny advisories, cargo-hack, and fuzz in a scheduled/manual hygiene job. A finding from any layer is either deleted or carries a written `#[expect]` reason — there is no third state. Generic guidance prefers staging these lints at `warn` in large workspaces; jackin' deliberately overrides that with the deny-always rule above — nothing here is a published library, so the caution that motivates staging does not apply.

## Goal: latest-and-greatest Rust baseline (operator directive, Round 3 — Defect 62)

jackin' always rides the latest stable Rust with the most modern workspace configuration. The executable plan is [Defect 62 in the checklist](/reference/roadmap/post-restructure-fixes-checklist/); the policy, summarized once here:

- **Edition 2024 + resolver 3.** The workspace is already on edition 2024 everywhere; the root manifest moves from `resolver = "2"` to `resolver = "3"` (the edition-2024 MSRV-aware resolver, set explicitly because a virtual workspace cannot infer it from a package edition).
- **One source of truth for manifest metadata.** `edition`/`rust-version`/`license`/`repository` are inherited from `[workspace.package]` via `edition.workspace = true` etc. — the 17 literal copies in crate manifests are a DRY drift, removed.
- **The canonical strict lint table** (the Defect 56 hoist target, contents in Defect 62.3/62.4): edition/compat groups at deny, the Defect 61 dead-code denies, the Rust 2024 unsafe-hygiene lints at deny, `unsafe_code = "forbid"` kept (stricter than the generic app-profile `deny`), `clippy::all` deny + `pedantic` warn + cherry-picked restriction lints, the wholesale `nursery` group dropped per Clippy's own guidance, groups at `priority = -1`. `warnings = "deny"` is deliberately **not** baked into the manifest — the charter's rule stands (`-D warnings` is a CI flag; a manifest-baked deny would brick local builds on every new toolchain warning, exactly the failure mode for a project that rides every new stable).
- **Toolchain freshness with reproducibility.** `rust-toolchain.toml` stays pinned to an exact latest-stable version (currently 1.96.0) and is raised within days of each 6-week release via an automated Renovate bump PR that runs the full gate suite; the MSRV (`rust-version`, verified by the `msrv` CI job) remains a separate, deliberately-raised floor — the toolchain pin and the MSRV are never conflated.
- **Supply-chain and feature-matrix hygiene (Defect 63).** `cargo-audit` now runs on every PR and push to `main`, checking the lockfile against RustSec with the same accepted advisory ignore recorded in `deny.toml`. `cargo-deny` remains the broader supply-chain policy gate — advisories in the scheduled hygiene workflow, licenses/bans/sources as deterministic PR blockers — with source policy covering the retired `vt100` fork removal. `cargo-hack --feature-powerset` compiles every feature combination in the scheduled hygiene job so no `#[cfg]` branch ships unverified. `cargo-vet`/`cargo-crev` are deliberately not adopted — they are shared-audit systems for organizations; under the solo-maintainer constraint they add ceremony without coverage beyond RustSec advisory checks. The decision and its retirement trigger are recorded.

## Why this exists

The crate restructure moved ~60K lines of Rust across fifteen crates and deleted the root `src/` binary. The mechanical moves preserved behavior, but the smoke pass over the rebuilt binary and the `jackin-tui-lookbook` browser surfaced a set of small, independent defects — TUI rendering inconsistencies, a build-path regression, and visual-polish gaps — that are each cheap to fix but collectively erode the "looks finished" bar. This item collects them so none is lost, each is fixed against the real current code (not the pre-restructure layout), and the docs/tests stay green per the repo rules.

Defects are grouped by surface in the checklist. Each entry names the exact file and the canonical pattern it should follow, so the fix is a convergence on an existing in-repo convention rather than a fresh invention.

## Surfaces in scope

- **`jackin-tui-lookbook`** — the contributor component browser (`cargo run -p jackin-tui-lookbook -- --terminal`). The lookbook is the reference implementation for every shared component; rendering inconsistencies here teach the wrong pattern. See <RepoFile path="crates/jackin-tui-lookbook/src/main.rs" />.
- **`jackin-tui` shared components** — when the lookbook reveals a missing affordance (scrollbar, backdrop, padding helper), the fix belongs in the library, not a lookbook-local workaround (per <RepoFile path="crates/jackin-tui-lookbook/AGENTS.md" />).
- **Build tooling** — `build-jackin-capsule` and any other path that hard-coded a layout assumption invalidated by the restructure.

## Fix the whole class, never one instance (hard rule)

Every defect below is a **class**, not a single broken widget. The components and surfaces share the same building blocks (the same `Panel`, the same scroll module, the same render helpers, the same padding idioms), so a defect that shows up on one component almost always exists on every sibling that uses the same block. Before fixing any entry, **scan the entire repository for every site that exhibits the same pattern and fix them all in one pass** — do not patch the one screenshot the operator happened to capture and leave the rest. A fix that lands on one story while the same flaw survives on five others is incomplete and reopens the defect.

Concretely, for each entry: grep the workspace for the construct (the helper call, the widget, the layout idiom), enumerate every call site, and converge all of them on the corrected shared pattern. When the right fix is a new or changed shared helper in `jackin-tui`, route **every** site through it so the correction cannot drift back apart.

## The checklist is a starting point, not the full truth (hard rule)

This checklist was written from operator screenshots and a first-pass scan; it is necessarily incomplete and occasionally inaccurate. **When you execute any entry, do a fresh analysis first — re-scan the repository for that defect class, confirm the listed sites are right, and find the sites the checklist missed — then fix everything you find, not only what is written down.** Treat each entry's site list and root cause as a hypothesis to verify, not a closed set. If your analysis turns up additional sites, additional root causes, or a better shared fix, expand the entry and fix the superset. A defect that the checklist under-described but you left half-fixed because "it wasn't on the list" is a miss, not a defense.
