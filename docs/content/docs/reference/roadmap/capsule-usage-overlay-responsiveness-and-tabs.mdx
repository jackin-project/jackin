## Guiding principles (binding for every item here)

1. **Correctness over ROI.** Fix what is wrong or inconsistent; never skip an item as "low value". The only reason to stop short is a *proven* impossibility.
2. **Root-cause, architectural fixes.** Every bug means the architecture permitted it. Remove the structural condition and fix the whole class, not the one symptom.
3. **Reuse, never copy-paste.** Recurring concerns (tab focus, hover, cursor, blocking I/O, per-provider headline) are fixed once as a shared mechanism applied everywhere.
4. **Never block.** Blocking work (network, subprocess, DB, file I/O) never runs on the event/input/render task â€” tokio / `spawn_blocking` / workers, independent loads joined.
5. **CodexBar is a reference to verify, not gospel.** Mirror [CodexBar](https://github.com/steipete/codexbar) by default, but verify every datum against its source; where jackin' has better data, diverge and document why.

## P0 â€” Claude and OpenAI full snapshots (the two regressions)

**The biggest lever â€” and a class of bug, not two isolated ones.** Live `usage verify` shows both providers resolve via **`source: "cli"` (`confidence: authoritative`)** with only two windows. The architecture *permits a silent degrade*: when the richer source (Claude OAuth, Codex RPC) is unavailable or not yet fetched, the snapshot quietly falls back to an impoverished `cli` snapshot still labelled `authoritative`, so the UI shows a confident-but-truncated result instead of an honest degraded state. **Root-cause / class fix:** the source layer must never present a reduced snapshot as `authoritative` â€” it must either produce the full snapshot from the available source or mark the result degraded with the reason. ### RC2 â€” `usage accounts` reports `resets_at: null` for every provider (reporter bug, fix alongside P0)

`usage accounts` JSON serializes `resets_at: null` for all rows even when the TUI renders resets correctly for six providers â€” so the reset data **is** in the in-memory snapshot but is dropped on the CLI report path. Consequences: (1) do **not** judge reset correctness from `usage accounts` â€” use the TUI render; (2) this is itself a bug â€” the architecture lets a render field be silently dropped on the CLI path, so the CLI and TUI disagree.

**Root cause (pinned).** `QuotaBucketView` carries only the formatted `reset_label` (e.g. `Resets in 3h`), not the raw reset epoch; the relative label cannot be reversed, so `account_snapshot_views_from_cache` (`usage.rs`) hardcodes `resets_at: None`. The window builders (`push_claude_window` / `push_codex_window` / `zai_bucket` / `kimi_bucket` / `minimax` / grok) compute the epoch and discard it after formatting.

**Shipped â€” `5bcc4ec1`.** Two parts, both landed: (a) `QuotaBucketView` gained `resets_at: Option<i64>` and a `timed_bucket` helper threads the epoch through every window builder (Claude/Codex/Grok/Z.AI/Kimi/MiniMax), so `account_snapshot_views_from_cache` emits it and the live `usage accounts` agrees with the TUI; (b) the telemetry store persists it both directions. **No schema migration was needed** â€” the earlier deferral assumed one, but the `account_usage_snapshots` table already had a `resets_at INTEGER` column (schema v4); the bucketâ†”row mapping simply dropped the value (`account_snapshot_rows` hardcoded `None`, the read never set it). Wiring `resets_at: bucket.resets_at` on write and `row.resets_at` on read closed it with no version bump. Locked by `account_snapshot_rows_carry_reset_epoch`.

## P3 â€” Usage-state lifecycle: never show a misleading state (F1â€“F3)

The usage segment must follow agent state, identically for **every** agent:

1. **Before the agent has started** â€” hide the usage segment entirely (today it shows `usage unavailable` during container init, before the agent exists).
2. **Agent started, usage not yet known** â€” show `refreshing` (one consistent word everywhere), not `usage unavailable`.
3. **Resolved** â€” the real headline.
4. **Genuinely unavailable** â€” only then `usage unavailable` (or the specific honest state: `needs login`, `stale`, `unsupported`, `daemon disconnected`).

### P3a â€” lifecycle states + clickable refresh

The segment is **clickable in the unknown/refreshing state**; a click forces a proactive load. These states are driven by the in-memory usage cache plus an agent-started flag, so P3a is implementable without the proxy below. Honest degradation (never a fabricated number):

### P3b â€” reliable agent-start signal (needs a design step before coding)

Detecting "agent started" reliably is the open architectural question behind steps 1â€“2. Before coding: identify how the agent is launched today (the process-launch path in the daemon/runtime) and confirm whether a proxy/hook is the right structural fix or whether an existing lifecycle signal can be used. **Design-step finding (PR 579 investigation).** The agent is *already* launched through jackin-capsule: attaching/opening a tab runs `/jackin/runtime/jackin-capsule new <agent>` (`crates/jackin-capsule/src/runtime/attach.rs`), whose `new` handler (`crates/jackin-capsule/src/main.rs`) sends a `SpawnRequest::Agent` to the daemon over the control socket; **the daemon owns the spawn** and creates the session/pane + agent process. So the exact start moment is already an in-daemon event â€” **no separate proxy process is needed**; the structural fix is to hook the daemon's existing `SpawnRequest::Agent` handling: mark that session's agent "started", kick a usage refresh, and let the usage view distinguish *before-start* (segment hidden) â†’ *started/unknown* (`refreshing`) â†’ *resolved*. This is the existing-lifecycle-signal option the design step was meant to choose between, and it is the lighter, correct one. **Pending operator ratification of this approach before P3a/P3b coding** (per the roadmap's "confirm before coding" gate).

## P5 â€” Shared tab-bar focus model + remaining visual (F14 / F15, Issue 1, Issue 15)

The tab bar reuses the shared `TabStrip` (`crates/jackin-tui/src/components/tab_strip.rs`), the same component as the console settings / workspace-editor surfaces. The model â€” already specified for that surface â€” must be applied identically on **every** tab bar in jackin' (console settings, workspace editor, capsule agent-tab bar, capsule usage overlay, status bar, lookbook):

- The **bright phosphor-green** accent is the single focus indicator and always marks where focus is â€” never in two places at once.
- **Tab bar focused:** the active tab shows the bright-green underline; `Left` / `Right` switch tabs; `Down` / `Tab`, or clicking the content, moves focus into the content.
- **Hover** a tab: the tab color changes AND the cursor changes to a pointer; clicking a tab moves focus onto the tab bar.
- **Content focused:** the tab underline drops to a neutral/dim color (not bright green) and the content block carries the green focus indicator instead.

**Capsule does not match today** â€” neither the top agent-tab bar nor the usage overlay strip shows the green-on-focus underline, arrow-nav while focused, the hover cursor, or the focus transition into content. The structural reason: the capsule agent-tab bar is a **hand-rolled painter** (`StatusBarWidget::paint_tab` in `crates/jackin-capsule/src/tui/components/chrome.rs`), separate from the shared `TabStrip` â€” that duplication is exactly why it diverges. Root-cause fix: fold the agent-tab bar (and the usage strip) onto the shared `TabStrip` so render + hover + per-tab hit-test + cursor + focus underline live in one place (Issue 1) and the focus model is unified across every surface (Issue 15) â€” not re-patch each painter.

**Still owed (capture + verify in a live smoke):** hover color + pointer cursor on every tab surface; status-bar consistency on every screen per the display matrix (debug + non-debug); no vertical jump on tab switch, centered strip, correct symmetric padding (content padded both sides, separators full-bleed); left/right tab switch instant even mid-refresh with no refetch within the TTL.

**Implementation plan (PR 579 investigation).** The shared `TabStrip` (`crates/jackin-tui/src/components/tab_strip.rs`) already carries the entire model: `.focused(bool)` drives `tab_underline_line` (bright phosphor-green underline only when focused, neutral otherwise), `.hovered(Some(idx))` drives `tab_cell_style` (hover colour), `hit_index_at(area, col, row)` is the shared click/hover hit-test, and `register_hover_targets(â€¦)` wires OSC-22 pointer cues + hover regions in one place. The capsule's divergence is purely that `StatusBarWidget::paint_tab` (`crates/jackin-capsule/src/tui/components/chrome.rs:37`) hand-paints cells instead of delegating. **Fold step:** replace `paint_tab`/the agent-tab loop with a `TabStrip` built from the `StatusTabCell`s, threading a tab-bar-focused flag and the hovered index, and route the existing `tab_regions` hover/click registration through `register_hover_targets`/`hit_index_at`; do the same for the usage-overlay strip. The hit-test and cell-mapping are unit-testable; the acceptance criteria above (underline colour on focus transition, hover colour, pointer cursor, centring, no vertical jump, symmetric padding) are **visual and must be confirmed in a live debug smoke** â€” they cannot be asserted headlessly, so this item is owed an interactive verification pass rather than landed blind.

## Reference â€” CodexBar extraction recipes (for P0 / P1)

The background refresher resolves a credential, calls the provider usage endpoint, and maps the response to quota buckets; render paths never call these. Only the recipes an open item still needs are kept here.

- **Anthropic / Claude.** Credential `~/.claude/.credentials.json` (`claudeAiOauth.accessToken`), forwarded from the host macOS Keychain service `Claude Code-credentials`. - **OpenAI / Codex.** Credential `$CODEX_HOME/auth.json` (default `~/.codex/auth.json`). Call `GET https://chatgpt.com/backend-api/wham/usage` with `Authorization: Bearer` + `ChatGPT-Account-Id`; or launch `codex -s read-only -a untrusted app-server` and call JSON-RPC `account/read` + `account/rateLimits/read`. Identity from the app-server RPC or `id_token` JWT, not a `tokens.email` field.
- **Claude credits (F8).** `claude.ai/api/organizations/{org}/usage` â€” spent, monthly cap, balance (currency-localized).

### Credential delivery into the container

The capsule runs as PID 1 inside the Linux container and cannot read the host Keychain; the runtime reads the host credential (file or Keychain) and forwards it into the container at a fixed path per provider (`crates/jackin-runtime/src/instance/auth.rs`). File-based providers (Claude, Codex, Grok, Amp) depend on that forwarding; env-key providers (Z.AI, Kimi, MiniMax) are carried in as environment variables regardless of mount mode. P0 depends on the Claude OAuth credential being forwarded reliably so the OAuth path can win over `cli`.
