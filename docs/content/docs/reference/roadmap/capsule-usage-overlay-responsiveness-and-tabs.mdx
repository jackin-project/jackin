## Guiding principles (binding for every item here)

1. **Correctness over ROI.** Fix what is wrong or inconsistent; never skip an item as "low value". The only reason to stop short is a *proven* impossibility.
2. **Root-cause, architectural fixes.** Every bug means the architecture permitted it. Remove the structural condition and fix the whole class, not the one symptom.
3. **Reuse, never copy-paste.** Recurring concerns (tab focus, hover, cursor, blocking I/O, per-provider headline) are fixed once as a shared mechanism applied everywhere.
4. **Never block.** Blocking work (network, subprocess, DB, file I/O) never runs on the event/input/render task ‚ÄĒ tokio / `spawn_blocking` / workers, independent loads joined.
5. **CodexBar is a reference to verify, not gospel.** Mirror [CodexBar](https://github.com/steipete/codexbar) by default, but verify every datum against its source; where jackin' has better data, diverge and document why.

## P0 ‚ÄĒ Claude and OpenAI full snapshots (the two regressions)

**The biggest lever ‚ÄĒ and a class of bug, not two isolated ones.** Live `usage verify` shows both providers resolve via **`source: "cli"` (`confidence: authoritative`)** with only two windows. The architecture *permits a silent degrade*: when the richer source (Claude OAuth, Codex RPC) is unavailable or not yet fetched, the snapshot quietly falls back to an impoverished `cli` snapshot still labelled `authoritative`, so the UI shows a confident-but-truncated result instead of an honest degraded state. **Root-cause / class fix:** the source layer must never present a reduced snapshot as `authoritative` ‚ÄĒ it must either produce the full snapshot from the available source or mark the result degraded with the reason. Fix that structural condition, not just the two tabs.

### P0a ‚ÄĒ Claude (regression; diagnose the credential path FIRST)

Symptom: Session/Weekly show `% left` only ‚ÄĒ no reset, no in-reserve, no `Lasts until reset`; Sonnet and Daily Routines absent. Claude resolves via `push_claude_cli_bucket` (`crates/jackin-capsule/src/usage.rs`), the impoverished path. The rich path is OAuth `/api/oauth/usage` ‚Üí `push_claude_window` (reset + in-reserve + Sonnet + Daily Routines).

The root cause is **not yet pinned ‚ÄĒ diagnose before fixing.** All seven providers passed in-container on June 24, so Claude OAuth **regressed**. First step, in-container on a debug launch:

- Check the forwarded Claude credential: `~/.claude/.credentials.json`, `$CLAUDE_CONFIG_DIR/.credentials.json`, `~/.claude.json`, and `ANTHROPIC_API_KEY` / `ANTHROPIC_AUTH_TOKEN` env.
- If the credential is absent or stale ‚Üí the OAuth fetch fails and the snapshot falls back to `cli`: the bug is in host‚Üícontainer credential forwarding (`crates/jackin-runtime/src/instance/auth.rs`). `git log` what changed Claude forwarding since the June-24 working state.
- If the credential is present but `cli` still wins ‚Üí the bug is in source selection: the `cli` path is preferred or OAuth is gated. Make OAuth win whenever its credential resolves.

Then restore the Claude sections (in order): **Session, Weekly, Sonnet, Daily Routines** (+ conditional Opus / Extra usage). Session/Weekly carry `Resets in <rel> (<local>)` + in-reserve + `Lasts until reset`; Sonnet carries a reset; Daily Routines shows remaining only.

### RC2 ‚ÄĒ `usage accounts` reports `resets_at: null` for every provider (reporter bug, fix alongside P0)

`usage accounts` JSON serializes `resets_at: null` for all rows even when the TUI renders resets correctly for six providers ‚ÄĒ so the reset data **is** in the in-memory snapshot but is dropped on the CLI report path (the reporter or its persistence). Consequences: (1) do **not** judge reset correctness from `usage accounts` ‚ÄĒ use the TUI render; (2) this is itself a bug ‚ÄĒ the architecture lets a render field be silently dropped on the CLI path, so the CLI and TUI disagree. Fix the reporter to serialize `resets_at` so both agree (same honest-output principle as P0's class fix).

## P1 ‚ÄĒ Unified account identity + auth-source header (F11 / F12)

Today the header mixes identity with auth method and is inconsistent per provider (`Claude OAuth`, `ZAI_API_KEY present`, `Kimi auth token`, `MiniMax API token`, real email for Codex/Amp/xAI). Separate the two concepts; **operator-approved layout, identical on every provider and everywhere in the dialog:**

- Row 1: provider name (left) ¬∑ account email (right; omit when the provider exposes none).
- Row 2: `Updated <rel> ¬∑ <state>` (left) ¬∑ `<username> ¬∑ <plan>` (right; username only when distinct from email).
- Row 3 (NEW): `Auth: <kind> ¬∑ <origin>` ‚ÄĒ the credential SOURCE, never the secret. kind = `OAuth` (native client) / `API token` (env) / `API key` (file); origin = the WINNING resolver arm (env ‚Üí `env <VAR_NAME>`; file ‚Üí the path; OAuth ‚Üí `keychain` / path).

**Per-provider data (verified against the `usage.rs` resolvers + parallax logs):**

| Provider | email | username | plan | Auth row |
|---|---|---|---|---|
| OpenAI / Codex | yes (`id_token` / RPC `account/read`) | yes | `Pro 20x` | `OAuth ¬∑ ~/.codex/auth.json` (or `/jackin` handoff) |
| Anthropic / Claude | needs new parse (`~/.claude.json` `oauthAccount`) | ‚ÄĒ | `Max` (`subscriptionType`) | `OAuth ¬∑ keychain` |
| Amp | yes (`email`) | yes | `Amp Free` | `API key ¬∑ amp secrets.json` |
| xAI / Grok | yes (`email`) | (user id) | `SuperGrok` | `OAuth ¬∑ ~/.grok/auth.json` |
| Z.AI | none | ‚ÄĒ | plan name | `API token ¬∑ env ZAI_API_KEY` |
| Kimi | none | ‚ÄĒ | ‚ÄĒ | `API token ¬∑ env KIMI_CODE_API_KEY` |
| MiniMax | none | ‚ÄĒ | plan title | `API token ¬∑ env MINIMAX_API_KEY` |

`FocusedAccountHeader` (`crates/jackin-protocol/src/control.rs`) has no field for credential origin ‚ÄĒ every snapshot resolver discards which `.or_else` arm won. Add a structured `credential_origin` (a small enum carrying the origin kind plus its detail: an environment variable with its name, a file with the path, the keychain, or an OAuth source), populate it at the resolver arm that actually wins (not a guessed default), split email/username/plan, and render the Auth row. For Claude, parse the account email from `~/.claude.json` `oauthAccount` (or a profile endpoint) ‚ÄĒ verify against CodexBar's Claude fetcher. Providers with no email show plan + auth row only; no fabricated email. Parallax logs confirm Z.AI/Kimi/MiniMax creds arrive as env vars (`ZAI_API_KEY` / `KIMI_CODE_API_KEY` / `MINIMAX_API_KEY`).

## P3 ‚ÄĒ Usage-state lifecycle: never show a misleading state (F1‚ÄďF3)

The usage segment must follow agent state, identically for **every** agent:

1. **Before the agent has started** ‚ÄĒ hide the usage segment entirely (today it shows `usage unavailable` during container init, before the agent exists).
2. **Agent started, usage not yet known** ‚ÄĒ show `refreshing` (one consistent word everywhere), not `usage unavailable`.
3. **Resolved** ‚ÄĒ the real headline.
4. **Genuinely unavailable** ‚ÄĒ only then `usage unavailable` (or the specific honest state: `needs login`, `stale`, `unsupported`, `daemon disconnected`).

### P3a ‚ÄĒ lifecycle states + clickable refresh

The segment is **clickable in the unknown/refreshing state**; a click forces a proactive load. These states are driven by the in-memory usage cache plus an agent-started flag, so P3a is implementable without the proxy below. Honest degradation (never a fabricated number):

| State | Rendering |
|---|---|
| Before agent start | usage segment hidden |
| Refreshing / unknown | `refreshing` (clickable ‚Üí force load) |
| Fresh | real quota headline |
| Stale | `stale` + `updated Xm ago`, keep prior data |
| Needs login / secret | the repair action name; no hidden repair |
| Unsupported | `unsupported` + reason |
| Daemon disconnected | `usage unavailable: daemon disconnected` |
| No focused agent | `usage unavailable: no focused agent session` |

### P3b ‚ÄĒ reliable agent-start signal (needs a design step before coding)

Detecting "agent started" reliably is the open architectural question behind steps 1‚Äď2. Intended direction: **route the agent launch through jackin-capsule as a proxy/hook** instead of launching the agent directly, so the capsule owns the exact start moment and triggers auth-verify + refresh then. Before coding: identify how the agent is launched today (the process-launch path in the daemon/runtime) and confirm whether a proxy/hook is the right structural fix or whether an existing lifecycle signal can be used. This is what makes "hide before start" and "`refreshing` on start" precise rather than approximated.

## P5 ‚ÄĒ Shared tab-bar focus model + remaining visual (F14 / F15, Issue 1, Issue 15)

The tab bar reuses the shared `TabStrip` (`crates/jackin-tui/src/components/tab_strip.rs`), the same component as the console settings / workspace-editor surfaces. The model ‚ÄĒ already specified for that surface ‚ÄĒ must be applied identically on **every** tab bar in jackin' (console settings, workspace editor, capsule agent-tab bar, capsule usage overlay, status bar, lookbook):

- The **bright phosphor-green** accent is the single focus indicator and always marks where focus is ‚ÄĒ never in two places at once.
- **Tab bar focused:** the active tab shows the bright-green underline; `Left` / `Right` switch tabs; `Down` / `Tab`, or clicking the content, moves focus into the content.
- **Hover** a tab: the tab color changes AND the cursor changes to a pointer; clicking a tab moves focus onto the tab bar.
- **Content focused:** the tab underline drops to a neutral/dim color (not bright green) and the content block carries the green focus indicator instead.

**Capsule does not match today** ‚ÄĒ neither the top agent-tab bar nor the usage overlay strip shows the green-on-focus underline, arrow-nav while focused, the hover cursor, or the focus transition into content. The structural reason: the capsule agent-tab bar is a **hand-rolled painter** (`StatusBarWidget::paint_tab` in `crates/jackin-capsule/src/tui/components/chrome.rs`), separate from the shared `TabStrip` ‚ÄĒ that duplication is exactly why it diverges. Root-cause fix: fold the agent-tab bar (and the usage strip) onto the shared `TabStrip` so render + hover + per-tab hit-test + cursor + focus underline live in one place (Issue 1) and the focus model is unified across every surface (Issue 15) ‚ÄĒ not re-patch each painter.

**Still owed (capture + verify in a live smoke):** hover color + pointer cursor on every tab surface; status-bar consistency on every screen per the display matrix (debug + non-debug); no vertical jump on tab switch, centered strip, correct symmetric padding (content padded both sides, separators full-bleed); left/right tab switch instant even mid-refresh with no refetch within the TTL.

## Reference ‚ÄĒ CodexBar extraction recipes (for P0 / P1)

The background refresher resolves a credential, calls the provider usage endpoint, and maps the response to quota buckets; render paths never call these. Only the recipes an open item still needs are kept here.

- **Anthropic / Claude.** Credential `~/.claude/.credentials.json` (`claudeAiOauth.accessToken`), forwarded from the host macOS Keychain service `Claude Code-credentials`. Call `GET https://api.anthropic.com/api/oauth/usage` with `Authorization: Bearer`, `anthropic-beta: oauth-2025-04-20`, `User-Agent: claude-code/<version>`. Windows: `five_hour`‚ÜíSession, `seven_day`‚ÜíWeekly, `seven_day_sonnet`‚ÜíSonnet, `seven_day_opus`‚ÜíOpus, `seven_day_routines` (and its `claude_routines` / `routines` aliases)‚ÜíDaily Routines, `extra_usage`‚ÜíExtra usage; each window carries `utilization` + `resets_at`. Plan prefers `subscriptionType`, falls back to `rateLimitTier`. Account email is NOT in this response ‚ÄĒ read `~/.claude.json` `oauthAccount` (F12).
- **OpenAI / Codex.** Credential `$CODEX_HOME/auth.json` (default `~/.codex/auth.json`). Call `GET https://chatgpt.com/backend-api/wham/usage` with `Authorization: Bearer` + `ChatGPT-Account-Id`; or launch `codex -s read-only -a untrusted app-server` and call JSON-RPC `account/read` + `account/rateLimits/read`. Map `rate_limit.primary_window`‚ÜíSession, `secondary_window`‚ÜíWeekly; `additional_rate_limits[]` `spark`‚ÜíCodex Spark 5-hour/Weekly; `credits`‚ÜíCredits; `plan_type`‚Üíplan (`Pro 20x`). Identity from the app-server RPC or `id_token` JWT, not a `tokens.email` field.
- **Claude credits (F8).** `claude.ai/api/organizations/{org}/usage` ‚ÄĒ spent, monthly cap, balance (currency-localized).

### Credential delivery into the container

The capsule runs as PID 1 inside the Linux container and cannot read the host Keychain; the runtime reads the host credential (file or Keychain) and forwards it into the container at a fixed path per provider (`crates/jackin-runtime/src/instance/auth.rs`). File-based providers (Claude, Codex, Grok, Amp) depend on that forwarding; env-key providers (Z.AI, Kimi, MiniMax) are carried in as environment variables regardless of mount mode. P0 depends on the Claude OAuth credential being forwarded reliably so the OAuth path can win over `cli`.
