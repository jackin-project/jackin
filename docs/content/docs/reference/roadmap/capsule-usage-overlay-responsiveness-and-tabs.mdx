## Guiding principles (binding for every item here)

1. **Correctness over ROI.** Judge each item by whether it *should* be done — is the current
   state wrong or inconsistent — never by cost/effort/"is it worth it." The only reason to stop
   short is a *proven* impossibility, not a hard or large change.
2. **Root-cause, architectural fixes.** A correct architecture has no bugs; every bug is
   evidence the architecture *permits* it. Before fixing, diagnose why the structure allowed it
   and whether it is one instance of a whole *class*. Prefer removing the structural condition
   over a symptom patch.
3. **Reuse, never copy-paste.** Recurring concerns (tab hover, cursor, blocking I/O) must be
   fixed once as a shared mechanism applied *everywhere*, so the same defect cannot reappear and
   the operator never has to re-explain the fix.
4. **Never block — tokio everywhere, loads in parallel (HARDCORE rule).** Everything must stay
   responsive. Blocking work (network, subprocess, DB, file I/O) must **never** run on the
   event/input/render task — use tokio / `spawn_blocking` / workers. Every data load is async; when
   independent, run them **in parallel** (join), never serially. Applies to **all of jackin'**, not
   just the usage overlay. Audit every load path and refactor any that blocks or runs serially.
5. **CodexBar is a reference to verify, not gospel.** Mirror CodexBar by default, but for every
   decision verify whether *our* data is more accurate or more up-to-date. CodexBar can have bugs or
   stale logic; if jackin' has the better/fresher data, display the correct thing and document why we
   diverge. Never copy a CodexBar behavior we cannot explain — first understand what a value means,
   how it is derived, and what it represents, then decide.

## Issue 1 — Tabs must show hover color + change the mouse cursor, everywhere (ARCHITECTURAL)

**Operator report.** Tabs should always respond to hover: hovering a tab must change its color,
and the mouse cursor must change too (pointer/hand). This must work the same on every surface
that has tabs. Do not fix it only for the usage overlay — fix it as an architectural issue so
the same behavior is applied to every tab surface, now and for anything built later, by reusing
one shared mechanism instead of copy-pasting. This is the first thing to plan.

**Verified root cause (it is a duplication / missing-wiring problem).**

- The shared widget <RepoFile path="crates/jackin-tui/src/components/tab_strip.rs" /> (`TabStrip`)
  is **render-only**. It accepts a precomputed `hovered: Option<usize>` and paints the matching
  background (`tab_strip.rs:63`, colors `crates/jackin-tui/src/theme.rs:47-50`), but it does
  **not** own hover state, does **not** expose per-tab click/hit regions (it calls
  `lay_out_tabs(labels, 0)` internally and discards the cells — `tab_strip.rs:51`), and does
  **not** decide or emit any cursor shape.
- Because hit-testing and cursor are not part of the shared widget, every surface re-does them:
  - Capsule status bar **hand-rolls its own tab painter** (`StatusBarWidget::paint_tab`,
    `crates/jackin-capsule/src/tui/components/chrome.rs:37`) with a **duplicated**
    `(active, hovered) → background` 4-state match (`chrome.rs:38-44` vs `tab_strip.rs:63`) and a
    duplicated underline (`chrome.rs:144` vs `tab_strip.rs:84`). Its click regions are computed
    separately (`status_bar.rs:168` `set_click_regions_from_plan`, `:155` `tab_at_col`).
  - Console editor/settings tabs use the shared `TabStrip` to paint but re-derive columns for
    hit-testing in `crates/jackin-console/src/tui/layout.rs:195` (`tab_cell_at_position`) and
    `:205` (`tab_hover_index_at_position`).
  - Hover state storage differs per surface: capsule `chrome_target: Option<HoverTarget>`
    (`crates/jackin-capsule/src/tui/app.rs:79`, enum `:112`) vs console per-screen `hovered_tab()`
    (`crates/jackin-console/src/tui/screens/settings/model.rs:195`).
  - Cursor (OSC 22) is decided/emitted two unrelated ways: capsule `PointerShape` enum
    (`app.rs:52`, `:86` `pointer_shape_for_state`, emit `tui/terminal.rs:73`,
    `daemon/mouse_input.rs:42`) vs console `console_pointer_hand` bool + direct stdout write of
    `POINTER_HAND`/`POINTER_DEFAULT` (`crates/jackin-tui/src/lib.rs:235-236`,
    `crates/jackin-console/src/tui/run.rs:407`, `crates/jackin/src/console/tui/run.rs:788-807`).
- **The usage overlay tabs are the odd one out: hover + hit-test + cursor are entirely absent.**
  - Rendered render-only via `TabStrip::new(..).focused(..).render(..)` —
    `.hovered()` is **never called** (`crates/jackin-capsule/src/tui/components/dialog_widgets.rs:431`),
    so the hover branch is dead for this surface.
  - The usage dialog returns no clickable area: `Self::Usage { .. } => false` in `clickable_at`
    (`crates/jackin-capsule/src/tui/components/dialog.rs:1594`), so tabs are never registered with
    any `HoverTracker` and `pointer_shape_for_state` yields the default arrow.
  - Tab switching is keyboard-only (`dialog.rs:645` `usage_provider_tab_target`,
    key handling `dialog.rs:966-1004`). No mouse path at all.

**What already exists and should be reused** (do not reinvent): `HoverTracker<K>`
(<RepoFile path="crates/jackin-tui/src/components/hover_tracker.rs" />), tab geometry
`lay_out_tabs` + `tab_at_column` (`crates/jackin-tui/src/geometry.rs:306`, `:330`), hover color
constants (`theme.rs:47-50`), OSC-22 escapes (`lib.rs:235-236`).

**Direction (architectural, to be planned first).** Make `TabStrip` the single tab component that
owns render **+** hover **+** per-tab hit regions **+** cursor declaration, so any surface that
renders tabs gets hover color and pointer cursor for free and cannot regress:
- Expose the laid-out cells / a `click_regions()` (hit-test) from `TabStrip` so callers stop
  re-deriving columns (`status_bar.rs:168`, `layout.rs:195-223`).
- Unify the cursor decision: one mechanism that lets a tab/clickable region declare "pointer"
  (reconcile capsule `PointerShape` with console `POINTER_HAND`).
- Fold the status bar's extras (state glyphs ○/●, brand pill, right menu button, overflow `›`,
  green/white focus underline) into `TabStrip` options so the status bar can drop its hand-rolled
  painter, OR factor the shared hover/cursor/hit-test out so even the hand-rolled painter consumes
  it. Either way: **one** definition of tab hover + cursor, consumed everywhere including the
  usage overlay.
- Then wire the usage overlay tabs through it: add click regions + hover + cursor + mouse switch.

**Scope:** cross-cutting — `jackin-tui` (shared), capsule status bar + usage overlay, console
editor/settings. This is the architectural item; the others below are usage-overlay-specific.

## Issue 2 — Usage dialog "jumps" vertically when switching tabs (USAGE DIALOG ONLY)

**Operator report.** Changing the size per tab is fine; the problem is the dialog *jumps* on
screen — middle, then up, then middle — every time a tab changes. Use the macOS approach: stick
the dialog to a fixed **top** position. The top line and the tab strip must **never move**. Only
the content inside adjusts: the **height** grows/shrinks downward to fit the selected tab's
content. (Width may adapt to content, but position stays anchored at the top.)

**Verified root cause.** The dialog is **vertically centered**, and the center is recomputed from
the per-tab height, so a changing height re-centers the box → it appears to jump.

- `box_rect` (`crates/jackin-capsule/src/tui/components/dialog.rs:1643`) computes the y origin at
  `dialog.rs:1712-1713`:
  ```rust
  let row = crate::tui::components::status_bar::STATUS_BAR_ROWS
      + (max_height.saturating_sub(height)) / 2;   // (available - height)/2  → recenters every height change
  ```
- The Usage height varies per tab: `natural_height` for `Self::Usage` comes from
  `usage_info_required_height` (`dialog.rs:1691-1693` →
  `crates/jackin-capsule/src/tui/components/dialog_widgets.rs:495-502`,
  `= usage_info_lines(state).len() + 5`), and `state` is rebuilt per selected tab (Overview is
  short, provider tabs are tall).
- The `row`/`col` centering at `dialog.rs:1712-1714` runs **unconditionally for every dialog
  variant** (`CommandPalette`, pickers, `ContainerInfo`, `GitHubContext`, `Usage`,
  `ConfirmAction`, …). `box_rect` currently branches per-variant only for `width`
  (`:1644-1660`) and `natural_height` (`:1665-1706`) — there is **no per-variant positioning
  hook**.

**Direction (scoped to Usage only) — DECIDED: fixed top offset.** Add a per-variant branch at the
`row` computation (`dialog.rs:1712-1713`) so `Self::Usage` anchors its top edge at a **constant top
row** (a fixed offset near the top); content grows straight down; the top edge + tab strip never
move. (Not centered-on-max-height — operator chose the fixed top offset.) All other variants keep
current centering. Because `box_rect` is the single geometry source for both render and hit-testing
(`dialog.rs:808, 862, 1330, 1562` + hover/pointer paths), the anchored `row` **must** be returned
from `box_rect` itself — do not compute a separate top offset only in the render helper, or render
and pointer math will disagree.

**Leak risk to avoid:** editing the shared `row =` line without a `Self::Usage` guard would shift
*every* dialog. Must be guarded.

## Issue 3 — Usage tab bar must be center-aligned (USAGE DIALOG ONLY)

**Operator report.** On this dialog specifically, the tab bar should be centered, not aligned to
the left.

**Verified root cause.** The strip is left-aligned to the dialog interior, and the shared
`TabStrip` always paints from col 0 of the rect it is given.

- Tab rect built at `crates/jackin-capsule/src/tui/components/dialog_widgets.rs:421-426` with
  `x: inner.x` and full `width: inner.width` → starts at the left interior column.
- `TabStrip` has no alignment option; `paragraph` lays out from `start_col = 0`
  (`tab_strip.rs:50-56`, `lay_out_tabs(labels, 0)`).

**Direction (scoped to Usage only).** Compute a centered x-offset in `render_usage_info` and pass
a centered rect to `TabStrip`:
`strip_w = usage_tab_strip_width(tabs)` (already exists, `dialog_widgets.rs:471-476`; it includes
a trailing `TAB_GAP` per cell, so subtract one trailing `TAB_GAP` for true visual centering),
`x_offset = (inner.width - strip_w) / 2` (saturating, clamp to 0), build `tab_area` at
`x: inner.x + x_offset`. Confined to `dialog_widgets.rs:421-426`.

**Do NOT** add centering to the shared `TabStrip`: the console/status strips rely on
`tab_at_column` assuming `start_col = 0` for click/hover hit-testing, so a shared centering mode
would break their mouse mapping. The usage strip is keyboard-only today, so shifting its x-origin
via the rect breaks nothing. *(Note: when Issue 1 wires mouse hit-testing into the usage strip,
its centered x-offset must be threaded into the new hit-test too — coordinate Issues 1 and 3.)*

## Issue 4 — Tab switching feels blocked / unresponsive while data refreshes (ROOT CAUSE: blocking work on the event loop)

**Operator report.** Switching tabs feels like a blocking operation — pressing left/right
sometimes does nothing, especially while data is refreshing/downloading; the whole thing feels
buggy and stuck. We must never block. Everything must be responsive — use tokio / tokio tasks
(spawn_blocking) as much as possible, always.

**Verified root cause (CONFIRMED).** The usage refresh runs **synchronously, inline inside the
daemon `tokio::select!` loop**, on the same single task that processes input and renders. While a
refresh tick runs, the loop cannot poll the input branch, so arrow-key bytes sit unread until it
returns — exactly the reported symptom.

- Daemon loop: `tokio::select!` at `crates/jackin-capsule/src/daemon.rs:655`. The usage ticker
  branch (`daemon.rs:1024-1029`) calls `mux.refresh_active_usage_account_snapshots(Instant::now())`
  at **`daemon.rs:1025` with no `.await`, no `spawn_blocking`, no worker** — a plain synchronous
  call. (Contrast: git-branch / PR-context lookups in the same loop ARE offloaded via
  `maybe_spawn_*`, `daemon.rs:1019, 1043` — the pattern exists but was not applied here.)
- The work it triggers is blocking and **sequential per target**:
  `refresh_active_usage_account_snapshots` (`multiplexer_utils.rs:273`) → `refresh_active_account_snapshots`
  (`usage.rs:218`) → `for target in targets` (`usage.rs:233`) → `focused_snapshot(force=true)`
  (`usage.rs:237`) → `build_snapshot` (`usage.rs:567`):
  - `reqwest::blocking` client, `PROVIDER_HTTP_TIMEOUT = 10s` (`usage.rs:25`, client
    `usage.rs:3114-3117`).
  - Blocking CLI subprocess poll loop with `thread::sleep(50ms)`, `PROVIDER_CLI_TIMEOUT = 10s`
    (`usage.rs:26`, `run_cli_with_timeout_full` `usage.rs:4364-4408`; amp `:4249`, claude CLI `:4288`).
  - Codex RPC subprocess: 8s init + 3s + 3s ≈ **14s** (`usage.rs:2265`, timeouts `:27,:28`).
  - Grok RPC subprocess: 8s init + 12s ≈ **20s** (`usage.rs:2592`, timeouts `:32,:33`).
  - Worst case per tick = **sum** of per-target ceilings (tens of seconds). Subprocess launches
    are rate-gated by a 30-min `ManagedCliLaunchGate` cooldown (`usage.rs:2062-2090`), so the
    14–20s subprocess cost is not paid every tick, but the HTTP path still parks the loop up to
    10s on a slow/hung network.
- Manual refresh (`r`/`R`) is even worse: `DialogAction::RefreshUsage` →
  `focused_usage_snapshot(true)` runs the **full blocking `build_snapshot` directly on the
  input-handling task** (`crates/jackin-capsule/src/daemon/input_dispatch.rs:146-155`).
- Synchronous DB bridge: `run_store` (`crates/jackin-capsule/src/telemetry_store.rs:59-74`) does
  `thread::spawn` + a fresh `new_current_thread` tokio runtime + reopen DB + re-run
  `initialize_schema` (`:76-91`) + `join()` **per call**. A tab switch does 1 such read
  (`input_dispatch.rs:158` → `usage.rs:161`, `force_refresh=false`); a mouse/resize event does 1
  (`input_dispatch.rs:682`); each forced refresh target does 2 (`usage.rs:202`, `:206`) plus the
  `materialize_accounts` file write (`usage.rs:198`).

**Already non-blocking (do not over-claim):** tab switch itself does *not* force a network fetch
(`force_refresh=false`); keyboard `Data` events do not call `focused_usage_status_label` (only
mouse/non-data do). The structural defect is that the periodic ticker and manual refresh execute
synchronously on the task that owns input + render.

**Direction (architectural).** No blocking work on the event/input/render task — ever:
- Offload the provider fetch (`refresh_active_account_snapshots` / `build_snapshot`) to a
  `tokio::task::spawn_blocking` or a dedicated worker, feeding results back via a channel and
  invalidating for redraw on completion — mirror `maybe_spawn_*` (`daemon.rs:1019, 1043`).
- Move the manual `r` refresh onto the same offloaded path.
- Run provider probes **concurrently** (join), not sequentially, so total latency is the slowest
  provider, not the sum.
- Take the synchronous telemetry-store bridge off the input/render path: read from the in-memory
  `UsageCache` for the status label / tab switch instead of a per-event DB reopen; if DB access is
  needed, hold one long-lived connection and run it off-task. (Pairs with the
  `focused_usage_view` double-read already removed in commit `402bbfa0`.)
- This is a *class* fix: the rule "blocking I/O never on the event loop" should be enforced for
  all daemon-loop work, not just usage.

**HARDCORE RULE (operator, always-apply).** Use tokio everywhere; every data load runs async and, when
there are independent loads, **in parallel** (join), never sequentially. **No operation ever blocks**
the event/input/render task. This is non-negotiable and applies to all of jackin', not just the usage
overlay.
- **Verification owed (audit across jackin'):** enumerate **every** data-load / I/O path (provider
  fetches, telemetry-store reads/writes, git/branch/PR context, image/build steps surfaced to the UI,
  any subprocess or network or file I/O reachable from a render/input task) and confirm each is off
  the event loop (spawn_blocking / worker + channel / async) and that independent loads are joined,
  not serial. Refactor any that block or run serially. The repo already disallows blocking calls on
  render/runtime threads via clippy (`Command::output`, `thread::sleep`, `File::open`,
  `OpenOptions::open` are `disallowed_methods` per `crates/AGENTS.md`) — treat any
  `#[expect(clippy::disallowed_methods)]` on a UI-reachable path as a finding to fix, not accept.
- Concretely for usage: the blocking `reqwest::blocking` client and the `thread::sleep` CLI poll loop
  (`usage.rs:3114, 4364-4408`) and the per-call telemetry-store thread+runtime bridge
  (`telemetry_store.rs:59-74`) are all on UI-reachable paths today and must move off-loop and run
  provider probes concurrently.

## Issue 5 — Stop over-refreshing while the overlay is open (rate-limit risk) (USAGE OVERLAY)

**Operator report.** It looks like we re-refresh every time a tab is opened/changed — too often.
If a provider's data was already loaded, keep it for a while instead of immediately refreshing.
While the dialog is open the operator is not using the agent, so the data cannot go stale; there
is no reason to keep refreshing. Aggressive refresh risks hitting provider rate limiters, which
has consequences we must avoid.

**Verified mechanism (to confirm precisely during implementation).** Switching a tab calls
`request_usage_refresh_for_provider` (`multiplexer_utils.rs:253`), which marks a target as
pending/due; the periodic `usage_account_ticker` (`daemon.rs:1024`) then performs the refresh.
There is a refresh schedule with `should_refresh` / cooldown logic
(`usage.rs:225` `in_flight`, `:234` `should_refresh`, launch cooldown `:2062-2090`), but the
effective cadence still re-fetches on tab interaction / open more often than the operator wants.

**Direction — DECIDED: no re-fetch while the overlay is open.** Once a provider's snapshot has been
loaded, do **not** re-fetch that provider again while the overlay stays open — the operator is idle
(not using the agent) so the data cannot go stale, and re-fetching risks provider rate limits. Serve
the cached value on tab open/switch and from the ticker for already-loaded providers. Manual `r`
stays the explicit override (force refresh). (Implement as a per-provider freshness gate; "while open"
effectively means an open-session cache — confirm whether a long TTL also applies when the overlay is
closed, but the binding rule is: no auto-refetch of a loaded provider while open.)

**Relationship:** pairs with Issue 4 — once refresh is offloaded and TTL-gated, the overlay is
both responsive and quiet. Verify the exact current trigger points (open vs switch vs ticker)
before choosing where the TTL gate lives.

## Issue 6 — Wrong horizontal padding: content lacks right padding; separator has a (wrong) left padding (USAGE OVERLAY, all tabs)

**Operator report.** Look at the overlay (seen on every provider tab, not just one). The padding
is wrong:
- **Content blocks** (header rows, the two-column lines, the meter bars) have padding on the
  **left** but **none on the right** — wrong. Content should always have padding on **both** the
  left and the right.
- **The separator line** (the horizontal `─` rule): its **right** side is correct (it reaches the
  edge, no padding), but its **left** side has a padding for some unknown reason — wrong. The
  separator should **not** use padding at all (full width, both sides).

So: content = padded both sides; separator = full-bleed, no padding either side. Today it is
inverted (content padded only left; separator padded left, full-bleed right).

**Verified root cause.** There is **no single padding policy** — every line builder in
<RepoFile path="crates/jackin-capsule/src/tui/components/dialog_widgets.rs" /> hand-codes a
2-space left indent (`Span::raw("  ")`) and sizes its content as `width - 2`, which accounts only
for the left indent and lets the content run to the right edge. The separator wrongly reuses the
same left indent.

- **Separator** (`usage_separator_line`, `dialog_widgets.rs:874-877`):
  ```rust
  let target = width.saturating_sub(2).max(1);
  Line::from(vec![Span::raw("  "), Span::styled("─".repeat(target), DIM)])
  ```
  → 2-space **left** pad (wrong) + `─` to the right edge (right is full-bleed). Operator wants
  full-bleed both sides: `"─".repeat(width)`, no leading `"  "`. (Wrapper `push_usage_separator`,
  `:868-872`.)
- **Two-column content rows** (`usage_header_two_column`, `:724-737`): `INDENT = 2` left
  (`Span::raw("  ")`), then `gap = width - (INDENT + left_cols + right_cols)`, so the right value
  ends at column `width` — the right edge, **no right padding**. Used by headers, detail rows,
  Limit Reset Credits (`:770, :793, :826, :834`).
- **Meter bars** (`usage_quota_bucket_lines` `:780-784` + `usage_full_width_meter` `:895-913`):
  `target = width - 2`, line = `"  "` + bar of `width - 2` → bar fills to the right edge, **no
  right padding**.
- **Section labels** and other rows all start with `Span::raw("  ")` (`:597, :620, :647-654,
  :731, :757, :782, :979`) — same left-only indent.
- This is why it is wrong on **all** provider tabs: every tab renders through these same shared
  builders. (`width` here is `body.width` = the dialog interior, passed from `render_usage_info`
  via `usage_info_lines_for_width(state, body.width)`, `dialog_widgets.rs:441`.)

**Direction (root-cause, one policy).** Define a single horizontal-padding policy for the usage
body instead of per-builder magic numbers:
- **Content** gets symmetric padding (e.g. 2 left **and** 2 right): content builders size to
  `width - (PAD_LEFT + PAD_RIGHT)`, two-column right values stop `PAD_RIGHT` columns before the
  edge, meter bars become `width - PAD_LEFT - PAD_RIGHT`.
- **Separators** are full-bleed: `"─".repeat(width)`, no indent.
- Prefer computing a single padded content sub-rect / a shared `content_width` + indent helper
  that all builders consume, so the padding is defined **once** and a new section cannot
  reintroduce the asymmetry. Update the snapshot tests
  (`crates/jackin-capsule/src/tui/components/dialog_widgets/tests.rs` and width snapshots) to lock
  the new padding.

**Scope:** the usage overlay body renderer (all tabs). Verify whether the same builders or the
generic `render_scrollable_dialog_body` are reused by other dialogs (ContainerInfo / GitHubContext)
before changing, so the symmetric-padding policy lands where intended and the separator full-bleed
change does not regress a surface that wants the current look.

## Issue 7 — Amp status-bar signal is wrong: must show Free % + credits (per-provider headline)

**Operator report.** For Amp, the status-bar signal shows the wrong thing. When there is no
subscription data, the status bar must show how much **Amp Free** is left **and** how many
**credits** are available — e.g. `Free 9% · $4.76` (the 9% free remaining and the $4.76 credit
balance). Today the bottom-right shows a generic `usage cached` for Amp instead.

**Verified root cause.** The status-bar headline is hardcoded to Claude/Codex vocabulary
(Session / Weekly) and has no per-provider notion of what each provider's headline metric is.

- `status_bar_label` (`crates/jackin-capsule/src/usage.rs:1421-1440`) calls
  `status_bar_quota_labels`, and if that is empty falls through to a status word —
  `UsageSnapshotStatus::Fresh => "usage cached"` (`usage.rs:1432`).
- `status_bar_quota_labels` (`usage.rs:1442-1456`) only looks for buckets named **"Session"** or
  **"Weekly"** (`status_bar_bucket_matches`, `:1458-1473` matches `session`/`5-hour`/`weekly`).
- Amp's snapshot (`amp_snapshot`, `usage.rs:902`) produces buckets **"Amp Free"** (with the 9%
  `remaining_percent`) and **"Credits"** (`Individual credits: $4.76`) — neither matches
  Session/Weekly, so `status_bar_quota_labels` returns empty and Amp falls through to
  `"usage cached"`.
- Note `status_bar_label`'s first parameter is `_surface: UsageSurface` — **already passed but
  unused**: the per-provider hook was anticipated but never implemented. This is the same
  per-provider special-casing class flagged in the earlier altitude review (provider vocabulary
  hardcoded in multiple disjoint sites).

**Direction (root-cause, per-provider headline).** Make the status-bar headline provider-aware:
derive it from each provider's own primary metric(s) rather than a fixed Session/Weekly lookup —
use the `surface`/provider the function already receives. For Amp specifically: compose
`Free {n}% · ${credits}` from the "Amp Free" bucket's `remaining_percent` and the "Credits"
balance. Prefer a single place that maps a provider → its status-bar headline (a table/trait keyed
by `UsageSurface`), so each provider's headline is defined once and no provider can fall through to
a meaningless `"usage cached"`. This pairs with the broader provider-generalization noted in the
earlier altitude review.

**Scope:** `usage.rs` status-bar label derivation (Amp now; structured so every provider gets a
correct headline). Verify the other providers' current headlines stay correct (Claude/Codex
Session·Weekly) when the lookup becomes provider-driven. Update the status-bar label tests
(`usage.rs:4708`, `:4742`, `:4923`).

## CodexBar reference (binding visual + data spec)

The operator's reference is the CodexBar macOS app. Each provider tab in our overlay must show the
same sections, in the same order, with the same per-bucket detail lines (remaining %, in-reserve,
reset countdown, "Lasts until reset") as CodexBar — quota only (we drop CodexBar's cost/spend grid,
sparkline, Top model, and account-action rows per the binding design in
<RepoFile path="docs/content/docs/reference/roadmap/capsule-usage-quota-overlay.mdx" />).

**MANDATORY RESEARCH BEFORE IMPLEMENTATION.** Always verify against the CodexBar application **and
its source code** to confirm exactly how each datum is obtained (which API/CLI/file, which field,
how reset countdowns, in-reserve, and Limit Reset Credits are computed). Do not guess. Cite the

### In-reserve (pace) — per-provider spec (VERIFIED, all 7 providers)

"In reserve" = `quota_remaining% − time_remaining%` (a pace buffer; see the explanation under
Issue 14). Researched against the CodexBar screenshots + source (`UsagePace`, `UsagePaceText`,
`MenuCardView`). CodexBar attaches the pace line (in-reserve / deficit / "Lasts until reset") **only**
to the primary subscription **burn** windows, and only once enough of the window has elapsed to be
meaningful:

| Provider | In-reserve shown on | Not on |
|---|---|---|
| OpenAI / Codex | **Session, Weekly** | Codex Spark 5-hour, Codex Spark Weekly, Limit Reset Credits, Credits |
| Anthropic / Claude | **Session, Weekly** | Sonnet, Daily Routines |
| xAI / Grok | — none | (single Weekly/credit window: % + reset only) |
| Z.AI | — none | Tokens, MCP, 5-hour (CodexBar uses detail text, not pace) |
| Kimi | **Rate Limit** (the 5-hour window) | Weekly (request-quota, count-based) |
| MiniMax | — none | General·5h, General·Weekly, Video (count-based `Usage: N/100`) |
| Amp | — none | Amp Free (time-to-full), Credits |

- **Gate:** CodexBar only shows pace once `expectedUsedPercent >= 3` (`UsagePaceText.swift:91`) — i.e.
  it suppresses pace on a freshly-reset / near-full window so you don't see a misleading "100% in
  reserve". jackin must adopt the same elapsed-threshold guard.
- **Why this set (verified, not mimicry):** pace is `remaining% − time%`, meaningful only on a
  time-windowed burn against a quota. It is omitted on count buckets (MCP, MiniMax `N/100`, Credits),
  on bonus allowances (Codex Spark), and on model/static sub-windows (Sonnet, Daily Routines) where a
  burn-vs-time buffer doesn't represent anything. CodexBar's set is internally consistent.
- **jackin bug = OVER-application:** jackin currently shows in-reserve where it shouldn't — e.g. Z.AI
  5-hour/Tokens (`48% / 62% in reserve`, Issue 11) and Kimi **Weekly** (Issue 14). The fix restricts
  pace to the table above via the per-provider section spec.
- **Principle 5 check (is CodexBar right / could we do better?):** in-reserve needs no extra data
  (just used%, window length, reset) so this is a *display-choice*, not a data-accuracy, question.
  CodexBar's set is the correct, coherent choice; **mirror it**. Optional future enhancement (NOT
  parity, flag before doing): jackin *could* also show pace on other genuine burn windows (Z.AI
  5-hour, Grok weekly, MiniMax 5h) since it is mathematically meaningful — but only if the window
  length is reliably known (ties to the Kimi window-duration parse bug). Default: match CodexBar.
- **Accuracy caveat:** the reserve % is only correct if the **window length** is correct — a wrong
  window makes the % wrong (this is the Kimi `300 TIME_UNIT_MINUTE` parse bug, Issue 14). Verify
  window length per provider when enabling pace.

## Issue 10 — Status-bar click opens the wrong provider tab (must open the focused pane's provider) (USAGE OVERLAY)

**Operator report.** Focused on the Amp pane, clicking the status bar opens the usage overlay but
it is selected on **OpenAI**, not Amp. The overlay must always open on the provider of the pane the
operator is focused on / clicked from — Amp in this case.

**Verified root cause.** When the focused pane's provider cannot be resolved, the overlay falls back
to "match everything, pick the most recently refreshed provider" instead of the focused one.

- Status-bar click → `branch_context_bar_click_action(hit)` →`Action::OpenUsage`
  (`crates/jackin-capsule/src/daemon/input_dispatch.rs:548-565, :223-227`).
- `Action::OpenUsage` builds the view from `focused_usage_snapshot(false)` and opens
  `Dialog::new_usage(view)` (`input_dispatch.rs:224-225`), which selects
  `UsageDialogTab::Provider` showing `view.focused_provider` (`dialog.rs:772-773`).
- `focused_usage_snapshot` → `focused_usage_snapshot_for_provider(None, false)` →
  `focused_agent_provider()` (`multiplexer_utils.rs`) → `focused_snapshot(agent, provider, …)` →
  (force=false) `telemetry_store::focused_usage_view(path, Some(agent), focused_provider, …)`
  (`usage.rs:159-183`).
- The Amp pane has agent `amp` with **no alt-provider label**, so `focused_provider` is `None`. In
  `select_provider_rows` (`crates/jackin-capsule/src/telemetry_store.rs:515-528`):
  ```rust
  let focused = focused_provider.unwrap_or_default();          // "" when None
  let mut matches = rows.into_iter()
      .filter(|row| provider_matches(focused, &row.provider))  // "" is contained in EVERY provider → matches all
      .collect::<Vec<_>>();
  let latest = matches.iter().map(|row| row.fetched_at).max()?;
  matches.retain(|row| row.fetched_at == latest);              // picks whichever provider refreshed most recently
  ```
  Empty focused matches **all** providers, then it keeps the latest-`fetched_at` one → OpenAI/Codex
  here. The focused **agent** (`amp`) is passed but never used to pick the provider.
- Structural cause: the focused-pane → provider mapping is unreliable (no agent→provider resolution
  when the pane carries no explicit provider label), and the empty-provider fallback silently
  resolves to the wrong provider instead of the focused pane's. Same agent↔provider taxonomy gap as
  Issue 7.

**Direction (root-cause).** Resolve the focused pane's provider deterministically before opening
the overlay: map the focused agent → its provider (shared resolver, see Issue 7), and pass that as
the explicit `focused_provider` so the overlay opens on the correct tab. Fix the empty-provider
fallback in `select_provider_rows` so an unresolved provider does not silently match-all-and-pick-
latest (it should use the focused agent's provider, or show that pane's provider explicitly). Cover
with a test: clicking the status bar on each provider's focused pane opens that provider's tab.

  - CodexBar `UsagePace.weekly` (`Sources/CodexBarCore/UsagePace.swift:68-74`):
    `expected = (elapsed/duration)*100` (= 100 − time-left%), `actual = usedPercent`,
    `delta = actual − expected`; "in reserve" = `expected − actual` (when behind pace).
  - jackin `quota_pace_label` (`crates/jackin-capsule/src/usage.rs:3993`):
    `delta = remaining_percent − time_left_percent`; "in reserve" when `delta > 0`.

    remaining% − time_left%`. Worked example (Kimi Rate Limit, 100% left, window 5h, reset in 2h20m):
    time-left = 8400/18000 = 46.7% → reserve ≈ **53%** (CodexBar shows 54%, rounding). Weekly
    (100% left, ~6d14h left of 7d): time-left ≈ 94% → reserve ≈ **6%**. So jackin's 6% is the correct
    *Weekly* reserve and CodexBar's 54% is the correct *Rate Limit* reserve.

    (`UsagePaceText.swift:91`); other providers get pace via `weeklyDetail`/`UsagePace.weekly`. The
    per-provider section spec should decide which buckets show pace.

**What "in reserve" means (explained — operator asked).** It is a **pace** signal, not a raw quota
number. For a time-windowed burn bucket: `reserve% = quota_remaining% − time_remaining%` (where
time_remaining% = seconds left in the window ÷ window length). Example: Kimi Rate Limit has 100%
quota left but only 47% of the 5h window's time left → **53% in reserve** = you have 53 points more
quota than a steady linear burn would have left by now (a buffer ahead of pace). Negative →
"in deficit" (burning faster than the clock). "Lasts until reset" → at the current burn you will not
run out before the window resets. It is only meaningful on time-windowed burn buckets
(Session/Weekly/Rate-Limit), not on count buckets (MCP) or already-full/static ones.

where CodexBar's is misleading); (c) that the window length we use is right (ties to the Kimi
window-duration parse bug above — a wrong window makes the reserve % wrong). Record the per-provider
verified decision back into this file before implementing. Do not blindly copy CodexBar's gating.
- **Reset format**: jackin absolute (`Resets 20:17`, `Resets Jul 1 at 08:17`); → Issue 13.

## Issue 18 — Meter bar empty cells must use `░` (light shade), not `·` (USAGE OVERLAY, all bars)

**Operator report.** The quota bars must render the empty/unfilled portion with `░` (light shade
block), matching CodexBar:

```text
██████████░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░   (want)
████████······················································   (current)
```

**Verified root cause.** The empty cell glyph is hardcoded as `·` (middle dot) in the single bar
builder, and several predicates detect a meter by `matches!(ch, '█' | '·')`, so the glyph must change
in lockstep:
- Build: `usage_full_width_meter` (`crates/jackin-capsule/src/tui/components/dialog_widgets.rs:911-912`)
  = `"█".repeat(filled)` + **`"·".repeat(rest)`** → change the empty run to `"░"`.
- Meter-detection predicates (treat a value as a bar): `dialog_widgets.rs:900, :1054, :1087, :1227`
  all use `matches!(ch, '█' | '·')` → must accept `'░'` (switch to `'█' | '░'`).
- Tests with literal meter strings: `dialog_widgets.rs:1172, :1176` use `·` → update to `░`.
- **Verify the meter source:** if any upstream (snapshot/`usage.rs`) emits the `█`/`·` string that
  these predicates parse, change it there too so the filled+empty glyphs are defined in one place.

**Direction (root-cause, one definition).** Define the two meter glyphs (`█` filled, `░` empty) as
single shared constants used by both the builder and every predicate/test, so the empty glyph can
never drift again. Keep full-width remaining bars (binding design). Update snapshot/width tests.
