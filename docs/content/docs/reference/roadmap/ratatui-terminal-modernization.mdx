---
title: "Ratatui terminal modernization"
---

**Status**: Open — research complete, re-verified against the tree, and extended with a skeptical review pass. The implementation branch is allowed to refactor broadly before merge. The work is no longer "upgrade Ratatui"; the tree is already on Ratatui 0.30.2. The work is to (a) exploit Ratatui 0.30 buffer/diff/width APIs that are available through the dependency but unused by jackin', (b) close concrete first-party fidelity gaps in `jackin-term` and the Capsule emit path that no upstream crate will ever fix for us, and (c) verify competing implementation approaches where the current roadmap and the skeptical pass disagree.

## Problem

jackin' depends on Ratatui for every rich terminal surface and on `jackin-term` for the Capsule PTY terminal model. The original framing of this item assumed `jackin-term` predated the Ratatui 0.26–0.30.2 work and that adopting it was a future dependency bump. That framing is now stale and has been corrected by reading the tree: the dependency bump already happened. The remaining work is narrower, deeper, and entirely first-party at the seam between jackin's owned model and Ratatui's compositor.

The goal is not compatibility preservation. jackin' is pre-1.0, and quality is more important than avoiding hard refactors. If Ratatui now owns a capability better than jackin's local implementation, prefer upstream Ratatui and reshape jackin' around it. If jackin's model is still stronger because it owns PTY state, passthrough policy, scrollback, or source-recorded damage, keep that local ownership and borrow only the proven idea. Where this document records competing views, the implementation branch should treat them as research hypotheses: build the smallest proof that can distinguish them, compare correctness and emitted bytes, then commit to the better design. The implementation should happen on a dedicated branch and merge only after the existing automated harnesses (extended with new fixtures) plus manual operator soak prove the new shape is strictly better. No runtime feature flag is required for the intended direction.

## Baseline reality — what is already true in the tree (verified)

This section exists to stop the roadmap from re-crediting work jackin' already gets for free. Every claim here was verified by reading the tree and the pinned upstream source in the cargo registry.

- **The tree is already on Ratatui 0.30.2.** `ratatui = "0.30"` is declared in <RepoFile path="crates/jackin-capsule/Cargo.toml" />, <RepoFile path="crates/jackin-tui/Cargo.toml" />, <RepoFile path="crates/jackin-console/Cargo.toml" />, <RepoFile path="crates/jackin-launch/Cargo.toml" />, <RepoFile path="crates/jackin-runtime/Cargo.toml" />, and <RepoFile path="crates/jackin/Cargo.toml" />. The lockfile resolves `ratatui 0.30.2`, `ratatui-core 0.1.2`, `ratatui-widgets 0.3.2`, `crossterm 0.29.0`. `termina 0.3.3` and `ratatui-termina 0.1.0` appear in the lockfile only as the Ratatui meta-crate's *optional* backend dependencies; the default feature set is `crossterm`, so Termina is not compiled into any jackin' binary today.
- **The custom backend is already on the modern Backend contract.** `SocketBackend` in <RepoFile path="crates/jackin-capsule/src/tui/socket_backend.rs" /> already declares `type Error = io::Error`, already implements `clear_region` for every `ClearType` variant, and its `draw` already consumes Ratatui's changed-cell iterator and builds no intermediate `Vec`. There is no per-frame changed-cell allocation to remove.
- **Ratatui's flush is already zero-allocation.** `ratatui_core` 0.1.2 `Terminal::flush` walks `previous_buffer.diff_iter(current_buffer)` — a zero-allocation `BufferDiff` iterator that already handles multi-width characters and VS16 trailing cells — and passes it straight to `Backend::draw`. jackin' receives that iterator unchanged.
- **The echo-back render conformance harness already exists.** Every frame the Capsule composes is replayed into a second `DamageGrid` standing in for the operator's outer terminal, asserting cell-exact equality (grapheme, attrs, wide flags) plus the frame-model cursor contract. The harness and its `VirtualClient` live in <RepoFile path="crates/jackin-capsule/src/daemon/tests.rs" />. Parser-side conformance lives in <RepoFile path="crates/jackin-term/tests/conformance.rs" />, allocation regression in <RepoFile path="crates/jackin-term/tests/allocation.rs" />, the fuzz target in <RepoFile path="crates/jackin-term/fuzz/src/damage_grid_process.rs" />, and benches in <RepoFile path="crates/jackin-term/benches/present_frame.rs" />, <RepoFile path="crates/jackin-term/benches/scroll_throughput.rs" />, and <RepoFile path="crates/jackin-capsule/benches/pane_body.rs" />. The modernization branch *extends* these; it does not build them.
- **Cursor position, visibility, and shape are already frame-level state.** The grid tracks the program-requested DECSCUSR cursor style (`CSI {n} SP q`) in <RepoFile path="crates/jackin-term/src/grid.rs" /> (set in <RepoFile path="crates/jackin-term/src/grid/perform.rs" />), and the compositor in <RepoFile path="crates/jackin-capsule/src/daemon/compositor.rs" /> reconciles cursor style per frame, emitting the escape only when it changes from the last asserted value. Cursor work in this item is a completeness audit, not a from-scratch build.
- **The new Ratatui 0.30 buffer APIs exist upstream but are unused by jackin'.** `ratatui_core::buffer::CellWidth`, `ratatui_core::buffer::CellDiffOption` (`None`, `Skip`, `AlwaysUpdate`, `ForcedWidth(NonZeroU16)`), the public `BufferDiff` iterator, and `Terminal::apply_buffer()` / `apply_buffer_with_cursor()` are all present in `ratatui-core` 0.1.2. A tree-wide search finds zero uses of `set_skip`, `set_diff_option`, `CellDiffOption`, `ForcedWidth`, `AlwaysUpdate`, or `apply_buffer` in jackin'. This unexploited surface is the real opportunity, but it is not enough by itself while the current compositor deliberately forces every cell to repaint.
- **The Capsule currently forces full-frame repaint every frame.** `compose_ratatui_frame()` in <RepoFile path="crates/jackin-capsule/src/daemon/compositor.rs" /> fills Ratatui's current buffer with a private-use sentinel (`U+E001`) and swaps buffers before every draw, so Ratatui sees every composed cell as changed. This is a deliberate convergence stopgap: the comments explain that until extra writers are deleted, Ratatui's previous buffer cannot be trusted as the client model. The consequence is important: Ratatui's `diff_iter` is zero-allocation, but jackin' does not currently get minimal changed-cell output. Any recommendation about `CellDiffOption`, `AlwaysUpdate`, hyperlink invalidation, or emitted-byte wins must be verified again after this sentinel path is removed.
- **There are two hyperlink paths today, not one.** UI/dialog hyperlinks flow through `SocketBackend::set_hyperlink_regions()` and are emitted only for diffed cells. Agent PTY OSC 8 hyperlinks are parsed in `jackin-term` as `PassthroughEvent::Hyperlink`, safety-filtered by the Capsule session, and queued as out-of-band bytes through `ClientWriter`. The roadmap therefore has two valid implementation hypotheses: keep UI hyperlinks as backend frame regions and make their diff invalidation precise, or unify PTY and UI hyperlinks as cell/frame metadata in the terminal model and render pipeline. The implementation branch should compare these instead of assuming the current split is final.

## Current jackin' model — keep it, modernize the seam

The architecture is sound and is not being discarded. The full engineering record is in <RepoFile path="crates/jackin-term/README.md" /> and the durable reference is [Capsule Terminal Model](/reference/capsule/terminal-model/).

```text
PTY bytes
  -> vte::Parser
  -> jackin_term::DamageGrid            (parser state machine + cell grid + scrollback + damage)
  -> GridView / GridSnapshot / SnapCell (observation APIs)
  -> PaneBodyWidget                     (blits model cells into a Ratatui Buffer)
  -> ratatui::Terminal (Fullscreen)     (owns double-buffer + zero-alloc diff_iter)
  -> SocketBackend                      (encodes the changed-cell iterator to ANSI)
  -> ClientWriter                       (one socket writer; ?2026 synchronized frame brackets)
```

`jackin-term` owns the PTY terminal model: parser state, primary and alternate screen, scrollback, cursor state, modes, source-recorded dirty rows, default-color replies, Kitty keyboard state, OSC/CSI passthrough policy, and snapshots. The load-bearing files are <RepoFile path="crates/jackin-term/src/lib.rs" />, <RepoFile path="crates/jackin-term/src/grid.rs" />, <RepoFile path="crates/jackin-term/src/grid/perform.rs" />, <RepoFile path="crates/jackin-term/src/damage.rs" />, <RepoFile path="crates/jackin-term/src/cell.rs" />, <RepoFile path="crates/jackin-term/src/snapshot.rs" />, and <RepoFile path="crates/jackin-term/src/passthrough.rs" />.

Ratatui does not replace that model. Ratatui's `Buffer` is a compositor buffer, not a PTY emulator: it does not own scrollback retention, application cursor mode, bracketed paste mode, Kitty keyboard stack depth, OSC 10/11 replies, DA/DSR replies, focus-event passthrough, or the distinction between bytes that update the pane grid and bytes returned to the agent PTY. Termina does not replace it either: Termina is a low-level terminal I/O and typed-escape library with event parsing, not a screen grid, damage tracker, scrollback store, or frame diff. The modernization target is therefore the *seam*: make the model's display-width contract explicit, feed Ratatui enough cell-width and protocol-visibility information to diff correctly, stop dropping terminal-visible SGR state at parse time, and evaluate Termina only at the raw terminal I/O boundary.

## Decision principles

- **Prefer upstream Ratatui when it owns the exact problem.** If Ratatui 0.30's diff iterator, `CellDiffOption`, or `CellWidth` solves a compositor problem better than local code, switch even if the refactor is large.
- **Keep first-party ownership where jackin's problem is different.** `DamageGrid` exists because the Capsule needs a PTY terminal model with scrollback, passthrough policy, query replies, and source-recorded damage. Ratatui and Termina do not provide that complete model.
- **Record competing designs instead of prematurely collapsing them.** When two directions are plausible — e.g. Ratatui-only cell options vs a parallel metadata grid, backend hyperlink regions vs model-owned OSC 8 cells, inferred scrolls vs source-recorded scroll ops — the branch should preserve both as explicit hypotheses until fixtures, echo-back, emitted bytes, and terminal soak choose one.
- **Test branch, not feature flag.** The branch may break internal APIs, reshape the Capsule render path, or replace dependencies. It merges only after automated and manual validation prove the new shape is strictly better.
- **One physical writer remains mandatory.** Direct `GridPatch` ANSI emission was retired because it could desync Ratatui's previous buffer from the outer terminal. Modernization must not reintroduce parallel writers. Damage is an invalidation and observation signal; committed output remains one composed frame written by `ClientWriter` in <RepoFile path="crates/jackin-capsule/src/client_writer.rs" />.
- **Measure before claiming performance.** Record only real wins: allocation profiles, frame timings, emitted bytes, and echo-back conformance.

## What is explicitly NOT the work (padding removed)

These were listed as future work in the first draft of this item. They are already satisfied by the version jackin' is on. Do not spend the branch's budget here:

- **Do not "implement the associated `Error` type and `clear_region`."** `SocketBackend` already does both. The only backend audit worth doing is the *semantic* one in Recommendation 3, not the trait-shape one.
- **Do not "remove the per-frame changed-cell `Vec`."** Both Ratatui's flush and `SocketBackend::draw` are already iterator-based and allocation-free on the diff path.
- **Do not "adopt Ratatui's `scrolling-regions` feature" for the Capsule.** That feature only affects `Terminal::insert_before` on an Inline viewport. The Capsule uses a Fullscreen viewport via `Terminal::new`, and the tree has zero `insert_before` calls and zero Inline viewports. The feature is inert for the Capsule re-emit path. The genuinely valuable idea hiding inside the old recommendation — emitting scroll regions in `SocketBackend` to cut bytes — is first-party work and is captured separately in Recommendation 6.

## Open implementation debates to preserve during the branch

The roadmap intentionally keeps multiple opinions where the code does not yet prove one design. This is not indecision; it is a research checklist for the future implementation branch.

|---|---|---|---|
| Ratatui diff modernization | Ratatui 0.30 `diff_iter`, `CellDiffOption`, and `CellWidth` are ready to exploit at the pane mapping seam. | The compositor currently forces full-frame repaint, so true diff behavior is hidden; restoring a trusted previous buffer must come first. | Add emitted-byte tests for unchanged frames before and after sentinel removal. ## Recommendations

Ranked by correctness value. Each one states why it matters, what it improves, the concrete implementation, and the acceptance coverage the branch must add before merge.

### 0. Restore a trustworthy Ratatui diff baseline by retiring the sentinel full-frame repaint

**Why this matters.** The current compositor intentionally fills Ratatui's current buffer with a sentinel and swaps buffers before every draw so every cell differs. That was a safe convergence stopgap while raw appenders and extra terminal-state writers could desynchronize Ratatui's previous buffer from the physical client screen. It is now the highest-leverage modernization blocker: it hides true Ratatui diff behavior, dominates emitted bytes, makes `AlwaysUpdate` meaningless in practice, and prevents reliable measurement of `CellDiffOption` and scroll-region improvements.

**What it improves.** Ratatui's previous buffer becomes the trusted physical-screen model again. Unchanged frames emit near-zero cell bytes, changed frames scale with actual damage, and every later recommendation can be judged against the real diff path instead of the forced repaint path.

**Two implementation views to compare.** The conservative view is to delete or model every extra writer that can mutate the client screen behind Ratatui, then remove the sentinel swap and rely on Ratatui's normal previous/current buffer diff. The more ambitious view is to introduce an explicit frame plan that commits every terminal-visible operation — cells, cursor state, hyperlinks, scroll ops, reset/wipe operations — through one ordered writer and updates Ratatui/physical state together. The first path is smaller and may be enough; the second path may be the cleaner long-term architecture if scroll ops, OSC 8 metadata, and backend-only SGR state all need first-class representation.

**How to implement.** Start by inventorying every path that writes bytes outside `SocketBackend::draw` and cursor/state reconciliation: title updates, out-of-band passthrough, mode prefaces, detach cleanup, dialog/chrome leftovers, and any raw appenders called from the compositor. Classify each as (a) not part of the cell model and safe at frame boundary, (b) terminal state that must be reconciled per frame, or (c) legacy raw paint that must move into Ratatui rendering or a frame operation. Once class (c) is gone, remove the sentinel fill/swap, keep real wipes only for first attach/resize/reset, and add diagnostics that fail or alert when an unchanged frame emits a full screen of cells.

**Acceptance coverage**: unchanged steady-state frame emits no cell body bytes or a tiny bounded amount; opening/closing dialogs clears all residue without sentinel repaint; selection highlight, scrollback view, focus swap, tab switch, resize, and detach/reset still converge; echo-back harness passes with sentinel disabled; emitted-byte counters show the expected drop for idle, typing, and small updates.

### 1. Define and enforce one display-width contract across `DamageGrid`, the Ratatui mapping, and the host terminal

**Why this matters.** This is the highest-priority correctness work and the one concrete divergence already proven in the tree. `jackin-term`'s cell width derives from `unicode-width` 0.2 with no special-casing (see <RepoFile path="crates/jackin-term/src/cell.rs" /> and the wide-handling in <RepoFile path="crates/jackin-term/src/grid/perform.rs" />). Ratatui 0.30's `ratatui_core::buffer::CellWidth` derives width from the same `unicode-width` 0.2 **plus** a terminal-compatibility adjustment: it adds `+1` for halfwidth katakana voiced/semi-voiced sound marks (`U+FF9E` / `U+FF9F`), which `unicode-width` reports as zero-width because they carry `Grapheme_Extend`. So for exactly those clusters, the model believes a cell is width 1 while Ratatui's buffer machinery believes it is width 2. When the model then writes the next cell at what it thinks is the adjacent column, Ratatui's diff — which assumes well-formed multi-width buffers — can compute a trailing-cell cleanup that the model never intended, producing stale tails, cursor drift, or right-edge corruption in a re-emitting multiplexer. This is precisely the class of bug that is invisible in unit tests and brutal to diagnose from logs.

**What it improves.** A single authoritative width per accepted grapheme cluster, shared by the parser, the model, the Ratatui mapping, and the bytes emitted to the host. No silent disagreement at the seam.

**How to implement.**
- Introduce a narrow width module owned by `jackin-term` (e.g. a `width` submodule, or a tiny internal crate if a second consumer appears) that is the single source of truth for "how many columns does this cluster occupy." Do not pull all of Ratatui into `jackin-term` for this; depend on `unicode-width` as today and, in tests only, cross-check the local function against `ratatui_core::buffer::CellWidth` so the two cannot drift.
- Decide the contract for each class deliberately and write it down: halfwidth katakana + dakuten/handakuten, VS16 emoji, emoji without VS16, ZWJ clusters, combining marks after a base, combining marks at column zero, wide glyph in the final column, and ambiguous-width code points. Where the target host terminals match Ratatui's compensation (the `+1` for `U+FF9E`/`U+FF9F`), port that behavior into the model so the model and the emitted bytes agree. Where jackin' intentionally diverges for emulator correctness, keep it local but record the divergence and test it.
- Wire `Cell::is_wide` / `Cell::is_wide_continuation` (already present in <RepoFile path="crates/jackin-term/src/cell.rs" />) to be derived from this one width function at parse time in <RepoFile path="crates/jackin-term/src/grid/perform.rs" />, so the model's wide flags and the width function can never disagree.
- Revisit `append_to_previous_cluster` in <RepoFile path="crates/jackin-term/src/grid.rs" />. Today cluster width stays the base width when VS16, combining marks, ZWJ, or subsequent ZWJ content joins the previous cell. If the canonical cluster width can grow or shrink after append, the implementation must update the lead width, add/remove continuation cells, dirty the affected span, and adjust cursor/pending-wrap semantics at the right edge. This is a correctness decision point, not an implementation detail to defer.

**Acceptance coverage** (added to <RepoFile path="crates/jackin-term/tests/conformance.rs" /> and the echo-back fixtures): CJK overwrite, CJK delete, CJK erase, emoji with VS16, emoji without VS16, ZWJ emoji clusters, combining marks after a base character, combining marks at column zero, halfwidth katakana dakuten and handakuten (the proven divergence), wide character in the final column, normal-width overwrite of a wide lead, normal-width overwrite of a wide continuation, right-edge deferred wrap plus SGR before the next printable, DSR cursor reply while wrap is pending, and scrollback snapshots containing all of the above.

### 2. Map `jackin-term` cells to Ratatui cells with `CellDiffOption`, instead of blanking continuation cells

**Why this matters.** Today the pane renderer in <RepoFile path="crates/jackin-capsule/src/tui/components/pane.rs" /> handles a wide character by writing the 2-column glyph into the lead Ratatui cell via `set_symbol`, and handling the continuation column with `buf_cell.reset()` — a literal blank space. That is only correct while the model and Ratatui agree on the lead glyph's width. Ratatui 0.30 added `CellDiffOption` precisely because the bytes stored in a buffer cell are not always what is visible: a cell may be model-known wide, a continuation that must not independently diff, or covered by protocol bytes such as an OSC 8 hyperlink. jackin' uses none of this and instead relies on width inference matching, which Recommendation 1 just proved is fragile.

**What it improves.** The mapping declares intent to Ratatui's diff rather than hoping inference agrees. The lead cell carries its model-known width; continuation and protocol-only cells are explicitly excluded from independent diffing; and the diff iterator's trailing-cleanup logic operates on correct widths.

**How to implement.** In the `render_cell` path of <RepoFile path="crates/jackin-capsule/src/tui/components/pane.rs" />, replace the implicit width handling with explicit `CellDiffOption` on the Ratatui `Cell`:
- Lead wide cell: set the symbol and style, then set `CellDiffOption::ForcedWidth(NonZeroU16::new(2)?)` so Ratatui treats it as two columns regardless of how its own width inference classifies the glyph.
- Continuation cell: keep it blank. The roadmap's first view allows `CellDiffOption::Skip` **only after** the new fixtures prove that wide-overwrite, wide-delete, and wide-erase still clear stale trailing background and style. The skeptical view is stricter: do not use `Skip` for ordinary wide continuations at all, because Ratatui already advances across a lead cell's effective width and `Skip` can hide a trailing clear after wide-to-narrow replacement. The branch should start with blank/default continuation cells, add `ForcedWidth` to the lead, then prove whether any `Skip` case is both necessary and safe.
- Protocol-only cells (e.g. cells that exist solely to carry an OSC 8 boundary, see Recommendation 3): `Skip` or `ForcedWidth` depending on their visible width.
- `CellDiffOption::AlwaysUpdate` is reserved for the side-channel state changes in Recommendation 3 and must be justified by the echo-back harness, never applied broadly. Forcing updates everywhere would mask equality bugs and inflate bytes per frame — the wrong outcome.

Keep this a small, commented mapping layer. The Ratatui pane conversion lives in `render_cell` in <RepoFile path="crates/jackin-capsule/src/tui/components/pane.rs" />. `snapshot_damagegrid_cells` in <RepoFile path="crates/jackin-capsule/src/tui/render.rs" /> is a selection/content snapshot rather than the Ratatui mapping path, but it still must agree on width and continuation semantics so copy/selection/snapshot behavior matches visible rendering.

**Acceptance coverage**: mapping tests from `jackin_term::Cell` / `SnapCell` to the Ratatui `Cell` for every cluster class in Recommendation 1, asserting the resulting `CellDiffOption`, plus a buffer well-formedness property (Recommendation 7) checked before diff.

### 3. Make OSC 8 hyperlink emission survive Ratatui's diff (latent bug, not in the first draft)

**Why this matters.** The hyperlink layer in <RepoFile path="crates/jackin-capsule/src/tui/socket_backend.rs" /> is a side channel: `set_hyperlink_regions` stores the frame's linked rects, and `draw` brackets linked cells with `emit_osc8_open` / `emit_osc8_close` (from the `ansi` module in <RepoFile path="crates/jackin-tui/src/lib.rs" />) **only for the cells Ratatui's diff actually yields**. If a link is added, removed, or retargeted over text whose glyph and style did not change, Ratatui's diff yields no cells for that region, so the OSC 8 state is never updated on the wire. The link silently fails to appear or fails to clear. Recommendation 2's `Skip` on continuation cells makes this worse, because cells covered by a wide glyph or by protocol bytes may never enter the diff at all.

**What it improves.** Hyperlink open/close boundaries become a tracked frame-level transition, correct independent of whether the underlying glyphs changed.

**How to implement.** There are two viable approaches, and the branch should compare them. The incremental approach tracks previous-frame and current-frame UI hyperlink membership at the backend or compositor level. When a cell's link membership or target changes between frames, force that cell into the diff by setting `CellDiffOption::AlwaysUpdate` on it in the mapping layer (Recommendation 2), so the `draw` loop re-emits the correct OSC 8 boundary. Keep the `AlwaysUpdate` set to exactly the boundary-changed cells — never all linked cells every frame — and measure the resulting bytes per frame. The more complete approach models OSC 8 as cell/frame metadata in `DamageGrid`: PTY OSC 8 open/close changes active hyperlink state, printed cells capture the active link, snapshots carry it, UI/dialog hyperlinks use the same metadata grid, and `SocketBackend` emits OSC 8 transitions from metadata rather than passthrough bytes. This unifies agent hyperlinks and jackin UI hyperlinks, but requires a broader model/snapshot/render change.

**Acceptance coverage**: link added over unchanged text; link removed over unchanged text; link target changed with unchanged text; PTY OSC 8 open/text/close in one batch; PTY OSC 8 spanning multiple writes; unsafe URI filtered without leaving an open link; link beginning and ending adjacent to wide/CJK/emoji cells; and an echo-back assertion (or recorded host capture) that the OSC 8 open/close sequence is correct after each transition. Re-run this after Recommendation 0 because the current sentinel full-frame repaint masks unchanged-cell invalidation bugs.

### 4. Stop dropping terminal-visible SGR state at parse time (attribute completeness)

**Why this matters.** The model's attribute set in <RepoFile path="crates/jackin-term/src/cell.rs" /> (`Attrs`) carries only foreground, background, bold, italic, underline, inverse, and dim. The encoder `apply_style` in <RepoFile path="crates/jackin-capsule/src/tui/socket_backend.rs" /> emits only those. Everything else an agent or program sets is dropped at parse time and can never reach the screen: **underline color** (`SGR 58`/`59`, which Ratatui's `underline-color` feature already models), underline styles (curly/dotted/dashed via `SGR 4:2`–`4:5`), strikethrough (`SGR 9`), blink (`SGR 5`/`6`), conceal (`SGR 8`), and overline (`SGR 53`). Modern coding agents and tools (e.g. error squiggles via colored curly underlines, diff strikethrough) use these. Given the quality-over-cost stance, dropping them is a fidelity regression, not a reasonable simplification.

**What it improves.** The Capsule faithfully reproduces what the agent's program actually rendered, instead of a lossy subset.

**How to implement.**
- Extend `Attrs` in <RepoFile path="crates/jackin-term/src/cell.rs" /> to carry the additional state. Underline color needs its own color field (default / indexed / RGB) since it is independent of foreground; the others are flags or a small underline-style enum.
- Preserve SGR subparameters before parsing the new cases. `csi_dispatch` in <RepoFile path="crates/jackin-term/src/grid/perform.rs" /> currently flattens every `vte::Params` entry to its first subparameter, which loses colon forms such as `SGR 4:2` and can lose colon-based color forms. The branch must keep enough structured SGR params for underline style and underline color to be parsed deliberately.
- Parse the additional SGR parameters in <RepoFile path="crates/jackin-term/src/grid/perform.rs" />, including the colon-subparameter form for underline style and `SGR 58`/`59` for underline color.
- Carry the new attributes through `SnapCell` in <RepoFile path="crates/jackin-term/src/snapshot.rs" /> and the `PaneCell` trait + `render_cell` in <RepoFile path="crates/jackin-capsule/src/tui/components/pane.rs" />. Ratatui's `Cell` already has `modifier` bits for `CROSSED_OUT`, `SLOW_BLINK`, `RAPID_BLINK`, and `HIDDEN`, and an `underline_color` field under the `underline-color` feature (enabled by default in the Ratatui meta-crate), so those can map to existing Ratatui fields. Ratatui `Cell` does not carry underline style variants or overline today; for those, compare two approaches: add a parallel render metadata grid consulted by `SocketBackend`, or deliberately keep them first-party/backend-only until upstream Ratatui gains native support.
- Emit the new state in `apply_style` in <RepoFile path="crates/jackin-capsule/src/tui/socket_backend.rs" />. Note that `apply_style` currently does a full `SGR 0` reset then re-applies; extend that re-apply set rather than adding incremental tracking, to keep it correct across cell runs.
- Decide and document one explicit non-goal if any attribute is intentionally excluded (for example, if blink is deliberately suppressed for the "no blinking" product value, model it but choose not to emit it, and write that down).

**Acceptance coverage**: parser round-trip for each new SGR (set, reset, and the colon-subparameter underline-style form); echo-back equality including the new attrs; and a golden wire-emit snapshot for a frame containing underline color + strikethrough.

### 5. Audit cursor and frame-terminal state for completeness

**Why this matters.** Cursor position, visibility, and DECSCUSR shape are already reconciled as frame-level state by the compositor in <RepoFile path="crates/jackin-capsule/src/daemon/compositor.rs" /> (a real strength — see Baseline reality). But "frame-level terminal state" is broader than the cursor glyph, and the branch should confirm nothing else is being emitted as scattered render bytes or dropped entirely.

**What it improves.** A single, auditable list of all terminal state that is reconciled per frame rather than written ad hoc, which is what keeps the one-writer invariant honest.

**How to implement.** Enumerate, in the compositor, every piece of state that must be diffed per frame: cursor row/col, cursor visibility, cursor shape (DECSCUSR), and evaluate whether cursor color (`OSC 12`) and cursor blink need modeling. Confirm that mouse mode, focus reporting, bracketed paste, and Kitty keyboard state remain reconciled as out-of-band sequences through `ClientWriter::enqueue_out_of_band` (see <RepoFile path="crates/jackin-capsule/src/client_writer.rs" />) and are never written mid-frame. The output is a documented state table plus tests for any gap found, not a rewrite of the working cursor path.

**Acceptance coverage**: extend the existing cursor tests in <RepoFile path="crates/jackin-capsule/src/daemon/tests.rs" /> (which already include a DECSCUSR case) to cover any newly modeled state. Assert the more precise invariant: Ratatui cell output and frame cursor reconciliation bytes are inside the `?2026` synchronized frame; out-of-band mode/passthrough bytes may remain outside the brackets only at frame boundaries and in the same socket write, never interleaved mid-frame.

### 6. Research first-party scroll-region (DECSTBM) emission to cut bytes — not upstream `scrolling-regions`

**Why this matters.** The Capsule repeatedly displays full-screen programs and agent output where large regions scroll. Re-emitting every changed cell of a scrolled region is bandwidth-expensive on SSH links, where low bandwidth is a core product value. Emitting a scroll-region escape (DECSTBM `CSI t;b r` plus index/reverse-index) can scroll the host's own screen and let the diff repaint only the newly exposed line. This is the real idea that the old "adopt Ratatui scrolling-regions" recommendation gestured at — but it is jackin-owned work, because Ratatui's feature only optimizes `insert_before`, which the Capsule does not use.

**What it improves.** Fewer emitted bytes for scroll-heavy output, with no visible flicker, on exactly the remote links where it matters most.

**How to implement (research-first, high-risk).** This is genuinely hard and must be proven, not assumed:
- Prefer source-recorded scroll operations over inferred buffer diffs. `DamageGrid` already owns DECSTBM, line feed at region bottom, insert/delete lines, CSI S/T, alternate screen, and scrollback mutation. Recording a semantic `ScrollOp` when the parser mutates rows gives the compositor exact intent (`rect`, direction, count, screen, reason) before Ratatui flattens the final frame. Inferring scroll from two completed buffers is possible, but likely more fragile and harder to prove around chrome, panes, wide cells, resize, and scrollback.
- The hard constraint is geometry. DECSTBM vertical margins scroll the **full terminal width** between the top and bottom margins. If a pane body does not span the full width, or shares rows with chrome (borders, status row, tab strip) or another pane, a scroll region will corrupt that chrome. DECSLRM left/right margins can restrict columns but are far less portable and carry their own mode state. So this optimization is only safe when the scroll rect spans the full terminal width, or when DECSLRM support is probed and reset safely.
- The second hard constraint is keeping Ratatui's previous buffer coherent. If the physical screen scrolls but Ratatui's previous buffer still describes the old rows, the next diff will corrupt or needlessly repaint. The branch must either mutate Ratatui's previous/current buffers to match the scroll (potentially via `Terminal::apply_buffer` for a controlled commit), or implement the scroll op below Ratatui with its own physical-state model. The latter risks reintroducing the parallel-writer desync the model deliberately retired — so it must stay inside the one-writer frame.
- Treat this as a candidate, not a default. Prototype it, measure emitted bytes against the current full-cell path on scroll-heavy fixtures, and adopt only if the bytes win is real and the echo-back harness still passes.

**Acceptance criterion (hard gate)**: after every candidate scroll-region change, the echo-back harness in <RepoFile path="crates/jackin-capsule/src/daemon/tests.rs" /> must still replay emitted bytes into the second `DamageGrid` and match the pane model inside the pane rect plus the cursor contract, across pane borders, status rows, tab strip, dialogs, scrollback view, alternate screen, primary screen, resize, and focus swaps. If parity cannot be proven, do not adopt scroll-region output.

### 7. Extend (do not rebuild) the conformance and echo-back harnesses, and add a buffer well-formedness invariant

**Why this matters.** The harnesses already exist (see Baseline reality). The risk is not their absence; it is fixture coverage and one missing structural invariant. A subtle trap: the echo-back harness replays emitted bytes into a *second `DamageGrid`*, which uses the same `jackin-term` width code as the source model. If both share a width bug, the harness false-passes. The width fixtures from Recommendation 1 are what make the harness able to catch emit-path width errors, because a wrong width changes the *emitted bytes*, which then misplace content when replayed.

**What it improves.** The existing safety net actually exercises the new failure modes (width classes, hyperlink transitions, new SGR), and a new property test catches malformed model→Ratatui buffers before they ever reach the diff.

**How to implement.**
- Add the width-class, hyperlink-transition, and SGR fixtures from Recommendations 1, 3, and 4 to <RepoFile path="crates/jackin-term/tests/conformance.rs" /> and the echo-back cases in <RepoFile path="crates/jackin-capsule/src/daemon/tests.rs" />.
- Add real recorded PTY fixtures (Claude Code, Codex, Amp, Kimi, OpenCode, `vim`, `tmux`, `htop`, shell prompts, `yes`, long `seq`, redraw storms) extracted from `--debug` run logs via the existing `jackin-xtask` PTY-fixture tool in <RepoFile path="crates/jackin-xtask/src/pty_fixture.rs" /> into the Capsule fixtures directory.
- Add a **buffer well-formedness property test** asserted on the Ratatui buffer *before* diff: every continuation has a matching lead; no non-blank cell sits inside a lead's effective width span; a wide glyph in the final column is handled explicitly; any `ForcedWidth` equals the model's physical width; and no stale continuation survives an overwrite/delete/erase/resize. This is independent of the backend's emitted bytes and catches mapping bugs at their source.
- Add a **true-diff baseline property** once Recommendation 0 lands: repeated rendering of an unchanged frame must not produce a full-screen diff, and any forced update must be explainable by metadata changes (hyperlink, cursor/state, external protocol coverage) rather than sentinel repaint.
- Keep the existing fuzz target in <RepoFile path="crates/jackin-term/fuzz/src/damage_grid_process.rs" /> and extend its corpus with the new cluster classes.

**Acceptance coverage**: the full extended automated set is green, including the well-formedness property over fuzzed inputs.

### 8. Prototype Termina at the raw terminal I/O boundary only (separate, lower priority)

**Why this matters.** Termina exposes terminal protocol details (typed CSI/OSC/DCS builders, raw/cooked mode, size queries, event parsing, and typed protocol queries/responses) more directly than Crossterm, and now has a Ratatui backend (`ratatui-termina`). It is promising for the host-side raw I/O and escape construction, but it is early (0.3.x), lower-level (no grid, diff, scrollback, or automatic application setup), and orthogonal to the model/diff correctness work above. It is not a replacement for `jackin-term` or for Ratatui.

**What it improves (if adopted).** Safer, typed reset/mode sequences and clearer protocol-reply handling at the boundary where jackin' currently uses Crossterm (`crossterm = "0.29"` in <RepoFile path="crates/jackin/Cargo.toml" />, <RepoFile path="crates/jackin-capsule/Cargo.toml" />, <RepoFile path="crates/jackin-console/Cargo.toml" />, <RepoFile path="crates/jackin-env/Cargo.toml" />, <RepoFile path="crates/jackin-diagnostics/Cargo.toml" />).

**How to implement.** Build a small `TerminalIo` abstraction and compare Crossterm and Termina for jackin's real surfaces: host console raw mode / alternate screen / mouse capture / event drain; launch TUI typed protocol queries/responses and size handling; Capsule attach raw mode, size reporting, reset, mouse/focus/bracketed-paste modes, and detach cleanup; and Kitty keyboard, SGR mouse, bracketed paste, focus events, OSC queries, and resize parity. Confirm whether `ratatui-termina` can preserve jackin's one-writer synchronized frame model and echo-back conformance. Adopt Termina only where it is clearly better for jackin's platform scope; otherwise keep Crossterm and borrow only Termina's explicit protocol style. Do not migrate all Crossterm usage in one unmeasured sweep.

**Acceptance coverage**: parity tests for every input/protocol surface above between the two backends before any swap.

### 9. Tighten the Ratatui dependency boundary and sweep widget ergonomics (secondary)

**Why this matters.** Ratatui 0.30 split into `ratatui-core`, `ratatui-widgets`, and backend crates. Keeping `jackin-term` free of a full Ratatui dependency (it depends only on `vte`, `unicode-width`, `compact_str`, `smallvec`) is correct and should be preserved; only the render crates need full Ratatui. Widget ergonomics (`Rect::layout`, centering helpers, layout cache, `Flex`, merged borders, `Table`/`List` selection, improved `TestBackend`) can simplify the TUI surfaces but are not terminal-model risks.

**How to implement.** Keep `jackin-term` on its current minimal deps; in tests only, cross-check the width module against `ratatui_core::buffer::CellWidth`. Opportunistically audit the console/launch/Capsule view code for widget sugar that removes local code or fixes UX, but do not let it distract from Recommendations 1–4. This is the last phase, not the first.

## Testing strategy for the implementation branch

The work is allowed to be a broad refactor, but terminal regressions are highly visible and hard to diagnose, so the bar is stricter than normal. The strategy is to extend existing infrastructure, not invent parallel infrastructure.

### Unit tests
- True-diff baseline tests after sentinel removal: unchanged frame emits no full-screen repaint, and any forced update has a recorded cause.
- Width-contract tests in `jackin-term` for every cluster class in Recommendation 1, cross-checked against `ratatui_core::buffer::CellWidth`.
- Mapping tests from `jackin_term::Cell` / `SnapCell` to Ratatui `Cell` + `CellDiffOption`, including continuation and forced-width cells.
- Parser round-trip tests for the new SGR attributes (Recommendation 4).
- Dirty-span tests proving wide overwrite, erase, delete, scroll, and resize mark the minimal correct spans or deliberately mark all when safer (extends <RepoFile path="crates/jackin-term/src/damage.rs" /> coverage).

### Integration and conformance tests
- Extend the parser conformance replay in <RepoFile path="crates/jackin-term/tests/conformance.rs" /> (one-shot vs byte-split determinism) with the new fixtures.
- Extend the fuzz target in <RepoFile path="crates/jackin-term/fuzz/src/damage_grid_process.rs" /> with the new cluster classes.
- Extend the Capsule echo-back harness in <RepoFile path="crates/jackin-capsule/src/daemon/tests.rs" /> with width, hyperlink-transition, and SGR cases, plus the buffer well-formedness property.
- Add real-PTY fixtures via <RepoFile path="crates/jackin-xtask/src/pty_fixture.rs" />.

### Performance and allocation checks
- Run <RepoFile path="crates/jackin-term/benches/present_frame.rs" />, <RepoFile path="crates/jackin-term/benches/scroll_throughput.rs" />, and <RepoFile path="crates/jackin-capsule/benches/pane_body.rs" /> before and after, plus the allocation regression in <RepoFile path="crates/jackin-term/tests/allocation.rs" />.
- Track concrete emitted-byte counters per frame, not just timings: total bytes, absolute cursor moves (`CSI H`), `SGR 0` reset count, OSC 8 open/close count, and full-screen erase count. The diagnostic scanner `scan_emitted_frame` in <RepoFile path="crates/jackin-capsule/src/client_writer.rs" /> already computes cursor-move, max-row/col, and erase counts behind the debug flag and is the right place to extend with SGR-reset and OSC 8 counters.
- Report frame timing p50/p95/p99 for idle, typing, streaming output, full-screen redraw, resize, scrollback view, and a scroll-heavy fixture, with a remote/high-latency profile where possible.

### Manual soak testing
Run the branch for normal work before merge, watching for flicker, stale cells, wrong cursor position/shape, mouse/focus breakage, paste behavior, scrollback correctness, resize behavior, dialog overlay cleanup, hyperlink rendering, and terminal reset after detach. Cover Ghostty, iTerm2, WezTerm, Alacritty, Kitty, and an SSH session, because the core product value includes no blinking, no flickering, and low bandwidth over remote links.

## Explicit non-goals

- Do not replace `DamageGrid` with Termina or with Ratatui's `Buffer`. Neither owns scrollback, query replies, passthrough policy, or terminal modes.
- Do not revive direct `GridPatch` ANSI emission as the live optimization path. It breaks the one-writer invariant and can desync Ratatui's previous buffer.
- Do not adopt Ratatui's `scrolling-regions` feature for the Capsule; it only affects `Terminal::insert_before`, which the Capsule does not use. Pursue first-party DECSTBM emission (Recommendation 6) instead.
- Do not rebuild the echo-back, conformance, fuzz, allocation, or bench harnesses; they exist. Extend them.
- Do not introduce a runtime feature flag as the primary validation mechanism. Validation is a branch with exhaustive tests plus manual soak, followed by replacing the old implementation if the new one is better.
- Do not apply `CellDiffOption::AlwaysUpdate` broadly or chase widget ergonomics before width/diff/output correctness lands.
- Do not migrate all Crossterm usage to Termina in one unmeasured sweep.

## Implementation phases

0. **True diff baseline.** Retire the sentinel full-frame repaint once every extra writer is modeled, moved into Ratatui rendering, or classified as safe boundary-only out-of-band state (Recommendation 0). Add unchanged-frame emitted-byte gates before judging later diff work.
1. **Width contract.** Implement the single display-width module (Recommendation 1), reconcile the proven halfwidth-katakana divergence, revisit cluster-width reflow in `append_to_previous_cluster`, and add the width fixtures to the existing conformance and echo-back harnesses.
2. **Ratatui cell mapping.** Replace implicit continuation blanking with explicit `CellDiffOption` (`ForcedWidth`, and `Skip` only if proven safe) in the pane mapping (Recommendation 2), and add the buffer well-formedness property test (Recommendation 7).
3. **Protocol-visible and attribute fidelity.** Compare backend-region OSC 8 invalidation with model-owned hyperlink metadata (Recommendation 3), preserve SGR subparameters, extend the SGR attribute set including underline color and backend-only attrs (Recommendation 4), and audit cursor/frame state for gaps (Recommendation 5).
4. **Scroll-op research.** Prototype source-recorded scroll operations and measure first-party DECSTBM emission under strict geometry/capability gating (Recommendation 6); adopt only on a proven byte win with echo-back parity and coherent Ratatui/physical buffer state.
5. **Termina prototype.** Compare Termina and Crossterm at the I/O boundary (Recommendation 8); adopt only where clearly better.
6. **Dependency boundary and ergonomics sweep.** Keep `jackin-term` minimal, cross-check width against `ratatui-core` in tests, and apply widget ergonomics where they remove local code (Recommendation 9).
7. **Soak and merge decision.** Run the full extended automated set, benchmark before/after with the byte counters, then perform manual operator soak long enough to catch flicker, stale cells, hyperlink and reset issues, and real-agent behavior. Merge only after the new path is the desired default.
