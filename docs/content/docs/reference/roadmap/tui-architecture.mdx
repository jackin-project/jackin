title: "TUI architecture - Ratatui application patterns"
---

**Status**: Partially implemented. The shared `jackin-tui` component crate exists, the launch cockpit is substantially extracted, and the capsule renders through Ratatui. The remaining work is to make `jackin-console`, `jackin-launch`, and `jackin-capsule` all follow one source layout and one Elm Architecture boundary.

## Goal

jackin' has three terminal applications:

- `jackin-console` - the host workspace console.
- `jackin-launch` - the launch progress cockpit.
- `jackin-capsule` - the in-container multiplexer.

All three should read like the same kind of Ratatui application. A contributor changing TUI behavior should know where to put state, messages, update logic, effects, subscriptions, rendering, terminal wiring, screen modules, and local components without searching through unrelated runtime, Docker, config, credential, or protocol code.

The target is not a visual redesign. The target is a clearer architecture: TUI code is grouped under the TUI boundary, non-visual product work stays outside it, and external work returns to the TUI through typed messages.

This is primarily a refactor. Existing operator workflows must keep working the same way unless a later PR explicitly calls out and justifies a user-visible behavior change. Structural moves may tune internal behavior and TUI rendering consistency, but they must not silently change command semantics, launch flow, capsule attach behavior, saved state, keyboard/mouse contracts, or error handling.

## Architecture rule

Each surface crate has one TUI boundary:

```text
crates/<surface-crate>/src/tui.rs
crates/<surface-crate>/src/tui/
```

`src/tui.rs` is the thin module root. It contains module declarations, intentional re-exports, and short `//!` orientation docs only.

`src/tui/` contains the whole TUI application layer for that surface, organized with The Elm Architecture:

| Layer | Owns |
|---|---|
| Model | Visible terminal state: active screen, focus owner, modal stack, scroll offsets, hover/click state, picker state, animation state, render-safe snapshots |
| Message | Semantic TUI events and results: key/mouse intent, resize, tick, background completion, dialog outcome, screen transition |
| Update | Deterministic state transitions: apply `Message` to the model and return typed effects |
| Effect / Subscription | Typed descriptions of non-TUI work and long-lived event sources; they describe work but do not execute external commands inside TUI code |
| View | Pure Ratatui/raw-ANSI frame composition from the current model |
| Terminal/run loop | Terminal lifecycle, backend adapter, event-loop wiring, and routing of completed effects back into messages |
| Components | Surface-local visual components that are not reusable enough for `jackin-tui` |

The shared component/design-system crate stays separate:

```text
crates/jackin-tui/src/
```

`jackin-tui` owns reusable visual primitives, tokens, theme adapters, geometry helpers, shared runtime abstractions, and lookbook-backed components. It must not contain console-, launch-, capsule-, Docker-, workspace-, credential-, or command-execution logic.

## Boundary rule

`src/tui/` is for terminal interaction. It is not a place for business logic or external command execution.

**Belongs in `src/tui/`:**

- TUI model structs and enums.
- TUI `Message` / `Action` enums.
- Update functions that mutate only TUI state.
- Effect and subscription type definitions.
- Event-loop wiring that maps terminal events and completed background work into messages.
- Terminal lifecycle/backend adapters, including raw mode, alternate screen, and capsule socket backends.
- Ratatui `Widget` / `StatefulWidget` implementations.
- Raw-ANSI rendering only where a surface still requires it.
- Frame composition, layout, geometry, hitboxes, focus rendering, hint/footer rendering, and local UI components.

**Stays outside `src/tui/`:**

- Docker/runtime command execution.
- Git, GitHub CLI, 1Password CLI, shell command, filesystem, and network operations.
- Config/schema loading, migration, and persistence.
- Workspace resolution, role loading, mount inspection, and durable state cleanup.
- Credential handling and token/domain logic.
- Capsule PTY/session/control-plane authority.
- Capsule wire protocol definitions.
- Pure non-visual product rules that do not depend on terminal interaction.

The allowed connection is typed effects:

```text
tui/update.rs
  -> returns Effect::RefreshInstances

effect executor / run loop
  -> calls non-TUI service/runtime code

non-TUI code
  -> performs Docker/runtime work
  -> returns typed data or typed error

tui/message.rs
  <- Message::InstancesRefreshed(result)

tui/update.rs
  -> updates the TUI model
```

No renderer, component, or update function should call Docker, git, `op`, `gh`, the filesystem, the network, or config persistence directly.

## Final layouts

These examples are the target shape. They intentionally use the Rust 2018 `foo.rs + foo/` module-root style used elsewhere in the repo.

```text
crates/jackin-tui/src/

|   |-- button_strip.rs
|   |-- confirm_dialog.rs
```

`jackin-tui` is the only place for cross-surface visual primitives. A surface-specific component starts in that surface's `src/tui/components/`; it graduates to `jackin-tui` only when more than one surface should consume the same implementation.

### Single-screen surface

Use this for `jackin-launch` and for `jackin-capsule` unless a surface gains multiple distinct screens later.

```text
crates/<surface>/src/
|-- lib.rs
|-- domain/                 # optional, pure non-visual rules
|-- services/               # optional, side-effect adapters
|-- tui.rs                  # thin module root
`-- tui/
    |-- app.rs              # TUI model
    |-- message.rs          # TUI message/action vocabulary
    |-- update.rs           # state transition + effect requests
    |-- effect.rs           # typed effect requests only
    |-- subscriptions.rs    # timers, background completions, long-lived streams
    |-- run.rs              # event-loop wiring
    |-- terminal.rs         # terminal lifecycle/backend/capabilities
    |-- view.rs             # top-level frame composition
    |-- components.rs       # thin local-component module root
    `-- components/
```

### Multi-screen surface

Use this for `jackin-console`.

```text
crates/jackin-console/src/
|-- lib.rs
|-- domain/                 # optional, pure non-visual console/product rules
|-- services/               # optional, command/IO adapters
|-- tui.rs                  # thin module root
`-- tui/
    |-- app.rs              # ConsoleApp, active screen, modal stack
    |-- message.rs          # top-level messages
    |-- update.rs           # top-level routing/update
    |-- effect.rs           # typed effect requests only
    |-- subscriptions.rs
    |-- run.rs
    |-- terminal.rs
    |-- view.rs             # top-level frame/chrome/modal composition
    |-- components.rs       # thin console-local component module root
    |-- components/         # console-local shared components
    |-- screens.rs          # thin screen module root
    `-- screens/
        |-- workspaces/
        |   |-- model.rs
        |   |-- message.rs
        |   |-- update.rs
        |   |-- effect.rs
        |   |-- view.rs
        |   `-- components/
        |-- editor/
        |   |-- model.rs
        |   |-- message.rs
        |   |-- update.rs
        |   |-- effect.rs
        |   |-- view.rs
        |   `-- components/
        `-- settings/
            |-- model.rs
            |-- message.rs
            |-- update.rs
            |-- effect.rs
            |-- view.rs
            `-- components/
```

Do not create `screens/` for a single-screen crate. Do not create nested `tui/screens/<screen>/tui/`; once code is inside `src/tui/screens/<screen>/`, the screen's TEA files already express the TUI layers.

4. **Keep non-TUI code outside.** External commands, filesystem/network/config/credential work, workspace resolution, and capsule protocol/session authority stay in root modules or future domain/service modules outside `src/tui/`.
5. **Replace direct work with effects.** If TUI update/render/component code needs Docker, git, `op`, `gh`, filesystem, network, config, or credential work, it returns a typed effect instead.
6. **Verify behavior after each move.** Run the affected unit tests, render snapshots/lookbook checks, docs link checks when docs changed, and focused manual terminal smoke for behavior a `TestBackend` cannot prove, especially console -> launch -> capsule handoff, raw-mode restore, mouse/focus behavior, and capsule attach/resize.
7. **Promote reusable visuals deliberately.** Surface-local components stay under the owning surface. Cross-surface components move to `jackin-tui` with lookbook coverage and snapshot/test coverage.
8. **Update docs as files move.** Keep [TUI Design Decisions](/reference/tui-design-decisions/) and [Codebase Map](/reference/codebase-map/) aligned with the new module shape.

## Acceptance criteria

- Each TUI surface has a thin `src/tui.rs` module root and the TUI application layer under `src/tui/`.
- `jackin-console` uses `src/tui/screens/<screen>/` only because it has multiple meaningful screens.
- `jackin-launch` and `jackin-capsule` do not grow `screens/` unless they gain multiple distinct screens.
- Existing operator-visible behavior is preserved unless an implementation PR explicitly documents and justifies a behavior change.
- TUI update/render/component code performs no external command execution, filesystem/network/config persistence, credential handling, or capsule protocol/session authority.
- External work requested by the TUI travels through typed effects/subscriptions and returns as typed messages.
- Shared visual primitives live in `jackin-tui`; surface-specific visual composition stays in the owning surface crate.
- Module-root files stay thin; no `lib.rs`, `tui.rs`, or other module root becomes a home for rendering, dispatch, state machines, or domain logic.
- Every implementation slice identifies the pre-existing behavior it protects and verifies it after the structural move with tests, snapshots, link checks, or manual terminal smoke as appropriate.
