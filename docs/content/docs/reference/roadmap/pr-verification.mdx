**Status**: Open — design proposal, no implementation committed. This page is the source of truth for the feature; it supersedes the earlier "standalone `jackin-dev` binary" sketch, which was dropped in favour of building the flow into the `jackin` binary itself. The standalone binary is gone, but its `dev` namespace survives as the `jackin dev …` maintainer subcommand group, keeping these tools off the operator-facing command surface while reusing one binary.

## Problem

jackin' pull requests carry a hand-written **Verify-locally** block: clone or update the repo, fetch the PR branch, build the right binaries, point a run at isolated state, run a smoke command. The block is copy-paste boilerplate, drifts between PRs, and every review re-derives the same steps by hand. With one human contributor and no second reviewer (<RepoFile path="AGENTS.md" /> → "Project staffing"), this manual loop runs on every PR. And the loop is not unique to jackin': every project developed inside a jackin' workspace has the same "check out this PR, build what it changed, run the smoke that proves it" ritual. jackin' verifying its own PRs is just the first case — the design is generic, and the per-project recipe that drives it lives in each project, not in jackin'.

Three things make a standalone tool the wrong answer and `jackin` itself the right one:

- **A workspace already encodes what to verify.** A jackin' workspace declares a primary `workdir` plus N project mounts and an active role — exactly "what I'm working on and what I'm verifying". The author's own jackin' workspace mounts eight sibling repos (the CLI, `agent-smith`, `homebrew-tap`, `validate-agent-action`, …) with one primary `workdir`. The workspace is the natural source of *addressing* — "which repo does PR 479 belong to" — while the build/run recipe lives in that repo, never in the workspace config.
- **A PR sometimes needs a companion role branch.** A change in the jackin' repo can require a matching change in a role repo (`the-architect`, `agent-smith`). Verifying that PR means running it with a specific role pinned to a specific branch — and jackin' already supports `--role-branch`.
- **Testing jackin' itself needs clean state.** Verifying a jackin' PR by running the freshly built binary must not read or mutate the operator's real `~/.config/jackin/` and `~/.jackin/`. jackin' already supports redirecting both with `JACKIN_CONFIG_DIR` / `JACKIN_HOME_DIR` (<RepoFile path="src/paths.rs" />), so an isolated verification home is a matter of wiring, not new machinery.

## Scope

**First focus: pull requests on project repos, verified host-side. Nothing more.** Branches as a standalone target, role-repo PRs as a standalone target, merge-preview, and coordinated multi-repo verification are explicitly deferred (see "Deferred"). The one branch-shaped capability kept in the first slices is **pinning a companion role to a branch while verifying a PR** — because that is a real need for PRs that pair a CLI change with a role change.

## Vision

PR verification is a **first-class `jackin dev` operation**, driven by the workspace as its source of identity and run **host-side**. The spine is **diff-driven**: jackin' reads what the PR changed (`gh pr diff <n> --name-only`) and surfaces the verification commands those changes call for — compile the crate a `src/**` edit touches, smoke the CLI (`target/debug/jackin console`), spin up the docs server and open the pages a `docs/**` edit touches. It checks the PR out into a managed worktree, points an isolated config + state home at it, and runs the repo's `jackin.tasks.toml` tasks **on the operator's machine** against that worktree. There is **one task shape**; the verbs are views over it: `build` runs the build-set tasks the diff lit up, `run <name>` runs any task on demand. No container is launched to run tasks. This verifies the built binary/project/docs with zero blast radius on the operator's real jackin' setup.

```bash
jackin dev pr verify 479                                  # PR on the workspace's primary repo
jackin dev pr verify agent-smith#12                       # PR on another mounted repo
jackin dev pr verify 479 --role the-architect@some-branch # pin a companion role to a branch for this PR's runs
```

Build/run verbs come from a `jackin.tasks.toml` **committed in the project repo**, so the recipe versions with the code and a PR that changes how it is built or verified updates the recipe in the same diff. The workspace config carries **no** recipe — it only *addresses* the target (resolves which repo and PR `479` or `agent-smith#12` names). That split is the generic contract: any project developed in a jackin' workspace gets PR verification by committing its own `jackin.tasks.toml`; jackin' verifying its own PRs is just the first user.

## CLI surface

Maintainer-facing, so it lives under a `dev` subcommand group that stays off the operator surface — noun-first within that group, mirroring `jackin workspace` / `jackin role` / `jackin config`:

```text
jackin dev pr verify <pr> [--role <role>[@<branch>]] [--isolation local-config|clean] [--workspace <name>] [--run <task>] [--all] [--no-interactive]
jackin dev pr build [name…]              # run jackin.tasks.toml build steps for the active verify
jackin dev pr run <task> [args…]         # run a named task, appending args verbatim
jackin dev pr refresh                    # re-fetch the PR head, reset the worktree
jackin dev pr refresh-build              # refresh then build
jackin dev pr list                       # cached verifies: number, title, isolation, staleness
jackin dev pr clean [<pr>|--all]         # drop worktree + isolated home/config, reclaim disk
```

`<pr>` is a PR number (`479`, the workspace's primary repo) or a `<repo>#<n>` form selecting another mounted repo. `--role <role>[@<branch>]` pins a companion role — and, with `@<branch>`, its branch — for any inner `jackin` launch a task performs (§4). Every invocation honours `--debug` and writes its firehose to the diagnostics run file, per <RepoFile path="AGENTS.md" /> → "Telemetry must be debuggable".

## Design

### 1. Target — a project-repo pull request

The workspace (active, or `--workspace <name>`) resolves the target. Bare `<n>` → the primary `workdir` repo's PR; `<repo>#<n>` → the mount whose `origin` slug matches `<repo>`. PR metadata comes from `gh pr view <n> --repo <slug> --json number,title,headRefName,headRefOid,baseRefName,state,isCrossRepository`. There is exactly one target kind in the first slices: a pull request on a repo the workspace mounts.

### 2. PR checkout store

jackin' owns one host directory for all verification checkouts, under the `~/.jackin/` host root (<RepoFile path="AGENTS.md" /> → path conventions):

```text
~/.jackin/pr/<owner>/<repo>/
├── repo/                  # one cached clone per repo (fetched, not per-PR)
└── <number>/
    ├── meta.json          # number, title, headRef, base, headSha, lastRefresh, state, isolation
    ├── worktree/          # git worktree detached at the PR head
    ├── config/            # JACKIN_CONFIG_DIR for the run (§5)
    └── home/              # JACKIN_HOME_DIR for the run (§5)
```

One cached clone per repo plus a **worktree per PR** (cheap to add/drop). Checkout uses `git fetch origin pull/<n>/head` (forks need no extra remote); `refresh` does fetch + `git reset --hard <new headSha>`, the correct response to a **force-pushed** PR branch. A lock file under each `<number>/` guards concurrent verifies of the same PR. `jackin dev pr clean` reclaims and reports disk used.

### 3. Execution model — host-side only

jackin' runs the repo's tasks on the operator's machine against the checkout worktree; it does not launch a container to run tasks:

1. Resolve and check out the PR head into `…/<n>/worktree/` (§2).
2. Set up the isolation profile (§5): export `JACKIN_CONFIG_DIR` / `JACKIN_HOME_DIR` for the run.
3. Run `jackin.tasks.toml` build steps from the worktree root, rendering their output into the surface's output pane and propagating exit codes.

**Which binary a task invokes matters.** When verifying jackin' itself, a `run console` task must execute the *freshly built* `target/debug/jackin`, not the operator's installed `jackin`. The runner prepends the worktree's build-output dir to `PATH` for task execution (tasks may also reference `${worktree}` / `${target}` placeholders), so `jackin console` inside a task resolves to the binary under test, whose own runs are sandboxed by the isolation profile (§5).

### 4. Companion role pinning (the branch-shaped need)

A PR that pairs a CLI change with a role change needs its inner `jackin` launches to use a specific role at a specific branch. `--role <role>[@<branch>]` records that pairing for the verify run: any task that launches jackin' (`jackin load …`, `jackin console`) is invoked with the role selected and, when `@<branch>` is given, `--role-branch <branch>` threaded through (<RepoFile path="src/cli/role.rs" />). This reuses the existing role-branch checkout path; no new mechanism. Without `--role`, tasks use whatever the (isolated) config/workspace would pick by default.

### 5. Isolation profiles (verifying jackin' with jackin')

jackin' separates two on-disk roots — config (`JACKIN_CONFIG_DIR` → `~/.config/jackin`: workspaces, mounts, auth, trust) and state/home (`JACKIN_HOME_DIR` → `~/.jackin`: instances, caches, role checkouts, sockets, agent homes) — and <RepoFile path="src/paths.rs" /> keeps the two overrides independent. A verification run drives both, scoped under the PR's store dir, via `--isolation <profile>`:

- **`local-config`** — verify against the operator's *real* configuration with *no* state. jackin' **copies** the real `~/.config/jackin` into the run's `config/` (real workspaces/mounts/auth as input), and `home/` starts **empty**. Every write the built binary makes (schema migrations, `last_role`, trust-on-first-use, new instance state) lands in the throwaway copy and empty home, discarded by `jackin dev pr clean`. The real config is never written; the real state is never read.
- **`clean`** — pure isolation: both `config/` and `home/` start empty. Clean-room jackin' from zero, for first-run behaviour and defaults.

<Aside type="caution">
**Copy, never reference, the real config.** A PR that bumps `CURRENT_CONFIG_VERSION` / `CURRENT_WORKSPACE_VERSION` migrates config files in place on load (<RepoFile path="AGENTS.md" /> → "Project status: pre-release"). Pointing `JACKIN_CONFIG_DIR` at the live `~/.config/jackin` would let a verification run rewrite the operator's real workspace files to the PR's schema version. `local-config` therefore snapshots config into the per-PR `config/` and runs against the snapshot.
</Aside>

Running against the operator's real config **and** real state in place is deliberately **not** a profile — it cannot satisfy the no-mutate guarantee and is the exact "broke everything I have" outcome the feature exists to prevent.

### 6. Task surface — repo-committed `jackin.tasks.toml`

Tasks live in a `jackin.tasks.toml` at the **root of each project repo**, so the recipe versions with the code. The primary `workdir` repo's file is the default; each mounted project can carry its own. There is **one task shape**, `[[task]]`; optional fields select behaviour rather than forking into separate kinds (<RepoFile path="AGENTS.md" /> → "Reuse before writing — DRY"):

- `build = true` — include in the bare `jackin dev pr build` set (ordered, stop-on-first-failure). Absent ⇒ runs only when named via `run`.
- `paths = [...]` — globs for relevance selection (§7).
- `serve = { … }` — mark a long-running server that opens the PR's changed docs pages (§8).

Trailing args are appended to any task's `run` verbatim. The verbs are views over the one list: `build` runs every `build = true` task; `run <name>` runs any task by name.

```toml
# jackin.tasks.toml — verification recipe for this repo
version = 1                          # plain guard, not a migrating schema

[[task]]
name = "cli"
run = "cargo build --bin jackin"
build = true                         # part of bare `jackin dev pr build`
paths = ["src/**", "Cargo.*"]        # §7 relevance selection

[[task]]
name = "docs-build"
run = "bun run build"
cwd = "docs"                         # relative to worktree root
build = true
paths = ["docs/**"]

[[task]]
name = "console"
run = "jackin console"               # resolves to the built binary on PATH; `run console --debug` appends args

[[task]]
name = "test"
run = "cargo nextest run"

[[task]]
name = "docs"
run = "bun run dev"                  # long-running; jackin keeps it up until the operator quits
cwd = "docs"
# presence of `serve` marks this task long-running + page-opening:
serve = { ready = "http://localhost:3000/", route_from = { dir = "docs/content/docs", strip_suffix = ".mdx", index = "index", base = "/" } }
```

`jackin.tasks.toml` is a self-contained task-runner shape: each `[[task]]` carries its own `run` string, so a repo is verifiable without any other task tooling. It is **related to** the [custom operator tools](/reference/roadmap/custom-operator-tools/) `[[tool]]` array but deliberately **not the same type**. `[[tool]]` is an operator hotkey bound in jackin' config (`type`/`key`/`exec`, fired from inside the console); `[[task]]` is a maintainer verification recipe committed in the repo under test (`name`/`run`/`build`/`paths`/`serve`, run host-side). Different home, different audience, different fields — per <RepoFile path="AGENTS.md" /> → "Reuse before writing — DRY", the divergence is structural, so the two stay separate rather than collapsing into a forced common shape; revisit a shared resolver only if [task source abstraction](/reference/roadmap/task-source-abstraction/) lands a concrete one. It is **not** a versioned jackin' schema: it lives in the repo under test, jackin' is pre-release, and breaking shape changes fail with the standard parser error.

### 7. Relevance selection — drive the command set from the diff

The spine of `verify`: map the PR's changed paths to the tasks they touch, so the operator sees *the commands this PR's changes call for* rather than a flat task list.

- jackin' computes changed paths once with `gh pr diff <n> --name-only` and matches them against each task's `paths` globs. Tasks whose globs match are the PR's **relevant set** — surfaced first and run by a bare `verify`.
- **Suggestion, never silent skip.** Non-matching tasks are still shown, just not auto-selected; `--all` runs everything and `run <name>` runs any task by name. An empty or unknown change set (no globs declared, fork diff unavailable) falls back to "everything is relevant". Per <RepoFile path="AGENTS.md" /> → "No silent caps", log which tasks were de-selected and why.
- Globs choose *which task* is relevant (the `cli` build vs `docs-build` vs the `console` smoke), **not** which source files a build recompiles — Cargo and the docs bundler already do file-level change detection inside a task. So a Cargo manifest or lockfile edit lights up every Rust task; a `docs/**` edit lights up the docs build and `serve` tasks.

**Rich interactive surface.** `verify` opens a **full-screen rich surface** (alternate screen), not a line-by-line CLI — this is the surface we expect to grow (a diff view, PR comments, several concurrent task panes, a log tail), so it starts rich. It reuses the console's render and layout machinery (<RepoFile path="src/console/mod.rs" />) and follows the [TUI design decisions](/reference/tui-design-decisions/) (navigation, focus, ARIA-tabs, palette, scroll). Layout: a PR context header, a left task list grouped relevant-first with per-task run-state badges, a right **captured-output pane**, and a key-hint footer.

```text
┌──────────────────────────────────────────────────────────────┐
│ jackin dev pr · #480 docs(roadmap): pr verification
│ base main ← head feature/… · OPEN · isolation local-config
├──────────────────────────────┬───────────────────────────────
│ Relevant to diff (docs/**)   │ docs-build · bun run build
│  ▸ docs-build  [build] ● ok  │ $ bun run build
│    docs        serve   ○     │ ✓ 142 pages built in 3.1s
│                              │ ✓ check:repo-links ok
│ Other tasks                  │ …
│    cli         [build] ○     │
│    console             ○     │
│    test                ○     │
├──────────────────────────────┴───────────────────────────────
│ enter run · a all-relevant · A all · r refresh · o open · q quit
└──────────────────────────────────────────────────────────────┘
```

Task output renders **inside** the output pane (run-state badges `○` idle, `◐` running, `●` pass, `✗` fail), never as raw stdout over the surface. `enter` runs the focused task, `a` every relevant task in declared order, `A` all tasks, `r` refreshes the PR head, `q` quits and tears down any `serve` child (§8).

<Aside type="note">
Because the surface is rich, `--debug` does not paint over it: the run — every external command (`cargo`, `bun`, `git`, `gh`) and its output — is captured to the diagnostics log file at `~/.jackin/data/diagnostics/runs/<run-id>.jsonl`, per [Debug output never reaches a rich full-screen TUI](/reference/tui-design-decisions/#debug-output-never-reaches-a-rich-full-screen-tui). Share the run id to triage a failed task instead of pasting scrollback.
</Aside>

**Non-interactive fallback.** A non-TTY (CI, a pipe) or `--no-interactive` skips the rich surface entirely: `verify` auto-runs the relevant build set with output streamed normally to stdout, prints the other relevant commands as copy-paste `jackin dev pr run <task>` lines, and exits non-zero on any failure. `--run <task>` / `--all` drive specific tasks without the surface.

### 8. Doc-page verification (`serve` tasks)

Some PRs change documentation, and the review need is "spin up the changed docs locally and open the pages this PR touched" — the PR template's **Documentation** walk, automated. A task carrying a `serve` field is run the same way as any other (`jackin dev pr run docs`), but its `serve` attributes tell jackin' to treat it as a long-running server and derive the pages to open:

1. `jackin dev pr run docs` (or `verify --run docs`) starts the task's `run` from `cwd`, rendering output into the pane, and waits until `serve.ready` responds (HTTP probe).
2. jackin' computes the PR's changed files (`gh pr diff <n> --name-only`, the same source as §7), filters to those under `route_from.dir`, and maps each to a page URL via `route_from` (strip `strip_suffix`, collapse the `index` segment, prefix `base`, join to the server's `ready` base). Example: a changed `…/content/docs/reference/roadmap/<page>.mdx` → `http://localhost:3000/reference/roadmap/<page>/`.
3. Each derived URL is opened with the `open` crate; if none match, jackin' prints the base URL. The server stays up until the operator quits (`q` / `Ctrl-C`), then is terminated cleanly.

The server command, port, and base URL are **declared in the recipe, never hardcoded in jackin'** — so a docs-engine change is a one-line edit to `jackin.tasks.toml`, not a jackin' code change. (This site's own move from Astro + Starlight to Fumadocs changed exactly those values — the dev-server command and port — which is precisely the kind of change the recipe absorbs.) Opening URLs should route through the same launcher resolution the [operator handler system](/reference/roadmap/operator-handler-system/) defines, rather than calling `open` directly in more than one place (<RepoFile path="AGENTS.md" /> → "Reuse before writing — DRY").

### 9. Data stored per PR

Everything under `~/.jackin/pr/<owner>/<repo>/<n>/` (§2, §5): `meta.json`, `worktree/`, and the per-run `config/` + `home/` (`config/` is a real-config snapshot under `local-config`, empty under `clean`; `home/` always starts empty). `meta.json` powers `jackin dev pr list` without re-hitting the network.

### 10. Console surface

A "Verify PR" `ConsoleInstanceAction` (<RepoFile path="src/console/mod.rs" />) prompts for a PR and opens the **same rich relevant-command surface** as the CLI `verify` (§7) — not a separate UI; it plugs into the same instance-action machinery as Reconnect / NewSession / Shell.

### Host-side effects

Per <RepoFile path="AGENTS.md" /> → "Never mutate the host machine silently", verification is operator-invoked, but its writes stay confined:

- **Writes only under `~/.jackin/pr/` and the role cache under `~/.jackin/roles/`.** Never the operator's working repos, never the real `~/.jackin/` or `~/.config/jackin/`.
- **The real config is copied, never written** — even a schema-bumping PR cannot migrate the operator's live workspace files (§5).
- **Read-only against host auth.** `gh auth token` / `gh auth status` are reads; verification never runs `gh auth setup-git` or rewrites host git config. Tasks that need auth inherit the host's `gh`/git credentials read-only.
- **The persisted workspace file is never mutated.**

### Risks, edge cases, failure modes

- **`gh` missing / unauthenticated** — pre-flight `gh auth status`; one-line install/login hint.
- **Slug resolution from a mount** — read each mount's `origin`; handle no-git mounts, multiple remotes, and SSH-vs-HTTPS URL forms when mapping `<repo>` → `owner/repo`.
- **Which binary a task runs** — the worktree build output must shadow the installed `jackin` on `PATH` (§3), or `run console` silently tests the wrong binary.
- **Build cache** — a per-PR `target/` is fully isolated but forces a cold rebuild on every PR; a shared host `target/` leans on Cargo's fingerprinting and is far faster for an iterating maintainer. Default to the **shared** cache with a `--fresh` flag for the rare fingerprint-bleed case, and log which was used (<RepoFile path="AGENTS.md" /> → "No silent caps").
- **Fork / cross-repo, force-push, closed/merged** — `pull/<n>/head` + detached worktree; `refresh` resets hard; `meta.json.state` warns but allows post-merge inspection.
- **Concurrency** — per-PR lock (§2).
- **No task file** — `build`/`run` with no `jackin.tasks.toml` print a one-line `no recipe in <repo>` error.
- **Serve lifecycle** — a `serve` task's server is a child that must be terminated on quit/`Ctrl-C`; spawn it in its own process group and signal the **group** on teardown so child workers (Vite, esbuild) don't orphan. The `ready` probe needs a timeout so a server that never binds fails loudly instead of hanging.
- **Doneness** — in `--no-interactive` mode `verify` prints a pass/fail summary and exits non-zero on any build/task failure so scripts and CI see it; the rich session shows the summary in the surface and returns the same code on quit. A `serve` task is interactive, not pass/fail, so it does not gate the exit code.

## Deferred

Explicitly out of the first slices, recorded so the design stays coherent when they land:

- **Branches as a standalone target** (`jackin dev branch verify <ref>`). PRs only for now. The store layout (§2) and runner already generalize to a `branch/` sibling.
- **Role-repo PRs as a standalone target** (`jackin dev pr verify the-architect#42`). Today roles enter only as the `--role <role>@<branch>` companion pin (§4); verifying a role repo's own PR end-to-end is the same `--role-branch` machinery and can be added later.
- **Merge preview** (`--merge` → `pull/<n>/merge`) — verify post-merge-into-base state. Default stays PR head.
- **Coordinated multi-repo verification** — one verify spanning several of the workspace's repos (the PR template's "Related pull requests"). The per-repo store keying already accommodates it.
- **In-container agent review** — launch the workspace container, rebind the matched mount to the PR worktree (launch-time overlay), and let an agent review in the multi-project context. Needs a new `jackin-capsule run -- <cmd>` verb and the per-mount isolation materialization (<RepoFile path="src/isolation/materialize.rs" />). The host-side model covers the daily "does this PR build and run" need; agent-assisted review is additive.

If implementation adds any field to `WorkspaceConfig` / `MountConfig` (e.g. a persisted default verify target), that touches a versioned schema and triggers the five-artifact migration requirement in <RepoFile path="AGENTS.md" /> → "Project status: pre-release". ## Prior art

No existing tool covers the host-side-verify + companion-role-branch + isolated-jackin'-state combination; the closest pieces would have to be glued by hand:

- [`gh pr checkout`](https://cli.github.com/manual/gh_pr_checkout) — checkout by number; no build/run recipe, no isolation.
- [`gh-worktree`](https://github.com/eikster-dk/gh-worktree) — checks a PR out into a worktree; worktree only.
- [HighReview](https://github.com/HighGarden-Studio/HighReview) — local PR review via worktree with IDE/AI analysis; diff-review focus, no configurable build/run tasks.
- [`gh-review`](https://crates.io/crates/gh-review) — terminal diff/comment/approve UI; review surface, not build/verify.
- Nx / Turborepo / Bazel / Buck2 — affected-target selective rebuild; prior art for §7, not PR-scoped review.
- `just` / `mise` — repo-committed task runners; tasks but no PR checkout or isolation.

The novel combination: host-side PR verification driven by the workspace, with companion-role-branch pinning and an isolated jackin' config/state home so jackin' can verify itself safely.

## Phased roadmap

- **Phase 0 — Tasks + store.** Parse `jackin.tasks.toml`. Implement the `~/.jackin/pr/<owner>/<repo>/` store: cached clone + `git fetch pull/<n>/head` + worktree, `meta.json`, per-PR lock, `jackin dev pr clean`.
- **Phase 1 — Host-side verify (run-all baseline).** `jackin dev pr verify <n>` / `<repo>#<n>`: resolve, checkout, run all `build = true` tasks host-side with the worktree binary on `PATH`. Pass/fail summary, non-zero exit on failure. Bare `verify` runs the whole build set until relevance selection lands in Phase 3.
- **Phase 2 — Verbs.** `jackin dev pr build`/`run <task> [args]`/`refresh`/`refresh-build`/`list`.
- **Phase 3 — Relevance selection + interactive surface (the diff-driven spine).** `gh pr diff <n> --name-only` × `paths` globs → the PR's relevant task set; the **rich** surface that lists relevant-first and lets the operator pick/run (`enter`/`a`/`A`/`r`/`q`) with output in a pane, plus `--all` / `--run` / `--no-interactive` for scripted use; suggestion-not-skip; de-selection logging (§7).
