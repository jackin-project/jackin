title: "Workspace-Native PR Verification"

**Status**: Open — design proposal, no implementation committed. This page is the source of truth for the feature. It describes the **experience and the flow** — what the operator sees, selects, and sets up — not the internal storage or code layout, which is expected to change as the console's state handling is reworked. The shape to preserve is the concept: PR verification is **console-driven** (select a workspace, pick one of its pull requests, choose how to verify it) and runs **host-side**, with no separate maintainer CLI verb.

## Problem

jackin' pull requests carry a hand-written **Verify-locally** block: clone or update the repo, fetch the PR branch, build the right binaries, point a run at isolated state, run a smoke command. The block is copy-paste boilerplate, drifts between PRs, and every review re-derives the same steps by hand. With one human contributor and no second reviewer, this manual loop runs on every PR. And the loop is not unique to jackin': every project developed inside a jackin' workspace has the same "check out this PR, build what it changed, run the smoke that proves it" ritual. jackin' verifying its own PRs is just the first case — the design is generic, and the per-project recipe that drives it lives in each project, not in jackin'.

Three things make the console the right home for this, with the workspace as the source of identity:

- **A workspace already encodes what to verify.** A jackin' workspace declares a primary working directory plus a set of project mounts and an active role — exactly "what I'm working on and what I'm verifying". The console already lists workspaces and lets the operator act on one, so "verify a pull request on this workspace's repo" is a natural action to hang off that selection. The workspace supplies the *addressing* — which repo a PR belongs to — while the build/run recipe lives in that repo, never in the workspace.
- **A PR sometimes needs a companion role branch.** A change in the jackin' repo can require a matching change in a role repo. Verifying that PR means running it with a specific role pinned to a specific branch — something jackin' already knows how to do.
- **Testing jackin' itself needs clean state.** Verifying a jackin' PR by running the freshly built binary must not read or mutate the operator's real configuration and state. jackin' can already redirect both to alternate locations, so an isolated verification environment is a matter of wiring, not new machinery.

## Scope

**First focus: open pull requests on a workspace's primary repo, picked and verified from the console, host-side. Nothing more.** PRs on the workspace's other mounts, branches as a standalone target, role-repo PRs as a standalone target, merge-preview, coordinated multi-repo verification, and any scripted/non-interactive entry are explicitly deferred (see "Deferred"). The one branch-shaped capability kept in the first slices is **pinning a companion role to a branch while verifying a PR** — because that is a real need for PRs that pair a CLI change with a role change.

## Vision

PR verification is a **first-class console flow**, driven by the workspace as its source of identity and run **host-side**. The operator opens the console, selects a workspace, picks one of its pull requests from a list, and chooses how to verify it — then jackin' does the work. The spine is **diff-driven**: jackin' reads what the PR changed and surfaces only the verification commands those changes call for — compile the code an edit touches, smoke the CLI, spin up the docs server and open the pages a docs edit touches. It checks the PR out into an isolated, throwaway location, points an isolated configuration and state home at it, and runs the repo's own verification recipe **on the operator's machine**. No container is launched to run the tasks. The result: the built binary, project, or docs are verified with zero blast radius on the operator's real setup.

The recipe comes from a file **committed in the project repo**, so it versions with the code — a PR that changes how the project is built or verified updates the recipe in the same diff. The workspace supplies no recipe; it only addresses the target (which repo the primary working directory names, and therefore which repo's PRs the picker lists). That split is the generic contract: any project developed in a jackin' workspace gets PR verification by committing its own recipe; jackin' verifying its own PRs is just the first user.

## What you'll see and do

The whole feature lives inside the console, as four steps. Each step is a screen; the operator moves forward by selecting and backward by going up a level.

### Step 1 — Pick a workspace

The console already opens to a list of workspaces (and the current directory). Verifying a PR is a new action offered on a selected workspace, alongside launching or attaching. It needs no running session — selecting it resolves the workspace's primary repo and moves to the PR list.

```text
┌─ jackin · workspaces ──────────────────────────────────────────┐
│ ▸ jackin          ~/Projects/jackin           the-architect      │
│   agent-smith     ~/Projects/agent-smith       agent-smith       │
│   docs-site       ~/Projects/docs-site         the-architect     │
├────────────────────────────────────────────────────────────────┤
│ enter open · v verify a pull request · n new · q quit            │
└────────────────────────────────────────────────────────────────┘
```

### Step 2 — Pick a pull request

jackin' lists the open pull requests on the workspace's primary repo, newest activity first. Each row shows the number, title, head branch, and draft/open state. A PR that has already been checked out for verification is flagged with how long ago it was last refreshed, so the picker doubles as the "what have I already got on disk" view — there is no separate management screen. The operator can filter as they type, reclaim the disk a cached verification used, and refresh the list against GitHub.

```text
┌─ Verify a pull request · workspace: jackin · repo: jackin-project/jackin ─┐
│   #493  fix(agent-binary): fall back to cached binaries   fix/…   OPEN     │
│ ▸ #480  docs(roadmap): workspace-native PR verification   feat/…  OPEN  ●  │  ● cached · 2h ago
│   #489  fix(console): show role load progress             fix/…   OPEN     │
│   #487  fix(net): send User-Agent to GitHub               net/…   DRAFT    │
├───────────────────────────────────────────────────────────────────────────┤
│ enter verify · / filter · x reclaim cached · r refresh list · q back        │
└───────────────────────────────────────────────────────────────────────────┘
```

### Step 3 — Choose how to verify

Selecting a PR opens a short setup screen. Everything here has a sensible default, so the operator can press enter to accept and go straight on, or adjust:

- **Isolation** — *Against my real config* (the operator's real configuration as read-only input, no state carried in — the default) or *Clean room* (empty configuration and state, for first-run behaviour and defaults). Verifying in place against the real configuration *and* real state is deliberately not offered, because it cannot keep the no-mutate promise.
- **Companion role** — *None* (default), or a role to pin for any jackin' launch this PR's tasks perform, optionally at a specific branch. This is the branch-shaped need: a PR that pairs a CLI change with a role change.
- **Build cache** — *Shared* (fast; leans on the compiler's own change detection — the default) or *Fresh* (cold build, for the rare case where a shared cache would bleed state between PRs).

```text
┌─ Verify #480 · docs(roadmap): workspace-native PR verification ─┐
│ Isolation       ▸ Against my real config   ( Clean room )        │
│ Companion role  ▸ None                     ( pick role… )        │
│ Build cache     ▸ Shared                   ( Fresh )             │
├──────────────────────────────────────────────────────────────────┤
│ enter start verify · ←/→ change · q back                          │
└──────────────────────────────────────────────────────────────────┘
```

### Step 4 — Verify

jackin' checks the PR out, reads what it changed, and opens the verification surface: a full-screen view with a PR context header, a task list that puts the commands this PR's changes call for **first**, a pane showing the output of whatever is running, and a footer of available keys. Tasks the diff did not touch are still listed (never silently hidden), just not pre-selected. Run-state badges show idle, running, passed, or failed. The operator runs the focused task, every relevant task in order, or all tasks; refreshes the PR to its latest pushed commit; opens any served pages (below); and quits — at which point a pass/fail summary of what ran is shown.

```text
┌──────────────────────────────────────────────────────────────┐
│ Verify · #480 docs(roadmap): workspace-native PR verification
│ base main ← head feature/… · OPEN · against my real config
├──────────────────────────────┬───────────────────────────────
│ Relevant to this PR (docs)   │ docs-build · bun run build
│  ▸ docs-build  [build] ● ok  │ $ bun run build
│    docs        serve   ○     │ ✓ 142 pages built in 3.1s
│                              │ ✓ link check ok
│ Other tasks                  │ …
│    cli         [build] ○     │
│    console             ○     │
│    test                ○     │
├──────────────────────────────┴───────────────────────────────
│ enter run · a all-relevant · A all · r refresh · o open · q quit
└──────────────────────────────────────────────────────────────┘
```

Output renders inside the pane (badges: `○` idle, `◐` running, `●` passed, `✗` failed), never as raw scrolling text over the surface. Because the surface is rich, debug mode does not paint over it: the full firehose of every command and its output is captured to a diagnostics run file, per [Debug output never reaches a rich full-screen TUI](/reference/tui-design-decisions/#debug-output-never-reaches-a-rich-full-screen-tui). To triage a failed task, the operator shares the run id rather than pasting scrollback.

## Setting up verification: the repo recipe

What jackin' can build, run, and serve for a repo is declared in a verification recipe committed at the **root of the project repo**, so it versions with the code. There is **one task shape**; optional fields select behaviour rather than forking into separate kinds:

- mark a task as part of the build set (run in order, stop on first failure);
- give it path globs, so jackin' can tell whether a PR's changes make it relevant;
- mark it as a long-running server that opens the PR's changed docs pages (below).

The surface's actions are views over this one list: "build" runs every build-set task; "run" runs any task by name; trailing arguments are appended verbatim.

```toml
# jackin.tasks.toml — verification recipe for this repo
version = 1

[[task]]
name = "cli"
run = "cargo build --bin jackin"
build = true                         # part of the build set
paths = ["src/**", "Cargo.*"]        # relevance: lit up by code changes

[[task]]
name = "docs-build"
run = "bun run build"
cwd = "docs"                         # relative to the checkout root
build = true
paths = ["docs/**"]

[[task]]
name = "console"
run = "jackin console"               # resolves to the freshly built binary, not the installed one

[[task]]
name = "test"
run = "cargo nextest run"

[[task]]
name = "docs"
run = "bun run dev"                  # long-running; kept up until the operator quits
cwd = "docs"
# marking a task as a server tells jackin' to wait for it and open the PR's changed pages:
serve = { ready = "http://localhost:3000/", route_from = { dir = "docs/content/docs", strip_suffix = ".mdx", index = "index", base = "/" } }
```

**Diff-driven relevance.** jackin' reads the PR's changed file list once and matches it against each task's path globs. The matching tasks are the PR's relevant set — surfaced first and run by "all relevant". Globs choose *which task* is relevant (the code build vs the docs build vs the smoke), not which source files a build recompiles — the compiler and the docs bundler already detect file-level changes inside a task. A manifest or lockfile edit lights up every code task; a docs edit lights up the docs build and the server. Non-matching tasks are shown but not auto-selected; "all" runs everything; an empty or unknown change set falls back to treating everything as relevant. Nothing is silently skipped — what was de-selected, and why, is logged.

When jackin' verifies itself, a task that launches jackin' (a console smoke, a load) must run the **freshly built** binary, not the operator's installed one; the runner arranges that, and the launched binary's own runs are sandboxed by the chosen isolation profile.

This recipe is self-contained — each task carries its own command, so a repo is verifiable without any other tooling. It is related to other task-shaped surfaces on the roadmap (operator hotkeys, a future task-source abstraction) but deliberately stays its own thing: a maintainer verification recipe committed in the repo under test, with a different audience and different fields from an operator's in-console hotkeys. A shared resolver is worth revisiting only once [task source abstraction](/reference/roadmap/task-source-abstraction/) defines a concrete shape. It is not a versioned jackin' schema: it lives in the repo under test, jackin' is pre-release, and a breaking shape change simply fails to parse.

## Isolation: verifying jackin' without risking your setup

jackin' keeps two things separate: its **configuration** (workspaces, mounts, auth, trust) and its **state** (running instances, caches, role checkouts, agent homes). A verification run can redirect both to throwaway locations, chosen on the setup screen:

- **Against my real config** (default) — verify against the operator's *real* configuration, with *no* state. jackin' takes a **copy** of the real configuration as read-only input and starts with empty state. Every write the built binary makes — schema migrations, "last role", trust-on-first-use, new instance state — lands in the throwaway copy and is discarded when the verification's disk is reclaimed. The real configuration is never written; the real state is never read.
- **Clean room** — both configuration and state start empty. jackin' from zero, for first-run behaviour and defaults.

The configuration is always **copied, never referenced**: a PR that bumps a configuration or workspace schema migrates files in place on load, and pointing a verification run at the live configuration would let it rewrite the operator's real workspace files to the PR's schema version. The copy is what makes a schema-bumping PR safe to verify.

## Companion role pinning

A PR that pairs a CLI change with a role change needs jackin's inner launches to use a specific role at a specific branch. The companion-role choice on the setup screen records that pairing: any task that launches jackin' is invoked with the chosen role and, when a branch is given, that role checked out at that branch. With no companion role chosen, tasks use whatever the isolated configuration would pick by default.

## Documentation preview

Some PRs change documentation, and the review need is "spin up the changed docs locally and open the pages this PR touched" — the PR template's documentation walk, automated. A task marked as a server is run like any other, but jackin' treats it specially: it starts the server, waits until it responds, then derives the URLs for the pages this PR changed (from the same changed-file list used for relevance) and opens each one. If none match, it opens the site root. The server stays up until the operator quits, then is shut down cleanly. The server command, port, and URL mapping are declared in the recipe, never hardcoded in jackin' — so a change of docs engine is a one-line edit to the recipe, not a jackin' code change. (This site's own move between docs frameworks changed exactly those values — the dev-server command and port — which is what the recipe absorbs.)

## What jackin' keeps per pull request

For each PR it verifies, jackin' keeps a self-contained, throwaway bundle, isolated from the operator's working repos and real setup:

- a checkout of the PR at its head commit;
- the isolated configuration (a copy of the real one, or empty) and the empty state home for the run;
- enough metadata (number, title, branch, base, head commit, last-refresh time, state, chosen isolation) to render the picker's cached/staleness badges without re-hitting the network.

Refreshing re-fetches the PR head and resets the checkout — the right response to a force-pushed branch. Reclaiming (the `x` key in the picker) drops a PR's bundle and reports the disk freed. A single PR cannot be verified twice at once. The exact on-disk location is intentionally left to the implementation, since how jackin' stores console state is being reworked; the guarantee to preserve is that the bundle is isolated, reclaimable, and never mixed with the operator's real config, state, or repos.

## Host-side guarantees

Per jackin's rule that it must never mutate the host machine silently, verification is operator-invoked, and its writes stay confined:

- it writes only inside its own throwaway verification bundles and the role checkout cache — never the operator's working repos, never the real configuration or state;
- the real configuration is copied, never written, so even a schema-bumping PR cannot migrate the operator's live workspace files;
- reading the repo's pull requests, the PR's diff, and the host's GitHub auth are all read-only — verification never rewrites host git config or auth;
- the saved workspace is never modified.

## Risks, edge cases, failure modes

- **GitHub CLI missing or unauthenticated** — checked before listing; the picker shows a one-line install/login hint instead of an empty list.
- **PR list slow or failing** — the picker shows a loading state, then an error row with the underlying reason; it never blocks the workspace screen.
- **Many open PRs** — the picker is scrollable and filter-as-you-type, newest activity first.
- **Resolving the repo from the workspace** — jackin' reads the primary working directory's remote; a directory with no resolvable GitHub repo disables the action with a one-line reason, and SSH-vs-HTTPS remote forms are normalized.
- **Which binary a task runs** — when verifying jackin' itself, tasks must run the freshly built binary, or a console smoke silently tests the wrong one.
- **Build cache** — shared is fast but can in rare cases bleed state between PRs; the *Fresh* setup option forces a cold build, and which was used is logged.
- **Fork, force-push, closed/merged** — checkout uses the PR head ref directly; refresh resets hard; a closed/merged PR is flagged but still openable for post-merge inspection.
- **No recipe in the repo** — the surface opens with a one-line "no recipe in this repo" notice and no runnable tasks.
- **Server lifecycle** — a server task is a child that must be torn down on quit, with its whole process group signalled so workers don't orphan, and a readiness timeout so a server that never binds fails loudly instead of hanging.
- **Doneness** — the surface shows a pass/fail summary on quit and badges a failed task `✗`. A server task is interactive, not pass/fail, so it does not count toward the verdict.

## Deferred

Explicitly out of the first slices, recorded so the design stays coherent when they land:

- **PRs on the workspace's other mounts.** The first slices list only the primary repo's PRs. A workspace mounts several repos; surfacing each mount's PRs (grouped by repo, or via a repo sub-picker) is the natural next slice.
- **Scripted / non-interactive entry.** The console is the only entry today, so CI and scripts cannot drive a verify. If that need arises, a thin non-interactive path over the same checkout-and-run core can be added without reshaping it.
- **Branches as a standalone target.** PRs only for now.
- **Role-repo PRs as a standalone target.** Today roles enter only as the companion pin; verifying a role repo's own PR end-to-end is the same machinery and can be added later.
- **Merge preview** — verify the post-merge-into-base state rather than the PR head.
- **Coordinated multi-repo verification** — one verify spanning several of the workspace's repos (the PR template's "Related pull requests").
- **In-container agent review** — launch the workspace container against the PR checkout and let an agent review in the multi-project context. The host-side model covers the daily "does this PR build and run" need; agent-assisted review is additive.

## Relationship to existing roadmap items

- [Console agent session control](/reference/roadmap/console-agent-session-control/) — "Verify a pull request" is a workspace-level console action; unlike the per-session actions, it needs no running session.
- [Custom operator tools](/reference/roadmap/custom-operator-tools/) and [task source abstraction](/reference/roadmap/task-source-abstraction/) — related task surfaces, but the verification recipe stays a distinct repo-committed thing; a shared resolver is worth revisiting only once task source abstraction defines a concrete shape.

## Prior art

No existing tool covers the host-side-verify + companion-role-branch + isolated-jackin'-state combination, driven from the workspace; the closest pieces would have to be glued by hand:

- [`gh pr checkout`](https://cli.github.com/manual/gh_pr_checkout) — checkout by number; no build/run recipe, no isolation.
- [`gh-worktree`](https://github.com/eikster-dk/gh-worktree) — checks a PR out into a worktree; worktree only.
- [HighReview](https://github.com/HighGarden-Studio/HighReview) — local PR review via worktree with IDE/AI analysis; diff-review focus, no configurable build/run tasks.
- [`gh-review`](https://crates.io/crates/gh-review) — terminal diff/comment/approve UI; review surface, not build/verify.
- Nx / Turborepo / Bazel / Buck2 — affected-target selective rebuild; prior art for diff-driven relevance, not PR-scoped review.
- `just` / `mise` — repo-committed task runners; tasks but no PR checkout or isolation.

The novel combination: host-side PR verification reached from the console and driven by the workspace, with companion-role-branch pinning and an isolated jackin' configuration and state home so jackin' can verify itself safely.

## Phased roadmap

- **Phase 0 — Recipe + checkout bundle.** Read the repo recipe. Check a PR out into an isolated, reclaimable bundle with metadata; support refresh and a single-verify-at-a-time guard.
- **Phase 1 — Console entry + PR picker + run-all baseline.** The "Verify a pull request" action on a workspace; resolve its primary repo; list open PRs in a picker; on select, check out and run the whole build set host-side against the freshly built binary, with a pass/fail summary.
- **Phase 2 — In-surface verbs + management.** Build / run-named / refresh actions in the verification surface; the picker doubles as the cached-verifies list with a reclaim key.
- **Phase 3 — Relevance + rich surface (the diff-driven spine).** Map the PR's changed files to the relevant tasks; the rich surface that lists relevant-first and lets the operator pick and run with output in a pane; suggestion-not-skip; de-selection logging.
- **Phase 4 — Documentation preview.** The server task: start it, wait for ready, derive and open the PR's changed pages, tear down cleanly on quit.
- **Phase 5 — Isolation profiles.** The setup choice "against my real config" vs "clean room".
- **Phase 6 — Companion role pinning.** The setup choice of a role and optional branch, threaded into inner jackin' launches.
- **Later — the items under "Deferred".**
