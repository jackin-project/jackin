title: "Supply-chain and dependency-hygiene CI"

**Status**: Open ŌĆö design proposal

## Problem

The workspace resolves **447 packages** in <RepoFile path="Cargo.lock" /> (85 direct dependencies in the root <RepoFile path="Cargo.toml" /> plus the four workspace crates). None of that tree is checked in CI today:

- No advisory scan ŌĆö a published `RUSTSEC` vulnerability in any of the 447 packages (the `bollard` + `rustls` networking stack alone pulls a deep tree) merges silently.
- No license gate ŌĆö nothing stops a future dependency bump from pulling a `GPL`/`AGPL` crate into a project that distributes binaries.
- No unused-dependency check ŌĆö 85 direct entries drift stale as features churn pre-release; removed code leaves its manifest entry behind.
- No duplicate-version or source check ŌĆö multiple semver-incompatible copies of the same crate, or a dependency silently switched to a git source, go unnoticed.

jackin's whole pitch is *"keep the agent from doing dangerous things to your host."* A tool that markets a security posture but has **no demonstrable supply-chain story for its own dependencies** carries a credibility gap. The one git-sourced dependency already in the tree ŌĆö `vt100` pinned to [`donbeave/vt100-rust`](https://github.com/donbeave/vt100-rust) (tracked under [Terminal emulation crate (`jackin-term`)](/reference/roadmap/terminal-emulation-crate/)) ŌĆö is exactly the kind of non-crates.io source a `sources` policy should make explicit rather than invisible.

This is the missing automated reviewer for the dependency tree. Per the solo-maintainer staffing model in <RepoFile path="AGENTS.md" />, there is no second human to notice a `RUSTSEC` advisory or a copyleft license sneaking in through a transitive bump. CI advisory + license + bans checks are the substitute, exactly as multi-agent review substitutes for the missing second reviewer elsewhere.

## License obligation ŌĆö verified

The operator's concern that copyleft licenses can force jackin' to adopt their terms is **correct** for the licenses that matter here:

- **`GPL` (and `AGPL`)** are strong copyleft. Rust statically links its dependencies by default, so a binary that links a GPL crate is a *combined / derivative work*. jackin' distributes binaries (the [`homebrew-tap`](https://github.com/jackin-project/homebrew-tap), GitHub release artifacts, the `jackin-role` validator). Distributing a binary that statically links GPL code obligates the **entire binary** to ship under GPL-compatible terms ŌĆö incompatible with the project's `Apache-2.0` license.
- **`AGPL`** adds a network-use clause: merely *operating* the software as a network service triggers the source-disclosure obligation, even without distributing a binary. Strictly worse for a tool that may grow daemon/remote surfaces.
- **`SSPL`, `BUSL`/`BSL` (non-permissive variants), `EUPL`, `CC-BY-SA`** are in the same "avoid for a permissively-licensed distributed binary" bucket and should be denied by default.

### Current tree is already clean

A `cargo metadata` license audit of the current 447-package tree (June 2026) found **zero `GPL` and zero `AGPL` dependencies**. There is **nothing to clean today** ŌĆö the ban is a *preventive guard against future bumps*, not a cleanup. Three edge cases were reviewed and are not problems:

| Crate | Pulled via | License | Verdict |
| --- | --- | --- | --- |
| `option-ext` | `dirs-sys` | `MPL-2.0` | **Allow.** File-level weak copyleft ŌĆö covers only the crate's own files, does not infect the larger work. Standard to permit. |
| `r-efi` (5.x, 6.x) | `getrandom` | `MIT OR Apache-2.0 OR LGPL-2.1-or-later` | **Allow.** An `OR` expression ŌĆö the project selects `MIT`/`Apache-2.0`, so the `LGPL` arm never applies. |
| (one crate) | ŌĆö | `WTFPL` | **Decide.** Permissive in intent but not OSI-clear; either allow explicitly or replace. Low stakes. |

The license check therefore lands green on the current tree with a small, documented allow-list ŌĆö no dependency removal required before it can be enabled.

## Proposed approach

Adopt **two tools**, both installed through `mise` (per the mise-only CI rule in <RepoFile path=".github/AGENTS.md" />), wired into <RepoFile path=".github/workflows/ci.yml" /> behind the existing `rust` paths-filter and folded into the `ci-required` aggregator:

### 1. `cargo-deny` ŌĆö advisories, licenses, bans, sources

`cargo-deny` is a superset of `cargo-audit`: its `advisories` check *is* the `RUSTSEC` database scan that `cargo-audit` performs, plus three more axes. Adopting `cargo-deny` alone ŌĆö **not** both ŌĆö avoids running the advisory scan twice and maintaining two tools.

A first `deny.toml`, written against the **current cargo-deny config schema** (verified against the upstream docs in June 2026 ŌĆö see [Upstream verification](#upstream-verification) below). Note there is no longer a `version = 2` marker in any section: the `version` field is deprecated and ignored, and the modern template documents each field by its default instead.

```toml
[advisories]
# Pull the RustSec advisory DB. As of current cargo-deny, every vulnerability,
# unsound, and notice advisory errors by default ŌĆö the old `vulnerability` /
# `unsound` / `notice` / `severity-threshold` lint-level fields were removed.
# Suppress a specific advisory only here, with a written justification:
# ignore = ["RUSTSEC-XXXX-XXXX"]
yanked = "deny"                 # default is "warn"; tighten to fail on yanked crates
unmaintained = "workspace"      # default "all"; scope to first-party + direct deps

[licenses]
# Closed allow-list, mirroring rust-lang/cargo's own deny.toml plus the extra
# permissive identifiers present in jackin's tree. Any license NOT listed is
# denied ŌĆö GPL/AGPL/LGPL/SSPL/EUPL/CC-BY-SA fail by omission. The old
# `copyleft` / `allow-osi-fsf-free` / `deny` / `default` toggles were removed,
# so omission from `allow` is now the ONLY way to ban a license.
allow = [
  "MIT", "MIT-0", "Apache-2.0", "Apache-2.0 WITH LLVM-exception",
  "BSD-2-Clause", "BSD-3-Clause", "ISC", "Zlib", "0BSD",
  "Unicode-3.0", "Unicode-DFS-2016", "CDLA-Permissive-2.0",
  "CC0-1.0", "BSL-1.0", "Unlicense", "MPL-2.0",
]
confidence-threshold = 0.9
# Defaults worth knowing: include-dev = false (dev-deps are not license-checked,
# since they are not in distributed binaries) and include-build = true.

[[licenses.exceptions]]
# WTFPL: decide allow-or-replace; pin the one crate that carries it here
# rather than widening the global allow-list.
# crate = "<crate>"
# allow = ["WTFPL"]

[bans]
multiple-versions = "warn"   # upstream + cargo's own default; surface duplicates without blocking
wildcards = "deny"

[sources]
unknown-registry = "deny"
unknown-git = "deny"
[[sources.allow-git]]
# vt100 fork until jackin-term retires it (see terminal-emulation-crate).
# url = "https://github.com/donbeave/vt100-rust"
```

The `licenses.allow` list is a closed allow-list: any license **not** named ŌĆö every `GPL`/`AGPL`/`LGPL` variant, `SSPL`, `EUPL`, `CC-BY-SA`, etc. ŌĆö fails the build by omission. That is the operator's requested ban, expressed as the safer "allow-list, deny-by-default" shape rather than an open-ended deny list that misses new SPDX identifiers. This is now the *only* mechanism: cargo-deny removed the dedicated `copyleft` and `allow-osi-fsf-free` toggles, so the allow-list is the license policy.

Two upstream behaviours make the allow-list robust:

- **OR-expressions pass if any operand is allowed.** A crate licensed `MIT OR Apache-2.0 OR LGPL-2.1-or-later` (jackin's `r-efi`, via `getrandom`) passes because `MIT`/`Apache-2.0` satisfy the expression ŌĆö the `LGPL` arm never triggers. The check fails only when *no* allowed license can satisfy the expression.
- **GNU licenses match pedantically (since cargo-deny 0.18.4).** Allowing `GPL-3.0-or-later` would *not* permit `GPL-3.0-only` ŌĆö each variant is exact-match. Irrelevant while banning all GNU licenses, but it means the ban cannot be accidentally widened by a loose identifier.

The `sources.allow-git` entry is what makes the `vt100` fork an explicit, reviewable decision instead of a silent exception; it is removed when `jackin-term` retires the fork.

### 2. `cargo-machete` ŌĆö unused direct dependencies

`cargo-machete` parses each crate manifest against actual source usage. It runs on the **stable** toolchain, needs no build, and finishes in well under a second ŌĆö cheap enough to run on **every** PR rather than "periodically" (a manual cadence a solo maintainer forgets). Macro-only / `cfg`-gated false positives are suppressed per-crate with `[package.metadata.cargo-machete] ignored = [...]`.

### Alternatives considered and not adopted

These verdicts are reasoned from ecosystem norms and the verified cargo-deny facts; unlike the cargo-deny config details, they were **not** independently triple-verified in the June 2026 research pass (see [Upstream verification](#upstream-verification)). Re-confirm tool maintenance status at implementation time.

| Tool | Decision | Reason |
| --- | --- | --- |
| `cargo-audit` | **Skip** | Its RustSec scan is the same DB cargo-deny's `advisories` check pulls (verified: same `RustSec/advisory-db`, same `P90D` staleness). Running both duplicates the scan and doubles maintenance. cargo-audit remains usable standalone, but for CI cargo-deny is the single gate. |
| `cargo-udeps` | **Skip** | Requires the **nightly** toolchain and a full build ŌĆö friction against the `rust-toolchain.toml` stable pin, slow in CI. `cargo-machete` covers the common case (unused *direct* deps) on stable for near-zero cost. Revisit only if precise compile-graph analysis is ever needed. |
| `cargo-vet` / `cargo-crev` | **Defer** | Human-audit-trail tooling whose value scales with reviewer count, or with importing an external trusted-audit set (Mozilla/Google). A solo maintainer cannot sustain a meaningful per-crate audit ledger, and importing audits without local re-review adds little; revisit if the project gains contributors. |
| `cargo-geiger` | **Defer** | `unsafe`-usage census is informative but not a pass/fail gate; produces a metric, not a policy. Out of scope for a CI gate. |
| `OpenSSF Scorecard` / GitHub `dependency-review-action` | **Defer** | Repo-health and PR-dependency-diff signals that overlap cargo-deny's advisory/license coverage for a Rust project; reconsider if the project later wants a published security-posture score rather than a pass/fail gate. |

## CI wiring

There are two integration paths, and they pull in different directions:

- **Upstream-canonical: `EmbarkStudios/cargo-deny-action@v2`.** This is the documented canonical CI integration ŌĆö a single step after `actions/checkout`, with the binary bundled by the action (the `@v2` major line currently ships cargo-deny 0.19.x). Lowest friction, but it installs a tool *outside* `mise`.
- **Project-consistent: install via `mise`, run the binary directly.** jackin's mise-only CI tool-install rule in <RepoFile path=".github/AGENTS.md" /> requires every CI tool to come from <RepoFile path="mise.toml" />, exactly like the existing `cargo:cargo-nextest` and `cargo:cargo-zigbuild` entries.

**Recommendation: the mise path**, to honour the project's hard rule and keep one tool-install mechanism. The `cargo-deny-action` is noted here as the upstream-canonical alternative, explicitly *not* adopted because it bypasses `mise` ŌĆö that trade-off is the kind of deviation the rule exists to prevent. A new `deny` job in <RepoFile path=".github/workflows/ci.yml" />, mirroring the existing `check` / `msrv` jobs:

```yaml
  deny:
    needs: changes
    if: needs.changes.outputs.rust == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@... # pinned SHA, matching sibling jobs
      - uses: jdx/mise-action@...  # installs cargo-deny + cargo-machete via mise.toml
      - run: cargo deny check advisories licenses bans sources
      - run: cargo machete
```

Tool versions are pinned in <RepoFile path="mise.toml" /> alongside the existing `cargo:cargo-nextest` / `cargo:cargo-zigbuild` entries:

```toml
"cargo:cargo-deny" = "<pinned>"      # e.g. the 0.19.x line current in mid-2026
"cargo:cargo-machete" = "<pinned>"
```

The `deny` job is added to the `ci-required` aggregator's `needs` list so branch protection keeps a single stable required check name.

### Advisory-freshness consideration

The `RUSTSEC` database updates independently of the repo, so a green PR can turn red later when a new advisory lands against an existing dependency ŌĆö with no code change. (cargo-deny fetches the same `RustSec/advisory-db` that `cargo-audit` uses, caches it under `$CARGO_HOME/advisory-dbs`, and treats a DB older than `maximum-db-staleness`, default `P90D` / 90 days, as an error ŌĆö so CI must refresh the DB rather than rely on a stale cache.) Two shapes resolve the green-turns-red problem:

- **Blocking on every PR + a scheduled run.** Keep `advisories` in the per-PR gate (catches newly-introduced vulnerable deps immediately) **and** add a daily/weekly `schedule:`-triggered run so an advisory against the existing tree surfaces as its own failing run rather than blocking an unrelated PR author. This is the recommended shape.
- **License + bans + sources block per-PR; advisories scheduled-only.** Lower friction, but a PR that introduces a known-vulnerable dependency would not be caught until the next scheduled run. Weaker; not recommended.

## Host-side effects

**None.** All work is CI configuration plus a new `deny.toml` and pinned tool entries in <RepoFile path="mise.toml" />. No host machine state, no operator config, no container changes. The tools run only in GitHub Actions and on a contributor's own checkout when they choose to run them.

## Steps

1. Add `deny.toml` at the repo root with the allow-list, bans, and sources config above; pin the `vt100` git source explicitly.
2. Resolve the `WTFPL` crate: add a scoped `licenses.exceptions` entry, or replace the dependency.
3. Add `cargo:cargo-deny` and `cargo:cargo-machete` to <RepoFile path="mise.toml" /> with pinned versions.
4. Add `[package.metadata.cargo-machete] ignored = [...]` entries where macro-only deps trip false positives; run `cargo machete` locally to find them.
5. Add the `deny` job to <RepoFile path=".github/workflows/ci.yml" /> behind the `rust` paths-filter and add it to the `ci-required` `needs` list.
6. Add the scheduled advisory run (`schedule:` trigger) so DB updates surface without blocking unrelated PRs.
7. Document the dependency-hygiene contract for contributors (which licenses are allowed, how to add a git source or an advisory ignore with justification) in the contributor docs.
8. Verify all checks pass green on the current tree before merging ŌĆö the audit above predicts they do, modulo the one `WTFPL` decision.

## Caveats

- `cargo-deny`'s `advisories` check needs network access in CI to fetch the `RUSTSEC` DB; the job must not run with networking disabled.
- The `licenses.allow` list must be re-checked whenever a dependency bump introduces a new SPDX identifier ŌĆö that is the intended friction (a new license is a decision, not a default).
- `multiple-versions = "warn"` starts non-blocking; tighten to `"deny"` (with a `[[bans.skip]]` list for the known-duplicate crates) only after the existing duplicate-version backlog is cleared, to avoid blocking unrelated PRs on pre-existing duplication.

## Upstream verification

The cargo-deny config schema, license model, and CI-integration claims in this item were verified against upstream sources in June 2026 (multi-source fan-out with adversarial claim verification). What was confirmed against primary sources:

- **Config schema dropped the `version` marker.** The `version` field in `[advisories]` / `[licenses]` is deprecated and ignored; the modern `deny.template.toml` documents fields by default value instead. ([cargo-deny advisories cfg](https://embarkstudios.github.io/cargo-deny/checks/advisories/cfg.html), [deny.template.toml](https://github.com/EmbarkStudios/cargo-deny/blob/main/deny.template.toml))
- **`[advisories]` lint-level fields removed.** `vulnerability` / `unsound` / `notice` / `severity-threshold` are gone ŌĆö all such advisories error by default; suppression is only via `ignore`. `yanked` defaults to `warn`; `unmaintained` is now an `all`/`workspace`/`transitive`/`none` enum (default `all`). ([cargo-deny advisories cfg](https://embarkstudios.github.io/cargo-deny/checks/advisories/cfg.html))
- **`[licenses]` consolidated on allow-list-only.** `copyleft` / `allow-osi-fsf-free` / `deny` / `default` / `unlicensed` removed; ban-by-omission is the only mechanism. `include-dev = false`, `include-build = true`. OR-expressions pass if any operand is allowed; GNU licenses match pedantically since 0.18.4. ([cargo-deny licenses cfg](https://embarkstudios.github.io/cargo-deny/checks/licenses/cfg.html))
- **Allow-list mirrors `rust-lang/cargo`'s own `deny.toml`** ŌĆö MIT, MIT-0, Apache-2.0, BSD-2/3-Clause, MPL-2.0, Unicode-3.0, CC0-1.0, ISC, Zlib ŌĆö with `Unicode-3.0` the current identifier replacing `Unicode-DFS-2016`. MPL-2.0 is file-level weak copyleft and safe to allow ([Mozilla MPL 2.0 FAQ](https://www.mozilla.org/en-US/MPL/2.0/FAQ/)). ([rust-lang/cargo deny.toml](https://github.com/rust-lang/cargo/blob/master/deny.toml))
- **`EmbarkStudios/cargo-deny-action@v2` is the upstream-canonical CI integration**, and `multiple-versions = "warn"` is the upstream + cargo default. ([cargo-deny-action](https://github.com/EmbarkStudios/cargo-deny-action))

What was **not** independently verified (reasoned synthesis only ŌĆö re-confirm at implementation): standalone cargo-audit's 2026 maintenance status and RustSec's official CI recommendation; cargo-machete vs cargo-udeps maintenance and false-positive rates; cargo-vet/cargo-crev practicality for a solo maintainer; and whether OpenSSF Scorecard / `dependency-review-action` / cargo-geiger / cargo-acl add value alongside cargo-deny. Tool versions are time-sensitive ŌĆö pin to a current release at implementation time and track the `@v2` major line if the action is ever chosen over the mise path.
