---
title: "ADR-011: Native macOS agent-usage menu bar (Rust truth, Swift shell)"
description: "Host menu-bar usage app reuses jackin-usage probes via UniFFI; Swift is display-only; Agent::ALL + existing UsageSurface only."
---

**Status**: Accepted

The macOS agent-usage menu bar is a **host process** that shows account quota / spend windows for jackin❯-supported agents without requiring an open Capsule session. **Rust owns all probing, cache, cooldown, severity, and view shaping** (`jackin-usage` + `jackin-protocol` contracts). **Swift owns only display and OS integration** (menu bar, popover, Settings, a11y, signing). The UniFFI facade is TableRock-shaped: synchronous coarse commands over a staticlib → XCFramework, with Swift polling immutable snapshots.

## Why this is recorded

CodexBar proves demand for a menu-bar usage product, but a pure-Swift probe stack would diverge from Capsule truth and double maintenance. TableRock already established the correct split for tailrocks native apps. This ADR freezes that split for jackin❯ and the hard agent/surface scope so contributors do not reintroduce provider sprawl or second probe matrices.

## Agent and surface scope (hard)

Only `jackin_core::Agent::ALL`: `claude`, `codex`, `amp`, `kimi`, `opencode`, `grok`.

Usage surfaces: existing `jackin_usage` surfaces — Claude, Codex, Amp, Grok, OpenCode, Kimi, plus **Z.AI / GLM** and **MiniMax** when modeled for routed providers. Never Cursor, Gemini, Copilot, or CodexBar’s provider zoo.

## Public probe entry points (`jackin-usage`)

| Entry | Role |
|---|---|
| `usage::{claude,codex,amp,grok,kimi,minimax,zai}_snapshot` (+ OpenCode) | Provider probes (crate-internal; host runtime calls via refresh) |
| `UsageCache::refresh_active_account_snapshots` | Parallel probes, locks, cooldown, shared snapshot write |
| `UsageCache::focused_snapshot` / `focused_status_bar_label` | Cached views + honest `refreshing` / `unavailable` |
| `UsageCache::request_account_refresh` | Force-due one surface |
| `usage::host::HostUsageRuntime` | Capsule-free host orchestration for menu bar + CLI |
| `run_claude_usage_diagnostic` | Existing Claude diagnostic helper |

Presentation never invents percentages: only render `FocusedUsageView` / bucket fields; `unavailable` / `refreshing` / `stale` stay honest.

## Architecture

```text
Swift `LSUIElement` menu bar (display only)
  → @MainActor presentation from immutable DTOs
  → UniFFI jackin-usage-ffi (staticlib / XCFramework)
  → HostUsageRuntime + jackin-usage probes (Rust)
```

Swift must not own HTTP/RPC/OAuth/CLI scrapes or a second provider matrix.

The glance popover is a display-only SwiftUI composition — a tab grid over the Rust glance rows, an Overview tab (one row per provider), a provider tab (header + account chips + a generic `displaySegments` bucket path), and a Refresh-only footer — hosted in the status item's transient `NSPopover`. Every provider, order, label, number, and segment comes from Rust; the `onOpenUsage` header seam is bound by plan 007.

Window entry paths are AppKit-only (plan 007). A pure, testable menu contract lives in the bridge target: `StatusItemMenuModel` fixes the static three-row context menu (Open Usage Window · Refresh · Quit) and `StatusItemMenuRouter` maps each row — plus a provider-header click — to injected host handlers with no AppKit dependency. `StatusBarController` installs one `NSStatusBar` button action per item sending on both mouse-up phases: a left-click toggles the transient popover on that provider, a right-click pops the static `NSMenu` built by `StatusItemMenu`. The menu's Refresh row invokes the exact same forced `PresentationStore.refreshAll()` the popover footer calls; Open Usage Window opens/focuses the window on its current selection (never re-selecting a provider — the menu is identical for every item); Quit terminates. A provider-header click routes through the `onOpenUsage` seam to `UsageWindowController.show(focusOn:)`, which sets `PresentationStore.selectUsageSurface(_:)`, lazily builds and retains the single `NSWindow` hosting `UsageWindowRoot`, and dismisses the popover. The window stays an accessory (`LSUIElement`) surface with no Dock icon.

The Usage window is Capsule-parity by construction (plan 008). Rust owns one `usage_detail_presentation(&FocusedUsageView)` builder producing an ordered `UsageDetailPresentation` — fixed `focused`/`header`/`provider`/`account`/`status`/`updated` rows, optional `username`/`plan`/`auth`, one position-based `bucket:<i>` row per source bucket (so duplicate provider labels stay distinct), then an optional `detail` row for `last_error`. Each row carries a Rust-owned `label`, already-grouped `layout_lines` (`leading`/`trailing`), a `display_label` equal to the joined line fields, and geometry-only `meter_percent`/`severity`. Both the Capsule usage dialog and the Desktop Usage window consume that single model verbatim: Capsule maps each row to a `ContainerInfoRow`, and the window's `ProviderCardView` renders `content.detail.rows` and their lines mechanically — no Swift splits `paceLabel`, indexes `displaySegments`, joins strings, or invents field/fallback copy. The pure `UsageWindowModel` (in the bridge target) resolves the Rust sidebar order and the incoming `usageSelection` to the selected surface's detail rows and account rows, falling back to Overview for a disabled/invalid selection. This removes the prior dual presentation model where Capsule and Swift assembled provider detail independently, so source scans can no longer let the two renderers drift.

The status bar is one AppKit `NSStatusItem` per auto-detected provider, reconciled from the Rust-owned seven-provider glance list (`provider_glance_rows`): an `@main` AppKit bootstrap + `StatusBarController` create/update/remove items by Rust `surfaceId` in canonical order, show a single jackin❯ fallback item when none is detected, dim stale/error, and host one transient popover. Each item displays the Rust `barLabel` verbatim — no SwiftUI `MenuBarExtra`, severity tint, mini meter, or percent math in Swift.

Rust owns semantic status-bar glance slots via `StatusSlot` (`session`, `daily`, `weekly`, `spend`); `daily` carries Amp Free's daily allowance. Amp's API and CLI paths share one `displayText` parser producing that Daily glance bucket, with individual/workspace credit balances as detail-only quota bounds — never a glance percentage or inferred plan.

The status-bar pace line can carry a Rust-produced Variant A run-out composite (`<pace> · Runs out in <duration>`), emitted only when the linear projection from the window start runs out before the reset; the TUI and Swift splitters split on `" · "` unchanged.

Grok's plan label comes from server-resolved truth (`x.ai/billing.subscription_tier`), not a client `auth_mode` heuristic — a Free-tier browser login is no longer mislabeled SuperGrok. One current decoder reads the nested `config` (preferred `creditUsagePercent`/`currentPeriod`, fallback `monthlyLimit`/`used`/billing period), emits exactly one Weekly headline with pace when the window is derivable (RPC path only), and exposes prepaid balance + on-demand cap/used as N3-permitted quota bounds (never a price or trend).

Claude credential resolution is Rust-only and reads the macOS Keychain before any file/env credential. The generic-password service is derived from the effective `CLAUDE_CONFIG_DIR` by one shared `jackin-core` helper (`claude_keychain_scope`) reused by instance provisioning and the usage probe — the default `~/.claude` uses the bare `Claude Code-credentials` service; a custom config uses `Claude Code-credentials-<sha256(path)[..8]>`, and a custom scope never reads the default account. The Keychain payload is the same `claudeAiOauth` JSON the file carries, parsed by the one existing parser. Reads serialize; an explicit operator denial is terminal for that service for the process (no prompt storm) and is a typed **local-only** resolution — like a missing credential or an anonymous credential with no proven identity, it blocks cached-quota restoration, shared adoption/coordination, persisted snapshots, and account materialization, with zero effect on other providers. A *missing* Keychain item is never cached, so a new `claude /login` is picked up without an app restart. A headless `errSecInteractionNotAllowed` result is absence (file/env fallback stays available), not denial.

## Distribution

Immutable GitHub Release artifact `jackin-desktop-<VERSION>-aarch64-apple-darwin.zip` (exactly `JackinDesktop.app`), produced only after Developer ID codesign (hardened runtime + secure timestamp), notarytool `Accepted`, staple, and Gatekeeper assessment. Supply-chain sidecars match CLI archives: `.sha256`, cosign `.bundle`, CycloneDX `.sbom.json`, plus a separate GitHub build-provenance attestation. Homebrew cask `jackin-desktop` advances with the stable formula in one tap PR; the first cask is never auto-merged. Identity supersession (2026-07-22): product is **jackin❯ Desktop** (`Jackin Desktop` / `com.jackin-project.desktop`); the previous pre-release names `JackinUsageMenuBar` / `jackin-usage-menu-bar` never shipped as a notarized release. Uninstall leaves no root daemon. v1 does **not** require the jackin❯ daemon.

## Consequences

- Capsule TUI usage dialog remains; both consumers share Rust views.
- New agents land via `Agent` / `UsageSurface` growth in Rust, then appear in the menu bar automatically through `HostUsageRuntime` surface lists.
- Notarization secrets may be operator-gated residual; host library + UniFFI + app shell ship without them.
