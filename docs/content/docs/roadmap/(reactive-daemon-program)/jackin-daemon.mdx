**Status**: Spiked — lifecycle, install, control-socket, security, redaction, and host-vs-Capsule boundaries are resolved in the research page; a feature-gated `daemon-spike` prototype exists, but no production daemon command is shipped yet. The reactive features that depend on it ([live auth sync](/roadmap/live-auth-sync/), [host bridge](/roadmap/host-bridge/), [agent attention prompts](/roadmap/agent-attention-prompts/)) still wait on the production lifecycle/socket foundation.

## Shipped facts

jackin❯ is still a purely short-lived CLI / TUI: every command (`jackin console`, `jackin load`, `jackin workspace …`, `jackin config …`) runs to completion and exits. There is no host-side `jackin daemon` subcommand, binary, or long-running host process in the codebase today. The only existing "daemon" is the in-container `jackin-capsule` PID 1 process (see [jackin❯ Capsule](/roadmap/jackin-capsule/)), which is a separate per-container control plane, not the per-host daemon this item proposes; Capsule's own Phase 4 (host daemon integration) depends on this item landing first.

The clearest concrete stand-in for a daemon-owned watcher remains the keep-awake reconciler in <RepoFile path="crates/jackin-host/src/caffeinate.rs">crates/jackin-host/src/caffeinate.rs</RepoFile>: it still runs at every CLI command boundary, acquiring a lock file and starting/stopping `caffeinate -imsu` based on whether any keep-awake-labelled container is running, exactly as this item's problem statement describes.

Research: [jackin❯ daemon — design proposal](/reference/research/infrastructure/jackin-daemon-design/) has the full design rationale and the Plan 042 spike resolution: selected lifecycle, install, protocol, security posture, redaction rules, host-vs-Capsule relationship, and implementation phasing.

The spike prototype is feature-gated and not user-wired: <RepoFile path="crates/jackin-runtime/src/reactive_daemon.rs" /> defines the newline-delimited JSON control request shape plus the minimal attention adapter proof.

## Remaining work

1. **Production lifecycle foundation.** Ship `jackin daemon serve` / `install` / `uninstall` / `start` / `stop` / `restart` / `status` / `logs` with no watchers — lifecycle, install, control socket, version-skew checks, log redaction, coredump policy, and same-UID socket hardening only. See <RepoFile path="plans/047-daemon-lifecycle-foundation.md" />.
2. **Attention adapter foundation.** Promote the spike's one-way `attention.snapshot` adapter into a production adapter after the daemon lifecycle lands. See <RepoFile path="plans/048-attention-adapter-foundation.md" />.
3. **Desktop-ready API.** Add the small versioned endpoints [jackin❯ Desktop Agent Hub](/reference/research/desktop/agent-hub/jackin-desktop-agent-hub/) needs first: `daemon/hello`, workspace/session discovery, GitHub PR lookup, account status, event subscription, Ghostty/browser open actions.
4. **Caffeinate migration.** Move the keep-awake reconciler from per-command into a daemon-owned Docker-events watcher.
5. **Live auth sync.** Add the per-axis watcher adapters for [live bidirectional auth sync](/roadmap/live-auth-sync/) only after the daemon foundation has shipped and log/core-dump redaction is proven.
6. **Further reactive adapters.** Container event watcher, cross-session notifications, and [host bridge](/roadmap/host-bridge/) each plug into the daemon's adapter surface once it exists; host bridge remains blocked until per-request approval and audit are specified.
