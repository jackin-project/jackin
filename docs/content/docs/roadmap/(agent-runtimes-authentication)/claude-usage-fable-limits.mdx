---
title: "Claude usage: Fable and the limits array"
---

**Status**: Partially implemented — the Capsule usage overlay now parses the Anthropic `limits` array, surfacing the per-model weekly limit for **Fable** (and any future model-scoped window) on both the OAuth and CLI paths, with the legacy named windows kept as a fallback. The Session and "All models" Weekly rows bind to the existing headline slots; Fable renders as a non-headline bucket with API-reported severity. See the *Implementation* section for what shipped and the remaining macOS-authoritative-path note.

## Problem

The Capsule Claude usage overlay reads `https://api.anthropic.com/api/oauth/usage` and decodes a fixed set of named window keys (`five_hour`, `seven_day`, `seven_day_sonnet`, `seven_day_opus`, `seven_day_routines`). Anthropic has migrated the authoritative shape to a `limits` array. The model-scoped weekly limit for **Fable** (and Sonnet/Opus/Routines going forward) now appears only inside that array, so it is silently dropped.

The claude.ai settings → usage page renders three rows under *Weekly limits* — *All models*, *Fable* (model-scoped), and so on. jackin❯ today shows only the *All models* / Session windows and loses the per-model row the operator most wants to see.

## Verified evidence (live API, 2026-07-03)

Called the real OAuth endpoint with the locally-resolved Claude Code token. Top-level keys returned:

```text
amber_ladder, cinder_cove, extra_usage, five_hour, iguana_necktie, limits,
member_dashboard_available, nimbus_quill, omelette_promotional, seven_day,
seven_day_cowork, seven_day_oauth_apps, seven_day_omelette, seven_day_opus,
seven_day_sonnet, spend, tangelo
```

The `limits` array is the authoritative new shape and carries Fable:

```json
"limits": [
  { "kind": "session",       "group": "session", "percent": 7,  "severity": "normal",
    "resets_at": "2026-07-03T03:19:59+00:00", "scope": null, "is_active": false },
  { "kind": "weekly_all",    "group": "weekly",  "percent": 28, "severity": "normal",
    "resets_at": "2026-07-03T07:00:00+00:00", "scope": null, "is_active": false },
  { "kind": "weekly_scoped", "group": "weekly",  "percent": 35, "severity": "normal",
    "resets_at": "2026-07-03T06:59:59+00:00",
    "scope": { "model": { "id": null, "display_name": "Fable" }, "surface": null },
    "is_active": true }
]
```

Key facts the implementation must respect:

- `weekly_scoped` is the per-model window; its label comes from `scope.model.display_name` (today `Fable`). New model-scoped limits will appear here without a code change per model — surfacing them generically is the whole point.
- `weekly_all` is the "All models" row (replaces the legacy `seven_day` semantics).
- `session` replaces the legacy `five_hour` semantics.
- The legacy `seven_day_sonnet` / `seven_day_opus` / `seven_day_cowork` keys are still present but **null** — they are vestigial. The model data lives only in `limits`.
- `is_active` marks the window currently contributing; `severity` (`normal`/`warn`/`danger`) drives meter color and should map to [`UsageSeverity`](/reference/capsule/).
- The CLI fallback (`claude -p /usage`) also reports Fable: `Current week (Fable): 35% used · resets …`. The existing parser only matches `Current week (Sonnet only):`, so Fable is dropped on the CLI path too.

## Design

### 1. Parse the `limits` array (authoritative path)

Add a `limits: Option<Vec<ClaudeOAuthLimit>>` field to `ClaudeOAuthUsageResponse` in <RepoFile path="crates/jackin-usage/src/usage/claude.rs">crates/jackin-usage/src/usage/claude.rs</RepoFile>:

```rust
#[derive(Debug, Deserialize)]
pub(crate) struct ClaudeOAuthLimit {
    pub(crate) kind: Option<String>,        // "session" | "weekly_all" | "weekly_scoped"
    pub(crate) group: Option<String>,       // "session" | "weekly"
    pub(crate) percent: Option<u8>,
    pub(crate) severity: Option<String>,
    pub(crate) resets_at: Option<String>,
    pub(crate) scope: Option<ClaudeOAuthLimitScope>,
    pub(crate) is_active: Option<bool>,
}

#[derive(Debug, Deserialize)]
pub(crate) struct ClaudeOAuthLimitScope {
    pub(crate) model: Option<ClaudeOAuthLimitModel>,
    pub(crate) surface: Option<String>,
}

#[derive(Debug, Deserialize)]
pub(crate) struct ClaudeOAuthLimitModel {
    pub(crate) id: Option<String>,
    pub(crate) display_name: Option<String>,
}
```

Because the struct already carries `#[serde(flatten)] other_windows`, today the `limits` array lands in `other_windows` and is skipped by `push_claude_dollar_windows` (its entries do not match `ClaudeOAuthUsageWindow`). Naming the field pulls it out of the flatten.

### 2. `into_buckets` prefers `limits`, falls back to legacy windows

When `limits` is non-empty, build buckets from it and skip the legacy named-window pushes (they are null on current accounts anyway):

- `kind == "session"` → Session bucket, `StatusSlot::Session`.
- `kind == "weekly_all"` → "All models" weekly bucket, `StatusSlot::Weekly`.
- `kind == "weekly_scoped"` → bucket labelled by `scope.model.display_name` (e.g. "Fable"), no status slot (non-headline). Surface every active scoped window so future models appear without per-model code.
- Map `severity` → `UsageSeverity` via the existing `severity_from_label`.
- Keep the legacy `push_claude_window` calls as a fallback for accounts whose API still returns populated `seven_day_*` and no `limits`, so no regression on older shapes.
- `spend` / `extra_usage` and the `push_claude_dollar_windows` codename handling stay unchanged.

### 3. CLI fallback: parse model-scoped weekly lines

In <RepoFile path="crates/jackin-usage/src/usage.rs">crates/jackin-usage/src/usage.rs</RepoFile> `parse_claude_usage_output`, generalize the `Current week (Sonnet only):` branch to `Current week (<label>):` and collect each into a `(label, percent)` vec. `ClaudeCliUsage` gains a `scoped_weekly: Vec<(String, f64)>` field rendered as per-model buckets (marked `Estimated`, matching the existing degraded-confidence policy for the CLI path).

### 4. macOS Keychain is already covered by host auth-forward

An earlier draft flagged the macOS Keychain as a blocker for the authoritative path. That was wrong: the host auth-forward resolver (<RepoFile path="crates/jackin-instance/src/auth.rs">crates/jackin-instance/src/auth.rs</RepoFile> `read_host_credentials`) already reads the macOS Keychain (`security find-generic-password -s "Claude Code-credentials" -w`, plus per-`CLAUDE_CONFIG_DIR` service names) and writes the credential blob to the `/jackin/claude/credentials.json` handoff. The Capsule reads that handoff as the last candidate in `claude_oauth_candidates`, so a normal jackin❯ launch on macOS resolves the OAuth token and reaches the authoritative `limits` path without any Capsule-side Keychain read. No change needed here.

## What shipped

1. `limits` structs (`ClaudeOAuthLimit` / `…Scope` / `…Model`) + a `limits: Vec<ClaudeOAuthLimit>` field on `ClaudeOAuthUsageResponse` in <RepoFile path="crates/jackin-usage/src/usage/claude.rs">crates/jackin-usage/src/usage/claude.rs</RepoFile>. A single normalized `ClaudeQuotaWindow` model + one bucket builder (`into_bucket`) now serves every utilization source: the `limits` array (authoritative), the legacy named windows (`seven_day*`, fallback for pre-`limits` responses), and the `claude -p /usage` CLI fallback. Fable is not a special case — it is a `weekly_scoped` entry normalized like any legacy model window, so Session / "All models" / Fable / Sonnet all take one path (headline slots, over-cap labels, pace, severity mirrored). The source preference (`limits` when present, else legacy) only avoids double-counting, since the live API returns both carriers with identical data; it is not a parallel implementation.
2. CLI fallback in <RepoFile path="crates/jackin-usage/src/usage.rs">crates/jackin-usage/src/usage.rs</RepoFile> `parse_claude_usage_output`: a generic `Current week (<label>):` branch feeds a new `ClaudeCliUsage::scoped_weekly: Vec<(String, f64)>`, so `Current week (Fable):` renders as a Fable bucket while the legacy `(Sonnet only)` line keeps its "Sonnet" label. CLI buckets reuse the same `ClaudeQuotaWindow::into_bucket` builder.
3. Tests in <RepoFile path="crates/jackin-usage/src/usage/tests.rs">crates/jackin-usage/src/usage/tests.rs</RepoFile>: the OAuth `limits` array surfacing Fable + "All models" with no legacy leakage, an unnamed-scoped-window omission guard, the CLI Fable line, and a direct equivalence test proving a legacy Sonnet window and a `weekly_scoped` Fable limit produce the same bucket through the one builder. All mirror the live 2026-07-03 response/output.

## Remaining

- Manual `--debug` validation inside a Capsule Claude session: confirm Session / All models / Fable buckets render with correct percents and the authoritative `provider` source tag (not `Estimated`) on an account that has a Fable limit.

## Out of scope

- Other top-level rotating-codename windows (`amber_ladder`, `cinder_cove`, …) — already handled generically by `push_claude_dollar_windows`; only acted on when they carry a `limit_dollars`.
- Non-Claude provider usage adapters (Codex/Amp/Grok/Kimi/MiniMax/Z.AI) — unaffected.
- Changing the refresh cadence or shared-snapshot cache — the new bucket set flows through the existing `UsageCache` unchanged.
