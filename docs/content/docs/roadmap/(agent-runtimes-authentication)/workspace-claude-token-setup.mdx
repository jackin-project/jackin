## What Shipped: `op_picker` Create Mode + Plain-Prompt `--interactive`

Two pieces shipped together, each on the surface where it belongs.

**Shared `op_picker` gained an opt-in Create mode.** <RepoFile path="crates/jackin-console/src/tui/op_picker.rs">crates/jackin-console/src/tui/op_picker.rs</RepoFile> now exposes `OpPickerMode::Create` alongside the default `Browse`. In Create mode the Item stage renders a `+ New item` sentinel and the Field stage renders `+ New field` and `+ New section` sentinels at the bottom of their lists (the standard `+ Add X` creation-sentinel convention), with naming sub-stages that reuse the shared `text_input` box. The picker commits an `OpPickerSelection` enum â€” `Existing(OpRef)` for an existing field, `NewItem { .. }` for a freshly named item, or `EditItemField { .. }` for a new field/section on an existing item. Browse mode is unchanged, so every existing caller keeps drilling Account â†’ Vault â†’ Item â†’ Field exactly as before. This replaced an earlier separate rich-dialog widget that had duplicated the drill-down state machine; the shared widget is now the single implementation.

**The CLI `--interactive` flow uses plain CLI prompts.** `jackin workspace claude-token setup <ws> --interactive` walks the operator through account â†’ vault â†’ item â†’ field using plain dialoguer Select/Input prompts â€” no rich ratatui TUI. At the item step the operator picks an existing item or `[ + New item ]`; at the field step they pick an existing field (token written in-place via `item_field_set`) or `[ + New field ]`. The flag is mutually exclusive with `--vault` and `--reuse`.

The principle this establishes: a rich ratatui TUI lives only inside `jackin console`; the CLI stays plain CLI. Scripted invocations keep using `--vault` / `--reuse`; `--interactive` is the plain-prompt path for an operator at a shell.

## What Shipped: Console Auth-tab Generate Action

The workspace editor's Auth tab can now mint a Claude token without leaving `jackin console`. On the workspace-level Claude `oauth_token` auth form, `G` opens a source step (Plain text / 1Password); the 1Password branch opens the shared `op_picker` in Create mode so the operator chooses (or creates, via `+ New item` / `+ New field` / `+ New section`) the location. Pressing `G` stashes the open auth form so the dialog can be re-mounted when the mint completes. On commit, the `run_console` loop suspends the TUI (leaves raw-mode + alt-screen, mirroring the terminal guard) and runs `token_setup::mint_token_value` â€” the interactive `claude setup-token` OAuth capture plus the 1Password create / read-back validation, but **no** config write â€” then resumes the TUI and re-mounts the Edit-auth dialog with the minted credential staged and focus on Save. The operator Saves to persist (the editor's normal save path), exactly like the *provide* flow where picking an existing value re-mounts the form on Save. Cancel leaves the config unwired (any op item already created is harmless). Errors surface through the standard `ErrorPopup`. The CLI keeps using `token_setup::run_setup` (mint **and** persist in one step) because it has no form to return to.

This is the console counterpart to the CLI `--interactive` storage picker. Role-level generate works the same way at the role layer: the operator opens that role's Claude auth form (via `+ Override for a role`) and presses `G` â€” the scope follows the auth-form target, so generate needs no separate role step. The CLI exposes the same scope via `jackin workspace claude-token setup <ws> --role <role>` (the interactive CLI prompts for the scope when `--role` is omitted).

Generate offers two storage targets via a source step (`G` â†’ Plain text / 1Password in the console; `--plain` flag or the interactive source prompt on the CLI): 1Password (mint â†’ create/edit an item â†’ wire the `op://` ref) or plain text (mint â†’ store the literal token in config at the chosen scope, no op item). The suspend/resume-around-PTY mint is exercised only by live use; it has no headless test.

## Remaining Work

1. **Canonical auth slot.** Replace the current two-key implementation (`auth_forward = "oauth_token"` plus managed `CLAUDE_CODE_OAUTH_TOKEN` env) with a dedicated `oauth_token` field on the Claude auth config. The launcher should synthesize `CLAUDE_CODE_OAUTH_TOKEN` from that slot.

2. **Pre-launch validity probe.** Add a lightweight validity check or cached doctor result so obviously invalid external rotations surface before Claude returns an API 401 inside the session. The design and implementation home for this probe is [Auth health and operator visibility](/roadmap/auth-health-and-visibility/); this item tracks the Claude-specific integration point.

3. **Apple Keychain backend.** Add a local OS-secret-store source after the cross-cutting [Credential Source Pattern](/roadmap/credential-source-pattern/) lands.

4. **Bulk migration.** Add a `--all` or equivalent workflow for migrating multiple workspaces to Claude OAuth-token mode.

## Design Notes

- The upstream `claude setup-token` browser consent remains the token minting step; jackinâťŻ owns capture, storage, config wiring, validation, and lifecycle commands.
- The canonical slot should be the future user-facing shape. The current managed env entry is an implementation bridge, not the target schema.
- Any future storage backend must preserve the current safety properties: no token in argv, no token in shell history, no config write before storage validation succeeds.
- The shared `op_picker` Create mode is the committed shape for in-place creation: the `+ New item` / `+ New field` / `+ New section` sentinels, their `text_input`-backed naming stages, and the `OpPickerSelection` commit enum (`Existing` / `NewItem` / `EditItemField`). The console Auth-tab generate action in Remaining Work item 1 reuses this widget rather than reintroducing a parallel one.
