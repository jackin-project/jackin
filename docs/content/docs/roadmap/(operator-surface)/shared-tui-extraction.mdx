# Shared TUI Extraction

**Status**: Open - research-backed cross-repository proposal

## Outcome

Extract the neutral component, terminal-runtime, lookbook CLI, generated SVG catalog, Fumadocs component documentation, and render-conformance foundation from `jackin-tui`, `jackin-tui-lookbook`, and the generic component docs into the independent Apache-2.0 `tailrocks/termrock` repository.

```text
Repository: tailrocks/termrock
Core crate: termrock
Lookbook:   termrock-lookbook
CLI:        termrock-lookbook
Docs:       same-repository Fumadocs catalog
```

The shared project targets Ratatui only, using Crossterm where terminal backend integration is required. It does not add a renderer-neutral abstraction or support alternative TUI frameworks. Components assume a modern truecolor terminal (Ghostty-class) with OSC 8/22/52 support; the library ships no capability-degradation paths.

The base crate should depend on `ratatui-core`, expose no default features, and keep Crossterm event/session integration behind one optional feature. The shared crate has no Tokio dependency; consumers own async execution, spawning, Tokio-backed subscriptions, panic/signal policy, diagnostics routing, and effects, while the executor-neutral update/view/subscription contracts and the `drive_frame` frame driver extract as one optional module.

Every decision in this plan is closed: the research chapters record decisions and execution steps, not open questions. The extraction ships only the existing donor component set and is bug-compatible — known rendering defects are recorded at freeze and fixed only after `jackin❯` parity passes, so `jackin❯`'s visual behavior never changes inside the ownership move.

Migrate `jackin❯` to the released project without visible behavior changes. Holla then becomes the second proving consumer. Future Velnor runner surfaces, justified Parallax interactive CLI workflows, and independent Tailrocks tools can consume the same neutral platform without importing `jackin❯` domain code.

The full donor audit, consumer boundaries, extraction classification, governance, and sequence are in [Shared TUI Extraction Research](/reference/research/shared-tui-extraction/).

## Why this belongs on the `jackin❯` roadmap

`jackin❯` is the donor and first compatibility consumer. The database workbench itself is not a `jackin❯` feature. The responsibility tracked here is limited to making the mature terminal system independently reusable and migrating the donor safely.

Current coupling prevents a direct publish:

- `jackin-tui` is `publish = false` and depends on `jackin-core`;
- neutral scrolling, dialog, ANSI, and bottom-chrome types are split or duplicated across the two crates;
- global palette constants mix semantic state, Tailrocks style, and product-specific colors;
- `BrandHeader`, container/agent helpers, product modal geometry, and animations embed `jackin❯` concepts;
- the lookbook harness is reusable, but many stories contain workspaces, roles, agents, containers, and product paths;
- known rendering defects (character-count width math in several components, a color-only panel focus cue) must be carried bug-compatible and fixed only after parity;
- the lookbook SVG drift check is not wired into CI, so the committed parity fixtures are not mechanically proven current.

## Design

### Repository surfaces

The external project owns four inseparable surfaces: the reusable Ratatui component crate, a lookbook library/CLI for interactive terminal preview and deterministic render/check operations, committed SVG component states, and a small Fumadocs catalog that documents every public component. Keeping them in one repository lets one PR change the API, story, preview, and documentation together. jackin❯ retains only product composition, interaction policy, and domain-specific conformance stories after migration.

### Git-first consumption

Initial adoption does not wait for crates.io. A consumer may point at a trusted feature branch or fork while developing a shared fix, but the branch is only an iteration pointer. Durable consumer changes pin the tested full commit SHA with Cargo's `rev` form and commit <RepoFile path="Cargo.lock" />; after upstream merge they repin to the canonical commit. This preserves the ability to unblock a consumer without making mutable branch heads the build or supply-chain boundary.

### Registry publication

An immutable Git revision is sufficient for the extraction preview and first migrations. Semver Git tags follow two-consumer proof; crates.io publication is a deliberately deferred, separate operator step. Neither blocks moving generic ownership out of jackin❯.

### Terminal baseline and parity sequencing

The shared library assumes a modern truecolor terminal (Ghostty-class): 24-bit color, alternate screen, SGR mouse, OSC 8 hyperlinks, OSC 22 pointer shapes, and OSC 52 clipboard. It ships no 256/16-color fallback, `NO_COLOR` handling, light-background adaptation, or RTL support; a consumer that needs degradation implements it above the library. Tiny and empty rectangles remain valid, panic-free inputs — terminal size is not a capability assumption.

Extraction is bug-compatible so jackin❯ never changes visually inside the ownership move. Known donor defects — character-count width math in several components and the color-only panel focus cue — are recorded in a freeze-time quality backlog, carried unchanged through extraction and migration, and fixed afterwards as reviewed `termrock` changes with regenerated fixtures that jackin❯ adopts through deliberate repins. Conformance gates split accordingly: an extraction tier required before the first consumer pin and a quality tier required before the first tag.

## Shared ownership

The external project owns:

- semantic themes and a default Tailrocks phosphor preset;
- display-width, terminal-text, key, focus, hover, scroll, layout, and ownership primitives;
- pure component/update/effect contracts that are product-neutral, plus the optional executor-neutral runtime module (update/view/subscription contracts and the frame driver);
- typed OSC hyperlink/pointer/clipboard requests with pure encoders (emission stays consumer-owned);
- panels, tabs, actions, dialogs, lists, inputs, hints, toasts, status surfaces, and diff presentation;
- terminal lifecycle guards where they can remain executor-neutral;
- interactive lookbook, deterministic renders, and conformance fixtures;
- terminal/list/render/check CLI behavior and a generated preview manifest;
- Fumadocs component API pages and committed neutral SVG previews;
- public API docs, semver, MSRV, Ratatui/Crossterm compatibility, and releases.

`jackin❯` retains:

- agent, role, workspace, container, Capsule, and host-specific types;
- OnePassword picker and all credential/product workflows;
- file browser, role/scope/mount editors, and other application components;
- brand composition, launch/closing animations, and product status policies;
- Tokio subscription implementations and spawn helpers (target home `jackin-console`), connected through the shared runtime module's closure adapter;
- product palette tokens (Capsule menu states, brand block, debug amber, rain colors) layered locally on the semantic theme;
- consumer stories and fixtures using `jackin❯` domain data.

## API rules

- no shared public type mentions agent, database, job, trace, connection, credential, or workspace;
- components perform no filesystem, network, process, secret, or telemetry I/O;
- components target Ratatui directly rather than a cross-renderer abstraction;
- components assume a modern truecolor terminal; no capability-degradation or detection code ships in the library;
- widgets never return raw escape bytes: hyperlink regions, pointer shapes, and clipboard writes are typed requests with pure encoders, and consumers own emission;
- consumers supply semantic theme, brand/header data, state, actions, and effects;
- widgets remain usable without Tokio; the shared crate exposes no async-runtime feature;
- a product-local component graduates only for a second concrete consumer or a clearly foundational primitive;
- every public component has keyboard, focus, non-color, Unicode, narrow-terminal, and lookbook evidence;
- consumers pin a release or immutable revision, never a moving default branch.

### API structure

The initial public modules are `text`, `input`, `interaction`, `layout`, `scroll`, `style`, `widgets`, `osc`, the optional executor-neutral `runtime` module, and optional `crossterm`. Rendering follows standard Ratatui `Widget for &T` / `StatefulWidget for &T` contracts with borrowed render data, stable consumer IDs, and state separated from domain records. Private fields/builders and non-exhaustive growing enums keep the pre-release API evolvable; broad root re-exports and compatibility facades are prohibited.

Every component and logical interaction API works without features. Enabling `crossterm` adds event conversion, backend construction, and an optional managed terminal-session builder; it cannot add alternate widgets, change component behavior, or become mandatory. Consumers may use only its event adapters, manage Crossterm manually, or use the convenience session while retaining ownership of their application loop, panic/signal policy, runtime, and effects.

TermRock is a toolkit rather than a mandatory application framework. It exposes no required root `Application`, `Model`, `Message`, `Component`, subscription, or effect-executor trait. Component-local logical actions and pure state updates return typed outcomes that a TEA-style or component-oriented consumer maps into its own application messages and effects. This follows Ratatui's documented application patterns without forcing jackin❯ runtime concepts into other consumers. The donor's executor-neutral contracts (dirty/update results, subscriptions with std-channel and closure adapters, component/view traits, and the `drive_frame` frame driver) extract as one optional module because jackin❯ mandates one shared frame driver across its console, Capsule, and launch surfaces and donor deletion would otherwise strand it; nothing requires the module, and Tokio-backed subscription helpers stay in jackin❯.

The external repository must include compile-checked direct, TEA, component-oriented, Flux, buffer-only, manual Crossterm, and managed Crossterm examples built from the same public widgets. The examples demonstrate integration and convenience patterns; none creates a privileged TermRock application runtime.

The current root palette, product wording, Tokio helpers, terminal-global atomics, and domain-specific modal/status/debug constructors are refactored before publication. The full contract and item-level ledger are in [Public API And Refactoring](/reference/research/shared-tui-extraction/06-public-api-and-refactoring/).
The file-by-file dispositions and replacement APIs are in [Component Redesign Catalog](/reference/research/shared-tui-extraction/09-component-redesign-catalog/).

### Repository engineering

The repository adopts jackin❯'s strict Rust lints, DCO, protected main branch, pinned tools/actions, committed <RepoFile path="Cargo.lock" />, Cargo dependency policy, REUSE/SPDX, Renovate, docs-in-the-same-PR, and aggregate required checks. It scales CI to a component library: default/all-feature/feature-powerset/MSRV/rustdoc/nextest, semver/package, dependency/advisory/provenance, deterministic render/catalog, Fumadocs build/link, and declared-platform terminal checks.

Registry and target caches are keyed by lockfile/toolchain/job purpose, compile jobs run offline after a locked fetch, Bun/Lychee cache only downloads, and PR caches are deleted on close. Generated SVGs, API reports, packages, and docs builds are artifacts rather than caches. `sccache` is added only after measured need. See [Repository Engineering](/reference/research/shared-tui-extraction/07-repository-engineering/).

## Acceptance

- the shared crate builds and tests without any Tailrocks product crate;
- the lookbook CLI previews and verifies every public component, and the Fumadocs catalog renders the same generated SVG states;
- the supported-terminal baseline (modern truecolor, OSC 8/22/52) is documented and no capability-degradation code exists in the library;
- known rendering defects were carried bug-compatible: no parity diff traces to an unrecorded fix, and quality-tier fixes landed only after parity with regenerated fixtures;
- `jackin❯` render and behavior parity passes before in-tree code is removed;
- Holla proves the API supports a different product state/effect model;
- no secret, agent, database, CI, or observability domain object crosses the shared API;
- terminal cleanup works on supported exit/failure paths;
- durable consumers pin immutable Git revisions or releases, while branch/fork references remain an explicit iteration mechanism;
- releases and revision snapshots are reproducible, documented, licensed, and attributed;
- public API, feature, MSRV, package contents, and render/catalog drift are mechanically reviewed;
- roadmap and canonical documentation in both repositories state one owner for every component and policy.

## Risks

- **Premature generalization:** solve with extract/parameterize/stay classification and the two-consumer rule.
- **Visual regression:** migrate `jackin❯` before redesign and compare deterministic fixtures.
- **Silent quality fixes during migration:** every parity diff must trace to a recorded quality-backlog item; an unrecorded rendering "improvement" fails the gate.
- **Dependency inversion:** reject imports from any consumer crate, feature-gate Crossterm integration, and keep Tokio out of the initial shared crate.
- **Cross-repository deadlock:** publish first, migrate one consumer at a time, and retain donor code until parity passes.
- **Brand leakage:** use semantic tokens and consumer-provided branding; keep product stories local.
- **History/license loss:** use a history-preserving extraction and independent provenance review.

## Not part of this item

- implementation of features in consuming products;
- Holla, Velnor, or Parallax product redesign beyond proving consumption;
- new components beyond the donor set (trees, progress indicators, data tables, command palettes, multi-line editors) — explicit follow-up work after extraction and migration;
- terminal capability degradation: 256/16-color fallback, `NO_COLOR` handling, light-background adaptation, and RTL/BiDi support;
- a universal Tailrocks state/runtime/IPC framework;
- a shared secrets crate;
- native macOS UI components;
- compatibility shims kept indefinitely after donor migration.
