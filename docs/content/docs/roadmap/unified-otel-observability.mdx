title: "Unified OpenTelemetry observability"
---

**Status**: Resolved

jackin❯ now has one governed application-observability contract. Each production process exports traces, logs, and metrics directly over OTLP/gRPC; the configured backend owns durable history. Product processes retain only bounded current-invocation state for operator UI and never write telemetry files, local logs, sidecars, spools, or Collector configuration.

The landed design uses the closed, Weaver-validated semantic-convention 1.43 registry in `jackin-telemetry`; generated constants and typed facade APIs prevent naming drift and raw emission bypasses. Operation spans are bounded. Lifecycle and state are events and metrics. W3C context crosses synchronous process boundaries, while detached work starts a new trace linked to its producer. Resource identity is stable per process, correlation identities stay out of Resource and metric dimensions, and allowlist-first validation rejects prohibited data before export.

`jackin diagnostics validate` is the only command that promises a delivery check. `--debug` remains operator troubleshooting output and may raise governed telemetry verbosity when OTLP is enabled; it does not create an artifact. Persistent daemons expose sanitized configuration fingerprints and typed current health.

## Resolution

The implementation removed the legacy JSONL run timeline, command-output telemetry sidecars, Capsule and host-daemon log files, local log readers/commands, file-specific telemetry configuration, legacy `jackin.*` and `parallax.run.id` telemetry namespaces, and the misleading telemetry-store name. Test-only PTY captures, render fixtures, and allocation artifacts remain explicitly outside production telemetry.

Profiles remain deferred until the OpenTelemetry Rust profiling signal is stable and receives a separate design. Backend deployment, storage, dashboards, alerts, authentication, and retention remain owned outside jackin❯.
