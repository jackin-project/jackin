title: "Unified OpenTelemetry observability"
---

**Status**: Open — design finalized with the operator and implementation-ready. This is an OTLP-only cross-crate refactor: product telemetry is exported to a configured OpenTelemetry endpoint and is never persisted to a jackin❯ telemetry file or rendered as a local telemetry stream

## Goal

Establish one governed, typed observability contract across every jackin❯ product crate, producing correlated OpenTelemetry logs, traces, and metrics across host and Capsule boundaries with bounded volume, explicit context propagation, and no telemetry leakage into operator output. "One contract" means one legal call-site facade and one semantic-conventions registry; internally, each signal uses its native OpenTelemetry API.

The resulting platform view must let an operator filter successfully delivered logs and traces by `parallax.run.id` and reconstruct the work causally owned by one invocation or console run: host and Capsule operations, the ordered chains of bounded operation and UI traces, named interaction and lifecycle events, correlated failures, and subprocess and Docker boundaries. Stable low-cardinality metrics describe product and exporter health; metric-to-trace correlation relies on shared bounded dimensions for now, because trace exemplars are deferred until the Rust SDK implements exemplar collection. Autonomous long-lived daemon work that belongs to no invocation remains queryable by its stable process identity and must never inherit a stale run id.

## Problem

jackin❯ already has an OpenTelemetry substrate, but most of the codebase does not emit through it. `RunDiagnostics` mints a run id, the optional `otlp` feature exports all three signals, and `jackin-diagnostics` contains a registry, redaction logic, screen taxonomy, and conformance tests. Alongside that substrate, prefix-tagged printf strings, direct terminal writes, hand-rolled JSONL diagnostics, and raw `tracing` sites remain separate observability interfaces.

The raw workspace inventory captured on 2026-07-15 found 284 `debug_log!` occurrences, 284 Capsule logging-macro occurrences, 127 `eprintln!` occurrences, 99 Tokio spawn/spawn-blocking occurrences, and 29 thread-spawn occurrences. These are search baselines, not all migration sites: definitions, tests, and legitimate operator output are included where applicable. Reproduce the inventory rather than copying the counts forward:

```bash
rg -n 'debug_log!' --glob '*.rs' crates | wc -l
rg -n '\b(clog|cdebug|ctrace_payload|cwarn|cerror)!' --glob '*.rs' crates/jackin-capsule crates/jackin-usage | wc -l
rg -n 'eprintln!' --glob '*.rs' crates | wc -l
rg -n 'tokio::spawn|spawn_blocking' --glob '*.rs' crates | wc -l
rg -n 'thread::spawn|std::thread::spawn' --glob '*.rs' crates | wc -l
```

The current split creates several concrete failures:

- **Message bodies carry dimensions.** Variable values are flattened into strings, so backends need regular expressions instead of typed attribute queries.
- **Events are not real OTel events.** The current function facade writes an `event.name` attribute instead of the top-level OpenTelemetry `EventName`, and dynamic attribute slices are stamped onto the active span rather than the emitted log record.
- **Crate ownership is lost.** A function-based facade reports `jackin-diagnostics` as the `tracing` target instead of the calling crate and module.
- **Identity is process-global where it must be request-scoped.** Capsule receives a launch-time `JACKIN_RUN_ID`, but its daemon can outlive that launch and serve later attaches belonging to other host runs.
- **The facade is in the wrong dependency tier.** `jackin-diagnostics` is Tier 2 while foundational crates such as `jackin-core` and `jackin-process` are Tier 0, so a universal facade in `jackin-diagnostics` would violate the enforced dependency direction.
- **Operator output and telemetry overlap.** Some terminal writes are product output, some are diagnostics, and some do both. A rich TUI cannot safely share its terminal with a telemetry formatter.
- **Background causality is accidental.** Async tasks, blocking tasks, threads, socket handlers, timers, and stream pumps routinely detach from the triggering context or retain context longer than intended.
- **The local logging surface is larger than one JSONL file.** Run JSONL, command-output sidecars, Capsule `multiplexer.log`, the host-daemon log, `jackin logs`, `jackin daemon logs`, failure-tail readers, reveal actions, and bundle writers form a second observability product with separate schemas, retention, and redaction behavior. This item removes that product instead of renaming only its primary artifact.
- **Capsule delivery crosses a security boundary.** Directly forwarding OTLP headers, client keys, or arbitrary Resource attributes would expose operator-owned backend credentials or metadata inside an agent-controlled container. Endpoint routing and telemetry context are safe to propagate; exporter secrets are not.
- **Persistent processes freeze exporter configuration.** A Capsule or host daemon may survive the CLI process that launched it. A later attach can supply a new run/trace context, but an OpenTelemetry provider and `tracing` subscriber cannot be silently replaced with a different endpoint or credential set mid-process.
- **The current topology conflates work with presentation.** A command or request can continue across several screens, while one screen can initiate several operations. Treating a screen visit as the operation trace makes ownership, outcome, and rotation ambiguous.

## Target view

- **One run, one query scope.** Every run-owned log and span carries `parallax.run.id`; filtering by it returns the complete causal story exported by host and Capsule processes. A run is a correlation scope, not necessarily one trace.
- **Stable resources, dynamic context.** Resource attributes identify the emitting process. Run, operator-session, screen, pane, and request identities live on the records and spans whose lifetimes they actually describe.
- **Two bounded trace families.** Operation traces describe commands, requests, and substantive background work. UI-interval traces describe screen visits and tab focus. A semantic UI action links to the operation it starts; neither trace family borrows the other's lifetime.
- **Named, typed events.** Interaction, lifecycle, transition, failure, and operator-notice mirrors are log-based events with a real top-level `EventName`, typed attributes, a redacted body, and current trace context.
- **Failures point at themselves.** Failed operations set span status and `error.type`; cancellation is distinguishable from failure; exception records correlate to the failed span; telemetry delivery failures never recurse into the same exporter.
- **Background work is useful, not noisy.** Meaningful scheduled work and state changes are observable; no-op ticks and continuous byte streams remain metrics rather than traces.
- **OTLP is the only production telemetry destination.** No product telemetry file, JSONL fallback, stdout exporter, stderr formatter, or TUI telemetry layer exists. Without a configured OTLP endpoint there is no retained telemetry.

The backend-neutral query contract is:

| View | Required join and shape |
|---|---|
| Run | Filter logs and spans by `parallax.run.id`; group by process Resource and `jackin.component`; show operation roots, linked UI intervals, failures, and external boundaries |
| Operation | Start from a root carrying registered `jackin.operation.name` and outcome; follow children and links; do not infer outcome from log severity |
| Session/UI | Filter by `session.id`; order screen/focus roots by explicit predecessor links and transition events rather than assuming clocks give a total order |
| Health | Query low-cardinality product/exporter metrics by stable Resource and component; never join metrics by run, session, attach, path, branch, or container name |

## Scope and consequences

This item owns the cross-crate API, signal semantics, context propagation, OTLP runtime, migration, and enforcement. It supersedes the observability-format fragments in the following items:

- [Launch Progress TUI](/roadmap/launch-progress-tui/) keeps current-run application state and the correlation run id visible; historical traces and cross-run comparison remain in the operator's configured backend because OTLP is an export protocol, not a query protocol.
- [Console resource panel](/roadmap/console-resource-panel/) may render live product state or separately queried backend data; it does not read a local telemetry artifact.
- [jackin❯ join & Debug Export](/roadmap/jackin-join-and-debug-bundle/) cannot automatically package telemetry after this cutover. A debug bundle may include non-telemetry diagnostics and redacted configuration, while telemetry remains in the operator's backend.
- [Sensitive-boundary code health](/roadmap/sensitive-boundary-code-health/) supplies the negative privacy assertions and issue-fingerprint policy consumed by the unified facade and OTLP conformance lane.

The loss of offline telemetry is deliberate. Normal commands continue when no endpoint is configured; telemetry levels and categories then have no signal sink, while `--debug` still controls redacted operator-facing error detail. Only a command that explicitly validates export, such as `jackin diagnostics validate`, fails when no endpoint is configured or a test export cannot flush. The render-conformance fixture recorder (`cargo xtask pty-fixture`) currently harvests PTY byte dumps from the run artifact and the Capsule log file; because PTY bytes are banned from telemetry and those files disappear, fixture recording moves to a dedicated development capture channel owned by the test harness — explicitly not telemetry and never enabled in production paths. Run JSONL, command-output sidecars, `multiplexer.log`, the host-daemon log, their readers/reveal/bundle commands, file-specific environment variables, and telemetry inclusion in debug bundles are removed during cutover with no migration shim under the pre-release policy. Contributor and CI telemetry validation runs against an in-memory test receiver or an operator-run Collector; jackin❯ itself persists nothing.

The documented recommended deployment shape — guidance only, jackin❯ ships and operates none of it — is a loopback-bound local agent Collector receiving all jackin❯ processes, exporting upstream to the operator's gateway or backend with batching, retries, and optional disk-backed queueing, and applying allowlist redaction as a defense-in-depth line behind jackin❯'s own in-process redaction. That shape is what the endpoint contract in decision 1 is designed for, and it is where operators add tail-sampling, routing, or credentialed upstream export. A local break-glass log file — the common mitigation for telemetry lost to `SIGKILL`, OOM, or power loss — is consciously rejected: crash evidence is limited to container-runtime stderr capture, exit codes, and whatever telemetry flushed before termination.

## Research basis

Verified against OpenTelemetry Rust, OTLP 1.10.0, semantic conventions 1.43.0, and OTel Weaver as of 2026-07-15.

- **Signal-native APIs.** OpenTelemetry Rust provides independent logger, tracer, and meter providers. The project facade uses the Logs API for named events, `tracing` plus `tracing-opentelemetry` for spans and context, and the Meter API for metrics. A shared contract does not require forcing all three signals through `tracing`.
- **Log-based events.** OpenTelemetry's stable [logs data model](https://opentelemetry.io/docs/specs/otel/logs/data-model/) defines `EventName` as a top-level field. OpenTelemetry [deprecated the span-events API](https://opentelemetry.io/blog/2026/deprecating-span-events/) for new instrumentation, so event-shaped data is emitted as named logs correlated to the current span.
- **OTLP transport.** The [OTLP specification](https://opentelemetry.io/docs/specs/otlp/) defines delivery, retry, partial-success, and duplicate-data behavior. jackin❯ keeps its existing OTLP/gRPC transport choice and rejects unsupported protocol configuration rather than silently changing transport.
- **Resources.** Resource attributes describe the entity producing telemetry and remain fixed for a provider lifetime. Dynamic run and attach context therefore belongs on individual records and spans, not the Capsule daemon's Resource.
- **CLI conventions.** The [CLI span conventions](https://opentelemetry.io/docs/specs/semconv/cli/cli-spans/) are Development status and apply only to short-lived programs. They require `process.executable.name`, `process.pid`, and `process.exit.code`; command arguments are withheld by default because they can contain secrets.
- **Session conventions.** The [session conventions](https://opentelemetry.io/docs/specs/semconv/general/session/) are Development and Opt-In. `session.previous_id` describes continuation on `session.start`; it is not stamped indiscriminately on every signal.
- **Context propagation.** W3C propagation carries and validates both `traceparent` and `tracestate`. Baggage is excluded from jackin❯ propagation until a separately reviewed, bounded use case exists.
- **Rust context timing.** `tracing-opentelemetry` can reject a parent assigned after a span has started, and thread-local OpenTelemetry context guards are unsafe to hold across arbitrary async suspension. Remote context must therefore be extracted before span creation and carried by async-aware wrappers or explicit tokens.
- **Exporter behavior.** OTLP partial success is success with rejected counts and must not be retried. In the current Rust exporter family, spec-aware gRPC retry is an opt-in maintained feature and partial-success detail is reported through internal diagnostics rather than a stable typed health callback.
- **TUI semantics.** TUI telemetry belongs at semantic message/action/effect boundaries, not raw key, mouse, text-input, clipboard, or PTY dispatch points.
- **Collector deployment.** The Collector project recommends an agent-to-gateway arrangement — a loopback-bound local agent Collector doing memory limiting early and batching late, with queued/retried upstream export — for resilient local collection. Gateway-side tail sampling buffers whole traces keyed by trace id, so it only works when traces are bounded, which the rotation limits below guarantee.

## Design

Settled with the operator on 2026-07-15. These decisions resolve the call-site API, dependency ownership, signal semantics, identity lifecycles, cross-process propagation, trace topology, delivery behavior, privacy boundary, governance, cutover, and completion gates.

### 1. OTLP is the only production telemetry output

The product binaries compile the OpenTelemetry API, SDK, and OTLP/gRPC exporters as baseline dependencies. Logs, traces, and metrics have separate SDK providers and processors but one runtime configuration contract. Only OTLP/gRPC export is supported; file, stdout, stderr, console, JSONL, TUI, and persistent queue/spool exporters do not exist. Delivery buffers are bounded and memory-only. The OTLP crate disables default features and enables only `trace`, `logs`, `metrics`, `grpc-tonic`, `experimental-grpc-retry`, `gzip-tonic`, `tls-aws-lc`, and `tls-roots`, so an unused OTLP/HTTP client is not linked into an allegedly gRPC-only product. The `internal-logs` feature stays disabled: the current exporter can render raw headers in an internal debug record, and exporter text is neither a privacy-safe nor a typed health API.

`OTEL_EXPORTER_OTLP_ENDPOINT` supplies the base endpoint for all three signals. A standard per-signal endpoint overrides that base for its signal; when no base is set, all three per-signal endpoints must be present, so a partially configured signal set fails initialization instead of silently omitting logs, traces, or metrics. The base and per-signal protocol variables, when set, must all resolve to `grpc`; exporter selectors must be absent or `otlp`; the propagator is fixed to `tracecontext`; and a conflicting sampler is rejected rather than silently weakening the run story. `OTEL_SDK_DISABLED=true` wins over endpoint presence and produces the same no-provider state as no endpoint. Standard timeout, gzip compression, Resource, service-name, header, and per-signal overrides are supported within the safety ceilings below.

Host and Capsule processes export independently; jackin❯ does not relay, proxy, or reserialize telemetry. At launch, the host propagates only normalized endpoint/protocol/compression routing, a sanitized Resource subset, and W3C context. OTLP headers, embedded endpoint userinfo, client keys/certificates, and raw `OTEL_RESOURCE_ATTRIBUTES` are never injected into an agent-visible environment. Host-side headers may authenticate the host exporter. An authenticated or mTLS upstream that must receive Capsule telemetry requires an operator-run OpenTelemetry Collector or gateway with a credential-free, container-reachable receiver and authenticated upstream export. jackin❯ neither ships nor operates that Collector.

Every distinct Capsule endpoint host is derived from the normalized per-signal endpoints and added to the jackin❯ infrastructure allowlist only when the effective network grant permits egress. `network=none` always wins: the launch continues with host-only telemetry and emits a safe host-side `telemetry.coverage.changed` event instead of weakening isolation or failing product work. A Collector on host loopback is rewritten per endpoint to the container-reachable host gateway. OTLP environment names are reserved from role/workspace overrides. *Why:* a pure OTLP path is useful only if it does not smuggle backend credentials across the trust boundary or override an explicit isolation decision.

When no endpoint is configured or the SDK is disabled, normal product flows use no-op tracer/meter behavior, leave the log dispatch uninstalled (the Rust Logs API deliberately has no global provider), and create no telemetry files, exporter runtime, worker thread, or network socket. `--debug` keeps its redacted operator-output role and, when OTLP is active, raises the telemetry level to DEBUG; it never makes an otherwise valid command depend on collector availability. `[telemetry].level`, `JACKIN_TELEMETRY_LEVEL`, and category controls are inert without an active provider. `jackin diagnostics validate` is the explicit delivery promise: it emits one marked record for each signal, force-flushes, and fails if configuration or delivery cannot be confirmed. Runtime unreachability during normal product work does not fail or block that work. *Why:* optional observability must not become a product availability dependency, while a purpose-built validation command can truthfully promise an export attempt.

Exporter configuration is immutable for one provider/process lifetime. A persistent host daemon or Capsule reports an opaque SHA-256 fingerprint of its canonical, length-delimited effective exporter configuration over its control protocol; secrets contribute to equality but are never rendered or returned. A later client with a different fingerprint does not reconfigure the process; host telemetry continues under the client's provider, the persistent process keeps its original provider, and coverage is marked degraded until that daemon/container is restarted. Dynamic run/session/trace context still changes per request. *Why:* OpenTelemetry providers and the global `tracing` subscriber are install-once process infrastructure, not request configuration.

### 2. Lowest-tier facade, higher-tier runtime

A new dependency-minimal `jackin-telemetry` crate becomes the lowest product architecture tier. It owns the generated names and attributes, typed values, log/span/metric call-site macros, operation outcome guard, explicit context token, and spawn helpers. It depends on no other jackin❯ crate. Existing foundational product crates that emit telemetry are re-tiered above it, and <RepoFile path="crates/jackin-xtask/src/arch.rs" /> remains the authority that proves every production dependency points downward. Re-tiering `jackin-core` and `jackin-process` above the new crate renumbers the whole tier table — including per-crate README tier sections and the header cross-check — a known mechanical consequence of this decision, not scope creep.

`jackin-telemetry` exposes an install-once process dispatch. Before installation, its enabled checks are false and all signal calls are no-ops; once installed, the dispatch holds only signal-facing traits and explicit context machinery, not `jackin-diagnostics` or SDK types. Call sites check enablement before formatting bodies or constructing dynamic attributes. `jackin-diagnostics` remains the process-level runtime owner: it builds the logger, tracer, and meter providers; installs the dispatch, span bridge, and propagator; applies filters and SDK limits; manages the dedicated export runtime; exposes in-memory exporter health; and coordinates flush/shutdown. A second installation is a fatal initialization error. *Why:* every product crate needs the facade, while provider construction, terminal ownership, and process lifecycle are application concerns that cannot sit at the bottom of the graph.

### 3. One call-site contract, three native signal APIs

Every product log is a registered log-based event; there is no unregistered free-form telemetry logger. The OpenTelemetry Logs API populates `EventName`, optional static/redacted body, severity number/text, typed attributes, timestamp, instrumentation scope, trace id, and span id in their native fields. Event names and attribute keys are generated constants, variable data lives only in typed attributes, and the event name is not duplicated as an attribute. The facade obtains correlation from the current `tracing` span or an explicit context token and sets the log record's trace context directly rather than relying on ambient thread-local context across async suspension.

Spans use facade macros over `tracing` and `tracing-opentelemetry`; metrics use facade helpers over the OTel Meter API. A log instrumentation scope is the emitting crate name/version plus the registry schema URL, not `module_path!()`; module, repo-relative compile-time source path, and line are code attributes. Shared product metrics use one generated `jackin-telemetry` meter scope so the same instrument does not split into different metric identities by caller; bounded component/operation attributes carry ownership. Spans bridged through `tracing` share one process-level tracer scope by construction, so the same registered code/owner attributes carry span ownership. Dynamic values support the OTel scalar and array types rather than being coerced to `String`.

Raw `tracing::event!`, raw OTel logger/meter calls, and unrestricted `#[instrument]` are forbidden outside the facade/runtime implementation and generated code. Facade span macros default to `skip_all` behavior and require registered attributes, preventing automatic argument capture. Project log events are not routed through the span-event conversion path, and conformance tests prove that each facade event produces one log-based event and zero OTel span events. *Why:* a single governed surface is compatible with signal-native data only when bypasses and double emission are both closed.

### 4. Weaver owns the telemetry contract

An OTel Weaver registry imports semantic conventions 1.43.0 and defines every project or vendor extension, including `jackin.*` and `parallax.run.id`, plus every event, span, metric, type, requirement level, stability marker, unit, histogram boundary, and allowed attribute set. Weaver 0.24.2 is pinned for the initial implementation; version upgrades are explicit schema changes. Project-owned Rust templates generate constants and typed definitions consumed by `jackin-telemetry`. The Resource carries `jackin.telemetry.schema.version=1`, all instrumentation scopes use the matching custom schema URL, and an incompatible registry diff increments that version so backends can distinguish mixed deployments. `weaver registry check`, a reviewed `registry diff`, generated-drift detection, and `weaver registry live-check` join CI.

Weaver complements rather than replaces runtime and behavioral enforcement. Fail-closed runtime validation, prohibited-key/body checks, privacy negatives, cardinality checks, export-volume ratchets, propagation tests, in-memory SDK tests, and an OTLP wire-capture conformance lane remain required. An unregistered event, span shape, metric, or attribute is rejected before it reaches any provider. *Why:* schema correctness, privacy, causality, and operational behavior are different properties and need different proofs.

### 5. Telemetry and operator output are separate ports

Operator output stays on the product output port and may render to stdout, stderr, or a TUI-owned surface according to the command contract. Telemetry never writes to those surfaces. Operator notices are not mirrored automatically: each typed notice declares `none` or a registered safe telemetry event, with `none` as the default. A mirrored event is independently redacted and carries safe structured fields; it never reuses an already-formatted terminal string.

Exporter initialization, queue, flush, and delivery failures use a minimal self-diagnostics control path that cannot call the telemetry facade. It may update an in-memory health snapshot and surface one debounced notice per health-state transition when terminal ownership permits, but it never serializes application telemetry or writes a diagnostics file. Capsule daemon boot and fatal-exit lines emitted before providers exist or after they shut down are bounded operator output on stderr and may be captured by the container runtime; jackin❯ does not mirror or copy them into a log file.

Transient command/build output required by an active TUI is held in a bounded in-memory ring and discarded when the owning operation ends. Machine-relevant failure state crosses typed result/control messages instead of being recovered by tailing a file. Automatically persisted run JSONL, command-output sidecars, `multiplexer.log`, and host-daemon logs are telemetry/log surfaces and are removed together with `jackin logs`, `jackin daemon logs`, reveal/follow/bundle actions, and failure-tail readers. Explicit user data, workspace state, caches, usage-history state that powers a product feature, and test-only fixture capture are not telemetry; misleading names such as `telemetry_store` are renamed to the product concept they actually persist. *Why:* product output must remain predictable, and an OTLP-only claim is false while jackin❯ still owns another automatic log store.

### 6. Identity has explicit lifecycles

| Identity | Lifetime | Placement |
|---|---|---|
| Process resource | One host or Capsule process/provider lifetime | `service.namespace=jackin`, stable service name, `service.version`, 128-bit `service.instance.id`, `process.pid`, runtime/process/container attributes on Resource |
| Run | One host CLI invocation or one `jackin console` process | `parallax.run.id` on every run-owned log and span; omitted from autonomous daemon telemetry and metric dimensions |
| Operator session | One top-level interactive client lifetime | 128-bit `session.id` on session-owned logs and spans; propagated to every attach initiated by that client; `session.start` and `session.end`; `session.previous_id` only on a continuing `session.start` |
| Attach | One host-to-Capsule handshake/connection lifecycle | 128-bit `jackin.attach.id` on attach-owned logs and spans; never a metric dimension |
| Agent session | One Capsule PTY/pane agent lifetime | `jackin.agent.session.id` plus bounded tab/pane attributes; never overloaded into `session.id` |
| Trace context | One invocation, screen/focus interval, request, or bounded background operation | Native trace/span ids plus W3C propagation |

Locally minted run, session, attach, agent-session, and process-instance identifiers use cryptographically random 128-bit values encoded as 32 lowercase hexadecimal characters. An external Parallax run id is trimmed, may lose one leading `run_` compatibility prefix, and must then contain 1–64 ASCII alphanumeric, hyphen, or underscore characters; invalid input is rejected instead of filtered or truncated. Resource never contains a run, session, attach, or dynamic component label. Metrics use the stable Resource plus bounded dimensions; run, session, attach, agent-session, container name, path, branch, user-entered values, and other unbounded identifiers are prohibited as metric attributes. Trace exemplars are explicitly deferred: the Rust SDK does not collect exemplars yet, so metric-to-trace correlation uses shared bounded dimensions until upstream support lands.

All three providers receive clones of the same immutable Resource value, and their instrumentation scopes use the registry schema URL. `service.name` is `jackin` for CLI/console processes, `jackin-daemon` for the host daemon, and `jackin-capsule` for Capsule; all use `service.namespace=jackin`. SDK process/service detectors and `OTEL_SERVICE_NAME`/`OTEL_RESOURCE_ATTRIBUTES` are honored with explicit policy: the operator may override `service.name`, but product-owned `service.namespace`, `service.version`, `service.instance.id`, process identity, and telemetry schema keys cannot be overridden; `parallax.run.id` is extracted into run context and removed from Resource; remaining operator attributes are string-only, redacted, count/size-capped, and rejected when malformed or credential-shaped. Only that normalized subset may be propagated to Capsule. *Why:* standard deployment metadata remains useful without allowing dynamic correlation ids, secrets, or spoofed process identity to fragment every signal.

Host run-id precedence is `PARALLAX_RUN_ID`, then a valid `parallax.run.id` parsed from `OTEL_RESOURCE_ATTRIBUTES` for Parallax compatibility, then a locally minted id. A run id exists even with telemetry disabled because the live UI and protocol use it as correlation state. Capsule startup may receive the launch run context, but every later attach/control request supplies its own validated context; no daemon-global `JACKIN_RUN_ID` may stamp unrelated work. Conflicting non-empty identities reject initialization or the request rather than being silently merged. A request carrying valid propagated context reuses the originating operator session; a new `session.id` is minted only for a new top-level interactive client, while every attach mints its own `jackin.attach.id`. *Why:* identity follows ownership and duration, which prevents stale correlation and metric-cardinality explosions while preserving the Parallax query key.

### 7. Context crosses every jackin❯ process boundary explicitly

The attach and control protocols gain a versioned `TelemetryContext` carrying `parallax.run.id`, `session.id` when applicable, `traceparent`, and `tracestate`. Outgoing request boundaries create a `CLIENT` span and use the official W3C `TextMapPropagator` to inject both fields into the protocol carrier. Capsule and host-daemon handlers extract before creating the corresponding `SERVER` span and set its remote parent before the span is entered or any field/event is recorded. Malformed/oversized W3C fields and `tracestate` without a valid `traceparent` are ignored without echoing raw values; the product request starts a safe local root and records only a bounded propagation-error classification. Malformed or conflicting run/session identities reject the request because those fields govern product correlation rather than optional trace parenting. When both W3C fields are absent, a valid run/session starts a local trace without reporting an error. Context-free autonomous work starts under process identity with no run or operator session.

Launch-time environment propagation parents Capsule startup only. Dynamic operations never rely on an environment variable that was fixed when the container started. `traceparent` and `tracestate` propagate together; baggage is not accepted or forwarded; upstream sampled flags are honored by the parent-based sampler. Context is passed explicitly across concurrent tasks rather than stored in a mutable process-global slot. A thread-local OpenTelemetry `ContextGuard` cannot live across `.await`; async helpers instrument the future or re-enter context only for a synchronous poll/emit boundary. *Why:* the persistent Capsule and host daemon can serve multiple runs over time, so request context must be scoped like request data and attached before the Rust bridge starts a span.

### 8. Runs contain bounded trace graphs

Runs contain two independent, linked trace families:

- **Operation traces** represent one top-level CLI command, semantic TUI operation, incoming request, or substantive background job. A short-lived CLI execution root follows the pinned CLI convention (`INTERNAL`, executable name, pid, exit code); daemon lifetimes do not. A UI action that starts work emits its action event on the current UI-interval span, then the new operation root links to that span context rather than inheriting the whole screen lifetime.
- **UI-interval traces** represent one console/launch screen visit or one Capsule tab-focus interval. They contain semantic UI actions/effects only; launch stages, Docker calls, and other operational spans stay in the operation trace. PTY output is never represented as spans.
- Every trace rotates after 30 minutes or 512 ended spans, whichever comes first. A successor root links to the predecessor root and carries the same run/session plus trace-family and operation/screen classification. These initial limits are registry-governed constants, not user configuration.
- After a successor exists, the predecessor context emits `screen.transition`, `focus.changed`, or `trace.rotated` with the successor trace id. Each interval emits a start event immediately and an end event on normal close. No design depends on a nonexistent transition span or a final event surviving fatal termination.

`parallax.run.id`, `session.id`, explicit links, and stable operation/screen names are the joins across the graph. Timestamps support display but never establish causal ordering across processes. A daemon lifetime is represented by its Resource identity, registered lifecycle events (`process.started`, `daemon.ready`, `daemon.draining`, `daemon.stopped`), and health metrics — never by a lifetime span. *Why:* work and presentation have different lifetimes; separating them preserves ownership and boundedness without backend-hostile monster traces.

### 9. Every concurrent task declares a context mode

Facade helpers cover `tokio::spawn`, runtime-handle/`JoinSet` spawning, `spawn_blocking`, `spawn_local`, `std::thread::spawn`, scoped/thread-builder spawning, TUI subscription helpers, socket/request handlers, timers, and long-lived stream tasks. Type-aware linting, not the baseline grep counts, is the completion authority. Every production spawn site declares exactly one mode:

1. **inherit** — bounded work whose result returns to the triggering interaction; it remains a child of the active operation.
2. **link** — detached work that outlives its trigger; it starts a new trace linked to the spawning span. When the detachment is a scheduled or queued hand-off, the scheduling span is `PRODUCER` and the processing root is `CONSUMER` (decision 11).
3. **cycle** — periodic maintenance; health is metrics, and state changes, failures, or slow completions are named logs. A trace is created only when substantive work is known before execution, such as a scheduled provider fetch, never retroactively after a no-op poll. A cycle trace carries aggregate counts (items scanned, changed, failed) as attributes; per-item spans and per-item events are prohibited.
4. **stream** — continuous pumps, watchers, accept loops, and reapers; the stream lifetime is not a span. Bounded handshakes/lifecycle operations are spans, notable occurrences are logs, and throughput/health are metrics.

High-frequency status ticks, branch polling, instance refresh, render loops, PTY pumps, and resource sampling cannot emit per-cycle or per-item spans. Bare spawn calls outside approved facade/runtime internals fail lint. *Why:* causality and volume must be deliberate at the creation point; tail-buffering every poll merely to decide whether it was interesting is unnecessary complexity.

### 10. TUI observability records semantic actions, never raw input

Console, launch, and Capsule instrumentation attaches at their semantic message, action, and effect boundaries after keyboard and mouse input has been interpreted. Events describe actions such as screen transition, widget activation, dialog open/confirm/cancel, focus change, and operation request. They may carry registered screen/action/widget identifiers but never raw keys, typed text, selected secret values, clipboard contents, PTY bytes, terminal escape sequences, command bodies, or unbounded labels.

The registry keeps the established `jackin.screen.name` contract and adds `jackin.ui.action.name` and bounded widget identifiers. Screen analytics ride metrics and events rather than span duration: each screen-exit event carries dwell time feeding a per-screen dwell histogram, transitions carry a bounded reason, and a monotonic `jackin.ui.navigation.sequence` orders navigation without relying on cross-process clocks. It does not claim browser/mobile `app.*` event names are stable TUI conventions; adoption of those Development conventions requires a future explicit registry migration. *Why:* semantic actions answer what the operator did without turning observability into input capture.

### 11. External boundaries use pinned semantic conventions

- Internal work uses `SpanKind::INTERNAL`, low-cardinality names, and registered parameters as attributes.
- Subprocess calls use `SpanKind::CLIENT` in the shared `jackin-process` boundary after its dependency re-tiering. Spans include executable name, child pid, exit code, and a documented low-cardinality error type. `process.command_args` and executable paths are omitted by default; a future sanitized allowlist requires separate privacy review.
- Docker Engine and provider HTTP work use the pinned client conventions with network/server attributes and a bounded custom external-system classification where the standard has no exact equivalent.
- Outgoing attach, control, daemon, and credential-resolver requests use `SpanKind::CLIENT`; the receiving process uses `SpanKind::SERVER`. Persistent PTY/byte streams are not request spans: only their bounded handshake, control operations, and close lifecycle are traced.
- Scheduling deferred or queued work uses `SpanKind::PRODUCER`; the later processing root uses `SpanKind::CONSUMER` with a creation-time link back to the producer span, so a retried or delayed job never inherits a completed trigger as its parent.

Log instrumentation scope or the registered code/owner attributes identify the emitting crate/module; shared metrics use their generated product scope and bounded ownership dimensions. A separate bounded `jackin.category` may classify operational domains; `target` is never overloaded as both ownership and category. *Why:* backend service graphs, ownership queries, and category filters need distinct, semantically correct fields.

### 12. Operation outcomes are result-aware

Operation wrappers close explicitly from `Result`, exit status, or a typed cancellation reason. Success leaves OTel span status unset and records `jackin.operation.outcome=success`; failure sets Error status, a stable low-cardinality `error.type`, and emits exactly one registered `operation.failed` or domain-failure event, with `exception.*` fields only when a real error value is available. Expected operator cancellation/detach records `cancelled` without Error status. A typed task abort records `cancelled`; an unclassified guard drop records `abandoned`, sets Error status, and uses a stable instrumentation-error type so missing completion paths are visible rather than mislabeled as user intent. Panic unwind records failure where Rust can observe it; process abort and `SIGKILL` make no flush or final-span guarantee.

Failure ownership is single: lower layers return typed errors and may attach attributes or DEBUG events, and only the layer that owns the operation outcome emits the one failure event — never one ERROR per layer for the same failure. Recovered degradation (retry scheduled, fallback selected, partial result, temporary disconnection) is WARN with a bounded classification; ERROR is reserved for the owning operation's terminal failure. Exception attributes and stack details pass the same redaction and size policies as every other event. Product binaries centralize exit-code selection so roots close and providers flush before the final `ExitCode`/`process::exit` boundary; no lower layer terminates the process directly. *Why:* `?`, user cancellation, task abort, panic, non-zero exit, and an instrumentation gap are different outcomes and must remain distinguishable in a backend.

### 13. Metrics have explicit instruments and cardinality budgets

Every metric registry entry declares instrument kind, unit, description, aggregation, cumulative temporality, histogram boundaries where applicable, and its complete allowed dimension set. Counters cover operations, failures, bytes, transitions, facade validation/cardinality drops, and cycles; histograms cover launch stages, render time, request latency, and exporter flush latency; observable gauges cover liveness, process uptime, and bounded resource/runtime state, and their callbacks read pre-computed atomics rather than performing I/O or taking async locks. Shared instruments use the generated `jackin-telemetry` meter scope regardless of the calling crate, so one metric identity cannot split accidentally across instrumentation scopes.

Metrics never contain run/session/attach ids, trace ids, paths, branch names, container names, user input, event bodies, or error messages. Every dimension is a registry enum or another explicitly bounded value; an instrument may have at most 256 active attribute sets per process between collection cycles. A new attribute set over that ceiling is rejected and counted as a cardinality drop rather than evicting an arbitrary live series. The interactive export interval starts at 30 seconds and every normal product shutdown force-flushes metrics before provider shutdown. Short-lived commands flush all three providers explicitly. *Why:* metrics remain aggregatable and affordable only when dimensions, series ceilings, and lifecycle behavior are part of the contract.

### 14. Filtering, sampling, and SDK limits are signal-specific

There is one OTLP destination but each signal keeps the policy native to it:

- Logs use severity and target/category filtering. INFO is the default product threshold; DEBUG activates only through the existing resolved telemetry-level contract after endpoint validation; TRACE remains a separately gated trusted diagnostic tier.
- DEBUG and TRACE remain semantic telemetry. Neither tier permits raw PTY, input, clipboard, command arguments, credentials, environment values, or other prohibited payloads.
- Traces use `ParentBased(AlwaysOn)`: jackin❯ roots are recorded, while a valid unsampled remote parent remains unsampled. Volume is controlled by bounded topology and task modes rather than probabilistic loss of the support story. Bounded rotation keeps every trace compatible with operator-side gateway tail sampling; if volume ever demands probabilistic root sampling, that is a future registry-governed policy change, not an environment tweak.
- Metrics use registered instruments, views, and cumulative temporality; log severity does not disable required health metrics.
- Product log scopes and the single product tracer scope export by default; registered metric instruments use the single generated meter scope. Dependency scopes are off unless explicitly allowlisted to prevent exporter/network recursion and dependency noise.

Initial SDK budgets are explicit ceilings: 4,096 queued log records, 2,048 queued spans, batches no larger than 512 records, a 1-second scheduled delay for logs and traces, a 5-second total export attempt including retry delays, a 30-second metric interval, a 5-second total coordinated shutdown budget, and one dedicated single-worker Tokio runtime per process for all exporters. Supported standard environment variables may lower these ceilings; values that are invalid or exceed them fail explicit telemetry initialization rather than being ignored or clamped invisibly. Queue exhaustion drops telemetry instead of blocking product hot paths; exact SDK queue-drop accounting is exposed only when the pinned SDK provides a stable typed hook. With no active provider, enabled checks return before message formatting, attribute allocation, span construction, or export-runtime access. A checked-in telemetry benchmark covers no-provider and active-provider launch, render, and PTY paths; the ratchet permits no more than a 5% regression from its reviewed baseline. Budget changes require conformance-volume and overhead evidence. *Why:* levels, sampling, metric views, and resource budgets solve different problems and cannot truthfully be one generic filter.

### 15. OTLP failure behavior is bounded and visible

Invalid endpoint, protocol, TLS, header, sampler, propagator, exporter selector, or over-budget SDK configuration fails initialization when the operator explicitly configured telemetry. Once initialized, the maintained spec-aware gRPC retry feature handles only retryable transport and server-throttle failures, honors retry delay metadata, uses exponential backoff with jitter, attempts at most three retries, and remains inside the per-batch 5-second export timeout. Authentication/configuration failures, other non-retryable status codes, and OTLP partial-success responses are never retried. Product work does not wait for retry exhaustion, and duplicate delivery is accepted as an OTLP property.

`TelemetryHealth` is deliberately limited to data available from stable typed SDK/exporter hooks: configured and active signals, configuration fingerprint, facade validation/cardinality/redaction drops, last export attempt, last successful export, last failed export, force-flush result, and shutdown result. SDK queue-drop counts, retry-attempt counts, exact partial-success, and backend-rejected counts are added only when the exporter family exposes stable typed callbacks; the implementation must not scrape or parse exporter diagnostic text to manufacture them. A jackin-owned exporter wrapper records typed outer export results without changing payloads or retry classification. Upstream internal logs stay disabled, and one debounced operator notice derives only from those typed state transitions.

`jackin diagnostics status` becomes a live configuration/health command rather than a local-run reader. It reports resolved current-process configuration plus typed health obtained over the existing control protocol from a selected live host daemon or Capsule: active signals, sanitized endpoint authority, protocol, queue ceilings, configuration fingerprint match, coverage state, and the fields above. It never prints headers or endpoint userinfo. The snapshot is not persisted, does not promise post-exit history, and does not query the backend. A TUI defers the one operator failure notice to its owned notice surface; telemetry never writes directly to the terminal.

Shutdown has one owner and one order: stop accepting new work; close operation and UI roots; force-flush metrics, logs, and traces while the export runtime is alive; shut down the three providers; then stop the runtime. The complete sequence shares the 5-second budget, records which phase timed out, and may drop remaining telemetry but may not deadlock or extend product shutdown indefinitely. *Why:* OTLP-only delivery gives the backend ownership of retention and query, while jackin❯ still needs truthful, non-recursive current health and a deterministic provider lifetime.

### 16. Redaction happens before every provider

The facade accepts typed safe values and validates/redacts bodies and attributes before invoking a logger, tracer, or meter. Provider processors enforce the prohibited-key and size policies again as defense in depth. Authentication headers, credentials, environment values, command arguments, PTY/input payloads, clipboard data, terminal escape content, full paths, and unbounded error/debug dumps are deny-by-default.

The initial registry ceilings are 32 attributes per log, 64 attributes and 8 links per span, 32 attributes per metric point, 4 KiB for a log body or redacted exception message, 1 KiB for one string attribute, 32 elements for one array value, and 128 bytes for an event, span, instrument, or registered enum name. Source paths, when enabled, are normalized repository-relative paths rather than absolute paths. Safe textual values are redacted first and then UTF-8-truncated with an explicit marker and original-length field; attribute-count, array-count, link-count, unknown-key, and invalid-enum overflow rejects the signal or metric update and increments typed self-health. There is no raw-value fallback.

Redaction failure rejects the signal and updates self-health; it never falls back to a raw formatted string. Endpoint and TLS errors sanitize authorities and never render headers or embedded credentials. *Why:* data that entered a provider can already reach an exporter, so the primary privacy boundary must be before provider submission rather than only in an exporter layer.

### 17. The cutover is staged but has one enforced ending

The coordinated track lands in dependency order: the registry and `jackin-telemetry` contract; the `jackin-diagnostics` providers and OTLP-only runtime; protocol identity and context propagation; shared subprocess/Docker/task/TUI boundaries; crate-by-crate call-site migration and operator-output classification; then removal of legacy macros, file logging, JSONL run artifacts, command-output sidecars, Capsule and host-daemon logs, log/reveal/follow/bundle readers, obsolete environment/config keys, and compatibility tests. `RunDiagnostics` is split so the correlation id and current progress model remain in memory without any artifact writer. The product usage database remains because it is application state rather than observability, but misleading `telemetry_store` naming is changed to a usage-domain name. DHAT output and render-conformance capture remain test-harness outputs that production code cannot enable. Multiple reviewable PRs may advance the one roadmap item, but the final cutover cannot retain a second production telemetry API or exporter.

Clippy `disallowed-macros`, existing print lints, grep backstops for macro blind spots, architecture-tier checks, and a telemetry-discipline dylint prohibit the deleted logging macros, raw print diagnostics, raw project `tracing`/OTel emission, bare spawns, unregistered attributes, and malformed external spans. Developer tooling binaries (`jackin-xtask`, `jackin-dev`, `jackin-pr-trailers`, and the TUI lookbook) remain outside product telemetry scope and keep intentional terminal output. *Why:* a large migration needs incremental review, while the shipped state still needs exactly one enforced approach.

## Acceptance criteria

The item is complete only when all of the following are true:

- Product builds contain the OTel API/SDK and OTLP/gRPC exporters without an `otlp` compile-time escape hatch; endpoint presence controls runtime activation.
- With no endpoint or with `OTEL_SDK_DISABLED=true`, normal product flows and the redacted operator-output role of `--debug` work without an export runtime, worker, socket, formatted telemetry body, or telemetry artifact. Only `jackin diagnostics validate` promises delivery and fails with an actionable no-endpoint error.
- Partial per-signal endpoint configuration, non-gRPC protocol configuration, unsupported exporter/propagator/sampler selection, embedded Capsule endpoint credentials, and an over-ceiling SDK budget fail initialization before product work begins.
- With a test OTLP receiver, host and Capsule export logs, traces, and metrics independently over gRPC. Host exporter headers never enter the Capsule environment or control protocol; an authenticated Capsule path works through an operator-run credential-free Collector receiver.
- `network=none` never gains telemetry egress and product work continues with host-only coverage. Allowlist/open modes derive every normalized per-signal endpoint host, and persistent-process fingerprint mismatch is reported as degraded coverage until restart.
- No production file/stdout/stderr/TUI exporter, persistent telemetry queue, run JSONL, command-output sidecar, `multiplexer.log`, host-daemon log, log/reveal/follow reader, or telemetry debug-bundle member remains.
- Each facade event arrives once as an OTel log with top-level `EventName`, typed attributes, instrumentation scope, and current trace context, with no corresponding OTel span event.
- One immutable, schema-versioned Resource is shared by all three providers in a process; product-owned identity cannot be overridden. Run, session, attach, trace, path, branch, and container-instance ids are absent from Resources and metric dimensions.
- Launch, reattach, concurrent control requests, and later attaches to a persistent Capsule carry validated W3C context plus the correct request-scoped run/session/attach identities; invalid context starts a safe local root, and autonomous daemon work carries no stale run id.
- Operation traces and UI-interval traces remain distinct, link across semantic actions, and rotate after 30 minutes or 512 ended spans with explicit predecessor links and events.
- Failure, cancellation, detach, task abort, panic unwind, non-zero subprocess exit, and guard abandonment produce their documented distinct outcomes.
- Semantic TUI interaction tests prove equivalent keyboard/mouse actions emit the same action event, while text input, secrets, clipboard contents, raw keys, and PTY bytes never appear in exported telemetry.
- Endpoint loss, retry, queue/cardinality saturation, partial success, flush timeout, and shutdown obey the documented limits, are non-recursive, and do not block product hot paths. Wire tests prove partial success is not retried; health tests assert only typed observations the exporter actually exposes.
- Coordinated shutdown closes roots, flushes all signals, shuts down providers, and stops the runtime in order inside the 5-second total budget, including unreachable-receiver and saturated-queue cases.
- Registry tests enforce the attribute, body, string, array, link, name, and metric-series ceilings; redaction/truncation happens before provider submission and prohibited data has no DEBUG or TRACE exception.
- Weaver check/diff/live-check, generated-drift, registry fail-closed, OTLP wire capture, privacy negatives, propagation, export-volume ratchet, and performance budgets pass. The checked-in benchmark proves the no-provider fast path and keeps the active-provider launch/render/PTY regression within 5% of its reviewed baseline.
- Type-aware lint and workspace searches find no legacy telemetry macros/prefixes, local log writer/reader/command, production raw facade bypass, or spawn site without an explicit context mode.
- The render-conformance fixture recorder works from its dedicated development capture channel with no dependency on telemetry files.
- The run telemetry guide, diagnostics reference, testing guide, engineering telemetry contract, debug-bundle scope, configuration reference, deprecated-surface catalog, and roadmap overview all describe the shipped OTLP-only behavior in the landing PR.

## Out of scope

- Shipping or operating an OpenTelemetry Collector or observability backend.
- Defining a generic backend query API; OTLP exports telemetry but does not provide backend-independent queries.
- Persisting, viewing, or bundling telemetry locally.
- Profiles until the OpenTelemetry Rust profiling signal is stable and separately designed; any future production profile export is OTLP-only.
- Shipping backend-specific dashboards, alerts, or saved queries. The backend-neutral fields and joins above are the compatibility contract.
- Redesigning the content and formatting of operator-facing command output beyond consolidating it onto the product output port and classifying safe telemetry mirrors.
- Telemetry for developer-only tooling binaries.
