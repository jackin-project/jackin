**Status**: Open — design finalized with the operator, implementation-ready. Consolidates the observability-format work previously scattered as fragments across other roadmap items into one cross-crate refactor

## Problem

jackin❯ has a real OpenTelemetry substrate — `RunDiagnostics` mints a run id per CLI invocation, writes JSONL run artifacts, and (behind the `otlp` feature) exports spans, logs, and metrics with an attribute-key registry, a screen taxonomy, and a redaction boundary in `jackin-diagnostics` — but most of the codebase does not emit through it. At least five logging conventions coexist today, and prefix-tagged printf strings are still the primary instrumentation interface:

1. **Capsule macros** — `clog!`, `cdebug!`, `ctrace_payload!`, `cwarn!`, `cerror!` (defined in <RepoFile path="crates/jackin-usage/src/logging.rs" />) render timestamped lines prefixed `[jackin-capsule]` / `[jackin-capsule debug]` / `[jackin-capsule trace]` to stderr and the capsule log file, then separately bridge the already-formatted string body to OTLP via <RepoFile path="crates/jackin-usage/src/telemetry.rs" />. Roughly 300 call sites across the capsule tier.
2. **Host debug macro** — `debug_log!` (port in <RepoFile path="crates/jackin-core/src/debug_log.rs" />, sink in <RepoFile path="crates/jackin-diagnostics/src/logging.rs" />) renders `[jackin debug {category}]` lines. Roughly 190 call sites across 13 crates.
3. **Freeform `[jackin]` lines** — ad-hoc `eprintln!`/`println!` operator-facing output scattered through the CLI app, `jackin-instance`, `jackin-isolation`, `jackin-env`, `jackin-runtime`, `jackin-image`, and `jackin-host` (over 120 non-test raw `eprintln!` calls workspace-wide).
4. **Subsystem prefixes** — `[jackin-exec]` in the exec wrapper and MCP server; test-only `[jackin-core]`.
5. **Direct `tracing`** — only five `#[tracing::instrument]` sites exist in the entire workspace; span coverage is real for launch stages and console screens but absent from most subsystems, and spawned tokio tasks are not instrumented, so async work detaches from its parent span.

The consequences compound:

- **Message bodies carry the data.** Variable values are interpolated into the message string, so OTLP and log-system consumers cannot filter or aggregate without regex; the prefix duplicates what `target`/instrumentation scope and resource attributes should carry.
- **Split-brain emission.** The same event is formatted twice — a prefix-tagged line for file/stderr and a prefix-free body for OTLP — with the structured-field version losing the fields because they were already flattened into the string.
- **Uneven identity stamping.** The run id is minted correctly and exported under the Parallax-compatible `parallax.run.id` key (a deliberate ecosystem contract), but the container boundary propagates it as `JACKIN_RUN_ID` while external ingestion uses `PARALLAX_RUN_ID`; session ids and traceparent propagation exist but not every signal is stamped with them.
- **Screen/widget context is partial.** The `Screen` taxonomy in <RepoFile path="crates/jackin-diagnostics/src/screen.rs" /> covers six console screens and `jackin.screen.name` exists as a key, but the console, capsule, and launch TUIs each carry their own screen/focus/pane/dialog enums that never reach telemetry — so most events cannot answer "what screen was the operator on, which block did they activate".
- **Levels and gating are per-convention.** `JACKIN_TELEMETRY_LEVEL`, per-sink level overrides, `--debug` fallback, and category filters apply differently depending on which of the five conventions a call site happens to use.

## Why It Matters

- **The run id is the debugging backbone.** Every signal from one CLI invocation or console session must join under one identity (`parallax.run.id`): <RepoFile path="TESTING.md">the testing guide</RepoFile> mandates `--debug` runs whose JSONL artifact is the triage evidence, the diagnostics viewer and redacted debug bundle ([jackin❯ join & Debug Export](/roadmap/jackin-join-and-debug-bundle/)) package that artifact, and [Launch Progress TUI](/roadmap/launch-progress-tui/) explicitly blocks further launch parallelization on trace-backed evidence. All of that only works if every crate emits into the same run-scoped, structured stream.
- **OTLP output is currently hostile to its consumers.** A log body of `[jackin-capsule debug] adopting prewarmed dind c3f2…` defeats the point of exporting to an OpenTelemetry backend; the same information as a plain message plus the emitting module (`target`), `jackin.container.id`, and severity is queryable.
- **Every new crate re-decides.** Without one mandated facade, each subsystem picks a convention arbitrarily, and the mess grows with the workspace (30 crates today).
- **Screen-aligned observability is a product requirement.** Understanding operator behaviour — which screen, which block, which keypress, foreground vs background work — requires the observability layer to align with the model/crate structure the TUIs already have.
- **Telemetry safety work needs one choke point.** The redaction gates and export conformance assertions tracked in [Sensitive-boundary code health](/roadmap/sensitive-boundary-code-health/) are only enforceable when there is a single emission path to gate.

## Target view

The end state this item exists to reach, stated as the experience an operator gets from any OpenTelemetry platform (or from the local artifact when no collector is configured) — effectively the TUI session replayed from telemetry:

- **One run, one story.** Filtering by `parallax.run.id` returns everything a run produced — the ordered chain of screen-visit and tab-focus traces, walkable backwards from the newest trace through span links to the very first intro screen, with per-visit durations visible at a glance.
- **Every span self-locates.** Opening any span answers, from attributes alone: which screen the operator was on (`jackin.screen.name`), which crate/module emitted it (`target`, `code.*`), whether the work was jackin❯-internal or an external call (`SpanKind` plus `service.peer.name`/`server.address` for Docker, `git`, `gh`, `op`, provider HTTP), and which run/session it belongs to.
- **Interactions are legible.** Widget activations, dialog opens/confirms, and screen transitions appear as named events inside the owning screen trace — what the user pressed, selected, and clicked, in order.
- **Failures point at themselves.** A failed operation is a red span with `error.type` and a correlated exception log record; a frozen screen is a heartbeat gap plus the flushed records of the unfinished trace; an abandoned operation reads `Cancelled`, never success.
- **Logs read clean.** Plain-sentence bodies, structured attributes, correct severity — no bracket prefixes, no regex archaeology — and the identical record shape whether viewed in a backend or in the local file artifact.
- **Background work is visible but not noisy.** Scheduled refreshes and status transitions show up as short traces with their causal links; no-op ticks and PTY throughput appear only as health metrics.

## Research notes (input for the design phase)

Verified against the OpenTelemetry specification, semantic-conventions registry, and the Rust ecosystem docs as of July 2026. These notes are the raw input that informed the decisions; where a note and the Design section differ, the Design governs.

- **Stack.** The opentelemetry-rust project recommends [`tracing`](https://github.com/open-telemetry/opentelemetry-rust) as the single instrumentation API; `tracing-opentelemetry` converts spans, [`opentelemetry-appender-tracing`](https://docs.rs/opentelemetry-appender-tracing/latest/opentelemetry_appender_tracing/) bridges events to OTel log records, `opentelemetry-otlp` exports all three signals. jackin❯ already depends on this exact stack — the refactor is about making it the *only* path, not adopting new crates. Logs and metrics API/SDK are stable upstream; traces are beta.
- **Structured logging.** Message = plain human sentence (becomes the log Body); every variable becomes a typed field (becomes Attributes); `target` is the category and maps to the instrumentation scope; `tracing` levels map onto OTel severity numbers ([logs data model](https://opentelemetry.io/docs/specs/otel/logs/data-model/)). Embedding data in the message string is the documented anti-pattern.
- **One invocation = one trace.** The [CLI semantic conventions](https://opentelemetry.io/docs/specs/semconv/cli/cli-spans/) model each one-shot invocation as one root span named after the executable (interactive sessions instead use the screen-scoped topology settled in Design decision 5), with `process.exit.code` required and `process.command_args` withheld by default (secrets in argv). Resource attributes `service.name`, `service.version`, `service.instance.id` identify the emitter; the [session conventions](https://opentelemetry.io/docs/specs/semconv/general/session/) (`session.id`, `session.previous_id`) fit interactive console/capsule sessions.
- **Spans.** Names must be low-cardinality classes of operation ([trace API spec](https://opentelemetry.io/docs/specs/otel/trace/api/)); ids/paths go in attributes. OpenTelemetry [deprecated the span-events API in 2026](https://opentelemetry.io/blog/2026/deprecating-span-events/) — events should be named log records correlated via context, and running the span layer and the log bridge together double-emits unless events are filtered out of one side.
- **Async.** Spans do not cross `tokio::spawn` implicitly; futures must be attached with `Instrument::instrument`/`in_current_span`, and creation-time span links model related-but-not-parent work ([tokio guidance](https://github.com/tokio-rs/tokio/discussions/6008)).
- **Screen/widget events.** No TUI semconv exists, but the client-side [app conventions](https://opentelemetry.io/docs/specs/semconv/app/app-events/) (`app.screen.click`, `app.widget.click`, `app.screen.id`/`app.screen.name`, `app.widget.id`/`app.widget.name`) adapt directly: stamp the current screen on every event, emit widget activation as named events, and treat screen transitions as their own span or event.
- **Process hygiene.** Batch exporters must be flushed/shut down on every exit path or short-lived runs silently drop their tail; a TUI on the alternate screen must never receive a stdout `fmt` layer ([ratatui guidance](https://ratatui.rs/recipes/apps/log-with-tracing/)) — human-readable output belongs to a file sink or an in-app surface.

## Design

Settled with the operator (2026-07-15). Each decision carries a one-line why; the research grounding lives in the notes above and in the linked sources. Decisions 1–4 are the foundation (pipeline, API, output split, identity), 5–8 the topology (traces, background work, screens, boundaries), 9–12 the signal semantics, and 13–14 the governance that keeps it the only approach.

### 1. One pipeline: everything rides OpenTelemetry rails

All observability — logs, spans, metrics — flows through one `tracing`-based subscriber stack into OpenTelemetry SDK exporters. Sinks differ only in exporter, never in API or data model: when an OTLP endpoint is configured, signals export over gRPC; when it is not, the same records land in the local run artifact as an OTLP-shaped file export (the JSONL layer is refactored to serialize the OTel log/span data model rather than its own schema v2, and the `jackin diagnostics` commands read that OTLP-shaped artifact — one schema, no second reader); the console debug render is one more layer over the same stream. Three consequences are part of the decision: the OTel SDK stops being feature-optional in the product binaries (`jackin`, `jackin-capsule`) — only the network exporter activates conditionally at runtime; sink selection (endpoint → gRPC, otherwise file, force-both override retained) is purely a runtime routing choice, never a shape change; and the shared redaction layer runs as a processor stage in this one pipeline ahead of every exporter, which is what makes the [Sensitive-boundary code health](/roadmap/sensitive-boundary-code-health/) gates enforceable. *Why:* the operator requirement is one standard with no parallel formats — a file sink that shares the OTLP data model means every consumer (viewer, debug bundle, collector) reads one shape.

### 2. Single emission API: the typed facade over `tracing`

The `operation_*` facade and fail-closed registry validation in `jackin-diagnostics` become the **only** legal call-site API, extended to cover everything the legacy macros do (levels, categories, guards, metrics, screen/widget events). It stays a thin layer: `tracing` is the substrate underneath, so `#[instrument]`, `Instrument::instrument` on spawned futures, and EnvFilter all keep working. The call-site surface is macro-based, not plain functions, so `target` and `code.*` are captured at the calling crate/module rather than inside `jackin-diagnostics` — this is what makes the crate attribution in decision 9 true (the current function-based facade mis-attributes both). `clog!`, `cdebug!`, `ctrace_payload!`, `cdebug_local!`, `cwarn!`, `cerror!`, and `debug_log!` are deleted — not deprecated, deleted — and their render prefixes (`[jackin-capsule]`, `[jackin debug {category}]`, `[jackin-exec]`) disappear with them; prefixes were doing the job of `target`, resource identity, and severity. *Why:* raw `tracing` at call sites cannot enforce the registry, redaction, or outcome discipline; the facade is where fail-closed validation and the RAII outcome guard (drop without `complete` records `Cancelled`, so `?`-exits never look like success) live.

### 3. Telemetry is not operator output

Roughly 70% of today's print sites are CLI subcommand output the operator is meant to read (tables, results, instructions); those are product surface, not telemetry, and stay on stdout/stderr — but through one output port (the existing `OperatorNoticeSink` port plus the capsule output helpers, consolidated to a single API that defers cleanly when a rich TUI owns the screen). The remaining ~30% — warnings, internal errors, diagnostics that happen to print — migrate to the facade. Every operator-facing notice the output port renders is also mirrored as a named log event on the telemetry stream, so the platform view includes what the operator was told without telemetry ever touching the product surface. No telemetry ever writes to stdout, and nothing writes to a TUI-owned terminal. *Why:* conflating the two is the root cause of the current mess; separating them is what lets the telemetry side be 100% structured.

### 4. Identity: run, session, service

- Run identity keeps the Parallax format: `parallax.run.id` is the canonical run-id attribute on every signal, with external run-id ingestion via `PARALLAX_RUN_ID` and `OTEL_RESOURCE_ATTRIBUTES` preserved. This is a deliberate compatibility contract with the Parallax observability ecosystem, not a legacy leftover — jackin❯ does not mint a competing `jackin.run.id` key.
- Resource attributes on every emitter: `service.name` (`jackin` host, `jackin-capsule` in-container), `service.version`, `service.instance.id`, plus `parallax.run.id`.
- Interactive console/capsule lifetimes use the OTel session conventions: `session.id` stamped on all signals, `session.previous_id` when a session continues an earlier one. `session.id` means the interactive surface's lifetime (one console process, one capsule daemon run) — the capsule's agent PTY "sessions" are a different concept and appear as tab/pane attributes, never as `session.id`.

*Why:* the run id is the join key the operator already uses (`jackin diagnostics summary <run>`), and keeping the Parallax key means jackin❯ runs correlate natively inside Parallax-based tooling; standard resource/session attributes make any OTel backend group signals correctly with zero custom configuration.

### 5. Trace topology: screen-scoped traces, chained with span links

One-shot CLI commands keep the shipped model: one root span per invocation, named per the CLI span conventions, with `process.exit.code` recorded at close. Interactive surfaces never become one monster session trace; instead traces are scoped to how long a surface naturally lives:

- **Short-term screens — one trace per screen visit.** The console intro, the workspace list, the workspace editor, workspace create, global settings, and the launch-progress surface each get their own trace per visit: the root span is the visit, and user actions, widget events, dialogs, and triggered operations are child spans inside it.
- **Long-term surfaces — one trace per tab-focus interval.** The capsule interface can host agents for hours or days, so a per-screen trace is unrealistic there. Instead, each tab-focus interval is a trace: switching tab focus closes the current trace and starts the next one. Operator actions inside the focused tab are child spans; agent PTY output and status streams flow as logs/metrics stamped with session/tab attributes rather than spans, so trace size tracks operator interaction, not agent runtime.
- **Rotation guard.** Any visit/focus trace that exceeds a duration or span-count cap rotates: the root closes, a successor trace starts with a link back — so a screen parked overnight can never grow an unbounded trace. Cap values are deliberately not fixed in this item.
- **Chaining.** Every boundary is span-level linked: the ending trace records a transition event carrying `jackin.screen.from`/`jackin.screen.to` and the successor trace id, and the successor's root span takes a creation-time span link back to that transition span. Walking the link chain from the newest trace reconstructs the full history — which screens, in what order, how long each visit took — without any single huge trace. `parallax.run.id` and `session.id` remain the query-side join across the whole run. Host-to-capsule propagation is two-tier: the launch-time `TRACEPARENT` env var parents only capsule startup (an env var is set once and cannot carry per-action context), while per-action trace context rides the attach/control protocol frames so container-side work parents under the currently active screen trace.
- **Crash/freeze evidence.** A visit root span is open while the screen is active; if the process freezes or dies, that root is lost — but logs and metrics flush independently of span end and carry the live trace/span ids, so the evidence trail (heartbeat gap plus the already-flushed records pointing into the unfinished trace) survives. The rotation guard bounds how much span data can ever be in flight.

*Why:* this keeps the operator's requirement — reconstruct the entire journey backwards from the latest point — while respecting backend limits (Tempo's default per-trace byte cap, Jaeger's rendering collapse on huge traces) and the long-span guidance in the research notes (interval spans + span links are the documented alternatives to never-ending spans); creation-time links are the OTel-native mechanism for related-but-not-parent traces.

### 6. Background and periodic work: every task declares a context mode

jackin❯ runs a lot of work no user action directly parents — the capsule daemon's 1 Hz state tick (agent-status arbitration and publish), the usage scheduler (30 s wake, ~5 min per-account provider fetches), the git branch context poll (1 s) and HEAD inotify watcher, the PR context lookup (60 s cache, `gh` subprocess), the resource sampler, the token-spend monitor, per-session PTY reader/writer/reaper tasks, socket accept loops, the console's instance-refresh cycle (500 ms `docker ps` + capsule snapshots), and fire-and-forget launch prewarms. Today all of it logs into the void with no causality. The unified layer gives every spawned task or timer exactly one of four **context modes**, chosen at the spawn point through facade spawn helpers:

1. **inherit** — operator-triggered async work whose result lands back in the triggering interaction (dialog-confirmed exec runs, on-demand usage refresh from the usage dialog, forced PR lookup from the GitHub-context dialog, console save/load subscriptions, launch-internal parallel steps): the spawned future is instrumented into the active screen/action trace and appears as a child span.
2. **link** — work that outlives or detaches from its trigger (background image/sidecar prewarm, any fire-and-forget spawn): its own trace, with a creation-time span link back to the spawning span, so the causal chain stays walkable without holding the parent open.
3. **cycle** — timer-driven maintenance: each *meaningful* execution is its own short trace (root span named for the task, `CLIENT` children for provider HTTP, `git`/`gh`/`docker` subprocess work); a cycle that does nothing emits nothing. Sub-30-second cycles (the 1 Hz status tick, the 1 s branch poll, the 500 ms instance refresh) emit a trace only on state change, failure, or a latency threshold breach; cycle health is carried by metrics (runs/skips counters, last-success gauge), never by no-op spans.
4. **stream** — continuous event-driven I/O (PTY pumps, attach client tasks, accept loops, inotify watchers, the SIGCHLD reaper): no per-event spans ever; discrete lifecycle operations get bounded spans (session spawn/exit, attach handshake), notable occurrences are named log events (session exited, attach failure — the existing typed attach events already have this shape), and throughput/health are metrics.

Cross-process rule: work arriving over a socket that carries `TRACEPARENT` joins the caller's trace as a `SERVER`-kind span (capsule attach handshake joining the host's attach trace, host-daemon requests from a CLI invocation, the jackin-exec credential resolver); without a traceparent it starts its own trace. Reference mapping for the systems named above:

| Background system | Mode | Trace shape |
|---|---|---|
| Agent-status tick (arbitrate → publish) | cycle | trace only on a status transition, carrying evidence/winner attributes; flap/stuck stay metrics + events |
| Usage refresh (scheduled) | cycle | trace per account fetch with `CLIENT` HTTP spans per provider |
| Usage refresh (dialog-triggered) | inherit | child of the usage-dialog action span |
| Branch context poll + HEAD watcher | cycle + stream | trace only when HEAD changed or cache expired; watcher itself emits nothing |
| PR context lookup (scheduled / dialog-forced) | cycle / inherit | `CLIENT` span for the `gh` subprocess |
| Resource sampler, token-spend monitor | cycle (degenerate) | metrics only; error events on failure |
| PTY reader/writer/reaper | stream | session spawn/exit spans, output volume as metrics |
| Attach handshake + client task | stream | handshake `SERVER` span joined via `TRACEPARENT`; task lifetime is not a span |
| Console instance refresh | cycle | trace on snapshot change or failure; cadence health as metrics |
| Launch parallel steps (`tokio::join!`, cleanup) | inherit | children of the launch trace, proving cleanup ordering |
| Background prewarms | link | own trace, linked to the launch span that spawned it |
| Host daemon / exec resolver requests | stream + `SERVER` spans | one trace (or joined trace) per request |

*Why:* the parent/child/link decision becomes explicit and reviewable at every spawn point instead of accidental; high-frequency tickers stop flooding backends with no-op traces; and bare `tokio::spawn`/`spawn_blocking` outside the facade helpers becomes lintable (decision 14), which is what makes "one approach" hold for background work too.

### 7. Screen and widget visibility

The `Screen` taxonomy generalizes to all three TUI surfaces (console — including the intro — capsule tabs, launch), and every signal emitted while a screen is active carries `jackin.screen.name`. Screen and tab-focus transitions are the trace boundaries defined in decision 5. Widget/block interactions emit named events shaped after the OTel client `app.screen.*`/`app.widget.*` conventions (`jackin.widget.id`, `jackin.widget.name`), wired at the single input choke points each TUI already has: the console's screen-sync and key/mouse dispatch in its run loop, the capsule's `apply_action`, and the launch TUI's `update_launch_view`. *Why:* the choke points already exist, so full interaction coverage is a wiring change, not an architecture change — and stamping screen context on everything is how mobile/browser instrumentation delivers "what was the user looking at" without unbounded spans.

### 8. Internal vs external spans

Span kind and standard attributes make the boundary obvious in any backend:

- App-internal work: `SpanKind::INTERNAL` (default), low-cardinality span names, parameters as attributes.
- Subprocess calls (`git`, `gh`, `op`, agent CLIs): `SpanKind::CLIENT` spans named after the executable per the CLI span conventions, with `process.executable.name`, `process.exit.code`, and sanitized-by-default command args. Instrumentation moves into the shared subprocess boundary (`jackin-process`) so every caller gets it — today only `ShellRunner` wraps spans and direct callers bypass it.
- Docker Engine API: `SpanKind::CLIENT` HTTP-client-convention spans on the bollard client, `server.address` = the socket, `service.peer.name: docker-engine` (`peer.service` is deprecated upstream).

*Why:* SpanKind is what backends use to derive service graphs and entry spans, and `service.peer.name`/`server.address` are the human-visible cue for "this span left jackin❯".

### 9. Crate attribution

Crate and module identity ride the `tracing` `target`, which is crate-prefixed by default: spans carry it as the `target` attribute plus `code.*` location attributes, and the Rust log bridge already surfaces the target as the OTLP instrumentation scope name per record. No hand-stamped "component" attribute duplicates what the ecosystem provides; `jackin.component` survives only where the target genuinely cannot express it (host vs capsule vs console process identity, which resource `service.name` also covers). *Why:* free, always correct, and one less convention to enforce.

### 10. Errors, stalls, and frozen screens

- Every failure path goes through the facade's error API: span status set to error, `error.type` per the OTel error conventions, exception detail as `exception.*` attributes on a log record correlated to the failing span — so "which span failed and froze the screen" is a direct query.
- The RAII outcome guard generalizes: any operation abandoned mid-flight (early return, panic unwind, task cancellation) closes its span as `Cancelled`, never silently as success.
- Liveness gets first-class signals: render-loop/tick heartbeat metrics per TUI surface and slow-render/stall events (the TUI analog of the client `app.jank` convention), so a frozen screen shows up as a gap in the heartbeat plus the flushed records of the in-flight screen trace — from the observability platform, without a debugger. The existing flap/stuck telemetry from the agent runtime status authority migrates onto the same conventions.
- Span events are not used (deprecated upstream); everything event-shaped is a named log record, and the span layer's event conversion is disabled to prevent double emission.

*Why:* this is exactly the operator's stated debugging story — see the failed span, understand why the screen froze, from the platform.

### 11. Metrics

Metrics go through typed facade helpers backed by the OTel Meter API (not the `tracing` metrics-prefix bridge): counters, histograms (launch stage durations, render times), and gauges defined in the registry with bounded-cardinality attributes. *Why:* the Meter API gives explicit instrument types and units; registry definition keeps cardinality reviewable.

### 12. Levels and gating

`tracing` levels keep their documented semantics and map to OTel severity. The existing resolution order (env `JACKIN_TELEMETRY_LEVEL` → config → `--debug` fallback) survives, but gating becomes uniform: every sink is a layer with its own per-sink filter (the per-sink `JACKIN_TELEMETRY_CONSOLE_LEVEL`/`_FILE_LEVEL` vars exist today but are never consumed in production — that gap closes), and category filtering becomes attribute/target-based filtering on the same stream instead of a bespoke string-match path. Two more rules complete the model: sampling is always-on — volume is controlled by design (cycle gating, stream mode, level filters), not probabilistic sampling, so the local artifact and a backend see the same complete story; and the hand-maintained per-crate export allowlist is replaced by a workspace-prefix rule (all `jackin*` targets export; dependency internals stay opt-in), so a new crate can never silently drop its signals. *Why:* one filtering model, no special-cased firehose, no hand-list to forget.

### 13. The jackin❯ semantic-conventions registry, owned by Weaver

All `jackin.*` attributes, event names, span shapes, and metrics are defined once in an [OTel Weaver](https://github.com/open-telemetry/weaver) YAML registry that imports the official conventions. Weaver generates the Rust constants (replacing the hand-rolled `otel_keys` and static `EventDef` tables as source of truth — the fail-closed runtime validation stays, now driven by generated definitions), `weaver registry check` with policy rules (every custom attribute starts with `jackin.`) and `weaver registry diff` gate CI, and `weaver registry live-check` scores the conformance lane's actual emitted telemetry against the registry. The current pre-migration tolerance for unregistered free-form event names ends with the cutover: after it, an unregistered name fails validation closed instead of emitting. *Why:* "one standard, never a second format" needs the standard to be machine-checked, diffable, and code-generated — exactly what Weaver is for, and it retires a hand-rolled schema per the library-first rule in <RepoFile path="ENGINEERING.md" />.

### 14. Cutover: one refactor, enforced ending

The migration is a single coordinated track, not a long coexistence: the facade and pipeline land first, call sites migrate crate by crate within the track, and the legacy macros/prefixes are deleted at the end of the same track. Enforcement then keeps it dead: clippy `disallowed-macros` for the print/legacy macro family (with the existing `print_stdout`/`print_stderr` denies and a grep backstop for proc-macro blind spots), the queued telemetry-discipline dylint lint in `jackin-lints` for facade bypasses (raw `tracing::event!` outside `jackin-diagnostics`, CLIENT spans missing `service.peer.name`), and the exporter-backed conformance lane extended with the Weaver live-check. *Why:* the operator's constraint is explicit — one approach everywhere, no second convention surviving the refactor — and pre-release policy (<RepoFile path="PRERELEASE.md" />) permits the breaking sweep.

### Out of scope for this item

- The diagnostics viewer/UX work (stays with [Launch Progress TUI](/roadmap/launch-progress-tui/)) and the redacted debug bundle (stays with [jackin❯ join & Debug Export](/roadmap/jackin-join-and-debug-bundle/)); both consume the unified stream.
- Operator-facing CLI output redesign beyond consolidating it onto one output port (content/formatting of subcommand output is untouched).
- Shipping or operating any collector/backend; jackin❯ emits, operators choose where it lands.
- Developer tooling binaries (`jackin-xtask`, `jackin-dev`, `jackin-pr-trailers`, the TUI lookbook) keep plain terminal output — the unification scope is the product binaries (`jackin`, `jackin-capsule`) and every crate they ship.
