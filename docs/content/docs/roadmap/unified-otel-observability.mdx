---
title: "Unified OpenTelemetry observability"
---

**Status**: Open — design finalized with the operator and implementation-ready. This is an OTLP-only cross-crate refactor: product telemetry is exported to a configured OpenTelemetry endpoint and is never persisted to a jackin❯ telemetry file or rendered as a local telemetry stream

## Goal

Establish one governed, typed observability contract across every jackin❯ product crate, producing correlated OpenTelemetry logs, traces, and metrics across host and Capsule boundaries with bounded volume, explicit context propagation, and no telemetry leakage into operator output. "One contract" means one legal call-site facade and one semantic-conventions registry; internally, each signal uses its native OpenTelemetry API.

The resulting platform view must let an operator filter logs and traces by `parallax.run.id` and reconstruct the work causally owned by one invocation or console run: host and Capsule operations, the ordered chain of bounded screen/focus traces, named interaction and lifecycle events, correlated failures, and subprocess and Docker boundaries. Stable low-cardinality metrics describe product and exporter health; trace exemplars may link a measurement to a sampled operation where the SDK and backend support them. Autonomous long-lived daemon work that belongs to no invocation remains queryable by its stable process identity and must never inherit a stale run id.

## Problem

jackin❯ already has an OpenTelemetry substrate, but most of the codebase does not emit through it. `RunDiagnostics` mints a run id, the optional `otlp` feature exports all three signals, and `jackin-diagnostics` contains a registry, redaction logic, screen taxonomy, and conformance tests. Alongside that substrate, prefix-tagged printf strings, direct terminal writes, hand-rolled JSONL diagnostics, and raw `tracing` sites remain separate observability interfaces.

The raw workspace inventory captured on 2026-07-15 found 284 `debug_log!` occurrences, 284 Capsule logging-macro occurrences, 127 `eprintln!` occurrences, 99 Tokio spawn/spawn-blocking occurrences, and 29 thread-spawn occurrences. These are search baselines, not all migration sites: definitions, tests, and legitimate operator output are included where applicable. Reproduce the inventory rather than copying the counts forward:

```bash
rg -n 'debug_log!' --glob '*.rs' crates | wc -l
rg -n '\b(clog|cdebug|ctrace_payload|cwarn|cerror)!' --glob '*.rs' crates/jackin-capsule crates/jackin-usage | wc -l
rg -n 'eprintln!' --glob '*.rs' crates | wc -l
rg -n 'tokio::spawn|spawn_blocking' --glob '*.rs' crates | wc -l
rg -n 'thread::spawn|std::thread::spawn' --glob '*.rs' crates | wc -l
```

The current split creates several concrete failures:

- **Message bodies carry dimensions.** Variable values are flattened into strings, so backends need regular expressions instead of typed attribute queries.
- **Events are not real OTel events.** The current function facade writes an `event.name` attribute instead of the top-level OpenTelemetry `EventName`, and dynamic attribute slices are stamped onto the active span rather than the emitted log record.
- **Crate ownership is lost.** A function-based facade reports `jackin-diagnostics` as the `tracing` target instead of the calling crate and module.
- **Identity is process-global where it must be request-scoped.** Capsule receives a launch-time `JACKIN_RUN_ID`, but its daemon can outlive that launch and serve later attaches belonging to other host runs.
- **The facade is in the wrong dependency tier.** `jackin-diagnostics` is Tier 2 while foundational crates such as `jackin-core` and `jackin-process` are Tier 0, so a universal facade in `jackin-diagnostics` would violate the enforced dependency direction.
- **Operator output and telemetry overlap.** Some terminal writes are product output, some are diagnostics, and some do both. A rich TUI cannot safely share its terminal with a telemetry formatter.
- **Background causality is accidental.** Async tasks, blocking tasks, threads, socket handlers, timers, and stream pumps routinely detach from the triggering context or retain context longer than intended.
- **The existing local artifact is a second telemetry product.** Its custom schema, adapters, viewer, retention, and redaction behavior must evolve separately from OTLP. This item removes that parallel surface instead of redesigning it.

## Target view

- **One run, one query scope.** Every run-owned log and span carries `parallax.run.id`; filtering by it returns the complete causal story exported by host and Capsule processes. A run is a correlation scope, not necessarily one trace.
- **Stable resources, dynamic context.** Resource attributes identify the emitting process. Run, operator-session, screen, pane, and request identities live on the records and spans whose lifetimes they actually describe.
- **Bounded linked traces.** A one-shot command has one invocation root. Interactive surfaces use multiple screen-visit or tab-focus traces linked in order, with fixed rotation limits.
- **Named, typed events.** Interaction, lifecycle, transition, failure, and operator-notice mirrors are log-based events with a real top-level `EventName`, typed attributes, a redacted body, and current trace context.
- **Failures point at themselves.** Failed operations set span status and `error.type`; cancellation is distinguishable from failure; exception records correlate to the failed span; telemetry delivery failures never recurse into the same exporter.
- **Background work is useful, not noisy.** Meaningful scheduled work and state changes are observable; no-op ticks and continuous byte streams remain metrics rather than traces.
- **OTLP is the only production telemetry destination.** No product telemetry file, JSONL fallback, stdout exporter, stderr formatter, or TUI telemetry layer exists. Without a configured OTLP endpoint there is no retained telemetry.

## Scope and consequences

This item owns the cross-crate API, signal semantics, context propagation, OTLP runtime, migration, and enforcement. It supersedes the observability-format fragments in the following items:

- [Launch Progress TUI](/roadmap/launch-progress-tui/) keeps current-run application state and the correlation run id visible; historical traces and cross-run comparison remain in the operator's configured backend because OTLP is an export protocol, not a query protocol.
- [Console resource panel](/roadmap/console-resource-panel/) may render live product state or separately queried backend data; it does not read a local telemetry artifact.
- [jackin❯ join & Debug Export](/roadmap/jackin-join-and-debug-bundle/) cannot automatically package telemetry after this cutover. A debug bundle may include non-telemetry diagnostics and redacted configuration, while telemetry remains in the operator's backend.
- [Sensitive-boundary code health](/roadmap/sensitive-boundary-code-health/) supplies the negative privacy assertions and issue-fingerprint policy consumed by the unified facade and OTLP conformance lane.

The loss of offline telemetry is deliberate. Normal commands continue when no endpoint is configured by installing no-op providers. An explicit telemetry request such as `--debug` or a telemetry diagnostics command fails before doing work when no endpoint is configured, naming the missing OTLP configuration instead of claiming that evidence was captured. The existing JSONL run artifact, file-specific environment variables, local summary/viewer behavior, and telemetry inclusion in debug bundles are removed during cutover with no migration shim under the pre-release policy. Contributor and CI workflows follow the same rule: manual `--debug` validation and the OTLP conformance lanes run against a locally operated collector endpoint, and the testing guide's evidence mandate is redefined against that backend — jackin❯ itself persists nothing.

## Research basis

Verified against OpenTelemetry Rust, OTLP 1.10.0, semantic conventions 1.43.0, and OTel Weaver as of 2026-07-15.

- **Signal-native APIs.** OpenTelemetry Rust provides independent logger, tracer, and meter providers. The project facade uses the Logs API for named events, `tracing` plus `tracing-opentelemetry` for spans and context, and the Meter API for metrics. A shared contract does not require forcing all three signals through `tracing`.
- **Log-based events.** OpenTelemetry's stable [logs data model](https://opentelemetry.io/docs/specs/otel/logs/data-model/) defines `EventName` as a top-level field. OpenTelemetry [deprecated the span-events API](https://opentelemetry.io/blog/2026/deprecating-span-events/) for new instrumentation, so event-shaped data is emitted as named logs correlated to the current span.
- **OTLP transport.** The [OTLP specification](https://opentelemetry.io/docs/specs/otlp/) defines delivery, retry, partial-success, and duplicate-data behavior. jackin❯ keeps its existing OTLP/gRPC transport choice and rejects unsupported protocol configuration rather than silently changing transport.
- **Resources.** Resource attributes describe the entity producing telemetry and remain fixed for a provider lifetime. Dynamic run and attach context therefore belongs on individual records and spans, not the Capsule daemon's Resource.
- **CLI conventions.** The [CLI span conventions](https://opentelemetry.io/docs/specs/semconv/cli/cli-spans/) are Development status and apply only to short-lived programs. They require `process.executable.name`, `process.pid`, and `process.exit.code`; command arguments are withheld by default because they can contain secrets.
- **Session conventions.** The [session conventions](https://opentelemetry.io/docs/specs/semconv/general/session/) are Development and Opt-In. `session.previous_id` describes continuation on `session.start`; it is not stamped indiscriminately on every signal.
- **Context propagation.** W3C propagation carries and validates both `traceparent` and `tracestate`. Baggage is excluded from jackin❯ propagation until a separately reviewed, bounded use case exists.
- **TUI semantics.** TUI telemetry belongs at semantic message/action/effect boundaries, not raw key, mouse, text-input, clipboard, or PTY dispatch points.

## Design

Settled with the operator on 2026-07-15. These decisions resolve the call-site API, dependency ownership, signal semantics, identity lifecycles, cross-process propagation, trace topology, delivery behavior, privacy boundary, governance, cutover, and completion gates.

### 1. OTLP is the only production telemetry output

The product binaries compile the OpenTelemetry API, SDK, and OTLP/gRPC exporters as baseline dependencies. Logs, traces, and metrics have separate SDK providers and processors but one runtime configuration contract. Only OTLP/gRPC export is supported; file, stdout, stderr, console, JSONL, TUI, and persistent queue/spool exporters do not exist. Delivery buffers are bounded and memory-only.

`OTEL_EXPORTER_OTLP_ENDPOINT` supplies the base endpoint for all three signals. A standard per-signal endpoint overrides that base for its signal; when no base is set, all three per-signal endpoints must be present, so a partially configured signal set fails initialization instead of silently omitting logs, traces, or metrics. The base and per-signal protocol variables, when set, must all resolve to `grpc`; conflicts or any other value are rejected. Standard headers and per-signal header overrides are supported, but authentication header names and values are never included in telemetry or operator-facing error details. *Why:* one wire protocol and no local telemetry schema removes the split-brain format, reader, retention, and redaction problem entirely, while an explicit configuration contract prevents accidental partial observability.

When no endpoint is configured, normal product flows use no-op providers and create no telemetry files. Commands that explicitly promise telemetry, including `--debug` and telemetry inspection/validation commands, fail before doing work and explain how to configure an OTLP endpoint. Runtime unreachability after successful configuration does not fail or block the product operation; it updates bounded in-memory exporter health and may produce one non-recursive operator notice. *Why:* optional observability must not become a product availability dependency, but an explicit request must never pretend capture succeeded.

### 2. Lowest-tier facade, higher-tier runtime

A new dependency-minimal `jackin-telemetry` crate becomes the lowest product architecture tier. It owns the generated names and attributes, typed values, log/span/metric call-site macros, operation outcome guard, explicit context token, and spawn helpers. It depends on no other jackin❯ crate. Existing foundational product crates that emit telemetry are re-tiered above it, and <RepoFile path="crates/jackin-xtask/src/arch.rs" /> remains the authority that proves every production dependency points downward.

`jackin-diagnostics` remains the process-level runtime owner: it builds the logger, tracer, and meter providers; installs the span bridge and propagator; applies filters and SDK limits; manages the dedicated export runtime; exposes in-memory exporter health; and coordinates flush/shutdown. Product binaries initialize it once and lower-tier libraries only emit through `jackin-telemetry`. *Why:* every product crate needs the facade, while provider construction, terminal ownership, and process lifecycle are application concerns that cannot sit at the bottom of the graph.

### 3. One call-site contract, three native signal APIs

Named events and logs use the OpenTelemetry Logs API so `EventName`, body, severity, typed attributes, instrumentation scope, trace id, and span id are populated in their native fields. Spans use facade macros over `tracing` and `tracing-opentelemetry`; metrics use facade helpers over the OTel Meter API. The call-site macros capture `module_path!()`, file, and line. Logs and metrics carry the calling module as their native instrumentation scope; spans bridged through `tracing` share one process-level tracer scope by construction, so span ownership rides the registered `code.*`/module attributes instead — no query may rely on a span's scope naming the emitting crate. Dynamic values support the OTel scalar and array types rather than being coerced to `String`.

Raw `tracing::event!`, raw OTel logger/meter calls, and unrestricted `#[instrument]` are forbidden outside the facade/runtime implementation and generated code. Facade span macros default to `skip_all` behavior and require registered attributes, preventing automatic argument capture. Project log events are not routed through the span-event conversion path, and conformance tests prove that each facade event produces one log-based event and zero OTel span events. *Why:* a single governed surface is compatible with signal-native data only when bypasses and double emission are both closed.

### 4. Weaver owns the telemetry contract

An OTel Weaver registry imports semantic conventions 1.43.0 and defines every custom `jackin.*` attribute, event, span, metric, type, requirement level, stability marker, unit, histogram boundary, and allowed attribute set. Weaver 0.24.2 is pinned for the initial implementation; version upgrades are explicit schema changes. Project-owned Rust templates generate constants and typed definitions consumed by `jackin-telemetry`. `weaver registry check`, a reviewed `registry diff`, generated-drift detection, and `weaver registry live-check` join CI.

Weaver complements rather than replaces runtime and behavioral enforcement. Fail-closed runtime validation, prohibited-key/body checks, privacy negatives, cardinality checks, export-volume ratchets, propagation tests, in-memory SDK tests, and an OTLP wire-capture conformance lane remain required. An unregistered event, span shape, metric, or attribute is rejected before it reaches any provider. *Why:* schema correctness, privacy, causality, and operational behavior are different properties and need different proofs.

### 5. Telemetry and operator output are separate ports

Operator output stays on the product output port and may render to stdout, stderr, or a TUI-owned surface according to the command contract. Telemetry never writes to those surfaces. Operator notices are not mirrored automatically: each typed notice declares `none` or a registered safe telemetry event, with `none` as the default. A mirrored event is independently redacted and carries safe structured fields; it never reuses an already-formatted terminal string.

Exporter initialization, queue, flush, and delivery failures use a minimal self-diagnostics control path that cannot call the telemetry facade. It may update an in-memory health snapshot and surface one bounded operator notice when terminal ownership permits, but it never serializes application telemetry or writes a diagnostics file. Capsule daemon boot and fatal-exit lines emitted before providers exist or after they shut down are bounded operator output on stderr — visible through `docker logs` — classified under the output port, never a telemetry exporter. *Why:* product output must remain predictable, and the telemetry system cannot diagnose its own failed exporter by recursively invoking that exporter.

### 6. Identity has explicit lifecycles

| Identity | Lifetime | Placement |
|---|---|---|
| Process resource | One host or Capsule process/provider lifetime | `service.name`, `service.version`, 128-bit `service.instance.id`, `process.pid`, runtime/process/container attributes on Resource |
| Run | One host CLI invocation or one `jackin console` process | `parallax.run.id` on every run-owned log and span; omitted from autonomous daemon telemetry and metric dimensions |
| Operator session | One interactive console lifetime or one host-to-Capsule attach lifecycle | 128-bit `session.id` on session-owned logs and spans; `session.start` and `session.end`; `session.previous_id` only on a continuing `session.start` |
| Agent session | One Capsule PTY/pane agent lifetime | `jackin.agent.session.id` plus bounded tab/pane attributes; never overloaded into `session.id` |
| Trace context | One invocation, screen/focus interval, request, or bounded background operation | Native trace/span ids plus W3C propagation |

Resource never contains `parallax.run.id`, `session.id`, or a dynamic component label. Metrics use the stable Resource plus bounded dimensions; run, session, container name, path, branch, user-entered values, and other unbounded identifiers are prohibited as metric attributes. Trace exemplars provide correlation where the SDK and backend support them.

Host run-id precedence is `PARALLAX_RUN_ID`, then a valid `parallax.run.id` parsed from `OTEL_RESOURCE_ATTRIBUTES` for Parallax compatibility, then a locally minted id. A value parsed from resource configuration is removed from the effective Resource and applied as run context. Capsule startup may receive the launch run context, but every later attach/control request supplies its own validated context; no daemon-global `JACKIN_RUN_ID` may stamp unrelated work. Conflicting non-empty identities fail initialization or reject the request rather than being silently merged. A request carrying valid propagated context reuses the originating operator session; a new `session.id` is minted only when an interactive surface starts without supplied context, such as a standalone hardline attach. *Why:* identity follows ownership and duration, which prevents stale correlation and metric-cardinality explosions while preserving the Parallax query key.

### 7. Context crosses every jackin❯ process boundary explicitly

The attach and control protocols gain a versioned `TelemetryContext` carrying `parallax.run.id`, `session.id` when applicable, `traceparent`, and `tracestate`. Host clients inject the current context into every causally related request; Capsule and host-daemon handlers validate and extract it, create a `SERVER` span, and bind an explicit context token only for that request or spawned work. Invalid or oversized context is rejected without becoming telemetry data. Context-free autonomous work starts under process identity with no run or operator session.

Launch-time environment propagation parents Capsule startup only. Dynamic operations never rely on an environment variable that was fixed when the container started. `traceparent` and `tracestate` propagate together; baggage is not accepted or forwarded. Context is passed explicitly across concurrent tasks rather than stored in a mutable process-global slot. *Why:* the persistent Capsule and host daemon can serve multiple runs over time, so request context must be scoped like request data.

### 8. Runs contain bounded trace graphs

One-shot CLI commands use one root execution span named according to the pinned CLI conventions and record `process.executable.name`, `process.pid`, `process.exit.code`, and low-cardinality `error.type` when failed. The CLI conventions do not apply to daemon lifetimes.

Interactive surfaces use trace graphs rather than one unbounded session trace:

- Short-lived console and launch surfaces create one trace per screen visit.
- Capsule creates one trace per tab-focus interval; PTY output is never represented as spans.
- A visit/focus trace rotates after 30 minutes or 512 ended spans, whichever comes first. These initial limits are constants governed by the registry and volume tests, not user configuration.
- A successor root links directly to the predecessor root. After the successor exists, the predecessor context emits a `screen.transition` or `trace.rotated` named log carrying from/to screen names and the successor trace id. No design depends on a nonexistent "transition span."
- Each visit emits `screen.visit.started` immediately and `screen.visit.ended` on normal close. An unfinished root may never export, and batch buffers may be lost on fatal termination; the design promises bounded best-effort evidence, never crash-proof delivery.

`parallax.run.id` and `session.id` are the query joins across the graph. *Why:* bounded linked traces preserve navigation history without backend-hostile monster traces or false crash-survival guarantees.

### 9. Every concurrent task declares a context mode

Facade helpers cover `tokio::spawn`, `spawn_blocking`, `std::thread::spawn`, socket/request handlers, timers, and long-lived stream tasks. Every production spawn site declares exactly one mode:

1. **inherit** — bounded work whose result returns to the triggering interaction; it remains a child of the active operation.
2. **link** — detached work that outlives its trigger; it starts a new trace linked to the spawning span.
3. **cycle** — periodic maintenance; health is metrics, and state changes, failures, or slow completions are named logs. A trace is created only when substantive work is known before execution, such as a scheduled provider fetch, never retroactively after a no-op poll.
4. **stream** — continuous pumps, watchers, accept loops, and reapers; the stream lifetime is not a span. Bounded handshakes/lifecycle operations are spans, notable occurrences are logs, and throughput/health are metrics.

High-frequency status ticks, branch polling, instance refresh, render loops, PTY pumps, and resource sampling cannot emit per-cycle or per-item spans. Bare spawn calls outside approved facade/runtime internals fail lint. *Why:* causality and volume must be deliberate at the creation point; tail-buffering every poll merely to decide whether it was interesting is unnecessary complexity.

### 10. TUI observability records semantic actions, never raw input

Console, launch, and Capsule instrumentation attaches at their semantic message, action, and effect boundaries after keyboard and mouse input has been interpreted. Events describe actions such as screen transition, widget activation, dialog open/confirm/cancel, focus change, and operation request. They may carry registered screen/action/widget identifiers but never raw keys, typed text, selected secret values, clipboard contents, PTY bytes, terminal escape sequences, command bodies, or unbounded labels.

The registry keeps the established `jackin.screen.name` contract and adds `jackin.ui.action.name` and bounded widget identifiers. It does not claim browser/mobile `app.*` event names are stable TUI conventions; adoption of those Development conventions requires a future explicit registry migration. *Why:* semantic actions answer what the operator did without turning observability into input capture.

### 11. External boundaries use pinned semantic conventions

- Internal work uses `SpanKind::INTERNAL`, low-cardinality names, and registered parameters as attributes.
- Subprocess calls use `SpanKind::CLIENT` in the shared `jackin-process` boundary after its dependency re-tiering. Spans include executable name, child pid, exit code, and a documented low-cardinality error type. `process.command_args` and executable paths are omitted by default; a future sanitized allowlist requires separate privacy review.
- Docker Engine and provider HTTP work use the pinned client conventions with network/server attributes and a bounded custom external-system classification where the standard has no exact equivalent.
- Incoming attach, control, daemon, and credential-resolver requests use `SpanKind::SERVER` only when they are actual request-serving boundaries.

Instrumentation scope (logs and metrics) or the registered module attributes (spans, per decision 3) identify the emitting crate/module. A separate bounded `jackin.category` may classify operational domains; `target` is never overloaded as both ownership and category. *Why:* backend service graphs, ownership queries, and category filters need distinct, semantically correct fields.

### 12. Operation outcomes are result-aware

Operation wrappers close explicitly from `Result`, exit status, or a typed cancellation reason. Success leaves OTel span status unset and records `jackin.operation.outcome=success`; failure sets Error status, a stable low-cardinality `error.type`, and emits one correlated exception log; expected operator cancellation/detach records `jackin.operation.outcome=cancelled` without Error status. Guard drop means abandoned work, not automatically user cancellation. Panic unwind records failure where Rust can observe it; process abort and `SIGKILL` make no flush or final-span guarantee.

Warnings never imply success or failure without an explicit outcome. Exception attributes and stack details pass the same redaction and size policies as every other event. *Why:* `?`, user cancellation, task abort, panic, and non-zero exit are different outcomes and must remain distinguishable in a backend.

### 13. Metrics have explicit instruments and cardinality budgets

Every metric registry entry declares instrument kind, unit, description, aggregation, temporality expectation, histogram boundaries where applicable, and its complete allowed dimension set. Counters cover operations, failures, bytes, transitions, queue drops, and cycles; histograms cover launch stages, render time, request latency, and exporter flush latency; observable gauges cover liveness and bounded resource/runtime state.

Metrics never contain run/session ids, paths, branch names, container names, user input, event bodies, or error messages. The interactive export interval starts at 30 seconds and every normal product shutdown force-flushes metrics before provider shutdown. Short-lived commands flush all three providers explicitly. *Why:* metrics remain aggregatable and affordable only when dimensions and lifecycle behavior are part of the contract.

### 14. Filtering, sampling, and SDK limits are signal-specific

There is one OTLP destination but each signal keeps the policy native to it:

- Logs use severity and target/category filtering. INFO is the default product threshold; DEBUG activates only through the existing resolved telemetry-level contract after endpoint validation; TRACE remains a separately gated trusted diagnostic tier.
- Traces use parent-based always-on sampling for jackin❯ roots. Volume is controlled by bounded topology and task modes rather than probabilistic loss of the support story.
- Metrics use registered instruments and views; log severity does not disable required health metrics.
- All `jackin*` instrumentation scopes export by default; dependency scopes are off unless explicitly allowlisted to prevent exporter/network recursion and dependency noise.

Initial SDK budgets are explicit implementation constants: 4,096 queued log records, 2,048 queued spans, batches no larger than 512 records, a 5-second export timeout, a 30-second metric interval, and a 5-second coordinated shutdown budget. Queue exhaustion drops telemetry rather than blocking product hot paths and increments in-memory/exportable self-health counters. Budget changes require conformance-volume and overhead evidence. *Why:* levels, sampling, and metric views solve different problems and cannot truthfully be one generic filter.

### 15. OTLP failure behavior is bounded and visible

Invalid endpoint, protocol, TLS, or header configuration fails initialization when the operator explicitly configured telemetry. Once initialized, exporter retries follow the OTLP exporter implementation and remain bounded by SDK queues and timeouts; product work does not wait for retry exhaustion. Duplicate delivery is accepted as an OTLP property. Partial success, rejected records, queue drops, last successful export time, and flush timeout are reflected in a process-local `TelemetryHealth` snapshot and exported as self-observability metrics whenever the endpoint is healthy.

`jackin diagnostics status` becomes a live configuration/health command rather than a local-run reader: it reports whether OTLP is configured, which signals are active, sanitized endpoint identity, protocol, queue limits, and health available in the current process/control connection. The health snapshot is never persisted, does not promise post-exit history, and does not query the backend. A TUI defers the one operator failure notice to its owned notice surface; telemetry never writes directly to the terminal. *Why:* OTLP-only delivery necessarily gives the backend ownership of retention and query, while jackin❯ still needs a non-recursive way to expose current exporter health.

### 16. Redaction happens before every provider

The facade accepts typed safe values and validates/redacts bodies and attributes before invoking a logger, tracer, or meter. Provider processors enforce the prohibited-key and size policies again as defense in depth. Authentication headers, credentials, environment values, command arguments, PTY/input payloads, clipboard data, terminal escape content, full paths, and unbounded error/debug dumps are deny-by-default. Attribute values, bodies, arrays, exception text, and link counts have registry-owned caps.

Redaction failure rejects the signal and updates self-health; it never falls back to a raw formatted string. Endpoint and TLS errors sanitize authorities and never render headers or embedded credentials. *Why:* data that entered a provider can already reach an exporter, so the primary privacy boundary must be before provider submission rather than only in an exporter layer.

### 17. The cutover is staged but has one enforced ending

The coordinated track lands in dependency order: the registry and `jackin-telemetry` contract; the `jackin-diagnostics` providers and OTLP-only runtime; protocol identity and context propagation; shared subprocess/Docker/task/TUI boundaries; crate-by-crate call-site migration and operator-output classification; then removal of legacy macros, file logging, JSONL run artifacts, adapters, local readers, obsolete environment/config keys, and compatibility tests. Multiple reviewable PRs may advance the one roadmap item, but the final cutover cannot retain a second production telemetry API or exporter.

Clippy `disallowed-macros`, existing print lints, grep backstops for macro blind spots, architecture-tier checks, and a telemetry-discipline dylint prohibit the deleted logging macros, raw print diagnostics, raw project `tracing`/OTel emission, bare spawns, unregistered attributes, and malformed external spans. Developer tooling binaries (`jackin-xtask`, `jackin-dev`, `jackin-pr-trailers`, and the TUI lookbook) remain outside product telemetry scope and keep intentional terminal output. *Why:* a large migration needs incremental review, while the shipped state still needs exactly one enforced approach.

## Acceptance criteria

The item is complete only when all of the following are true:

- Product builds contain the OTel API/SDK and OTLP/gRPC exporters without an `otlp` compile-time escape hatch; endpoint presence controls runtime activation.
- With no endpoint, normal product flows succeed with no-op providers and no telemetry files; explicit `--debug` and telemetry validation flows fail early with an actionable configuration error. Partial per-signal endpoint configuration and non-gRPC protocol configuration also fail early.
- With a test OTLP collector, host and Capsule export logs, traces, and metrics over gRPC, and no production file/stdout/stderr/TUI exporter or persistent telemetry queue is installed.
- Each facade event arrives once as an OTel log with top-level `EventName`, typed attributes, instrumentation scope, and current trace context, with no corresponding OTel span event.
- Resources contain only stable process/entity attributes. Run and operator-session ids are absent from Resources and metric dimensions.
- Launch, reattach, concurrent control requests, and later attaches to a persistent Capsule carry the correct request-scoped run/session/trace context; autonomous daemon work carries no stale run id.
- One-shot commands, screen visits, focus intervals, transitions, and rotations match the documented root/link topology and fixed limits.
- Failure, cancellation, detach, task abort, panic unwind, non-zero subprocess exit, and guard abandonment produce their documented distinct outcomes.
- Semantic TUI interaction tests prove equivalent keyboard/mouse actions emit the same action event, while text input, secrets, clipboard contents, raw keys, and PTY bytes never appear in exported telemetry.
- Endpoint loss, partial success, queue saturation, retry, flush timeout, and shutdown are bounded, non-recursive, and do not block product hot paths; health and drop accounting remain observable when delivery recovers.
- Weaver check/diff/live-check, generated-drift, registry fail-closed, OTLP wire capture, privacy negatives, propagation, export-volume ratchet, and performance budgets pass.
- Workspace searches find no legacy telemetry macros/prefixes, local telemetry artifact writer/reader, production raw facade bypass, or unapproved spawn site.
- The run telemetry guide, diagnostics reference, testing guide, engineering telemetry contract, debug-bundle scope, configuration reference, deprecated-surface catalog, and roadmap overview all describe the shipped OTLP-only behavior in the landing PR.

## Out of scope

- Shipping or operating an OpenTelemetry Collector or observability backend.
- Defining a generic backend query API; OTLP exports telemetry but does not provide backend-independent queries.
- Persisting, viewing, or bundling telemetry locally.
- Redesigning the content and formatting of operator-facing command output beyond consolidating it onto the product output port and classifying safe telemetry mirrors.
- Telemetry for developer-only tooling binaries.
