title: "Unified OpenTelemetry observability"

**Status**: Open — application design finalized and implementation-ready. This is an OTLP-only refactor of jackin❯ product code. It does not deploy or configure a Collector, gateway, backend, dashboard, or storage system

## Goal

Refactor jackin❯ into a convention-led Rust application that emits correlated OpenTelemetry logs, traces, and metrics directly to the OTLP endpoint supplied by Parallax. The application must use the standard Rust `tracing` and OpenTelemetry integrations, standard semantic conventions wherever they apply, bounded in-memory SDK pipelines, explicit async and cross-process context propagation, and no local telemetry files or terminal telemetry stream.

The governing model is:

> A long-running CLI invocation is a correlation domain, not one enormous trace.

A two-second command and a week-long console process use the same model. Stable process, invocation, session, and job identifiers join many bounded traces and named log events. Metrics describe aggregate behavior. No process-lifetime, invocation-lifetime, session-lifetime, screen-lifetime, daemon-lifetime, or stream-lifetime span exists.

The resulting Parallax view must support:

- an invocation timeline across all participating host and Capsule processes;
- one readable waterfall for a user action, command, synchronous daemon request, connection attempt, job attempt, or meaningful background cycle;
- session and screen history reconstructed from named events rather than a giant trace;
- navigation, action, render, job, connection, and process health through low-cardinality metrics;
- direct parentage for synchronous work and links for detached, delayed, retried, fan-out, or independently owned work;
- log-to-trace and trace-to-log navigation using native trace and span IDs.

OpenTelemetry is the application instrumentation and transport layer. Parallax owns telemetry storage, search, visualization, retention, and backend policy.

## Scope

This item changes jackin❯ application code only:

- product-crate instrumentation APIs and call sites;
- Resource, identity, event, span, metric, and error contracts;
- Rust subscriber/provider initialization and shutdown;
- OTLP/gRPC export to the configured Parallax endpoint;
- host, Capsule, daemon, subprocess, Docker, TUI, async-task, and control-protocol propagation;
- removal of every jackin❯-owned production log/telemetry file and reader;
- conformance, privacy, propagation, volume, performance, and lifecycle tests.

It does **not** add Collector configuration, an agent/gateway topology, backend queries, dashboards, alerts, tail-sampling rules, storage queues, or Parallax infrastructure. jackin❯ neither ships nor operates an OpenTelemetry Collector.

## Current conflicts

jackin❯ already has an optional OpenTelemetry substrate, but the application does not have one coherent instrumentation model. The workspace combines `debug_log!`, Capsule prefix macros, raw `tracing`, terminal writes, diagnostics JSONL, command-output sidecars, `multiplexer.log`, a host-daemon log, local log commands/readers, and a partial OTLP bridge. These surfaces disagree about identity, redaction, retention, event names, crate ownership, and failure behavior.

The inventory captured on 2026-07-15 found 284 `debug_log!` occurrences, 284 Capsule logging-macro occurrences, 127 `eprintln!` occurrences, 99 Tokio spawn/spawn-blocking occurrences, and 29 thread-spawn occurrences. The counts include definitions, tests, and legitimate operator output, so implementation must classify the sites rather than replace them mechanically:

```bash
rg -n 'debug_log!' --glob '*.rs' crates | wc -l
rg -n '\b(clog|cdebug|ctrace_payload|cwarn|cerror)!' --glob '*.rs' crates/jackin-capsule crates/jackin-usage | wc -l
rg -n 'eprintln!' --glob '*.rs' crates | wc -l
rg -n 'tokio::spawn|spawn_blocking' --glob '*.rs' crates | wc -l
rg -n 'thread::spawn|std::thread::spawn' --glob '*.rs' crates | wc -l
```

The current roadmap direction also contained conflicts that this design resolves:

- A run, session, or screen was treated as a trace even though those lifetimes can span hours or days and cross unrelated operations.
- Screen visits were modeled as long spans even though a visible screen mostly represents state or idle time.
- A custom dispatch and direct Logs API were proposed where the maintained Rust logs bridge already maps `tracing` event names to native OpenTelemetry `EventName`.
- Product-specific `jackin.*` attributes duplicated concepts already covered by OpenTelemetry semantic conventions.
- Direct OTLP export and a local Collector were both described as the target. This item now owns direct application export only.
- OTLP-only behavior was claimed while JSONL, sidecars, Capsule/daemon logs, log commands, and reveal/follow readers remained.
- `--debug` was allowed to become an export requirement even though it also controls redacted operator-facing troubleshooting output.
- Exact partial-success, queue-drop, and retry counters were promised even where the Rust exporter exposes no stable typed callback.

## Semantic convention policy

OpenTelemetry semantic conventions 1.43.0 are the naming authority. The implementation imports and pins them in one schema module and uses their standard names, types, units, requirement levels, and well-known values. There is no `jackin.*` attribute, event, span, or metric prefix.

Use the standard fields first:

| Purpose | Contract |
|---|---|
| Process/service identity | `service.name`, `service.namespace`, `service.version`, `service.instance.id`, `process.pid`, `process.executable.name`, and applicable `container.*`, `os.*`, and runtime attributes |
| Interactive session | `session.id`, `session.previous_id`, `session.start`, and `session.end` |
| UI context | `app.screen.id`, `app.screen.name`, `app.widget.id`, and `app.widget.name` |
| AI agent context | `gen_ai.agent.name`/`gen_ai.agent.id` for the hosted coding agent, `gen_ai.conversation.id` for one agent session/thread, and `gen_ai.provider.name` (well-known values include `anthropic` and `openai`) on provider calls |
| Failure | span Status plus stable low-cardinality `error.type`; `exception.type`, `exception.message`, and `exception.stacktrace` only for a real redacted exception |
| Source ownership | standard `code.*` attributes and the `tracing` target/module metadata |
| Network/RPC/process work | the applicable RPC, HTTP, network, server, process, and CLI conventions |
| Causality | native trace/span IDs, W3C `traceparent`/`tracestate`, parent-child relationships, and span links |

Where OpenTelemetry 1.43 has no applicable convention, the schema may define a small neutral extension without a product/vendor prefix. The initial gaps are `cli.invocation.id`, `cli.command.name`, `job.id`, `job.type`, `ui.action.name`, `ui.screen.visit.id`, and `ui.navigation.sequence`. These retain a generic semantic-domain namespace because unqualified keys collide, but never add a jackin❯ or other product prefix. The screen and widget attributes remain the standard `app.*` fields. Each listed gap is verified against 1.43: no convention defines a CLI invocation id (`cicd.pipeline.run.id` is CI/CD-scoped and `faas.invocation_id` serverless-scoped), no screen entered/exited, navigation-sequence, or dwell convention exists, and no general-purpose outcome attribute exists — outcome *values* nevertheless mirror the CICD result enum (`success`, `failure`, `error`, `timeout`, `skip`, `cancellation`), the one place the industry defines cancellation as a first-class result. New extensions require proof that no current semantic convention fits, a bounded type/value set, privacy review, and a migration note for adopting a later standard.

`cli.invocation.id` is the application's correlation key from the first migrated release, with no transition period and no compatibility minting of the old key. `parallax.run.id` is not an application-defined replacement for `cli.invocation.id`: if Parallax supplies it, jackin❯ validates and preserves it as external backend correlation metadata, and otherwise it simply does not appear. jackin❯ never mints it, overloads it as Resource identity, or copies it into metric dimensions. The same rule applies to every attribute: an invented key is fully replaced by the applicable convention in the same cutover — no invented key survives alongside its standard equivalent.

Development-status conventions — Session, App, CLI, and the GenAI registry (maintained in its own `semantic-conventions-genai` repository since 2026, where `gen_ai.system` is already deprecated in favor of `gen_ai.provider.name`) — are isolated behind the schema module. An OpenTelemetry version update changes their mapping centrally instead of leaving old names throughout the codebase. OTel Weaver validates imported conventions and the small extension registry; generated Rust constants prevent stringly typed keys and drift.

## jackin❯ attribute contract by case

The registry binds each real jackin❯ surface to explicit fields. Values shown are the initial seed observed in the codebase on 2026-07-15; the generated registry owns the authoritative enums, and only cases that exist in jackin❯ belong in it:

| jackin❯ case | Attribute contract |
|---|---|
| Every process | Resource: `service.namespace=jackin`; `service.name` = `jackin`, `jackin-daemon`, or `jackin-capsule`; `service.version`; opaque `service.instance.id` per process; `process.pid`, `process.executable.name`; Capsule adds stable `container.id` when known |
| CLI command | Root span per the CLI convention; opaque `cli.invocation.id`; `cli.command.name` ∈ {`load`, `console`, `hardline`, `status`, `doctor`, `eject`, `prune`, `workspace`, `role`, `config`, `diagnostics`, …}; `process.exit.code`; `error.type` on failure |
| Console/launch screens | `app.screen.name` ∈ {`intro`, `list`, `settings`, `editor`, `create`, `launch`, `capsule`}; `app.screen.id` for the registry screen type; opaque `ui.screen.visit.id`; `ui.navigation.sequence` |
| Capsule tabs/panes | Same screen fields with `app.screen.name=capsule`; a tab-focus interval is a screen visit; pane/tab control types ride `app.widget.id` |
| Widget interaction | `app.widget.id` (registry control id such as `workspace-row`, `mount-editor`, `auth-tab-generate`, `dialog-confirm`) and `app.widget.name`; `ui.action.name` for the semantic command (`launch_role`, `open_settings`, `save_workspace`, `switch_tab`, `confirm_exit`, …) |
| Hosted agent session | `gen_ai.agent.name` ∈ {`claude-code`, `codex`, `amp`, `kimi`, `opencode`, `grok`}; opaque `gen_ai.conversation.id` per PTY agent lifetime |
| Provider calls | HTTP client conventions plus `gen_ai.provider.name` (`anthropic`, `openai`, and documented custom values for Z.AI, MiniMax, Moonshot, xAI); also the bounded provider dimension on usage-refresh metrics |
| Subprocess | CLI caller-convention `CLIENT` span named for the executable ∈ {`git`, `gh`, `op`, `docker`, `caffeinate`, agent binaries}; `process.executable.name`, `process.exit.code`, `error.type`; arguments and paths omitted by default |
| Docker Engine API | HTTP client conventions over the Unix socket: `http.request.method`, low-cardinality route in the span name, `server.address` for the socket, `network.transport=unix`, target `container.id` |
| Attach/control protocol | RPC conventions: `rpc.system=jackin` (documented custom value), `rpc.method` ∈ {`attach`, `displace`, `status`, `exec_resolve`, …}, `CLIENT`/`SERVER` spans, `network.transport=unix` |
| Launch pipeline | One operation trace per launch; child spans named per stage ∈ {`identity`, `role`, `credentials`, `construct`, `agent_binaries`, `derived_image`, `workspace`, `network`, `sidecar`, `capsule`, `hardline`}; the stage enum is the launch-duration metric dimension |
| Background work | Opaque durable `job.id`; `job.type` ∈ {`image_prewarm`, `sidecar_prewarm`, `usage_refresh`, `pr_context`, `branch_context`, `instance_refresh`, `status_tick`, …}; `PRODUCER`/`CONSUMER` spans for queued hand-offs |
| Failure classification | `error.type` from the bounded registry set (`timeout`, `permission_denied`, `connection_refused`, `protocol_decode`, `resource_exhausted`, `invalid_configuration`, `docker_unavailable`, `auth_failed`, …) |

Workspace names, role identifiers, repository names and paths, container names, and branch names are user data — never attribute values; bounded kind enums classify them where classification is needed.

## Research basis

Verified against OpenTelemetry Rust 0.32, `tracing-opentelemetry` 0.33, OpenTelemetry semantic conventions 1.43.0, and the OTLP specification as of 2026-07-15:

- The [OpenTelemetry Rust project](https://github.com/open-telemetry/opentelemetry-rust) recommends `tracing` for new application instrumentation. `opentelemetry-appender-tracing` maps a `tracing` event's metadata name to native log `EventName`, while `tracing-opentelemetry` maps spans and, unless filtered, events to OTel spans/span events.
- A bridged tracing span is finalized when it closes, so an invocation-, session-, or screen-lifetime span delays completion and conflicts with bounded span data. Bounded operation spans are the unit of tracing; history is a correlated log/event view.
- The [Session conventions](https://opentelemetry.io/docs/specs/semconv/general/session/) define a session as a collection of logs, events, and spans carrying `session.id`, with `session.start`/`session.end`; they do not define a session root span.
- The [App conventions](https://opentelemetry.io/docs/specs/semconv/app/app-events/) provide `app.screen.*` and `app.widget.*` fields and click events, but no screen entered/exited, dwell, navigation-sequence, or transport-neutral semantic-action contract. Those exact gaps justify the neutral `ui.*` extensions above.
- The [CLI conventions](https://opentelemetry.io/docs/specs/semconv/cli/cli-spans/) apply to short-lived commands and define `INTERNAL` callee and `CLIENT` caller spans with process attributes. They do not apply a process-lifetime root to an interactive console or daemon.
- The [OTLP specification](https://opentelemetry.io/docs/specs/otlp/) defines retry, partial-success, and duplicate-delivery behavior. Partial success is not retryable, and telemetry transport cannot be treated as an audit or crash-durability channel.
- Rust async context uses `#[instrument(skip_all)]`, `Future::instrument`, `or_current()`, or synchronous `in_scope`; a span/context enter guard must not live across `.await`, and an extracted remote parent must be applied before the receiving span starts.

## Rust instrumentation architecture

### 1. Use the maintained Rust stack

The initial implementation pins the compatible family:

```toml
opentelemetry = "0.32"
opentelemetry_sdk = "0.32"
opentelemetry-otlp = "0.32"
opentelemetry-appender-tracing = "0.32"
opentelemetry-semantic-conventions = "0.32"
tracing-opentelemetry = "0.33"
tracing = "0.1"
tracing-subscriber = "0.3"
```

Product modules instrument spans and events through `tracing`. The OpenTelemetry API is used directly only for metric instruments, W3C context, links, and explicit status/attributes not represented safely by `tracing`. Only process composition roots depend on the SDK, OTLP exporter, logs bridge, tracing bridge, and subscriber construction.

A new lowest-tier `jackin-telemetry` crate contains the convention registry, generated constants, thin macros/helpers that expand to `tracing`, typed metric handles, operation outcome guard, context carrier, and async spawn helpers. It is not a second logging framework and has no provider dispatch of its own. `jackin-diagnostics` remains the composition root that builds and owns providers, layers, exporter runtime, health, flush, and shutdown.

`#[tracing::instrument]` is permitted only with `skip_all`, a registered static span name, and explicit approved fields. Dynamic values never enter span names. Automatic argument capture, raw project OTel logger calls, raw metric construction outside the registry, unrestricted raw `tracing` calls, and application formatter layers are prohibited by lint.

### 2. Route events once

`OpenTelemetryTracingBridge` converts governed `tracing` events to OpenTelemetry log records. The `tracing` metadata name becomes native `EventName`; `message` becomes the optional body; level becomes native severity; fields remain typed attributes; and active trace context supplies TraceId and SpanId.

The `tracing-opentelemetry` layer converts governed spans to OpenTelemetry spans. Per-layer filters accept all registered spans but reject the target class reserved for log-based events, while the logs bridge accepts that event class. Conformance tests prove one event produces one OTel log and zero OTel span events. Error-event-to-status and error-record-to-exception inference are disabled; the operation owner sets status and exception fields explicitly.

The logs bridge stays unconfigured for experimental span-attribute copying. Important log events carry their searchable invocation, session, screen, and job correlation explicitly, so behavior does not depend on experimental enrichment or ambient thread-local state.

### 3. Keep operator output separate

Operator output remains a separate typed product port that may own stdout, stderr, or a TUI surface. Telemetry never renders there. A notice may have an independently defined safe telemetry event, but an already formatted terminal string is never mirrored automatically.

`--debug` continues to enable redacted operator-facing troubleshooting detail. When OTLP is active it also enables governed DEBUG telemetry; it never requires an endpoint. TRACE remains semantic and privacy constrained: it cannot contain raw keys, text input, prompts, clipboard data, PTY bytes, terminal frames, command arguments, environment values, credentials, or file contents.

## Telemetry model

### Resource and correlation lifetimes

| Concept | Lifetime and representation |
|---|---|
| Process instance | One random UUID per OS process start in `service.instance.id`; never reused after restart |
| Service | `service.namespace=jackin`; `service.name=jackin`, `jackin-daemon`, or `jackin-capsule`; package version in `service.version` |
| CLI invocation | One random UUID in `cli.invocation.id` for every command or console process; stable for the complete process, even if it lives for a week |
| Interactive session | One random UUID in `session.id` from TUI/attach terminal ownership until detach or process exit; reattach emits a new `session.start` with `session.previous_id`, after the prior `session.end`; no idle-time rotation in the initial implementation |
| Background job | Durable opaque `job.id` plus bounded `job.type`; attempts may use separate traces and process instances |
| Screen visit | Opaque `ui.screen.visit.id` plus monotonic per-session `ui.navigation.sequence`; events and metrics, not a span |
| Hosted agent session | One coding-agent lifetime in a Capsule pane: `gen_ai.agent.name` plus `gen_ai.conversation.id`; never overloaded into `session.id` |
| External Parallax run | Validated incoming `parallax.run.id` when supplied; record correlation only, never Resource or metric identity |

All providers in one process receive clones of one immutable Resource value. Invocation, session, job, screen, and Parallax run identifiers belong on relevant spans/logs, never the Resource. Noninteractive one-shot commands omit `session.id`. A persistent daemon has no stale invocation/session on autonomous work and applies request correlation only inside that request.

### Bounded trace boundaries

One trace represents one independently understandable operation with one owner and one outcome:

| Activity | Trace design |
|---|---|
| One-shot CLI command | One `INTERNAL` execution root following the CLI convention, with low-cardinality command classification and final exit code |
| Interactive startup/shutdown | Separate `app.startup` and `app.shutdown` roots; no process-lifetime parent |
| User action | Root `ui.action` with `ui.action.name` and standard `app.screen.*`/`app.widget.*` context |
| Synchronous daemon/control call | `CLIENT` span under the current operation and a remote `SERVER` span in the same trace |
| Delayed/detached job | `PRODUCER` scheduling span; a separate `CONSUMER` attempt trace linked to the producer and joined by `job.id` |
| Meaningful periodic work | One root `background.cycle` per substantive cycle; no-op timer ticks are metrics only |
| Connection/reconnect | One `connection.attempt` trace per attempt; retry scheduling is a WARN event, not an open parent |
| Long-lived stream/watcher | No lifetime span; bounded open/handshake/control/close operations are spans, state changes are logs, throughput/liveness are metrics |
| Subprocess | `CLIENT` span following the CLI caller convention; executable name, pid, exit code, and `error.type`, with args/path omitted by default |

Provider quota and API calls carry `gen_ai.provider.name` with its well-known values, so per-provider behavior uses the industry key rather than an invented provider label. Direct execution ancestry uses parent-child spans. Delayed, independently retryable, fan-out, batch, or multiply caused work starts a new root with links. A durable job that lasts for days is divided into scheduling and bounded attempt/lease/checkpoint traces; `job.id` supplies continuity. No trace is kept open merely to preserve history.

Successful spans keep Status Unset and omit `error.type`. A terminal operation failure sets Error status and one stable `error.type`; only the owning operation emits the ERROR log. Expected cancellation emits a named cancellation log and leaves status unset. Deadline expiry or dependency-driven cancellation is an error. Guard abandonment is an instrumentation error, not user cancellation.

### Screens and TUI actions

Screens are application state, not operations. Entering a screen emits `ui.screen.entered`; leaving emits `ui.screen.exited`; both use a registry-bounded screen type in standard `app.screen.id`/`app.screen.name`, the per-visit id, session id, invocation id, and navigation sequence. The exit event records a bounded reason, and `ui.screen.dwell.duration` records dwell seconds by `app.screen.name`. Stable registry IDs identify screen/widget types; visit IDs never replace them. Focus changes use the same event-plus-duration approach and never create hour-long focus spans.

An action is instrumented only after raw keyboard, mouse, paste, or protocol input has been interpreted. Equivalent keyboard and mouse paths produce the same `ui.action` trace. Standard `app.widget.id` and `app.widget.name` identify the control; `ui.action.name` identifies the semantic command. Raw key values, coordinates, typed text, selections, prompts, clipboard contents, PTY data, and terminal escape sequences are not telemetry.

A screen transition caused by an action is a child `ui.screen.transition` span inside that action trace and is accompanied by exited/entered events. Rendering caused directly by a bounded action may be a child `ui.render` span. Continuous render loops and idle refreshes are metric-only unless an individual slow/failing render crosses a reviewed diagnostic threshold.

The current OpenTelemetry App conventions define screen/widget fields and click events for client applications, but not screen entered/exited, navigation sequence, or a transport-neutral semantic action. The neutral `ui.*` gaps above are therefore registry extensions, not renamed copies of an existing standard. If OpenTelemetry adds suitable conventions, the schema module migrates to them.

### Logs and events

Logs answer what changed, what degraded, what finally failed, and which invocation/session/job/trace was involved. Every product log has a registered static event name and structured fields; free-form diagnostic logging is not a production API.

| Level | Contract |
|---|---|
| ERROR | The owning operation finally failed, data may be lost, or operator action is required |
| WARN | Degraded but handled behavior, retry scheduling, fallback, partial result, or temporary disconnect |
| INFO | Lifecycle and semantic state changes: startup/shutdown, session, navigation, connection, and job transitions |
| DEBUG | Bounded diagnostic decisions and state summaries |
| TRACE | Rare semantic internals under an explicit gate; never raw payload capture |

Lower layers return typed errors and enrich their spans; they do not repeat the final ERROR at every stack layer. Human-readable messages may be redacted log bodies, but grouping and metrics use stable `error.type`, never an error message. Exception records follow the exceptions-on-logs conventions: stable `exception.type`/`exception.message`/`exception.stacktrace` fields, operation-specific failure events using a `.exception`-suffixed event name, and the bare `exception` name reserved for a global last-resort handler.

### Metrics

Metrics use the OpenTelemetry Meter API, registered views, cumulative temporality, explicit units/descriptions, and fixed histogram boundaries. Initial application instruments cover:

- invocation and action count/duration/failure;
- screen transition count, dwell duration, action duration, and render duration;
- job scheduled/started/completed/active, attempt duration, and queue depth;
- connection attempts/reconnects and duration;
- daemon request count/active/duration;
- launch-stage duration and cache/reuse decisions;
- process uptime, CPU, memory, and bounded runtime state;
- facade validation, redaction, and cardinality rejection.

Metric names use the neutral domain (`cli.*`, `ui.*`, `job.*`, `connection.*`, `daemon.*`, `process.*`, or applicable standard instrument names), never a jackin❯ prefix. Metric attributes are bounded enums such as command, screen, action, job type, operation, reason, `gen_ai.provider.name`, and `error.type`. They never contain invocation/session/Parallax run/trace/span/job/visit IDs, paths, branches, repository/container names, raw URLs, commands, error messages, or user input.

Each metric stream allows at most 256 active attribute sets per process. A new set over the limit is rejected and counted rather than evicting an arbitrary live series. Observable callbacks read atomics or cheap snapshots only; they do not perform I/O, block on async locks, scan files, or enter a Tokio runtime. The initial export interval is 30 seconds, and graceful shutdown force-flushes metrics.

## Async and distributed context

W3C `traceparent` and `tracestate` are the only trace propagation format. Baggage is disabled until a separately reviewed use case exists. Attach, control, daemon, resolver, and queued-job envelopes carry a versioned string map plus the required invocation/session/job correlation fields.

The caller creates the `CLIENT` or `PRODUCER` span, injects its OpenTelemetry context, and sends the envelope. The receiver extracts before constructing the `SERVER` or `CONSUMER` span and sets the parent or link before that span starts. Malformed W3C data is ignored without echoing it and starts a safe local root; malformed product correlation IDs reject the request. A remote unsampled decision is honored by the parent-based sampler.

Never hold `Span::enter()`, `.entered()`, or an OpenTelemetry `ContextGuard` across `.await`. Async functions use `#[instrument(skip_all)]`, `Future::instrument`, or synchronous `in_scope` sections. Joined tasks use a child span instrumented with `or_current()`; detached tasks create a new root and link. Facade helpers cover Tokio spawn, handles, `JoinSet`, blocking/local tasks, standard/scoped threads, TUI subscriptions, socket handlers, timers, and stream tasks. Type-aware lint requires every production spawn to declare joined, detached, cycle, or stream ownership.

## OTLP runtime contract

All product binaries compile the three OpenTelemetry APIs/SDKs and OTLP/gRPC exporters as baseline dependencies. `opentelemetry-otlp` disables default features and enables only traces, logs, metrics, tonic gRPC, gzip, project-standard TLS roots/provider, and the maintained gRPC retry feature. HTTP, file, stdout, stderr, JSONL, TUI, and persistent spool exporters are absent. The exporter `internal-logs` feature is disabled because current internal debug records can render headers and do not form a typed health API.

`OTEL_EXPORTER_OTLP_ENDPOINT` is the normal Parallax endpoint. Standard per-signal endpoints override it; all three signals must resolve when telemetry is enabled. Only `grpc` is accepted. Standard timeout, compression, TLS, header, Resource, and service-name configuration is validated before providers start. The initial sampler is fixed to `ParentBased(AlwaysOn)`: local roots are recorded and a valid unsampled remote parent remains unsampled. A conflicting `OTEL_TRACES_SAMPLER` fails initialization; probabilistic application sampling is a later evidence-led policy, while any Parallax-side sampling remains outside this item. Programmatic safety ceilings win over larger environment values. `OTEL_SDK_DISABLED=true` disables all three providers.

Host, daemon, and Capsule processes export independently and directly to their resolved endpoint. jackin❯ does not relay or proxy OTLP. Exporter credentials are resolved per trust boundary: host headers/client keys are never copied automatically into an agent-visible Capsule environment. Capsule export requires endpoint/auth material explicitly classified as Capsule-safe by the Parallax launch environment; otherwise product work continues with host-only telemetry and live coverage health reports the gap. `network=none` always prevents Capsule telemetry egress and never gets weakened for observability.

With no endpoint or with the SDK disabled, normal commands use no-op instrumentation and create no exporter runtime, worker, socket, or telemetry artifact. Enabled checks occur before body formatting and dynamic field allocation. `--debug` still works as operator output. `jackin diagnostics validate` is the only command that promises an export attempt: it emits one marked log, trace, and metric, force-flushes, and fails when configuration or delivery cannot be confirmed.

Exporter/provider configuration is immutable for a process lifetime. Persistent daemons report a sanitized configuration fingerprint and typed current health over their control protocol; later clients do not replace a subscriber/provider in place. Health includes only typed observations exposed by jackin-owned validation or stable SDK/exporter hooks: active signals, outer export attempt/success/failure, facade rejection, flush, and shutdown. It does not parse exporter text or promise exact SDK queue-drop, retry-attempt, partial-success, or backend-rejection counts when no typed hook exists.

The SDK queues are bounded and memory-only: 4,096 log records, 2,048 spans, batches no larger than 512, a 1-second schedule delay, a 5-second total export-attempt timeout including at most three retries, a 30-second metric interval, one dedicated single-worker export runtime, and a 5-second coordinated shutdown budget. Saturation drops telemetry rather than blocking product work. Retry handles only retryable transport/throttle failures; authentication/configuration failures and OTLP partial success are not retried.

Graceful shutdown has one owner: stop accepting actions/requests; cancel and time-bound background work; emit final session/job/process events; end remaining operation spans; force-flush all signals; shut down tracer, logger, then meter providers while the runtime is alive; stop the runtime. Abort, OOM, `SIGKILL`, and power loss have no durability guarantee. No local log is retained as a fallback.

## Privacy and limits

Instrumentation is allowlist-first. Passwords, API tokens, authorization headers, cookies, keys, environment values, raw command lines, clipboard contents, keystrokes, typed text, prompts, PTY/terminal content, file contents, full paths, raw URLs, repository/user names, and arbitrary debug representations are prohibited by default at every level.

The initial ceilings are 32 attributes per log, 64 attributes and 8 links per span, 32 attributes per metric point, 4 KiB for a redacted body or exception message, 1 KiB for one string attribute, 32 elements for one array, and 128 bytes for an event/span/instrument name. Safe text is redacted before UTF-8 truncation. Invalid names, keys, enums, counts, or prohibited values reject the signal and update typed health; no raw fallback exists.

Processors validate the allowlist again before export. Endpoint/TLS errors render sanitized authorities only and never headers, embedded credentials, or client-key paths. Parallax-side redaction may add defense in depth, but application correctness cannot depend on backend cleanup.

## Cutover

The implementation lands in dependency order:

1. Pin the Rust/semantic-convention family and add the generated convention registry plus thin `tracing`/metrics/context helpers.
2. Build the three providers, two subscriber bridges with non-duplicating filters, Resource, direct OTLP runtime, health, validation command, and ordered shutdown.
3. Separate `RunDiagnostics` into in-memory invocation/progress state and remove JSONL ownership. Mint `cli.invocation.id` independently of optional incoming `parallax.run.id`.
4. Version protocol propagation for invocation/session/job identity and W3C context; add synchronous parent and detached link conformance tests.
5. Instrument the shared subprocess, Docker, RPC/control, job, connection, async-task, and TUI boundaries before migrating leaf call sites.
6. Replace screen/focus spans with entered/exited/focus events, dwell metrics, navigation sequence, and action-root traces.
7. Classify every raw print as operator output, error output, protocol output, or telemetry; migrate only telemetry and leave legitimate product output typed and separate.
8. Remove legacy macros, raw bypasses, run JSONL, command-output sidecars, `multiplexer.log`, host-daemon logs, `jackin logs`, `jackin daemon logs`, reveal/follow/bundle readers, file-specific telemetry configuration, and compatibility tests.
9. Rename product usage state currently called `telemetry_store`; retain genuine application state and test-only PTY/DHAT fixture output outside production telemetry.
10. Enable architecture, lint, Weaver, generated-drift, privacy, wire, propagation, cardinality, volume, soak, and performance gates as migration completion authority.

Multiple reviewable PRs may advance this item, but the final shipped state has one application instrumentation path and one production destination: OTLP to the configured Parallax endpoint.

## Acceptance criteria

- Product code uses governed `tracing` spans/events, explicit OpenTelemetry metrics/context, and composition-root SDK setup; no custom provider dispatch or raw emission bypass remains.
- Workspace and wire tests find no `jackin.*` attribute/event/span/metric namespace. Standard semantic conventions are used where available; neutral extension keys are limited to the documented gaps. Incoming `parallax.run.id` is the only vendor key and is never minted by jackin❯.
- A one-shot command, week-long console simulation, daemon lifetime, screen visit, focus interval, job, and stream never create a lifetime trace. Each meaningful action/request/attempt/cycle has a bounded trace and searchable correlation fields.
- Synchronous CLI-to-daemon/control calls use one trace with `CLIENT`/`SERVER` parentage. Detached jobs use `PRODUCER`/`CONSUMER` traces with a valid link and shared `job.id`.
- Each registered event arrives once as an OTel log with native `EventName`, severity, typed fields, Resource, and current TraceId/SpanId, and does not also arrive as an OTel span event.
- Screen history uses `ui.screen.entered`/`ui.screen.exited`, standard `app.screen.*`, visit id, navigation sequence, and dwell metrics. Equivalent keyboard/mouse actions produce the same `ui.action` trace; raw input and PTY content never appear.
- Hosted-agent and provider telemetry uses the GenAI conventions (`gen_ai.agent.name`, `gen_ai.conversation.id`, `gen_ai.provider.name`); no invented agent or provider keys remain.
- Resource values are stable for one process and cloned across all providers. Invocation/session/job/screen/Parallax IDs are absent from Resource and metric dimensions.
- Error ownership, span status, `error.type`, cancellation, deadline, abandonment, and exception behavior match the documented contract without duplicate ERROR records.
- With a test OTLP receiver representing Parallax, host, daemon, and a Capsule-safe configuration export all three signals directly over gRPC. No Collector config or OTLP relay is added to jackin❯.
- With no endpoint, SDK disabled, `network=none`, endpoint loss, partial success, queue saturation, or slow export, product work remains bounded and usable; `--debug` remains operator output and only `jackin diagnostics validate` promises delivery.
- Workspace searches find no production telemetry/log file writer or reader, JSONL run artifact, command-output sidecar, Capsule/daemon log, local log command, stdout/stderr/TUI telemetry layer, or persistent telemetry spool.
- W3C propagation, disabled-child `or_current()`, link, malformed-context, concurrent-session, persistent-daemon, graceful-shutdown, privacy-negative, cardinality, and accelerated soak tests pass.
- A checked-in benchmark proves the disabled fast path avoids formatting/allocation and keeps active telemetry launch/render/PTY overhead within 5% of the reviewed baseline.
- Operator and contributor documentation describes direct OTLP-to-Parallax behavior, backend-owned history, in-memory current-run UI state, and the absence of local telemetry files.

## Out of scope

- Deploying or configuring an OpenTelemetry Collector, gateway, agent, sidecar, backend, dashboard, alert, or saved query.
- Parallax ingestion, authentication, storage, retention, sampling, redaction, routing, or visualization changes.
- A backend-neutral query/viewer API inside jackin❯.
- Local telemetry persistence, crash log, debug-bundle telemetry, audit trail, or disk-backed queue.
- Profiles until the OpenTelemetry Rust profiling signal is stable and separately designed; any future production profile export remains OTLP-only.
- Instrumentation of developer-only binaries such as `jackin-xtask`, `jackin-dev`, `jackin-pr-trailers`, and the TUI lookbook.
