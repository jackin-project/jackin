title: "Unified OpenTelemetry observability"
---

**Status**: Open - application design finalized and implementation-ready

## Problem

jackin❯ has an optional OpenTelemetry substrate, but it does not yet have one coherent application-observability model. Product code currently combines `debug_log!`, Capsule prefix macros, raw `tracing`, terminal writes, diagnostics JSONL, command-output sidecars, `multiplexer.log`, a host-daemon log, local log commands/readers, and a partial OTLP bridge. These paths disagree about identity, redaction, event names, retention, crate ownership, async propagation, and failure behavior.

The current instrumentation also models some long-lived state as tracing work. A CLI invocation, interactive session, screen visit, daemon lifetime, Capsule attachment, or PTY stream can last for hours or days, but it is not one operation with one outcome. Keeping any of those lifetimes open as a span delays export, consumes bounded span storage, makes sampling impractical, and produces a waterfall that describes history rather than causality.

The roadmap previously left contradictory implementation choices unresolved:

- direct OTLP export and a local Collector were both described as the target;
- OTLP-only behavior was claimed while jackin❯-owned JSONL, sidecars, Capsule/daemon logs, and log readers remained;
- product-specific `jackin.*` attributes duplicated concepts already covered by OpenTelemetry semantic conventions;
- custom event dispatch and direct Logs API use were proposed even though the maintained Rust bridge already converts `tracing` events into OpenTelemetry logs;
- screen visits were treated as root traces and Capsule tab focus as a new screen, although screens, widgets, actions, and focus have different meanings;
- periodic refresh cycles, joined async work, detached prewarm work, socket streams, and durable jobs were grouped under one background-job model;
- `--debug` operator output was conflated with remote telemetry delivery;
- exporter retry, partial-success, and queue counters were promised even where the Rust SDK has no stable typed callback.

The workspace inventory captured on 2026-07-15 found 284 `debug_log!` occurrences, 284 Capsule logging-macro occurrences, 127 `eprintln!` occurrences, 99 Tokio spawn/spawn-blocking occurrences, and 29 thread-spawn occurrences. Those counts include definitions, tests, and legitimate operator output, so implementation must classify each site rather than replace calls mechanically.

## Why It Matters

jackin❯ is a multi-process Rust CLI/TUI. A single operator action can cross the host console, host daemon, Capsule daemon, a Unix-socket protocol, Docker Engine, provider HTTP APIs, subprocesses, detached prewarm work, and one or more PTY-hosted coding agents. Without one identity and causality contract, Parallax cannot reliably answer which action triggered a failure, which process handled it, whether a retry recovered, or which later job was caused by an earlier action.

The governing model is:

> A long-running CLI invocation is a correlation domain, not one enormous trace.

A two-second command and a week-long console process use the same model. Process, invocation, session, screen-visit, agent-conversation, and job identifiers correlate many bounded operations. Named log events provide chronological history. Metrics describe aggregate health and performance. No process-lifetime, invocation-lifetime, session-lifetime, screen-lifetime, daemon-lifetime, focus-lifetime, or stream-lifetime span exists.

The resulting Parallax experience must provide:

- an invocation and session timeline across the participating host, daemon, and Capsule processes;
- one readable waterfall for a user action, command, synchronous request, connection attempt, detached job attempt, or meaningful background cycle;
- screen and widget-focus history reconstructed from named events and duration metrics;
- action, render, launch, job, connection, agent-state, and process health through low-cardinality metrics;
- direct parentage for synchronous work and span links for detached, delayed, retried, fan-out, or independently owned work;
- native log-to-trace and trace-to-log navigation through TraceId and SpanId;
- bounded product memory and no dependency on a local telemetry file when Parallax is unavailable.

OpenTelemetry is the jackin❯ instrumentation and transport layer. Parallax owns ingestion, storage, search, visualization, retention, and backend policy.

## Design

### Scope and completion boundary

This item refactors jackin❯ application code only:

- product-crate instrumentation APIs and call sites;
- Resource, correlation, event, span, metric, and error contracts;
- Rust subscriber/provider initialization, exporter runtime, flush, and shutdown;
- direct OTLP/gRPC export to the Parallax endpoint supplied at runtime;
- host, daemon, Capsule, subprocess, Docker, TUI, async-task, and protocol context propagation;
- removal of every jackin❯-owned production telemetry/log file and its reader;
- conformance, privacy, propagation, cardinality, volume, lifecycle, soak, and performance verification.

It does not deploy or configure an OpenTelemetry Collector, gateway, agent, sidecar, backend, dashboard, alert, saved query, storage queue, or Parallax service. No Collector configuration belongs in the repository for this item.

### Standards and research basis

The design is verified against OpenTelemetry Rust 0.32, `tracing-opentelemetry` 0.33, OpenTelemetry semantic conventions 1.43.0, and the OTLP specification as of 2026-07-15:

- The [OpenTelemetry Rust project](https://github.com/open-telemetry/opentelemetry-rust) recommends `tracing` for new application instrumentation. `opentelemetry-appender-tracing` maps a `tracing` event's metadata name to native log `EventName`; `tracing-opentelemetry` maps spans and, unless filtered, events to OTel spans/span events.
- A bridged tracing span is finalized when it closes, so process-, invocation-, session-, screen-, focus-, and stream-lifetime spans have poor visibility and conflict with bounded span data.
- The [Session conventions](https://opentelemetry.io/docs/specs/semconv/general/session/) define a session as a collection of logs, events, and spans carrying `session.id`, with `session.start` and `session.end`; they do not define a session root span.
- The [App conventions](https://opentelemetry.io/docs/specs/semconv/app/app-events/) provide `app.screen.*` and `app.widget.*` fields and UI interaction events, but no screen entered/exited, visit, navigation-sequence, dwell, or transport-neutral semantic-action contract.
- The [CLI conventions](https://opentelemetry.io/docs/specs/semconv/cli/cli-spans/) apply to command execution and caller operations; they do not require an interactive console or daemon to keep a process-lifetime root span.
- Semantic conventions 1.43 use `rpc.system.name` and a fully qualified `rpc.method`. The old `rpc.system` name is replaced, and the removed `rpc.service` is incorporated into `rpc.method`.
- The [OTLP specification](https://opentelemetry.io/docs/specs/otlp/) defines retry, partial-success, and duplicate-delivery behavior. Partial success is not retryable, and asynchronous OTLP is not an audit or crash-durability channel.
- Rust async context uses `#[instrument(skip_all)]`, `Future::instrument`, `or_current()`, or synchronous `in_scope`; a span/context enter guard must never live across `.await`.

### Semantic convention policy

OpenTelemetry semantic conventions 1.43.0 are the naming authority. One schema module imports and pins standard fields, types, units, requirement levels, and well-known values. OTel Weaver validates imported conventions and the extension registry; generated Rust constants prevent string drift. Product code contains no `jackin.*` attribute, event, span, or metric namespace.

Use standard fields whenever they match the meaning:

| Purpose | Standard contract |
|---|---|
| Process/service identity | `service.name`, `service.namespace`, `service.version`, `service.instance.id`, `process.pid`, `process.executable.name`, and applicable `container.*`, `os.*`, and runtime fields |
| Interactive session | `session.id`, `session.previous_id`, `session.start`, `session.end` |
| UI context | `app.screen.id`, `app.screen.name`, `app.widget.id`, `app.widget.name` |
| Coding agent/provider | `gen_ai.agent.name`, `gen_ai.conversation.id`, `gen_ai.provider.name`, and applicable GenAI usage instruments |
| Failure | span Status plus low-cardinality `error.type`; `exception.*` only for a real redacted exception |
| Source ownership | applicable `code.*` fields and `tracing` target/module metadata |
| Network/integration | applicable RPC, HTTP, network, server, database, process, container, and CLI conventions |
| Causality | native TraceId/SpanId, W3C `traceparent`/`tracestate`, parent-child relationships, and span links |

Where 1.43 has no applicable convention, the schema defines only the following neutral semantic-domain extensions. None carries a project or vendor prefix:

| Extension | jackin❯ use and bounded values |
|---|---|
| `app.mode` | Resource value `one_shot`, `interactive`, `daemon`, or `capsule` |
| `cli.invocation.id` | Opaque UUID minted once for every top-level CLI launch; never a metric dimension |
| `cli.command.name` | Registry-backed command path; values are listed in the CLI case below |
| `ui.action.name` | Completed semantic action after input interpretation |
| `ui.screen.visit.id` | Opaque UUID for one screen visit; never a metric dimension |
| `ui.navigation.sequence` | Monotonic per-session integer on screen lifecycle events |
| `ui.transition.reason` | `action`, `launch`, `attach`, `detach`, `back`, `cancel`, `completion`, `failure`, or `shutdown` |
| `job.id`, `job.type` | Opaque detached-job id and `image_prewarm` or `sidecar_prewarm`; `job.id` is never a metric dimension |
| `outcome` | `success` for the intended result; `failure` for a completed negative domain result such as invalid configuration or rejected trust; `error` for an execution/infrastructure fault; `timeout`; `skip`; or operator/system `cancellation` |
| `launch.stage.name` | One of the eleven `LaunchStage` values listed below |
| `launch.target.kind` | `workspace` or `directory` |
| `background.cycle.name` | `branch_context`, `pr_context`, `usage_account`, `provider_probe`, `instance_refresh`, or `agent_status` |
| `connection.peer.type` | `host_daemon`, `capsule_control`, `capsule_attach`, `docker`, `provider`, or `parallax` |
| `agent.state` | `working`, `blocked`, `done`, `idle`, or `unknown` |
| `agent.status.source` | `none`, `visible_screen`, `shell_integration`, `foreground_process`, or `reported` |
| `agent.status.confidence` | `unknown`, `weak`, `strong`, or `authoritative` |
| `agent.status.stuck` | Boolean derived by the status authority, never from raw text in the telemetry layer |
| `auth.mode`, `credential.source.type` | Auth mode `sync`, `api_key`, `oauth_token`, or `ignore`; source type `environment`, `agent_home`, `onepassword`, `github_cli`, `oauth_store`, or `none`; never a credential name/value |
| `workspace.isolation.mode` | `shared`, `worktree`, or `clone` |
| `network.mode`, `dind.mode` | `none`, `allowlist`, or `open`; and `none`, `rootless`, or `privileged` |
| `config.scope`, `config.operation` | `global` or `workspace`; and `load`, `validate`, `migrate`, or `save` |
| `config.schema.version.from`, `config.schema.version.to`, `config.migration.step_count` | `legacy` or `v1alpha1` through `v1alpha9` for global config; `legacy` or `v1alpha1` through `v1alpha8` for workspace config; integer migration count; no config values or paths |
| `trust.decision`, `trust.source.type` | `granted`, `revoked`, or `rejected`; and `builtin` or `external` |
| `cache.name`, `cache.result` | Registry cache class and `hit`, `miss`, `stale`, `reuse`, or `bypass` |
| `pty.exit.reason` | `clean`, `signal`, `nonzero_exit`, `wait_failed`, or `cancelled` |
| `stream.direction` | `input` or `output` on PTY/terminal byte instruments; absent on directionless frame/cursor/mouse instruments |
| `telemetry.signal` | `log`, `trace`, or `metric` on facade/export health only |
| `telemetry.rejection.reason` | Bounded facade rejection reason such as `unknown_name`, `unknown_attribute`, `invalid_value`, `privacy`, `cardinality`, or `size_limit` |

These are closed registries, not permission to create arbitrary generic keys. A new extension requires proof that no current semantic convention fits, a bounded type/value set, privacy review, and a migration rule for adopting a later standard. Development-status Session, App, CLI, and GenAI conventions stay isolated behind the schema module for the same reason.

The legacy `parallax.run.id` and every `jackin.*` telemetry key are removed. `cli.invocation.id` is the application correlation key. The Parallax destination does not change the semantic schema, and jackin❯ does not mint, preserve, or depend on backend-specific attributes in product code.

### Attribute contract for actual jackin❯ cases

The schema maps only product cases present in the workspace. Registry values below are the implementation seed from the 2026-07-15 codebase scan.

#### Identity and correlation

| jackin❯ case | Required attributes |
|---|---|
| Product process | Resource `service.namespace=jackin`; `service.name` is `jackin`, `jackin-role`, `jackin-daemon`, or `jackin-capsule`; package `service.version`; random `service.instance.id` per process start; `process.pid`; `process.executable.name`; `app.mode`; applicable `container.id` in Capsule |

| jackin❯ case | Required attributes |
|---|---|
| Host screen visit | `app.screen.id` is `workspace.list`, `workspace.editor`, `settings`, `workspace.create`, `launch.progress`, or `capsule`; stable `app.screen.name`; `ui.screen.visit.id`; `ui.navigation.sequence`; `session.id`; `cli.invocation.id` |
| Editor/settings navigation | The screen remains `workspace.editor` or `settings`; `app.widget.id` identifies the bounded tab (`general`, `mounts`, `roles`, `secrets_environments`, `auth`, or `trust` as applicable); tab focus emits widget-focus events and duration, not a new screen visit |

Raw keys, mouse coordinates, typed text, selections, prompts, clipboard data, file contents, paths, PTY bytes, terminal cells, and escape sequences never become telemetry. Equivalent keyboard and mouse inputs must resolve to the same semantic action.

Workspace names, role identifiers, repositories, branches, container names, tab labels, account identifiers, and paths are user data, not attributes. Instrumenting a case not listed above requires extending this contract first.

### Rust instrumentation architecture

The implementation pins the compatible maintained family:

```toml
opentelemetry = "0.32"
opentelemetry_sdk = "0.32"
opentelemetry-otlp = "0.32"
opentelemetry-appender-tracing = "0.32"
opentelemetry-semantic-conventions = "0.32"
tracing-opentelemetry = "0.33"
tracing = "0.1"
tracing-subscriber = "0.3"
```

Product modules emit spans and events through `tracing`. Direct OpenTelemetry API use is limited to metric instruments, W3C context, links, and explicit span status/attributes that cannot be represented safely through `tracing`. Only binary composition roots depend on the SDK, OTLP exporter, logs bridge, tracing bridge, and subscriber construction.

A new lowest-tier `jackin-telemetry` crate owns the schema, generated constants, thin helpers that expand to `tracing`, typed metric handles, an operation outcome guard, the propagation carrier, and async spawn helpers. It is not another logging framework and does not dispatch to providers. `jackin-diagnostics` remains the composition root that owns providers, layers, the exporter runtime, typed health, flush, and shutdown.

`#[tracing::instrument]` is allowed only with `skip_all`, a registered static span name, and explicit allowlisted fields. Dynamic values never enter span/event/instrument names. Lint rejects automatic argument capture, raw product OTel logger calls, unregistered metric construction, unrestricted raw `tracing` calls, and application formatter layers.

`OpenTelemetryTracingBridge` converts governed `tracing` events to OpenTelemetry log records. The metadata name becomes native `EventName`, `message` becomes an optional body, level becomes native severity, attributes stay typed, and the active context supplies TraceId/SpanId. The `tracing-opentelemetry` layer converts registered spans only. Per-layer target filters ensure one event becomes one OTel log and zero duplicate span events. Automatic error-event status and exception inference are disabled; the operation owner records outcome, error status, and exception fields explicitly.

Operator output remains a separate typed product port that may own stdout, stderr, or a TUI surface. Telemetry never renders there, and a formatted terminal string is never mirrored automatically. `--debug` continues to enable redacted operator troubleshooting without requiring an endpoint; when OTLP is enabled it may also enable governed DEBUG telemetry. TRACE remains structured and subject to the same privacy contract.

### Resource and correlation lifetimes

| Concept | Lifetime and representation |
|---|---|
| Process instance | Random `service.instance.id` per OS process start, never reused |
| CLI invocation | Random `cli.invocation.id` for the top-level process lifetime, including a week-long console |
| Interactive session | Random `session.id` from TUI/attach ownership until detach or process exit; reattach starts a new session |
| Screen visit | Random `ui.screen.visit.id` plus per-session navigation sequence; events and metrics, not a span |
| Coding-agent conversation | `gen_ai.conversation.id` for one PTY-hosted agent lifetime, distinct from the operator session |
| Detached job | Durable `job.id` for image or sidecar prewarm; each attempt may be a separate linked trace |

All providers in a process receive clones of one immutable Resource. Invocation, session, screen-visit, conversation, and job identifiers belong only on relevant spans/logs, never the Resource or metric dimensions. A persistent daemon applies request correlation inside that request and does not retain stale invocation/session context for autonomous cycles.

### Bounded traces, logs, and events

One trace represents one independently understandable operation with one owner and outcome:

| Activity | Trace design |
|---|---|
| One-shot CLI command | One `cli.command` execution root with command classification and final exit code |
| Interactive startup/shutdown | Separate `app.startup` and `app.shutdown` roots |
| User action | Root `ui.action` with semantic action and current screen/widget context |
| Synchronous daemon/control call | `CLIENT` child plus remote `SERVER` span in the same trace |
| Image/sidecar prewarm | `PRODUCER` scheduling span plus separate linked `CONSUMER` attempt trace |
| Periodic work | One `background.cycle` per substantive cycle; no-op tick is metric-only |
| Connection/reconnect | One `connection.attempt` trace per attempt; retry scheduling is a WARN log |
| Long-lived stream/watcher | No lifetime span; bounded open/handshake/control/close work is traced, state is logged, throughput/liveness are metrics |
| Subprocess | `process.command` `CLIENT` span with process result and no argv/path capture |

Successful spans keep Status Unset and omit `error.type`. Terminal failure sets Error status and one stable `error.type`; only the operation owner emits the ERROR log. Recovered degradation is WARN. Expected cancellation uses `outcome=cancellation` and leaves status unset. Deadline expiry and dependency-driven cancellation are errors. Guard abandonment is an instrumentation fault.

Every log has a registered static event name and typed fields. INFO records semantic lifecycle/state changes, WARN records handled degradation, ERROR belongs to the final failing operation, DEBUG records bounded diagnostic decisions, and TRACE records rare semantic internals behind an explicit gate. Lower layers return typed errors and enrich spans rather than repeating ERROR at every level. Redacted human messages may be log bodies, but grouping and metrics use `error.type`, never an error message.

Screens are state, not operations. Entering emits `ui.screen.entered`; leaving emits `ui.screen.exited`; focus changes emit `ui.widget.focused`/`ui.widget.unfocused`. Screen exit and widget unfocus record a bounded reason, and duration histograms record dwell/focus time. A transition caused by an action is a child `ui.screen.transition` and is accompanied by lifecycle events. There is no `ui.screen.visit` span.

### Metrics

Metrics use the OpenTelemetry Meter API, registered views, cumulative temporality, explicit units/descriptions, and fixed histogram boundaries. Initial instruments are limited to actual jackin❯ behavior:

| Metric family | Bounded dimensions |
|---|---|
| CLI invocation count/duration/failure | `cli.command.name`, `outcome`, `error.type` |
| UI transition/action/dwell/focus/render | `app.screen.id`, bounded widget/action/reason, `outcome`, `error.type` |
| Launch stage duration/cache reuse | `launch.stage.name`, `launch.target.kind`, `cache.name`, `cache.result`, `outcome` |
| Detached prewarm count/active/duration | `job.type`, `outcome`, `error.type` |
| Background cycle count/duration | `background.cycle.name`, `outcome`, `error.type` |
| Connection and daemon/RPC count/active/duration | `connection.peer.type` or `rpc.method`, `outcome`, `error.type` |
| Agent-state transitions/stuck/flap | `gen_ai.agent.name`, `agent.state`, bounded source/confidence |
| PTY/terminal/render throughput | separate byte, frame, cursor, mouse, render-duration, and painted-cell instruments; `stream.direction` only where applicable; never content or identity |
| Process/runtime health | standard process uptime/CPU/memory plus established Tokio runtime instruments |
| Telemetry facade health | `telemetry.rejection.reason`, `telemetry.signal` |

Metric attributes never contain invocation/session/trace/span/job/visit/conversation IDs, workspace/role/repository/container/account identity, paths, branches, raw URLs, command lines, error messages, model strings, or user input. Each metric stream permits at most 256 active attribute sets per process. A new set above the limit is rejected and counted rather than evicting an arbitrary live series. Observable callbacks read atomics or cheap snapshots only; they perform no I/O, async locking, filesystem scan, or runtime entry. The initial export interval is 30 seconds, and graceful shutdown force-flushes metrics.

### Async and cross-process context

W3C `traceparent` and `tracestate` are the only trace propagation format. Baggage is disabled. Attach, control, daemon, and detached-job envelopes carry a versioned string map plus only the required invocation/session/job correlation fields.

The caller creates the `CLIENT` or `PRODUCER` span, injects its context, and sends the envelope. The receiver extracts before constructing the `SERVER` or `CONSUMER` span and applies the parent or link before that span starts. Malformed W3C data is ignored without echo and starts a local root; malformed product correlation IDs reject the request. A valid remote unsampled decision is honored.

No `Span::enter()`, `.entered()`, or OpenTelemetry `ContextGuard` may cross `.await`. Async functions use `#[instrument(skip_all)]`, `Future::instrument`, or synchronous `in_scope`. Joined tasks use an instrumented child with `or_current()`; detached work creates a root with a link; periodic cycles and streams declare their ownership explicitly. Helpers cover Tokio spawn/handles/`JoinSet`, blocking/local tasks, threads, TUI subscriptions, socket handlers, timers, and streams. Architecture lint requires each production spawn to declare joined, detached, cycle, or stream ownership.

### Direct OTLP runtime contract

All product binaries compile traces, logs, and metrics plus the OTLP/gRPC exporters as baseline dependencies. `opentelemetry-otlp` disables default features and enables only the three signals, `grpc-tonic`, `gzip-tonic`, project-standard TLS roots/provider, and the pinned `experimental-grpc-retry` feature. Retry construction remains isolated in `jackin-diagnostics` so a later stable API can replace the experimental surface without changing product instrumentation. HTTP, file, stdout, stderr, JSONL, TUI, and persistent-spool exporters are absent. Exporter `internal-logs` is disabled.

`OTEL_EXPORTER_OTLP_ENDPOINT` is the normal Parallax endpoint; standard per-signal endpoints override it, and all three signals must resolve when telemetry is enabled. Only `grpc` is accepted. Standard timeout, compression, TLS, header, Resource, and service-name configuration is validated before providers start. The initial sampler is `ParentBased(AlwaysOn)`: local roots are recorded and valid unsampled remote parents remain unsampled. A conflicting `OTEL_TRACES_SAMPLER` fails initialization. `OTEL_SDK_DISABLED=true` disables all three providers.

Host, daemon, and Capsule processes export independently and directly. jackin❯ does not relay or proxy OTLP. Host headers/client keys are never copied automatically into an agent-visible Capsule environment. Capsule export requires endpoint/auth material classified as Capsule-safe by the Parallax launch environment; otherwise product work continues with host-only telemetry and typed health reports the coverage gap. `network.mode=none` always prevents Capsule telemetry egress and is never weakened for observability.

With no endpoint or with the SDK disabled, normal commands use no-op instrumentation and create no exporter runtime, worker, socket, or telemetry artifact. Enabled checks occur before body formatting and dynamic-field allocation. `jackin diagnostics validate` is the only command that promises a delivery check: it emits one marked log, trace, and metric, force-flushes, and fails when configuration or delivery cannot be confirmed.

Provider configuration is immutable for the process lifetime. Persistent daemons expose a sanitized configuration fingerprint and typed current health over the control protocol; clients do not replace the subscriber/provider. Health includes only observations owned by jackin❯ or exposed by stable typed hooks: active signals, outer export attempt/success/failure, facade rejection, flush, and shutdown. It does not parse exporter text or promise exact SDK queue-drop, retry-attempt, partial-success, or backend-rejection counts.

SDK queues are bounded and memory-only: 4,096 log records, 2,048 spans, batches no larger than 512, 1-second schedule delay, 5-second total export-attempt timeout with at most three retries, 30-second metric interval, one dedicated single-worker export runtime, and a 5-second coordinated shutdown budget. Saturation drops telemetry instead of blocking product work. Retry applies only to retryable transport/throttle failures; authentication/configuration failures and OTLP partial success are not retried.

Graceful shutdown has one owner: stop accepting work; cancel and time-bound background work; emit final session/job/process events; end remaining operation spans; force-flush all signals; shut down tracer, logger, then meter providers while the runtime is alive; stop the runtime. Abort, OOM, `SIGKILL`, and power loss have no durability guarantee. No local file is retained as fallback.

### Privacy and limits

Instrumentation is allowlist-first. Passwords, tokens, authorization headers, cookies, keys, environment values, raw command lines, clipboard contents, keystrokes, typed text, prompts, PTY/terminal content, file contents, full paths, raw URLs, repositories, users, workspace/role/container names, tab labels, account ids, and arbitrary debug representations are prohibited by default.

The initial ceilings are 32 attributes per log, 64 attributes and 8 links per span, 32 attributes per metric point, 4 KiB for a redacted body or exception message, 1 KiB for one string attribute, 32 elements for one array, and 128 bytes for an event/span/instrument name. Safe text is redacted before UTF-8 truncation. Invalid names, keys, enums, counts, or prohibited values reject the signal and update typed health; there is no raw fallback.

Application processors validate the allowlist again before export. Endpoint/TLS errors render sanitized authorities only and never headers, embedded credentials, or client-key paths. Parallax may add defense-in-depth redaction, but application correctness does not depend on backend cleanup.

### Required jackin❯ end state

This section states refactor boundaries, not an implementation task list:

| Current surface | Required end state |
|---|---|
| `jackin-diagnostics` string registries and partial facade | Generated standard/extension schema and thin typed APIs in lowest-tier `jackin-telemetry`; provider composition stays in `jackin-diagnostics` |
| Raw `debug_log!`, Capsule prefix macros, and ungoverned `tracing` | Each call classified as typed operator output or a registered span/log/metric; no raw emission bypass |
| Screen root spans and `ScreenGuard` history | Screen lifecycle events, widget-focus events, action-root traces, transition children, dwell/focus metrics |
| Run JSONL and `RunDiagnostics` artifact ownership | In-memory current-invocation/progress state only; `cli.invocation.id` owns correlation |
| Build logs, command-output sidecars, `multiplexer.log`, host-daemon log | Removed as production telemetry/log destinations and readers |
| `jackin logs`, `jackin daemon logs`, diagnostics reveal/follow/summary/compare/bundle readers | Removed with local artifacts; `jackin diagnostics validate` remains as the direct OTLP delivery check |
| Protocol envelopes without standard context | Versioned W3C context plus required correlation; synchronous parentage and detached links tested across serialization |
| Ad hoc subprocess, Docker, HTTP, database, launch, and async instrumentation | Shared boundary instrumentation using the explicit case contracts above before leaf call sites migrate |
| Product usage state called `telemetry_store` | Renamed to its actual usage-state domain; product usage persistence remains and is not treated as telemetry storage |
| File-specific telemetry configuration and compatibility tests | Removed in the same cutover; there is one production destination, direct OTLP/gRPC to Parallax |

Test-only PTY fixtures, render-conformance output, and DHAT allocation artifacts remain outside production telemetry. Multiple reviewable PRs may advance this item, but the shipped state must not retain parallel old/new telemetry contracts.

### Acceptance criteria

- Product code uses governed `tracing` spans/events, explicit OpenTelemetry metrics/context, and composition-root SDK setup; no custom provider dispatch or raw emission bypass remains.
- Workspace and wire tests find no `jackin.*`, `parallax.*`, or other project/vendor attribute/event/span/metric namespace. Standard semantic conventions are used where available; neutral extensions are limited to the closed registry above.
- A one-shot command, week-long console simulation, daemon lifetime, screen visit, widget-focus interval, PTY conversation, job, and stream never create a lifetime trace. Every meaningful action/request/attempt/cycle has a bounded trace and the documented correlation.
- Synchronous CLI-to-daemon/control calls use one trace with `CLIENT`/`SERVER` parentage. Image and sidecar prewarm use `PRODUCER`/`CONSUMER` traces with a valid link and shared `job.id`. Periodic refresh cycles do not mint job ids.
- Each registered event arrives once as an OTel log with native EventName, severity, typed fields, Resource, and current TraceId/SpanId, and not as a duplicate span event.
- Screen history uses standard `app.screen.*`, entered/exited events, visit id, sequence, and dwell metrics. Tabs/panes use standard widget fields plus focus events/duration. Equivalent keyboard/mouse actions produce the same semantic `ui.action`; raw input and PTY content never appear.
- Agent/provider telemetry uses `gen_ai.agent.name`, `gen_ai.conversation.id`, and `gen_ai.provider.name`; no tab label, agent codename, account id, prompt, or arbitrary model name is exported.
- Resource values are stable for one process and cloned across all providers. Invocation/session/job/screen-visit/conversation ids are absent from Resource and metric dimensions.
- Error ownership, Status, `error.type`, cancellation, deadline, abandonment, and exception behavior match the contract without duplicate ERROR logs.
- A test OTLP receiver representing Parallax receives all three signals directly over gRPC from host, daemon, and a Capsule-safe configuration. No Collector configuration or OTLP relay is added.
- With no endpoint, SDK disabled, network disabled, endpoint loss, partial success, saturation, or slow export, product work remains bounded and usable. `--debug` remains operator output; only `jackin diagnostics validate` promises delivery validation.
- Workspace searches find no production telemetry/log file writer or reader, JSONL run artifact, command-output sidecar, Capsule/daemon log, local log command, stdout/stderr/TUI telemetry layer, or persistent telemetry spool.
- W3C propagation, disabled-child `or_current()`, links, malformed context, concurrent sessions, persistent daemon behavior, graceful shutdown, privacy negatives, cardinality, volume, and accelerated soak tests pass.
- A checked-in benchmark proves the disabled fast path avoids formatting/allocation and active launch/render/PTY telemetry stays within 5% of the reviewed baseline.
- Operator and contributor documentation describes direct OTLP-to-Parallax behavior, backend-owned history, in-memory current-run UI state, and the absence of local telemetry files.

### Out of scope

- Deploying or configuring an OpenTelemetry Collector, gateway, agent, sidecar, backend, dashboard, alert, or saved query.
- Parallax ingestion, authentication, storage, retention, sampling, redaction, routing, or visualization changes.
- A backend-neutral query/viewer API inside jackin❯.
- Local telemetry persistence, crash log, debug-bundle telemetry, audit trail, or disk-backed queue.
- Profiles until the OpenTelemetry Rust profiling signal is stable and separately designed; any future production profile export remains OTLP-only.
- Developer-only binaries such as `jackin-xtask`, `jackin-dev`, `jackin-pr-trailers`, `build-jackin-capsule`, and the TUI lookbook.
