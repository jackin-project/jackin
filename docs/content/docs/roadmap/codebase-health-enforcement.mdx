The thesis, in one line: **the structure of the code should reveal what the app does and where each thing lives — and that property has to be enforced, not remembered.** jackin❯ is ~20 crates and ~300k lines; this umbrella tracker makes the layout honest (crate name → layer → allowed dependencies) and keeps it that way with machine teeth.

<Aside type="caution">
**Structure only — never behavior.** Every change under this item is a pure structural reorganization: moving code between files/crates, relocating types, tightening lint config, adding a CI gate. It must not alter business logic, control flow, public behavior, or **performance**. The reorganization is *correct* only if the existing test suite and the `runtime-launch` / `op-picker` behavioral specs pass **unchanged** — no test rewritten to accommodate a move. A real bug spotted during a split is filed and fixed separately, never folded into a structural PR.
</Aside>

## Target architecture (the spec the remaining work serves)

A layered workspace with dependencies pointing **inward**, validated against [rust-analyzer](https://rust-analyzer.github.io/book/contributing/architecture.html), [matklad's "Large Rust Workspaces"](https://matklad.github.io/2021/08/22/large-rust-workspaces.html), and the [Rust "small crates" pattern](https://rust-unofficial.github.io/patterns/patterns/structural/small-crates.html). Principles:

- **IO-free core.** `jackin-core` (and the schema half of `jackin-config`, `jackin-manifest`, `jackin-protocol`) hold types and pure logic only; Docker/PTY/1Password/file calls live in adapter crates that depend on core, never the reverse.
- **Dependencies point inward.** Presentation → application → infrastructure → domain. A lower layer never depends on a higher one — the single still-violated edge is R1.
- **Named facade crates** (pure core; the host runtime; the shared TUI), each documenting its allowed dependencies in its `//!` Architecture-Invariant header.
- **One outermost crate owns serialization/presentation;** DTOs live at the edges.
- **Screaming names** — a crate/module name announces its responsibility.
- **Right-size crates** — carve god-crates along real subsystem seams; do not multiply crates needlessly.

The end-state layer graph (L0 lowest, dependencies strictly inward):

- **L0 domain (IO-free, owns the ports):** `jackin-core`, `jackin-config` (schema), `jackin-manifest`, `jackin-protocol`.
- **L1 application:** `jackin-runtime` (launch/bootstrap pipeline), `jackin-instance`, `jackin-isolation`, `jackin-env`, `jackin-image`.
- **L2 infrastructure:** `jackin-docker`, `jackin-host`, `jackin-diagnostics`, `jackin-term`.
- **L3 presentation:** `jackin-tui` (design system + shared Elm runtime), `jackin-launch-tui`, `jackin-console`.
- **L4 entry/glue:** `jackin` (CLI), `jackin-capsule` (PID1 + PTY daemon), `jackin-usage`.

<Aside type="caution">
**Performance guardrail.** Release builds enable `lto = "thin"` (E0); generics and `#[inline]` fns optimize across crates without it, but ordinary functions do not. Keep performance-critical, tightly-coupled code within one crate; any hot path that crosses a new boundary is perf-checked against the recorded launch/attach baseline. The R1 relocations and the R3 carve both ride this gate.
</Aside>

## Design decisions (reference)

The decisions the remaining slices follow, settled through brainstorming. Each carries a one-line *why*.

- **D1 — full layered topology.** Explicit per-concern layer crates with strictly inward dependencies — the most honest map; churn is acceptable because every move is behavior-preserving and independently shippable. Binding condition: `lto = "thin"` + perf-check before hot paths cross new boundaries.
- **D2 — boundary pattern: trait ports (hexagonal).** When a lower layer must hand data to a higher one, it depends on a **trait port** defined in `jackin-core`; the higher layer provides the adapter; the entry layer injects it. Not an event channel, not a relocated global. Governs the *streaming sinks* (build-log, launch progress, and the R1 launch-output calls); misplaced *types* are fixed by plain relocation, no port. This is the literal Dependency-Inversion fix and maps cleanly to the arch-gate ban.
- **D3 — four new crates** (`jackin-instance`, `jackin-isolation`, `jackin-host`, `jackin-usage`) by carving the two god-crates, so every crate name is a true subsystem boundary. `jackin-instance`/`jackin-isolation` are on the launch hot path → gated on E0; the cold carves are not.
- **D4 — `jackin-launch` → `jackin-launch-tui`.** The name then announces a presentation surface; the launch *logic* lives in `jackin-runtime`. Establishes the `-tui` suffix convention for app UI surfaces, distinct from the shared `jackin-tui` design-system crate.
- **D5 — one in-house shared Elm runtime in `jackin-tui`, not `tui-realm`.** One central `update(&mut Model, Message)` per app (canonical TEA); components translate input events → messages. In-house keeps full control over jackin❯'s custom terminal infrastructure and adds no dependency to clear against the `deny.toml` gate.
- **D6 — no binary facade layer.** Callers import directly from the owning crate root; each library `lib.rs` is its single curated facade. One home per concept.
- **D7 — one predictable stem vocabulary.** TUI stacks use the canonical TEA stems (`model`/`message`/`update`/`view`/`effect`, with `run`/`input`); `jackin-config`'s `app_config_*` fan-out converts to the standard coordinator + sibling layout. *Open sub-decision:* the `persist.rs` IO edge — see R10.

## Refactor safety — zero behavior change

Manage this as continuous, disciplined [Fowler-style](https://martinfowler.com/articles/refactoring-2nd-ed.html) refactoring, never an ad-hoc rewrite. [Branch by Abstraction](https://martinfowler.com/bliki/BranchByAbstraction.html) for the R1 port; [Parallel Change](https://martinfowler.com/bliki/ParallelChange.html) for relocations and the R3 move; [Strangler Fig](https://shopify.engineering/refactoring-legacy-code-strangler-fig-pattern) for the carve; [characterization tests](https://en.wikipedia.org/wiki/Characterization_test) where coverage is thin. Every PR lands with the suite and the `runtime-launch` / `op-picker` specs **green and unmodified** — that is the proof behavior is unchanged.

## Per-PR checklist (every remaining slice)

- [ ] Scope is exactly one slice — structure-only: no logic, control-flow, behavior, or performance change.
- [ ] `cargo fmt --check` · `cargo clippy --workspace --all-targets --all-features --locked -- -D warnings` · `cargo nextest run --all-features` all green.
- [ ] Behavioral specs (`runtime-launch`, `op-picker`) pass **unmodified**.
- [ ] `cargo xtask lint` green (file-size + test-layout + dependency-direction); refresh the relevant ratchet and delete the fixed entry when a file drops under cap.
- [ ] Any new or renamed crate carries `[lints] workspace = true` and an Architecture-Invariant `//!` header.
- [ ] Docs synced same PR: `PROJECT_STRUCTURE.md` + [Codebase Map](/reference/getting-oriented/codebase-map/), this item's checkbox, and the worklist file.
- [ ] DCO sign-off (`-s`); push immediately.
- [ ] *(carve / hot-path slices only)* the launch/attach benchmark shows no regression vs the E0 baseline.
