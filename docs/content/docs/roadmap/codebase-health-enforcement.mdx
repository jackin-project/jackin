The umbrella tracker for making jackin❯ understandable and reviewable as it grows. The thesis, in one line: **the structure of the code should reveal what the app does and where each thing lives — and that property has to be enforced, not remembered.** jackin❯ is now ~20 crates and ~300k lines. Crate separation exists and helps, but a new developer still cannot reliably answer "where does feature X live?" — the architecture has accumulated indirection (re-export shims), inverted dependencies, misleading crate names, god-crates, and duplicated subsystems that send the same question to several different files. <Aside type="caution">
**Structure only — never behavior.** Every change under this item is a pure structural reorganization: moving code between files/crates, relocating types, tightening lint config, adding a CI gate. It must not alter business logic, control flow, public behavior, or anything an operator can observe. Usability, workability, and **performance** stay identical. The reorganization is *correct* only if the existing test suite and behavioral specs pass unchanged — no test is rewritten to accommodate a move, no logic is "improved" along the way. If a real bug or genuine improvement is spotted during a split, it is filed separately and fixed in its own behavior-changing PR, never folded into a structural one.
</Aside>

## Target architecture

The model is a layered workspace with dependencies pointing **inward**, validated against the most directly comparable real-world Rust codebase — [rust-analyzer](https://rust-analyzer.github.io/book/contributing/architecture.html) (~200k+ lines, strict layered crates) — plus [matklad's "Large Rust Workspaces"](https://matklad.github.io/2021/08/22/large-rust-workspaces.html) and the [Rust "small crates" pattern](https://rust-unofficial.github.io/patterns/patterns/structural/small-crates.html). Principles, each mapped to jackin❯:

- **IO-free core.** The domain/vocabulary layer does no filesystem, network, process, or container IO and therefore cannot fail on IO. rust-analyzer's invariant: *"core parts (ide/hir) don't interact with the outside world and thus can't fail. Only parts touching LSP are allowed to do IO."* For jackin❯: `jackin-core` (and the schema half of `jackin-config`, `jackin-manifest`, `jackin-protocol`) hold types and pure logic only; Docker/PTY/1Password/file calls live in adapter crates that depend on core, never the reverse.
- **Dependencies point inward.** Presentation → application → infrastructure → domain. A lower layer never depends on a higher one. Today several edges run backwards (see below); fixing them is Workstream 1.
- **A few named facade crates ("API Boundaries").** rust-analyzer designates specific crates as *"the façade you'll be talking to."* jackin❯ should name 2–3 obvious entry facades (pure core; the host runtime; the shared TUI) and document each crate's allowed dependencies in its `//!` header as an **Architecture Invariant** — a convention reviewers uphold (rust-analyzer does exactly this; it is *not* a build-time check on its own — Workstream 4 adds the CI teeth).
- **One outermost crate owns serialization/presentation.** rust-analyzer: *"rust-analyzer is the only crate that knows about LSP and JSON serialization… types in lower layers are not serializable by design."* For jackin❯: keep wire/`serde`/config-format and terminal-rendering concerns out of the domain; DTOs and their conversions live at the edges (`jackin-protocol` for the container channel, the CLI/console crates for display).
- **Screaming names.** A crate/module name should announce its responsibility. `jackin-launch` currently fails this (it is a TUI, not the launch orchestrator) — Workstream 1 renames it.
- **Right-size crates — do not maximize crate count.** Smaller single-purpose crates aid comprehension and parallel compilation (the crate is Rust's compilation unit), but more crates is not strictly better: Tokio merged its many small crates back into one. jackin❯'s problem is bimodal — a few **god-crates** that are too big *and* a shim layer that adds indirection without value. The fix is to carve the god-crates along real subsystem seams and delete the pointless shims, not to multiply crates.

<Aside type="caution">
**Performance guardrail (must-preserve).** Release builds do not perform cross-crate LTO by default (`lto = false`), so ordinary (non-generic, non-`#[inline]`) functions are **not** inlined across crate boundaries — *"two small crates may be less optimized than one large one."* Any split that puts a hot, tightly-coupled code path behind a new crate boundary can cost runtime performance. Rule for this item: keep performance-critical, tightly-coupled code **within one crate**; if a hot path must cross a new boundary, enable `lto = "thin"` in the release profile for that build and measure. Generics and `#[inline]` fns still optimize across crates without LTO.
</Aside>

## Design

Concrete decisions for the open workstreams (W1, W4, W6, and the remaining W5 slices), settled through brainstorming. Each decision carries a one-line *why*.

### D1 — Restructure scope: full layered topology

**Decision:** pursue the rust-analyzer-style layered topology — explicit per-concern layer crates with strictly inward dependencies — over the lighter module-first or hybrid options. **Why:** it gives the most honest map (crate name → layer → allowed dependencies), which is the stated goal; the larger churn is acceptable because every move is behavior-preserving and independently shippable.

**Binding condition (performance).** Because release builds do not cross-crate LTO by default, splitting hot, tightly-coupled paths across new crate boundaries can cost runtime performance. Therefore: (a) the release profile enables `lto = "thin"` workspace-wide before hot paths cross new boundaries, and (b) each god-crate carve is perf-checked against a representative launch/attach measurement to confirm no regression. No carve lands on an unmeasured hot path. This operationalizes the performance guardrail above; performance is preserved, not assumed.

### D2 — Boundary pattern: trait ports (hexagonal), not channels or relocated globals

**Decision:** when a lower layer must hand data to a higher one, the lower layer depends on a **trait port** defined in a neutral low crate (`jackin-core`), the higher layer provides the adapter (`impl` of the port), and the entry/runtime layer injects it. Not an event channel, not a relocated global sink. **Why:** the port is an abstraction the domain owns — the literal Dependency-Inversion fix — it removes the global mutable state (channels and ports both do; a relocated global does not), it is the most testable (pass a `Vec`-backed fake), and it maps cleanly to a `cargo-deny` ban because the dependency target is a core-owned trait. Per-call dynamic dispatch (`&dyn`) is negligible on these paths (per-line build output, not per-byte); use `impl Trait` for static dispatch if a path is ever measured hot.

Worked example — the `build_log` inversion (P2): define `trait BuildLogSink` in `jackin-core`; `jackin-docker`'s shell runner takes `&dyn BuildLogSink` and calls `push_line` on it; `jackin-launch` (the UI) implements it; the runtime injects the UI's sink. Result: `jackin-docker → jackin-core` only.

**Scope clarification.** This pattern governs only the *streaming sinks* — `build_log` and the launch `progress` updates. The other inversions are **misplaced types, not data flows**, and are fixed by plain relocation, no port needed: `PromptResult` (env → launch) and the `progress` value types (runtime → tui) move to `jackin-core`; the terminal-ownership / debug-surface state moves out of `jackin-diagnostics` to a presentation/terminal crate (P7). A trait port is used *only* where a lower layer actively pushes data upward at runtime.

### D3 — Target crate graph: full carve into four new crates

**Decision:** the end-state topology adds **four new crates** by carving the two god-crates, plus relocations/dedup. Layers, with dependencies pointing strictly inward (L0 lowest):

- **L0 domain (IO-free, owns the ports):** `jackin-core` (cleaned of presentation helpers; gains the trait ports + relocated `PromptResult`/`progress` types), `jackin-config` (schema), `jackin-manifest`, `jackin-protocol`.
- **L1 application/orchestration:** `jackin-runtime` (shrinks to the launch/bootstrap pipeline), **`jackin-instance`** *(new — instance identity/manifest/index/recovery)*, **`jackin-isolation`** *(new — mount isolation: shared/worktree/clone)*, `jackin-env`, `jackin-image` (absorbs `runtime/image.rs`, killing P5).
- **L2 infrastructure/adapters:** `jackin-docker` (implements the ports), **`jackin-host`** *(new — host OS integration: desktop, clipboard, caffeinate/keep-awake)*, `jackin-diagnostics` (observability only), `jackin-term`.
- **L3 presentation:** `jackin-tui` (design system + unified Elm runtime), `jackin-launch` (renamed/folded — see D4), `jackin-console`.
- **L4 entry/glue:** `jackin` (CLI; shim maze deleted), `jackin-capsule` (PID1 + PTY daemon), **`jackin-usage`** *(new — usage/pricing/telemetry/token monitors)*.

**Why full carve over the cooler hybrid:** it is the only option that makes every crate name a true subsystem boundary, which is the goal; the navigability win on `instance`/`isolation` is worth the cost.

**Binding condition — `jackin-instance` and `jackin-isolation` are on the launch hot path.** Unlike the cold `jackin-host`/`jackin-usage` carves, these split a tight `jackin load` sequence across crate boundaries, so per D1: enable `lto = "thin"` in the release profile **and** establish a launch/attach benchmark **before** these two crates land; the carve PR must show no measurable regression. The cold carves (`jackin-host`, `jackin-usage`) carry no such gate. `jackin-console`'s own internal carve rides with [unify-settings-editor-surfaces](/roadmap/unify-settings-editor-surfaces/), not this topology change.

### D4 — `jackin-launch` → `jackin-launch-tui`

**Decision:** once its value types relocate to `jackin-core` (D2/D3) the crate holds only the launch-cockpit UI; rename it `jackin-launch-tui`. **Why:** the name then announces it is a presentation surface, so no one expects the launch *logic* (which lives in `jackin-runtime`) here — resolving P3 fully rather than relying on a doc comment. Establishes a `-tui` suffix convention for app UI surfaces, distinct from the shared `jackin-tui` design-system crate. Not folded into `jackin-console`: the launch cockpit and the interactive console are separate surfaces with independent lifecycles, and folding would bloat the already-oversized console crate.

### D5 — TUI runtime (W6): one in-house shared Elm runtime in `jackin-tui`, not `tui-realm`

**Decision:** build the shared runtime in-house — distill the common `Model`/`Message`/`update` plus a `Component`/`View` contract out of the four existing Elm stacks into `jackin-tui`; the stacks (`jackin-console`, the capsule TUI, `jackin-launch-tui`, and the per-screen `editor`/`settings`/`workspaces` stacks) become components/screens on that one runtime. Do **not** adopt `tui-realm`. **Why:** jackin❯ already has substantial *custom* terminal infrastructure (`jackin-term`'s VT model, the capsule re-emitting multiplexer, the attach-capability matrix) that an opinionated external framework would have to coexist with and could fight; in-house keeps full control, fits that model exactly, adds no dependency to clear against the strict `deny.toml` license/supply-chain gate, and is incremental (four working implementations already exist to distill). The research's own framing: the *pattern*, not the `tui-realm` dependency, is the takeaway.

**Contract shape.** The runtime centers on **one** `update(&mut Model, Message)` per app (canonical TEA), with components translating input events → messages — *not* per-component co-located update loops (the "co-location reduces duplication" claim was adversarially refuted; the single central `update` is the actual de-duplication). Sequence with [unify-settings-editor-surfaces](/roadmap/unify-settings-editor-surfaces/), which already collapses two of the four stacks (editor + settings) and is the first consumer of the shared contract.

### D6 — Shim-maze end-state (P1): direct imports from crate roots, no binary facade layer

**Decision:** delete all 19 `pub use jackin_X::*` shim modules in the `jackin` binary; callers import directly from the owning crate root (`use jackin_manifest::RoleManifest`). Each library crate's `lib.rs` is its single facade — keep those crate-root re-exports curated so the public surface *is* the facade. No `jackin::prelude` glob, no per-module re-export layer. **Why:** one home per concept — the crate whose name matches — which is the entire navigability goal; a binary-side prelude would reintroduce a second home and a glob indirection ([matklad cautions against exactly this](https://matklad.github.io/2021/08/22/large-rust-workspaces.html)). Fix the memorize-this exceptions along the way (e.g. `DriftDetection` "import from `runtime::drift` directly") so there are zero special cases. The import-site churn is mechanical and behavior-preserving (a [Parallel Change](https://martinfowler.com/bliki/ParallelChange.html): callers migrate to the crate path, then the shim is deleted).

### D7 — Naming convention (P8): one predictable stem vocabulary

**Decision:** a file's stem must predict its role, with one vocabulary per subsystem kind. Two concrete rules:

- **TUI stacks** all use the canonical TEA stems (following from D5's shared contract): `model.rs` (state), `message.rs` (the `Message` enum), `update.rs` (the central `update`), `view.rs` (render), `effect.rs` (side effects), with `run.rs`/`input.rs` where needed. Retire the `state.rs` / `render.rs` synonyms so "where is the update loop / the view?" has one answer on every surface.
- **`jackin-config`'s `app_config_*` fan-out** (`app_config_mounts.rs`, `app_config_persist.rs`, `app_config_roles.rs`, `app_config_workspaces.rs` — a `type_aspect.rs` stem used nowhere else) converts to the workspace's standard coordinator + sibling layout: `app_config.rs` coordinator + `app_config/{mounts,persist,roles,workspaces}.rs`.

**Why:** a predictable convention is the difference between "navigate by knowing the rule" and "navigate by grepping"; the project already standardized on coordinator + self-named siblings (no `mod.rs`), so this extends one existing rule rather than inventing a second.

**Deferred sub-decisions (settle during implementation, not blocking):** (a) whether `jackin-config`'s file-IO persistence (`persist.rs`) splits into an L2 adapter to keep the schema crate strictly IO-free, or the schema crate keeps a thin documented IO edge; (b) the W4 unused-dep tool overlap (`cargo-shear` already runs — reconcile with `cargo-machete` rather than run both); (c) the cross-crate `FakeOpWriter` test-helper dedup (needs a shared `test-support` feature like the one `config_with_agents` now uses).

## Workstreams

### W1 — Architecture & boundaries (the core restructuring)

Behavior-preserving moves that make the layout match the target. This section is the *thematic* summary; the **ordered, per-PR execution** of these moves is the [Implementation plan](#implementation-plan) below — Phase A (boundary fixes) through Phase F (naming), with the god-crate carves in Phases C and E.

- [ ] **Relocate the misplaced types that cause the inverted dependencies (P2).** Move `PromptResult` into `jackin-core` (a domain value). Replace `jackin-docker`'s direct `build_log::push_line` calls with an emit **port** the runtime owns (Docker emits a log event; the launch UI subscribes) — a [Branch by Abstraction](https://martinfowler.com/bliki/BranchByAbstraction.html) move. Result: `jackin-env` and `jackin-docker` no longer depend on `jackin-launch`.
- [ ] **Rename `jackin-launch` → a name that screams "launch cockpit UI"** (e.g. `jackin-launch-tui`), or merge it into the shared TUI crate; move its non-UI `progress`/`build_log` value types to a neutral layer (P3).
- [ ] **Delete the shim maze (P1).** For each of the 19 `pub use jackin_X::*` modules, repoint call sites at the owning crate and remove the shim. Where a genuine facade is wanted, keep exactly one documented re-export at the crate boundary — not a per-module parallel layer. Fix the documented exceptions (e.g. `DriftDetection` "import from `runtime::drift` directly") so there are no memorize-this special cases.
- [ ] **Carve the god-crates along subsystem seams (P4)** — keeping hot paths intact per the performance guardrail. `jackin-runtime`: separate `instance`, `isolation`, and host desktop/clipboard from the bootstrap pipeline. `jackin-capsule`: separate PID1/daemon, the in-container TUI, and usage/telemetry. Each carve is independently shippable.
- [ ] **Collapse duplicated concept homes (P5)** — one home each for image build, env resolution, `op_cache`, and resource naming; the loser becomes a re-export or is deleted.
- [ ] **Move leaked concerns to their layer (P7)** — presentation helpers out of `jackin-core`; terminal-ownership state out of `jackin-diagnostics` into a presentation/terminal crate.
- [ ] **Normalize naming (P8)** — one module-stem convention per subsystem; retire the `app_config_*` fan-out.

### W2 — Stricter Clippy

The reviewability levers are currently OFF. CI already runs `cargo clippy … -D warnings`, so flipping a lint to `warn` enforces it immediately. Measure first, set thresholds at the current maximum, grandfather with narrow `#[expect(…, reason = "…")]`, ratchet down. Never blanket-`allow`.

- [x] **W2.0 measure** — apply the lints below locally; capture violation counts + current maxima to set non-breaking initial thresholds.
- [x] Enable `too_many_lines` (`allow` → `warn`) + set `too-many-lines-threshold`.
- [x] Enable `cognitive_complexity` (nursery, cherry-pick) + threshold.
- [x] Enable `excessive_nesting` (nursery; default 0 disables — set e.g. 5) + threshold.
- [x] Flip `fn_params_excessive_bools` and `struct_excessive_bools` `allow` → `warn`.
- [x] Tune `clippy.toml` thresholds: `too-many-arguments`, `type-complexity`, `enum-variant-size`.
- Decision: `*/tests.rs` exceed limits by nature — enforce on production code, `#[expect]` the large test fns, handle test size under W3.

### W3 — File-size ratchet gate

Clippy has **no** max-file-length lint; this is the bespoke check it cannot provide. Same shape as the existing `bun run check:roadmap-sidebar`.

- [x] **Hard ceiling** — no production `.rs` over an absolute cap (proposed 2000L); new files born under it.
- [x] **High-water-mark allowlist (the ratchet)** — a committed budget (`file-size-budget.toml` or a `cargo xtask lint files` subcommand) lists every current over-cap file with its line count; the gate fails if a listed file *grows*, or if an unlisted file exceeds the cap. Numbers may only decrease; an entry is deleted when its file drops under the cap. **You cannot add lines to an already-too-big file**, which makes decomposition self-enforcing.
- [x] Wire into <RepoFile path=".github/workflows/ci.yml">.github/workflows/ci.yml</RepoFile> as a required job; seed from today's offenders.
- [x] Separate, higher budget for `*/tests.rs` (`launch/tests.rs` 8603L, `daemon/tests.rs` 7612L) so production enforcement isn't blocked on the test-infra work ([test-infra-behavioral-specs](/roadmap/test-infra-behavioral-specs/)).

### W4 — Enforce the architecture in CI

Documented invariants are upheld by reviewers ([rust-analyzer's are not a build check](https://rust-analyzer.github.io/book/contributing/architecture.html)); add machine teeth so the inversions cannot come back.

- [ ] **Dependency-direction bans** via [`cargo-deny` bans](https://embarkstudios.github.io/cargo-deny/checks/bans/index.html): once the inversions are fixed, deny `jackin-env`/`jackin-docker` → `jackin-launch*`, deny domain crates → presentation, etc., in `deny.toml`. A re-inversion fails CI.

- [ ] **Unused-dep detection** — [`cargo-machete`](https://github.com/bnjbvr/cargo-machete) (fast, source-scan, CI-friendly exit codes) on every PR; [`cargo-udeps`](https://github.com/est31/cargo-udeps) (compile-based, nightly) periodically. ### W6 — Unify the TUI runtime (P6)

Stop hand-rolling the loop per screen/app. [The Elm Architecture](https://ratatui.rs/concepts/application-patterns/the-elm-architecture/) centralizes transitions in **one** `Message` enum + **one** `update` (ratatui ships no loop, which is why the duplication exists). A shared runtime like [tui-realm](https://github.com/veeso/tui-realm) implements `update` once with a central `View` owning mount/focus/event routing.

- [ ] Extract one shared Elm runtime (Model/Message/update + a `Component`/`View` contract) into `jackin-tui`; the four stacks become screens/components on it.
- [ ] Land [unify-settings-editor-surfaces](/roadmap/unify-settings-editor-surfaces/) on top of it so the editor/settings models are one implementation.

## Refactor safety — zero behavior change

Manage this as continuous, disciplined refactoring — [Fowler's](https://martinfowler.com/articles/refactoring-2nd-ed.html) catalog of small, named, mechanically-repeatable, behavior-preserving steps — never an ad-hoc rewrite. Techniques, by situation:

- **[Branch by Abstraction](https://martinfowler.com/bliki/BranchByAbstraction.html)** for the dependency-inversion fixes (introduce the port, migrate callers, remove the old path).
- **[Parallel Change / expand–contract](https://martinfowler.com/bliki/ParallelChange.html)** for type relocations and shim removal (add the new home, migrate, delete the old).
- **[Strangler Fig](https://shopify.engineering/refactoring-legacy-code-strangler-fig-pattern)** for carving god-crates incrementally.
- **[Characterization / golden-master tests](https://en.wikipedia.org/wiki/Characterization_test)** where coverage is thin before a move, so behavior is pinned first.

Every PR under this item is structure-only and must land with the existing test suite and behavioral specs (the `runtime-launch` and `op-picker` specs are the oracle) **green and unmodified** — that is the proof behavior is unchanged. A move that forces a test edit changed behavior; back it out.

### Reusable recipe — carve a subsystem into its own crate

Used by every C/E slice. Pure move; contents are not rewritten.

1. Create `crates/<new>/` with a manifest (`[lints] workspace = true`, minimal dependency set) and a `//!` Architecture-Invariant header.
2. `git mv` the subsystem's modules across verbatim — byte-identical bodies.
3. Adjust visibility only as the boundary forces: the crate's public surface becomes `pub`; everything else stays `pub(crate)`. No signature or logic edits.
4. Repoint importers to the new crate root ([Parallel Change](https://martinfowler.com/bliki/ParallelChange.html) — add the new path, migrate callers, delete the old).
5. Add the crate to the workspace `members`; add the old crate's dependency on it only if it still calls inward.
6. Add the `cargo-deny` ban entries that encode the new allowed edges (W4), so the boundary cannot invert later.
7. Update `PROJECT_STRUCTURE.md` + Codebase Map.
8. Run the Per-PR checklist. For a hot-path carve, run the launch/attach benchmark against the E0 baseline and attach the numbers to the PR.

## Why it matters

A new developer should locate any functionality by reading crate names and following one obvious inward dependency path — not by guessing which of several homes owns a concept. Stricter Clippy and a size ratchet keep files reviewable; fixing the layering, names, and duplication makes the *map* itself honest; CI bans keep both from regressing. The payoff is concrete and measurable: "where is the Codex usage parser?" → `usage/codex.rs`; "what does launch do?" → the crate whose name says launch; "can `jackin-docker` touch the UI?" → no, CI forbids it.
