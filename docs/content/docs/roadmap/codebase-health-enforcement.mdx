## Target architecture

The model is a layered workspace with dependencies pointing **inward**, validated against the most directly comparable real-world Rust codebase â€” [rust-analyzer](https://rust-analyzer.github.io/book/contributing/architecture.html) (~200k+ lines, strict layered crates) â€” plus [matklad's "Large Rust Workspaces"](https://matklad.github.io/2021/08/22/large-rust-workspaces.html) and the [Rust "small crates" pattern](https://rust-unofficial.github.io/patterns/patterns/structural/small-crates.html). Principles, each mapped to jackinâťŻ:

- **IO-free core.** The domain/vocabulary layer does no filesystem, network, process, or container IO and therefore cannot fail on IO. rust-analyzer's invariant: *"core parts (ide/hir) don't interact with the outside world and thus can't fail. Only parts touching LSP are allowed to do IO."* For jackinâťŻ: `jackin-core` (and the schema half of `jackin-config`, `jackin-manifest`, `jackin-protocol`) hold types and pure logic only; Docker/PTY/1Password/file calls live in adapter crates that depend on core, never the reverse.
- **Dependencies point inward.** Presentation â†’ application â†’ infrastructure â†’ domain. A lower layer never depends on a higher one. Today several edges run backwards (see below); fixing them is Workstream 1.
- **A few named facade crates ("API Boundaries").** rust-analyzer designates specific crates as *"the faĂ§ade you'll be talking to."* jackinâťŻ should name 2â€“3 obvious entry facades (pure core; the host runtime; the shared TUI) and document each crate's allowed dependencies in its `//!` header as an **Architecture Invariant** â€” a convention reviewers uphold (rust-analyzer does exactly this; it is *not* a build-time check on its own â€” Workstream 4 adds the CI teeth).
- **One outermost crate owns serialization/presentation.** rust-analyzer: *"rust-analyzer is the only crate that knows about LSP and JSON serializationâ€¦ types in lower layers are not serializable by design."* For jackinâťŻ: keep wire/`serde`/config-format and terminal-rendering concerns out of the domain; DTOs and their conversions live at the edges (`jackin-protocol` for the container channel, the CLI/console crates for display).
- **Screaming names.** A crate/module name should announce its responsibility. `jackin-launch` currently fails this (it is a TUI, not the launch orchestrator) â€” Workstream 1 renames it.
- **Right-size crates â€” do not maximize crate count.** Smaller single-purpose crates aid comprehension and parallel compilation (the crate is Rust's compilation unit), but more crates is not strictly better: Tokio merged its many small crates back into one. jackinâťŻ's problem is bimodal â€” a few **god-crates** that are too big *and* a shim layer that adds indirection without value. The fix is to carve the god-crates along real subsystem seams and delete the pointless shims, not to multiply crates.

<Aside type="caution">
**Performance guardrail (must-preserve).** Release builds do not perform cross-crate LTO by default (`lto = false`), so ordinary (non-generic, non-`#[inline]`) functions are **not** inlined across crate boundaries â€” *"two small crates may be less optimized than one large one."* Any split that puts a hot, tightly-coupled code path behind a new crate boundary can cost runtime performance. Rule for this item: keep performance-critical, tightly-coupled code **within one crate**; if a hot path must cross a new boundary, enable `lto = "thin"` in the release profile for that build and measure. Generics and `#[inline]` fns still optimize across crates without LTO.
</Aside>

## Design

Concrete decisions for the open workstreams (W1, W4, W6, and the remaining W5 slices), settled through brainstorming. Each decision carries a one-line *why*.

### D1 â€” Restructure scope: full layered topology

**Decision:** pursue the rust-analyzer-style layered topology â€” explicit per-concern layer crates with strictly inward dependencies â€” over the lighter module-first or hybrid options. **Why:** it gives the most honest map (crate name â†’ layer â†’ allowed dependencies), which is the stated goal; the larger churn is acceptable because every move is behavior-preserving and independently shippable.

**Binding condition (performance).** Because release builds do not cross-crate LTO by default, splitting hot, tightly-coupled paths across new crate boundaries can cost runtime performance. Therefore: (a) the release profile enables `lto = "thin"` workspace-wide before hot paths cross new boundaries, and (b) each god-crate carve is perf-checked against a representative launch/attach measurement to confirm no regression. No carve lands on an unmeasured hot path. This operationalizes the performance guardrail above; performance is preserved, not assumed.

### D2 â€” Boundary pattern: trait ports (hexagonal), not channels or relocated globals

**Decision:** when a lower layer must hand data to a higher one, the lower layer depends on a **trait port** defined in a neutral low crate (`jackin-core`), the higher layer provides the adapter (`impl` of the port), and the entry/runtime layer injects it. Not an event channel, not a relocated global sink. **Why:** the port is an abstraction the domain owns â€” the literal Dependency-Inversion fix â€” it removes the global mutable state (channels and ports both do; a relocated global does not), it is the most testable (pass a `Vec`-backed fake), and it maps cleanly to a `cargo-deny` ban because the dependency target is a core-owned trait. Per-call dynamic dispatch (`&dyn`) is negligible on these paths (per-line build output, not per-byte); use `impl Trait` for static dispatch if a path is ever measured hot.

Worked example â€” the `build_log` inversion (P2): define `trait BuildLogSink` in `jackin-core`; `jackin-docker`'s shell runner takes `&dyn BuildLogSink` and calls `push_line` on it; `jackin-launch` (the UI) implements it; the runtime injects the UI's sink. Result: `jackin-docker â†’ jackin-core` only.

**Scope clarification.** This pattern governs only the *streaming sinks* â€” `build_log` and the launch `progress` updates. The other inversions are **misplaced types, not data flows**, and are fixed by plain relocation, no port needed: `PromptResult` (env â†’ launch) and the `progress` value types (runtime â†’ tui) move to `jackin-core`; the terminal-ownership / debug-surface state moves out of `jackin-diagnostics` to a presentation/terminal crate (P7). A trait port is used *only* where a lower layer actively pushes data upward at runtime.

### D3 â€” Target crate graph: full carve into four new crates

**Decision:** the end-state topology adds **four new crates** by carving the two god-crates, plus relocations/dedup. Layers, with dependencies pointing strictly inward (L0 lowest):

- **L0 domain (IO-free, owns the ports):** `jackin-core` (cleaned of presentation helpers; gains the trait ports + relocated `PromptResult`/`progress` types), `jackin-config` (schema), `jackin-manifest`, `jackin-protocol`.
- **L1 application/orchestration:** `jackin-runtime` (shrinks to the launch/bootstrap pipeline), **`jackin-instance`** *(new â€” instance identity/manifest/index/recovery)*, **`jackin-isolation`** *(new â€” mount isolation: shared/worktree/clone)*, `jackin-env`, `jackin-image` (absorbs `runtime/image.rs`, killing P5).
- **L2 infrastructure/adapters:** `jackin-docker` (implements the ports), **`jackin-host`** *(new â€” host OS integration: desktop, clipboard, caffeinate/keep-awake)*, `jackin-diagnostics` (observability only), `jackin-term`.
- **L3 presentation:** `jackin-tui` (design system + unified Elm runtime), `jackin-launch` (renamed/folded â€” see D4), `jackin-console`.
- **L4 entry/glue:** `jackin` (CLI; shim maze deleted), `jackin-capsule` (PID1 + PTY daemon), **`jackin-usage`** *(new â€” usage/pricing/telemetry/token monitors)*.

**Why full carve over the cooler hybrid:** it is the only option that makes every crate name a true subsystem boundary, which is the goal; the navigability win on `instance`/`isolation` is worth the cost.

**Binding condition â€” `jackin-instance` and `jackin-isolation` are on the launch hot path.** Unlike the cold `jackin-host`/`jackin-usage` carves, these split a tight `jackin load` sequence across crate boundaries, so per D1: enable `lto = "thin"` in the release profile **and** establish a launch/attach benchmark **before** these two crates land; the carve PR must show no measurable regression. The cold carves (`jackin-host`, `jackin-usage`) carry no such gate. `jackin-console`'s own internal carve rides with [unify-settings-editor-surfaces](/roadmap/unify-settings-editor-surfaces/), not this topology change.

### D4 â€” `jackin-launch` â†’ `jackin-launch-tui`

**Decision:** once its value types relocate to `jackin-core` (D2/D3) the crate holds only the launch-cockpit UI; rename it `jackin-launch-tui`. **Why:** the name then announces it is a presentation surface, so no one expects the launch *logic* (which lives in `jackin-runtime`) here â€” resolving P3 fully rather than relying on a doc comment. Establishes a `-tui` suffix convention for app UI surfaces, distinct from the shared `jackin-tui` design-system crate. Not folded into `jackin-console`: the launch cockpit and the interactive console are separate surfaces with independent lifecycles, and folding would bloat the already-oversized console crate.

### D5 â€” TUI runtime (W6): one in-house shared Elm runtime in `jackin-tui`, not `tui-realm`

**Decision:** build the shared runtime in-house â€” distill the common `Model`/`Message`/`update` plus a `Component`/`View` contract out of the four existing Elm stacks into `jackin-tui`; the stacks (`jackin-console`, the capsule TUI, `jackin-launch-tui`, and the per-screen `editor`/`settings`/`workspaces` stacks) become components/screens on that one runtime. Do **not** adopt `tui-realm`. **Why:** jackinâťŻ already has substantial *custom* terminal infrastructure (`jackin-term`'s VT model, the capsule re-emitting multiplexer, the attach-capability matrix) that an opinionated external framework would have to coexist with and could fight; in-house keeps full control, fits that model exactly, adds no dependency to clear against the strict `deny.toml` license/supply-chain gate, and is incremental (four working implementations already exist to distill). The research's own framing: the *pattern*, not the `tui-realm` dependency, is the takeaway.

**Contract shape.** The runtime centers on **one** `update(&mut Model, Message)` per app (canonical TEA), with components translating input events â†’ messages â€” *not* per-component co-located update loops (the "co-location reduces duplication" claim was adversarially refuted; the single central `update` is the actual de-duplication). Sequence with [unify-settings-editor-surfaces](/roadmap/unify-settings-editor-surfaces/), which already collapses two of the four stacks (editor + settings) and is the first consumer of the shared contract.

### D6 â€” Shim-maze end-state (P1): direct imports from crate roots, no binary facade layer

**Decision:** delete all 19 `pub use jackin_X::*` shim modules in the `jackin` binary; callers import directly from the owning crate root (`use jackin_manifest::RoleManifest`). Each library crate's `lib.rs` is its single facade â€” keep those crate-root re-exports curated so the public surface *is* the facade. No `jackin::prelude` glob, no per-module re-export layer. **Why:** one home per concept â€” the crate whose name matches â€” which is the entire navigability goal; a binary-side prelude would reintroduce a second home and a glob indirection ([matklad cautions against exactly this](https://matklad.github.io/2021/08/22/large-rust-workspaces.html)). Fix the memorize-this exceptions along the way (e.g. `DriftDetection` "import from `runtime::drift` directly") so there are zero special cases. The import-site churn is mechanical and behavior-preserving (a [Parallel Change](https://martinfowler.com/bliki/ParallelChange.html): callers migrate to the crate path, then the shim is deleted).

### D7 â€” Naming convention (P8): one predictable stem vocabulary

**Decision:** a file's stem must predict its role, with one vocabulary per subsystem kind. Two concrete rules:

- **TUI stacks** all use the canonical TEA stems (following from D5's shared contract): `model.rs` (state), `message.rs` (the `Message` enum), `update.rs` (the central `update`), `view.rs` (render), `effect.rs` (side effects), with `run.rs`/`input.rs` where needed. Retire the `state.rs` / `render.rs` synonyms so "where is the update loop / the view?" has one answer on every surface.
- **`jackin-config`'s `app_config_*` fan-out** (`app_config_mounts.rs`, `app_config_persist.rs`, `app_config_roles.rs`, `app_config_workspaces.rs` â€” a `type_aspect.rs` stem used nowhere else) converts to the workspace's standard coordinator + sibling layout: `app_config.rs` coordinator + `app_config/{mounts,persist,roles,workspaces}.rs`.

**Why:** a predictable convention is the difference between "navigate by knowing the rule" and "navigate by grepping"; the project already standardized on coordinator + self-named siblings (no `mod.rs`), so this extends one existing rule rather than inventing a second.

**Deferred sub-decisions (settle during implementation, not blocking):** (a) whether `jackin-config`'s file-IO persistence (`persist.rs`) splits into an L2 adapter to keep the schema crate strictly IO-free, or the schema crate keeps a thin documented IO edge; (b) the W4 unused-dep tool overlap (`cargo-shear` already runs â€” reconcile with `cargo-machete` rather than run both); (c) the cross-crate `FakeOpWriter` test-helper dedup (needs a shared `test-support` feature like the one `config_with_agents` now uses).

## Refactor safety â€” zero behavior change

Manage this as continuous, disciplined refactoring â€” [Fowler's](https://martinfowler.com/articles/refactoring-2nd-ed.html) catalog of small, named, mechanically-repeatable, behavior-preserving steps â€” never an ad-hoc rewrite. Techniques, by situation:

- **[Branch by Abstraction](https://martinfowler.com/bliki/BranchByAbstraction.html)** for the dependency-inversion fixes (introduce the port, migrate callers, remove the old path).
- **[Parallel Change / expandâ€“contract](https://martinfowler.com/bliki/ParallelChange.html)** for type relocations and shim removal (add the new home, migrate, delete the old).
- **[Strangler Fig](https://shopify.engineering/refactoring-legacy-code-strangler-fig-pattern)** for carving god-crates incrementally.
- **[Characterization / golden-master tests](https://en.wikipedia.org/wiki/Characterization_test)** where coverage is thin before a move, so behavior is pinned first.

Every PR under this item is structure-only and must land with the existing test suite and behavioral specs (the `runtime-launch` and `op-picker` specs are the oracle) **green and unmodified** â€” that is the proof behavior is unchanged. A move that forces a test edit changed behavior; back it out.

### Reusable recipe â€” carve a subsystem into its own crate

Used by every C/E slice. Pure move; contents are not rewritten.

1. Create `crates/<new>/` with a manifest (`[lints] workspace = true`, minimal dependency set) and a `//!` Architecture-Invariant header.
2. `git mv` the subsystem's modules across verbatim â€” byte-identical bodies.
3. Adjust visibility only as the boundary forces: the crate's public surface becomes `pub`; everything else stays `pub(crate)`. No signature or logic edits.
4. Repoint importers to the new crate root ([Parallel Change](https://martinfowler.com/bliki/ParallelChange.html) â€” add the new path, migrate callers, delete the old).
5. Add the crate to the workspace `members`; add the old crate's dependency on it only if it still calls inward.
6. Add the `cargo-deny` ban entries that encode the new allowed edges (W4), so the boundary cannot invert later.
7. Update `PROJECT_STRUCTURE.md` + Codebase Map.
8. Run the Per-PR checklist. For a hot-path carve, run the launch/attach benchmark against the E0 baseline and attach the numbers to the PR.
