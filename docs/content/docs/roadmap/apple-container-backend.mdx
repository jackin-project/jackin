Run <RepoFile path="scripts/phase0-apple-container.sh" /> there to validate it; nothing in this branch claims Phase 0 has passed, and the launch dispatch currently starts then reclaims the Docker sidecar pending the Phase-0 reorder.

## What This Is

This item tracks the implementation of the `apple-container` backend for jackin❯. Apple Container (`apple/container`, macOS 26 Tahoe) gives each role container its own dedicated Linux VM via Virtualization.framework, natively on Apple Silicon, with zero install friction — it is built into macOS 26.

The Apple Container backend is the primary path to close the **kernel boundary gap**: today all jackin❯ sessions share a single OrbStack Linux VM kernel. Apple Container gives each session its own kernel, so a kernel exploit in one session cannot reach other sessions or the host.

For the complete picture of why this backend exists and how it fits with the other security layers, see [Agent isolation architecture](/roadmap/agent-isolation-architecture/).

## Prerequisites (Both Must Complete Before Phase 0 Starts)

### Prerequisite 1: Docker Hardening Rootless DinD Validation

Apple Container does not support `--privileged`. The current jackin❯ DinD sidecar requires `--privileged` and cannot run inside an apple/container VM as-is. Rootless DinD is the compatibility path — it replaces `--privileged` with specific capability grants (`--cap-add`).

Before Apple Container Phase 0 can start, the [Docker runtime hardening contract](/roadmap/docker-runtime-hardening-contract/) work must answer:

1. Does apple/container allow `--cap-add CAP_SYS_ADMIN`? (rootless DinD needs it for mount namespaces)
2. Does rootless DinD pass `docker build`, Compose, and Testcontainers inside an apple/container VM?

If rootless DinD fails inside apple/container — either `CAP_SYS_ADMIN` is blocked or Docker workflows break — the Apple Container backend cannot support full jackin❯ role workflows. In that case, fall back to [smolvm backend research](/reference/research/isolation/smolvm-backend/), which has a documented Docker-in-VM path (though constrained).

### Prerequisite 2: `JACKIN_CAPSULE_FORCE_DAEMON` Capsule Patch

Apple Container's `vminitd` runs as PID 1 inside every VM. The jackin-capsule daemon mode now accepts `JACKIN_CAPSULE_FORCE_DAEMON=1` in <RepoFile path="crates/jackin-capsule/src/main.rs" /> so the apple-container backend can run capsule as PID 2+ while still entering daemon mode.

The daemon-mode check is:

```rust
// Before (line 22):
let is_pid1 = std::process::id() == 1;

// After:
let is_pid1 = std::process::id() == 1
    || std::env::var("JACKIN_CAPSULE_FORCE_DAEMON").is_ok();
```

The Apple Container backend launch path passes `-e JACKIN_CAPSULE_FORCE_DAEMON=1` as a flag to `container run` — not as a static `ENV` in the derived Dockerfile. Setting it in the Dockerfile would break the Docker backend where capsule IS PID 1 and client-mode invocations (e.g. `jackin-capsule status`) must not enter daemon mode. The env var must be injected only when launching via the `apple-container` backend path. This must be validated in Phase 0: does SIGTERM propagation, exit-code visibility, and Ctrl+C passthrough all still work correctly when capsule is not PID 1?

## Architecture

### Runtime Stack

```
macOS 26 ARM
  → apple/container VM (own kernel, Virtualization.framework)
    → vminitd (PID 1, gRPC/vsock init system)
      → role container entrypoint (jackin-capsule, PID 2+, JACKIN_CAPSULE_FORCE_DAEMON=1)
        → agent process (Claude Code, Codex, Amp, ...)
        → rootless DinD (inner Docker daemon, --cap-add instead of --privileged)
```

### Attach Transport

`vminitd` exposes a gRPC API over vsock. It provides I/O streaming, signal forwarding, and process supervision. The `container exec` CLI command uses this internally. jackin❯ shells out to:

```bash
container exec -it <container-name> jackin-capsule
```

This gives a proper PTY with SIGWINCH forwarding and signal passthrough — the exact requirements for jackin❯s interactive TUI session. This is an advantage over smolvm's CLI-based attach where PTY behavior was undocumented.

### Lifecycle CLI Surface

jackin❯ shells out to the `container` CLI for all lifecycle operations:

```bash
container run --name jackin-<instance> \
  -e JACKIN_CAPSULE_FORCE_DAEMON=1 \
  -v /host/workspace:/workspace \
  --cap-add <required-caps> \
  <role-image> jackin-capsule

container exec -it jackin-<instance> jackin-capsule   # attach
container stop jackin-<instance>                       # stop
container rm jackin-<instance>                         # delete
container ps --format json                             # list/inspect
container logs jackin-<instance>                       # logs
```

No Rust-native API exists for apple/container; the programmatic surface is Swift XPC-native. Shell-out to `container` CLI is the correct integration pattern for V1.

### Networking

Each apple/container VM gets its own dedicated IP address via vmnet — no port mapping needed. Services inside the role container are reachable from macOS at the container's IP. Known rough edges in v0.11.0: container-to-container networking on the same bridge has DNS hiccups after macOS sleep/wake. jackin❯ should detect DNS failures and prompt the operator to reconnect.

## `AppleContainerResources` — Instance Manifest Shape

<RepoFile path="crates/jackin-runtime/src/instance/manifest.rs" /> now has a backend-neutral `BackendResources` enum.

The `AppleContainerResources` variant needed for this backend:

```rust
pub struct AppleContainerResources {
    /// Name of the apple/container container: "jackin-<instance-id>"
    pub container_name: String,
    /// OCI image ref used to start the container
    pub role_image_ref: String,
    /// Whether an inner Docker daemon (rootless DinD) is running
    pub inner_docker_enabled: bool,
}
```

`DockerResources` has four fields: `role_container: String`, `dind_container: String`, `network: String`, `certs_volume: String`.

The `BackendResources` tagged union, added alongside the existing `DockerResources`:

```rust
#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum BackendResources {
    Docker(DockerResources),
    AppleContainer(AppleContainerResources),
}
```

The change is **additive, not a replacement**: `InstanceManifest` keeps its existing `docker: DockerResources` field (still the source of truth for the Docker path and for resource naming) and gains an optional `backend: Option<BackendResources>` field. `InstanceManifest::new` leaves `backend` as `None` (Docker), and `InstanceManifest::new_with_backend` sets it for apple-container launches. The optional field means old manifests deserialize unchanged.

**`INSTANCE_MANIFEST_VERSION`**: bumped from `1` to `2`.

### Docker Client Architecture Note

<RepoFile path="crates/jackin-docker/src/docker_client.rs" /> uses **bollard** — a typed async Rust API client for Docker, not a CLI shell-out. All Docker lifecycle calls (create container, start, stop, remove, network, volumes) go through bollard's `Docker` trait. The Apple Container backend **cannot** use bollard. It shells out to the `container` CLI (`container run`, `container exec`, `container stop`, `container rm`). The Apple Container client module (<RepoFile path="crates/jackin-runtime/src/apple_container_client.rs" />) uses `tokio::process::Command` for lifecycle calls — a fundamentally different pattern from `docker_client.rs`.

## Known Limitations (v0.11.0)

| Limitation | Workaround |
|---|---|
| `--privileged` not supported | rootless DinD via `--cap-add` (Phase 0 validation required) |
| Multi-container bridge networking rough edges | DinD inner container networking may be affected; test in Phase 0 |
| DNS hiccuping after macOS sleep/wake | Detect DNS failure in capsule; surface "reconnect required" to operator |
| No health checks (`--health-cmd`) | jackin-capsule liveness probes instead |
| Apple Silicon only | Acceptable — jackin❯ targets macOS 26 ARM only |
| macOS 26 required | Acceptable — jackin❯ targets macOS 26+ |

## Implementation Phases

### Phase 0 — empirical validation (prerequisite gate)

Run on a real macOS 26 ARM machine with apple/container v0.11.0+. This phase is empirical only — no production code lands, but the validation scaffold is in place.

**Run the validation script:**

```bash
./scripts/phase0-apple-container.sh [--role-image <image>]
```

The script (<RepoFile path="scripts/phase0-apple-container.sh" />) automates all checks, records PASS/FAIL for each, and produces a `scripts/phase0-results.md` decision gate summary. Checks covered:

1. Prerequisites: `container` CLI installed, Apple Silicon ARM64, macOS ≥ 26
2. `JACKIN_CAPSULE_FORCE_DAEMON=1` — non-PID-1 daemon mode activates correctly
3. Role OCI image boots inside apple/container VM
4. `container exec -it` PTY works (SIGWINCH and Ctrl+C require manual verification)
5. Workspace bind mounts: read, write-back to host, read-only enforcement
6. `--cap-add CAP_SYS_ADMIN` — **critical gate for rootless DinD**
7. Rootless DinD (if cap check passed): dockerd starts, `docker build`/Compose/Testcontainers
8. Cold start latency: container run → capsule socket ready (target: < 5 s)
9. DNS stability after macOS sleep/wake (manual step)

**Decision gate**: if rootless DinD passes → Phase 2 launch path is unblocked. If it fails → fall back to smolvm Phase 0 (see [smolvm backend research](/reference/research/isolation/smolvm-backend/)).

### Phase 1 — backend-neutral instance registry

- Add `BackendResources` tagged union to <RepoFile path="crates/jackin-runtime/src/instance/manifest.rs" /> (DockerResources variant, AppleContainerResources variant)
- Teach `hardline --inspect` to report manifest state for non-Docker backends
- Extract backend-kind probe from Docker-specific naming in <RepoFile path="crates/jackin-runtime/src/runtime/launch.rs" />

### Phase 2 — experimental `apple-container` backend

- Add `apple-container` as a hidden/experimental backend flag in the CLI
- Shell out to `container run` for launch
- Shell out to `container exec` for attach
- Apply `JACKIN_CAPSULE_FORCE_DAEMON=1` env var at launch
- Use rootless DinD inside the VM (validated in Phase 0)
- Mount only operator-approved workspace/global paths

### Phase 3 — full lifecycle (reconnect, eject, purge)

- Reconnect to stopped/running containers via `container exec`
- `stop` → `container stop`
- `eject` → stop container, preserve instance manifest
- `purge` → `container rm`, delete instance manifest
- Detect and handle DNS hiccup state after sleep/wake
- Preserve worktree cleanup behavior on eject

### Phase 4 — session contract and security output

- Launch/session contract output for backend=apple-container, kernel boundary, inner Docker, caps granted, mounts, residual risks
- Report DNS enforcement state
- Fail-closed if required caps are not available

## AppleContainerApi Trait (Phase 1)

Phase 1 adds <RepoFile path="crates/jackin-runtime/src/apple_container_client.rs" />. Unlike <RepoFile path="crates/jackin-docker/src/docker_client.rs" /> which uses the **bollard** crate (typed async Rust API), Apple Container has no Rust API — all operations shell out to the `container` CLI using `tokio::process::Command`. The trait shape for the apple-container backend:

```rust
// Native async-fn-in-trait (no `async_trait` macro needed on this toolchain).
pub trait AppleContainerApi {
    async fn run_container(&self, name: &str, spec: &AppleContainerSpec) -> anyhow::Result<()>;
    async fn exec_attach(&self, name: &str) -> anyhow::Result<tokio::process::Child>;
    async fn stop_container(&self, name: &str) -> anyhow::Result<()>;
    async fn remove_container(&self, name: &str) -> anyhow::Result<()>;
    async fn inspect_container(&self, name: &str) -> anyhow::Result<Option<AppleContainerInfo>>;
    async fn list_containers(&self, name_prefix: &str) -> anyhow::Result<Vec<AppleContainerInfo>>;
}

pub struct AppleContainerSpec {
    pub image: String,
    pub env: Vec<(String, String)>,        // (-e KEY=VALUE flags)
    pub mounts: Vec<(PathBuf, PathBuf)>,   // (-v host:container flags)
    pub caps_add: Vec<String>,             // (--cap-add flags)
}

pub struct AppleContainerInfo {
    pub name: String,
    pub status: String,  // "running", "stopped", etc. — parsed from `container ps` output
}
```

All methods shell out to `container run`, `container exec -it`, `container stop`, `container rm`, `container ps` respectively. `exec_attach` returns the child process handle so the caller can pipe stdio (mirrors how `docker exec -it` is handled in the Docker backend via `runner.run`).

## Schema Version Sequencing

The on-demand credential fields (jackin-exec: `on_demand` on `OpRef`, the `Extended` `EnvValue` variant) and the per-workspace runtime backend field (apple-container: `[workspaces.X.runtime]`) are both additive workspace-schema changes and land together under a single workspace version bump, alongside the config `default_backend` bump:

| Schema change | Source version | Target version |
|---|---|---|
| `CURRENT_WORKSPACE_VERSION` (on-demand credentials + runtime backend) | `v1alpha6` | `v1alpha7` |
| `CURRENT_CONFIG_VERSION` (`[runtime] default_backend`) | `v1alpha6` | `v1alpha7` |

Per the AGENTS.md one-bump-per-PR rule, a single PR introduces exactly one version bump per file kind; multiple additive fields land under that one bump. The config and workspace constants are independent, so both transitions land in the same PR. An implementing agent extending this further must check the current value of `CURRENT_WORKSPACE_VERSION` in <RepoFile path="crates/jackin-config/src/migrations.rs" /> when the PR is opened and target the next version.
