It replaces the old phase leaves for setup docs, file splits, behavioral-spec chores, module-contract chores, visibility cleanup, split-runtime/split-editor tracking, settings/editor unification, test-infrastructure tracking, and the stale complexity-debt plan folder.

## Goal

Make jackin❯ exceptionally safe to refactor by humans and AI agents: strict compiler/tooling gates catch predictable bugs before review, crate and module boundaries make the right edit location obvious, critical behavior has executable characterization coverage, performance hot paths are measured instead of guessed, and every surviving exception is justified rather than inherited. The program also measures its own effect: escaped defects convert into new gates, and agent task success on this codebase is tracked as a first-class health metric rather than assumed (Phase 7). And because "how did we get into this state" must be answerable for the running product, not only the source, every run tells one structured, redacted, correlated story in an OpenTelemetry backend (Phase 8).

The project is pre-release, so large restructures are acceptable. The priority is not minimizing migration cost; the priority is lowering future defect rate and making the codebase easy for agents to understand, modify, verify, and review.

The operating assumption for this program is that the most frequent contributor is an autonomous agent, not a human at a keyboard. That flips the usual lint calibration. A `warn` is a human grace period — a hint that something *could* be fixed later. An agent does not need a grace period: it can read the lint message, write the fix, and re-run the gate in the same turn. So the target ruleset is **deny by default, with `reason =`-justified `#[expect]` as the only escape hatch**, even for lints that conventional projects leave at `warn` because their fix is "tedious." The 2026 Rust discourse on stricter Clippy for coding agents makes the same point from the opposite direction: panics that kill a Tokio worker, dropped futures, swallowed errors, and lossy numeric casts are exactly the defects the compiler cannot catch but Clippy can, and they are exactly the defects an agent is likeliest to introduce. This program adopts that stricter posture as the destination, not the starting point.

## Shipped foundation

- The workspace uses Rust 2024, resolver v3, explicit MSRV, a strict workspace lint table, `unsafe_code = "forbid"`, `unreachable_pub = "deny"`, `clippy::all = deny`, `pedantic`/`cargo` warnings promoted by CI, and hard denials for `unwrap`, `expect`, `panic`, stdout/stderr printing, `todo`, `unimplemented`, `mod_module_files`, and selected blocking methods.
- The current ratchet headroom is explicit and bounded. `clippy.toml` complexity thresholds sit at the measured pre-Workstream-A maxima (`too-many-lines = 150`, `cognitive-complexity = 60`, `excessive-nesting = 5`, `too-many-arguments = 7`) and the four `disallowed-methods` entries are the only blocked-API list. A deliberate set of pedantic/restriction lints is still `allow` (`needless_pass_by_value`, `large_futures`, `unused_async`, `assigning_clones`, `match_same_arms`, `drop_non_drop`, `unused_self`, `format_push_string`, `option_option`, `float_cmp`, `struct_field_names`, and more), and the panic-coverage / silent-failure / async-safety restriction families are not yet adopted at all. There is **no `[workspace.lints.rustdoc]` table**, no `missing_docs` on API-bearing crates, no `disallowed_types`/`disallowed_macros`, no committed public-API snapshot, and no layered-architecture gate. Phase 1 and Phase 2 close exactly these gaps.
- The codebase is split into workspace crates with Rust 2024 self-named module files, no `mod.rs`, no inline test modules, and a shrink-only test-layout allowlist.
- CI already runs fmt, clippy across all targets/features, schema checks, file-size/test-layout/agent-file gates, a dependency-direction (`arch`) gate, `cargo shear`, `cargo audit`, `cargo deny` policy checks, nextest shards, fuzz smoke, benchmark build checks, MSRV checks, and optional Docker-backed E2E.
- The workflow layer is already partially hardened and instrumented: every action is SHA-pinned, `actionlint` + `shellcheck` gate workflow and script edits, the clippy/check lanes upload `cargo --timings` and sccache-stats artifacts, a layered `cargo-depgraph` SVG artifact is rendered per run, the scheduled hygiene lane runs `cargo deny check advisories`, the feature powerset, doc tests, and a long fuzz pass, a native macOS smoke lane builds and tests off-Linux, and the release pipeline pins `cosign` and `syft` for artifact signing and SBOM generation. Later phases build on these facts instead of re-introducing them.
- A layered-architecture gate already exists: <RepoFile path="crates/jackin-xtask/src/arch.rs">`cargo xtask lint arch`</RepoFile> (Workstream 4) walks the `cargo metadata` graph and runs strict in CI. Its forbidden-edge list is currently empty because every known inversion was already resolved by lifting port traits (`CommandRunner`, `BuildLogSink`, `OperatorNoticeSink`, `DebugLogSink`) into `jackin-core`. The remaining work is to evolve its model, not to build the gate — see Phase 2.
- The file-size and test-layout ledgers were burned down to zero grandfathered entries, and the old PR-specific merge-readiness audit is closed.
- Contributor orientation exists in the [Codebase Map](/reference/getting-oriented/codebase-map/), TUI rules in [TUI Design Decisions](/reference/tui/), and behavioral specs for runtime launch and op-picker under [Behavioral Specs](/reference/developer-reference/specs/).
- **Per-crate orientation is authoritative.** Every workspace crate carries a README (purpose, what it owns, architecture tier + allowed dependencies, `src/` structure, public API, how to verify) and a per-crate AGENTS file holding **only the non-derivable rules** an agent cannot read from the code, the README, the workspace lint table, or the `cargo xtask lint arch` dependency gate — never a duplicate of tier/dependency/structure/API info, because duplicating it measurably reduces agent task success and wastes context tokens. `CLAUDE.md` is symlinked to it. The convention, templates, the "update the README on any structural change" rule, and the non-derivable-only AGENTS rule live in <RepoFile path="crates/AGENTS.md">crates/AGENTS.md</RepoFile>, and the `cargo xtask lint agents` gate enforces the AGENTS + `CLAUDE.md` symlink pair in every `crates/*/` member. A cold reader — human or agent — opens a crate and understands it from its README without consulting the root structure doc; pre-implementation design records (e.g. `jackin-term`) live in the internal docs page, not the README, so the README stays a current-state map.

## Principles for the remaining work

- **Ratchet, do not posture.** Add a strict gate only after a measured dry run proves the signal is actionable; otherwise start as an advisory artifact or scheduled hygiene lane.
- **Prefer source-of-truth boundaries over wrappers.** If host and capsule need the same auth/path/process rule, extract one shared contract below both rather than copying policy.
- **Refactor with characterization first.** Launch, capsule daemon, isolation materialization, auth, and TUI input changes need fast tests around observable behavior before body extraction.
- **Make exceptions auditable.** Every surviving `allow`/`expect` should have a narrow scope and a reason; unreasoned broad suppressions are debt even when CI is green.
- **Optimize for AI navigation.** A future agent should know which crate owns a concept from the codebase map, should see one canonical API to copy, and should have a focused verification command for the touched surface.
- **Deny by default; optimize the ruleset for autonomous enforcement.** Because the primary contributor is an agent, calibrate every gate toward `deny` + reasoned `#[expect]`, not `warn`. A lint that "only an agent would violate" is the strongest argument *for* denying it, not against. Prefer allowlist-style adoption (turn a category on, then justify each suppression) over denylist-style cherry-picking once the Phase 0 inventory is in hand; a missed lint is a missed defect, and an allowlist makes a missed lint impossible.
- **Make the compiler and the lints the oracle.** Maximize the share of "is this correct?" that an answerable machine check decides before a human or agent guesses. Every invariant that can be a type, a trait bound, a lint, or an architecture rule should be; invariants that live only in prose are invariants an agent will quietly violate.
- **Diagnostics are prompts.** A gate that fails without stating the fix wastes an agent turn. Every first-party gate (xtask lints, ratchets, policy tests) must print, on failure: which rule fired, why it exists, the exact edit or command that clears it, and the narrowest rerun command — and must offer `--format json` plus GitHub problem-matcher annotations so agents parse results instead of scraping logs. A gate's quality bar is "an agent clears it in one turn without reading the gate's source."
- **Every escaped defect becomes a gate.** A bug that reaches an operator — or a panic that reaches the installed panic hooks — is evidence of a missing oracle, not just a missing fix. Closing it requires a characterization test plus either a new lint/gate/budget that would have caught the class, or a written reason none is practical, recorded in the Phase 7 defect ledger.

## Phase 1 — Tighten Rust and CI strictness

### PR-time gates to promote after dry run

1. **Feature powerset gate**: move at least a scoped `cargo hack check --workspace --feature-powerset --all-targets --locked` lane from scheduled-only hygiene into PR CI, starting with crates that actually expose optional behavior.
2. **Rustdoc documentation tests**: add `cargo test --doc --workspace --locked` to PR CI if the current scheduled job is green enough to promote.
3. **Deny advisories parity**: run `cargo deny check advisories` in PR dependency policy so `deny.toml`, `.cargo/audit.toml`, yanked policy, and advisory ignores cannot drift until scheduled hygiene.
4. **Snapshot hygiene**: pin `cargo-insta` or add a cheap pending-snapshot/diff gate so TUI snapshot changes cannot leave `*.pending-snap` files or unreviewed `.snap` diffs behind.

### Advisory or scheduled lanes first

1. **Coverage**: add `cargo-llvm-cov` as an artifact/report, not a global PR fail threshold. Start with crate-group trends for parser/protocol/config/terminal crates and only later ratchet specific critical surfaces.
2. **Miri**: run a narrow scheduled/manual lane for pure crates and deterministic parser/protocol/config tests. Do not attempt full-workspace Miri across Docker, PTY, process, or OS-heavy code.
3. **Sanitizers**: keep PR fuzz fast, but add scheduled sanitizer runs where fuzz targets already exist, especially terminal/parser paths.
4. **Mutation testing**: evaluate `cargo-mutants` only on pure crates with fast deterministic tests, such as protocol/config/manifest/terminal parsing. Do not run it across Docker/TUI integration code.
5. **Public API drift**: add `cargo-public-api` or `cargo-semver-checks` as an informational report for selected API-bearing crates. Because jackin❯ is pre-release, the gate should make breakage intentional rather than forbid it.
6. **Transitive unsafe and workspace-hack experiments**: use `cargo geiger` and `cargo hakari` only as advisory investigations. First-party unsafe is already forbidden; workspace-hack value should be proven with cargo timing artifacts before adding generated maintenance surface.

### Lint ratchets and suppression hygiene

1. Convert broad `#[allow(...)]` suppressions to narrow `#[expect(..., reason = "...")]` where the expectation is stable across targets. Keep `allow` only where `expect` would be flaky under cfg/test/target differences.
2. Add an xtask or grep gate that blocks new `allow`/`expect` attributes without `reason =`, except for documented allowlist patterns such as generated code or cross-target imports.
3. Revisit globally allowed maintainability lints one at a time: `needless_pass_by_value`, `large_futures`, `unused_async`, `assigning_clones`, `match_same_arms`, `drop_non_drop`, `unused_self`, and other explicitly allowed pedantic lints. Promote only low-noise lints; keep noisy ones documented.
4. Expand `clippy::disallowed_methods` after a discovery pass to cover more blocking process/filesystem APIs where they can hit render/runtime paths, with local expectations for startup/build/test-owned boundaries.
5. Ratchet `too_many_lines`, `cognitive_complexity`, `excessive_nesting`, and `too_many_arguments` thresholds only after the fresh inventory identifies the next safe lower bucket.
6. Require every non-Apache/MIT license exception in `deny.toml` to carry a rationale or grouped policy note so the current strict supply-chain posture stays reviewable.
7. Add a per-lint suppression-count ratchet: a shrink-only ledger (same mechanism as the file-size budget, folded into the Phase 7 unified ratchet engine) recording how many `#[expect]`s exist per lint per crate. Suppression debt becomes visible, bounded, and monotonically shrinking between category adoptions, instead of drifting upward unnoticed.

### Panic-coverage and silent-failure restriction lints — the agent defect surface

The workspace already denies `unwrap_used`, `expect_used`, `panic`, `todo`, and `unimplemented`, but that misses the panic families a coding agent introduces most often: byte-boundary string/array slicing, swallowed results, dropped futures, and lossy numeric casts. The motivating class is the same one behind the 2025 "unwrap that broke the internet" incident and the UTF-8-slicing Tokio-worker kills described in the 2026 stricter-Clippy writing — defects the compiler cannot see but Clippy can. Adopt these as the long-term `deny` target, each behind a Phase 0 dry run:

- **Don't panic (slice/index coverage):** `string_slice`, `indexing_slicing`, `get_unwrap`, `unwrap_in_result`, `unchecked_time_subtraction`, `panic_in_result_fn`. These complement the existing unwrap/expect/panic denials by catching `&s[a..b]`, `arr[i]`, and panics inside `Result`-returning functions. `string_slice` and `indexing_slicing` will fire widely on day one; route every hit through `.get()`/`.split_at_checked()`/iterator APIs. Treat the wide blast radius as a feature — each site is a latent panic an agent would otherwise ship.
- **Don't fail silently:** `let_underscore_future`, `let_underscore_must_use` (jackin❯ already denies `let_underscore_drop`; close the family), `unused_result_ok`, `map_err_ignore`, `assertions_on_result_states`. These catch `let _ = future`, `r.ok();`, and `.map_err(|_| …)` — silent error loss that is invisible to tests.
- **Don't do bad async:** `await_holding_lock`, `await_holding_refcell_ref`. jackin❯ runs the capsule daemon and runtime on Tokio with non-`Send` guards held across `.await` in places; these are the deadlock lint pair and are high-value against the capsule control plane. (`if_let_mutex` is edition-2024-resolved and can stay off.)
- **Don't do incorrect things with numbers:** `float_cmp_const`, `lossy_float_literal`, `cast_sign_loss`, `invalid_upcast_comparisons`, and re-evaluate the current `float_cmp = "allow"`. Treat `cast_possible_truncation`/`cast_precision_loss`/`cast_possible_wrap` as the optional "force a documented invariant on every lossy cast" tier once the loud set is clean.
- **Easy to avoid:** `rc_mutex`, `debug_assert_with_mut_call`, `expl_impl_clone_on_copy`, `infallible_try_from`, `iter_not_returning_iterator`.

Add the matching `clippy.toml` escape valves (`allow-unwrap-in-tests`, `allow-expect-in-tests`, `allow-panic-in-tests`, `allow-print-in-tests` are the ones present today; add `allow-indexing-slicing-in-tests`, `allow-dbg-in-tests`, and the rest as each lint lands) so the denials apply to production and tests stay ergonomic.

### Meta-suppression discipline — make every `allow` a reasoned `expect`

The single highest-leverage, lowest-noise addition for an agent-authored codebase:

1. **`allow_attributes = "deny"`** — every `#[allow(...)]` must become `#[expect(...)]`. An `expect` is self-documenting and fails the build if the underlying lint ever stops firing, so suppressions cannot outlive their reason. This directly enforces the "no silent host writes / no silent suppressions" posture as a gate.
2. **`allow_attributes_without_reason = "deny"`** — every `#[expect(...)]` (and any surviving `#[allow]`) must carry `reason = "…"`. An agent that disables a lint must state why; the build rejects a bare suppression.
3. The existing xtask grep that blocks `allow`/`expect` without `reason =` becomes redundant once these two lints deny; keep it only for the generated-code / cross-target allowlist carve-outs the lints cannot express.

### Rustdoc and missing-docs gates

jackin❯ has **no `[workspace.lints.rustdoc]` table** and only `missing_debug_implementations = "warn"` from the `rust` group. Close the documentation-as-navigation gap:

1. Add `[workspace.lints.rustdoc]` with `broken_intra_doc_links = "deny"`, `private_intra_doc_links = "warn"`, `redundant_explicit_links = "warn"`, and `unescaped_quotes_in_doc_comment = "warn"` so rustdoc is a real check, not decoration.
2. Promote `missing_debug_implementations` to `deny` once the remaining gaps are filled (it is already CI-promoted via `-D warnings`, so the delta is making it local-fail too).
3. Turn on `missing_docs` (the `rust` lint) for the foundational, API-bearing crates — `jackin-core`, `jackin-config`, `jackin-manifest`, `jackin-protocol`, `jackin-env`, `jackin-term` — not for binaries or `jackin-xtask`. An undocumented public item is a navigation hole for an agent copying a canonical API; `missing_docs` makes the hole a build failure. Gate crate-by-crate behind the dry run rather than workspace-wide, because the first adoption sweep is large.
4. Keep `cargo test --doc --workspace` (Phase 1 PR-time gate) as the executable half of the rustdoc gate.

### Disallowed types, macros, and the blocked-surface list

`disallowed_methods` already blocks four blocking process/filesystem APIs. Extend the same mechanism, configured in `clippy.toml`:

1. **`disallowed-types`** — block footgun types in favor of workspace-canonical alternatives: `std::sync::Mutex`/`std::sync::RwLock` → `parking_lot` (already a dependency), `Rc<RefCell<_>>` in async code, `Cell`/`RefCell` on `Send` types, `VecDeque` where a ring-buffer abstraction exists, bare `String`/`Vec<u8>` in protocol frames where `compact_str`/`bytes` is the contract. Each entry carries a `reason` naming the replacement.
2. **`disallowed_macros`** — block `println!`/`eprintln!`/`dbg!`/`todo!`/`unimplemented!`/`print!` everywhere production code can reach (tests exempt), and any crate-local macro that has been superseded. This makes the existing `print_stdout`/`print_stderr`/`dbg_macro` denials structural instead of per-call.
3. Keep the blocked-surface list versioned in `clippy.toml` with a one-line `reason` per entry so the gate doubles as documentation for agents.

### Custom project lints via dylint

When a jackin❯-specific invariant cannot be expressed by any shipped lint, encode it as a workspace-owned [dylint](https://github.com/trailofbits/dylint) library run from CI, not as a prose rule an agent will drift from. Candidate first-party lints, each justified by a real footgun:

- **Process-execution boundary:** no direct `tokio::process::Command` / `std::process::Command` use outside the designated execution-boundary crate (the Phase 2 "Shared contracts" extraction); every call site must go through the shared capture/timeout/retry/redaction model.
- **Render-thread purity:** no blocking I/O, allocation storms, or `std::sync` locks reachable from the terminal/console draw path; the lint enforces what `disallowed_methods` cannot express structurally.
- **Foundational-crate surface:** every `pub` type in a foundational crate implements `Debug`, and extension traits are sealed.
- **Config/manifest field consistency:** new `jackin.role.toml` / `config.toml` fields carry a `#[serde]` + a doc + a migration entry together; the lint flags a field added to a struct without the matching schema-version bump.
- **Telemetry discipline:** exporting call sites go through the typed operation API and the semconv name registry (Phase 8), not ad-hoc `log::`/`tracing::` free-form strings; `clog!`/`cdebug!` stay legal as console/file renderers only.

Author the lint library under `crates/jackin-lints/`, gate it advisory first (warn in a scheduled lane), and promote to PR-blocking once the false-positive rate is measured. dylint lints compile against a nightly-internal `rustc` API, so isolate them in their own crate and run them in the scheduled lane or a pinned-toolchain CI job rather than the default PR toolchain.

### Whole-category "Stricter-er" as an adoption track

Two defensible philosophies exist for adopting restriction lints: cherry-pick (Schwartz) or turn whole categories on and justify each suppression (the "Stricter-er" allowlist argument). For jackin❯, run the Phase 0 inventory, then pilot the **allowlist track** on one low-risk pure crate: set `restriction = { level = "warn", priority = -1 }` for that crate only, fix or `#[expect(…, reason)]` every hit, and measure the ongoing maintenance cost. If the maintenance cost is low and the defect catch rate high — the expected outcome for an agent-authored codebase where the friction is the point — roll the allowlist posture workspace-wide and invert the existing allow-set: most current `allow`s become either justified `#[expect]`s or accepted warns. Track the decision and its snapshot date in this page's research notes so the posture is auditable, not inherited.

### Toolchain, workflow, and supply-chain canaries

The deny-wall gets stricter every phase, which raises the cost of surprises arriving *with the toolchain or the CI substrate* rather than with the code. Buy the surprises early and cheap:

1. **Beta-toolchain clippy canary.** A scheduled lane runs `cargo clippy --all-targets --all-features` (and optionally the test suite) on the next Rust beta, so new lints, lint renames, and behavior changes surface weeks before the pinned toolchain moves. Fixes or `#[expect]`s land ahead of the upgrade instead of blocking it; on a deny-by-default workspace, a toolchain bump should be a planned chore, never a surprise red main.
2. **Workflow security lint.** Add [zizmor](https://github.com/zizmorcore/zizmor) beside the existing `actionlint` gate: template injection, credential persistence, mutable `uses:` references, and excessive `permissions:` across `.github/workflows/`. The workflows are code an agent edits; lint them like code.
3. **Secret scanning.** Run [gitleaks](https://github.com/gitleaks/gitleaks) over the tree and each PR's commits. jackin❯ forwards real credentials (`op`, `gh`, agent auth) through env and config surfaces, and agent-authored fixtures are the likeliest leak vector; a committed token is a revocation incident even when review catches it. This is the commit-side half of the risk; the Phase 3 telemetry-redaction policy tests cover the runtime side.
4. **Dependency review beyond advisories.** `cargo deny`/`cargo audit` catch *known-bad*; they say nothing about unreviewed code. <RepoFile path="crates/AGENTS.md">crates/AGENTS.md</RepoFile> records a deliberate cargo-vet/cargo-crev no-adopt while jackin❯ is solo-maintainer; this item is that decision's named revisit trigger. Trial [cargo-vet](https://mozilla.github.io/cargo-vet/) as an advisory lane in exemptions-heavy mode, importing shared audits (Mozilla, Google, kernel) so the marginal cost of an explicit audit record per new/updated dependency stays near zero. Promote to a gate only if audit-import coverage proves the burden negligible, and update the AGENTS entry when the trial starts.
5. **Auditable binaries.** Build releases with [cargo-auditable](https://github.com/rust-secure-code/cargo-auditable) so the exact dependency graph is embedded in the shipped binary and `cargo audit bin` works on artifacts in the wild — the binary-side complement of the syft SBOM the release pipeline already produces.

### Public API surface and dependency-direction gates

The current structure relies on `unreachable_pub = "deny"` and reviewer memory to keep the public surface small and the dependency graph acyclic. Promote both to machine-checked gates so an agent cannot accidentally widen the API or reverse a dependency:

1. **Sealed traits as the extension-point boundary.** Any trait that downstream crates or role authors are not meant to implement must be sealed (private supertrait). Sealing is what makes a trait "internal" real to `cargo-semver-checks`, so unsealed `pub trait`s are a semver liability and an agent footgun. Audit the foundational and protocol crates, seal everything that is not an intended extension point, and add a first-party dylint lint (Phase 1) that flags `pub trait` without a seal in API-bearing crates.
2. **Committed public-API snapshot.** Run `cargo public-api` over the workspace and commit the generated API surface as a checked-in artifact (per API-bearing crate). A PR that changes the public API must update the snapshot; the diff in the PR review is then the API change, made intentional and visible. Because jackin❯ is pre-release, breaking changes are allowed — the snapshot's job is to force awareness and review, not to forbid breakage. Promote `cargo-public-api` and `cargo-semver-checks` from the Phase 1 "informational report" tier to a real PR gate once the baseline snapshot is committed.
3. **Barrel-file re-export discipline.** Each foundational crate exposes one canonical surface from its `lib.rs` — intentional root re-exports of the supported API, with `pub(crate)` implementation modules behind them — so an agent copying an API copies the canonical path, not an internal one. Pair this with the Phase 1 `missing_docs` gate so every re-exported item is documented. Deprecate accidental `pub mod` surfaces by narrowing them to `pub(crate)` or re-exporting the intended subset.
4. **Evolve the layered-architecture gate from an allowlist to a tier graph.** The dependency-direction gate already exists and runs strict in CI (`cargo xtask lint arch`, see Shipped foundation); today it forbids a hardcoded list of specific inverted edges. Move it to a tier-graph model — declare each crate's tier (from its `lib.rs` Architecture Invariant) and fail when *any* edge points upward, so a new crate gets a rule automatically instead of waiting for a manual forbidden-edge entry. Add separate production vs dev-dependency edge rules so test-support crates can depend up without polluting the production graph, plus an explicit cycle check. `arch-lint` / `layered-crate` remain options if their rule model fits better than the in-repo xtask. This supersedes the "Architecture tests" line above with the existing tool and a tier source of truth.
5. **`pub`-surface ratchet report.** The Phase 0 code-health dashboard (see Phase 0) reports each crate's public-item count and flags foundational crates whose surface grew without a corresponding snapshot update. Growth is information, not failure, but unreviewed growth is the debt to catch.

### Type-system, newtype, and error-handling discipline

The strongest structural tool Rust gives an agent is the type system; making the *wrong* API impossible to write is better than documenting the right one. Treat primitive-obsession, error fuzziness, and missing `#[non_exhaustive]` as debt on the same ledger as a broad `pub`:

1. **Domain newtypes over primitives.** Replace bare `String`/`PathBuf`/`u64`/`Vec<u8>` primitives at module/contract boundaries with named newtypes (`WorkspaceName`, `RoleRef`, `SessionId`, `ContainerId`, `TokenCount`, `MountPath`) that carry their validation at construction. The constructor is where "is this a legal value?" is decided once, so no downstream site re-checks or mis-checks. This is the highest-leverage agent-navigation move: a function signature `fn launch(name: WorkspaceName)` cannot be called with the wrong string, where `fn launch(name: String)` can. Bound the sweep to foundational and contract crates; do not newtype internal hot loops.
2. **Error taxonomy with a hard thiserror/at-binary-boundary rule.** Library and foundational crates define concrete error enums with `thiserror` (typed, `#[from]` source chains, machine-matchable variants); `anyhow` is permitted *only* at binary/`main.rs`/xtask boundaries where errors are reported, not handled. Ban `Box<dyn Error>` and stringly-typed errors (`String`/`&'static str` error variants) in library crates — an agent cannot programmatically recover from a string. A Phase 1 dylint lint can flag `Box<dyn std::error::Error>` and bare-string error variants in `crates/`.
3. **`#[non_exhaustive]` on public enums and structs in API-bearing crates.** Pre-release, every public enum/struct that could gain a variant/field should be `#[non_exhaustive]` now, so a future addition is not an accidental breaking change against the committed public-API snapshot (Phase 2). This is cheap insurance that pays out at the first 1.0.
4. **`Result`-first APIs; no panicking constructors.** Public constructors that can fail take input that has already been validated by the newtype, or return `Result`; the panic-coverage lints (Phase 1) back this up by denying `unwrap`/`expect`/indexing in those paths.
5. **Adopt `async fn in trait` deliberately.** The workspace currently sets `async_fn_in_trait = "allow"`. Before relying on native AFIT broadly, decide the Send-bound story (native AFIT traits are not auto-`Send`, which breaks Tokio `spawn` of trait objects) and prefer returning `Pin<Box<dyn Future + Send>>` or the eventual `trait_upcaster`/return-position-impl-trait-in-trait shape where a `Send` future is required. Document the rule so an agent does not introduce a non-`Send` future that compiles but deadlocks a runtime.
6. **Typestate for phase pipelines.** Where Phase 2 extracts phase contracts (launch, image build, capsule attach/teardown), encode the phase *order* in types: each phase consumes the previous phase's output struct by value and returns the next (`ValidatedProfile → MaterializedWorkspace → ResolvedImage → RunningContainer`), with `#[must_use]` on every phase output. Skipping, reordering, or double-running a phase then fails `cargo check`, and the phase structs double as natural seams for characterization tests and the Phase 3 simulation hooks. This is "make the compiler the oracle" applied to control flow, not just data.

1. Finish or remove the shared TUI runtime half-layer. The preferred path is to dispatch console, launch, and capsule loops through the shared `jackin_tui::runtime` traits so the abstraction is operational, not just type-level.
2. Fold the old `split-input-editor` and settings/editor unification leaves into one config-surface cleanup: extract the remaining editor mounts input handling, then converge settings and workspace editor around shared per-domain edit models, validation, render rows, input/mouse handling, persistence handoff, and parity tests.
3. Replace repeated type-complexity suppressions in settings/editor view builders with named row/view model structs and shared form-section builders.
4. Continue the successful `jackin-console-oppicker` extraction pattern until picker model/planning lives in the pure crate and console/root own only side-effect adapters.

### Deterministic simulation and fault injection

The daemon control plane (client registry, attach/displace, session supervision, protocol routing) is concurrent, I/O-heavy, and exactly the surface where "tests pass, production deadlocks" lives. Deterministic simulation testing is the strongest known tool for that class — FoundationDB and TigerBeetle treat it as their primary correctness strategy, and sled documents the same pattern at smaller scale. Adopt it scoped, not wholesale:

1. **Sim-first control plane.** Run capsule control-protocol and session-supervision tests under [turmoil](https://github.com/tokio-rs/turmoil) (or `madsim` if the API fit proves better): simulated network, deterministic scheduling, seeded randomness, so a failing interleaving replays exactly from its seed. This depends on the Phase 2 daemon decomposition keeping I/O behind ports — the simulation swaps the ports, the logic under test stays identical. Wall-clock-free by construction, so the suite is fast and cannot flake.
2. **Property-based state machines.** Model the client registry, active-client policy, and persistence/reattach as [proptest-state-machine](https://docs.rs/proptest-state-machine) specs: random operation sequences run against a small reference model, shrinking to the minimal failing sequence. This finds the "attach during teardown during resize" interleavings no hand-written test enumerates.
3. **Fault-injection points.** Add [fail](https://docs.rs/fail)-style failpoints at the Docker/PTY/filesystem/process edges, behind a test-only feature, so tests can force the rare branches: container dies mid-launch, control socket drops mid-attach, PTY EOF mid-resize, disk full during materialization. Target: every cleanup-classification branch is reachable by a deliberate failpoint test, not only by luck.
4. **Chaos lane for Docker E2E.** Extend the existing Docker-backed E2E with a scheduled chaos variant that kills containers, drops the control socket, and SIGKILLs the capsule at randomized-but-seeded points, asserting the invariants that must survive: no orphaned containers, no stale state dirs, correct cleanup classification, reattach either works or fails clean. Log the seed so any failure replays.
5. **Scope honestly.** `loom` and `kani` stay out of scope until jackin❯ owns a custom lock-free primitive or a proof-worthy pure invariant; simulation + property + fuzz covers the actual risk profile. Revisit if the daemon ever grows consensus-like state.

### Product invariants as executable policy

The repo-level hard rules in <RepoFile path="AGENTS.md">AGENTS.md</RepoFile> are currently prose backed by review. Prose binds an agent only while it stays in context; tests bind forever. Convert each hard rule into a named policy-test suite that CI runs like any other test:

1. **No silent host writes.** Property/table tests over the mount planner and Docker-args builder: for arbitrary valid configs, the generated mounts and writable paths never include host dotfiles, `~/.config/gh`, `~/.gitconfig`, `.git` control dirs, or host remotes unless the corresponding opt-in is set — and when it is set, the opt-in must be visible in the launch-summary model. The forbidden-path table lives in one source of truth shared between the test and the implementation.
2. **Container paths under `/jackin/` only.** Assert every container-side path any builder emits is under `/jackin/`; a new FHS-root path fails the suite, not review.
3. **Telemetry and error redaction.** Golden/property tests feed canary credentials (tokens, PATs, `op` session values, URL-embedded secrets) through launch/auth paths and assert credential-shaped strings never appear in `clog!`/`cdebug!` output, error chains, or diagnostics JSONL. This is the runtime half of the Phase 1 secret-scanning canary.
4. **Control-socket authorization.** Tests pin who may connect/attach/displace and what an unauthenticated peer can observe, so a protocol refactor cannot silently widen the control surface.

A hard rule without a failing test is a request; these make each one a fact. New hard rules land with their policy test in the same PR.

### Flakes, corpora, and suite hygiene

1. **Flake quarantine as a ratchet.** Configure [nextest](https://nexte.st/) retries to *detect* flakes — a pass-on-retry is reported, never silently absorbed — and maintain the quarantine list as a shrink-only ledger with an owner and reason per entry; new unquarantined flakes fail CI. A flaky test is a lie an agent will believe; bounded quarantine keeps the lie visible until fixed.
2. **Suite wall-time budget and slowest-test report.** Publish per-shard slowest-test artifacts and hold total suite wall time under a Phase 7 ratchet budget. Test latency is agent inner-loop time (Phase 6) as much as CI cost.
3. **Protocol golden corpus and skew tests.** Commit a golden frame corpus for `jackin-protocol` — every frame kind plus adversarial truncations promoted from fuzz finds — and decode it in both directions. The host binary and the capsule binary can skew across an upgrade window; corpus tests plus an explicit version-negotiation test make "old capsule, new host" a tested state instead of an assumed-identical one.
4. **Config migration corpus.** Commit one fixture per historical `config.toml`/`jackin.role.toml` schema version and replay the full migration chain in tests: idempotence, post-migration validity, unknown-field policy. This is the executable half of the five-artifacts-per-schema-change rule in <RepoFile path="PRERELEASE.md">PRERELEASE.md</RepoFile>.
5. **Fuzz corpus management.** Commit minimized fuzz corpora per target and re-seed CI fuzz smoke from them, so the scheduled long fuzz run compounds coverage instead of restarting cold; promote crash finds into the golden corpus.

## Phase 4 — Measure and protect performance

1. Convert benchmark build-only coverage into measured, non-flaky performance reporting for terminal pane rendering, launch/attach, grid resize, scrollback snapshotting, materialization, and diagnostics parsing. Start scheduled/advisory; require only once thresholds are stable.
2. Use deterministic, measurement-noise-free regression gates for CI, not wall-clock criterion. Criterion stays the local statistical harness; for PR gates adopt [iai-callgrind](https://docs.rs/iai-callgrind) (Valgrind/Cachegrind instruction+cache counts, identical across noisy cloud runners) for the pure hot paths — terminal parser, grid render, scrollback snapshot, diagnostics JSONL, env/manifest resolution — and reserve criterion for the integration paths where determinism is impossible. iai-callgrind supports per-event regression limits, which become the perf budget: a PR that regresses instruction count past the limit fails. Track the two-tier shape explicitly so agents know which harness a given path is gated by.
3. Commit per-hot-path perf budgets alongside the file-size budget, as a shrink-only ratchet mirroring `file-size-budget.toml`. A budget entry records the accepted instruction-count/regression limit; exceeding it is a failure, and a measured improvement must shrink the entry. This makes "performance is measured, not guessed" a CI gate, not an aspiration.
4. Avoid cloning full usage views during account materialization by serializing from borrowed views or iterators.
5. Add terminal resize fast paths that reuse row storage for height-only or same-width changes and benchmark resize storms.
6. Parallelize isolated mount materialization only after partitioning by dependency/order group and host repository to avoid `.git` lock contention.
7. Stream diagnostics JSONL into typed borrowed records and parse nested detail only for event kinds that need it.
8. Split local CI orchestration into AI-friendly aliases that mirror CI partitions: lint, policy, tests, snapshots, docs, MSRV, E2E. Preserve full `cargo xtask ci` semantics, but make the smallest correct command discoverable.
9. **Allocation budgets on the existing dhat harnesses.** `dhat` is already wired into `jackin-capsule` and `jackin-term`; turn its output into per-hot-path allocation-count/peak-bytes budgets in the Phase 7 ratchet engine, beside the iai-callgrind instruction budgets. Instruction counts are blind to allocator behavior; the pair covers both axes deterministically.
10. **Operator-perceived latency budgets.** Track CLI cold-start (`jackin --help`, console first frame) with [hyperfine](https://github.com/sharkdp/hyperfine) in a scheduled lane, and input-to-frame latency in a headless TUI harness. These are the numbers an operator feels; budget them like instruction counts, scheduled/advisory first.

## Phase 6 — Optimize for autonomous navigation and feedback speed

The other phases make the code correct and structured; this phase makes an agent *fast* at moving through it. An agent's quality is bounded by its edit→compile→lint→test loop time and by how little of the codebase it must read to find the right edit location. Treat both as first-class engineering targets, not side effects.

1. **Crate ownership headers (hard rule).** Every `lib.rs` (and `main.rs`) opens with a `//!` block stating, in three lines: *what this crate owns*, *its tier in the dependency graph*, and *the one entry point an agent should copy*. This is the short, always-current in-source version of the per-crate README (the long-form version shipped under Shipped foundation); an agent opening any crate root learns its boundary without leaving the file. Enforce presence with a Phase 1 dylint lint or a `cargo xtask lint headers` check.
2. **Narrowest-verification map, generated.** Generate (or audit) a machine-checkable map from each crate/module to its narrowest passing verification command — targeted `cargo nextest run -p <crate>`, `cargo clippy -p <crate>`, the snapshot lane, etc. Surface it in <RepoFile path="TESTING.md">TESTING.md</RepoFile> as a table an agent can grep, so the smallest correct command for a touched surface is discoverable in one lookup instead of reconstructed by trial. Phase 4 item 8 (CI aliases) is the operator-facing half of the same idea.
3. **Feedback-loop budget.** Measure and protect the clean-build and incremental-build times for the crates an agent touches most (runtime, capsule, console, term, config). Keep the agent-facing inner loop under a stated budget; regressions trip the same kind of gate as the file-size budget. Levers: keep crates small and dependency-narrow so incremental recompilation is localized (this is also why Phase 2 crate splitting matters for *speed*, not just structure); enable `sccache` and a fast linker (`mold`/`lld`) in CI and document the local setup; keep `codegen-units` and `profile.dev` tuned for compile speed over micro-opt in dev; prefer `cargo nextest` (already adopted) for parallel test discovery. Compile time is agent throughput — budget it like any other resource.
4. **Make the type system decide.** Where a runtime invariant can instead be a type, a sealed trait, a `#[non_exhaustive]` enum, a newtype, or a state-machine type, do that. The goal is to maximize the fraction of "is this change safe?" that `cargo check` answers before any agent or human reasons about it — the strongest agent-velocity lever in Rust, and the through-line of the 2026 "AI writes Rust" consensus.
5. **Determinism over cleverness.** Prefer code that behaves identically across machines and toolchain patches: no wall-clock-dependent tests (use an injected clock, as Phase 3's test-support crate already plans), no filesystem-order-dependent iteration (sorted/globbed), no floating-point in equality assertions. Deterministic tests are tests an agent can trust to mean what they say.
6. **Machine-readable gates with fix-stating output.** Implement the "diagnostics are prompts" principle across every `cargo xtask lint`/ratchet gate: `--format json`, GitHub problem-matcher annotations with file:line on the violating item, and failure text that states the clearing edit plus the narrowest rerun command. Measure gate quality by whether an agent clears a red gate in one turn without opening the gate's source.
7. **Context-economy budgets.** Token-count budgets (a bytes/4 approximation is sufficient) for each AGENTS file, crate README, and `lib.rs` ownership header, tracked in the Phase 7 ratchet engine. The repo's own finding is that duplicated orientation prose measurably reduces agent task success, and every agent pays these files' token cost every session — a budget forces curation over accretion. The [agents.md](https://agents.md/) convention's minimalism guidance points the same way.
8. **Archaeology and IDE parity.** Add a `.git-blame-ignore-revs` file fed by mass-mechanical commits (fmt sweeps, rename sweeps) so `git blame` stays a useful agent tool through the Phase 2 restructures, and add an advisory rust-analyzer cleanliness lane — the workspace loads without proc-macro or config errors — because agents increasingly navigate through the LSP, and a broken rust-analyzer workspace silently downgrades them to grep.

## Phase 7 — Close the loop: the program measures itself

Phases 0–6 install static oracles. This phase installs the feedback loops that generate the *next* oracles and verify the program is actually working — the difference between a strict codebase and a self-tightening one.

1. **Defect→gate ledger.** An append-only in-repo ledger, one row per escaped defect: symptom, root cause, the characterization test added, and the gate/lint/budget adopted — or the written reason none is practical. Sources: operator reports, CI-missed regressions, and the panic hooks already installed in `jackin-usage`/`jackin-diagnostics`, whose captured incidents should flow into triage rather than logs nobody reads. Review the ledger when choosing the next lint or category adoption, so gate selection follows observed failure classes instead of taste. Discipline: no defect closes without leaving a gate or a reason behind.
2. **Unified ratchet engine.** One declarative `ratchet.toml` plus one xtask engine implementing shrink-only semantics once, powering every budget this page names: file size (exists today and is the model), per-lint suppression counts, complexity thresholds, perf/allocation budgets, suite wall time, doc token budgets, public-item counts. Each entry declares its metric source, bound, and direction; the engine owns comparison, fix-stating failure text, JSON output, and the shrink rule. N budgets, one implementation, zero bespoke gate code per new metric.
3. **Self-tightening ratchets.** A scheduled lane recomputes each ratchet's actual current value and, where measured headroom exists (largest file well under budget, a lint's suppression count fell), opens a PR tightening the bound to the new floor — including proposing `clippy.toml` threshold reductions when the measured maxima drop below the configured ones. The ratchet turns itself; humans review a diff, not a dashboard.
4. **Agent-legibility benchmark ("golden agent tasks").** The program's stated goal is that agents modify jackin❯ safely and fast — measure that directly instead of inferring it from lint counts. A scheduled, cost-bounded lane (weekly or monthly) runs a pinned coding-agent configuration against a small fixed task suite in a sandboxed checkout: add a config field end-to-end with its schema artifacts, add a TUI keybinding with docs, thread a new protocol frame host↔capsule, fix a seeded bug drawn from the defect ledger. Score per task: gates-green success, turns, tokens, wall time. Trend it across main: a structure or docs change that degrades task success is a navigability regression this program caused and must be able to see. Honest framing: nondeterministic and priced, so it is a trend instrument, never a PR gate; pin the agent and model version per baseline and re-baseline deliberately.
5. **Health-history telemetry.** Append per-main-commit metrics — suppression counts, ratchet values, coverage trends, benchmark scores — as a JSONL artifact or data branch, and render the trend in the Phase 0 dashboard. Status claims on this page then cite measured history, and "did Phase N help?" becomes an answerable question.
6. **Agent-operated hygiene lane.** Once gates are machine-readable (Phase 6) and ratchets unified, run the maintenance itself through a scheduled agent: re-justify or remove stale `#[expect]`s, apply ratchet tightenings, refresh README/`lib.rs` headers after structural changes, propose the next lint adoption with a measured dry-run report. Every such PR passes the full gate wall like any other contributor's — the enforcement program becomes the first consumer of the agent-safe codebase it builds.

## Phase 8 — Telemetry overhaul: one observable story per run

jackin❯ already streams every run to an OpenTelemetry backend keyed by a run id ([Run Telemetry](/guides/run-telemetry/)), and [Parallax](https://github.com/tailrocks/parallax) is the reference backend — jackin❯ is its first real product integration, and the [parallax-telemetry-playground](https://github.com/tailrocks/parallax-telemetry-playground) is the instrumentation-quality bar this phase borrows from. A live-data audit against Parallax (2026-07-03) measured what the propagation produced as of that date: 968,593 `DEBUG` log records versus 8,977 `INFO`, 4 `WARN`, and 1 `ERROR` across 34 runs; console prefixes like `[jackin debug docker]` baked into exported bodies; one recurring attach failure fingerprinted into 14 separate single-event issues by container/user/run ids; twelve identical `launch_stage` waterfall rows; 109,579 dependency-internal `emit_insn` spans; and Docker-inspect payloads with token-shaped values reaching the backend. The full diagnostic record, the layered-verbosity model, the typed-operation design, and the staged implementation plan with acceptance checks live in [Parallax Observability Findings](/reference/research/agent-telemetry/parallax-observability-findings/) — that dossier is the canonical detailed plan; this phase tracks the program and installs the gates that keep it fixed once fixed.

The substrate is real and stays: OTLP/gRPC export is a default feature of the binary (spans, a tracing→logs bridge, and a periodic metrics reader with process CPU/memory, tokio runtime gauges, and cache counters), launch stages export as spans carrying `otel.name = launch.<stage>` with `ERROR` status on failure, every console screen opens a root trace span-linked to the previous screen, `traceparent` propagates host→capsule, the JSONL event schema already carries `event.name`/`event.outcome`/`jackin.component`/`jackin.operation`/`jackin.category`, `JACKIN_OTEL_INTERNAL` gates dependency-internal export, `JACKIN_TELEMETRY_LEVEL`/`JACKIN_TELEMETRY_CATEGORIES` exist beside `JACKIN_DEBUG`, both host and capsule install panic hooks that bridge an `ERROR` record and flush on exit, and a wrong OTLP protocol is a fatal `E016` instead of silent non-delivery. The remaining program is convergence and discipline, not a green field:

1. **One telemetry facade over the two macro stacks.** The capsule logs through `clog!`/`cdebug!` into a plain-text `multiplexer.log` (`jackin-usage`) while the host logs through `debug_log!` into the JSONL run (`jackin-diagnostics`) — different macros, prefixes, sinks, and files, joined only by a path pointer in the `container_started` event. Unify behind the dossier's typed operation API (`operation_span` / `operation_log` / `operation_error` / `operation_metric`) with per-sink renderers, adopted at the shared choke points first — `ShellRunner`, the HTTP/download helpers, Docker lifecycle calls, the launch stage/timing API, and the capsule attach/session state machine — so every long or failure-prone operation gets a stably named span, a typed outcome (`success`, `failure`, `timeout`, `cancelled`, `cache_hit`, `expected_close`), a duration, and low-cardinality attributes on both sides of the container boundary. Console keeps its bracket prefixes at the rendering boundary; exported bodies are concise text over structured fields. The two-tier rule in <RepoFile path="ENGINEERING.md">ENGINEERING.md</RepoFile> — and the per-crate AGENTS rules that pin the current macro contract — are rewritten to the layered model in the same PR that lands this.
2. **Layered verbosity with per-sink filters.** Finish retiring the binary `--debug`/`JACKIN_DEBUG` as a telemetry control in favor of `JACKIN_TELEMETRY_LEVEL=info|debug|trace` plus `JACKIN_TELEMETRY_CATEGORIES` (the level/category plumbing exists; the launch path still injects `JACKIN_DEBUG=1` into containers), and give each sink — console, diagnostics file, OTLP logs, OTLP spans — its own filter so console compactness and backend richness stop being one knob. `TRACE` owns the byte/frame/keypress firehose; `DEBUG` answers "why did it choose this path" (cache decisions, plan selection, retry summaries, config-source resolution); `INFO` alone must reconstruct a normal launch on the backend's run page.
3. **Correlation must survive without a backend.** Cross-screen links, session ids, and `traceparent` propagation are compiled and active only on the OTLP path today, the JSONL `trace_id` is just the run id, and `multiplexer.log` lines carry no run/session context — so the default file-mode story cannot be stitched. Weave run/session/trace context into the file sinks, make the JSONL carry real trace/span ids, and collapse the duplicated host-vs-capsule OTLP layer construction (the code itself warns the two init paths must be edited in tandem) into one shared builder so they cannot drift.
4. **Trace shape: inherited context and linked subtraces everywhere.** Logs emitted inside a stage must inherit the active span context — a launch trace showing `Logs: 0` is the defect — and the shipped screen/launch/capsule link model extends to the remaining big subsystems: launch ↔ BuildKit/buildx build, capsule session ↔ agent sessions, usage refresh ↔ provider probes, all via standard OTel span links (backends may derive parent-trace navigation fields from them). Dependency-internal spans (`emit_insn`, storage-engine loops) stay out of default export now that `JACKIN_OTEL_INTERNAL` is the opt-in.
5. **Per-crate, per-step, per-screen, per-feature dimensions.** Per-crate visibility falls out of `tracing` targets plus per-crate filters; per-step is the workflow span taxonomy; per-screen builds on the shipped screen traces by stamping an `app.screen.name`-style attribute on every log, span, and interaction metric emitted while a screen is active (console screen, capsule tab, modal id) so cohort questions slice by surface; per-feature means feature-flag and config-toggle decisions are emitted as flag-evaluation events (OTel feature-flag semconv shape: key, variant, provider) for the isolation profile, auth-forward mode, keep-awake, and the compiled cargo feature set — "what was enabled when this failed" becomes a query, not archaeology.
6. **Metrics own the high-frequency internals.** The runtime gauges shipped; extend the metric set so terminal bytes, render/frame durations, painted cells, mouse-event counts, and DB statement counts become counters/histograms instead of log rows, from both host and capsule. Metric names follow `domain.thing.unit` with low-cardinality labels only.
7. **Error semantics that group.** Real failures emit an `ERROR` log plus span status `ERROR` plus a stable `error.type`; expected shutdowns are typed outcomes, not `failed`-shaped `INFO` bodies; issue fingerprints exclude container ids, run ids, paths, uids, and command text so one recurring failure is one issue with N events. The panic hooks feed this path — and the Phase 7 defect ledger.
8. **Redaction before every sink.** One shared redaction layer ahead of console, file, and OTLP: key-pattern and value-pattern scrubbing, typed summaries instead of raw Docker inspect JSON / env arrays / generated Dockerfiles, artifact links for large payloads. The Phase 3 telemetry-redaction policy tests are the gate: canary credentials pushed through launch/auth paths must never reach any sink.
9. **Semconv registry and a telemetry conformance suite as the gate.** A workspace semconv-style constants module becomes the single source for attribute/event/metric names so they cannot drift per crate, and a conformance lane runs a standard launch against an in-process OTLP capture exporter asserting the dossier's acceptance checks as tests: no `[jackin debug` bodies, no token-shaped values, bounded span-name cardinality, `ERROR` on forced failure, logs correlated to their spans, default-mode `DEBUG` volume within budget. That per-run export-volume budget joins the Phase 7 ratchet engine so the firehose cannot creep back, and the Phase 1 dylint telemetry lint enforces the typed API + registry at exporting call sites.
10. **Run-id and sink contract stays honest.** `parallax.run.id` rides every signal; a wrapper-injected id wins over a self-minted one; the JSONL diagnostics file is the fallback sink only; no surface shows a "reveal diagnostics" path for a file that was never written in OTLP mode; every failure surface prints the backend query (`parallax run <id>`, `parallax logs --run <id>`, or the backend-neutral equivalent). The env-var contract is shipped in the [Run Telemetry](/guides/run-telemetry/) guide; the remaining work is the UI-surface honesty items in the dossier's sink-contract section (the capsule container-context fallback still synthesizes a diagnostics path the host never wrote). Agent-session GenAI telemetry (LLM/MCP spans from the agents themselves) stays tracked in the [platform ideas backlog](/reference/research/backlog/platform-ideas/), not here.

### Reference instrumentation shapes

The canonical shapes every Phase 8 implementation PR copies. They compose the idioms already proven in-tree (the screen-trace link model, `otel.name` on stage spans, the OTel meter API with semconv units) with the Parallax conventions and the playground's bootstrap library. Attribute and event names come from the semconv registry module (item 9); the snippets inline string literals only for readability. Everything here follows one rule: **names are stable and low-cardinality; dynamic values are attributes; big payloads are artifacts.**

**An operation span** — one span per long or failure-prone operation, fields declared up front and recorded at completion, error status set through the span (never only body text):

```rust
use tracing::field::Empty;
use tracing_opentelemetry::OpenTelemetrySpanExt as _; // set_status / set_attribute

let span = tracing::info_span!(
    "process.exec",                    // stable name: waterfall-scannable, never interpolated
    otel.kind = "client",
    "jackin.component" = "host",
    "jackin.operation" = "process.exec",
    "process.command" = %program,      // "docker", "gh" — the program, not the argv
    "process.args_redacted" = Empty,   // recorded after redaction
    "process.exit_code" = Empty,
    "event.outcome" = Empty,           // success | failure | timeout | cancelled | expected_close
    "error.type" = Empty,              // stable machine key — this drives issue fingerprints
);

async move {
    let result = spawn_and_wait(command).await;
    let span = tracing::Span::current();
    span.record("process.args_redacted", redact_argv(&argv).as_str());
    match &result {
        Ok(output) => {
            span.record("process.exit_code", output.exit_code);
            span.record("event.outcome", "success");
        }
        Err(err) => {
            span.record("event.outcome", "failure");
            span.record("error.type", err.stable_kind()); // "spawn_failed", not err.to_string()
            span.set_status(opentelemetry::trace::Status::error(err.stable_kind()));
        }
    }
    result
}
.instrument(span)
.await
```

**A log record** — fields carry the structure, the body stays a short human phrase, and the console renderer (not the exporter) adds bracket prefixes:

```rust
// Wrong — the current shape: category and facts baked into the exported body,
// so a backend can only text-search it:
//   body = "[jackin debug docker] inspect container jk-abc -> running"

// Right — the export carries dimensions; the console layer may still render
// the bracketed line for humans:
tracing::debug!(
    target: "jackin::docker",          // tracing target = per-crate/module filtering for free
    "event.name" = "docker.container.inspect",
    "event.outcome" = "success",
    "jackin.component" = "host",
    "log.category" = "docker",
    "container.name" = %name,
    "container.state" = %state,
    "container inspected"              // body: concise, prefix-free
);
```

**An error that groups** — severity, span status, and a stable `error.type` say "failure"; dynamic identifiers stay attributes so one recurring failure is one issue:

```rust
tracing::error!(
    "event.name" = "capsule.attach.failed",
    "jackin.operation" = "capsule.attach",
    "jackin.stage" = "attach",
    "error.type" = "attach_socket_eof", // fingerprint input: stable across runs
    "container.name" = %container,      // evidence only — never part of the fingerprint
    "capsule attach failed"
);
// …and the active attach span gets Status::error("attach_socket_eof"), so the
// trace, the log, and the derived issue all agree on the same typed failure.
```

**A linked subtrace** — the shipped screen→screen pattern generalized: an independent subsystem gets its own trace root, navigably linked to the workflow that caused it:

```rust
let span = tracing::info_span!("image.build", otel.name = "launch.build_derived_image");
drop(span.set_parent(opentelemetry::Context::new()));  // detach: this is its own trace root
span.add_link(parent_span.context().span().span_context().clone()); // parent↔child across traces
span.set_attribute("build.reason", "capsule_version_changed"); // why the build ran — the question
                                                               // the trace exists to answer
```

**Metrics for the hot path** — high-frequency internals are counters/histograms with semconv units, never per-event log rows (the shipped process/tokio gauges already use this meter):

```rust
let bytes_sent = meter
    .u64_counter("jackin.terminal.bytes_sent")
    .with_unit("By")
    .build();
let render = meter
    .f64_histogram("jackin.render.duration")
    .with_unit("ms")
    .build();

bytes_sent.add(chunk.len() as u64, &[KeyValue::new("jackin.component", "capsule")]);
render.record(elapsed_ms, &[KeyValue::new("app.screen.name", screen.as_str())]);
```

**A feature-flag decision** — config toggles emit an evaluation event so "what was enabled" is a filter, not archaeology (verify field names against the current OTel feature-flag semconv when implementing):

```rust
tracing::info!(
    "event.name" = "feature_flag.evaluation",
    "feature_flag.key" = "isolation.profile",
    "feature_flag.provider.name" = "jackin-config",
    "feature_flag.result.variant" = %profile,   // "standard" | "rootless-dind" | …
    "flag evaluated"
);
```

**The conformance assertion** — the item-9 suite runs a launch against an in-process OTLP capture exporter and pins these shapes as tests, so the reference above is enforced, not aspirational:

```rust
let telemetry = capture.finish();
assert!(telemetry.log_bodies().all(|b| !b.contains("[jackin debug")));
assert!(telemetry.all_strings().all(|s| !s.contains(CANARY_TOKEN)));
assert!(telemetry.span_names().all(|n| semconv::REGISTERED_SPANS.contains(&n)));
assert_eq!(telemetry.error_logs("attach_socket_eof").count(), 1);
```

## Research and prior art

External references that shaped the stricter-Clippy, public-API-surface, simulation-testing, and agent-loop direction of this page. Snapshot: 2026-07. Re-verify each before borrowing an implementation detail; blog posts are context, not the only evidence.

- [Your Clippy Config Should Be Stricter](https://emschwartz.me/your-clippy-config-should-be-stricter/) (Evan Schwartz, 2026-04-30) — the motivating taxonomy of panic-coverage, silent-failure, async-safety, and numeric-correctness restriction lints, framed explicitly as guardrails for coding agents. The Phase 1 "Panic-coverage and silent-failure restriction lints" subsection adopts its groupings.
- [Your Clippy Config Should Be Stricter-er](https://billylevin.dev/posts/clippy-config/) (Billy Levin, 2026-04-30) — the counter-argument for allowlist adoption: enable whole `pedantic`/`restriction` categories and justify each suppression, so a useful lint can never be overlooked. The Phase 1 "Stricter-er adoption track" pilots this posture.
- [egui workspace lint configuration](https://github.com/emilk/egui/blob/main/Cargo.toml) — a widely-referenced, production-grade curated lint table; used as a cross-check for which restriction/nursery lints are worth the noise in a real workspace.
- [epage `_rust` template](https://github.com/epage/_rust) — a maintained lint baseline used as the merge base across many Rust projects.
- [dylint](https://github.com/trailofbits/dylint) and the [Trail of Bits introduction](https://blog.trailofbits.com/2021/11/09/write-rust-lints-without-forking-clippy/) — the mechanism behind the Phase 1 "Custom project lints" plan: enforce jackin❯-specific invariants without forking Clippy.
- [cargo-semver-checks](https://github.com/obi1kenobi/cargo-semver-checks) and Predrag Gruevski's notes on [sealed traits](https://predr.ag/blog/four-challenges-cargo-semver-checks-has-yet-to-tackle/) and [`doc(hidden)` API surface](https://predr.ag/blog/checking-semver-for-doc-hidden-items/) — why sealing and an explicit public-API snapshot (Phase 2) make the surface machine-checkable.
- [cargo-public-api](https://github.com/Enselic/cargo-public-api) — the committed-snapshot mechanism for the Phase 2 public-API gate.
- [iai-callgrind](https://docs.rs/iai-callgrind) — deterministic, Valgrind-based regression counts for the Phase 4 perf budget; chosen over wall-clock criterion for CI gates.
- [arch-lint](https://crates.io/crates/arch-lint), [layered-crate](https://crates.io/crates/layered-crate), and [boundary-rust](https://lib.rs/crates/boundary-rust) — candidate tools for the Phase 2 layered-architecture gate, evaluated against a `cargo metadata` + petgraph xtask.
- r/rust [curated lint list (2026-03)](https://www.reddit.com/r/rust/comments/1rxbygj/curated_list_of_clippy_lints_worth_adding_to_the/) — community consensus on the high-impact baseline (`unreachable_pub`, `unused_qualifications`, `missing_docs`, rustdoc `broken_intra_doc_links`), several of which close jackin❯ gaps.
- [Large Rust Workspaces](https://matklad.github.io/2021/08/22/large-rust-workspaces.html) (matklad) — crate-boundary and layout guidance that motivates the per-crate README + AGENTS authority model and the slim root structure doc.
- [Best practice for documenting crates: README vs doc comments](https://users.rust-lang.org/t/best-practice-for-documenting-crates-readme-md-vs-documentation-comments/124254) — the split the per-crate convention follows: the README is the GitHub/crates.io + agent-landing record (purpose, structure, public API); rustdoc `//!`/`///` comments feed docs.rs and the Phase 1 doc gates. The two are cross-referenced, not duplicated.
- [turmoil](https://github.com/tokio-rs/turmoil) (tokio-rs) — deterministic network simulation for Tokio; the Phase 3 sim-first control-plane harness. [FoundationDB's testing story](https://apple.github.io/foundationdb/testing.html), [sled's simulation guide](https://sled.rs/simulation.html), and [Antithesis](https://antithesis.com/) are the prior art for treating deterministic simulation as the primary correctness strategy for concurrent control planes.
