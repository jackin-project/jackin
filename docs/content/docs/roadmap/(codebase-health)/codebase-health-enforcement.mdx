## Goal

Make jackin❯ easy to refactor by humans and AI agents: compiler/tooling gates catch predictable bugs before review, crate and module boundaries make the right edit location obvious, critical behavior has executable characterization coverage, performance hot paths are measured instead of guessed, and every surviving exception is justified rather than inherited. This page stays focused on ordinary Rust quality, structure, tests, docs, and performance.

The project is pre-release, so large restructures are acceptable. The priority is not minimizing migration cost; the priority is lowering future defect rate and making the codebase easy for agents to understand, modify, verify, and review.

The operating assumption for this program is that the most frequent contributor is an autonomous agent, not a human at a keyboard. That flips the usual lint calibration. A `warn` is a human grace period — a hint that something *could* be fixed later. An agent does not need a grace period: it can read the lint message, write the fix, and re-run the gate in the same turn. So the target ruleset is **deny by default, with `reason =`-justified `#[expect]` as the only escape hatch**, even for lints that conventional projects leave at `warn` because their fix is “tedious.”

## Principles for the remaining work

- **Ratchet, do not posture.** Add a strict gate only after a measured dry run proves the signal is actionable; otherwise start as an advisory artifact or scheduled hygiene lane.
- **Refactor with characterization first.** Launch, capsule daemon, isolation materialization, and TUI input changes need fast tests around observable behavior before body extraction.
- **Make exceptions auditable.** Every surviving `allow`/`expect` should have a narrow scope and a reason; unreasoned broad suppressions are debt even when CI is green.
- **Optimize for AI navigation.** A future agent should know which crate owns a concept from the codebase map, should see one canonical API to copy, and should have a focused verification command for the touched surface.
- **Deny by default; optimize the ruleset for autonomous enforcement.** Because the primary contributor is an agent, calibrate every gate toward `deny` + reasoned `#[expect]`, not `warn`. Prefer allowlist-style adoption (turn a category on, then justify each suppression) over denylist-style cherry-picking once the Phase 0 inventory is in hand.
- **Make the compiler and the lints the oracle.** Maximize the share of “is this correct?” that an answerable machine check decides before a human or agent guesses. Every invariant that can be a type, a trait bound, a lint, or an architecture rule should be; invariants that live only in prose are invariants an agent will quietly violate.
- **Diagnostics are prompts.** A gate that fails without stating the fix wastes an agent turn. Every first-party gate (xtask lints, ratchets, policy tests) must print, on failure: which rule fired, why it exists, the exact edit or command that clears it, and the narrowest rerun command — and must offer `--format json` plus GitHub problem-matcher annotations so agents parse results instead of scraping logs.
- **Every escaped defect becomes a gate.** A bug that reaches an operator — or a panic that reaches the installed panic hooks — is evidence of a missing oracle, not just a missing fix. Closing it requires a characterization test plus either a new lint/gate/budget that would have caught the class, or a written reason none is practical, recorded in the Phase 7 defect ledger.

## Phase 1 — Tighten Rust and CI strictness

### PR-time gates to promote after dry run

1. **Feature powerset gate**: move at least a scoped `cargo hack check --workspace --feature-powerset --all-targets --locked` lane from scheduled-only hygiene into PR CI, starting with crates that actually expose optional behavior.
2. **Rustdoc documentation tests**: add `cargo test --doc --workspace --locked` to PR CI if the current scheduled job is green enough to promote.
3. **Snapshot hygiene**: pin `cargo-insta` or add a cheap pending-snapshot/diff gate so TUI snapshot changes cannot leave `*.pending-snap` files or unreviewed `.snap` diffs behind.

### Advisory or scheduled lanes first

1. **Coverage**: add `cargo-llvm-cov` as an artifact/report, not a global PR fail threshold. Start with crate-group trends for parser/protocol/config/terminal crates and only later ratchet specific critical surfaces.
2. **Miri**: run a narrow scheduled/manual lane for pure crates and deterministic parser/protocol/config tests. Do not attempt full-workspace Miri across Docker, PTY, process, or OS-heavy code.
3. **Sanitizers**: keep PR fuzz fast, but add scheduled sanitizer runs where fuzz targets already exist, especially terminal/parser paths.
4. **Mutation testing**: evaluate `cargo-mutants` only on pure crates with fast deterministic tests, such as protocol/config/manifest/terminal parsing. Do not run it across Docker/TUI integration code.
5. **Public API drift**: add `cargo-public-api` or `cargo-semver-checks` as an informational report for selected API-bearing crates. Because jackin❯ is pre-release, the gate should make breakage intentional rather than forbid it.
6. **Workspace-hack experiment**: use `cargo hakari` only as an advisory investigation. Workspace-hack value should be proven with cargo timing artifacts before adding generated maintenance surface.

### Lint ratchets and suppression hygiene

1. Convert broad `#[allow(...)]` suppressions to narrow `#[expect(..., reason = "...")]` where the expectation is stable across targets. Keep `allow` only where `expect` would be flaky under cfg/test/target differences.
2. Add an xtask or grep gate that blocks new `allow`/`expect` attributes without `reason =`, except for documented allowlist patterns such as generated code or cross-target imports.
3. Revisit globally allowed maintainability lints one at a time: `needless_pass_by_value`, `large_futures`, `unused_async`, `assigning_clones`, `match_same_arms`, `drop_non_drop`, `unused_self`, and other explicitly allowed pedantic lints. Promote only low-noise lints; keep noisy ones documented.
4. Ratchet `too_many_lines`, `cognitive_complexity`, `excessive_nesting`, and `too_many_arguments` thresholds only after the fresh inventory identifies the next safe lower bucket.
5. Add a per-lint suppression-count ratchet: a shrink-only ledger (same mechanism as the file-size budget, folded into the Phase 7 unified ratchet engine) recording how many `#[expect]`s exist per lint per crate. Suppression debt becomes visible, bounded, and monotonically shrinking between category adoptions.

### Panic-coverage and silent-failure restriction lints — the agent defect surface

The workspace already denies `unwrap_used`, `expect_used`, `panic`, `todo`, and `unimplemented`, but that misses the panic families a coding agent introduces most often: byte-boundary string/array slicing, swallowed results, dropped futures, and lossy numeric casts. Adopt these as the long-term `deny` target, each behind a Phase 0 dry run:

- **Don't panic (slice/index coverage):** `string_slice`, `indexing_slicing`, `get_unwrap`, `unwrap_in_result`, `unchecked_time_subtraction`, `panic_in_result_fn`. These complement the existing unwrap/expect/panic denials by catching `&s[a..b]`, `arr[i]`, and panics inside `Result`-returning functions. `string_slice` and `indexing_slicing` will fire widely on day one; route every hit through `.get()`/`.split_at_checked()`/iterator APIs.
- **Don't fail silently:** `let_underscore_future`, `let_underscore_must_use` (jackin❯ already denies `let_underscore_drop`; close the family), `unused_result_ok`, `map_err_ignore`, `assertions_on_result_states`. These catch `let _ = future`, `r.ok();`, and `.map_err(|_| …)` — silent error loss that is invisible to tests.
- **Don't do bad async:** `await_holding_lock`, `await_holding_refcell_ref`. jackin❯ runs the capsule daemon and runtime on Tokio with non-`Send` guards held across `.await` in places; these are the deadlock lint pair and are high-value against the capsule control plane. (`if_let_mutex` is edition-2024-resolved and can stay off.)
- **Don't do incorrect things with numbers:** `float_cmp_const`, `lossy_float_literal`, `cast_sign_loss`, `invalid_upcast_comparisons`, and re-evaluate the current `float_cmp = "allow"`. Treat `cast_possible_truncation`/`cast_precision_loss`/`cast_possible_wrap` as the optional “force a documented invariant on every lossy cast” tier once the loud set is clean.
- **Easy to avoid:** `rc_mutex`, `debug_assert_with_mut_call`, `expl_impl_clone_on_copy`, `infallible_try_from`, `iter_not_returning_iterator`.

Add the matching `clippy.toml` escape valves (`allow-unwrap-in-tests`, `allow-expect-in-tests`, `allow-panic-in-tests`, `allow-print-in-tests` are the ones present today; add `allow-indexing-slicing-in-tests`, `allow-dbg-in-tests`, and the rest as each lint lands) so the denials apply to production and tests stay ergonomic.

### Meta-suppression discipline — make every `allow` a reasoned `expect`

1. **`allow_attributes = "deny"`** — every `#[allow(...)]` must become `#[expect(...)]`. An `expect` is self-documenting and fails the build if the underlying lint ever stops firing, so suppressions cannot outlive their reason.
2. **`allow_attributes_without_reason = "deny"`** — every `#[expect(...)]` (and any surviving `#[allow]`) must carry `reason = "…"`. An agent that disables a lint must state why; the build rejects a bare suppression.
3. A reason-gate xtask lane (Plan 011) becomes redundant once these two lints deny; keep it only for the generated-code / cross-target allowlist carve-outs the lints cannot express.

### Rustdoc and missing-docs gates

1. Add `[workspace.lints.rustdoc]` with `broken_intra_doc_links = "deny"`, `private_intra_doc_links = "warn"`, `redundant_explicit_links = "warn"`, and `unescaped_quotes_in_doc_comment = "warn"` so rustdoc is a real check, not decoration.
2. Promote `missing_debug_implementations` to `deny` once the remaining gaps are filled (it is already CI-promoted via `-D warnings`, so the delta is making it local-fail too).
3. Turn on `missing_docs` (the `rust` lint) for the foundational, API-bearing crates — `jackin-core`, `jackin-config`, `jackin-manifest`, `jackin-protocol`, `jackin-env`, `jackin-term` — not for binaries or `jackin-xtask`. Gate crate-by-crate behind the dry run rather than workspace-wide, because the first adoption sweep is large.
4. Keep `cargo test --doc --workspace` (Phase 1 PR-time gate) as the executable half of the rustdoc gate.

### Custom project lints via dylint

When a jackin❯-specific invariant cannot be expressed by any shipped lint, encode it as a workspace-owned [dylint](https://github.com/trailofbits/dylint) library run from CI, not as a prose rule an agent will drift from. Routine first-party candidates:

- **Render-thread purity:** no blocking I/O, allocation storms, or `std::sync` locks reachable from the terminal/console draw path; the lint enforces what method-level lists cannot express structurally.
- **Foundational-crate surface:** every `pub` type in a foundational crate implements `Debug`, and extension traits are sealed.
- **Config/manifest field consistency:** new `jackin.role.toml` / `config.toml` fields carry a `#[serde]` + a doc + a migration entry together; the lint flags a field added to a struct without the matching schema-version bump.
- **Telemetry discipline:** exporting call sites go through the typed operation API and the semconv name registry (Phase 8), not ad-hoc `log::`/`tracing::` free-form strings; `clog!`/`cdebug!` stay legal as console/file renderers only.

Author the lint library under `crates/jackin-lints/`, gate it advisory first (warn in a scheduled lane), and promote to PR-blocking once the false-positive rate is measured. dylint lints compile against a nightly-internal `rustc` API, so isolate them in their own crate and run them in the scheduled lane or a pinned-toolchain CI job rather than the default PR toolchain.

### Whole-category “Stricter-er” as an adoption track

Two defensible philosophies exist for adopting restriction lints: cherry-pick or turn whole categories on and justify each suppression. For jackin❯, run the Phase 0 inventory, then pilot the **allowlist track** on one low-risk pure crate: set `restriction = { level = "warn", priority = -1 }` for that crate only, fix or `#[expect(…, reason)]` every hit, and measure the ongoing maintenance cost. If the maintenance cost is low and the defect catch rate high, roll the allowlist posture workspace-wide and invert the existing allow-set: most current `allow`s become either justified `#[expect]`s or accepted warns.

### Toolchain canaries

1. **Beta-toolchain clippy canary.** A scheduled lane runs `cargo clippy --all-targets --all-features` (and optionally the test suite) on the next Rust beta, so new lints, lint renames, and behavior changes surface weeks before the pinned toolchain moves. Fixes or `#[expect]`s land ahead of the upgrade instead of blocking it; on a deny-by-default workspace, a toolchain bump should be a planned chore, never a surprise red main.
2. **Auditable dependency timing.** Continue using cargo timing artifacts to justify dependency or workspace-shape changes instead of adding generated maintenance surface on taste alone.

### Type-system, newtype, and error-handling discipline

1. **Domain newtypes over primitives.** Replace bare `String`/`PathBuf`/`u64`/`Vec<u8>` primitives at module/contract boundaries with named newtypes (`WorkspaceName`, `RoleRef`, `SessionId`, `ContainerId`, `TokenCount`, `MountPath`) that carry their validation at construction. Bound the sweep to foundational and contract crates; do not newtype internal hot loops.
2. **Error taxonomy with a hard thiserror/at-binary-boundary rule.** Library and foundational crates define concrete error enums with `thiserror` (typed, `#[from]` source chains, machine-matchable variants); `anyhow` is permitted only at binary/`main.rs`/xtask boundaries where errors are reported, not handled. Ban `Box<dyn Error>` and stringly-typed errors (`String`/`&'static str` error variants) in library crates.
3. **`#[non_exhaustive]` on public enums and structs in API-bearing crates.** Pre-release, every public enum/struct that could gain a variant/field should be `#[non_exhaustive]` now, so a future addition is not an accidental breaking change against the committed public-API snapshot.
4. **`Result`-first APIs; no panicking constructors.** Public constructors that can fail take input that has already been validated by the newtype, or return `Result`; the panic-coverage lints (Phase 1) back this up by denying `unwrap`/`expect`/indexing in those paths.
5. **Adopt `async fn in trait` deliberately.** The workspace currently sets `async_fn_in_trait = "allow"`. Before relying on native AFIT broadly, decide the Send-bound story and prefer returning `Pin<Box<dyn Future + Send>>` or the eventual return-position-impl-trait-in-trait shape where a `Send` future is required.
6. **Typestate for phase pipelines.** Where Phase 2 extracts phase contracts (launch, image build, capsule attach/teardown), encode the phase order in types: each phase consumes the previous phase's output struct by value and returns the next (`ValidatedProfile → MaterializedWorkspace → ResolvedImage → RunningContainer`), with `#[must_use]` on every phase output.

### Deterministic simulation and fault injection

1. **Sim-first control plane.** Run capsule control-protocol and session-supervision tests under [turmoil](https://github.com/tokio-rs/turmoil) (or `madsim` if the API fit proves better): simulated network, deterministic scheduling, seeded randomness, so a failing interleaving replays exactly from its seed. This depends on the Phase 2 daemon decomposition keeping I/O behind ports — the simulation swaps the ports, the logic under test stays identical.
2. **Property-based state machines.** Model the client registry, active-client policy, and persistence/reattach as [proptest-state-machine](https://docs.rs/proptest-state-machine) specs: random operation sequences run against a small reference model, shrinking to the minimal failing sequence.
3. **Fault-injection points.** Add [fail](https://docs.rs/fail)-style failpoints at the Docker/PTY/filesystem/process edges, behind a test-only feature, so tests can force rare cleanup and teardown branches.
4. **Chaos lane for Docker E2E.** Extend the existing Docker-backed E2E with a scheduled chaos variant that kills containers, drops the control socket, and sends SIGKILL to the capsule at randomized-but-seeded points, asserting the invariants that must survive: no orphaned containers, no stale state dirs, correct cleanup classification, reattach either works or fails clean. Log the seed so any failure replays.
5. **Scope honestly.** `loom` and `kani` stay out of scope until jackin❯ owns a custom lock-free primitive or a proof-worthy pure invariant; simulation + property + fuzz covers the actual risk profile.

### Product invariants as executable policy

The repo-level hard rules in <RepoFile path="AGENTS.md">AGENTS.md</RepoFile> are currently prose backed by review. Prose binds an agent only while it stays in context; tests bind forever. Convert hard rules into named policy-test suites that CI runs like any other test:

1. **Container paths under `/jackin/` only.** Assert every container-side path any builder emits is under `/jackin/`; a new FHS-root path fails the suite, not review.
2. **Role/config schema invariants.** Every schema-changing PR updates the five required artifacts from <RepoFile path="PRERELEASE.md">PRERELEASE.md</RepoFile> and proves migration idempotence.
3. **Behavioral spec coverage.** Behavioral specs cite tests for each section, and the spec↔test linkage gate in Phase 5 keeps those links alive.

A hard rule without a failing test is a request; these make each one a fact.

### Flakes, corpora, and suite hygiene

1. **Flake quarantine as a ratchet.** Configure [nextest](https://nexte.st/) retries to detect flakes — a pass-on-retry is reported, never silently absorbed — and maintain the quarantine list as a shrink-only ledger with an owner and reason per entry; new unquarantined flakes fail CI.
2. **Suite wall-time budget and slowest-test report.** Publish per-shard slowest-test artifacts and hold total suite wall time under a Phase 7 ratchet budget. Test latency is agent inner-loop time (Phase 6) as much as CI cost.
3. **Protocol golden corpus and skew tests.** Commit a golden frame corpus for `jackin-protocol` — every frame kind plus adversarial truncations promoted from fuzz finds — and decode it in both directions. Corpus tests plus an explicit version-negotiation test make “old capsule, new host” a tested state instead of an assumed-identical one.
4. **Config migration corpus.** Commit one fixture per historical `config.toml`/`jackin.role.toml` schema version and replay the full migration chain in tests: idempotence, post-migration validity, unknown-field policy.
5. **Fuzz corpus management.** Commit minimized fuzz corpora per target and re-seed CI fuzz smoke from them, so the scheduled long fuzz run compounds coverage instead of restarting cold; promote crash finds into the golden corpus.

## Phase 4 — Measure and protect performance

1. Convert benchmark build-only coverage into measured, non-flaky performance reporting for terminal pane rendering, launch/attach, grid resize, scrollback snapshotting, materialization, and diagnostics parsing. Start scheduled/advisory; require only once thresholds are stable.
2. Use deterministic, measurement-noise-free regression gates for CI, not wall-clock criterion. Criterion stays the local statistical harness; for PR gates adopt [iai-callgrind](https://docs.rs/iai-callgrind) for the pure hot paths — terminal parser, grid render, scrollback snapshot, diagnostics JSONL, env/manifest resolution — and reserve criterion for the integration paths where determinism is impossible.
3. Commit per-hot-path perf budgets alongside the file-size budget, as a shrink-only ratchet mirroring `file-size-budget.toml`. A budget entry records the accepted instruction-count/regression limit; exceeding it is a failure, and a measured improvement must shrink the entry.
4. Avoid cloning full usage views during account materialization by serializing from borrowed views or iterators.
5. Add terminal resize fast paths that reuse row storage for height-only or same-width changes and benchmark resize storms.
6. Parallelize isolated mount materialization only after partitioning by dependency/order group and host repository to avoid `.git` lock contention.
7. Stream diagnostics JSONL into typed borrowed records and parse nested detail only for event kinds that need it.
8. Split local CI orchestration into AI-friendly aliases that mirror CI partitions: lint, policy, tests, snapshots, docs, MSRV, E2E. Preserve full `cargo xtask ci` semantics, but make the smallest correct command discoverable.
9. **Allocation budgets on the existing dhat harnesses.** `dhat` is already wired into `jackin-capsule` and `jackin-term`; turn its output into per-hot-path allocation-count/peak-bytes budgets in the Phase 7 ratchet engine, beside the iai-callgrind instruction budgets.
10. **Operator-perceived latency budgets.** Track CLI cold-start (`jackin --help`, console first frame) with [hyperfine](https://github.com/sharkdp/hyperfine) in a scheduled lane, and input-to-frame latency in a headless TUI harness. These are the numbers an operator feels; budget them like instruction counts, scheduled/advisory first.

## Phase 5 — Keep docs and maps authoritative

1. Keep the [Codebase Map](/reference/getting-oriented/codebase-map/) generated or audited against `cargo metadata` so crate tiers and “where to start” entries match reality.
2. Keep route-level roadmap docs focused on active work. Resolved setup records should live in this page's closed ledger or in git history, not as sidebar leaves.
3. Keep behavioral specs in `/reference/developer-reference/specs/`; use this roadmap page to track missing specs or missing tests, not duplicate the spec content.
4. Keep user-facing docs free of internal refactor admissions; code-health details belong under roadmap/reference only.
5. **Per-crate authority; the Codebase Map becomes an index.** The per-crate README is now the authoritative source for each crate's purpose, tier, allowed dependencies, structure, and public API. Evolve the single large `codebase-map.mdx` from a hand-maintained module map into a slim index that links into per-crate pages, instead of duplicating per-module structure prose that drifts from the code.
6. **Auto-extract crate README files into Fumadocs.** Add a build-time pipeline (a `scripts/*.ts` sync wired into the docs build) that renders each `crates/*/README.md` as a page under a “Behind jackin❯ — crates” section. The README is the single source of truth; the docs page is generated from it.
7. **Slim <RepoFile path="PROJECT_STRUCTURE.md">PROJECT_STRUCTURE.md</RepoFile>.** Reduce the root structure doc to its irreducible job — the multi-repo ecosystem table and code↔docs pointers — and route every “where does the code for X live / what does crate Y do” question to the per-crate README files and the generated crate pages.
8. **README-freshness gate.** Once the auto-extraction pipeline exists, add a cheap heuristic check that a crate whose `src/` module layout changed in a PR also touched its README, so the “update README on structural change” rule in <RepoFile path="crates/AGENTS.md">crates/AGENTS.md</RepoFile> is enforced, not just documented.
9. **Brand and prose gates.** The `jackin❯` brand rule in <RepoFile path="RULES.md">RULES.md</RepoFile> is mechanically checkable: an xtask docs lint that rejects misspellings across prose surfaces (MDX, README, AGENTS files), with an allowlist for code identifiers, paths, and the documented plaintext fallback. Extend to other RULES.md prose rules as they prove greppable.
10. **Spec↔test linkage gate.** Behavioral specs cite the tests covering each section; add an xtask check that every cited test id still exists, so specs cannot silently rot into fiction.
11. **Documented-command and config-key drift gates.** Extract `jackin …` invocations from docs code fences and `try_parse` them against the real clap command tree in a test, so a renamed flag fails docs CI instead of an operator. Where docs tables describe config keys, generate or check them against the schema artifacts the `schema-check` gate already versions.

## Phase 6 — Optimize for autonomous navigation and feedback speed

The other phases make the code correct and structured; this phase makes an agent fast at moving through it. An agent's quality is bounded by its edit→compile→lint→test loop time and by how little of the codebase it must read to find the right edit location. Treat both as first-class engineering targets, not side effects.

1. **Crate ownership headers (hard rule).** Every `lib.rs` (and `main.rs`) opens with a `//!` block stating, in three lines: what this crate owns, its tier in the dependency graph, and the one entry point an agent should copy. Enforce presence with a Phase 1 dylint lint or a `cargo xtask lint headers` check.
2. **Narrowest-verification map, generated.** Generate (or audit) a machine-checkable map from each crate/module to its narrowest passing verification command — targeted `cargo nextest run -p <crate>`, `cargo clippy -p <crate>`, the snapshot lane, etc. Surface it in <RepoFile path="TESTING.md">TESTING.md</RepoFile> as a table an agent can grep.
3. **Feedback-loop budget.** Measure and protect the clean-build and incremental-build times for the crates an agent touches most (runtime, capsule, console, term, config). Keep the agent-facing inner loop under a stated budget; regressions trip the same kind of gate as the file-size budget.
4. **Make the type system decide.** Where a runtime invariant can instead be a type, a sealed trait, a `#[non_exhaustive]` enum, a newtype, or a state-machine type, do that. The goal is to maximize the fraction of “is this change safe?” that `cargo check` answers before any agent or human reasons about it.
5. **Determinism over cleverness.** Prefer code that behaves identically across machines and toolchain patches: no wall-clock-dependent tests (use an injected clock, as Phase 3's test-support crate already plans), no filesystem-order-dependent iteration (sorted/globbed), no floating-point in equality assertions.
6. **Machine-readable gates with fix-stating output.** Implement the “diagnostics are prompts” principle across every `cargo xtask lint`/ratchet gate: `--format json`, GitHub problem-matcher annotations with file:line on the violating item, and failure text that states the clearing edit plus the narrowest rerun command.
7. **Context-economy budgets.** Token-count budgets (a bytes/4 approximation is sufficient) for each AGENTS file, crate README, and `lib.rs` ownership header, tracked in the Phase 7 ratchet engine. The repo's own finding is that duplicated orientation prose measurably reduces agent task success.
8. **Archaeology and IDE parity.** Add a `.git-blame-ignore-revs` file fed by mass-mechanical commits (fmt sweeps, rename sweeps) so `git blame` stays a useful agent tool through the Phase 2 restructures, and add an advisory rust-analyzer cleanliness lane — the workspace loads without proc-macro or config errors.

## Phase 7 — Close the loop: the program measures itself

1. **Defect→gate ledger.** An append-only in-repo ledger, one row per escaped defect: symptom, root cause, the characterization test added, and the gate/lint/budget adopted — or the written reason none is practical. Review the ledger when choosing the next lint or category adoption, so gate selection follows observed failure classes instead of taste.
2. **Unified ratchet engine.** One declarative `ratchet.toml` plus one xtask engine implementing shrink-only semantics once, powering every budget this page names: file size (exists today and is the model), per-lint suppression counts, complexity thresholds, perf/allocation budgets, suite wall time, doc token budgets, public-item counts. Each entry declares its metric source, bound, and direction.
3. **Self-tightening ratchets.** A scheduled lane recomputes each ratchet's actual current value and, where measured headroom exists, opens a PR tightening the bound to the new floor — including proposing `clippy.toml` threshold reductions when the measured maxima drop below the configured ones.
4. **Agent-legibility benchmark (“golden agent tasks”).** A scheduled, cost-bounded lane runs a pinned coding-agent configuration against a small fixed task suite in a sandboxed checkout: add a config field end-to-end with its schema artifacts, add a TUI keybinding with docs, thread a new protocol frame host↔capsule, fix a seeded bug drawn from the defect ledger. Score per task: gates-green success, turns, tokens, wall time. It is a trend instrument, never a PR gate.
5. **Health-history telemetry.** Append per-main-commit metrics — suppression counts, ratchet values, coverage trends, benchmark scores — as a JSONL artifact or data branch, and render the trend in the Phase 0 dashboard. Status claims on this page then cite measured history.
6. **Agent-operated hygiene lane.** Once gates are machine-readable (Phase 6) and ratchets unified, run the maintenance itself through a scheduled agent: re-justify or remove stale `#[expect]`s, apply ratchet tightenings, refresh README/`lib.rs` headers after structural changes, propose the next lint adoption with a measured dry-run report.

## Phase 8 — Telemetry overhaul: one observable story per run

jackin❯ already streams every run to an OpenTelemetry backend keyed by a run id ([Run Telemetry](/guides/run-telemetry/)), and [Parallax](https://github.com/tailrocks/parallax) is the reference backend. The full diagnostic record, layered-verbosity model, typed-operation design, and staged implementation plan with acceptance checks live in [Parallax Observability Findings](/reference/research/agent-telemetry/parallax-observability-findings/) — that dossier is the canonical detailed plan; this phase tracks the observability program.

1. **One telemetry facade over the two macro stacks.** The capsule logs through `clog!`/`cdebug!` into a plain-text file while the host logs through `debug_log!` into the JSONL run — different macros, prefixes, sinks, and files. Unify behind the dossier's typed operation API (`operation_span` / `operation_log` / `operation_error` / `operation_metric`) with per-sink renderers, adopted at the shared choke points first.
2. **Layered verbosity with per-sink filters.** Finish retiring the binary `--debug`/`JACKIN_DEBUG` as a telemetry control in favor of `JACKIN_TELEMETRY_LEVEL=info|debug|trace` plus `JACKIN_TELEMETRY_CATEGORIES`, and give each sink — console, diagnostics file, OTLP logs, OTLP spans — its own filter so console compactness and backend richness stop being one knob.
3. **Correlation must survive without a backend.** Weave run/session/trace context into the file sinks, make the JSONL carry real trace/span ids, and collapse the duplicated host-vs-capsule OTLP layer construction into one shared builder so they cannot drift.
4. **Trace shape: inherited context and linked subtraces everywhere.** Logs emitted inside a stage must inherit the active span context, and the shipped screen/launch/capsule link model extends to the remaining big subsystems via standard OTel span links.
5. **Per-crate, per-step, per-screen, per-feature dimensions.** Per-crate visibility falls out of `tracing` targets plus per-crate filters; per-step is the workflow span taxonomy; per-screen stamps an `app.screen.name`-style attribute on every log, span, and interaction metric emitted while a screen is active; per-feature emits feature-flag-style decision events for major config toggles.
6. **Metrics own the high-frequency internals.** The runtime gauges shipped; extend the metric set so terminal bytes, render/frame durations, painted cells, mouse-event counts, and DB statement counts become counters/histograms instead of log rows, from both host and capsule.
7. **Error semantics that group.** Real failures emit an error log plus span status plus a stable `error.type`; expected shutdowns are typed outcomes, not failure-shaped bodies; issue fingerprints exclude per-run identifiers so one recurring failure is one issue with N events.
8. **Semconv registry and a telemetry conformance suite as the gate.** A workspace semconv-style constants module becomes the single source for attribute/event/metric names so they cannot drift per crate, and a conformance lane runs a standard launch against an in-process OTLP capture exporter asserting the dossier's acceptance checks as tests. Export-volume budgets join the Phase 7 ratchet engine so firehose regressions cannot creep back.
9. **Run-id and sink contract stays honest.** `parallax.run.id` rides every signal; a wrapper-injected id wins over a self-minted one; the JSONL diagnostics file is the fallback sink only; every failure surface prints the backend query (`parallax run <id>`, `parallax logs --run <id>`, or the backend-neutral equivalent). Agent-session GenAI telemetry stays tracked in the [platform ideas backlog](/reference/research/backlog/platform-ideas/), not here.

## Research and prior art

External references that shaped the stricter-Clippy, public-API-surface, simulation-testing, and agent-loop direction of this page. Snapshot: 2026-07. Re-verify each before borrowing an implementation detail; blog posts are context, not the only evidence.

- [Your Clippy Config Should Be Stricter](https://emschwartz.me/your-clippy-config-should-be-stricter/) (Evan Schwartz, 2026-04-30) — the motivating taxonomy of panic-coverage, silent-failure, async-safety, and numeric-correctness restriction lints, framed explicitly as guardrails for coding agents. The Phase 1 “Panic-coverage and silent-failure restriction lints” subsection adopts its groupings.
- [Your Clippy Config Should Be Stricter-er](https://billylevin.dev/posts/clippy-config/) (Billy Levin, 2026-04-30) — the counter-argument for allowlist adoption: enable whole `pedantic`/`restriction` categories and justify each suppression, so a useful lint can never be overlooked. The Phase 1 “Stricter-er adoption track” pilots this posture.
- [egui workspace lint configuration](https://github.com/emilk/egui/blob/main/Cargo.toml) — a widely-referenced, production-grade curated lint table; used as a cross-check for which restriction/nursery lints are worth the noise in a real workspace.
- [epage `_rust` template](https://github.com/epage/_rust) — a maintained lint baseline used as the merge base across many Rust projects.
- [dylint](https://github.com/trailofbits/dylint) and the [Trail of Bits introduction](https://blog.trailofbits.com/2021/11/09/write-rust-lints-without-forking-clippy/) — the mechanism behind the Phase 1 “Custom project lints” plan: enforce jackin❯-specific invariants without forking Clippy.
- [cargo-semver-checks](https://github.com/obi1kenobi/cargo-semver-checks) and Predrag Gruevski's notes on [sealed traits](https://predr.ag/blog/four-challenges-cargo-semver-checks-has-yet-to-tackle/) and [`doc(hidden)` API surface](https://predr.ag/blog/checking-semver-for-doc-hidden-items/) — why sealing and an explicit public-API snapshot (Phase 2) make the surface machine-checkable.
- [cargo-public-api](https://github.com/Enselic/cargo-public-api) — the committed-snapshot mechanism for the Phase 2 public-API gate.
- [iai-callgrind](https://docs.rs/iai-callgrind) — deterministic, Valgrind-based regression counts for the Phase 4 perf budget; chosen over wall-clock criterion for CI gates.
- [arch-lint](https://crates.io/crates/arch-lint) and [layered-crate](https://crates.io/crates/layered-crate) — candidate tools for the Phase 2 layered-architecture gate, evaluated against a `cargo metadata` + petgraph xtask.
- r/rust [curated lint list (2026-03)](https://www.reddit.com/r/rust/comments/1rxbygj/curated_list_of_clippy_lints_worth_adding_to_the/) — community consensus on the high-impact baseline (`unreachable_pub`, `unused_qualifications`, `missing_docs`, rustdoc `broken_intra_doc_links`), several of which close jackin❯ gaps.
- [Large Rust Workspaces](https://matklad.github.io/2021/08/22/large-rust-workspaces.html) (matklad) — crate-boundary and layout guidance that motivates the per-crate README + AGENTS authority model and the slim root structure doc.
- [Best practice for documenting crates: README vs doc comments](https://users.rust-lang.org/t/best-practice-for-documenting-crates-readme-md-vs-documentation-comments/124254) — the split the per-crate convention follows: the README is the GitHub/crates.io + agent-landing record (purpose, structure, public API); rustdoc `//!`/`///` comments feed docs.rs and the Phase 1 doc gates.
- [turmoil](https://github.com/tokio-rs/turmoil) (tokio-rs) — deterministic network simulation for Tokio; the Phase 3 sim-first control-plane harness. [FoundationDB's testing story](https://apple.github.io/foundationdb/testing.html), [sled's simulation guide](https://sled.rs/simulation.html), and [Antithesis](https://antithesis.com/) are the prior art for treating deterministic simulation as the primary correctness strategy for concurrent control planes.
- [proptest-state-machine](https://docs.rs/proptest-state-machine) — reference-model state-machine property testing, targeted at the daemon registry and attach/displace policies in Phase 3.
- [fail](https://docs.rs/fail) — failpoint injection for forcing rare cleanup/teardown branches.
- [cargo-nextest](https://nexte.st/) retries and flaky-test detection — the mechanism behind the Phase 3 flake-quarantine ratchet.
- [hyperfine](https://github.com/sharkdp/hyperfine) — CLI cold-start measurement for the Phase 4 operator-perceived latency budgets.
- [agents.md](https://agents.md/) — the cross-tool agent-instructions convention; its minimalism guidance backs the Phase 6 context-economy budgets.
- [Parallax](https://github.com/tailrocks/parallax) — the reference OpenTelemetry backend for Phase 8: OTLP-native ingest, derived error events with normalized fingerprints, run-scoped views under `parallax.run.id`, and evidence bundles for coding agents. Its published conventions page and its jackin❯ integration recipe define the backend half of the run-telemetry contract.
- [parallax-telemetry-playground](https://github.com/tailrocks/parallax-telemetry-playground) — the maximum-fidelity OTel sample the Phase 8 patterns are borrowed from: one shared telemetry bootstrap, a semconv wire-name registry module, span links for async hops, OpenFeature flag-evaluation events, `app.screen.name` cohort fields, tokio runtime metric lanes, and telemetry-health scenarios.
