## Guiding principles (binding for every item here)

1. **Correctness over ROI.** Fix what is wrong or inconsistent; never skip an item as "low value". The only reason to stop short is a *proven* impossibility.
2. **Root-cause, architectural fixes.** Every bug means the architecture permitted it. Remove the structural condition and fix the whole class, not the one symptom.
3. **Reuse, never copy-paste.** Recurring concerns (tab focus, hover, cursor, blocking I/O, per-provider headline) are fixed once as a shared mechanism applied everywhere.
4. **Never block.** Blocking work (network, subprocess, DB, file I/O) never runs on the event/input/render task ‚Äî tokio / `spawn_blocking` / workers, independent loads joined.
5. **CodexBar is a reference to verify, not gospel.** Mirror [CodexBar](https://github.com/steipete/codexbar) by default, but verify every datum against its source; where jackin‚ùØ has better data, diverge and document why.

## P0 ‚Äî Claude and OpenAI full snapshots (the two regressions)

**The biggest lever ‚Äî and a class of bug, not two isolated ones.** Live `usage verify` shows both providers resolve via **`source: "cli"` (`confidence: authoritative`)** with only two windows. The architecture *permits a silent degrade*: when the richer source (Claude OAuth, Codex RPC) is unavailable or not yet fetched, the snapshot quietly falls back to an impoverished `cli` snapshot still labelled `authoritative`, so the UI shows a confident-but-truncated result instead of an honest degraded state. **Root-cause / class fix:** the source layer must never present a reduced snapshot as `authoritative` ‚Äî it must either produce the full snapshot from the available source or mark the result degraded with the reason. Fix that structural condition, not just the two tabs.

### P0a ‚Äî Claude (regression; diagnose the credential path FIRST)

Symptom: Session/Weekly show `% left` only ‚Äî no reset, no in-reserve, no `Lasts until reset`; Sonnet and Daily Routines absent. Claude resolves via `push_claude_cli_bucket` (`crates/jackin-usage/src/usage.rs`), the impoverished path. The rich path is OAuth `/api/oauth/usage` ‚Üí `push_claude_window` (reset + in-reserve + Sonnet + Daily Routines).

The root cause is **not yet pinned ‚Äî diagnose before fixing.** All seven providers passed in-container on June 24, so Claude OAuth **regressed**. First step, in-container on a debug launch:

- Check the forwarded Claude credential: `~/.claude/.credentials.json`, `$CLAUDE_CONFIG_DIR/.credentials.json`, `~/.claude.json`, and `ANTHROPIC_API_KEY` / `ANTHROPIC_AUTH_TOKEN` env.
- If the credential is absent or stale ‚Üí the OAuth fetch fails and the snapshot falls back to `cli`: the bug is in host‚Üícontainer credential forwarding (`crates/jackin-instance/src/auth.rs`). `git log` what changed Claude forwarding since the June-24 working state.
- If the credential is present but `cli` still wins ‚Üí the bug is in source selection: the `cli` path is preferred or OAuth is gated. Make OAuth win whenever its credential resolves.

Then restore the Claude sections (in order): **Session, Weekly, Sonnet, Daily Routines** (+ conditional Opus / Extra usage). Session/Weekly carry `Resets in <rel> (<local>)` + in-reserve + `Lasts until reset`; Sonnet carries a reset; Daily Routines shows remaining only. A section appears **only when the API returns data for it** ‚Äî a `null` window (an unused Sonnet/Opus/Daily Routines) is omitted, never rendered as `100% left` (honest degradation; an intentional divergence from CodexBar, which shows null Daily Routines at 100%).

**Resolved** (verified in-container). Two root causes, both proven: (1) `claude_snapshot` never read the runtime handoff `/jackin/claude/credentials.json` (the home copy lacked `claudeAiOauth.accessToken`), so OAuth was skipped and the snapshot dropped to the impoverished CLI fallback ‚Äî commit `90acd5e2` adds the handoff read via a shared `first_credential` resolver (home-first, handoff-last) reused by Claude/Codex/Amp so no provider is missed; (2) with the token resolving, the OAuth 200 body failed to decode because `seven_day_oauth_apps` was aliased onto the `seven_day` field and the API now returns both keys (serde "duplicate field") ‚Äî commit `3ff657e1` drops the alias. Claude now resolves via OAuth (`provider_api`, authoritative); Session/Weekly carry utilization + reset; Sonnet/Daily Routines appear when present.

### RC2 ‚Äî `usage accounts` reports `resets_at: null` for every provider (reporter bug, fix alongside P0)

`usage accounts` JSON serializes `resets_at: null` for all rows even when the TUI renders resets correctly for six providers ‚Äî so the reset data **is** in the in-memory snapshot but is dropped on the CLI report path. Consequences: (1) do **not** judge reset correctness from `usage accounts` ‚Äî use the TUI render; (2) this is itself a bug ‚Äî the architecture lets a render field be silently dropped on the CLI path, so the CLI and TUI disagree.

**Root cause (pinned).** `QuotaBucketView` carries only the formatted `reset_label` (e.g. `Resets in 3h`), not the raw reset epoch; the relative label cannot be reversed, so `account_snapshot_views_from_cache` (`usage.rs`) hardcodes `resets_at: None`. The window builders (`push_claude_window` / `push_codex_window` / `zai_bucket` / `kimi_bucket` / `minimax` / grok) compute the epoch and discard it after formatting.

**Shipped ‚Äî `5bcc4ec1`.** Two parts, both landed: (a) `QuotaBucketView` gained `resets_at: Option<i64>` and a `timed_bucket` helper threads the epoch through every window builder (Claude/Codex/Grok/Z.AI/Kimi/MiniMax), so `account_snapshot_views_from_cache` emits it and the live `usage accounts` agrees with the TUI; (b) the telemetry store persists it both directions. **No schema migration was needed** ‚Äî the earlier deferral assumed one, but the `account_usage_snapshots` table already had a `resets_at INTEGER` column (schema v4); the bucket‚Üîrow mapping simply dropped the value (`account_snapshot_rows` hardcoded `None`, the read never set it). Wiring `resets_at: bucket.resets_at` on write and `row.resets_at` on read closed it with no version bump. Locked by `account_snapshot_rows_carry_reset_epoch`.

## P1 ‚Äî Unified account identity + auth-source header (F11 / F12)

Today the header mixes identity with auth method and is inconsistent per provider (`Claude OAuth`, `ZAI_API_KEY present`, `Kimi auth token`, `MiniMax API token`, real email for Codex/Amp/xAI). Separate the two concepts; **operator-approved layout, identical on every provider and everywhere in the dialog:**

- Row 1: provider name (left) ¬∑ account email (right; omit when the provider exposes none).
- Row 2: `Updated <rel> ¬∑ <state>` (left) ¬∑ `<username> ¬∑ <plan>` (right; username only when distinct from email).
- Row 3 (NEW): `Auth: <kind> ¬∑ <origin>` ‚Äî the credential SOURCE, never the secret. kind = `OAuth` (native client) / `API token` (env) / `API key` (file); origin = the WINNING resolver arm (env ‚Üí `env <VAR_NAME>`; file/OAuth ‚Üí the resolved path, home-prefix collapsed to `~`). There is no keychain reader in the capsule ‚Äî every OAuth token resolves from a file, so the origin is always that file's path.

**Per-provider data (verified against the `usage.rs` resolvers + parallax logs):**

| Provider | email | username | plan | Auth row |
|---|---|---|---|---|
| OpenAI / Codex | yes (`id_token` / RPC `account/read`) | yes | `Pro 20x` | `OAuth ¬∑ ~/.codex/auth.json` (or `/jackin` handoff) |
| Anthropic / Claude | needs new parse (`~/.claude.json` `oauthAccount`) | ‚Äî | `Max` (`subscriptionType`) | `OAuth ¬∑ ~/.claude/.credentials.json` (or `~/.claude.json` / `/jackin` handoff) |
| Amp | yes (`email`) | yes | `Amp Free` | `API key ¬∑ amp secrets.json` |
| xAI / Grok | yes (`email`) | (user id) | `SuperGrok` | `OAuth ¬∑ ~/.grok/auth.json` |
| Z.AI | none | ‚Äî | plan name | `API token ¬∑ env ZAI_API_KEY` |
| Kimi | none | ‚Äî | ‚Äî | `API token ¬∑ env KIMI_CODE_API_KEY` |
| MiniMax | none | ‚Äî | plan title | `API token ¬∑ env MINIMAX_API_KEY` |

`FocusedAccountHeader` (`crates/jackin-protocol/src/control.rs`) has no field for credential origin ‚Äî every snapshot resolver discards which `.or_else` arm won. Add a structured `credential_origin` (a small enum carrying the origin kind plus its detail: an environment variable with its name, or a file with the path), populate it at the resolver arm that actually wins (not a guessed default), split email/username/plan, and render the Auth row. For Claude, parse the account email from `~/.claude.json` `oauthAccount` (or a profile endpoint) ‚Äî verify against CodexBar's Claude fetcher. Providers with no email show plan + auth row only; no fabricated email. Parallax logs confirm Z.AI/Kimi/MiniMax creds arrive as env vars (`ZAI_API_KEY` / `KIMI_CODE_API_KEY` / `MINIMAX_API_KEY`).

**Shipped (partial) ‚Äî `8df471f4`.** `FocusedAccountHeader` gains `username` + `credential_origin` (a pre-formatted display string); `account_label` now holds the email only (empty, never a fabricated identity, for the no-email providers); the render (`usage_header_lines`) shows email (omitted when blank ‚Äî no "account unavailable"), `username ¬∑ plan` on line 2, and the `Auth: <origin>` line. Z.AI/Kimi/MiniMax move their environment-variable string out of `account_label` into `credential_origin` (precise: the exact env var). Claude/Codex/Amp/Grok keep their email and gain the Auth line. Authoritative render verified by the `usage_dialog_renders_auth_source_and_omits_blank_email` snapshot test; no provider-resolution regression in-container (OpenAI 5 rows).

**Completed follow-ups.** (1) Exact winning-arm precision shipped (`8262c728`): Codex (home `auth.json` vs handoff vs env `OPENAI_API_KEY`), Amp (env `AMP_API_KEY` vs `secrets.json` vs handoff), Grok (home vs handoff vs env `XAI_API_KEY`/`GROK_DEPLOYMENT_KEY`) each derive the origin from the arm that actually resolved, not a constant; Codex also drops its `OPENAI_API_KEY` string out of `account_label`. (2) Claude email parse shipped (`d558c31f`): `account_label` reads `oauthAccount.emailAddress` from `~/.claude.json` (`load_claude_account_email`), with per-arm origin: the OAuth file path that actually won (`~/.claude/.credentials.json`, `$CLAUDE_CONFIG_DIR`, `~/.claude.json`, or the `/jackin` handoff ‚Äî home-prefix collapsed to `~`) when an OAuth token resolves, else the env token (`env ANTHROPIC_API_KEY`/`ANTHROPIC_AUTH_TOKEN`). The OAuth arm is named first because the displayed numbers always come from the file token, never the env key. (3) `username` deliberately stays `None` ‚Äî no provider exposes a handle distinct from the email (CodexBar surfaces email too); fabricating one would break the absent-data rule, and the field + render are ready if a source appears. (4) Structured-enum origin and (5) store-persisting the two header fields are optional polish: the tested display string is the user-facing contract, and the header re-resolves live on every refresh.

The Session/Weekly headline slot is now a semantic `StatusSlot` tag on `QuotaBucketView`, set by each provider where it builds the window (Claude/Codex `Session`/`Weekly`, Z.AI `5-hour`/`Tokens`, Kimi `Rate Limit`/`Weekly`, MiniMax general `5h`/`Weekly`, Grok billing cycle ‚Üí Weekly) and read by `status_bar_quota_labels`, so a label rename can no longer silently break the headline ‚Äî locked by `status_bar_reads_session_weekly_slots_from_tags` plus a per-provider tagging assertion in each provider's bucket test. `provider_matches_usage_label` now compares the canonical `UsageSurface` each label resolves to (`surface_from_text`) instead of a synonym OR-chain. The slot is not persisted: the headline is stored as `status_bar_label`, so a store-restored bucket carries no slot and the next live refresh re-tags it.

## P3 ‚Äî Usage-state lifecycle: never show a misleading state (F1‚ÄìF3)

The usage segment must follow agent state, identically for **every** agent:

1. **Before the agent has started** ‚Äî hide the usage segment entirely (today it shows `usage unavailable` during container init, before the agent exists).
2. **Agent started, usage not yet known** ‚Äî show `refreshing` (one consistent word everywhere), not `usage unavailable`.
3. **Resolved** ‚Äî the real headline.
4. **Genuinely unavailable** ‚Äî only then `usage unavailable` (or the specific honest state: `needs login`, `stale`, `unsupported`, `daemon disconnected`).

### P3a ‚Äî lifecycle states + clickable refresh

The segment is **clickable in the unknown/refreshing state**; a click forces a proactive load. These states are driven by the in-memory usage cache plus an agent-started flag, so P3a is implementable without the proxy below. Honest degradation (never a fabricated number):

| State | Rendering |
|---|---|
| Before agent start | usage segment hidden |
| Refreshing / unknown | `refreshing` (clickable ‚Üí force load) |
| Fresh | real quota headline |
| Stale | `stale` + `updated Xm ago`, keep prior data |
| Needs login / secret | the repair action name; no hidden repair |
| Unsupported | `unsupported` + reason |
| Daemon disconnected | `usage unavailable: daemon disconnected` |
| No focused agent | `usage unavailable: no focused agent session` |

### P3b ‚Äî reliable agent-start signal (needs a design step before coding)

Detecting "agent started" reliably is the open architectural question behind steps 1‚Äì2. Intended direction: **route the agent launch through jackin-capsule as a proxy/hook** instead of launching the agent directly, so the capsule owns the exact start moment and triggers auth-verify + refresh then. Before coding: identify how the agent is launched today (the process-launch path in the daemon/runtime) and confirm whether a proxy/hook is the right structural fix or whether an existing lifecycle signal can be used. This is what makes "hide before start" and "`refreshing` on start" precise rather than approximated.

**Design-step finding (PR 579 investigation).** The agent is *already* launched through jackin-capsule: attaching/opening a tab runs `/jackin/runtime/jackin-capsule new <agent>` (`crates/jackin-capsule/src/runtime/attach.rs`), whose `new` handler (`crates/jackin-capsule/src/main.rs`) sends a `SpawnRequest::Agent` to the daemon over the control socket; **the daemon owns the spawn** and creates the session/pane + agent process. So the exact start moment is already an in-daemon event ‚Äî **no separate proxy process is needed**; the structural fix is to hook the daemon's existing `SpawnRequest::Agent` handling: mark that session's agent "started", kick a usage refresh, and let the usage view distinguish *before-start* (segment hidden) ‚Üí *started/unknown* (`refreshing`) ‚Üí *resolved*. This is the existing-lifecycle-signal option the design step was meant to choose between, and it is the lighter, correct one. **Pending operator ratification of this approach before P3a/P3b coding** (per the roadmap's "confirm before coding" gate).

## P5 ‚Äî Shared tab-bar focus model + remaining visual (F14 / F15, Issue 1, Issue 15)

The tab bar reuses the shared `TabStrip` (`crates/jackin-tui/src/components/tab_strip.rs`), the same component as the console settings / workspace-editor surfaces. The model ‚Äî already specified for that surface ‚Äî must be applied identically on **every** tab bar in jackin‚ùØ (console settings, workspace editor, capsule agent-tab bar, capsule usage overlay, status bar, lookbook):

- The **bright phosphor-green** accent is the single focus indicator and always marks where focus is ‚Äî never in two places at once.
- **Tab bar focused:** the active tab shows the bright-green underline; `Left` / `Right` switch tabs; `Down` / `Tab`, or clicking the content, moves focus into the content.
- **Hover** a tab: the tab color changes AND the cursor changes to a pointer; clicking a tab moves focus onto the tab bar.
- **Content focused:** the tab underline drops to a neutral/dim color (not bright green) and the content block carries the green focus indicator instead.

**Capsule does not match today** ‚Äî neither the top agent-tab bar nor the usage overlay strip shows the green-on-focus underline, arrow-nav while focused, the hover cursor, or the focus transition into content. The structural reason: the capsule agent-tab bar is a **hand-rolled painter** (`StatusBarWidget::paint_tab` in `crates/jackin-capsule/src/tui/components/chrome.rs`), separate from the shared `TabStrip` ‚Äî that duplication is exactly why it diverges. Root-cause fix: fold the agent-tab bar (and the usage strip) onto the shared `TabStrip` so render + hover + per-tab hit-test + cursor + focus underline live in one place (Issue 1) and the focus model is unified across every surface (Issue 15) ‚Äî not re-patch each painter.

**Still owed (capture + verify in a live smoke):** hover color + pointer cursor on every tab surface; status-bar consistency on every screen per the display matrix (debug + non-debug); no vertical jump on tab switch, centered strip, correct symmetric padding (content padded both sides, separators full-bleed); left/right tab switch instant even mid-refresh with no refetch within the TTL.

**Implementation plan + scope (PR 579 investigation).** The shared `TabStrip` (`crates/jackin-tui/src/components/tab_strip.rs`) carries the whole model: `.focused(bool)` drives `tab_underline_line` (bright phosphor-green underline only when focused, neutral otherwise), `.hovered(Some(idx))` drives `tab_cell_style` (hover colour), `hit_index_at` is the shared hit-test, `register_hover_targets` wires OSC-22 pointer cues in one place.

**The usage-overlay strip is already folded** ‚Äî `dialog_widgets.rs` builds `TabStrip::new(&tab_refs).focused(tab_bar_focused)` for render and `hit_index_at` for clicks (the overlay carries a real `tab_bar_focused` state). So P5's remaining gap is **only the agent-tab bar** (`StatusBarWidget` in `crates/jackin-capsule/src/tui/components/chrome.rs`), and it is *not* a clean drop-in:

1. **It is a composite, not a plain strip** ‚Äî a brand pill, per-tab status glyphs (`‚óã` done / `‚óè` blocked, the blocked one bright-red over-painted), and a right-side menu button surround the tab cells. `TabStrip`/`TabCell` model plain label tabs with no glyph, so folding needs either glyph support added to `TabStrip` or the glyphs kept as a post-paint over the strip ‚Äî and the hit-test only matches if `TabStrip` lays the cells out at the same columns the glyphs assume.
2. **It has no tab-bar-focus mode** ‚Äî the agent terminal always holds input focus, so the row-1 underline is *always* phosphor-green. Making it the focus indicator (green only when the bar is focused, neutral when the agent content is) needs a new capsule interaction mode (enter tab-bar focus, `Left`/`Right` to switch, `Down`/click to return to the agent) ‚Äî an interaction-design decision, not just a render swap.
3. **Acceptance is visual/interactive** ‚Äî underline colour on focus transition, hover colour, pointer cursor, centring, no vertical jump, symmetric padding ‚Äî none assertable headlessly.

So the agent-tab-bar fold is owed (a) a `TabStrip` glyph/‚Äãcomposite reconciliation, (b) the new focus-mode interaction (operator design), and (c) a live debug smoke. It is deliberately not landed blind.

## Reference ‚Äî CodexBar extraction recipes (for P0 / P1)

The background refresher resolves a credential, calls the provider usage endpoint, and maps the response to quota buckets; render paths never call these. Only the recipes an open item still needs are kept here.

- **Anthropic / Claude.** Credential `~/.claude/.credentials.json` (`claudeAiOauth.accessToken`), forwarded from the host macOS Keychain service `Claude Code-credentials`. Call `GET https://api.anthropic.com/api/oauth/usage` with `Authorization: Bearer`, `anthropic-beta: oauth-2025-04-20`, `User-Agent: claude-code/<version>`. Windows: `five_hour`‚ÜíSession, `seven_day`‚ÜíWeekly, `seven_day_sonnet`‚ÜíSonnet, `seven_day_opus`‚ÜíOpus, `seven_day_routines` (and its `claude_routines` / `routines` aliases)‚ÜíDaily Routines, `extra_usage`‚ÜíExtra usage; each window carries `utilization` + `resets_at`. Plan prefers `subscriptionType`, falls back to `rateLimitTier`. Account email is NOT in this response ‚Äî read `~/.claude.json` `oauthAccount` (F12).
- **OpenAI / Codex.** Credential `$CODEX_HOME/auth.json` (default `~/.codex/auth.json`). Call `GET https://chatgpt.com/backend-api/wham/usage` with `Authorization: Bearer` + `ChatGPT-Account-Id`; or launch `codex -s read-only -a untrusted app-server` and call JSON-RPC `account/read` + `account/rateLimits/read`. Map `rate_limit.primary_window`‚ÜíSession, `secondary_window`‚ÜíWeekly; `additional_rate_limits[]` `spark`‚ÜíCodex Spark 5-hour/Weekly; `credits`‚ÜíCredits; `plan_type`‚Üíplan (`Pro 20x`). Identity from the app-server RPC or `id_token` JWT, not a `tokens.email` field.
- **Claude credits (F8).** `claude.ai/api/organizations/{org}/usage` ‚Äî spent, monthly cap, balance (currency-localized).

### Credential delivery into the container

The capsule runs as PID 1 inside the Linux container and cannot read the host Keychain; the runtime reads the host credential (file or Keychain) and forwards it into the container at a fixed path per provider (`crates/jackin-instance/src/auth.rs`). File-based providers (Claude, Codex, Grok, Amp) depend on that forwarding; env-key providers (Z.AI, Kimi, MiniMax) are carried in as environment variables regardless of mount mode. P0 depends on the Claude OAuth credential being forwarded reliably so the OAuth path can win over `cli`.
