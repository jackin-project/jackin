# Claude Auth — `auth_forward = "token"` Mode Using `CLAUDE_CODE_OAUTH_TOKEN`

**Status:** Proposed
**Date:** 2026-04-23
**Scope:** `jackin` crate only
**PR:** 3 of 3 in the Claude auth strategy series
**Depends on:** PR 2 (workspace env resolver) landing first.

## Problem

`sync` (PR 1's new default) eliminates startup drift for copied OAuth state, but it does not solve the underlying fragility of forwarding Claude Code subscription OAuth into long-lived or concurrent containers. Refresh-token rotation across the host and multiple jackin sessions can still invalidate any given container's credentials mid-session, producing the same `API Error: 401 Invalid authentication credentials` that motivated the auth strategy roadmap.

Claude Code's official authentication precedence order (documented at <https://code.claude.com/docs/en/iam>) places `CLAUDE_CODE_OAUTH_TOKEN` **above** subscription `/login` credentials. If the env var is present, Claude Code inside the container uses it and ignores the bind-mounted `.claude/.credentials.json`. This makes the long-lived OAuth token (generated by `claude setup-token`, valid ~1 year) a much better fit for concurrent jackin sessions than copied subscription state.

This PR adds `auth_forward = "token"` as a first-class mode. It does not create or store tokens; it only asserts that `CLAUDE_CODE_OAUTH_TOKEN` is resolvable via PR 2's operator-env system and forwards it.

## Goals

1. Add `AuthForwardMode::Token` as a fourth variant (after PR 1 shrinks the enum, the set is `Ignore`, `Sync`, `Token`).
2. In token mode, provision the container with an empty `.claude.json` and no `.credentials.json` — the same filesystem shape as `Ignore` mode.
3. Require that `CLAUDE_CODE_OAUTH_TOKEN` appears in the resolved operator env (from PR 2). Token mode without the env var is a hard error at launch.
4. Rely on PR 2 to actually read, resolve, and inject the env var. PR 3 does not duplicate that path.
5. Print a clear one-line diagnostic at launch so operators can verify which auth mode is active and where the token came from.
6. Document the recommended setup: `claude setup-token` → add `CLAUDE_CODE_OAUTH_TOKEN = "op://..."` to global or workspace env → `auth_forward = "token"`.

## Non-Goals

- Generating, storing, or rotating OAuth tokens. Anthropic's `claude setup-token` does the generation and intentionally does not persist. jackin follows the upstream posture.
- Adding a `jackin config auth setup-token` helper. The operator runs `claude setup-token` directly; our docs link to it. Wrapping upstream tooling adds maintenance for no operator gain.
- Keychain / libsecret integration for the token. Operators who want native OS storage use 1Password (or similar) via PR 2's `op://` source.
- Per-token validation. jackin passes whatever string the env resolver produces; Claude Code inside the container validates.
- Token mode as the new default. `sync` stays default (PR 1). Flipping default to `token` is a future decision once operator adoption data justifies it.
- Solving `--bare` Claude Code mode. Upstream docs explicitly note that bare mode does not read `CLAUDE_CODE_OAUTH_TOKEN`. jackin does not invoke bare mode; operators who do must configure `ANTHROPIC_API_KEY` or `apiKeyHelper` in-container themselves.

## Design

### Config model

`AuthForwardMode` gains a `Token` variant (PR 1 has already removed `Copy`):

```rust
#[derive(Debug, Default, Clone, Copy, PartialEq, Eq, Serialize, Deserialize)]
#[serde(rename_all = "lowercase")]
pub enum AuthForwardMode {
    Ignore,
    #[default]
    Sync,
    Token,
}
```

`Display` and `FromStr` gain the `"token"` case. No new config fields beyond the existing `auth_forward` key — token mode consumes the env resolver from PR 2, so there is nothing new to store here.

Per-agent override (`[roles.<name>.claude.auth_forward = "token"]`) works for free via the existing resolution path in `src/config/agents.rs::resolve_auth_forward_mode`.

### Provisioning behavior

`AgentState::provision_claude_auth` in `src/instance/auth.rs` gains a `Token` match arm. Behavior mirrors `Ignore`:

1. Ensure `.claude.json` exists with `{}` (same bootstrap file requirement — Docker bind-mount requires a regular file).
2. Ensure no `.credentials.json` is present. If a previous mode had written one, remove it.
3. Repair permissions (same safeguards already applied in the existing arms).
4. Return `AuthProvisionOutcome::TokenMode` (a new variant) so the caller can distinguish.

The mode transitions existing tests cover (`copy → ignore`, `sync → ignore`) must gain parallel coverage for `sync → token` and `token → sync` / `token → ignore` to prove the old credentials are revoked when switching away from any forwarded state and back.

### Launch-time verification

After PR 2's resolver produces the operator env map, and before the container starts, `src/runtime/launch.rs` checks:

```text
if auth_mode == Token && resolved_env does not contain CLAUDE_CODE_OAUTH_TOKEN:
    bail with a hard error
```

The error message lists the remediation paths:

```
error: auth_forward is "token" for <agent> but CLAUDE_CODE_OAUTH_TOKEN is not in the resolved
env for workspace <workspace>. Add it to your jackin config:

  [env]
  CLAUDE_CODE_OAUTH_TOKEN = "op://Personal/Claude Code Token/credential"

  # Or if you prefer host pass-through:
  CLAUDE_CODE_OAUTH_TOKEN = "$CLAUDE_CODE_OAUTH_TOKEN"

Generate a token via `claude setup-token`. See
https://code.claude.com/docs/en/iam for details.
```

If PR 2's resolver itself failed (missing `op`, bad reference, unset host env), that error surfaces first and launch never reaches this check.

### Launch-time diagnostic

One line, printed before the container starts:

```
claude auth: OAuth token (CLAUDE_CODE_OAUTH_TOKEN ← op://Personal/Claude Code Token/credential)
```

or:

```
claude auth: OAuth token (CLAUDE_CODE_OAUTH_TOKEN ← $CLAUDE_CODE_OAUTH_TOKEN)
```

or (literal — uncommon, but possible if an operator puts the token directly in config):

```
claude auth: OAuth token (CLAUDE_CODE_OAUTH_TOKEN ← literal)
```

This surfaces the *source reference*, never the *value*. PR 2's resolver tracks the source kind per key for exactly this purpose.

For symmetry, `sync` and `ignore` modes get matching one-liners:

```
claude auth: forwarded host session (sync)
claude auth: none (manual /login required) — ignore
```

These come from the roadmap's "Suggested Operator UX" section and are mechanical additions alongside token mode.

### Env resolver dependency

PR 3 does not call `op` or read host env vars directly. It relies entirely on PR 2's `resolve_operator_env`. The call in `src/runtime/launch.rs` looks roughly like:

```rust
let operator_env = resolve_operator_env(&config, &workspace, &class)?;
// ... existing manifest resolution ...
if auth_mode == AuthForwardMode::Token {
    if !operator_env.vars.contains_key("CLAUDE_CODE_OAUTH_TOKEN") {
        bail_with_actionable_error();
    }
}
// ... existing docker run -e loop uses the combined map ...
```

If PR 2 has not yet landed when PR 3 is reviewed, do not merge PR 3 first.

### CLI

`jackin config auth set token` works via the same dispatch as `sync` and `ignore`. Help text for `jackin config auth set` is updated to list `sync | ignore | token` with a brief one-line description of each.

`jackin config auth show` prints `token` when active.

No new subcommands. Token generation stays with upstream `claude setup-token`.

## Failure Modes

| Condition                                                                         | Behavior                                                                                       |
| --------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| `auth_forward = "token"` but `CLAUDE_CODE_OAUTH_TOKEN` not in resolved operator env | Hard error at launch with actionable remediation (see above)                                   |
| `op://` resolution fails (`op` missing, `op read` fails, timeout)                 | Surfaced by PR 2's resolver before this check fires                                            |
| Token is present but invalid / expired / revoked                                  | Surfaces as a 401 **inside** the container from Claude Code itself. jackin cannot pre-validate |
| Operator sets `auth_forward = "token"` and also declares env at a non-global layer that does not include the token | Hard error — the merged env for that (workspace, agent) must include the token |
| Mode switch `sync → token`                                                        | Provisioning removes `.credentials.json` (revoke of forwarded creds), writes `{}` to `.claude.json` |
| Mode switch `token → ignore`                                                      | Same as `sync → ignore` today — `.claude.json` reset to `{}`, no credentials file              |
| Mode switch `token → sync`                                                        | `sync` re-seeds from host; existing symlink / permission defenses apply                        |

Token validity is Claude Code's responsibility. A 401 inside the container with token mode active tells the operator to regenerate via `claude setup-token` and update their 1Password item (or their shell env). jackin's diagnostic line makes this triage straightforward.

## Security Notes

- jackin never stores, logs, or displays the token value. Only the *reference* (env var name or `op://` URI) is shown in diagnostics.
- The token is injected via `docker run -e CLAUDE_CODE_OAUTH_TOKEN=<value>`. Docker treats this as environment for the container process; the value is not persisted to any host file by jackin.
- When token mode is active, jackin writes `.claude.json = "{}"` and deletes any pre-existing `.credentials.json`. This actively revokes previously forwarded subscription OAuth state for the same container, preventing a stale copy from silently being used if the token later becomes the explicit mode.
- The token survives in container process env for the lifetime of the container. That matches how `CLAUDE_CODE_OAUTH_TOKEN` is expected to be used per the upstream docs; it is not jackin's responsibility to rotate or refresh it mid-session.
- Symlink traversal protections in `src/instance/auth.rs::reject_symlink` continue to apply to the `.claude.json` write path unchanged.

## Test Plan

### Unit tests

- `AuthForwardMode::from_str("token")` returns `Token`; round-trip via `Display`.
- `provision_claude_auth(..., AuthForwardMode::Token, ...)` writes `{}` to `.claude.json`, removes `.credentials.json` if present, returns `AuthProvisionOutcome::TokenMode`.
- Mode-transition matrix: `sync → token` removes credentials file; `token → sync` re-copies; `token → ignore` ends at empty state.
- Launch-time: with `Token` mode and `CLAUDE_CODE_OAUTH_TOKEN` absent from the resolved env, launch bails with the expected error string.
- Launch-time: with `Token` mode and the env var present (any source), launch proceeds.
- Diagnostic line formatting: each of the three source kinds produces the expected one-liner.

### Integration tests

- End-to-end launch with a mocked-op-returning-a-fixed-string env var: container receives `CLAUDE_CODE_OAUTH_TOKEN` in its env and `.credentials.json` does not exist in agent state.
- Switching from `sync` to `token` on an existing agent state directory removes the previously copied `.credentials.json`.
- Per-agent override: `[roles.<name>.claude.auth_forward = "token"]` takes effect for that agent and not others.

### Pre-commit

`cargo fmt -- --check && cargo clippy && cargo nextest run` is clean.
