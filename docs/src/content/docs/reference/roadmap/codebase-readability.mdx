**Status**: Open — active program (Phase 1 → Phase 3)

## Goal

Make AI-generated code verifiable. 59% of source files have no `//!` module contracts; no behavioral specs exist; no systematic way to answer "did the AI implement this correctly?" PR review is hard because files are large and concerns are mixed.

The work is organized in three phases. Each item below links to its dedicated page with steps, file references, and confirmation requirements.

## Execution order (recommended)

1. **PR #177** — merge what's in flight first
2. **Phase 2 file splits** — biggest readability gain per effort; do before workspace split
3. **Per-directory README + AGENTS.md** — orient AI agents in the new structure
4. **Greenfield workspace split** — enforce the existing clean `console/`↔`runtime/` boundary with the compiler
5. **Phase 1 documentation items** — `//!` docs, behavioral specs, ADRs become authoritative against the clean structure

## Key structural insight

`console/` has **zero imports from `runtime/`** — this pre-existing clean boundary is the most important structural asset. The greenfield workspace split can enforce this boundary with the Rust compiler without moving any logic, only directories.
