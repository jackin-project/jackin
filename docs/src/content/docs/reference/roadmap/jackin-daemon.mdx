**Status**: Implemented — phase 1 base daemon landed

## Problem

jackin' has been a short-lived CLI / TUI from day one: every command (`jackin console`, `jackin load`, `jackin workspace …`, `jackin config …`) runs to completion and exits. Reconcilers that need to keep state warm, react to host events, or hold a per-host singleton currently piggy-back on the next command boundary, which is the wrong shape for a growing class of features.

A handful of these features have already accumulated:

- **Keep-awake reconciler.** When a workspace opts into `keep_awake = true`, jackin' spawns `caffeinate -imsu` while any role from a keep-awake workspace is running. Today the reconciler runs at every CLI command boundary — leaning on the operator to type a `jackin` command often enough to keep the lock alive, and tearing down + respawning the lock on each invocation. Works in practice, but is structurally a stand-in for a long-running watcher.
- **Live bidirectional auth sync** ([dedicated roadmap item](/reference/roadmap/live-auth-sync/)). Needs to watch the host's token stores (Keychain notifications on macOS, inotify on Linux, periodic `gh auth token` polls), reconcile a flock-protected shared store, and push updates into running containers within seconds — without waiting for the operator to type a command. There is no way to do this from a CLI that exits after each command.
- **Container event watcher (future).** Surface launch failures, OOM kills, agent process exits to the operator without requiring an open `jackin console` session. Today these only surface when the operator next runs `jackin console` or `gh pr checks`-style polling on the CLI.
- **Cross-session notifications (future).** Ping the operator when an agent in a backgrounded session needs attention (sensitive-mount confirmation, auth re-prompt, network-policy violation).

The pattern: each new feature that wants to **react to events instead of waiting for the next command** independently invents its own "what if jackin had a daemon" workaround. That doesn't scale.

This roadmap item introduced **the jackin daemon** as a first-class concept the rest of the project can compose on. The initial implementation ships the daemon bones and the first host adapters:

- `jackin daemon serve` / `start` / `stop` / `restart` / `status` / `logs` / `install` / `uninstall`.
- A per-user Unix domain socket at `~/.jackin/run/jackin-daemon.sock`.
- A singleton lock at `~/.jackin/run/jackin-daemon.lock`.
- Auto-start from `jackin load` and `jackin console`, with version skew detection. If the CLI version differs from the running daemon version, the CLI stops the old daemon and starts the current binary.
- Daemon-owned keep-awake reconciliation, using the existing `caffeinate` state machine.
- Periodic cache warmup for the construct image, `docker:dind`, and any cached role manifest `published_image`.
- macOS Notification Center notifications through the daemon via `jackin daemon notify` and the in-container `jackin-notify` helper. High urgency requests the native `Glass` sound.
- A container bridge: running roles receive `JACKIN_DAEMON_SOCKET=/jackin/daemon.sock` and the host daemon socket bind-mounted at that path when the daemon is available.

## What the daemon owns

The daemon is the long-running, per-operator-user, per-host process that holds anything jackin needs to:

1. **Watch host state.** Filesystem inotify, macOS Keychain notifications, periodic polls — for any host source jackin reacts to.
2. **Watch container state.** Docker events stream (`docker events`) for launch failures, exits, OOM kills.
3. **Push reactive updates** into running containers via the existing bind-mount channels (no new container-side privilege).
4. **Hold cross-process locks** that have to outlive any one CLI invocation (the live-auth-sync shared store flock, future "only one `jackin load` at a time per role class" mutex, etc.).
5. **Reconcile per-host singleton state.** Today's keep-awake `caffeinate` lock is the obvious example — only one caffeinate process per host, regardless of how many `jackin` invocations want it alive.
6. **Run periodic tasks** the operator opted into — credential refresh, session timeout warnings, garbage-collect on dead containers.

Anything that fits this shape moves into the daemon over time. Features that don't (one-shot container launch, config edits, git operations) stay on the CLI / TUI.

## What the daemon does NOT do

To keep the surface manageable and the security posture reviewable:

- It does **not** replace the CLI / TUI. `jackin console`, `jackin load`, `jackin workspace`, `jackin config` continue to be short-lived processes the operator types. The daemon is a back-end the CLI / TUI can talk to over a control socket.
- It does **not** introduce a network surface. The daemon listens on a Unix domain socket scoped to the operator's user account, never on TCP.
- It does **not** run as root. Same UID as the operator who launched it.
- It does **not** survive operator logout. Lifecycle is tied to the operator's session (launchd LaunchAgent on macOS, systemd user unit on Linux), not a system-level service.
- It does **not** hold credentials in persistent storage of its own. Anything the daemon needs at rest lives in the per-feature shared store (e.g. `~/.jackin/auth-shared/<axis>/` for live auth sync), not in a daemon-owned database.

## Design decisions

### Lifecycle

- **Start trigger.** `jackin load` and `jackin console` auto-start the daemon. `jackin daemon install` additionally registers it with the host's per-user service manager so it starts on login.
- **Restart policy.** launchd / systemd own crash restart when installed. CLI fallback restarts on the next daemon-aware command.
- **Stop semantics.** `jackin daemon stop` sends a protocol shutdown over the socket. If the socket is stale, it falls back to the recorded pid file and cleans stale runtime files.
- **Upgrade across jackin versions.** Every daemon start/status path checks the daemon's reported `JACKIN_VERSION`. A mismatch causes the CLI to stop the old daemon and start the current executable, so the first command after `brew upgrade jackin` moves the daemon forward.

### Install method

- **macOS.** `jackin daemon install` writes `~/Library/LaunchAgents/com.jackin.daemon.plist` and loads it with `launchctl load -w`.
- **Linux.** `jackin daemon install` writes `~/.config/systemd/user/jackin-daemon.service` and enables it with `systemctl --user enable --now`.
- **Fallback for systems without launchd / systemd.** `jackin daemon start` backgrounds `jackin daemon serve`, writes a pid file, and logs to `~/.jackin/run/jackin-daemon.log`.
- **Container / CI environments.** The daemon is per-user and host-scoped. CI can avoid `load`/`console` or explicitly stop it; future CI detection can make this silent.

### CLI surface

Implemented subcommands under `jackin daemon`:

- `jackin daemon install` — write the launchd / systemd unit and start the daemon.
- `jackin daemon uninstall` — stop and remove the unit.
- `jackin daemon start` / `stop` / `restart` — manual lifecycle controls.
- `jackin daemon status` — running state, version, pid, uptime, socket, log, keep-awake adapter, cache warmup adapter.
- `jackin daemon logs [-n N]` — print recent daemon log lines.
- `jackin daemon serve` — run the daemon in the foreground.
- `jackin daemon warm` — ask the daemon to warm Docker cache inputs immediately.
- `jackin daemon notify --title ... --body ... [--urgency low|normal|high]` — send a host macOS Notification Center notification.

The console / TUI gets a one-line daemon-status indicator in its footer, mirroring the launch-summary pattern the auth axes use today.

### Control protocol

- **Transport.** Unix domain socket. Path per-operator: `~/.jackin/run/jackin-daemon.sock`.
- **Wire format.** JSON Lines — one JSON object per line, request/response.
- **Authentication.** Filesystem permissions on the runtime directory (`0700`) and same-UID Unix socket access. No TCP listener and no root daemon.
- **Versioning.** Each request carries a protocol version. The daemon rejects protocol mismatches and the CLI restarts daemon binary version mismatches. The `daemon/hello` request is the compatibility handshake Desktop and future daemon clients must call first: it returns the jackin binary version, supported protocol range, and the daemon capabilities currently available on that socket.
- **Container bridge.** Role containers receive the socket as `/jackin/daemon.sock` and can use `jackin-notify --title ... --body ...` from hooks or agent-created scripts.
- **Notifications.** macOS is the v1 target. The daemon shells out to `/usr/bin/osascript` with `display notification`, so notifications are native Notification Center toasts but do not require a bundled `.app`, Apple Developer account, certificate, or notarization for local developer-machine use. Linux and Windows adapters are deferred.

Initial protocol capabilities:

| Method | Purpose |
|---|---|
| `daemon/hello` | Return jackin version, supported protocol range, and capability list. |
| `daemon/status` | Return daemon pid, uptime, socket/log paths, keep-awake, cache warmup, and notification adapter status. |
| `daemon/shutdown` | Stop the daemon cleanly. |
| `daemon/warm_cache` | Ask the daemon to warm construct, DinD, and cached role images. |
| `workspace/list` | Return saved workspace summaries for launch-only Desktop surfaces: name, workdir, mount counts, default/last role, default agent, and keep-awake state. |
| `session/list` | Return running role containers with Docker-label metadata for workspace, role, agent, branch, primary repo, display name, and container status. |
| `github/my_open_prs` | Return open pull requests authored by the operator, sorted newest first. |
| `github/project_inbox` | Return the operator's open pull requests filtered by owner and/or repository. |
| `github/repository_prs` | Return open pull requests for one repository, sorted newest first. |
| `workspace/launch` | Open Ghostty and run `jackin load` against a saved workspace, with optional role and agent override. |
| `desktop/open` | Open a browser URL through macOS `open`, or open Ghostty and run `jackin hardline` for a running session. |
| `account/status` | Return cached Claude, Codex, and Amp credential-source presence without exposing token values; `refresh` bypasses the daemon cache. |
| `event/subscribe` | Return the first Desktop event contract. Streaming is explicitly disabled for now; the response names the poll methods and click-route target fields Desktop should use for attention and ready-for-review routing. |
| `notification/send` | Dispatch a host macOS Notification Center notification. |

Most Desktop-specific methods are deliberately absent until their slices land. The next phase turns the event subscription placeholder into a real stream.

### Security posture

- **Credential lifetime in memory.** Live auth sync requires the daemon to hold tokens in memory while watching for changes. Document the threat model: any process running as the operator can already read those tokens from the host's stores — the daemon does not widen the attack surface, but it does centralise it.
- **Secret-redacted logs.** Daemon log output must redact tokens the same way `tracing::debug!` already does for `GithubAuthContext` and `GithubProvisionOutcome` (manual `Debug` impls; see <RepoFile path="src/instance/mod.rs" />).
- **Crash dumps and panic backtraces.** Tokens in memory can leak through coredumps. Either disable coredumps for the daemon process (`prctl(PR_SET_DUMPABLE, 0)` on Linux, equivalent on macOS) or redact tokens before any panic-backtrace path can format them.
- **Fork-safety.** Daemon does not exec subprocesses with token env vars set; `gh auth token` shellouts use a clean env.

### Migration of today's reconcilers

- **Keep-awake `caffeinate`.** The daemon runs the existing reconciler every five seconds. CLI command-boundary reconciliation remains as a compatibility fallback for now; once daemon install confidence is high, those call sites can be removed.
- **Live auth sync.** First feature designed to require the daemon — see the [live-auth-sync roadmap item](/reference/roadmap/live-auth-sync/) for the per-axis adapter plan. The daemon hosts the shared store flocks and the host-side watchers.
- **Desktop Agent Hub.** The native macOS app depends on daemon state rather
  than scraping Jackin config or Docker directly. The daemon is the primary
  data plane for Desktop status bar and window state; CLI calls are fallback or
  one-shot paths only when daemon communication is unavailable or would hurt the
  operator experience. The daemon should expose versioned capabilities for
  workspace/session discovery, GitHub PR lookup, Ghostty/browser open actions,
  account status, and event subscription as those slices land.
- **Future reactive watchers.** Container exit notifications, sensitive-mount confirmations across sessions, etc. — each adds an adapter under the daemon's plug-in surface, not its own ad-hoc workaround.

## Why this is one item, not many

Each of the dependent features (live auth sync, keep-awake migration, container event watcher) could in theory ship its own narrow "we need a daemon for this" PR. That would produce three different daemon shapes, three different lifecycle stories, three different security postures — operationally awful.

The daemon's bones (lifecycle, install, control socket, security model, logs) need to be designed once. Each feature plugs into that base. This roadmap item exists to make that design explicit and to be the place where "do we have a daemon yet?" is answered for any future proposal.

## Proposed implementation phasing

Rough order of operations — not committed; to be confirmed during the design pass:

1. **Phase 0 — design.** Complete.
2. **Phase 1 — daemon base.** Complete: lifecycle, install, singleton, socket protocol, logs, version skew restart.
3. **Phase 2 — first adapters.** Started: keep-awake reconciliation, cache warmup, and host notification dispatch are daemon-owned.
4. **Phase 3 — Desktop-ready daemon API.** Started: `daemon/hello` now provides a versioned capability handshake; read-only workspace/session discovery is available through `workspace/list` and `session/list`; initial GitHub PR lookup is available through `github/my_open_prs`, `github/project_inbox`, and `github/repository_prs`; Ghostty/browser actions are available through `workspace/launch` and `desktop/open`; Claude/Codex/Amp credential-source presence is available through `account/status`; the first `event/subscribe` response publishes the poll and click-routing contract. Next, replace the placeholder with a real event stream.
5. **Phase 4 — live auth sync.** Not started. Add per-axis host watchers and shared-store flocks on top of this daemon.
6. **Phase 5+ — container event watcher and beyond.** Add Docker event subscriptions, attention prompt MCP integration, click-to-focus, and richer host actions.
