## Goal

jackin' renders three connected terminal surfaces: the host operator console (`jackin console`), the launch progress cockpit, and the in-container multiplexer (`jackin-capsule`). The first two now use Ratatui, and the console-triggered launch path now keeps one host alternate screen across console ŌåÆ launch ŌåÆ capsule attach. That shipped backbone is the right product direction, but the architecture behind it is still not the Ratatui application pattern this codebase should standardize on: the host console remains a procedural draw-everything-every-frame loop over god-structs, the launch cockpit owns a parallel mini-app under <RepoFile path="src/runtime/progress.rs" />, and the capsule still uses **no Ratatui at all** ŌĆö it hand-rolls ANSI escape writes and a row diff that re-implements Ratatui's own `Buffer::diff`.

This item defines the **architectural target** for all three surfaces: a clear Model / Update / View separation, an explicit `Action`/`Message` type instead of inline state mutation in event handlers, a pure view layer with no I/O, state decomposed into domain vs. UI vs. ephemeral concerns, `StatefulWidget`-based reusable components, and background I/O that feeds the update loop through a channel rather than blocking the draw. It also commits `jackin-capsule` to **rendering through Ratatui** instead of its current hand-rolled ANSI, so the in-container multiplexer shares the same component library as the host. Throughout, the organizing goal is **reusability**: a single, recognizable component vocabulary that every terminal surface shares, instead of the parallel copy-paste implementations the codebase accumulates today.

This is a north-star design item, not a big-bang rewrite. The work lands incrementally, one surface and one layer at a time, behind the snapshot-test regression net, with **no operator-visible behavior change** at any step.

## Scope ŌĆö one continuous Ratatui experience

This item covers **every terminal surface jackin' renders**, unified under Ratatui and one shared component library:

1. **The operator console (`jackin console`)** ŌĆö the full flow from console start through workspace / role / agent selection. Already Ratatui; this item restructures it (see the host target below).
2. **The loading flow (console ŌåÆ launch ŌåÆ container).** The [Launch Progress TUI](/reference/roadmap/launch-progress-tui/) work shipped the product backbone for this: a Ratatui cockpit, structured stages, diagnostics run IDs, a console-owned host alternate screen, and a capsule handoff flag that prevents nested alt-screen toggles. The remaining target is to make that shipped backbone architectural rather than ad hoc: one app-level TUI session model, one update/effect boundary, one shared component library, no direct imports from `src/console/widgets` into runtime launch code, and no split where launch progress is a separate mini-app beside the console.
3. **Ratatui inside the container.** The `jackin-capsule` multiplexer daemon renders the in-container experience. It moves from hand-rolled ANSI to **rendering through Ratatui** (the attach client stays a thin byte relay ŌĆö see the capsule target below).
4. **The capsule, fully Ratatui.** Every capsule surface ŌĆö pane chrome, status bar, tab strip, dialogs, pickers, confirms ŌĆö becomes a Ratatui widget, and those widgets are the *same* shared components the host console uses.

The connective tissue across all four is the shared design-component crate (`jackin-tui`): one `ConfirmDialog`, one `FilterList`, one `TabStrip`, one `Panel`, one `HintBar`, reused everywhere rather than reimplemented per surface ŌĆö catalogued and rendered in a browsable **lookbook** (the design-system section below).

## Design principles

**Reusability is the first principle of this item, not one of several ŌĆö every other goal here is downstream of it.** The single biggest source of TUI entropy today is copy-paste: the same confirm dialog, picker, tab strip, scrollbar, and palette wrapping exist as parallel implementations across the host console and the capsule, kept consistent only by the procedural rules in [TUI Design Decisions](/reference/tui-design-decisions/). Ending that failure mode is the whole point of this item. The non-negotiable rule: **no code is duplicated and no component is copied. Anything ŌĆö widget, block, dialog, layout, or UI logic ŌĆö that can appear more than once is implemented exactly once and instantiated everywhere it is needed, with one source of truth, so a change in that one place changes every instance.** When new functionality would repeat existing code, the work is to refactor the existing thing into a reusable component *first* and then use it ŌĆö never to paste a second copy and "keep them in sync." We do not tolerate multiple places where the same thing works slightly differently. Every TUI improvement under this item begins with the question "what is the reusable shape?", and that includes the whole **transition experience** (console ŌåÆ loading ŌåÆ capsule ŌåÆ exit), which must itself be composed from shared components, not per-surface copies. The structural fix is to make reuse the path of least resistance ŌĆö a real design-component crate, not a rulebook.

- **One pattern, one component ŌĆö zero duplication.** If a visual pattern, widget, block, or piece of UI logic can appear twice, it is a single shared implementation with one source of truth, instantiated per use site ŌĆö never a second copy kept in sync by hand. Copying code to produce a variant is forbidden; the correct move is to parameterize or extend the one implementation so all instances advance together. This operationalizes the existing "Component Reuse ŌĆö Strict Rule" in [TUI Design Decisions](/reference/tui-design-decisions/) structurally rather than by review discipline: with the shared component crate in place, the duplication becomes hard to introduce, not merely discouraged.
- **One design system, every terminal surface.** The host console, launch cockpit, and capsule render the same brand language (PHOSPHOR palette, tight chrome, footer-only hints, identical confirm / picker / modal shapes). They must consume the *same* components from the *same* crate, so a confirm dialog is literally the same code everywhere ŌĆö not two copies a rule says must match.
- **Recognizable, not bespoke.** A contributor (or agent) reading the code should recognize the component vocabulary immediately: `Panel`, `ScrollableBlock`, `ConfirmDialog`, `FilterList`, `TabStrip`, `HintBar`, `Modal`. New screens compose existing components; a genuinely new component is added only when none fits, with operator sign-off per the strict-reuse rule.
- **Idiomatic Ratatui.** Components are `StatefulWidget` / `WidgetRef` implementations with their own `State`, the shape the Ratatui ecosystem recognizes and the [component template](https://ratatui.rs/templates/component/project-structure/) is built around ŌĆö so the codebase reads like a Ratatui app, not a one-off renderer.
- **One source of truth for tokens.** Color values, spacing, tab gap, and hint formatting are defined once and converted per surface (raw-ANSI `Rgb` today, `ratatui::Color` for widgets), never re-declared inline.
- **Async-first rendering ŌĆö never block the loop.** Every animated or progress-bearing surface keeps drawing on its own cadence *while work is in flight*. Rendering and long-running work (Docker calls, image pulls, network waits, git, daemon round-trips, disk inspection) run on independent async tasks: an interval-/`Message`-driven loop repaints shared state, and the work mutates that state from a `tokio::select!` arm, a `tokio::spawn`d task, or `spawn_blocking` ŌĆö never inline with the draw. A surface that only repaints *between* sequential blocking steps freezes the instant a step is slow: the digital rain stalls, the active-stage pulse stops, the spinner hangs, and the operator cannot tell whether jackin' is working or wedged. Steady motion regardless of how long the underlying step takes is what makes the TUI feel native, snappy, and alive ŌĆö and that "snappy and alive" feel is an explicit operator goal for this item, not a nice-to-have side effect. The acceptance test for any new long-running step on an animated surface: *does the animation keep moving smoothly while the step runs?* If it stutters, the step is on the render loop and must move off it. The concrete sites to migrate are enumerated in the [blocking-work inventory](#blocking-work-inventory--async-migration-targets) below.

### Anti-patterns vs. Ratatui best practice

1. **God-structs that mix four concerns.** `ManagerState` (Ōēł30 fields) holds domain data (`instances`, `workspaces`, `current_dir`), layout geometry (`cached_term_size`, `list_split_pct`, eight separate scroll-offset fields), navigation/focus state (`selected`, `expanded_workspaces`, `list_scroll_focus`, `preview_focused`), and infrastructure (`op_cache`, `instances_last_refresh`) in one struct. Adding one scrollable block touches the state struct, a render signature, the mouse handler, and a focus enum across four files with no single seam to audit.

2. **No `Action`/`Message` type.** Event handling mutates state directly inside the handler. `handle_key` in <RepoFile path="src/console/manager/input/mod.rs" /> is a ~240-line `if`-precedence chain that returns an `InputOutcome` only for routing decisions; all state changes are side effects before the return. Nothing is replayable, loggable, or testable without constructing a full `ManagerState` and driving a real terminal.

3. **The view is not pure.** Render functions take `&mut state`. `render_scrollable_block` clamps `scroll_x`/`scroll_y` in place every frame; the editor tab renderers write `tab_content_width`/`tab_content_height` back into `EditorState` so the mouse handler can later read them. "Draw" mutates observable state, which couples render output to input handling and is the root of a class of stale-geometry scroll bugs.

4. **Filesystem I/O and instance refresh on the render loop.** `mount_info::inspect` (<RepoFile path="src/console/manager/mount_info.rs" />) still runs from render/input paths and stats the disk / reads `.git` pointers for visible mounts. `refresh_instances` still does blocking index/manifest reads and a per-container socket fan-out on the event-loop thread (self-throttled to 500 ms, but still blocking the draw when it fires).

5. **Unconditional full-frame redraw.** The loop draws every tick regardless of whether anything changed. No dirty flag, no skipped frames.

6. **No `StatefulWidget`/`WidgetRef`.** Every renderer is a free `fn(frame, area, &mut state)`. The `unstable-widget-ref` feature is enabled in <RepoFile path="Cargo.toml" /> but unused ŌĆö there are no reusable widget components implementing the Ratatui widget traits.

7. **Three coexisting modal-stack shapes.** A `Vec<Modal>` stack (correct), legacy stash slots (`pending_auth_form_return`, `pending_env_key`, ŌĆ”), and a bare `Option<Modal>` slot the [TUI Design Decisions](/reference/tui-design-decisions/) page already flags as an invalid shape that cannot represent a parent-under-child.

8. **DRY drift from symmetric variants.** `sidebar_inputs_for_workspace` vs. `sidebar_inputs_for_current_dir` in <RepoFile path="src/console/manager/render/list.rs" /> are near-identical bodies; the same workspace-vs-current-dir distinction is repeated in `focused_block_still_scrollable`, `clamp_list_scroll_for_area`, and the mouse `update_scroll_focus` handler ŌĆö four call sites that must stay in lock-step.

9. **Component reuse is happening in the wrong layer.** The launch cockpit now reuses host console widgets such as `select_list`, `status_bar`, `hints`, `scrollable`, and `error_popup`, which is the right instinct. But the dependency direction is wrong: runtime launch code reaches sideways into `src/console/widgets` instead of every terminal surface consuming components from <RepoFile path="crates/jackin-tui/Cargo.toml" />. This is new evidence that `jackin-tui` must become the shared component crate, not just a token/helper crate.

## Current state ŌĆö `jackin-capsule` multiplexer

The in-container control plane (the [jackin' Capsule](/reference/roadmap/jackin-capsule/) program) is a small tmux-like PTY multiplexer. It uses `crossterm` only for `enable_raw_mode`/`disable_raw_mode` and `terminal::size()` in <RepoFile path="crates/jackin-capsule/src/client.rs" />. **It does not depend on Ratatui.**

### Anti-patterns

1. **All rendering is hand-rolled raw ANSI** into a `Vec<u8>` ŌĆö direct `write!(buf, "\x1b[{};{}H", ŌĆ”)` cursor moves and inline SGR byte sequences in <RepoFile path="crates/jackin-capsule/src/render.rs" />, <RepoFile path="crates/jackin-capsule/src/dialog.rs" />, and the status bar. There is a hand-rolled SGR serializer and a hand-rolled CSI re-serializer (the latter untested) in <RepoFile path="crates/jackin-capsule/src/session.rs" />.

2. **A hand-rolled row diff re-implements `Buffer::diff`.** `PaneBodyCache` builds a `Vec<RowSnapshot>` from the `vt100::Screen`, compares it cell-by-cell against the cached previous snapshot, and emits only changed rows. This is exactly the job Ratatui's `Buffer` double-buffer does ŌĆö re-built by hand.

3. **A ~6,900-line god-struct.** `Multiplexer` in <RepoFile path="crates/jackin-capsule/src/daemon.rs" /> owns rendering, input parsing, network I/O, git/PR context polling, session lifecycle, the dialog stack, and mouse drag/selection in one file and one compilation unit.

4. **No top-level mode state machine.** "Which mode is the multiplexer in?" is answered by combining scattered booleans (`dialog_open()`, `drag.is_some()`, `selection.is_some()`, `input_parser.is_awaiting_prefix()`) at each call site. There is no `MuxMode` enum making invalid combinations unrepresentable ŌĆö and at least one arm checks `dialog_open()` but not `drag.is_some()`, a latent bug class.

5. **`compose_full_frame` vs. `compose_partial_frame` duplication** ŌĆö ~160 lines of near-duplicate pane-loop / box-draw / scrollbar code that must be kept symmetric by hand.

6. **No tests** for `handle_input` dispatch, `apply_dialog_action`, the dialog `handle_key`/`handle_click`, the drag/selection state machines, or the compositor. The parser, protocol, and `render_pane` are tested in isolation; the daemon's decision layer is not.

<Aside type="note">
The capsule already has one piece of the target in place: `Dialog::handle_key`/`handle_click` return a `DialogAction` enum (`Command`, `SpawnAgent`, `RenameTab`, `Dismiss`, ŌĆ”) rather than mutating state directly. That is exactly the `Action` pattern this item generalizes to the whole dispatch layer.
</Aside>

## Current state ŌĆö the console ŌåÆ launch ŌåÆ container flow

The three surfaces are still built differently, but #452 removed the biggest product seam for the console-triggered launch path. `jackin console` now creates a `HostScreen` in <RepoFile path="src/app/mod.rs" /> before entering `run_console`; `run_console` detects that an outer guard owns the terminal and draws into it; `runtime::LaunchProgress` draws the cockpit into the same alternate screen; and `runtime::attach` passes `JACKIN_HOST_ALT_SCREEN=1` into the capsule client so the child skips its own alternate-screen enter/leave. The remaining problem is no longer the old plain-output launch premise; it is that the shipped continuity is implemented as several local contracts and globals instead of one TUI application architecture.

| Segment | Technology | Terminal state | Anchor |
|---|---|---|---|
| Console command wrapper | `HostScreen` guard + Ratatui | raw mode, host alternate screen, mouse capture | <RepoFile path="src/app/mod.rs" />, <RepoFile path="src/console/mod.rs" /> |
| Console manager | Ratatui, **stdout** | draws into the inherited host screen when present; standalone `run_console` still owns its own guard | `run_console` in <RepoFile path="src/console/mod.rs" /> |
| Rare interim prompts | dialoguer / cooked terminal | `HostScreen::suspend` leaves and re-enters the alt screen for sensitive-mount confirm or agent choice | <RepoFile path="src/app/mod.rs" /> |
| Launch progress | Ratatui rich cockpit with compact fallback | rich renderer draws into the inherited host screen; non-rich / non-interactive paths emit compact deterministic lines | <RepoFile path="src/runtime/progress.rs" />, <RepoFile path="src/runtime/launch.rs" /> |
| ŌåÆ attach | `docker exec -it ŌĆ” /jackin/runtime/jackin-capsule` | host screen remains owned; env flag tells capsule client not to nest alt-screen | <RepoFile path="src/runtime/attach.rs" /> |
| In-container client | raw byte relay | raw mode; skips its own alt-screen only when `JACKIN_HOST_ALT_SCREEN` is set | <RepoFile path="crates/jackin-capsule/src/client.rs" /> |
| Daemon render | hand-rolled ANSI | ŌĆö | <RepoFile path="crates/jackin-capsule/src/daemon.rs" /> |

This is a real improvement and should be preserved. The architectural debt is what the improvement exposed:

- `HostScreen` is a terminal-lifecycle guard, not an app session model. It coordinates sub-surfaces through `crate::tui::host_screen_owned()` and `set_rich_surface_active()` globals instead of typed ownership.
- The launch cockpit has its own `LaunchView`, renderer task, picker loop, input drain, and failure modal rather than sharing the console's eventual `Message` / `update` / `Effect` loop.
- The launch cockpit imports reusable pieces from `src/console/widgets`, so the first cross-surface reuse exists but lives below the wrong module boundary.
- `jackin load` still creates the launch cockpit as a standalone rich surface; only the console path has the outer `HostScreen` that spans console ŌåÆ launch ŌåÆ attach.
- The capsule render path is still independent raw ANSI. The host can preserve a single alt-screen across handoff, but the in-container UI is not yet part of the shared Ratatui component system.

## Ratatui application patterns (the target)

The Ratatui project documents three application patterns and recommends one for immediate-mode apps:

- **The Elm Architecture (TEA)** ŌĆö [Model / Update / View](https://ratatui.rs/concepts/application-patterns/the-elm-architecture/). State lives in a Model; events map to a `Message` enum; an `update` function applies a message to the model; a **pure** `view` function renders the model. Ratatui calls TEA "the best architectural approach for immediate-mode rendering."
- **[Component Architecture](https://ratatui.rs/concepts/application-patterns/component-architecture/)** ŌĆö components own their local state and event handlers. The [recommended project structure](https://ratatui.rs/templates/component/project-structure/) is `app.rs` ┬Ę `action.rs` ┬Ę `event.rs` ┬Ę `tui.rs` ┬Ę `components/`, with the guidance that "you shouldn't need to change anything outside the `components` folder."
- **[Flux](https://ratatui.rs/concepts/application-patterns/flux-architecture/)** ŌĆö unidirectional data flow; an alternative to TEA for larger apps.

Supporting practices:

- **`StatefulWidget` / `WidgetRef`** separate a widget's drawing logic from a persistent `State` object ŌĆö the idiomatic way to build reusable, testable components.
- **Immediate-mode discipline**: the view is a pure function of state, redrawn each frame; mutation and I/O belong in `update`, never in `view`.
- **I/O off the render/update thread**: long-running work (disk, network, subprocess) runs in background tasks that send `Message`s back to the loop, so the draw never blocks.
- For PTY rendering specifically, [`tui-term`](https://github.com/a-kenji/tui-term) renders a `vt100::Screen` into a Ratatui `Buffer` (the same approach [used by Turborepo](https://github.com/vercel/turborepo/pull/9123) and [Ratty](https://blog.orhun.dev/introducing-ratty/)) ŌĆö which means the capsule *could* keep its `vt100` model while letting Ratatui own the buffer diff and the chrome.

<Aside type="caution">
Dogmatic TEA clones the whole model on every update. jackin's state is large, so a clone-per-keystroke model is impractical. The realistic target is Ratatui's documented **mutable** variant: `fn update(&mut self, msg: Message)` mutating the model in place, paired with a strictly **pure** `fn view(&self, frame)`. The win is the `Message` boundary and the pure view, not literal immutability.
</Aside>

## Blocking-work inventory ŌĆö async migration targets

The async-first principle becomes actionable as a concrete list. Every site below either blocks the render/update loop today or sleeps synchronously on a render-adjacent path; the migration target for each is the same ŌĆö move it off the loop so the draw keeps ticking. Phases 2, 3, and 5 are where these land.

**Host console ŌĆö work on the event-loop thread:**

| Site | What blocks | Target |
|---|---|---|
| `mount_info::inspect` (<RepoFile path="src/console/manager/mount_info.rs" />) | per-frame `stat()` + `.git` reads for every mount, at 20 Hz, uncached | cached `MountFacts`, refreshed by a background `MountInspected` message |
| `refresh_instances` (<RepoFile path="src/console/manager/state.rs" />) | blocking index/manifest reads + per-container socket fan-out (self-throttled to 500 ms, still blocks the draw when it fires) | background task ŌåÆ `InstancesRefreshed` message |
| 1Password loads / snapshot fan-out | synchronous `OpRunner` + socket calls invoked from input handlers | background task ŌåÆ result message |
| `event::poll(50 ms)` loop | wakes 20├Ś/s even when idle; full-frame redraw every tick | `tokio::select!` over an `EventStream` + the `Message` channel; dirty/animation-tick redraw |

**Launch flow ŌĆö the cockpit render cadence:**

The launch cockpit now has a background render task in <RepoFile path="src/runtime/progress.rs" />. It ticks every ~33 ms, snapshots a shared `LaunchView`, drains cockpit-local input, and draws the Ratatui frame while <RepoFile path="src/runtime/launch.rs" /> awaits Docker/network/workspace futures. `LaunchProgress::while_waiting` is now deliberately a thin await wrapper because the renderer no longer needs each launch step to interleave drawing manually. That means the old "wrap every step in a select loop" plan is stale.

The remaining migration target is different: fold the cockpit's parallel mini-app into the same app-level message/effect architecture as the console. Launch stage mutations become messages; forced-choice pickers and failure acknowledgement become effects/input states; the background render task becomes the shared animation-tick source instead of a launch-only thread. Any synchronous work that still runs before `LaunchProgress::new`, between stage mutations, or during rich-surface input handling must move behind effects so the cockpit and future shared session stay responsive.

**Synchronous `std::thread::sleep` on render-adjacent paths** ŌĆö each should become `tokio::time::sleep`, an interval tick, or move off the render task:

| Site | Sleep | Note |
|---|---|---|
| <RepoFile path="src/tui/output.rs" /> | 25 ms shimmer ┬Ę 1500 ms deploy hold | animation/pacing on the launch path ŌĆö drive from an interval, not a blocking sleep |
| <RepoFile path="src/runtime/snapshot.rs" /> | 20 ms ┬Ę 10 ms | retry pacing |
| <RepoFile path="src/runtime/logs.rs" /> | 250 ms | poll pacing |
| <RepoFile path="src/console/widgets/op_picker/mod.rs" /> | 2 ms ├Ś2 | worker poll pacing |

**Capsule:** PTY and git/PR context already run on background tasks; the gap is the `std::thread` vs. `spawn_blocking` inconsistency the gap-analysis table notes, plus the compose path running on the loop. The Ratatui-frame move (Phase 6) puts the buffer diff on the framework and the compose work behind the same `Action`/update boundary.

Line references and timings drift ŌĆö re-grep (`rg 'thread::sleep' src/`, `rg 'while_waiting' src/`) before acting on any row.

## Target architecture ŌĆö host operator console

Introduce the Model/Update/View seam without changing behavior:

1. **One dispatch type + central `update`.** Define the dispatch `enum` ŌĆö `Message` on the host, the generalized `Action` on the capsule, **converged on a single name** so the shared vocabulary stays recognizable ŌĆö covering input intents (`MoveSelection`, `OpenModal`, `CommitSave`, `LaunchWorkspace`, ŌĆ”), background results, and their failures (`InstancesRefreshed`, `MountInspected`, `OpLoadFailed(e)`, ŌĆ”) so errors flow as messages routed to the `ErrorPopup`, never as `unwrap`/panic. Map every `crossterm` event ŌĆö key, mouse, **paste, focus, and resize** ŌĆö to messages in a thin layer; route every state change through `fn update(&mut self, msg: Message) -> Vec<Effect>` ŌĆö one update may emit several effects (redraw, spawn a task, quit), so it returns a list, not a single `Option`. Handlers stop mutating state directly.

2. **Decompose state.** Split the god-structs along the seams that already exist conceptually: a domain `Model` (workspaces, instances, pending config), a `UiState` (selection, focus, scroll offsets, open modal), and ephemeral per-frame scratch. Geometry that the renderer derives (content width/height) becomes computed-and-cached state owned by `update`, not written from inside `view`.

3. **Pure view.** Convert renderers to `fn(frame, area, &Model, &UiState)` with no `&mut` and no I/O. Move scroll clamping into `update` (on resize / scroll messages). Replace per-frame `mount_info::inspect` with a cached `MountFacts` map refreshed by a background `MountInspected` message.

4. **Background I/O ŌåÆ channel.** Move instance refresh, 1Password loads, snapshot fan-out, and mount inspection onto background tasks that send `Message`s. `run_console` is already an `async fn`, so the target loop is a `tokio::select!` over a crossterm `EventStream` and the background `Message` channel ŌĆö replacing today's synchronous `event::poll(50 ms)` ŌĆö so the draw never blocks on disk/socket work and no longer wakes 20├Ś/s just to spin. (Fallback if `EventStream` proves awkward: a dedicated input thread feeding the same `Message` channel via `std::sync::mpsc`.)

5. **`StatefulWidget` components.** Extract the list, the editor (per tab), settings, and the modal family into components implementing the Ratatui widget traits, each with its own `State`. This is where the [Split `input/editor.rs`](/reference/roadmap/split-input-editor/) and [Split `app/mod.rs`](/reference/roadmap/split-app-mod/) work should land ŌĆö the splits target component boundaries, not arbitrary smaller files.

6. **Converge the modal stack** onto the single `Vec<Modal>` shape and retire the stash slots and bare `Option<Modal>`, satisfying the existing [TUI Design Decisions](/reference/tui-design-decisions/) sub-dialog-stack rule by construction.

Every existing rule in [TUI Design Decisions](/reference/tui-design-decisions/) (W3C ARIA tabs, the focusability/affordance contract, the shared `render_scrollable_block`, modal backdrop dimming, the PHOSPHOR palette, settingsŌåöeditor parity) is preserved ŌĆö the refactor changes the *structure behind* those rules, never the rules themselves.

## Target architecture ŌĆö `jackin-capsule`

Two questions are independent and should be decided separately:

**(1) Structural cleanup ŌĆö clear win, do regardless of rendering backend.**

- Add a `MuxMode` enum (`Normal`, `PrefixAwait`, `Dialog`, `Drag`, `Select`) so invalid combinations are unrepresentable and dispatch reads from one place.
- Generalize the existing `DialogAction` into a single `Action` type for the whole `handle_input` dispatch ŌĆö the *same dispatch type the host converges on* (see the host target) ŌĆö so the daemon's decision layer is testable without a live PTY.
- Split <RepoFile path="crates/jackin-capsule/src/daemon.rs" /> into modules: event loop, session lifecycle, render/compose, input dispatch, dialog. Merge `compose_full_frame`/`compose_partial_frame` onto a shared pane-composite helper.
- Add dispatch/dialog/drag tests once the `Action` boundary exists.

**(2) Render through Ratatui ŌĆö the decided direction.**

The capsule moves to **full Ratatui**: the daemon builds frames in a Ratatui `Buffer` and renders chrome, status bar, tab strip, dialogs, pickers, and confirms as Ratatui widgets ŌĆö the *same* shared widgets the host console uses (see the shared design system below). The attach client in <RepoFile path="crates/jackin-capsule/src/client.rs" /> stays a thin byte relay; Ratatui renders inside the **daemon** via a custom backend that serializes the `Buffer` diff to the attach socket instead of a real TTY, replacing the hand-rolled `PaneBodyCache` row diff with Ratatui's own `Buffer::diff`.

The one open question is the **pane-body** mechanism ŌĆö the cells that mirror a child program's `vt100::Screen`, the perf-critical hot path:

| Pane-body option | Shape | Trade-off |
|---|---|---|
| `tui-term` | render the `vt100::Screen` into the Ratatui `Buffer` via the pseudo-terminal widget | maximal reuse of Ratatui's buffer/diff; `tui-term` is self-described "work in progress" with a known per-cell allocation cost on high-throughput PTYs |
| thin custom cell widget | a small `Widget` that blits the `vt100::Screen` into the `Buffer` directly | keeps the proven fast path while still living inside the Ratatui frame; a little custom code, no new dependency risk |

**Resolve the pane-body question with a throughput spike** (a later phase) ŌĆö both options keep the capsule fully inside a Ratatui frame, so chrome / dialog / picker reuse lands regardless. That benchmark is the only part of the capsule that stays a measurement question; everything else is decided. The shared widgets live in the existing <RepoFile path="crates/jackin-tui/Cargo.toml" /> crate (today Ratatui-free), which grows a Ratatui dependency to host them.

## Target architecture ŌĆö one owned terminal session

The first version of this exists now, so the target is no longer "move ownership up" in the abstract. The target is to formalize the shipped `HostScreen` / launch renderer / capsule handoff into one app-level TUI session spine:

- Keep the #452 behavior: the console manager renders into an outer host screen, the launch cockpit draws into that same screen, and the capsule handoff preserves the host alternate screen with `JACKIN_HOST_ALT_SCREEN` so the child does not enter a nested alt-screen.
- Replace the global coordination flags (`host_screen_owned`, `rich_surface_active`) with typed session ownership: an `AppTuiSession`/`TerminalSession` object owns raw mode, alt-screen, mouse capture, debug buffering, rich-surface diagnostics routing, suspend/resume, and the handoff contract.
- Make console and launch views siblings in one app model. "Operator picked a workspace" ŌåÆ "watching the launch cockpit" becomes a `Message`/`Effect` transition, not `run_console` returning an outcome to a separate launch mini-app.
- Convert launch progress updates (`stage_started`, `stage_done`, failures, build-log overlay, forced-choice picker, failure acknowledgement) into the same message/effect vocabulary as the console. The existing background render task is preserved conceptually as the animation tick source, but it is not launch-private.
- Move reusable launch widgets out of `src/console/widgets` and into <RepoFile path="crates/jackin-tui/Cargo.toml" />. Runtime code must not depend on console internals just to render a status bar, hint row, select list, scrollable build log, or error popup.
- Keep the rare cooked prompts as explicit `SuspendForPrompt` effects. The current `HostScreen::suspend` behavior is useful, but the reason for leaving the rich surface should be typed and visible in the update loop.
- Preserve panic-safe restore. The RAII guard remains the ultimate owner of terminal restoration; the refactor changes where ownership is modeled, not the guarantee that raw mode and alternate screen are restored on every exit path.

This item now owns the consolidation of the shipped backbone. The [Launch Progress TUI](/reference/roadmap/launch-progress-tui/) item owns the operator-facing launch cockpit and diagnostics behavior that already landed; this item owns making that behavior a reusable architecture rather than a special case.

## Shared design system & lookbook (`jackin-tui`)

A shared crate already exists: <RepoFile path="crates/jackin-tui/Cargo.toml" />. Today it is a **Ratatui-free crate of raw-ANSI primitives and pure helpers** ŌĆö the PHOSPHOR palette as `Rgb`, an `ansi` module (`fg` / `bg` / `move_to`), `TextField` (a text-input model), `HintSpan` + `hint_row_cols`, `TabCell` + `lay_out_tabs`, `vertical_thumb`, and unicode-width helpers. The capsule consumes it directly; the host console consumes only the **token layer** ŌĆö <RepoFile path="src/console/widgets/mod.rs" /> wraps `jackin-tui`'s `Rgb` palette values into `ratatui::Color`.

That token layer is the model to follow: one source of truth for the values, a thin per-surface conversion. The gap is that **every component above the token layer is reimplemented per surface or borrowed through the wrong module boundary** ŌĆö the host has its own tab strip, scrollbar, text input, hint model, and confirm dialog; the launch cockpit imports host widgets directly from `src/console/widgets`; and the capsule hand-rolls each again in raw ANSI, kept in sync only by the procedural rules in [TUI Design Decisions](/reference/tui-design-decisions/). **Target: elevate `jackin-tui` into the canonical TUI design-component crate** every terminal surface consumes; the crate that already owns the tokens grows to own the components. Do not create a new crate.

This is a design-system problem, and the web ecosystem has a mature playbook for it. An audit found **roughly 45 repeatable visual patterns** ŌĆö an approximate figure; the enumerated drift table below is the verifiable subset ŌĆö about a dozen implemented twice and several with measurable cross-surface drift. The fix is a single token set, a single component library, and a browsable **lookbook** that renders every component and its states so the operator can point at one and say "use this." The canonical *rules* stay in [TUI Design Decisions](/reference/tui-design-decisions/); the lookbook renders the *components* and never restates a rule ŌĆö its Do / Don't rows link to the relevant rule anchor. The catalog below, organized by atomic-design layer, is the seed inventory; each row becomes a lookbook page.

### Rendering TUI components in the docs

The lookbook is only trustworthy if its previews are the *real* components, not hand-drawn art that drifts. The plan, strongest first:

- **Primary ŌĆö render the real widget to SVG.** Draw each component (and each state) through `ratatui::backend::TestBackend` into a `Buffer`, then serialize the buffer's cells (char + fg/bg/modifiers) to inline **SVG** (or styled HTML). A small `cargo run --bin tui-lookbook` emits one SVG per component-state from the actual widget code; MDX embeds the generated SVGs; CI regenerates them so they never drift. This reuses the exact `TestBackend` + `insta` infrastructure from [Snapshot tests for TUI render](/reference/roadmap/snapshot-tests-tui/) ŌĆö the same render that guards regressions also produces the lookbook art. [`term-transcript`](https://docs.rs/term-transcript) (Rust; renders terminal output to SVG/HTML) is a candidate serializer.
- **Motion and raw-ANSI surfaces** ŌĆö for the digital rain, spinners, and (until they move to Ratatui) the capsule's raw-ANSI dialogs, record an asciicast and convert with [`vhs`](https://github.com/charmbracelet/vhs), [`termsvg`](https://github.com/MrMarble/termsvg), [`svg-term`](https://github.com/marionebl/svg-term-cli), or [`termtosvg`](https://nbedos.github.io/termtosvg/) to an animated SVG embeddable in MDX.

## Terminal capability & degradation

A single token layer makes graceful degradation a property of the tokens, not of every call site:

- **Color depth.** Tokens resolve through Ratatui's color handling for truecolor; on 256- or 16-color terminals the palette maps to the nearest supported color, and `NO_COLOR` / `COLORTERM` are honored so the UI stays legible without a redesign.
- **Glyph fallback.** The box-drawing, scrollbar, and selection glyphs (`ŌöīŌöĆŌöÉŌöéŌööŌöś`, `Ō¢ł`, `┬Ę`, `Ō¢Ė`, `Ō¢æ`) get an ASCII fallback set for terminals that cannot render them, selected once at startup rather than per call site.
- **Size.** Components clamp to a minimum usable size and degrade (hide chrome, collapse columns) on narrow terminals instead of panicking or overflowing.
- **Reduced motion.** Animations (digital rain, spinners, shimmer) honor the reduced-motion handling owned by [Launch Progress TUI](/reference/roadmap/launch-progress-tui/); the design system exposes the same gate so static fallbacks stay consistent.

**Non-goals (this refactor).** Screen-reader / assistive-technology support and alternate themes (light, high-contrast) are out of scope ŌĆö but the single token set makes both a future *configuration* change rather than a rewrite, which is the point of centralizing them.

## Constraints and non-goals

- **No behavior change.** Snapshot tests gate every step; the operator must not see a difference. This is a structure refactor, not a redesign ŌĆö visual/UX changes are out of scope and belong to items like [Launch Progress TUI](/reference/roadmap/launch-progress-tui/).
- **Pre-release freedom.** Internal types may change freely; no migration shims or compatibility layers (these are not versioned schemas).
- **Incremental only.** No big-bang rewrite. One surface, one layer, behind tests. A half-migrated state (some handlers on `Message`, some not) must compile and pass tests at every commit.
- **DRY.** Extend the shared scrollable/widget helpers and the `jackin-tui` crate; do not fork parallel implementations.

## Implementation phases

### Phase 0 ŌĆö Safety net and decision record

Land the [Snapshot tests for TUI render](/reference/roadmap/snapshot-tests-tui/) item (the regression net) and extend [ADR-003 "ratatui selection"](/reference/roadmap/architecture-decision-records/) to record the capsule's current no-Ratatui status and the pane-body rendering decision. This page is the design artifact.

### Phase 1 ŌĆö Host: `Message` boundary

Introduce the `Message` enum and central `update`; route existing key/mouse handlers through it without moving the renderers. Behavior identical; the win is a testable update layer.

### Phase 2 ŌĆö Host: pure view

Remove `&mut` and I/O from renderers. Move scroll clamping into `update`. Replace per-frame `mount_info::inspect` with cached facts refreshed by a message. Introduce event-driven (dirty) redraw ŌĆö draw on a `Message` or an animation tick, not unconditionally every 50 ms ŌĆö so an idle console does no work.

### Phase 3 ŌĆö Host: background I/O

Move instance refresh, 1Password loads, snapshot fan-out, and mount inspection onto background tasks that send `Message`s. The draw never blocks.

### Phase 4 ŌĆö Host: components + modal-stack convergence

Extract `StatefulWidget` components (list, editor-per-tab, settings, modals) into the shared <RepoFile path="crates/jackin-tui/Cargo.toml" /> design crate; land the [Split `input/editor.rs`](/reference/roadmap/split-input-editor/) and [Split `app/mod.rs`](/reference/roadmap/split-app-mod/) work against these seams; converge the modal stack onto `Vec<Modal>`. Token and layout-helper unification (the option-independent floor) can land earlier, in Phase 1.

### Phase 5 ŌĆö App: one owned terminal session

Consolidate the `HostScreen` behavior that [Launch Progress TUI](/reference/roadmap/launch-progress-tui/) shipped into a typed app-level `TerminalSession`. The console manager and launch cockpit already share one host alternate screen on the console path; this phase removes the ad hoc globals and return-to-app handoff, models console and launch as sibling views in one app model, makes launch updates messages/effects, and moves the launch-used widgets from `src/console/widgets` into <RepoFile path="crates/jackin-tui/Cargo.toml" />. The visible behavior stays the same: console ŌåÆ launch ŌåÆ capsule remains continuous, with cooked prompts represented as explicit suspend effects.

### Phase 6 ŌĆö Capsule: structural cleanup + Ratatui frame

Add `MuxMode`; generalize `Action`; split <RepoFile path="crates/jackin-capsule/src/daemon.rs" />; merge the full/partial compositors; add dispatch/dialog tests. Then move the daemon to render into a Ratatui `Buffer` via a custom attach-socket backend, replacing `PaneBodyCache` with `Buffer::diff`, and render chrome / dialogs / pickers as the shared <RepoFile path="crates/jackin-tui/Cargo.toml" /> widgets.

### Phase 7 ŌĆö Capsule: pane-body spike + final reuse

Benchmark `tui-term` against a thin custom cell widget for pane bodies on a high-throughput PTY and pick the mechanism. Land the shared confirm / picker / tab-strip / panel / hint-bar widgets so the host console and the capsule render identical components.

### Phase 8 ŌĆö Design-system lookbook

Stand up the lookbook: a `tui-lookbook` binary that renders each `jackin-tui` component and its states to SVG from the real widget code, one MDX page per component following the story shape above, and CI regeneration so previews never drift. Reuses the [Snapshot tests for TUI render](/reference/roadmap/snapshot-tests-tui/) `TestBackend` infrastructure. Reconcile the host/capsule drift table as components consolidate into shared widgets.

## Open questions & risks

- **Handoff feasibility.** The single-host-alt-screen approach (the host keeps its alt-screen across `docker exec -it`; the capsule client skips its own `?1049` toggle via `JACKIN_HOST_ALT_SCREEN`) is implemented by the [Launch Progress TUI](/reference/roadmap/launch-progress-tui/) work and must keep being confirmed across real terminals ŌĆö alt-screen survival, raw mode, SIGWINCH, and the stale `Buffer` full-redraw on resume. If a terminal mishandles the shared screen, the fallback is to tear the host backend down before the exec and re-initialize on return, with the flash minimized.
- **Global session flags.** `host_screen_owned` and `rich_surface_active` encode real product state but are not a durable architecture. Phase 5 must replace them with typed ownership/effects without regressing diagnostics routing, compact fallback output, or panic-safe terminal restore.
- **Async loop shape.** `EventStream` + `tokio::select!` vs. a dedicated input thread + `std::sync::mpsc`. Decide in Phase 1; the `Message` boundary keeps the rest of the design independent of the choice.
- **Pane-body rendering cost.** `tui-term`'s per-cell copy on high-throughput PTYs vs. a thin custom cell widget ŌĆö the Phase 7 benchmark.
- **`ratatui-textarea` vs. `TextField`.** *Decision:* consolidate the host text input onto the in-house `TextField` (already shared via `jackin-tui`) and drop the third-party, pre-1.0 `ratatui-textarea`, so there is one text-input model. Revisit only if `TextField` lacks needed editing features.
- **`unstable-widget-ref`.** Enabled but unused today. *Decision:* build components as `StatefulWidget` and drop the unstable feature unless `WidgetRef` proves genuinely necessary, to avoid depending on an unstable API.
- **Lookbook SVG toolchain.** `term-transcript` vs. a custom `Buffer`ŌåÆSVG serializer for widgets, and `vhs` / `termsvg` for motion ŌĆö pick in Phase 8.
- **Dirty redraw vs. animation.** Event-driven redraw must still drive spinner / rain animation ticks, so the redraw scheduler needs an animation-timer source, not only input `Message`s.

## Host-side effects

None. This is a pure internal refactor of in-process TUI code. It introduces no new writes to the operator's host machine, no new files, and no new container paths.

## Acceptance criteria

- Host event handling routes through a `Message` enum and a central `update`; no renderer takes `&mut state` or performs disk/socket I/O.
- Per-frame filesystem inspection is eliminated; mount facts and instance state are refreshed by background tasks that message the loop.
- Reusable screens are `StatefulWidget` components; the modal stack is a single `Vec<Modal>` shape.
- The console ŌåÆ launch ŌåÆ handoff remains one continuous Ratatui session on the console path, but the continuity is owned by a typed app-level `TerminalSession` rather than global flags; the capsule handoff still preserves the host alt-screen with `JACKIN_HOST_ALT_SCREEN`, and the in-container experience continues the same visual language. *(Manual-smoke criterion ŌĆö a real-terminal flash is not catchable by `TestBackend` snapshots.)*
- The capsule has a `MuxMode` state machine, a generalized `Action` dispatch with tests, and a decomposed `daemon.rs`; the full/partial compositors share one helper.
- The capsule renders through Ratatui; the pane-body mechanism (`tui-term` vs a custom cell widget) is recorded in an ADR with benchmark evidence.
- Every rule in [TUI Design Decisions](/reference/tui-design-decisions/) still holds; snapshot tests show no render diff across the refactor.
- Reusable patterns live in <RepoFile path="crates/jackin-tui/Cargo.toml" /> as single implementations; no visual pattern that appears on multiple terminal surfaces has two copies or a runtime dependency on `src/console/widgets`. The confirm dialog, picker, tab strip, scrollbar, panel, hint bar, status/footer bar, and modal frame each have exactly one implementation all surfaces consume.
- Token and layout-helper unification lands without a rendering-backend change; full widget sharing follows the capsule's move to a Ratatui frame.
- Every repeatable visual pattern is a named `jackin-tui` component with a lookbook page rendering its states from the real widget code; no host/capsule drift remains for shared patterns.
- Lookbook previews are generated from `TestBackend` renders (or recorded SVG for motion) and regenerated in CI, so they cannot drift from the shipped components.
- No phase introduces an operator-visible behavior change.
