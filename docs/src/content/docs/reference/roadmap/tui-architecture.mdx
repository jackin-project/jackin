**Status**: Open — design proposal (re-analyzed after Launch Progress TUI landed in May 2026; no implementation for this item committed)

## Goal

jackin' renders three connected terminal surfaces: the host operator console (`jackin console`), the launch progress cockpit, and the in-container multiplexer (`jackin-capsule`). The first two now use Ratatui, and the console-triggered launch path now keeps one host alternate screen across console → launch → capsule attach. That shipped backbone is the right product direction, but the architecture behind it is still not the Ratatui application pattern this codebase should standardize on: the host console remains a procedural draw-everything-every-frame loop over god-structs, the launch cockpit owns a parallel mini-app under <RepoFile path="src/runtime/progress.rs" />, and the capsule still uses **no Ratatui at all** — it hand-rolls ANSI escape writes and a row diff that re-implements Ratatui's own `Buffer::diff`.

This item defines the **architectural target** for all three surfaces: a clear Model / Update / View separation, an explicit `Action`/`Message` type instead of inline state mutation in event handlers, a pure view layer with no I/O, state decomposed into domain vs. UI vs. ephemeral concerns, `StatefulWidget`-based reusable components, and background I/O that feeds the update loop through a channel rather than blocking the draw. It also commits `jackin-capsule` to **rendering through Ratatui** instead of its current hand-rolled ANSI, so the in-container multiplexer shares the same component library as the host. Throughout, the organizing goal is **reusability**: a single, recognizable component vocabulary that every terminal surface shares, instead of the parallel copy-paste implementations the codebase accumulates today.

This is a north-star design item, not a big-bang rewrite. The work lands incrementally, one surface and one layer at a time, behind the snapshot-test regression net, with **no operator-visible behavior change** at any step.

## What this roadmap must deliver

This item exists to make future TUI work obvious to place, easy to read, and hard to block at runtime. Every phase below should be judged against these three outcomes:

| Outcome | Target | How an implementation proves it |
|---|---|---|
| **Clear structure** | Every TUI concern has one obvious owner: app session/runtime, app model, update/effects, reusable components, surface-specific domain state, terminal capability handling, and tests/lookbook. | A contributor changing a footer, modal, launch stage, capsule dialog, or async refresh knows which module owns it without searching unrelated surfaces. `src/runtime` no longer imports `src/console/widgets`; capsule UI code no longer lives as raw ANSI copies beside daemon business logic. |
| **Understandable, idiomatic Ratatui code** | The host console, launch cockpit, and capsule all read like Ratatui applications: typed messages, central update, pure view, `StatefulWidget` components, explicit effects, decomposed state, and snapshot-backed render tests. | Event handlers do not directly mutate god-structs; render functions do not take `&mut state`; repeated visual patterns are named `jackin-tui` components; module boundaries follow Model / Update / View and component seams rather than file-size-only splits. |
| **Non-blocking UX** | The TUI keeps responding and animating while Docker, git, 1Password, filesystem, capsule socket, diagnostics, and launch work is in flight. | Long-running work runs as typed effects or subscriptions and reports back through messages. The only draw owner is the TUI runtime. No disk, socket, subprocess, or sleep runs inside render or on the update path unless it is demonstrably trivial and bounded. |

The intended operator-visible result is not a redesign. The screens should look the same unless a later UX roadmap item deliberately changes them. The visible improvement from this item is responsiveness and consistency: no frozen progress cockpit, no event loop blocked by refresh work, no host/capsule drift in common components, and no terminal handoff flash introduced by the refactor.

One UX behavior is part of the architecture, not optional polish: every surface must have a single visible focus owner. The active interaction container gets the bright `PHOSPHOR_GREEN` border; background and sibling containers use the dark border. Opening a dialog moves the bright border to that dialog; closing it restores the bright border to the underlying focused block; clicking a scrollable detail block moves the bright border from the workspace list to that block. The component library and state model must make this focus-visible rule the default, matching [TUI Design Decisions](/reference/tui-design-decisions/#focus-visible-border-one-active-container-owns-bright-green/).

## Scope — one continuous Ratatui experience

This item covers **every terminal surface jackin' renders**, unified under Ratatui and one shared component library:

1. **The operator console (`jackin console`)** — the full flow from console start through workspace / role / agent selection. Already Ratatui; this item restructures it (see the host target below).
2. **The loading flow (console → launch → container).** The [Launch Progress TUI](/reference/roadmap/launch-progress-tui/) work shipped the product backbone for this: a Ratatui cockpit, structured stages, diagnostics run IDs, a console-owned host alternate screen, and a capsule handoff flag that prevents nested alt-screen toggles. The remaining target is to make that shipped backbone architectural rather than ad hoc: one app-level TUI session model, one update/effect boundary, one shared component library, no direct imports from `src/console/widgets` into runtime launch code, and no split where launch progress is a separate mini-app beside the console.
3. **Ratatui inside the container.** The `jackin-capsule` multiplexer daemon renders the in-container experience. It moves from hand-rolled ANSI to **rendering through Ratatui** (the attach client stays a thin byte relay — see the capsule target below).
4. **The capsule, fully Ratatui.** Every capsule surface — pane chrome, status bar, tab strip, dialogs, pickers, confirms — becomes a Ratatui widget, and those widgets are the *same* shared components the host console uses.

The connective tissue across all four is the shared design-component crate (`jackin-tui`): one `ConfirmDialog`, one `FilterList`, one `TabStrip`, one `Panel`, one `HintBar`, reused everywhere rather than reimplemented per surface — catalogued and rendered in a browsable **lookbook** (the design-system section below).

## Design principles

**Reusability is the first principle of this item, not one of several — every other goal here is downstream of it.** The single biggest source of TUI entropy today is copy-paste: the same confirm dialog, picker, tab strip, scrollbar, and palette wrapping exist as parallel implementations across the host console and the capsule, kept consistent only by the procedural rules in [TUI Design Decisions](/reference/tui-design-decisions/). Ending that failure mode is the whole point of this item. The non-negotiable rule: **no code is duplicated and no component is copied. Anything — widget, block, dialog, layout, or UI logic — that can appear more than once is implemented exactly once and instantiated everywhere it is needed, with one source of truth, so a change in that one place changes every instance.** When new functionality would repeat existing code, the work is to refactor the existing thing into a reusable component *first* and then use it — never to paste a second copy and "keep them in sync." We do not tolerate multiple places where the same thing works slightly differently. Every TUI improvement under this item begins with the question "what is the reusable shape?", and that includes the whole **transition experience** (console → loading → capsule → exit), which must itself be composed from shared components, not per-surface copies. The structural fix is to make reuse the path of least resistance — a real design-component crate, not a rulebook.

- **One pattern, one component — zero duplication.** If a visual pattern, widget, block, or piece of UI logic can appear twice, it is a single shared implementation with one source of truth, instantiated per use site — never a second copy kept in sync by hand. Copying code to produce a variant is forbidden; the correct move is to parameterize or extend the one implementation so all instances advance together. This operationalizes the existing "Component Reuse — Strict Rule" in [TUI Design Decisions](/reference/tui-design-decisions/) structurally rather than by review discipline: with the shared component crate in place, the duplication becomes hard to introduce, not merely discouraged.
- **One design system, every terminal surface.** The host console, launch cockpit, and capsule render the same brand language (PHOSPHOR palette, tight chrome, footer-only hints, identical confirm / picker / modal shapes). They must consume the *same* components from the *same* crate, so a confirm dialog is literally the same code everywhere — not two copies a rule says must match.
- **Frontend-team workflow, terminal edition.** Agents should work like a frontend team using Storybook and a component library: pick an existing component from the catalog, pass props/state, wire messages/effects, and ship. They should not invent a new dialog, picker, footer, list row, or panel while implementing a feature.
- **Recognizable, not bespoke.** A contributor (or agent) reading the code should recognize the component vocabulary immediately: `Panel`, `ScrollableBlock`, `ConfirmDialog`, `FilterList`, `TabStrip`, `HintBar`, `Modal`. New screens compose existing components; a genuinely new component is added only when none fits, with operator sign-off per the strict-reuse rule.
- **Idiomatic Ratatui.** Components are `StatefulWidget` / `WidgetRef` implementations with their own `State`, the shape the Ratatui ecosystem recognizes and the [component template](https://ratatui.rs/templates/component/project-structure/) is built around — so the codebase reads like a Ratatui app, not a one-off renderer.
- **One source of truth for tokens.** Color values, spacing, tab gap, and hint formatting are defined once and converted per surface (raw-ANSI `Rgb` today, `ratatui::Color` for widgets), never re-declared inline.
- **Async-first rendering — never block the loop.** Every animated or progress-bearing surface keeps drawing on its own cadence *while work is in flight*. Rendering and long-running work (Docker calls, image pulls, network waits, git, daemon round-trips, disk inspection) run on independent async tasks: an interval-/`Message`-driven loop repaints shared state, and the work mutates that state from a `tokio::select!` arm, a `tokio::spawn`d task, or `spawn_blocking` — never inline with the draw. A surface that only repaints *between* sequential blocking steps freezes the instant a step is slow: the digital rain stalls, the active-stage pulse stops, the spinner hangs, and the operator cannot tell whether jackin' is working or wedged. Steady motion regardless of how long the underlying step takes is what makes the TUI feel native, snappy, and alive — and that "snappy and alive" feel is an explicit operator goal for this item, not a nice-to-have side effect. The acceptance test for any new long-running step on an animated surface: *does the animation keep moving smoothly while the step runs?* If it stutters, the step is on the render loop and must move off it. The concrete sites to migrate are enumerated in the [blocking-work inventory](#blocking-work-inventory--async-migration-targets) below.

## Product goal — component library first

The end state should feel like a mature frontend codebase:

- A contributor opens the **TUI lookbook**, finds `ConfirmDialog`, `FilterListPicker`, `StatusFooter`, `ScrollablePanel`, or `TabStrip`, and uses it.
- Component pages show real rendered states, accepted props/state, emitted messages, keyboard/mouse contract, layout constraints, accessibility notes, screenshots/SVGs, and usage examples.
- New feature work starts from "which existing component expresses this?" not "draw a block and spans."
- Design-system work has its own review path: add or change a component once, update its stories/tests, then consume it in product screens.
- Product screens compose components; product screens do not own palette constants, border glyphs, hint rendering, button styling, modal chrome, list-row highlight rules, scrollbar math, or text-input behavior.

The practical rule: **feature PRs may use components; component PRs may create components.** If a feature needs a new reusable widget, split that into a component-library step first unless the feature is a tiny one-off with a written reason. This mirrors frontend teams that keep design-system primitives separate from app screens, and it makes agents less likely to create near-copy UI.

### Component maturity ladder

Every repeated TUI pattern should move through this ladder:

| Level | State | Allowed use |
|---|---|---|
| 0 — scattered | Inline spans/layout in product code | Existing debt only |
| 1 — local helper | Helper in `src/console/widgets` or capsule raw ANSI | May be used by one surface, not copied |
| 2 — shared model | Shared state/input model in `jackin-tui`, per-surface renderer | Good for capsule-before-Ratatui transition |
| 3 — shared Ratatui widget | `StatefulWidget` / pure renderer in `jackin-tui` | Default for host + launch |
| 4 — catalogued component | Widget has stories, snapshot/SVG lookbook, usage docs, tokens, states | Required before broad reuse |
| 5 — cross-surface canonical | Host, launch, capsule all consume one implementation | Target state |

This lets the migration be incremental without pretending partial sharing is complete.

### Component contract

Each `jackin-tui` component should expose the same shape:

| Contract piece | Purpose |
|---|---|
| `Props` | Immutable inputs from product model: labels, rows, mode, disabled state, severity |
| `State` | Local UI state: focus, filter text, scroll offset, hover, caret, selected row |
| `Message` / `Outcome` | Semantic output: `Selected(id)`, `Cancelled`, `FilterChanged`, `Scrolled`, `Copied` |
| `Effect` hooks | Optional typed requests: clipboard copy, diagnostics note, async load trigger |
| `render(&Props, &State, area)` | Pure Ratatui render, no I/O, no mutation |
| `update(&mut State, ComponentEvent) -> ComponentOutcome` | Key/mouse/focus handling local to the component |
| focus-visible styling | Active component/container uses `PHOSPHOR_GREEN` border; inactive/background containers use `PHOSPHOR_DARK`; passive scroll blocks only claim the bright border when overflow makes scroll focus meaningful |
| stories/tests | Render all states through `TestBackend`; pin keyboard/mouse behavior |

Current code already hints at this contract: many widgets have `State`, `handle_key`, `render`, and `ModalOutcome`. The refactor should preserve that useful shape and promote it into `jackin-tui` instead of replacing it with a vague abstraction.

The same audit found concrete places where a component library would pay off immediately:

| Drift source | Evidence | Fix |
|---|---|---|
| Local palette constants | `Color::Rgb(0,255,65)` is re-declared in save/source/scope widgets, file browser, save input code, capsule dialogs/status bar, and runtime progress animation code | only token modules may define RGB values; all product code imports semantic tokens |

### Proposed `jackin-tui` package layout

Keep the existing crate and grow it deliberately:

```text
crates/jackin-tui/src/
  tokens.rs          # Rgb, semantic tokens, spacing, glyphs, timing
  theme.rs           # Ratatui + ANSI adapters, terminal capability fallback
  geometry.rs        # centering, clamp, scrollbar, tab layout, text measurement
  runtime/
    message.rs       # shared event/message traits and common input events
    effect.rs        # typed side-effect vocabulary used by host runtime
    dirty.rs         # Dirty flags and redraw scheduler types
    subscription.rs  # ticks, terminal events, long-lived external streams
  components/
    panel.rs
    modal.rs
    hint_bar.rs
    status_footer.rs
    button_strip.rs
    text_input.rs
    filter_list.rs
    scrollable.rs
    tab_strip.rs
    confirm.rs
    error_dialog.rs
    file_browser.rs  # composed, only after primitives are stable
```

The `runtime/` module can start as small shared traits and types if moving the full host loop into the crate is too much at first. The important boundary is that product crates own domain data and effects, while `jackin-tui` owns reusable UI state machines, rendering, tokens, and layout math.

### Target ownership map

The refactor should leave a contributor-facing map like this. The exact filenames can evolve, but the ownership boundaries should not:

| Owner | Lives in | Owns | Must not own |
|---|---|---|---|
| App TUI session | `src/app/` plus shared runtime types in `jackin-tui` | terminal lifecycle, raw mode, alternate screen, mouse capture, diagnostics routing, suspend/resume, console → launch → attach handoff | product state, component drawing, Docker/git/1Password work inline |
| Host console app | `src/console/` | console domain model, workspace/editor/settings state, host-specific messages/effects, adapters from shared component outcomes to console actions | shared visual primitives, palette literals, blocking refresh work in render/update |
| Launch app state | `src/runtime/launch.rs` / `src/runtime/progress.rs` until folded into the app session | launch-stage domain state, launch-specific effects, forced choices, failure acknowledgement, build-log state | its own private TUI runtime, direct dependency on `src/console/widgets` |
| Capsule app | `crates/jackin-capsule/src/` | PTY/session model, mux mode, attach protocol, capsule-specific messages/effects, pane-body rendering decision | duplicated confirm/picker/tab/status implementations, raw ANSI chrome once Ratatui frame lands |
| Shared design system | `crates/jackin-tui/src/` | tokens, themes/adapters, geometry, reusable component state machines, Ratatui widgets, common runtime contracts, lookbook stories | host workspace semantics, container session lifecycle, direct terminal writes |
| Tests/lookbook | `tests/`, crate-local tests, generated docs assets | message/update tests, snapshot renders, component stories, Buffer/SVG generation, terminal handoff smoke notes | hand-drawn component previews that can drift from real widgets |

This map is the answer to "where does TUI code go?" New work under this roadmap should either move code toward one of these owners or explain why a temporary bridge is still needed.

### Component extraction order

1. **Freeze the component inventory.** Add a tracked table of components, owners, current call sites, states, and maturity level. New TUI work must name the component it uses or the reason it cannot.
2. **Promote tokens and geometry.** Move every inline RGB value, spacing constant, border glyph, scrollbar helper, tab layout helper, and centering helper behind `jackin-tui` tokens/adapters.
3. **Promote pure models.** `TextField`, `HintSpan`, `TabCell`, `ModalOutcome`, picker row IDs, button focus, and scroll state become reusable model types before renderers move.
4. **Promote primitives.** `Panel`, `ModalFrame`, `HintBar`, `StatusFooter`, `ButtonStrip`, `TextInput`, `FilterInput`, `ScrollablePanel`, `Scrollbar`, and `TabStrip` land in `jackin-tui` with `TestBackend` snapshots.
5. **Promote dialogs.** Rebuild confirm, error, text-input, filter-list, choice, save/discard, and status popups from primitives. Existing app-specific dialogs become thin props/state adapters.
6. **Consolidate launch.** Launch progress stops importing `src/console/widgets`; it consumes `jackin-tui` components and emits app messages/effects.
7. **Bridge capsule.** Capsule consumes shared tokens/models with raw-ANSI adapters until Phase 6 gives it a Ratatui frame.
8. **Replace capsule renderers.** Once capsule renders through Ratatui, delete raw dialog/picker/status renderer copies and consume canonical components.
9. **Generate lookbook.** Every canonical component gets stories, snapshots, generated SVG previews, and usage examples.

### Guardrails that make reuse enforceable

These are implementation rules for future PRs under this item:

- No new direct `Color::Rgb(...)` in product code. RGB values belong in `jackin-tui` tokens or a documented adapter test.
- No new raw ANSI color literals for jackin-owned chrome. Capsule may have an adapter, but it consumes shared `Rgb` tokens.
- No product screen hand-rolls a bordered modal, button row, footer hint line, tab strip, filter row, scrollbar, or selectable list once the matching component exists.
- No component invents its own focus-border semantics. The shared components own the "one active bright border" rule so host, launch, and capsule cannot drift.
- No new `src/runtime` dependency on `src/console/widgets`; both surfaces import `jackin-tui`.
- No component becomes canonical without keyboard/mouse tests, `TestBackend` snapshots, and a lookbook story.
- No app-specific component is promoted until its domain data is separated from UI state. Example: the 1Password picker can keep vault/item domain loading locally, but list selection, filter text, loading/error chrome, and button behavior must be shared.

## Scope & sequencing

This is a large item — eight phases across the host console, launch cockpit, capsule, session spine, and lookbook tooling. It is scoped as a single roadmap item for now, but if it grows it should graduate to a multi-leaf program like the [Codebase readability program](/reference/roadmap/codebase-readability/), with one leaf per track (host refactor, launch/session consolidation, capsule Ratatui, design system / lookbook).

Critical-path dependencies:

- **Phase 0 (snapshot tests) blocks every refactor phase** — without the regression net, no behavior-preserving claim is verifiable.
- The host track (Phases 1–4) and the capsule track (Phases 6–7) are largely independent and can proceed in parallel.
- The [file-split items](/reference/roadmap/split-input-editor/) land *against* the Phase 1/4 component seams, not before them — splitting first just produces smaller files that still mutate a god-struct.
- Phase 5 (session-spine consolidation) depends on Phase 1 (the `Message` loop) and starts from the `HostScreen` / launch-cockpit / `JACKIN_HOST_ALT_SCREEN` behavior shipped by [Launch Progress TUI](/reference/roadmap/launch-progress-tui/), not from a blank slate.
- Capsule Ratatui (Phase 6) blocks full widget sharing (Phase 7) and the lookbook's capsule coverage (Phase 8).

### Anti-patterns vs. Ratatui best practice

1. **God-structs that mix four concerns.** `ManagerState` (≈30 fields) holds domain data (`instances`, `workspaces`, `current_dir`), layout geometry (`cached_term_size`, `list_split_pct`, eight separate scroll-offset fields), navigation/focus state (`selected`, `expanded_workspaces`, `list_scroll_focus`, `preview_focused`), and infrastructure (`op_cache`, `instances_last_refresh`) in one struct. Adding one scrollable block touches the state struct, a render signature, the mouse handler, and a focus enum across four files with no single seam to audit.

2. **No `Action`/`Message` type.** Event handling mutates state directly inside the handler. `handle_key` in <RepoFile path="src/console/manager/input/mod.rs" /> is a ~240-line `if`-precedence chain that returns an `InputOutcome` only for routing decisions; all state changes are side effects before the return. Nothing is replayable, loggable, or testable without constructing a full `ManagerState` and driving a real terminal.

3. **The view is not pure.** Render functions take `&mut state`. `render_scrollable_block` clamps `scroll_x`/`scroll_y` in place every frame; the editor tab renderers write `tab_content_width`/`tab_content_height` back into `EditorState` so the mouse handler can later read them. "Draw" mutates observable state, which couples render output to input handling and is the root of a class of stale-geometry scroll bugs.

4. **Filesystem I/O and instance refresh on the render loop.** `mount_info::inspect` (<RepoFile path="src/console/manager/mount_info.rs" />) still runs from render/input paths and stats the disk / reads `.git` pointers for visible mounts. `refresh_instances` still does blocking index/manifest reads and a per-container socket fan-out on the event-loop thread (self-throttled to 500 ms, but still blocking the draw when it fires).

5. **Unconditional full-frame redraw.** The loop draws every tick regardless of whether anything changed. No dirty flag, no skipped frames.

6. **No `StatefulWidget`/`WidgetRef`.** Every renderer is a free `fn(frame, area, &mut state)`. The `unstable-widget-ref` feature is enabled in <RepoFile path="Cargo.toml" /> but unused — there are no reusable widget components implementing the Ratatui widget traits.

7. **Three coexisting modal-stack shapes.** A `Vec<Modal>` stack (correct), legacy stash slots (`pending_auth_form_return`, `pending_env_key`, …), and a bare `Option<Modal>` slot the [TUI Design Decisions](/reference/tui-design-decisions/) page already flags as an invalid shape that cannot represent a parent-under-child.

8. **DRY drift from symmetric variants.** `sidebar_inputs_for_workspace` vs. `sidebar_inputs_for_current_dir` in <RepoFile path="src/console/manager/render/list.rs" /> are near-identical bodies; the same workspace-vs-current-dir distinction is repeated in `focused_block_still_scrollable`, `clamp_list_scroll_for_area`, and the mouse `update_scroll_focus` handler — four call sites that must stay in lock-step.

9. **Component reuse is happening in the wrong layer.** The launch cockpit now reuses host console widgets such as `select_list`, `status_bar`, `hints`, `scrollable`, and `error_popup`, which is the right instinct. But the dependency direction is wrong: runtime launch code reaches sideways into `src/console/widgets` instead of every terminal surface consuming components from <RepoFile path="crates/jackin-tui/Cargo.toml" />. This is new evidence that `jackin-tui` must become the shared component crate, not just a token/helper crate.

## Current state — `jackin-capsule` multiplexer

The in-container control plane (the [jackin' Capsule](/reference/roadmap/jackin-capsule/) program) is a small tmux-like PTY multiplexer. It uses `crossterm` only for `enable_raw_mode`/`disable_raw_mode` and `terminal::size()` in <RepoFile path="crates/jackin-capsule/src/client.rs" />. **It does not depend on Ratatui.**

### Anti-patterns

1. **All rendering is hand-rolled raw ANSI** into a `Vec<u8>` — direct `write!(buf, "\x1b[{};{}H", …)` cursor moves and inline SGR byte sequences in <RepoFile path="crates/jackin-capsule/src/render.rs" />, <RepoFile path="crates/jackin-capsule/src/dialog.rs" />, and the status bar. There is a hand-rolled SGR serializer and a hand-rolled CSI re-serializer (the latter untested) in <RepoFile path="crates/jackin-capsule/src/session.rs" />.

2. **A hand-rolled row diff re-implements `Buffer::diff`.** `PaneBodyCache` builds a `Vec<RowSnapshot>` from the `vt100::Screen`, compares it cell-by-cell against the cached previous snapshot, and emits only changed rows. This is exactly the job Ratatui's `Buffer` double-buffer does — re-built by hand.

3. **A ~6,900-line god-struct.** `Multiplexer` in <RepoFile path="crates/jackin-capsule/src/daemon.rs" /> owns rendering, input parsing, network I/O, git/PR context polling, session lifecycle, the dialog stack, and mouse drag/selection in one file and one compilation unit.

4. **No top-level mode state machine.** "Which mode is the multiplexer in?" is answered by combining scattered booleans (`dialog_open()`, `drag.is_some()`, `selection.is_some()`, `input_parser.is_awaiting_prefix()`) at each call site. There is no `MuxMode` enum making invalid combinations unrepresentable — and at least one arm checks `dialog_open()` but not `drag.is_some()`, a latent bug class.

5. **`compose_full_frame` vs. `compose_partial_frame` duplication** — ~160 lines of near-duplicate pane-loop / box-draw / scrollbar code that must be kept symmetric by hand.

6. **No tests** for `handle_input` dispatch, `apply_dialog_action`, the dialog `handle_key`/`handle_click`, the drag/selection state machines, or the compositor. The parser, protocol, and `render_pane` are tested in isolation; the daemon's decision layer is not.

<Aside type="note">
The capsule already has one piece of the target in place: `Dialog::handle_key`/`handle_click` return a `DialogAction` enum (`Command`, `SpawnAgent`, `RenameTab`, `Dismiss`, …) rather than mutating state directly. That is exactly the `Action` pattern this item generalizes to the whole dispatch layer.
</Aside>

## Current state — the console → launch → container flow

The three surfaces are still built differently, but #452 removed the biggest product seam for the console-triggered launch path. `jackin console` now creates a `HostScreen` in <RepoFile path="src/app/mod.rs" /> before entering `run_console`; `run_console` detects that an outer guard owns the terminal and draws into it; `runtime::LaunchProgress` draws the cockpit into the same alternate screen; and `runtime::attach` passes `JACKIN_HOST_ALT_SCREEN=1` into the capsule client so the child skips its own alternate-screen enter/leave. The remaining problem is no longer the old plain-output launch premise; it is that the shipped continuity is implemented as several local contracts and globals instead of one TUI application architecture.

This is a real improvement and should be preserved. The architectural debt is what the improvement exposed:

- `HostScreen` is a terminal-lifecycle guard, not an app session model. It coordinates sub-surfaces through `crate::tui::host_screen_owned()` and `set_rich_surface_active()` globals instead of typed ownership.
- The launch cockpit has its own `LaunchView`, renderer task, picker loop, input drain, and failure modal rather than sharing the console's eventual `Message` / `update` / `Effect` loop.
- The launch cockpit imports reusable pieces from `src/console/widgets`, so the first cross-surface reuse exists but lives below the wrong module boundary.
- `jackin load` still creates the launch cockpit as a standalone rich surface; only the console path has the outer `HostScreen` that spans console → launch → attach.
- The capsule render path is still independent raw ANSI. The host can preserve a single alt-screen across handoff, but the in-container UI is not yet part of the shared Ratatui component system.

## Ratatui application patterns (the target)

The Ratatui project documents three application patterns and recommends one for immediate-mode apps:

- **The Elm Architecture (TEA)** — [Model / Update / View](https://ratatui.rs/concepts/application-patterns/the-elm-architecture/). State lives in a Model; events map to a `Message` enum; an `update` function applies a message to the model; a **pure** `view` function renders the model. Ratatui calls TEA "the best architectural approach for immediate-mode rendering."
- **[Component Architecture](https://ratatui.rs/concepts/application-patterns/component-architecture/)** — components own their local state and event handlers. The [recommended project structure](https://ratatui.rs/templates/component/project-structure/) is `app.rs` · `action.rs` · `event.rs` · `tui.rs` · `components/`, with the guidance that "you shouldn't need to change anything outside the `components` folder."
- **[Flux](https://ratatui.rs/concepts/application-patterns/flux-architecture/)** — unidirectional data flow; an alternative to TEA for larger apps.

Supporting practices:

- **`StatefulWidget` / `WidgetRef`** separate a widget's drawing logic from a persistent `State` object — the idiomatic way to build reusable, testable components.
- **Immediate-mode discipline**: the view is a pure function of state, redrawn each frame; mutation and I/O belong in `update`, never in `view`.
- **I/O off the render/update thread**: long-running work (disk, network, subprocess) runs in background tasks that send `Message`s back to the loop, so the draw never blocks.
- For PTY rendering specifically, [`tui-term`](https://github.com/a-kenji/tui-term) renders a `vt100::Screen` into a Ratatui `Buffer` (the same approach [used by Turborepo](https://github.com/vercel/turborepo/pull/9123) and [Ratty](https://blog.orhun.dev/introducing-ratty/)) — which means the capsule *could* keep its `vt100` model while letting Ratatui own the buffer diff and the chrome.

<Aside type="caution">
Dogmatic TEA clones the whole model on every update. jackin's state is large, so a clone-per-keystroke model is impractical. The realistic target is Ratatui's documented **mutable** variant: `fn update(&mut self, msg: Message)` mutating the model in place, paired with a strictly **pure** `fn view(&self, frame)`. The win is the `Message` boundary and the pure view, not literal immutability.
</Aside>

## Additional research pass — stronger architecture ideas

The post-#452 re-analysis should not stop at "move files into components." A broader research pass across Ratatui, Bubble Tea, Textual, `tui-realm`, and `tui-term` suggests a better target: **a small jackin-owned TUI runtime**, not a generic framework dependency and not a loose convention.

### Source ideas worth adopting

| Source | Useful idea | jackin' adaptation |
|---|---|---|
| [Ratatui application patterns](https://ratatui.rs/concepts/application-patterns/) | Ratatui explicitly frames TEA, component architecture, and Flux as app-level patterns rather than renderer features. | Treat Ratatui as the renderer; jackin' owns the app runtime around it. |
| [Ratatui FAQ — async](https://ratatui.rs/faq/#when-should-i-use-tokio-and-asyncawait) | `terminal.draw` is the only Ratatui call that must touch the terminal; async mainly matters for input and app work. | Enforce one draw owner; all Docker, 1Password, filesystem, capsule, and diagnostics work is an effect that returns a message. |
| [Ratatui component template](https://ratatui.rs/templates/component/project-structure/) | `action.rs`, `app.rs`, `tui.rs`, and `components/` keep update, terminal lifecycle, and components apart. | Use the same separation, but adapted to the existing crate layout instead of importing a template wholesale. |
| [Bubble Tea](https://github.com/charmbracelet/bubbletea) | `Model.Init`, `Model.Update`, `Model.View`, plus `Cmd` for async side effects, gives a concise runtime contract; Bubbles separates shared components from apps. | Add typed `Effect`s returned by `update`; put reusable UI in `jackin-tui`, with app-specific state in host/capsule modules. |
| [Textual reactivity](https://textual.textualize.io/guide/reactivity/) | Smart refresh: state changes mark widgets dirty and coalesce refreshes; watch hooks react to changes. | Do not add magic descriptors in Rust; do add explicit invalidation: messages declare `Dirty::Layout`, `Dirty::View(ComponentId)`, or `Dirty::Animation`. |
| [`tui-realm`](https://github.com/veeso/tui-realm) | A React/Elm-inspired ratatui framework proves the shape is viable: components, event updates, declarative rendering. | Use as a design reference, not a dependency: jackin' needs container attach, diagnostics, and host-suspend effects that are too project-specific. |
| [`tui-term`](https://docs.rs/tui-term/) | A Ratatui pseudo-terminal widget already maps `vt100::Screen` into Ratatui. | Keep it as the benchmark baseline for capsule pane bodies; decide with throughput and allocation measurements. |

### The higher bar: `TuiRuntime`, not just `Message`

The roadmap should target an internal runtime with these contracts:

1. **One draw owner.** A `TuiRuntime` owns the terminal backend, raw mode, alternate screen, mouse capture, diagnostics routing, and `terminal.draw`. No component, launch step, or capsule adapter may call `draw` directly.
2. **Messages are facts, not commands.** `Message` values describe what happened (`KeyPressed`, `StageDone`, `InstancesRefreshed`, `CapsuleDetached`, `BuildLogClicked`). They do not perform I/O.
3. **Effects are the only side-effect door.** `update(&mut Model, Message) -> Effects` returns typed effects such as `DockerBuild`, `OpLoad`, `InspectMounts`, `SuspendForPrompt`, `AttachCapsule`, `WriteDiagnostics`, `SetTerminalTitle`, `ClipboardCopy`, `TickAfter`, or `Quit`. Effects run outside the draw/update path and return messages.
4. **Subscriptions own long-lived inputs.** Keyboard/mouse/focus/resize events, animation ticks, build-log updates, capsule socket events, Docker instance refresh, and diagnostics lifecycle updates are `Subscription`s. The runtime starts/stops them based on the active view and component tree.
5. **Dirty state is explicit.** Every message/update marks what changed. The runtime coalesces dirty flags and redraws only when state, layout, or animation actually changed. Animations register their own tick subscription; idle views do not redraw at 20 Hz.
6. **Components are pure renderers plus local update.** A component owns its local UI state, handles component-scoped messages, emits effects, and renders from immutable input. Cross-component mutation happens only through parent messages.
7. **Terminal suspension is an effect.** Cooked prompts, token generation, external editors, and `docker exec -it` attach are `SuspendTerminal` / `AttachTerminal` effects with explicit restore semantics, not inline calls sprinkled through event loops.
8. **Diagnostics are a runtime concern.** Since a rich surface owns the terminal, debug output routes through the active diagnostics run from the runtime. Components emit structured diagnostics events; they never print.
9. **Replay is possible by construction.** A recorded sequence of messages plus effect results should replay the state machine in a headless `TestBackend`. This unlocks regression tests for console → launch → attach transitions that screenshots alone cannot cover.

### Why not adopt a framework?

The external frameworks validate the pattern, but jackin' should not depend on one wholesale:

- The host flow has project-specific effects: sensitive host prompts, container attach, diagnostics run files, Capsule socket events, host-alt-screen inheritance, and no-debug-spew guarantees.
- The capsule needs a custom backend that serializes Ratatui buffer diffs to an attach socket, not a normal local terminal.
- The shared `jackin-tui` crate already exists and already owns tokens and pure layout helpers. The right move is to grow it into the project-specific runtime/component layer rather than route through a generic framework and then fight its assumptions.

The implementation can still borrow proven vocabulary: Bubble Tea's `Cmd` becomes jackin's typed `Effect`; Textual's smart refresh becomes explicit dirty invalidation; Ratatui's component template supplies the file boundary; `tui-term` supplies the pane-body benchmark baseline.

### Async architecture contract

"Async-first" does not mean every function becomes `async fn`. It means the render/update path never waits for unbounded work, and the terminal has one clear draw owner. The implementation contract:

- **Only the runtime draws.** `terminal.draw` and Ratatui buffer flushing happen in the app runtime / capsule backend, never inside components, background tasks, launch steps, or domain helpers.
- **Update is fast and bounded.** `update(&mut model, msg) -> Vec<Effect>` may mutate memory and enqueue effects; it must not call Docker, git, 1Password, the filesystem for unbounded inspection, the capsule socket, network APIs, or `thread::sleep`.
- **Effects own work.** Blocking subprocesses, filesystem walks, Docker CLI calls, `op` calls, GitHub/PR context loading, diagnostics writes, and retry loops run as effects. Async-capable work uses `tokio::spawn`; CPU-bound or blocking library work uses `spawn_blocking`; already-threaded capsule PTY work reports back through the same message boundary.
- **Subscriptions own streams.** Terminal input, resize/focus, animation ticks, launch logs, instance refresh cadence, capsule protocol events, and diagnostics lifecycle updates are subscriptions that send messages. They can be started/stopped based on the active view.
- **Cancellation is explicit.** Effects that outlive the view that requested them carry a generation id, request id, or cancellation handle so stale results cannot mutate a newer screen.
- **Backpressure is deliberate.** High-volume streams (PTY output, build logs, animation ticks) coalesce or drop superseded redraw messages instead of growing an unbounded queue.
- **Locks never cross await points.** Shared view snapshots may use short-lived locks, but no update/effect path may hold a mutex while awaiting Docker, socket, filesystem, or channel work.
- **Errors are messages.** Background failures return typed messages and render through the normal error component. They do not panic, print over a rich surface, or disappear into logs only.

The practical test is simple: while a long image build, instance refresh, git/PR context load, or 1Password lookup is slow, key input still queues, animations still tick, diagnostics still record, and the terminal can still redraw or show cancellation/error state.

## Blocking-work inventory — async migration targets

The async-first principle becomes actionable as a concrete list. Every site below either blocks the render/update loop today or sleeps synchronously on a render-adjacent path; the migration target for each is the same — move it off the loop so the draw keeps ticking. Phases 2, 3, and 5 are where these land.

**Host console — work on the event-loop thread:**

| Site | What blocks | Target |
|---|---|---|
| `mount_info::inspect` (<RepoFile path="src/console/manager/mount_info.rs" />) | per-frame `stat()` + `.git` reads for every mount, at 20 Hz, uncached | cached `MountFacts`, refreshed by a background `MountInspected` message |
| `refresh_instances` (<RepoFile path="src/console/manager/state.rs" />) | blocking index/manifest reads + per-container socket fan-out (self-throttled to 500 ms, still blocks the draw when it fires) | background task → `InstancesRefreshed` message |
| 1Password loads / snapshot fan-out | synchronous `OpRunner` + socket calls invoked from input handlers | background task → result message |
| `event::poll(50 ms)` loop | wakes 20×/s even when idle; full-frame redraw every tick | `tokio::select!` over an `EventStream` + the `Message` channel; dirty/animation-tick redraw |

**Launch flow — the cockpit render cadence:**

The launch cockpit now has a background render task in <RepoFile path="src/runtime/progress.rs" />. It ticks every ~33 ms, snapshots a shared `LaunchView`, drains cockpit-local input, and draws the Ratatui frame while <RepoFile path="src/runtime/launch.rs" /> awaits Docker/network/workspace futures. `LaunchProgress::while_waiting` is now deliberately a thin await wrapper because the renderer no longer needs each launch step to interleave drawing manually. That means the old "wrap every step in a select loop" plan is stale.

The remaining migration target is different: fold the cockpit's parallel mini-app into the same app-level message/effect architecture as the console. Launch stage mutations become messages; forced-choice pickers and failure acknowledgement become effects/input states; the background render task becomes the shared animation-tick source instead of a launch-only thread. Any synchronous work that still runs before `LaunchProgress::new`, between stage mutations, or during rich-surface input handling must move behind effects so the cockpit and future shared session stay responsive.

**Synchronous `std::thread::sleep` on render-adjacent paths** — each should become `tokio::time::sleep`, an interval tick, or move off the render task:

| Site | Sleep | Note |
|---|---|---|
| <RepoFile path="src/tui/output.rs" /> | 25 ms shimmer · 1500 ms deploy hold | animation/pacing on the launch path — drive from an interval, not a blocking sleep |
| <RepoFile path="src/runtime/snapshot.rs" /> | 20 ms · 10 ms | retry pacing |
| <RepoFile path="src/runtime/logs.rs" /> | 250 ms | poll pacing |
| <RepoFile path="src/console/widgets/op_picker/mod.rs" /> | 2 ms ×2 | worker poll pacing |

**Capsule:** PTY and git/PR context already run on background tasks; the gap is the `std::thread` vs. `spawn_blocking` inconsistency the gap-analysis table notes, plus the compose path running on the loop. The Ratatui-frame move (Phase 6) puts the buffer diff on the framework and the compose work behind the same `Action`/update boundary.

## Target architecture — host operator console

Introduce the Model/Update/View seam without changing behavior:

1. **One dispatch type + central `update`.** Define the dispatch `enum` — `Message` on the host, the generalized `Action` on the capsule, **converged on a single name** so the shared vocabulary stays recognizable — covering input intents (`MoveSelection`, `OpenModal`, `CommitSave`, `LaunchWorkspace`, …), background results, and their failures (`InstancesRefreshed`, `MountInspected`, `OpLoadFailed(e)`, …) so errors flow as messages routed to the `ErrorPopup`, never as `unwrap`/panic. Map every `crossterm` event — key, mouse, **paste, focus, and resize** — to messages in a thin layer; route every state change through `fn update(&mut self, msg: Message) -> Vec<Effect>` — one update may emit several effects (redraw, spawn a task, quit), so it returns a list, not a single `Option`. Handlers stop mutating state directly.

2. **Decompose state.** Split the god-structs along the seams that already exist conceptually: a domain `Model` (workspaces, instances, pending config), a `UiState` (selection, focus, scroll offsets, open modal), and ephemeral per-frame scratch. Geometry that the renderer derives (content width/height) becomes computed-and-cached state owned by `update`, not written from inside `view`.

3. **Pure view.** Convert renderers to `fn(frame, area, &Model, &UiState)` with no `&mut` and no I/O. Move scroll clamping into `update` (on resize / scroll messages). Replace per-frame `mount_info::inspect` with a cached `MountFacts` map refreshed by a background `MountInspected` message.

4. **Background I/O → channel.** Move instance refresh, 1Password loads, snapshot fan-out, and mount inspection onto background tasks that send `Message`s. `run_console` is already an `async fn`, so the target loop is a `tokio::select!` over a crossterm `EventStream` and the background `Message` channel — replacing today's synchronous `event::poll(50 ms)` — so the draw never blocks on disk/socket work and no longer wakes 20×/s just to spin. (Fallback if `EventStream` proves awkward: a dedicated input thread feeding the same `Message` channel via `std::sync::mpsc`.)

5. **`StatefulWidget` components.** Extract the list, the editor (per tab), settings, and the modal family into components implementing the Ratatui widget traits, each with its own `State`. This is where the [Split `input/editor.rs`](/reference/roadmap/split-input-editor/) and [Split `app/mod.rs`](/reference/roadmap/split-app-mod/) work should land — the splits target component boundaries, not arbitrary smaller files.

6. **Converge the modal stack** onto the single `Vec<Modal>` shape and retire the stash slots and bare `Option<Modal>`, satisfying the existing [TUI Design Decisions](/reference/tui-design-decisions/) sub-dialog-stack rule by construction.

Every existing rule in [TUI Design Decisions](/reference/tui-design-decisions/) (W3C ARIA tabs, the focusability/affordance contract, the shared `render_scrollable_block`, modal backdrop dimming, the PHOSPHOR palette, settings↔editor parity) is preserved — the refactor changes the *structure behind* those rules, never the rules themselves.

## Target architecture — `jackin-capsule`

Two questions are independent and should be decided separately:

**(1) Structural cleanup — clear win, do regardless of rendering backend.**

- Add a `MuxMode` enum (`Normal`, `PrefixAwait`, `Dialog`, `Drag`, `Select`) so invalid combinations are unrepresentable and dispatch reads from one place.
- Generalize the existing `DialogAction` into a single `Action` type for the whole `handle_input` dispatch — the *same dispatch type the host converges on* (see the host target) — so the daemon's decision layer is testable without a live PTY.
- Split <RepoFile path="crates/jackin-capsule/src/daemon.rs" /> into modules: event loop, session lifecycle, render/compose, input dispatch, dialog. Merge `compose_full_frame`/`compose_partial_frame` onto a shared pane-composite helper.
- Add dispatch/dialog/drag tests once the `Action` boundary exists.

**(2) Render through Ratatui — the decided direction.**

The capsule moves to **full Ratatui**: the daemon builds frames in a Ratatui `Buffer` and renders chrome, status bar, tab strip, dialogs, pickers, and confirms as Ratatui widgets — the *same* shared widgets the host console uses (see the shared design system below). The attach client in <RepoFile path="crates/jackin-capsule/src/client.rs" /> stays a thin byte relay; Ratatui renders inside the **daemon** via a custom backend that serializes the `Buffer` diff to the attach socket instead of a real TTY, replacing the hand-rolled `PaneBodyCache` row diff with Ratatui's own `Buffer::diff`.

The one open question is the **pane-body** mechanism — the cells that mirror a child program's `vt100::Screen`, the perf-critical hot path:

| Pane-body option | Shape | Trade-off |
|---|---|---|
| `tui-term` | render the `vt100::Screen` into the Ratatui `Buffer` via the pseudo-terminal widget | maximal reuse of Ratatui's buffer/diff; `tui-term` is self-described "work in progress" with a known per-cell allocation cost on high-throughput PTYs |
| thin custom cell widget | a small `Widget` that blits the `vt100::Screen` into the `Buffer` directly | keeps the proven fast path while still living inside the Ratatui frame; a little custom code, no new dependency risk |

**Resolve the pane-body question with a throughput spike** (a later phase) — both options keep the capsule fully inside a Ratatui frame, so chrome / dialog / picker reuse lands regardless. That benchmark is the only part of the capsule that stays a measurement question; everything else is decided. The shared widgets live in the existing <RepoFile path="crates/jackin-tui/Cargo.toml" /> crate (today Ratatui-free), which grows a Ratatui dependency to host them.

## Target architecture — one owned terminal session

The first version of this exists now, so the target is no longer "move ownership up" in the abstract. The target is to formalize the shipped `HostScreen` / launch renderer / capsule handoff into one app-level TUI session spine:

- Keep the #452 behavior: the console manager renders into an outer host screen, the launch cockpit draws into that same screen, and the capsule handoff preserves the host alternate screen with `JACKIN_HOST_ALT_SCREEN` so the child does not enter a nested alt-screen.
- Replace the global coordination flags (`host_screen_owned`, `rich_surface_active`) with typed session ownership: an `AppTuiSession`/`TerminalSession` object owns raw mode, alt-screen, mouse capture, debug buffering, rich-surface diagnostics routing, suspend/resume, and the handoff contract.
- Make console and launch views siblings in one app model. "Operator picked a workspace" → "watching the launch cockpit" becomes a `Message`/`Effect` transition, not `run_console` returning an outcome to a separate launch mini-app.
- Convert launch progress updates (`stage_started`, `stage_done`, failures, build-log overlay, forced-choice picker, failure acknowledgement) into the same message/effect vocabulary as the console. The existing background render task is preserved conceptually as the animation tick source, but it is not launch-private.
- Move reusable launch widgets out of `src/console/widgets` and into <RepoFile path="crates/jackin-tui/Cargo.toml" />. Runtime code must not depend on console internals just to render a status bar, hint row, select list, scrollable build log, or error popup.
- Keep the rare cooked prompts as explicit `SuspendForPrompt` effects. The current `HostScreen::suspend` behavior is useful, but the reason for leaving the rich surface should be typed and visible in the update loop.
- Preserve panic-safe restore. The RAII guard remains the ultimate owner of terminal restoration; the refactor changes where ownership is modeled, not the guarantee that raw mode and alternate screen are restored on every exit path.

This item now owns the consolidation of the shipped backbone. The [Launch Progress TUI](/reference/roadmap/launch-progress-tui/) item owns the operator-facing launch cockpit and diagnostics behavior that already landed; this item owns making that behavior a reusable architecture rather than a special case.

## Shared design system & lookbook (`jackin-tui`)

A shared crate already exists: <RepoFile path="crates/jackin-tui/Cargo.toml" />. Today it is a **Ratatui-free crate of raw-ANSI primitives and pure helpers** — the PHOSPHOR palette as `Rgb`, an `ansi` module (`fg` / `bg` / `move_to`), `TextField` (a text-input model), `HintSpan` + `hint_row_cols`, `TabCell` + `lay_out_tabs`, `vertical_thumb`, and unicode-width helpers. The capsule consumes it directly; the host console consumes only the **token layer** — <RepoFile path="src/console/widgets/mod.rs" /> wraps `jackin-tui`'s `Rgb` palette values into `ratatui::Color`.

That token layer is the model to follow: one source of truth for the values, a thin per-surface conversion. The gap is that **every component above the token layer is reimplemented per surface or borrowed through the wrong module boundary** — the host has its own tab strip, scrollbar, text input, hint model, and confirm dialog; the launch cockpit imports host widgets directly from `src/console/widgets`; and the capsule hand-rolls each again in raw ANSI, kept in sync only by the procedural rules in [TUI Design Decisions](/reference/tui-design-decisions/). **Target: elevate `jackin-tui` into the canonical TUI design-component crate** every terminal surface consumes; the crate that already owns the tokens grows to own the components. Do not create a new crate.

This is a design-system problem, and the web ecosystem has a mature playbook for it. An audit found **roughly 45 repeatable visual patterns** — an approximate figure; the enumerated drift table below is the verifiable subset — about a dozen implemented twice and several with measurable cross-surface drift. The fix is a single token set, a single component library, and a browsable **lookbook** that renders every component and its states so the operator can point at one and say "use this." The canonical *rules* stay in [TUI Design Decisions](/reference/tui-design-decisions/); the lookbook renders the *components* and never restates a rule — its Do / Don't rows link to the relevant rule anchor. The catalog below, organized by atomic-design layer, is the seed inventory; each row becomes a lookbook page.

### The web playbook, mapped to a TUI

| Web concept | What it is | jackin' equivalent |
|---|---|---|
| [Design tokens](https://dev.to/psqrrl/managing-design-tokens-using-storybook-5975) | the smallest brand primitives (color, spacing, timing) in a tech-independent source | the `Rgb` palette + spacing/timing constants in <RepoFile path="crates/jackin-tui/src/lib.rs" /> |

### How deep the sharing goes depends on the capsule's rendering decision

- **Tokens and pure layout math** (palette, `lay_out_tabs`, `vertical_thumb`, text measurement, `box_rect` centering) can be shared **today**, regardless of the capsule's pane-body rendering decision. This is the floor — already partly in place — and unifying the remaining token wrappers and layout helpers is an early, low-risk win that needs no rendering-backend change.
- **Full `StatefulWidget` components** (`Panel`, `ScrollableBlock`, `ConfirmDialog`, `FilterList`, `TabStrip`, `HintBar`) require the capsule to render through a Ratatui frame (Phase 6). Until that lands, the capsule shares only tokens and layout math and keeps separate render code — the weakest form of reuse, which is why moving the capsule into a Ratatui frame is committed scope here, not an optional follow-up.

### What each lookbook page contains

Each component gets one page with a fixed "story" shape so the operator reads them all the same way:

1. **Name + one-line purpose** — what it represents and when it appears.
2. **Anatomy** — a labelled diagram of the rendered component (border, title, content, scrollbar, buttons).
3. **Tokens used** — which palette / spacing tokens, linked to the token page.
4. **States / variants gallery** — focused / unfocused / scrollable-fit / error / default-focus / hover, each rendered.
5. **Where used** — the screens that compose it, so a change's blast radius is visible.
6. **Do / Don't** — links to the [TUI Design Decisions](/reference/tui-design-decisions/) rule anchors that apply (never restated inline).
7. **Live preview** — the actual rendered component (mechanism below).

### Rendering TUI components in the docs

The lookbook is only trustworthy if its previews are the *real* components, not hand-drawn art that drifts. The plan, strongest first:

- **Primary — render the real widget to SVG.** Draw each component (and each state) through `ratatui::backend::TestBackend` into a `Buffer`, then serialize the buffer's cells (char + fg/bg/modifiers) to inline **SVG** (or styled HTML). A small `cargo run --bin tui-lookbook` emits one SVG per component-state from the actual widget code; MDX embeds the generated SVGs; CI regenerates them so they never drift. This reuses the exact `TestBackend` + `insta` infrastructure from [Snapshot tests for TUI render](/reference/roadmap/snapshot-tests-tui/) — the same render that guards regressions also produces the lookbook art. [`term-transcript`](https://docs.rs/term-transcript) (Rust; renders terminal output to SVG/HTML) is a candidate serializer.
- **Motion and raw-ANSI surfaces** — for the digital rain, spinners, and (until they move to Ratatui) the capsule's raw-ANSI dialogs, record an asciicast and convert with [`vhs`](https://github.com/charmbracelet/vhs), [`termsvg`](https://github.com/MrMarble/termsvg), [`svg-term`](https://github.com/marionebl/svg-term-cli), or [`termtosvg`](https://nbedos.github.io/termtosvg/) to an animated SVG embeddable in MDX.
- **Interim bridge** — a small MDX `<TuiPreview>` component that paints hand-authored Unicode mockups using the docs theme's existing `--jk-brand` PHOSPHOR CSS tokens. Fast to write but hand-drawn and drift-prone, so it is a bridge to the generated-SVG approach, not the destination.

### The component selector (end state)

The lookbook is the operator's component catalog. The intended flow: the operator browses it, points at a component — "use `ConfirmDialog` here", "use `FilterList` with section headers", "this should be a `Panel`, not a bare block" — and the agent composes the screen from the shared `jackin-tui` widgets instead of hand-rolling a near-copy. That is the Storybook "pick the component" experience brought to jackin's TUI: a named, visible, reusable vocabulary instead of ~45 patterns reinvented per surface.

## Terminal capability & degradation

A single token layer makes graceful degradation a property of the tokens, not of every call site:

- **Color depth.** Tokens resolve through Ratatui's color handling for truecolor; on 256- or 16-color terminals the palette maps to the nearest supported color, and `NO_COLOR` / `COLORTERM` are honored so the UI stays legible without a redesign.
- **Glyph fallback.** The box-drawing, scrollbar, and selection glyphs (`┌─┐│└┘`, `█`, `·`, `▸`, `░`) get an ASCII fallback set for terminals that cannot render them, selected once at startup rather than per call site.
- **Size.** Components clamp to a minimum usable size and degrade (hide chrome, collapse columns) on narrow terminals instead of panicking or overflowing.
- **Reduced motion.** Animations (digital rain, spinners, shimmer) honor the reduced-motion handling owned by [Launch Progress TUI](/reference/roadmap/launch-progress-tui/); the design system exposes the same gate so static fallbacks stay consistent.

**Non-goals (this refactor).** Screen-reader / assistive-technology support and alternate themes (light, high-contrast) are out of scope — but the single token set makes both a future *configuration* change rather than a rewrite, which is the point of centralizing them.

## Constraints and non-goals

- **No behavior change.** Snapshot tests gate every step; the operator must not see a difference. This is a structure refactor, not a redesign — visual/UX changes are out of scope and belong to items like [Launch Progress TUI](/reference/roadmap/launch-progress-tui/).
- **Pre-release freedom.** Internal types may change freely; no migration shims or compatibility layers (these are not versioned schemas).
- **Incremental only.** No big-bang rewrite. One surface, one layer, behind tests. A half-migrated state (some handlers on `Message`, some not) must compile and pass tests at every commit.
- **DRY.** Extend the shared scrollable/widget helpers and the `jackin-tui` crate; do not fork parallel implementations.

## Implementation phases

### Phase 0 — Safety net and decision record

Land the [Snapshot tests for TUI render](/reference/roadmap/snapshot-tests-tui/) item (the regression net) and extend [ADR-003 "ratatui selection"](/reference/roadmap/architecture-decision-records/) to record the capsule's current no-Ratatui status and the pane-body rendering decision. This page is the design artifact.

### Phase 1 — Host: `Message` boundary

Introduce the `Message` enum and central `update`; route existing key/mouse handlers through it without moving the renderers. Behavior identical; the win is a testable update layer.

### Phase 2 — Host: pure view

Remove `&mut` and I/O from renderers. Move scroll clamping into `update`. Replace per-frame `mount_info::inspect` with cached facts refreshed by a message. Introduce event-driven (dirty) redraw — draw on a `Message` or an animation tick, not unconditionally every 50 ms — so an idle console does no work.

### Phase 3 — Host: background I/O

Move instance refresh, 1Password loads, snapshot fan-out, and mount inspection onto background tasks that send `Message`s. The draw never blocks.

### Phase 4 — Host: components + modal-stack convergence

Extract `StatefulWidget` components (list, editor-per-tab, settings, modals) into the shared <RepoFile path="crates/jackin-tui/Cargo.toml" /> design crate; land the [Split `input/editor.rs`](/reference/roadmap/split-input-editor/) and [Split `app/mod.rs`](/reference/roadmap/split-app-mod/) work against these seams; converge the modal stack onto `Vec<Modal>`. Token and layout-helper unification (the option-independent floor) can land earlier, in Phase 1.

### Phase 5 — App: one owned terminal session

Consolidate the `HostScreen` behavior that [Launch Progress TUI](/reference/roadmap/launch-progress-tui/) shipped into a typed app-level `TerminalSession`. The console manager and launch cockpit already share one host alternate screen on the console path; this phase removes the ad hoc globals and return-to-app handoff, models console and launch as sibling views in one app model, makes launch updates messages/effects, and moves the launch-used widgets from `src/console/widgets` into <RepoFile path="crates/jackin-tui/Cargo.toml" />. The visible behavior stays the same: console → launch → capsule remains continuous, with cooked prompts represented as explicit suspend effects.

### Phase 6 — Capsule: structural cleanup + Ratatui frame

Add `MuxMode`; generalize `Action`; split <RepoFile path="crates/jackin-capsule/src/daemon.rs" />; merge the full/partial compositors; add dispatch/dialog tests. Then move the daemon to render into a Ratatui `Buffer` via a custom attach-socket backend, replacing `PaneBodyCache` with `Buffer::diff`, and render chrome / dialogs / pickers as the shared <RepoFile path="crates/jackin-tui/Cargo.toml" /> widgets.

### Phase 7 — Capsule: pane-body spike + final reuse

Benchmark `tui-term` against a thin custom cell widget for pane bodies on a high-throughput PTY and pick the mechanism. Land the shared confirm / picker / tab-strip / panel / hint-bar widgets so the host console and the capsule render identical components.

### Phase 8 — Design-system lookbook

Stand up the lookbook: a `tui-lookbook` binary that renders each `jackin-tui` component and its states to SVG from the real widget code, one MDX page per component following the story shape above, and CI regeneration so previews never drift. Reuses the [Snapshot tests for TUI render](/reference/roadmap/snapshot-tests-tui/) `TestBackend` infrastructure. Reconcile the host/capsule drift table as components consolidate into shared widgets.

## Open questions & risks

- **Handoff feasibility.** The single-host-alt-screen approach (the host keeps its alt-screen across `docker exec -it`; the capsule client skips its own `?1049` toggle via `JACKIN_HOST_ALT_SCREEN`) is implemented by the [Launch Progress TUI](/reference/roadmap/launch-progress-tui/) work and must keep being confirmed across real terminals — alt-screen survival, raw mode, SIGWINCH, and the stale `Buffer` full-redraw on resume. If a terminal mishandles the shared screen, the fallback is to tear the host backend down before the exec and re-initialize on return, with the flash minimized.
- **Global session flags.** `host_screen_owned` and `rich_surface_active` encode real product state but are not a durable architecture. Phase 5 must replace them with typed ownership/effects without regressing diagnostics routing, compact fallback output, or panic-safe terminal restore.
- **Async loop shape.** `EventStream` + `tokio::select!` vs. a dedicated input thread + `std::sync::mpsc`. Decide in Phase 1; the `Message` boundary keeps the rest of the design independent of the choice.
- **Pane-body rendering cost.** `tui-term`'s per-cell copy on high-throughput PTYs vs. a thin custom cell widget — the Phase 7 benchmark.
- **`ratatui-textarea` vs. `TextField`.** *Decision:* consolidate the host text input onto the in-house `TextField` (already shared via `jackin-tui`) and drop the third-party, pre-1.0 `ratatui-textarea`, so there is one text-input model. Revisit only if `TextField` lacks needed editing features.
- **`unstable-widget-ref`.** Enabled but unused today. *Decision:* build components as `StatefulWidget` and drop the unstable feature unless `WidgetRef` proves genuinely necessary, to avoid depending on an unstable API.
- **Lookbook SVG toolchain.** `term-transcript` vs. a custom `Buffer`→SVG serializer for widgets, and `vhs` / `termsvg` for motion — pick in Phase 8.
- **Dirty redraw vs. animation.** Event-driven redraw must still drive spinner / rain animation ticks, so the redraw scheduler needs an animation-timer source, not only input `Message`s.

## Acceptance criteria

- Host event handling routes through a `Message` enum and a central `update`; no renderer takes `&mut state` or performs disk/socket I/O.
- Per-frame filesystem inspection is eliminated; mount facts and instance state are refreshed by background tasks that message the loop.
- Reusable screens are `StatefulWidget` components; the modal stack is a single `Vec<Modal>` shape.
- The console → launch → handoff remains one continuous Ratatui session on the console path, but the continuity is owned by a typed app-level `TerminalSession` rather than global flags; the capsule handoff still preserves the host alt-screen with `JACKIN_HOST_ALT_SCREEN`, and the in-container experience continues the same visual language. *(Manual-smoke criterion — a real-terminal flash is not catchable by `TestBackend` snapshots.)*
- The capsule has a `MuxMode` state machine, a generalized `Action` dispatch with tests, and a decomposed `daemon.rs`; the full/partial compositors share one helper.
- The capsule renders through Ratatui; the pane-body mechanism (`tui-term` vs a custom cell widget) is recorded in an ADR with benchmark evidence.
- Every rule in [TUI Design Decisions](/reference/tui-design-decisions/) still holds; snapshot tests show no render diff across the refactor.
- Reusable patterns live in <RepoFile path="crates/jackin-tui/Cargo.toml" /> as single implementations; no visual pattern that appears on multiple terminal surfaces has two copies or a runtime dependency on `src/console/widgets`. The confirm dialog, picker, tab strip, scrollbar, panel, hint bar, status/footer bar, and modal frame each have exactly one implementation all surfaces consume.
- Token and layout-helper unification lands without a rendering-backend change; full widget sharing follows the capsule's move to a Ratatui frame.
- Every repeatable visual pattern is a named `jackin-tui` component with a lookbook page rendering its states from the real widget code; no host/capsule drift remains for shared patterns.
- Lookbook previews are generated from `TestBackend` renders (or recorded SVG for motion) and regenerated in CI, so they cannot drift from the shipped components.
- No phase introduces an operator-visible behavior change.
