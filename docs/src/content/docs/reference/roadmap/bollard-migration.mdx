**Status**: Partially implemented — Phase 1 (lifecycle, cleanup, inspect, non-interactive exec, image inspect, and pull via Bollard) shipped; `docker build` and interactive `docker run -it` remain deferred to Phase 2.

## What Shipped (Phase 1)

Phase 1 introduced a `DockerApi` trait in <RepoFile path="src/docker_client.rs" /> with `BollardDockerClient` (production) and `FakeDockerClient` (tests). The following operations were migrated from `ShellRunner` CLI calls to the bollard 0.21 typed API:

- Container lifecycle: `inspect_container_state`, `remove_container`, `create_container`, `start_container`
- Network lifecycle: `create_network`, `remove_network`, `list_networks`, `inspect_network`
- Volume cleanup: `remove_volume`
- Container listing: `list_containers`
- Image management: `list_image_tags`, `remove_image`, `inspect_image_label`, `pull_image`
- Non-interactive exec (DinD readiness checks, tmux session listing): `exec_capture`

`ContainerState` was extended from 4 variants to 9, covering all states bollard can return (`Running`, `Paused`, `Restarting`, `Removing`, `Created`, `Dead`, `Stopped`, `NotFound`, `InspectUnavailable`). Error handling throughout cleanup and inspect paths now uses bollard's typed HTTP status codes — 404 for missing resources, transport errors propagated as `InspectUnavailable` — eliminating all Docker CLI stderr string-matching. Closed open review findings #13 and #14.

## What Stays CLI

- `docker build` — bollard's build API streams a tarball over HTTP and is meaningfully different to integrate; Phase 2.
- `docker run -it` (foreground agent attach with TTY) — `attach` via the API requires non-trivial TTY/raw-mode handling; Phase 2.
- `docker run --rm` for version extraction and one-shot commands — CLI is the appropriate tool here.
- `docker exec -it` for interactive tmux attach — inherently TTY-bound; CLI only.

## Phase 2 — Build and interactive attach (deferred)

Migrate `docker build` and interactive `docker run -it` / `docker exec -it` to bollard. Remains deferred — value-per-effort is lower than Phase 1 and these call sites have well-understood CLI shapes with smaller failure-mode risk. Pick up when a concrete review finding or operational pain motivates it.
