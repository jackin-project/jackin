**Status**: Resolved

## Problem

Operators who routinely want raw container output had to pass `--debug` on every invocation: `jackin --debug`, `jackin load --debug`, `jackin console --debug`. There was no way to make debug-by-default sticky for a shell session, account, or CI job, so the flag had to be remembered and added repeatedly.

## Why It Matters

- **Operator ergonomics.** A persistent verbosity preference is a one-time setup in unix shells (`RUST_LOG`, `NO_COLOR`, etc.). jackin's `--debug` was the odd one out.
- **CI / agent-loop debugging.** Long-running operator workflows that wrap `jackin` benefit from setting verbose mode once at the environment level, not threading a `--debug` flag through every wrapper.
- **No new TOML.** The configuration alternative (`jackin config debug enable`) would have added a new `ConfigCommand` arm, an `AppConfig` field, and an editor method â€” all to express a single boolean developer preference. An env var matches the unix idiom for verbosity knobs and avoids the new persistence surface.

## Resolution

`--debug` is a single `#[arg(global = true)]` flag on the top-level `Cli` struct in <RepoFile path="src/cli/mod.rs" /> with `env = "JACKIN_DEBUG"`. `global = true` makes clap accept and propagate the flag to every subcommand, so `jackin hardline --debug`, `jackin eject --debug`, `jackin purge --debug`, etc. all work without per-subcommand wiring. clap's `ArgAction::SetTrue` env semantics handle precedence and value parsing without bespoke code:

1. `--debug` on the command line â†’ `true`
2. CLI flag absent, env var set to a truthy value (`1`, `true`, `yes`, `on`, â€¦) â†’ `true`
3. CLI flag absent, env var set to a falsy value (`0`, `false`, `no`, `off`, empty string) â†’ `false`
4. CLI flag absent, env var unset â†’ `default_value_t = false`

## Operator usage

```bash
# Make debug sticky for a shell or user account
echo 'export JACKIN_DEBUG=1' >> ~/.zshrc

# Now every invocation defaults to debug-on â€” including all subcommands
jackin load
jackin console
jackin hardline
jackin eject agent-smith
jackin purge agent-smith
jackin                          # bare â†’ console (on a TTY)

# Disable for one invocation when env is sticky
JACKIN_DEBUG=0 jackin load

# CLI flag still wins; accepted before or after the subcommand
jackin load --debug
jackin hardline --inspect k7p9m2xq --debug
```

The `--help` output for every subcommand automatically advertises the env binding via clap's standard `[env: JACKIN_DEBUG=]` annotation, so the variable is discoverable through normal CLI exploration.

## Test strategy

Because `unsafe_code = "forbid"` rules out in-process `std::env::set_var` for testing env-driven clap parsing, the integration coverage in <RepoFile path="tests/cli_debug_env.rs" /> spawns the real binary with a controlled env and verifies:

- `--help` for `jackin load`, `jackin console`, and bare `jackin` advertises the `[env: JACKIN_DEBUG=]` annotation (proves the binding is correctly attached to `--debug` on every entry point).
- Truthy, falsy, and empty `JACKIN_DEBUG` values do not break clap parsing â€” the existing console-on-non-TTY error is reached, which means parsing succeeded.

clap's value-resolution semantics for `ArgAction::SetTrue` env vars (`FalseyValueParser`) are clap's contract and not retested here.
