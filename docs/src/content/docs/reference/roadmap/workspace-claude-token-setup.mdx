## What is shipped

- `[workspaces.X.claude].oauth_token` canonical slot on `AgentAuthConfig` — accepts `EnvValue::Plain` or `EnvValue::OpRef`, rejected at parse-time on `[codex]`.
- `[workspaces.X].op_account` field on `WorkspaceConfig` — pins every `op` invocation made on behalf of the workspace to a specific 1P account via `op --account <id>`.
- Env resolver synthesis: a non-empty canonical slot at any of the 3 Claude layers (global, workspace, workspace × role) materialises `CLAUDE_CODE_OAUTH_TOKEN` at the same layer; standard last-wins merge across layers.
- `host_claude::probe_claude_cli()` — host-side CLI probe.
- `host_claude::capture_setup_token()` — PTY-wrapped `claude setup-token` driver. Token captured into `secrecy::SecretString`, redactor swaps the matching span for `<redacted>` in the operator's stderr stream so the URL / instructions still display but the token never echoes.
- `OpWriteRunner` trait + `OpCli` impl: `item_create` (JSON template on stdin, secret never on argv) and `item_delete`. `item_edit_field` is intentionally unimplemented (no documented stdin form on upstream `op item edit`); rotation uses create + delete instead.
- `workspace::token_setup` orchestrator: `run_setup`, `run_revoke`, `run_doctor`, with `_with_runner` injection seams for hermetic unit tests. Stamps `notesPlain` with workspace / host / created / `expires_estimate` / `token_sha256_prefix`.
- Post-write SHA-256 round-trip validation gating config persistence; on validation failure the orphan 1P item is best-effort deleted and the bail message reports the cleanup outcome via the `OrphanCleanup` enum (`Deleted` / `UnparseableRef` / `DeleteFailed { hint }`). Recovery `op item delete <id> --vault <vault>` command is rendered via `OpReferenceParts::manual_delete_hint`.
- Rotate flow including same-`op://`-ref guard, prior-vault defaulting via `vault_for_rotate`, and a copy-pasteable manual-delete recovery hint when `delete_prior_op_item` fails.
- Revoke `--delete-op-item` is a hard error on literal-token slots and unparseable URIs (silent no-op would let the secret survive).
- In-container onboarding skeleton — `provision_claude_auth` writes `{"hasCompletedOnboarding":true}` to `account.json` under `OAuthToken` mode and bind-mounts it via `agent_mounts`; `credentials.json` is removed and never mounted.
- Expiry stamp cache + launch banner — `expiry_days_for_launch` reads `<cache_dir>/claude-token-expiry/<workspace>` and `tui::output::format_expiry_suffix` renders red/yellow/dim countdowns. Malformed-stamp errors surface a one-shot warning on the next launch.
- Deletion guards on the canonical slot — `jackin workspace env unset <ws> CLAUDE_CODE_OAUTH_TOKEN` and the TUI Environments tab `D` key both refuse while `auth_forward = oauth_token` is active, pointing the operator at `jackin workspace claude-token revoke`.
- CLI surface — `jackin workspace claude-token setup|rotate|revoke|doctor`, with `--vault`, `--item-name`, `--op-account`, `--reuse`, `--delete-op-item`. Help text in <RepoFile path="src/cli/workspace.rs" />.

## What is NOT yet shipped

- TUI integration — `[G] Generate token →` action on the auth panel modal so the operator never leaves `jackin console`. The auth panel today surfaces `jackin workspace claude-token setup <workspace> --vault <vault>` as a copy-pasteable hint instead of driving the orchestrator from inside the TUI.
- Pre-launch validity probe (a doctor-style structural check on every launch). The shipped expiry banner is best-effort; a real upstream-validity probe would catch externally-rotated tokens before the agent hits API 401.
- Apple Keychain backend (`keychain://` source) — gated on the cross-cutting [Credential Source Pattern](/reference/roadmap/credential-source-pattern/) landing.
- Bulk `--all` migration flag.

## Problem

Token mode (`auth_forward = "oauth_token"`) is the recommended choice for long-lived or concurrent jackin' sessions, but the operator setup path is currently a five-step manual chore:

1. Run `claude setup-token` on the host.
2. Copy the printed token from terminal output.
3. Open 1Password (or another secret store), create a new item, paste the value into the right field.
4. Edit `~/.config/jackin/config.toml` by hand to add `CLAUDE_CODE_OAUTH_TOKEN = "op://..."` to the right workspace-scoped `[env]` block.
5. Edit the same file again to set `auth_forward = "oauth_token"` under the matching `[workspaces.<ws>.claude]` block.

That flow has every classic foot-gun: secrets in shell history, secrets in scrollback, two coupled config edits, and zero validation that the final stack actually works until the next launch fails with `API Error: 401`.

The pain compounds for operators with several workspaces. Each workspace ideally wants its own long-lived token so revocation stays granular, but nobody wants to repeat the five steps five times.

## Why It Matters

- Manual steps are where secrets leak — clipboards, terminals, log capture, accidentally-committed dotfiles.
- jackin's value proposition is that the operator workflow is smooth. Token mode is *the* recommended mode for production-style usage but has the worst onboarding ergonomics of any auth mode shipped today.
- Several already-deferred items (granular 1P account scope, token rotation, expiry warnings) all become solvable if a single jackin-owned setup flow exists to anchor them.
- The current config shape couples two unrelated keys (`auth_forward` mode + a magic env-var name). A guided setup path is the natural opportunity to introduce a single canonical slot.

## Background

The [Claude auth strategy](/reference/roadmap/claude-auth-strategy/) roadmap shipped token mode (`auth_forward = "oauth_token"`) but deliberately did not wrap the upstream `claude setup-token` flow — the operator was expected to run the upstream command, copy the token by hand, and write it into config.

That call was correct for the minimum viable token-mode release. With token mode in production, three follow-on requirements have surfaced that the manual flow does not address:

1. **Workspace-scoped tokens** (one OAuth token per workspace, stored under a deterministic 1P name, revocable independently).
2. **Multi-1P-account operators** (token for `Personal` workspace lives in `Personal` vault; token for `Work` workspace lives in `Work` vault — `op` CLI default-account context cannot resolve both in the same session).
3. **Token lifecycle** (rotation before expiry, revocation on workspace delete, drift detection when 1P value rotated externally).

This proposal therefore reverses the original non-goal: jackin' should own a workspace-scoped setup, rotate, revoke, and validate flow, with the upstream `claude setup-token` invoked as the inner step that actually mints the token.

## Goal

1. One command — `jackin workspace claude-token setup` — moves a workspace from "no token configured" to "token mode active and end-to-end validated", with no copy-paste of secret material.
2. Tokens never touch shell history, scrollback, or any file on disk that jackin writes. The token transits jackin's process memory once and lands in 1Password (or an alternative secret backend).
3. Every subsequent token operation (rotate, revoke, doctor) is a single command and stays workspace-scoped.
4. The flow is idempotent: re-running it updates the existing 1P item in place rather than creating duplicates.
5. Multi-account 1P operators can pin a workspace to a specific `op` account so resolution is deterministic.
6. The TUI Auth tab gains a peer "Generate token" action so the operator never needs to drop to the CLI.

## Non-Goals

- Replacing or forking `claude setup-token`. jackin always shells out to the upstream binary; what jackin owns is the surrounding capture, storage, wiring, and validation.
- Generating tokens without operator interaction. The browser-based consent step in `claude setup-token` stays. jackin only removes the copy-paste *after* consent.
- Bringing token rotation onto a schedule (daemon, cron). Rotation is always operator-initiated by `jackin workspace rotate claude-token`. An optional pre-launch expiry banner is in scope; an automatic rotator is not.
- Creating a generic secrets platform. The setup flow is Claude-token specific; the underlying primitives (op-write, host-CLI probe, PTY-wrapped capture) are factored generically so a future `gh auth token` or `codex login` flow can reuse them, but nothing in this proposal ships a cross-runtime secret pipeline.
- Solving Codex equivalent. Codex has no analog to `CLAUDE_CODE_OAUTH_TOKEN`; this proposal is Claude-only.

## Design Options

### Option 1: Status quo (manual five-step setup)

Keep the current state. Documentation already explains the steps.

Pros:

- zero implementation work
- preserves the original "wrap nothing" stance from the token-mode spec

Cons:

- secret-in-clipboard / secret-in-scrollback risk on every setup
- two-key config edit invites partial states (mode set, env missing, or vice versa) that only fail at next launch
- no idempotency story for rotation
- no multi-account 1P story

### Option 2: Setup helper that prints instructions only

A `jackin workspace claude-token setup --plan` style command that prints a personalized step list for the current workspace, but does not perform the steps.

Pros:

- minimal new code
- avoids touching `op` write paths

Cons:

- still has every copy-paste foot-gun of Option 1
- false sense of automation; operators expect actions when commands exist with imperative verbs

### Option 3: Full guided setup with PTY-captured token + 1P write (recommended)

`jackin workspace claude-token setup` runs the full pipeline: host-CLI probe → PTY-wrapped `claude setup-token` → in-process token capture → `op item create`/`op item edit` (or alternative backend) → config wiring → end-to-end validation.

Pros:

- zero copy-paste of secret material
- single canonical 1P naming (`jackin · <workspace> · claude-token`) so revocation and rotation become trivial
- launch-time failure modes shifted to setup-time, where they are actionable
- naturally extends to other secret-bearing CLIs (`gh auth token`, `codex login` if/when it gains a non-interactive path)

Cons:

- requires a new `OpWriteRunner` trait (the existing `OpStructRunner` is read-only by SAFETY contract — see <RepoFile path="src/console/op_cache.rs" />)
- requires a PTY harness (`portable-pty` or equivalent) to host the interactive `claude setup-token` flow without echoing the captured token to the parent TTY
- adds a dependency on the host having Claude CLI installed, with install-hint plumbing for the case where it is missing

### Option 4: Use OS keychain instead of 1Password for storage

Store the captured token in Apple Keychain (macOS) or libsecret (Linux), with config referencing it via a future `keychain://<service>` source URI.

Pros:

- no 1P CLI dependency for solo operators
- aligns with the [Credential Source Pattern](/reference/roadmap/credential-source-pattern/) cross-cutting roadmap

Cons:

- needs the cross-cutting `CredentialSource` enum landed first
- operators who already standardise on 1P (the dominant cohort today) get nothing
- shared / team workspace scenarios benefit much more from a 1P shared-vault path than from per-host keychain entries

Ship as a follow-up source after Option 3, gated on the credential source pattern landing.

## Recommendation

Implement Option 3 now and treat Option 4 as a follow-up source on top of the same setup pipeline. The setup orchestrator is backend-agnostic; only the storage step changes when keychain support arrives.

## Suggested Operator UX

### CLI surface

```bash
# Interactive end-to-end (writes to 1P, edits config, validates)
jackin workspace claude-token setup

# Non-interactive: reuse an existing op-ref, skip generation
jackin workspace claude-token setup --reuse op://Personal/foo/token

# Pin to a specific 1P account (multi-account operators)
jackin workspace claude-token setup --op-account Personal

# Override default item naming
jackin workspace claude-token setup --item-name "jackin · {ws} · claude"

# Rotate: regenerate, overwrite same 1P field, update config slot if shape changed
jackin workspace rotate claude-token

# Revoke: zero out env entry + flip mode → ignore + (optionally) delete 1P item
jackin workspace revoke claude-token --delete-1p-item

# Doctor: validate without changing anything
jackin workspace doctor claude-token
```

`jackin workspace ...` is a new top-level subcommand group that implicitly targets the current-directory workspace; an explicit `--workspace <name>` flag overrides.

### TUI integration

The Auth tab in `jackin console` (<RepoFile path="src/console/widgets/auth_panel/render.rs" />) gains a single new action on a credential row when mode = `oauth_token` and no credential is set:

```text
[G] Generate new token →
```

Selecting it triggers the same orchestrator used by the CLI flow. The modal renders progress for steps 1–4 inline so the operator never leaves the console.

### Config shape

Introduce a single canonical slot inside the per-(workspace × agent) auth block:

```toml
[workspaces.chainargos-platform.claude]
auth_forward = "oauth_token"
oauth_token  = "op://Personal/jackin · chainargos-platform · claude-token/token"
```

The resolver synthesises `CLAUDE_CODE_OAUTH_TOKEN` from `oauth_token` at the **same layer** the slot was declared on, then runs the standard four-layer last-wins merge (global → role → workspace → workspace × role). Synthesis runs after the explicit `[env]` of the same layer, so when the operator declares both at the same layer the canonical slot wins on intra-layer collision; cross-layer ordering is unchanged. The "two-key" form (declaring `CLAUDE_CODE_OAUTH_TOKEN` directly inside `[workspaces.X.env]`) is just a normal env entry and behaves like every other env var:

```toml
# Two ways to spell the same thing — pick whichever block makes the
# auth wiring more obvious in your config.

# Canonical slot (used by `jackin workspace claude-token setup`):
[workspaces.chainargos-platform.claude]
auth_forward = "oauth_token"
oauth_token  = "op://Personal/jackin · chainargos-platform · claude-token/token"

# Or, equivalently, as a regular env entry:
[workspaces.chainargos-platform.claude]
auth_forward = "oauth_token"

[workspaces.chainargos-platform.env]
CLAUDE_CODE_OAUTH_TOKEN = "op://Personal/Anthropic/token"
```

There is no special "redundant entry" warning — per the pre-release schema rule in <RepoFile path="AGENTS.md" />, jackin does not ship tolerance handlers. If both forms are set at the same layer, the canonical slot wins (synthesis after explicit env); that is the entire collision contract.

A new optional field pins workspace 1P account scope:

```toml
[workspaces.chainargos-platform]
op_account = "Personal"  # forwarded as `op --account Personal ...`
```

### Launch-time messaging

The existing diagnostic line (see
<RepoFile path="src/tui/output.rs" />, function `format_auth_line`)
gains an expiry suffix when the canonical slot is in use:

```text
claude auth: OAuth token (oauth_token ← op://Personal/jackin · …/token, expires 2027-05-08)
```

Within 30 days of expiry, the suffix renders yellow; within 7 days, red. Beyond expiry, launch refuses unless `--force` is passed.

## Security Notes

- **PTY isolation.** `claude setup-token` runs under a `portable-pty` child. Stdout is consumed in-process, parsed for the token line, and never forwarded to the parent TTY or any tracing sink.
- **Heap minimization.** The token is held in a `secrecy::SecretString` for the entire pipeline; `Debug` and `Display` are scrubbed; the buffer is zeroed before the captured slot is dropped.
- **No process-arg leakage.** The token is *never* passed to `op` via argv — `op item create` reads it from stdin (`op item create … password=- < <SecretString>`).
- **Provenance metadata only.** The 1P item stores a SHA-256 prefix (first 12 hex chars) of the token in `notesPlain`; `jackin workspace doctor claude-token` recomputes the prefix from the retrieved value to detect external rotation. The full hash is *never* recorded — prefix-only avoids constructing a confirmation oracle while still letting drift be visible.
- **Filesystem hygiene.** The orchestrator emits zero temp files. If PTY capture fails midway, the partial child process is killed and the operator instructed to re-run; no half-written 1P item or config edit persists.
- **Revoke completeness.** `revoke` flips mode to `ignore`, removes the canonical slot, and (optionally) deletes the 1P item. It does *not* call any Anthropic-side revocation API — that surface does not exist publicly today; the operator must rotate via Anthropic dashboard if true revocation is required.

## Open Questions

1. **Token-line parsing contract.** Is the upstream output of `claude setup-token` stable enough to regex-parse, or should the wrapper require a `--print-token-only` upstream flag? If the latter, should jackin ship a small upstream PR to Anthropic?
2. **Concurrent setup safety.** Two `jackin workspace setup claude-token` runs against the same workspace at once — should the orchestrator hold a flock on `~/.config/jackin/config.toml`?
3. **Default `op_account` discovery.** When the operator has multiple 1P accounts and no workspace pin, should the wizard prompt every time, or remember the last choice per workspace?
4. **Doctor verbosity.** Should `doctor` print the `op://` reference in full, partially redacted, or only the resolution result? (Full path makes audits easy; partial redaction is friendlier in shared-screen demos.)
5. **Apple Keychain follow-up timing.** Does Option 4 (keychain backend) ship in the same release or wait for the [Credential Source Pattern](/reference/roadmap/credential-source-pattern/) to land first?
6. **Bulk migration.** Is `jackin workspace claude-token setup --all` worth shipping in v1, or does serial per-workspace remain sufficient until adoption demands batch?
