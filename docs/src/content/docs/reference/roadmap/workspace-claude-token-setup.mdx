## What Shipped: `op_picker` Create Mode + Plain-Prompt `--interactive`

Two pieces shipped together, each on the surface where it belongs.

**Shared `op_picker` gained an opt-in Create mode.** <RepoFile path="src/console/widgets/op_picker/mod.rs" /> now exposes `OpPickerMode::Create` alongside the default `Browse`. In Create mode the Item stage renders a `+ New item` sentinel and the Field stage renders `+ New field` and `+ New section` sentinels at the bottom of their lists (the standard `+ Add X` creation-sentinel convention), with naming sub-stages that reuse the shared `text_input` box. The picker commits an `OpPickerSelection` enum — `Existing(OpRef)` for an existing field, `NewItem { .. }` for a freshly named item, or `EditItemField { .. }` for a new field/section on an existing item. Browse mode is unchanged, so every existing caller keeps drilling Account → Vault → Item → Field exactly as before. This replaced an earlier separate rich-dialog widget that had duplicated the drill-down state machine; the shared widget is now the single implementation.

**The CLI `--interactive` flow uses plain CLI prompts.** `jackin workspace claude-token setup <ws> --interactive` walks the operator through account → vault → item → field using plain dialoguer Select/Input prompts — no rich ratatui TUI. At the item step the operator picks an existing item or `[ + New item ]`; at the field step they pick an existing field (token written in-place via `item_field_set`) or `[ + New field ]`. The flag is mutually exclusive with `--vault` and `--reuse`.

The principle this establishes: a rich ratatui TUI lives only inside `jackin console`; the CLI stays plain CLI. Scripted invocations keep using `--vault` / `--reuse`; `--interactive` is the plain-prompt path for an operator at a shell.

## What Shipped: Console Auth-tab Generate Action

The workspace editor's Auth tab can now mint a Claude token without leaving `jackin console`. On the workspace-level Claude `oauth_token` auth form, `G` opens the shared `op_picker` in Create mode so the operator chooses (or creates, via `+ New item` / `+ New field` / `+ New section`) the 1Password location. On commit, the `run_console` loop suspends the TUI (leaves raw-mode + alt-screen, mirroring the terminal guard), runs `token_setup::run_setup` — the same orchestrator the CLI uses, which drives the interactive `claude setup-token` OAuth capture, writes to 1Password, and wires the workspace config — then resumes the TUI and syncs the editor so the Auth row reflects the wired credential. Errors surface through the standard `ErrorPopup`; success is silent (the populated row is the confirmation).

This is the console counterpart to the CLI `--interactive` storage picker. Role-level generate (the form's per-role override layer) and a plain-text generate target are not yet wired — the action is gated to the workspace-level Claude `oauth_token` slot. The suspend/resume-around-PTY mint is exercised only by live use; it has no headless test.

## Remaining Work

1. **Canonical auth slot.** Replace the current two-key implementation (`auth_forward = "oauth_token"` plus managed `CLAUDE_CODE_OAUTH_TOKEN` env) with a dedicated `oauth_token` field on the Claude auth config. The launcher should synthesize `CLAUDE_CODE_OAUTH_TOKEN` from that slot.

2. **Pre-launch validity probe.** Add a lightweight validity check or cached doctor result so obviously invalid external rotations surface before Claude returns an API 401 inside the session.

3. **Apple Keychain backend.** Add a local OS-secret-store source after the cross-cutting [Credential Source Pattern](/reference/roadmap/credential-source-pattern/) lands.

4. **Bulk migration.** Add a `--all` or equivalent workflow for migrating multiple workspaces to Claude OAuth-token mode.

## Design Notes

- The upstream `claude setup-token` browser consent remains the token minting step; jackin' owns capture, storage, config wiring, validation, and lifecycle commands.
- The canonical slot should be the future user-facing shape. The current managed env entry is an implementation bridge, not the target schema.
- Any future storage backend must preserve the current safety properties: no token in argv, no token in shell history, no config write before storage validation succeeds.
- The shared `op_picker` Create mode is the committed shape for in-place creation: the `+ New item` / `+ New field` / `+ New section` sentinels, their `text_input`-backed naming stages, and the `OpPickerSelection` commit enum (`Existing` / `NewItem` / `EditItemField`). The console Auth-tab generate action in Remaining Work item 1 reuses this widget rather than reintroducing a parallel one.
