## Problem

<RepoFile path="PROJECT_STRUCTURE.md" /> drifted since PR #166 because there is no automated check. PRs that add new `.rs` module files can merge without mentioning them in the navigation map.

## Implementation

Add a step to the `check` job in <RepoFile path=".github/workflows/ci.yml" />:

```yaml
- name: Check PROJECT_STRUCTURE.md covers new modules
  run: |
    missing=()
    while IFS= read -r f; do
      stem="${f%.rs}"
      dir="${stem%/mod}"
      basename="${dir##*/}"
      if ! grep -qF "$basename" PROJECT_STRUCTURE.md 2>/dev/null; then
        missing+=("$f")
      fi
    done < <(git diff --name-only origin/main...HEAD -- 'src/**/*.rs')
    if [ ${#missing[@]} -gt 0 ]; then
      printf 'Not mentioned in PROJECT_STRUCTURE.md: %s\n' "${missing[@]}"
      exit 1
    fi
```

## Design decisions

- **Scoped to PR diff only** — checks only files added in the current PR, not all 94 existing files. Avoids requiring retroactive fixes before the gate can be enabled.
- **Greps for basename** (e.g., `op_picker` for `src/console/widgets/op_picker/mod.rs`) — PROJECT_STRUCTURE.md uses prose descriptions containing the module name.
- **False negative risk**: a file named `mod.rs` won't be caught (too generic). Only named modules are checked.

## Steps

1. Do the project structure update item first (fix existing staleness before enabling the gate).
2. Add the YAML step above to <RepoFile path=".github/workflows/ci.yml" /> after the `cargo fmt --check` step.
3. Add a note to <RepoFile path="CONTRIBUTING.md" />: "When adding a new `.rs` module file, update PROJECT_STRUCTURE.md."
4. Verify the step passes on the current branch before merging.

## Caveats

The script uses `git diff --name-only origin/main...HEAD` which requires the repo to have `origin/main` available in CI — already true since `actions/checkout` fetches it for the PR check job.
