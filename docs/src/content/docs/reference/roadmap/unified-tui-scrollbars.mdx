## Problem

jackin' has multiple terminal scrollbar paths that share the same visual intent but not the same state model, clamp policy, geometry, or hit-testing rules. The visible failure is the Docker build log overlay: scrolling up can leave a stored value beyond the current maximum, rendering clamps only a temporary top-offset, and the first downward wheel events are spent consuming invisible overshoot instead of moving visible content. The fix is one shared scroll model used by renderers, keyboard input, mouse wheel handling, drag/click hit testing, launch progress, and capsule scrollback.

<Aside type="note">
CSS/browser scrollbars in the docs landing page and Starlight theme are out of scope. This item covers ratatui and raw-terminal scrollbars only.
</Aside>

## Final Approach

Use [`tui-scrollbar`](https://crates.io/crates/tui-scrollbar) for scrollbar metrics and interaction. Add it at the `jackin-tui` boundary and compose its `ScrollLengths`, `ScrollMetrics`, `ScrollBarInteraction`, and `ScrollCommand` types instead of reimplementing those concepts locally.

Keep jackin-owned rendering. Do not adopt `tui-scrollbar`'s fractional renderer, arrow endcaps, glyph sets, or default styles as the default output. The shared code owns state, metrics, clamping, drag/click mapping, and offset conversion only. It must not make horizontal and vertical scrollbars visually identical.

Preserve the current visual split by design. Console horizontal scrollbars continue to render as a bottom-border-adjacent dotted track with a `━` thumb. Console vertical scrollbars continue to render as a right-side dotted track with a full-cell `█` thumb. Capsule scrollbars continue to paint only thumb cells into the pane's right border column so the border itself remains the track. This asymmetry is intentional and must be covered by surface tests.

Use one canonical coordinate model per axis. Normal content panels store top-relative offsets: `0 = first row/column`, `max = last reachable row/column`. Live-tail surfaces use a small `TailScroll` adapter in `jackin_tui::scroll` that preserves `0 = newest/live tail` externally while clamping through `tui-scrollbar` metrics before every render or mutation.

Clamp on every mutation. Wheel, keyboard, page movement, drag, click, resize, content refresh, cursor-follow updates, and render entry all write back a clamped value. Rendering may clamp defensively, but it must not be the only clamp.

Do not create a parallel local metrics API. `jackin_tui::scroll` is an integration layer: constructors from jackin' viewport/content values, `TailScroll`, full-cell thumb conversion for existing visuals, and small helpers that compose `tui-scrollbar` rather than replacing it.

## Implementation Steps

1. Add `tui-scrollbar` to `jackin-tui`.
2. Add `jackin_tui::scroll` with `TailScroll`, constructors for `tui_scrollbar::ScrollMetrics`, full-cell thumb conversion, clamped delta/page helpers, and cursor-follow helpers.
3. Move pure geometry/clamp tests from console/capsule into `jackin-tui` and add coverage for zero viewport, content fits, one-row overflow, exact max offset, overscroll clamp, midpoint drag, thumb end reach, tail-relative conversion, and cursor-follow selection.
4. Convert <RepoFile path="src/console/widgets/scrollable.rs" /> to use `jackin_tui::scroll` while preserving existing visuals.
5. Convert manager render/input/state paths listed in the affected inventory.
6. Convert the launch build-log overlay to `TailScroll` and add the sticky-top regression test.
7. Convert capsule scrollback and border-column rendering to shared metrics.
8. Update <RepoFile path="docs/src/content/docs/reference/tui-design-decisions.mdx" /> to name the new source of truth.

## Acceptance Criteria

- All TUI scrollbars use `tui-scrollbar` metrics through `jackin_tui::scroll` for thumb length, thumb position, max offset, clamping, and drag/click mapping.
- No scroll offset remains above the current maximum after wheel, key, page, drag, click, resize, content refresh, cursor-follow, or render.
- The Docker build log overlay no longer sticks at the top: with a saturated top value, the first downward wheel event moves visible content toward the tail.
- Horizontal and vertical scrollbar behavior is consistent across list view, editor tabs, settings tabs, picker lists, confirm-save details, launch progress, and capsule panes.
- Existing visual styling remains stable: focused scrollable blocks keep green focus, inactive tracks keep the dialog-scroll palette, console horizontal scrollbars keep dotted tracks with `━` thumbs, console vertical scrollbars keep dotted tracks with `█` thumbs, and capsule scrollbars keep right-border-column thumb-only rendering.
- Horizontal and vertical scrollbars are not normalized into a single visual treatment. Their metrics are shared; their visual representation remains intentionally different.
- Fractional glyphs, arrows, and other `tui-scrollbar` renderer visuals are not introduced by this refactor.
- Pure geometry tests live in `jackin-tui`; surface tests cover symbols, palette, focus, hit areas, state routing, and the build-log sticky-top repro.
