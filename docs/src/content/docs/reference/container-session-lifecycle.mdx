## Overview

Every jackin container runs `jackin-container` as PID 1. The daemon owns the PTY sessions, renders the in-container multiplexer, serves `/run/jackin/jackin.sock`, and exits when the last live session ends. The host attaches with a separate `docker exec -it <container> /usr/local/bin/jackin-container` client.

There are two session types:

| Session type | Agent slug | Command source |
|---|---|---|
| Agent pane | `claude`, `codex`, `amp`, `kimi`, or `opencode` | `/jackin/runtime/entrypoint.sh` |
| Shell pane | none | `/bin/zsh` |

`JACKIN_AGENT` is per agent pane. The daemon sets it only when spawning `/jackin/runtime/entrypoint.sh`; shell panes do not receive it.

## Container Startup (`jackin load`)

When `jackin load` provisions a new instance, the container launch happens in two steps:

1. `docker run -d ... <image> <agent>` starts the container detached. The image entrypoint is `/usr/local/bin/jackin-container`; the trailing `<agent>` argv tells PID 1 which initial agent tab to spawn. It is not exported as a container environment variable.
2. jackin checks that PID 1 did not exit before attach. If it exited, jackin captures the last 40 lines of container logs and surfaces an actionable error.
3. jackin runs `docker exec -it <container> /usr/local/bin/jackin-container`. The client connects to the daemon socket and blocks the operator terminal for the duration of the attach.

Implementation: <RepoFile path="src/runtime/launch.rs" /> (`launch_role_runtime`, `diagnose_premature_exit`) and <RepoFile path="crates/jackin-container/src/main.rs" /> (`resolve_initial_agent`).

## Reconnecting (`jackin hardline`)

`jackin hardline` inspects the target container and chooses the reconnect path:

- **Container is running**: attach with `docker exec -it <container> /usr/local/bin/jackin-container`.
- **Container is running and the operator requested a new agent session**: run `docker exec --workdir <workdir> -it <container> /usr/local/bin/jackin-container new <agent>`. The agent slug travels as argv; `JACKIN_AGENT` is set later inside the spawned PTY.
- **Container is running and the operator requested a shell**: run `docker exec -it <container> /usr/local/bin/jackin-container new`. No agent slug means a shell pane.
- **Container is stopped with crash state**: restore the required DinD sidecar, network, and certs when needed, restart the role container, then attach.
- **Container is gone but indexed recoverable state exists**: rebuild the runtime around jackin-managed local state.

Implementation: <RepoFile path="src/runtime/attach.rs" /> (`reconnect_or_create_session`, `spawn_agent_session`, `spawn_shell_session`).

## Session Inventory

jackin queries the daemon through its socket by executing:

```bash
/usr/local/bin/jackin-container status
```

The status output is line-oriented and parsed by the host into `AgentSession` records. If the container is not running, inventory is reported as `sessions:not_running` without contacting Docker.

Implementation: <RepoFile path="src/runtime/attach.rs" /> (`inspect_agent_sessions`) and <RepoFile path="crates/jackin-container/src/client.rs" /> (`run_status`).

## Container Shutdown

### Normal Exit

1. An agent or shell process exits.
2. The daemon removes the pane and redraws the multiplexer.
3. If no live sessions remain, PID 1 drains the final frame and exits.
4. The host finalizer observes the stopped container and runs cleanup:
   - `docker rm -f <role-container>`
   - `docker rm -f <dind-container>`
   - `docker volume rm <certs-volume>`
   - `docker network rm <network>`
5. Keep-awake state is reconciled after the managed containers are gone.

### Detach

When the operator detaches or the terminal closes while sessions are still alive, the attach client exits but PID 1 and the session PTYs continue running. `jackin hardline` can reconnect to the daemon later. The DinD sidecar, network, and certs volume remain while the role container is preserved.

### Crash Or Forced Stop

If the role container exits non-zero or is OOM-killed, jackin records crash state and removes the DinD sidecar, network, and certs volume. If Docker sends `SIGTERM` to the role container, PID 1 shuts down the daemon and sessions as the container namespace exits.

## What Persists Across Session Exit

| Thing | Persists? | Notes |
|---|---|---|
| Agent conversation history | Yes | Stored in the per-instance durable home mount on the host |
| Agent auth tokens | Yes (per auth mode) | Depends on the configured auth forwarding mode |
| Files written in the mounted workspace | Yes | Host-side mount |
| In-container packages installed ad hoc | No | Writable layer is lost on container stop/removal |
| DinD images | No | DinD state is recreated with the sidecar |
| Multiplexer session state | No | Sessions are recreated by `hardline --new` or the next `load` |

## Interaction Diagram

```text
jackin load
  -> docker run -d ... <image> claude
       -> jackin-container PID 1 spawns initial Claude PTY
  -> docker exec -it <container> jackin-container
       -> attach client renders the daemon session

running container
  -> hardline
       -> docker exec -it <container> jackin-container
  -> hardline --new --agent codex
       -> docker exec -it <container> jackin-container new codex
       -> daemon spawns Codex PTY with JACKIN_AGENT=codex
  -> hardline --shell
       -> docker exec -it <container> jackin-container new
       -> daemon spawns zsh PTY without JACKIN_AGENT

last live PTY exits
  -> daemon exits
  -> Docker reports container stopped
  -> host cleanup removes role container, DinD, cert volume, and network
```
